BLE 廣播與連線機制全解析

理解 BLE 如何在擁擠的 2.4GHz 頻段中「聰明生存」，並建立穩定的點對點連線。

📢 1. 廣播機制：我是誰？我在哪？

在連線建立之前，設備必須主動宣告自己的存在。這就是廣播 (Advertising)。這就像在喧鬧的派對中大喊自己的名字。

📡 避開 Wi-Fi 干擾

BLE 廣播固定在頻道 37, 38, 39。這三者完美避開了 Wi-Fi 常用的頻道 1, 6, 11，確保訊號能被快速搜尋。

📦 封包載荷 (Payload)

包含設備名稱、MAC 地址與最重要的 Service UUID。這讓手機一眼看出「我是一台遙控車」而非藍牙耳機。

⏱️ Adv Interval

  • ● 靈敏模式 (20ms): 極速被搜尋，但功耗顯著。
  • ● 省電模式 (1s): 極致續航，但搜尋需等待。
  • 💡 廣播是 BLE 最耗電的行為之一。

🤝 2. 掃描與連線流程

1
掃描 (Scanning)

手機 (Central) 在廣播頻道間切換，傾聽空中的廣播封包。

2
連線請求 (Req)

手機送出時序指令：「之後我們建立專屬頻跳頻，別再廣播了」。

3
獨佔連線

進入 1 對 1 高速交換模式。這就是為什麼 A 同學連線後，B 同學就看不到該車。

🔁

跳頻技術 (AFH)

連線後，設備會在剩餘的 37 個數據頻道中高速跳轉。這讓 BLE 即使在 Wi-Fi 全開的環境下依然穩健，不受干擾。

🔋 3. 工作週期：省電的關鍵

BLE 的低功耗奇蹟，源自於對工作週期 (Duty Cycle) 的極致管理。

限時廣播模式

開機僅廣播 60 秒，若無人連線則關閉藍牙進入休眠。下次需實體 Reset 喚醒。

事件觸發模式

僅在按下「配對紐」時才啟動廣播，平常完全關閉天線，達成年級別的待機續航。

"BLE 並非傳輸不快，而是絕大多數時間都在睡覺。"

核心任務：掌控廣播與連線的藝術

從修改封包標識開始，到實作具有場域感知能力的硬體連線回饋。

🆔 30 min

【作業 5-1】

「身份識別封包實戰」

修改 BLE 名稱包含個人座號。利用 nRF Connect 解析空中封包，記錄該設備的 MAC 地址與 UUID 關鍵特徵。

🔦 45 min

【作業 5-2】

「自動重連狀態控制」

實作 Server Callbacks。連線亮綠燈，斷線亮紅燈並自動呼叫 startAdvertising。確保系統具備斷線自我修復能力。

📏 45 min

【作業 5-3】

「RSSI 場域感知應用」

記錄不同距離下的訊號強度。實作「極近快閃提示」邏輯，並探討環境遮擋對無線通訊穩定度的劇烈影響。

BLE 廣播與連線機制實驗室

任務 1：「改名大師」：自定義廣播封包 (Name Master: Advertising)

角色扮演：品牌設備工程師 | 難度：⭐️ | 估計時長：30 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 識別標籤設定：修改 BLEDevice::init() 的呼叫名稱，精確包含個人座號（如 Vibe_Car_07）。
  3. 空中抓包分析：使用手機 nRF Connect App 掃描設備，截圖並記錄 MAC 地址與 UUID 關鍵參數。
  4. 記錄與提交：在 README 中彙整截圖，簡述此名稱是如何在連線前被「宣告」出去的。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 識別正確性：設備名稱是否符合格式規範且能在眾多設備中被唯一辨識？
  • [ ] 參數理解：能否從掃描結果中正確讀取該設備實體層的 MAC 地址？

任務 2：「連線守門員」：Server 回呼機制 (Connection Guard: Callbacks)

角色扮演：通訊系統開發員 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 回呼類別實作：繼承 BLEServerCallbacks 並實作 onConnect 與 onDisconnect。
  2. 狀態同步回饋：撰寫邏輯使連線時綠燈常亮，斷線時紅燈急閃並在 Serial 印出斷線日誌。
  3. 自動修復廣播：在 onDisconnect 函式結尾加入 BLEDevice::startAdvertising()，確保斷線能立即重新被搜尋。
  4. 最終 Commit：封裝 Server 事件管理模組並 Push。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 狀態實時性：板端的硬體回饋 (LED/Serial) 是否與連線狀態保持高度同步？
  • [ ] 斷線修復能力：斷線後是否具備「零手動介入」的自動重新廣播功能？

任務 3：「距離與強度的秘密」：RSSI 測距實驗 (RSSI Distance Lab)

角色扮演：場域感知工程師 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 梯度採樣記錄：分別在距離 10cm、1m、3m 處記錄並計算 RSSI 的平均值變化。
  2. 臨界觸發實作：實作邏輯：若 pServer->getRSSI() > -50dBm，則快閃 LED 提示進入「極近場域」。
  3. 實驗總結撰寫：在 README 中彙整「RSSI vs 距離」對應表，並論述環境遮擋對無線訊號強度的決定性干擾。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 數據趨勢性：所記錄的 RSSI 數據是否能正確反應無線訊號隨距離衰減的物理規律？
  • [ ] 應用邏輯準確度：近場閃爍提示是否運作正常，且能與長距離訊號做有效區隔？

GitHub Classroom 詳細操作指南

本單元教學重點：掌握 BLE 生態系中「發現 -> 握手 -> 維護」的核心生命週期。

  1. 重點 1：宣告即存在 - 引導學員理解廣播封包的實體意義，與為何需要 MAC 指標。
  2. 重點 2：非對稱式通訊 - 雖然連線是 1 對 1，但 Server 必須在斷線後主動重啟廣播，這是初學者最常忽略的可靠性設計。
  3. 重點 3：無線物理特性 - 利用 RSSI 實驗，建立學員對於 2.4GHz 頻段不穩定性與干擾源的正確工程觀念。
