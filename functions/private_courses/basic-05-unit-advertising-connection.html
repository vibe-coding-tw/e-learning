簡介

BLE 設備透過廣播封包（Advertising）在固定頻道上宣告自身存在，手機掃描後發起連線請求，建立專屬的 1 對 1 通訊。 本單元將帶你掌握廣播封包的結構、掃描與連線的三步握手流程，以及如何透過限時廣播與事件觸發策略優化功耗管理。

廣播機制：宣告存在

BLE 設備在尚未與任何 Client 建立連線之前，會透過廣播（Advertising）機制向周圍宣告自身的存在。 這就像是一家新開的店面掛出招牌——路過的人（手機）看到招牌後，才能決定是否要進門（建立連線）。

廣播是一種單向、無連線的通訊方式。ESP32 會週期性地在特定頻道上發送廣播封包（Advertising Packet）， 任何處於掃描狀態的設備都可以接收到這些封包，但此時雙方並未建立專屬的通訊通道。

廣播頻道設計

BLE 使用 2.4 GHz ISM 頻段，共劃分為 40 個頻道（編號 0–39）。其中，頻道 37、38、39 被指定為專用廣播頻道， 其餘 37 個頻道用於連線後的數據傳輸。

廣播頻道 中心頻率 設計用意
Ch 37 2402 MHz 避開 Wi-Fi Ch 1 的干擾峰值
Ch 38 2426 MHz 位於 Wi-Fi Ch 1 與 Ch 6 之間的間隙
Ch 39 2480 MHz 避開 Wi-Fi Ch 11 的干擾峰值
備註 BLE 的廣播頻道特別選在 Wi-Fi 頻道的「間隙」位置，大幅降低了與 Wi-Fi 的互相干擾。這也是為何即使在 Wi-Fi 密集的環境中，BLE 廣播仍能被快速搜尋到的原因。

廣播封包結構

每個廣播封包的最大長度為 31 bytes（BLE 4.x）或 255 bytes（BLE 5.0 Extended Advertising）。 在有限的空間中，開發者需要精心選擇要在封包中攜帶哪些資訊：

  • Flags（必要）：標示設備類型與發現模式
  • Local Name：設備名稱，如 "Vibe-Car-07"
  • Service UUID：公開的服務識別碼，讓手機篩選目標
  • TX Power Level：發射功率，用於估算距離
  • Manufacturer Data：廠商自定義數據

ESP32 廣播配置

// 初始化 BLE 並設定設備名稱
BLEDevice::init("Vibe-Car-07");

// 取得廣播物件並配置封包內容
BLEAdvertising *pAdvertising = BLEDevice::getAdvertising();
pAdvertising->addServiceUUID(SERVICE_UUID);
pAdvertising->setScanResponse(true);
pAdvertising->setMinPreferred(0x06);

// 開始廣播
BLEDevice::startAdvertising();
提示 將設備名稱加入座號（如 "Vibe-Car-07"）是課堂中區分多台 ESP32 的最佳做法。在 nRF Connect 中，你可以透過名稱快速找到自己的設備。

掃描與連線

當手機開啟藍牙掃描時，它會在三個廣播頻道上被動傾聽（Passive Scanning）或主動詢問（Active Scanning）廣播封包。 一旦發現目標設備，手機便可發起連線請求（Connection Request），將雙方從「廣播模式」切換到「連線模式」。

連線建立三步流程

1

掃描（Scanning）

手機（Central）在廣播頻道上傾聽封包。若使用 Active Scanning，還會發送 Scan Request 要求設備回傳額外資訊（Scan Response）。

2

連線請求（Connection Request）

手機選定目標後，在廣播頻道上發送 CONNECT_REQ 封包，內含連線參數（間隔、延遲、超時等），ESP32 收到後停止廣播。

3

獨佔連線（Connected State）

雙方進入 1 對 1 的專屬連線。數據傳輸改用其餘 37 個數據頻道，並透過跳頻（Frequency Hopping）避免干擾。此時其他手機無法看到該設備。

備註 BLE 連線是排他性的——一旦 ESP32 與一台手機建立連線，它就會停止廣播，其他設備無法發現它。若需要支援多裝置連線，需使用 BLE 5.0 的 Multi-Connection 特性或手動管理重新廣播。

Server Callbacks 事件處理

ESP32 作為 BLE Server，可以透過繼承 BLEServerCallbacks 類別來監聽連線與斷開事件。 這是實現「硬體狀態同步」的關鍵——讓 LED 即時反映藍牙連線狀態：

class MyServerCallbacks : public BLEServerCallbacks {
  void onConnect(BLEServer* pServer) {
    digitalWrite(LED_PIN, HIGH); // 連線時亮燈
    Serial.println("Client connected!");
  }

  void onDisconnect(BLEServer* pServer) {
    digitalWrite(LED_PIN, LOW); // 斷線時熄燈
    pServer->startAdvertising(); // 重啟廣播！
  }
};
警告 onDisconnect 中必須呼叫 pServer->startAdvertising() 重新啟動廣播。若省略此步驟，手機斷線後設備將永遠「消失」，只能透過重新上電才能恢復——這是學員最常犯的錯誤。

連線參數

連線建立時，Central 會提議一組連線參數，影響通訊的即時性與功耗：

參數 說明 典型值
Connection Interval 兩次通訊事件之間的間隔 7.5ms – 4000ms
Slave Latency Peripheral 可跳過的通訊事件次數 0 – 499
Supervision Timeout 無通訊多久後判定斷線 100ms – 32s

工作週期與功耗管理

BLE 之所以被稱為「低功耗藍牙」，核心秘密在於其精心設計的工作週期（Duty Cycle）管理機制。 設備不需要持續廣播或保持連線——透過合理的週期配置，可以將電池壽命從數小時延長到數月甚至數年。

對於無人車應用，雖然電池容量通常較充裕，但理解功耗管理仍然至關重要——尤其是在設備閒置待命的場景中。

廣播間隔（Advertising Interval）

廣播間隔決定了 ESP32 多久發送一次廣播封包。間隔越短，手機越容易快速發現設備，但功耗也越高：

廣播間隔 發現速度 功耗 適用場景
20ms – 100ms 極快（<1秒） 高 需要立即被發現（展示、教學）
100ms – 500ms 快（1–3秒） 中 一般消費產品
1000ms – 10000ms 慢（3–30秒） 極低 資產追蹤、長待機 Beacon

功耗優化策略

以下兩種策略可大幅降低廣播階段的電力消耗：

策略一：限時廣播（Timed Advertising）

設備開機後僅廣播固定時間（如 60 秒），若無人連線則自動進入深度休眠（Deep Sleep）。 使用者按下按鈕時才喚醒並重新廣播：

// 設定廣播持續 60 秒後自動停止
unsigned long advStartTime = millis();
const unsigned long ADV_TIMEOUT = 60000; // 60 秒

void loop() {
  if (!connected && millis() - advStartTime > ADV_TIMEOUT) {
    Serial.println("No connection. Entering deep sleep...");
    esp_deep_sleep_start();
  }
}

策略二：事件觸發廣播（Event-Triggered Advertising）

設備平時處於休眠狀態，僅在外部事件（如按下按鈕、感測器觸發）發生時才啟動廣播。 ESP32 支援 GPIO 作為喚醒源，即使在 Deep Sleep 下也能監聽引腳狀態：

// 設定 GPIO 0 (BOOT 鍵) 為喚醒源
esp_sleep_enable_ext0_wakeup(GPIO_NUM_0, 0); // LOW 觸發
esp_deep_sleep_start();
提示 ESP32 在 Deep Sleep 模式下的電流消耗僅約 10μA，相較於正常運行時的 80–240mA，省電效果超過 10000 倍。對於電池供電的遙控裝置，合理使用休眠機制是延長續航的關鍵。

RSSI 信號強度感知

RSSI（Received Signal Strength Indicator）是手機接收到 BLE 廣播封包時量測的信號強度，單位為 dBm。 RSSI 隨距離增加而衰減，可用於粗略估算設備間的物理距離：

  • -30 ~ -50 dBm：極近距離（10cm 以內）
  • -50 ~ -70 dBm：近距離（0.5m – 2m）
  • -70 ~ -90 dBm：中距離（2m – 5m）
  • < -90 dBm：遠距離或訊號微弱
警告 RSSI 受環境影響極大——人體遮擋、金屬反射、多路徑效應都會造成讀數波動。切勿將 RSSI 視為精確距離量測工具，它只適合做「近場/遠場」的粗略判斷。實務中建議取 10 組以上的平均值來減少雜訊。

BLE 廣播機制與連線流程實驗

本實驗旨在引導學員掌握 BLE (Bluetooth Low Energy) 的核心運作機制，包括設備廣播、連線事件處理以及 RSSI 信號感知。透過本實驗，你將學習如何讓硬體根據連線狀態做出反應，並理解無線信號在空間中的傳播規律。

🎯 實驗目標

  1. 自定義廣播封包：修改設備名稱與 UUID，並使用偵錯工具驗證。
  2. 實作連線回呼 (Callbacks)：處理連線與斷開事件，確保設備具備自動恢復廣播的能力。
  3. RSSI 空間交互：記錄不同距離下的信號強度，並實作基於強度的近場互動。

🛠 實驗任務

任務 1：「改名大師」自定義廣播封包

  • 目標：修改 include/ble_app.h 中的 DEVICE_NAME 為 Vibe-Student-XX (XX 為你的座號)。
  • 驗證：使用手機 App（如 nRF Connect）掃描到你的設備，並記錄其 MAC 地址。

任務 2：「連線守門員」Server 回呼機制

  • 目標：在 src/main.cpp 中補全 BLEServerCallbacks 的 onConnect 與 onDisconnect 邏輯。
    • 連線時：板載 LED 恆亮。
    • 斷開時：板載 LED 閃爍，且必須調用 pServer->startAdvertising() 以重啟廣播。
  • 驗證：測試連線後再斷開，手機是否能立即再次搜尋到該設備。

任務 3：「距離強度」RSSI 測距實驗

  • 目標：在 10cm, 50cm, 150cm 處分別記錄 RSSI 值。
  • 加分題：實作當 RSSI 大於 -50dBm 時，觸發 LED 快速閃爍。

任務 3：「距離強度」RSSI 測距實驗 (RSSI Distance Lab)

  • 目標：體會無線信號隨物理空間衰減的非線性規律，實作基於信號強度的「空間交互」。
  • Classroom 部署建議：
    • 數據採樣量測：分別在 10cm, 50cm, 150cm 處記錄 10 組 RSSI 平均值。
    • 近場提示 (Proximity Recall)：撰寫邏輯，當 RSSI 大於 -50dBm 時（代表極近距離），觸發 LED 快速閃爍作為感知互動。
    • 驗證要點：在報告中描述：為何手機放很近時 RSSI 是 -40 而非 0？並說明遮擋物（如手掌）對 RSSI 的影嚮。

3. 無線通訊與生命週期點評標準 (Wireless Lifecycle & UX Standards)

此單元的價值在於 「對無線物理特性的掌握與連線魯棒性的設計」：

  • [ ] 連線自癒能力：斷開連線後，設備是否能不需要重開機就自動重新進入廣播模式？
  • [ ] 硬體同步意識：LED 的視覺回饋是否精準對應了藍牙內部的狀態機 (State Machine)？
  • [ ] 信號洞察力：RSSI 實驗日誌是否呈現出了信號隨距離增加而對數衰減的趨勢？
