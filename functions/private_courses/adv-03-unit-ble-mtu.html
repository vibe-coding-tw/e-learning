📏 1. 什麼是 MTU (Maximum Transmission Unit)？ (20 min)

核心定義：通訊鏈路中單個封包所能承載的最大位元組數。MTU 決定了你一次「對話」能講多少個字。

BLE 的物理限制

傳統 BLE 預設 MTU 通常只有 23 Bytes。扣除 3 Bytes 的 L2CAP 標頭後，實際可用的數據載荷僅約 20 Bytes。

生活比喻：電梯限重

MTU 就像一台電梯的載重限制。如果你有一台鋼琴（大的 JSON 數據）要送上樓，你不能直接塞進去，必須先將其拆解成零件分別運送。

🤔 常見問題

「為什麼我發送一段 JSON 字串，ESP32 只收到前面的一小段？」—— 這通常就是因為長度超過了 MTU，導致後續數據被直接丟棄。

⚙️ 2. 分段傳輸與 MTU 交涉 (25 min)

🤝 1. MTU 交涉 (Negotiation)

在連線建立後，客戶端可以請求提升 MTU 限額（例如提升至 247 Bytes）。這就像是打電話給管委會，請求更換一台更大的貨梯。

注意：這取決於手機硬體與 ESP32 藍牙棧的共同支援。

🧩 2. 軟體分段 (Software Segmentation)

在程式碼層面將大字串切割成數個小塊（Chunks），逐一發送，並在接收端重新組裝。這是最保險的做法，不依賴硬體交涉。

🏁 3. 數據完整性與終止符 (15 min)

標記結尾：接收方收到零散的零件時，該如何知道「鋼琴」已經送完了，可以開始組裝了？

🚀 預習任務：終止符的重要性

思考在字串末端加入特定符號（如 \n 或 #）作為「結尾符」的重要性。

  • 接收方持續將片段累加到 Buffer。
  • 直到看見終止符，才觸發 JSON.parse()。
  • 這樣能避免解析到「殘缺不全」的 JSON 導致程式崩潰。

任務 1：「電梯拆解工」：手動分段發送器 (MTU Fragmenter Lab)

角色扮演：通訊底層工程師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 實作分段：撰寫 fragmentData(text, size) 函式，將長 JSON 字串依據 MTU 限制（如 20 Bytes）進行切分。
  3. 邊界驗證：測試當字串長度非倍數時，最後一個片段是否被正確發送。
  4. Push Final：提交具備「手動分段能力」且能穩定發送長數據的通訊專案。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 片段完整性：最後一個片段是否被遺漏或截斷？
  • [ ] 時序穩定性：片段間是否加入了適當延遲以防止發送端 Buffer 溢位？

任務 2：「拼圖高手」：緩衝區重組器 (Puzzle Reassembler)

角色扮演：前端系統整合員 | 難度：⭐️⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 緩衝機制：在接收端建立臨時累加 Buffer，持續接收並存儲零散片段。
  2. 終止符偵測：實作對 \n 或自定義結束符號的自動掃描。
  3. 原子解析：確保僅在收到完整終止符後，才觸發 JSON.parse() 以防止崩潰。
  4. Push Final：提交具備「抗殘缺、防崩潰」能力的重組引擎。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 魯棒性測試：當接收到不完整 JSON 時，代碼是否能有效捕獲錯誤而非導致程式鎖死？
  • [ ] 內存管理：Buffer 是否在每次成功解析後正確清空，避免數據疊加污染？

任務 3：MTU 請求與效能分析 (MTU Request & Analysis)

角色扮演：效能優化專家 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 權限提升：嘗試在連線後調用 requestMTU(247) 並紀錄系統響應。
  2. 效率對比：紀錄並對比「多小包」與「單一大包」在同一段長數據下的傳輸總耗時。
  3. 差異化分析：在 README 中紀錄不同手機系統對 MTU 請求的成功率與行為差異。
  4. Push Final：提交包含「MTU 效能權衡報告」的最終優化方案。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 實驗嚴謹度：報告中是否針對多次實驗結果取平均值，並標註測試環境？
  • [ ] 理論理解：能否原理解釋為何增加 MTU 能降低傳輸負擔 (Protocol Overhead)？
