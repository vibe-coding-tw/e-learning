進階 13 - 系統健壯性

軟體健壯性
與錯誤處理

打造牢不可破的防禦鏈：從防禦性編程到自動跳閘機制，為一切意外做好準備。

🛡️ 1. 防禦性編程 (Defensive Programming)

永遠假設外部輸入是不可信的。在執行核心硬體指令前，必須進行嚴格的「邊界檢查」與「格式驗證」，防止系統因單一錯誤封包而癱瘓。

Safety Patterns

  • 輸入驗證 (Input Validation)：檢查數值是否在合法硬體區間 (如 PWM 0-255)。
  • 數值鉗位 (Clamping)：強制將溢位值拉回安全邊界，防止馬達反轉。
  • 故障隔離 (Sandboxing)：確保通訊模組的錯誤不應導致主控制迴圈鎖死。

Runtime Guard Logic

ILLEGAL_INPUT: "999,999" (Corruption)
⬇️ SHIELD ⬇️
CLAMPED_RESULT: 255 (Safe)

DATA RECOVERY FILTERING ACTIVE

💓 2. 心跳機制與自動煞車 (Fail-safe)

若無線連線不幸中斷，車子必須在「一秒內」自動停止。這就是 Fail-safe：讓系統在失敗時，自動進入預定義的最安全狀態（通常是切斷馬達電力）。

Watchdog Timer Logic

\[if (millis() - last_heartbeat > 1500ms)\]

執行 stopMotors() 煞車指令：為軟體異常加上物理保險絲

⚠️ 3. 異常狀態回報介面 (UX/DX)

一個優秀的工業系統不應在出錯時保持「沈默」。UI 必須能讓操作者即時區分是通訊不良、電力負載過重還是內部組件失效。

Diagnostic UI Layers

  • 延遲度量 (Latency)：當連線延遲過高時變更配色，給予操作者視覺預警。
  • 標準錯誤代碼：定義 E01 (通訊)、E02 (超載) 等異常類別機制。
  • 故障日誌紀錄：在本地端緩存最近 10 次的異常事件，協助遠端除錯。

單元課程 進階 13 - 系統健壯性

Robustness 實驗室：邊界防禦與異常恢復

任務 1：非法數據壓力測試 (Input Clamping Lab)

角色扮演：安全性驗證工程師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 硬性邊界實作：在 motor_write() 前加入 constrain(val, 0, 255)。
  3. 異常解析攔截：實作 JSON 校驗，對於格式不完整或包含亂碼的數據包進行主動過濾。
  4. Push Final：提交具備「數據洗滌能力」的強韌型核心邏輯。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 邊界嚴密性：嘗試從前端發送 \(-500\) 或 \(800\)，觀察韌體是否能穩健地將其修正為 \(0\) 與 \(255\)？
  • [ ] 抗死機能力：在高速發送隨機字符時，ESP32 韌體是否會發生 Kernel Panic 重啟？

任務 2：通訊心跳與失效保護 (Heartbeat Fail-safe Lab)

角色扮演：預警系統工程師 | 難度：⭐️⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 計時閥門部署：實作全局變數 lastMsgTime。每次成功解析指令時重置該計時器。
  2. 超時跳閘邏輯：在主迴圈檢測 millis() - lastMsgTime > 1200，觸發 lockMotors()。
  3. 復歸安全性驗證：實作「手動解除鎖定」邏輯，防止在連線剛恢復時因舊緩存指令導致馬達暴衝。
  4. Push Final：提交具備「自動跳閘與重置保護」的安全專案。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 觸發準確度：模擬斷網後，馬達是否在約 1.2 秒（誤差 \(\pm 200ms\)）內停止轉動？
  • [ ] 代碼邏輯性：心跳檢測是否獨立於任務迴圈，不受其他耗時函數（如 Serial Print）干擾？

任務 3：診斷介面與延遲優化 (Diagnostic Dashboard Lab)

角色扮演：數據可視化專家 | 難度：⭐️⭐️ | 估計時長：40 min

🛠️ 執行技術流程 (Git Workflow)

  1. 遙測封包設計：在回傳 JSON 中加入 rssi (訊號強度) 與 error_code 欄位。
  2. 界面預警實作：實現動態延遲波形圖，並在訊號低於 \(-80dBm\) 時彈出警告提示。
  3. 故障回溯報告：實作「異常紀錄區」，將斷斷續續的通訊事件記錄在 UI 下方，方便現場維護人員診斷。
  4. Push Final：完成具備「自我診斷、外部感知」特質的最終整合專案。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 反饋及時性：當通訊發生異常時，UI 面板是否能在 1 秒內呈現出相應的視覺警告？
  • [ ] 代碼架構感：診斷邏輯是否與主控制邏輯解耦，易於未來擴展其他感測器診斷？

GitHub Classroom 詳細操作指南

本單元教學重點：從「功能優先」轉向「穩定優先」。引導學生理解在物聯網系統中，容錯 (Fault Tolerance) 比完美更重要。

  1. 重點 1：垃圾進，垃圾出 (GIGO) - 讓學生體會：如果對輸入不做「洗滌 (Sanitization)」，代碼隨時會從內部崩潰。
  2. 重點 2：心跳是系統的靈魂 - 透過 Fail-safe 實驗，讓學生感受到軟體逻辑如何控制物理世界的「安全性邊界」。
  3. 重點 3：工程式的信任感 - 強調 UI 診斷不是裝飾，而是操作者對系統產生信心（或進行維修）的唯一窗口。
