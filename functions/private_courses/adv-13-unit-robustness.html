進階 13 - 系統健壯性

軟體健壯性
與錯誤處理

打造牢不可破的防禦鏈：從防禦性編程到自動跳閘機制，為意外做好準備。

🛡️ 1. 防禦性編程 (Defensive Programming)

永遠假設輸入是錯誤的，假設使用者會亂按。在執行核心指令前，必須進行嚴格的「邊界檢查」與「格式驗證」。

Safety Patterns

  • 輸入驗證 (Input Validation)：檢查數值是否在合法區間 (e.g., PWM 0-255)。
  • 數值鉗位 (Clamping)：強制將異常值拉回邊界，防止計算溢位。
  • 故障隔離：局部代碼錯誤不應導致整個韌體當機。

💓 2. 心跳機制與自動煞車 (Fail-safe)

若連線不幸中斷，車子必須在「一秒內」自動停止。這就是 Fail-safe：讓系統在失敗時，自動進入最安全的狀態。

Timeout Detection

\[if (millis() - last_heartbeat > 2000)\]

執行緊急煞車 stopMotors()：為意外加上保險絲

⚠️ 3. 異常狀態回報介面 (UX)

一個好的系統不應在錯誤時「沈默」。UI 必須能讓使用者即時知道是通訊不良、電力不足還是指令格式錯誤。

Diagnostic UI Layers

  • 視覺警報：當延遲 (Latency) 過高時變更配色，給予使用者預判。
  • 錯誤代碼系統：定義 E01 (通訊)、E02 (超載) 等標準化回傳機制。
  • 歷史日誌：紀錄最近幾次的斷線時間與原因，供開發者除錯。

Robustness Lab：斷網與壓力測試

任務 1：「暴力輸入」壓力測試 (Input Stress Test)

角色扮演：全端資安開發員 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 漏洞掃描：檢查 setMotor() 等硬體控制函式，確認是否對 PWM 範圍 [0-255] 做了 Clamping。
  3. 防禦性重構：實作 isValidJSON() 驗證，攔截格式不符的字串（模擬駭客胡亂發送）。
  4. 壓力報告：在 README 中說明在極端輸入（如 -9999 或 亂碼）下，你的程式如何優雅地保持穩態而非重啟。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 範疇檢查合規性：所有的 PWM 輸出點是否都具備了 constrain() 函式？
  • [ ] 穩定性驗證：面對非法字串注入時，ESP32 韌體是否能不中斷並持續運作？

任務 2：「斷網救援」模擬演習 (Connection Fail-safe Simulation)

角色扮演：嵌入式安全工程師 | 難度：⭐️⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 心跳機制實作：在主迴圈內建立時間戳變數 lastReceived。當收到藍牙封包時立即更新。
  2. 自動跳閘邏輯：判斷 millis() - lastReceived > 1500。若成立，立刻調用 emergencyStop()。
  3. 安全性復歸：增加一個「安全鎖」變數。斷線煞車後，必須收到「RESET」指令才能重啟馬達，避免盲目恢復造成事故。
  4. 結果 Push：將具备 Fail-safe 保護的核心邏輯 Commit 並 Push。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] Fail-safe 觸發速度：通訊中斷後的煞車響應時間是否小於 2 秒？
  • [ ] 邏輯死鎖排除：當正常通訊時，心跳機制是否能保持隱形成員身分，不誤觸發煞車？

任務 3：異常狀態診斷 UI (Error Diagnosis Dashboard)

角色扮演：UI/UX 開發者 | 難度：⭐️⭐️ | 估計時長：40 min

🛠️ 執行技術流程 (Git Workflow)

  1. 遙測系統擴展：在前端 Javascript 中實作每秒計算一次連線成功率與指令延遲 (RTT)。
  2. 視覺預警設計：當延遲超過 500ms 時，UI 指令區塊變為紅色且禁用發送，並給出「連線不穩」提示。
  3. 異常日誌功能：實作一個 Console 區域，收集並顯示來自 ESP32 的錯誤代碼（如 E01: BLE_LOST）。
  4. PR 發起：提交完整的 Dashboard 診斷擴充功能。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 異常感知程度：UI 是否能讓使用者在「不看開發者日誌」的情況下理解當前系統發生的具體錯誤？
  • [ ] 回報精確度：顯示的延遲數據是否能反映真實的通訊狀況？

GitHub Classroom 詳細操作指南

本單元教學重點：從「功能優先」轉向「安全優先」。引導學員建立商務產品的核心穩定性觀念。

  1. 重點 1：為失敗而設計 (Design for Failure) - 引導學員接受故障是必然的，關鍵在於如何系統化地處理故障而非消滅它。
  2. 重點 2：數據完整性與邊界 - 指導學員理解為什麼任何外部輸入都應被視為「有毒的」，必須經過洗滌（校驗）才能使用。
  3. 重點 3：視覺反饋的力量 - 強調 UI 診斷面板在調試複雜物聯網系統中的不可替代性。
