🛡️ 1. 防禦性編程 (Defensive Programming)

永遠假設外部輸入是不可信的。在執行核心硬體指令前，必須進行嚴格的「邊界檢查」與「格式驗證」，防止系統因單一錯誤封包而癱瘓。

Safety Patterns

  • 輸入驗證 (Input Validation)：檢查數值是否在合法硬體區間 (如 PWM 0-255)。
  • 數值鉗位 (Clamping)：強制將溢位值拉回安全邊界，防止馬達反轉。
  • 故障隔離 (Sandboxing)：確保通訊模組的錯誤不應導致主控制迴圈鎖死。

Runtime Guard Logic

ILLEGAL_INPUT: "999,999" (Corruption)
⬇️ SHIELD ⬇️
CLAMPED_RESULT: 255 (Safe)

💓 2. 心跳機制與自動煞車 (Fail-safe)

若無線連線不幸中斷，車子必須在「一秒內」自動停止。這就是 Fail-safe：讓系統在失敗時，自動進入預定義的最安全狀態（通常是切斷馬達電力）。

Watchdog Timer Logic

⚠️ 3. 異常狀態回報介面 (UX/DX)

一個優秀的工業系統不應在出錯時保持「沈默」。UI 必須能讓操作者即時區分是通訊不良、電力負載過重還是內部組件失效。

Diagnostic UI Layers

  • 延遲度量 (Latency)：當連線延遲過高時變更配色，給予操作者視覺預警。
  • 標準錯誤代碼：定義 E01 (通訊)、E02 (超載) 等異常類別機制。
  • 故障日誌紀錄：在本地端緩存最近 10 次的異常事件，協助遠端除錯。

專案：系統健壯性 (adv-13-unit-robustness)

本單元是從「創作者」轉向「工程師」的核心分水嶺。你將學習如何防止車子失控，而不僅僅是讓它動起來。這涉及到輸入過濾、通訊守門狗 (Watchdog) 以及失效保護 (Fail-safe) 的工業級設計理念。

學習目標

  • 實作防禦性編程 (Defensive Programming) 以處理非法數據。
  • 建立失效保護機制，應對通訊中斷。
  • 實作診斷介面，提升系統透明度。

任務清單

任務 1：非法數據壓力測試 (Input Sanitization Lab)

確保系統不會因為不可信的外部指令而崩潰。

  • 檔案：src/main.ino 中的 handleCommand()。
  • 要求：
    • 驗證輸入是否為有效數字。
    • 使用 constrain() 將數值限制在安全範圍 (0-255)。
    • 在 README.md (本文件下方) 記錄當發送非法數值 (如 999, -500) 時，系統的反應。

任務 2：通訊心跳與失效保護 (Heartbeat Fail-safe Lab)

為無線設備建立自動安全機制。

  • 檔案：src/main.ino 中的 checkHeartbeat()。
  • 要求：
    • 當通訊中斷超過 1.2 秒時，自動執行 triggerEmergencyStop()。
    • 實作「手動解鎖」邏輯，防止連線恢復後立即衝撞。
    • 錄製影片展示：當車輛全速行駛時斷開網路，車輛必須在約 1.2 秒內煞停。

任務 3：診斷介面與延遲感知 (Diagnostic Feedback Lab)

讓操作者能即時監控系統健康狀況。

  • 檔案：src/main.ino 中的 sendTelemetry() 與 web/vui_shield.html。
  • 要求：
    • 在回傳封包中加入 RSSI (訊號強度) 與延遲數據。
    • 在網頁介面實作顏色警告：正常 (綠色)、訊號弱 (黃色)、通訊中斷 (紅色)。
    • 實作異常日誌，記錄最近 10 次斷線時間。

專案結構

  • include/safety_config.h: 定義安全閾值與物理邊界。
  • src/main.ino: 核心邏輯實作。
  • web/vui_shield.html: 診斷與控制面板。
  • STRESS_TEST_CHECKLIST.md: 壓力測試紀錄表。
  • tests/stress_test.js: 自動化測試脚本。

繳交要求

  1. 完成 src/main.ino 中的所有 TODO。
  2. 更新 STRESS_TEST_CHECKLIST.md 中的測試數據。
  3. 提交具備工業級診斷面板的最終程式碼。
  4. 將更新後的程式碼 push 到你的程式庫。

針對 adv-13-unit-robustness (系統健壯性) 單元，這是從「創作者」轉向「工程師」的核心分水嶺。它要求學員不再只關注「如何讓車子動」，而是轉向關注 「如何防止車子失控」。這涉及到輸入過濾、通訊守門狗 (Watchdog) 以及失效保護 (Fail-safe) 的工業級設計理念。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 配置建議

健壯性測試需要模擬異常與邊界條件，建議範本包含：

  • 📂 src/RobustCore.ino：主控制核心。預置通訊接收邏輯，但故意不加防護，留下 TODO 讓學員實作邊界檢查與格式校驗。
  • 📂 include/security_limits.h：安全限制定義檔。定義馬達最大 PWM、通訊逾時門檻 (e.g., 1.2s) 以及電流/負載的安全邊際。
  • 📂 web/diagnostics_panel.html：診斷介面基礎。這是一個網頁端框架，學員需在此實作延遲圖表與異常日誌顯示。
  • 📂 STRESS_TEST_CHECKLIST.md：壓力測試清單。提供標準化表格，讓學員手動輸入非法數值（如 999, -500）並紀錄系統的「生存反應」。

2. 作業任務部署細節

任務 1：非法數據壓力測試 (Input Sanitization Lab)

  • 目標：實踐「防禦性編程」，確保系統不會因不可信的外部指令而崩潰。
  • Classroom 部署建議：
    • 洗滌邏輯實作：學員需在接收指令處加入 constrain(val, 0, 255) 與數據類型檢查。
    • 邊界防禦驗證：要求學員在 README 中記錄：當透過網頁發送非數字或溢位值時，韌體如何進行「數據清洗」而不發生馬達逆轉或程序重啟。
    • Autograding：透過腳本發送多組惡意 JSON 封包，檢核 ESP32 韌體是否依然能穩定運行。

任務 2：通訊心跳與失效保護 (Heartbeat Fail-safe Lab)

  • 目標：為無線設備建立「意外中斷」後的自動安全機制。
  • Classroom 部署建議：
    • 心跳守門狗部署： 
      // 學生需實作：通訊監視器
if (millis() - lastValidMsgTime > 1200) {
    triggerEmergencyStop();
    systemState = SAFE_LOCK;
}
    • 斷連演示影片：學員需錄製影片展示：當車輛全速行駛時，強行關閉遙控器或斷開網路，車輛必須在約 1.2 秒內自動緊急煞停。
    • 復歸安全邏輯：要求實作「解鎖按鈕」，防止連線恢復瞬間因舊指令殘留導致的二次衝撞。

任務 3：診斷介面與延遲感知 (Diagnostic Feedback Lab)

  • 目標：提昇系統透明度，讓操作者能即時診斷系統「健康狀況」。
  • Classroom 部署建議：
    • 遙測數據擴充：學員需在回傳封包中加入 RSSI (訊號強度) 與延遲數據。
    • 視覺預警系統：在網頁端實作顏色警告：正常 (綠色)、訊號弱 (黃色)、通訊中斷 (紅色)。
    • 異常報告日誌：在 UI 下方實作一個清單，記錄最近 10 次的斷線時間點，方便工程檢修。
    • Push Final：提交具備「工業級診斷面板」的最終整合型控制大腦。

3. 安全工程導師點評標準 (Security & Robustness Benchmarks)

此單元的價值在於 「對系統穩定性的極端執著」：

  • [ ] 輸入過濾完備度：是否處理了所有可能的極端邊界與非正常類型輸入？
  • [ ] 保護機制可靠性：Fail-safe 觸發時間是否穩定？是否具備「安全解鎖」流程？
  • [ ] 系統透明度：診斷介面是否真的能幫助人類一眼看出失敗原因？
