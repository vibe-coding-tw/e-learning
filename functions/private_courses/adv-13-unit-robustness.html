🛡️ 1. 防禦性編程 (Defensive Programming)

在嵌入式系統與物聯網裝置中，防禦性編程 (Defensive Programming) 是確保系統穩定運行的第一道防線。 其核心理念非常直覺：永遠不要信任外部輸入。無論資料來自藍牙通訊、網頁介面或感測器讀數， 在將其用於控制硬體（例如馬達 PWM 輸出）之前，都必須經過嚴格的驗證與清洗。

在遙控車系統中，使用者透過 Web Dashboard 發送的指令經由藍牙傳輸到 ESP32。這條傳輸鏈路上的任何環節都可能產生異常—— 無線雜訊可能損毀封包內容、惡意使用者可能發送超出合理範圍的數值、甚至 JSON 解析程式本身也可能因格式錯誤而崩潰。 防禦性編程的目標就是讓系統在面對這些「意外」時，仍能安全、可預測地運行，而非直接當機或做出危險動作。

三大防禦策略

1. 輸入驗證 (Input Validation)

在使用任何外部數據之前，首先檢查其格式是否正確以及數值是否在合理範圍內。 對於遙控車的馬達控制，PWM 訊號的合法範圍為 0–255。任何超出此範圍的數值（例如 999 或 -500）都應被視為無效輸入， 系統應記錄警告並拒絕處理，而非盲目轉發到硬體。

2. 數值鉗位 (Clamping)

有時候，直接丟棄無效輸入可能導致系統停擺（例如車子收不到任何速度指令而完全不動）。 更優雅的做法是使用鉗位函數 (Clamping Function)，將超出範圍的數值強制「夾」回安全邊界。 Arduino 提供了內建的 constrain() 函數，能將數值限制在指定的最小值與最大值之間。

3. 故障隔離 (Fault Isolation / Sandboxing)

系統中的各個模組應盡可能彼此隔離，確保某一模組的故障不會「傳染」到其他模組。 例如，藍牙通訊解析器的崩潰不應導致馬達控制迴圈被鎖死。 透過將各功能封裝在獨立的函數中，並設置適當的超時機制與錯誤回傳碼，可以有效實現故障隔離。

實作範例：安全的指令處理函數

以下程式碼展示了如何在 ESP32 韌體中實作一個具備防禦性的指令處理函數：

void handleCommand(String rawInput) {
  // Step 1: 格式驗證 — 確認輸入非空且為有效數字
  if (rawInput.length() == 0) {
    Serial.println("[WARN] Empty input — ignored");
    return;
  }

  int value = rawInput.toInt();

  // Step 2: 數值鉗位 — 強制限制在 PWM 安全範圍
  value = constrain(value, 0, 255);

  // Step 3: 安全地應用到硬體
  analogWrite(MOTOR_PIN, value);
}
注意 Arduino 的 String.toInt() 在無法解析時會回傳 0，這在某些場景下可能被誤解為合法輸入（例如「速度 = 0」）。 在生產級韌體中，建議使用更嚴謹的解析函數（如 C 標準庫的 strtol()）， 並透過其回傳的 endptr 參數明確檢查解析是否成功。

常見攻擊向量與防禦對照

異常輸入類型未防禦時的後果防禦策略

PWM 值 = -500馬達可能反轉失控取絕對值後鉗位至合法範圍

💓 2. 心跳機制與自動煞車 (Fail-safe)

即使你的防禦性編程做得再完美，仍有一種致命的失敗模式無法僅靠輸入驗證來解決——通訊中斷。 當遙控車正以全速行駛，而藍牙連線突然因為距離過遠、訊號干擾或手機來電而中斷時， 車輛會繼續執行最後一次收到的指令：全速前進。這就是為什麼我們需要失效保護 (Fail-safe) 機制。

Fail-safe 的核心理念是：當系統偵測到自身處於無法安全運行的狀態時，應自動進入預定義的最安全狀態。 對於遙控車而言，最安全的狀態就是「停止所有馬達」。實現這個機制的關鍵技術就是心跳計時器 (Heartbeat Timer)， 也常被稱為「看門狗 (Watchdog)」。

心跳機制的運作原理

心跳機制的邏輯非常直覺：控制端（手機或 Web Dashboard）需要定期向車輛發送「心跳訊號」， 證明連線仍然有效。車輛端則持續追蹤「上次收到有效訊息的時間」。一旦超過預設的逾時門檻（例如 1200 毫秒）， 系統就判定通訊已中斷，並立即觸發緊急停車程序。

unsigned long lastHeartbeat = 0;
const unsigned long HEARTBEAT_TIMEOUT = 1200; // 1.2 秒
bool safetyLocked = false;

void checkHeartbeat() {
  if (millis() - lastHeartbeat > HEARTBEAT_TIMEOUT) {
    triggerEmergencyStop();
    safetyLocked = true;
    Serial.println("[SAFETY] Heartbeat lost — motors locked");
  }
}

void triggerEmergencyStop() {
  analogWrite(MOTOR_L, 0);
  analogWrite(MOTOR_R, 0);
}
關鍵設計 逾時門檻的選擇至關重要：設得太短（如 200ms），正常的網路抖動就會頻繁觸發誤報； 設得太長（如 5 秒），車輛在真正斷線時會失控衝出太遠。建議值為 1000–1500 毫秒， 並根據實際測試環境的藍牙品質進行微調。

安全復歸：防止二次衝撞

當通訊恢復時，系統不應立刻執行佇列中殘留的舊指令，否則可能因為「過時的全速前進指令」導致二次衝撞。 正確的做法是引入一個手動解鎖 (Manual Unlock) 流程： 連線恢復後，車輛仍保持鎖定狀態，操作者必須明確發送「解鎖」指令後才能恢復控制。

系統狀態條件允許的操作
NORMAL 正常運行心跳持續有效接受所有控制指令
TIMEOUT 心跳逾時超過 1200ms 無心跳自動鎖定馬達，拒絕控制指令
SAFE_LOCK 安全鎖定通訊恢復但尚未解鎖僅接受「解鎖」指令
UNLOCKED 已解鎖操作者確認解鎖清除舊指令佇列，回到 NORMAL
測試方法 驗證 Fail-safe 的最有效方式：讓車輛全速行駛，然後直接關閉手機的藍牙。 觀察車輛是否能在 1.2 秒內自動煞停。錄製這段影片，就是實作任務 13-2 的核心交付物。

⚠️ 3. 異常狀態回報介面 (UX/DX)

一個具備工業水準的控制系統，在發生異常時絕不能保持「沉默」。 如果車子突然停下來，操作者需要在第一時間知道原因—— 是藍牙訊號太弱？是馬達電流過載？還是韌體內部發生了未預期的例外？ 異常狀態回報介面的目標就是讓系統的「健康狀況」對人類完全透明。

這不僅是使用者體驗 (UX) 的問題，更是開發者體驗 (DX) 的核心需求。 當你在除錯階段需要遠端診斷一台行為異常的車輛時， 如果 Dashboard 能即時顯示 RSSI 訊號強度、指令延遲數值與最近的故障日誌， 你就能在幾秒內定位問題根源，而不是盲目猜測。

三層診斷架構

1. 延遲度量 (Latency Monitoring)

追蹤每個指令從發送到執行的往返時間 (Round-Trip Time, RTT)。 當延遲超過警告門檻時，Dashboard 應改變顯示顏色，給予操作者視覺預警。 正常延遲通常在 20–50ms，超過 200ms 就會明顯影響操控體驗。

2. 標準化錯誤代碼 (Error Code System)

定義一套統一的錯誤分類機制，使不同類型的異常能被快速識別與處理。 這對於多人協作的開發團隊尤其重要——每個人都能看懂錯誤代碼的含義，無需深入程式碼。

錯誤代碼分類描述嚴重程度
E01通訊藍牙連線中斷或心跳逾時嚴重
E02硬體馬達電流超過安全閾值嚴重
W01通訊RSSI 訊號低於 -80 dBm警告
W02效能指令延遲超過 200ms警告
I01系統安全鎖已觸發，等待解鎖資訊

3. 故障日誌紀錄 (Fault Logging)

在本地端維護一個循環緩衝區 (Circular Buffer)，記錄最近 10 次的異常事件， 包含時間戳記、錯誤代碼與當時的系統狀態快照。這些日誌可透過 WebSocket 即時推送到 Dashboard， 也可在重新連線後批次上傳，協助工程師進行事後分析。

// 遙測封包結構：在回傳數據中加入診斷資訊
void sendTelemetry() {
  String json = "{";
  json += "\"rssi\":" + String(WiFi.RSSI()) + ",";
  json += "\"latency\":" + String(lastRTT) + ",";
  json += "\"errCode\":\"" + currentError + "\",";
  json += "\"uptime\":" + String(millis()) + "}";
  ws.textAll(json);
}
提示 前端 Dashboard 應根據 RSSI 與延遲數值動態改變顯示顏色： 綠色（RSSI > -60 dBm，延遲 < 100ms）、 黃色（RSSI -60 ~ -80 dBm，延遲 100–300ms）、 紅色（RSSI < -80 dBm 或延遲 > 300ms）。 這套三色預警系統能讓操作者在異常發生前就採取預防措施。
進階技巧 將故障日誌存入 ESP32 的 SPIFFS 或 LittleFS 檔案系統，即使重開機也不會遺失。 這在實際部署中非常實用——當設備在遠端現場發生故障時， 工程師可以在事後連線讀取「黑盒子」紀錄來還原事發經過。

專案：系統健壯性 (adv-13-unit-robustness)

本單元是從「創作者」轉向「工程師」的核心分水嶺。你將學習如何防止車子失控，而不僅僅是讓它動起來。這涉及到輸入過濾、通訊守門狗 (Watchdog) 以及失效保護 (Fail-safe) 的工業級設計理念。

學習目標

  • 實作防禦性編程 (Defensive Programming) 以處理非法數據。
  • 建立失效保護機制，應對通訊中斷。
  • 實作診斷介面，提升系統透明度。

任務清單

任務 1：非法數據壓力測試 (Input Sanitization Lab)

確保系統不會因為不可信的外部指令而崩潰。

  • 檔案：src/main.ino 中的 handleCommand()。
  • 要求：
    • 驗證輸入是否為有效數字。
    • 使用 constrain() 將數值限制在安全範圍 (0-255)。
    • 在 README.md (本文件下方) 記錄當發送非法數值 (如 999, -500) 時，系統的反應。

任務 2：通訊心跳與失效保護 (Heartbeat Fail-safe Lab)

為無線設備建立自動安全機制。

  • 檔案：src/main.ino 中的 checkHeartbeat()。
  • 要求：
    • 當通訊中斷超過 1.2 秒時，自動執行 triggerEmergencyStop()。
    • 實作「手動解鎖」邏輯，防止連線恢復後立即衝撞。
    • 錄製影片展示：當車輛全速行駛時斷開網路，車輛必須在約 1.2 秒內煞停。

任務 3：診斷介面與延遲感知 (Diagnostic Feedback Lab)

讓操作者能即時監控系統健康狀況。

  • 檔案：src/main.ino 中的 sendTelemetry() 與 web/vui_shield.html。
  • 要求：

針對 adv-13-unit-robustness (系統健壯性) 單元，這是從「創作者」轉向「工程師」的核心分水嶺。它要求學員不再只關注「如何讓車子動」，而是轉向關注 「如何防止車子失控」。這涉及到輸入過濾、通訊守門狗 (Watchdog) 以及失效保護 (Fail-safe) 的工業級設計理念。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 配置建議

健壯性測試需要模擬異常與邊界條件，建議範本包含：

  • 📂 src/RobustCore.ino：主控制核心。預置通訊接收邏輯，但故意不加防護，留下 TODO 讓學員實作邊界檢查與格式校驗。
  • 📂 include/security_limits.h：安全限制定義檔。定義馬達最大 PWM、通訊逾時門檻 (e.g., 1.2s) 以及電流/負載的安全邊際。
  • 📂 web/diagnostics_panel.html：診斷介面基礎。這是一個網頁端框架，學員需在此實作延遲圖表與異常日誌顯示。
  • 📂 STRESS_TEST_CHECKLIST.md：壓力測試清單。提供標準化表格，讓學員手動輸入非法數值（如 999, -500）並紀錄系統的「生存反應」。

2. 作業任務部署細節

任務 1：非法數據壓力測試 (Input Sanitization Lab)

  • 目標：實踐「防禦性編程」，確保系統不會因不可信的外部指令而崩潰。
  • Classroom 部署建議：
    • 洗滌邏輯實作：學員需在接收指令處加入 constrain(val, 0, 255) 與數據類型檢查。
    • 邊界防禦驗證：要求學員在 README 中記錄：當透過網頁發送非數字或溢位值時，韌體如何進行「數據清洗」而不發生馬達逆轉或程序重啟。
    • Autograding：透過腳本發送多組惡意 JSON 封包，檢核 ESP32 韌體是否依然能穩定運行。

任務 2：通訊心跳與失效保護 (Heartbeat Fail-safe Lab)

  • 目標：為無線設備建立「意外中斷」後的自動安全機制。
  • Classroom 部署建議：
    • 心跳守門狗部署： 
      // 學生需實作：通訊監視器
if (millis() - lastValidMsgTime > 1200) {
    triggerEmergencyStop();
    systemState = SAFE_LOCK;
}
    • 斷連演示影片：學員需錄製影片展示：當車輛全速行駛時，強行關閉遙控器或斷開網路，車輛必須在約 1.2 秒內自動緊急煞停。
    • 復歸安全邏輯：要求實作「解鎖按鈕」，防止連線恢復瞬間因舊指令殘留導致的二次衝撞。

任務 3：診斷介面與延遲感知 (Diagnostic Feedback Lab)

  • 目標：提昇系統透明度，讓操作者能即時診斷系統「健康狀況」。
  • Classroom 部署建議：
    • 遙測數據擴充：學員需在回傳封包中加入 RSSI (訊號強度) 與延遲數據。
    • 視覺預警系統：在網頁端實作顏色警告：正常 (綠色)、訊號弱 (黃色)、通訊中斷 (紅色)。
    • 異常報告日誌：在 UI 下方實作一個清單，記錄最近 10 次的斷線時間點，方便工程檢修。
    • Push Final：提交具備「工業級診斷面板」的最終整合型控制大腦。

3. 安全工程導師點評標準 (Security & Robustness Benchmarks)

此單元的價值在於 「對系統穩定性的極端執著」：

  • [ ] 輸入過濾完備度：是否處理了所有可能的極端邊界與非正常類型輸入？
  • [ ] 保護機制可靠性：Fail-safe 觸發時間是否穩定？是否具備「安全解鎖」流程？
  • [ ] 系統透明度：診斷介面是否真的能幫助人類一眼看出失敗原因？
