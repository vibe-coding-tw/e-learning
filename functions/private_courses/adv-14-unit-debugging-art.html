🦆 1. 除錯心理學與科學方法

除錯不僅是技術，更是嚴謹的心理戰。我們常被「剛才明明還是好的」這種經驗主義蒙蔽。學會利用科學檢驗流程與「橡皮鴨除錯法」來系統化定位問題。

Scientific Cycle

  • 現象記錄：不帶成見地觀察 Bug 觸發的精確硬體與軟體條件。
  • 建立假說：邏輯推演可能的原因（如：整數溢位、反向映射、競態條件）。
  • 隔離實驗：縮小代碼範圍，驗證假說是否正確，直到鎖定「犯罪現場」。

🛡️ 3. 防禦性重構：寫出「不壞」的代碼

除錯的最高境界是「不需要除錯」。透過邊界驗證、異常捕獲 (Try-Catch) 與非同步流程管理，建立代碼的免疫能力。

Robustness Archetypes

  • 邊界防禦 (Edge Case Guard)：主動預測並攔截 NaN、Null 與負值溢位。
  • 競態保護 (Race Buffer)：確保非同步指令依序執行，防止任務衝突。
  • 感知型報錯 (Verbose UX)：故障發生時給予診斷引導，而非靜默當機。

adv-14-unit-debugging-art (除錯藝術)

歡迎來到「除錯藝術」單元！在這個單元中，你將學習如何從「只是修好 Bug」轉變為建立一套嚴謹的 科學除錯流程 與 防禦性程式設計 (Defensive Design) 體系。這是在開發複雜系統時不可或缺的核心技能。

學習目標

  • 掌握科學定位 Bug 的方法：現象 ➡ 假說 ➡ 隔離驗證 ➡ 證偽/證實。
  • 學習使用專業除錯工具：代碼斷點、網路檢視器、抓包工具。
  • 實作防禦性重構，讓你的程式庫具備更高的韌性與免疫力。

任務說明

任務 1：命案現場：代碼陷阱偵查 (Crime Scene Forensics Lab)

在 src/main.ino 中有幾個精心設計的陷阱（如整數溢位、變數作用域問題）。

  • 目標：不要憑感覺修復，請使用「科學方法」定位故障點。
  • 要求：在 docs/Investigation_Log.md 中詳細紀錄你的除錯過程。重點在於：你觀察到什麼現象？你的假說是什麼？你是如何驗證或推翻該假說的？

任務 2：通訊黑盒診斷與抓包 (Full-Link Packet Sniffing)

診斷前後端通訊協議中的隱形錯誤。

  • 目標：掌握數據鏈路的觀測能力。
  • 要求：利用瀏覽器的 Network 面板或 WebSerial 工具進行封包比對，找出發送端的 JSON 字串與接收端的解析期待之間的落差。

任務 3：防禦性重構實戰 (Defensive Hardening Lab)

加固你的代碼，從根本上消滅 Bug 產生的土壤。

  • 目標：寫出具備「免疫力」的程式碼。
  • 要求：
    • 前端：實作 try-catch 捕捉異常，並加入 UI 層級的異常警告 (Toast)。
    • 韌體端：在 include/hard_logic.h 中實作安全防護閥，確保馬達指令永遠在安全範圍內（例如 0-255）。

程式庫結構

  • src/main.ino: 韌體主要邏輯與 Bug 修復。
  • include/hard_logic.h: 韌體端的防禦性邏輯加固。
  • web/: 控制介面，需實作具備容錯檢查的高強度 UI。
  • docs/Investigation_Log.md: 必填，詳列你的除錯日誌。
  • tests/: 壓力測試工具，用來檢核你加固後的代碼韌性。

繳交方式

  1. 完成所有程式碼修復與重構。
  2. 確保 docs/Investigation_Log.md 紀錄完整。
  3. 將所有更改 git commit 並 git push 到你的程式庫中。

掌握除錯藝術後，你將擁有了應對「生命攸關」或「大規模複雜系統」開發的最核心工程師直覺！加油！

針對 adv-14-unit-debugging-art (除錯藝術) 單元，這是從「學生」邁向「高階工程師」的最終心態轉型。它不只是教如何修好 Bug，更是教如何建立一套嚴謹的 科學除錯流程 以及 防禦性程式設計 (Defensive Design) 體系。這是在複雜系統（如 Vibe Coding 的前後端無線控制）中生存的必備技能。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫配置建議

除錯教學需要「模擬真實犯罪現場」，建議範本包含：

  • 📂 src/BuggyCore.ino：故障現場專案。預埋 3 個精心設計的 Bug：一個是「整數溢位 (Integer Overflow)」、一個是「變數作用域污染」、一個是「隱形通訊協議拼寫不一致」。
  • 📂 docs/INVESTIGATION_LOG.md：除錯偵調日誌模板。強制學員依照「現象 ➡ 假說 ➡ 隔離驗證 ➡ 證偽/證實」的科學循環紀錄心路歷程。
  • 📂 web/js/defensive_shield.js：防禦重構區域。學員需在此實作 try-catch 與 UI 層級的異常警告彈窗 (Toast)。
  • 📂 tests/stress_injector.js：壓力注入工具。自動發送 NaN、null 與非法字元，用來檢核學員加固代碼後的防禦力。

2. 作業任務部署細節

任務 1：命案現場：代碼陷阱偵查 (Crime Scene Forensics Lab)

  • 目標：練習「非經驗主義」的科學定位法。
  • Classroom 部署建議：
    • 故障重現紀錄：學員需紀錄在何種輸入下系統會崩潰（例如：當 PID Target 設定為 0 時是否發生除零錯誤？）。
    • 橡皮鴨報告遞交：在 README 中提交日誌。重點在於：你嘗試過哪些「錯誤的猜測」？你是如何透過二分搜尋法 (Binary Search) 縮小故障範圍的？
    • PR 核核：點評其找出的 Bug 是否準確命中根因，而非只是暫時修飾現象。

任務 2：通訊黑盒診斷與抓包 (Full-Link Packet Sniffing)

  • 目標：掌握「數據鏈路」的透明化觀測能力。
  • Classroom 部署建議：
    • 抓包證據收集：利用瀏覽器 Network 面板或 WebSerial 進行「封包比對」。學員需截圖展示發送端的 JSON 字串與接收端的解析期待，找出那個「隱形的拼寫錯誤」。
    • 故障隔離分析：判斷 Bug 是發生在「前端封包封裝」、「頻寬丟包」還是「韌體 JSON 段落解析」？
    • 提交結果：提交修復後的通訊解析代碼以及抓包證據圖。

任務 3：防禦性重構實戰 (Defensive Hardening Lab)

  • 目標：寫出具備「免疫力」的代碼，從根本上消滅 Bug 產生的土壤。
  • Classroom 部署建議：
    • 加固加載實作：重構 PID 與馬達寫入函式。 
      // 學生需實作：防禦性邏輯
try {
    if (isNaN(input)) throw "Input Error!";
    let safeVal = Math.min(255, Math.max(0, input));
    sendToRobot(safeVal);
} catch (e) {
    showToast("⚠️ 偵測到非法指令：" + e);
}
    • 感知型報錯 UX：在網頁端實作 Alert。當機器人偵測到異常重啟或數據損毀時，UI 面板需主動變成紅色警告色，而非讓操作者對著毫無反應的車子發呆。
    • 最終推送：完成具備「健壯門檻、主動申告」特質的全鏈路代碼優化。

3. 工程師素養導師點評標準 (Scientific Debugging Benchmarks)

此單元的價值在於 「解決問題的邏輯與代碼的韌度」：

  • [ ] 除錯科學性：偵調日誌是否有邏輯地排除了各種干擾項，而非不斷地打掉重練？
  • [ ] 工具應用力：是否能在 PR 中正確使用代碼斷點或網路檢視器作為故障定位的證據？
  • [ ] 邊界防護力：重構後的代碼是否能擋住惡意字串（如 "abc" 代替轉數）的衝擊？
