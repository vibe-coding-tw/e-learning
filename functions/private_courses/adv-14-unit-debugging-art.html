🦆 1. 除錯心理學與科學方法

除錯不僅是技術，更是心理戰。我們常被「剛才明明還是好的」這種假設蒙蔽。學會「橡皮鴨除錯法」與「科學檢驗流程」。

Scientific Cycle

  • 現象觀察：不帶成見地記錄 Bug 觸發的精確步驟。
  • 建立假設：可能是馬達反向映射，也可能是整數溢位？
  • 最小實驗：隔離特定函數，驗證你的假設是否正確。

🛡️ 3. 防禦性重構：寫出「不壞」的代碼

除錯的最高境界是「不需要除錯」。透過輸入驗證、異常捕獲 (Try-Catch) 與自動超時保護，建立系統的免疫力。

Robustness Archetypes

  • 邊界檢查 (Edge Cases)：處理 NaN、負值與極端溢位。
  • 非同步處理 (Race Conditions)：確保指令按序執行，防止競態。
  • 有意義的錯誤 UI：當 Bug 發生時，明確告訴使用者為什麼，而非靜默崩潰。

除錯劇場：命案現場與防禦性設計

任務 1：「命案現場」：預埋 Bug 偵查挑戰 (Crime Scene Detective)

角色扮演：代碼偵探 | 難度：⭐️⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 現象捕獲：運行 broken-code 專案，觀察：馬達在轉向時為何突然失效？Web 端按鈕是否正確關聯？
  3. 假設定位：利用 console.log() 追蹤變數狀態。判斷是「數據型別錯誤 (Type Error)」還是「映射邏輯溢位 (Integer Overflow)」。
  4. 偵報撰寫：在 README 中詳細描述調查過程、修正前後的代碼差異，以及如何透過一個「實驗」確信問題已修復。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 邏輯洞見：是否能正確找出代碼中那 3 個故意設置的陷阱（如負值映射错误）？
  • [ ] 排查過程紀錄：日誌是否有呈現在找到解答前的「證偽」過程？（展現出科學除錯觀念）。

任務 2：「通訊黑盒」：網路封包抓包與分析 (Network Blackbox Analysis)

角色扮演：網路維運工程師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 模擬通訊故障：在 Web 面板操作，但 ESP32 沒反應。
  2. 前端監測：利用 DevTools > Network 截圖，確認指令包 (Payload) 是否正確送出？HTTP Status Code 是 200 還是 4xx？
  3. 封包比對：比對 Web 發出的 JSON 欄位與 ESP32 解析器預期的欄位，找出因大小寫或關鍵字不一致導致的解析失敗。
  4. 診斷提交：在 PR 附上抓包證據截圖，並邏輯推論故障點發生的具體層次。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 技術工具熟練度：能否從 Network 面板中提取出傳輸數據內容進行分析？
  • [ ] 問題隔離逻辑：是否展現出「排除法」思路，明確切分出前端代碼與韌體解析的邊界？

任務 3：防禦性重構實戰 (Defensive Refactoring)

角色扮演：系統穩定性專家 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 脆弱面掃描：回頭審視自己的 PID 運算函數：若收到 NaN 或 Null 數據，系統會崩潰嗎？
  2. 加固實作：加入 isNaN() 檢查與數值上限鉗位 (Clamping)。實作逾時心跳檢測，確保指令丟失時自動煞車。
  3. 異常反饋 UI：實作一個 Toast 通知。當封包解析失敗時，在畫面上顯示紅色的具體錯誤原因。
  4. Push Final：推送具备「自我保護意識」的最終代碼架構。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 代碼健壯性程度：隨機輸入極端異常值（如：10,000, "abc"）測試，系統是否能無誤拋出警告且不當機？
  • [ ] 用戶告知機制：當捕獲到錯誤時，介面是否能提供清晰的「診斷資訊」引導使用者解決問題？

GitHub Classroom 詳細操作指南

本單元教學重點：將「Debug」視為一種科學實驗。引導學生建立從觀測到驗證的嚴謹邏輯環結。

  1. 重點 1：看懂「犯罪現場」 - 強調不要看到 Bug 就急著改，要先能重複現象。沒被重複的 Bug 是沒被抓到的幽靈。
  2. 重點 2：工具是延伸的大腦 - 指導學生使用 Breakpoints 代替盲目的打印。雖然 print 快速，但斷點能提供更完整的時空上下文。
  3. 重點 3：為安全性買單 - 建立「防禦性重構」的觀念。讓學員體悟高品質的代碼不是寫出來的，是「重構」出來的。
