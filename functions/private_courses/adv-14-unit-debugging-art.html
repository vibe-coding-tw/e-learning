🦆 1. 除錯心理學與科學方法

除錯（Debugging）不僅是一項技術操作，更是一場嚴謹的心理與邏輯博弈。多數開發者在遇到 Bug 時，第一反應是「剛才明明還是好的」——這種依賴記憶與直覺的經驗主義，恰恰是除錯效率低落的根本原因。真正的專業工程師，會將除錯過程視為一場科學實驗：從觀察現象出發，建立假說，設計實驗來驗證或推翻假說，最終精準鎖定根因。

本單元將帶你建立一套可重複、可驗證的科學除錯流程，讓你從「碰運氣修 Bug」升級為「系統化獵殺 Bug」。這套方法論不僅適用於嵌入式系統開發，在任何軟體工程場景中都是頂尖工程師的核心素養。

科學除錯的三步循環

所有有效的除錯行為，都遵循以下三個階段的迭代循環。每一次迭代都會將問題空間縮小，直到精確定位到根因為止。

1

現象記錄（Observe）

不帶任何成見地觀察 Bug 觸發的精確條件。記錄硬體狀態、輸入數據、觸發時序與錯誤訊息。關鍵在於區分「事實」與「推測」——「馬達不轉」是事實，「馬達驅動壞了」是推測。

2

建立假說（Hypothesize）

根據觀察到的現象，邏輯推演可能的原因。常見的假說方向包括：整數溢位、變數作用域污染、競態條件（Race Condition）、通訊協議欄位不一致等。每次只針對一個假說進行驗證。

3

隔離驗證（Isolate & Test）

透過縮小代碼範圍或控制變因，設計最小化實驗來驗證假說。如果假說被推翻，回到步驟 1 重新觀察；如果假說被證實，你就找到了「犯罪現場」。

提示 「橡皮鴨除錯法」（Rubber Duck Debugging）要求你把問題完整地解釋給一隻橡皮鴨（或任何非技術對象）聽。在組織語言的過程中，你往往會自行發現邏輯漏洞。這個看似荒謬的方法，在 Google、Microsoft 等公司被廣泛實踐。

二分搜尋除錯法（Binary Search Debugging）

當問題涉及大量代碼且難以直接定位時，二分搜尋法是最有效率的隔離策略。其核心思想是：每次將可疑代碼範圍對半切割，判斷 Bug 出現在哪一半，再繼續對該半段進行切割，直到精確鎖定問題行。

// 二分搜尋除錯策略示意
// 假設 500 行的 loop() 出現異常

// 第 1 輪：註解掉後半段（第 250-500 行），測試前半段
// → 問題消失 → Bug 在後半段

// 第 2 輪：恢復後半段，註解第 375-500 行
// → 問題仍在 → Bug 在第 250-375 行

// 第 3 輪：繼續對半，縮小到第 310-375 行
// → 最終定位到第 342 行：整數溢位！

int speed = sensorValue * 256; // 當 sensorValue > 127 時溢位！

常見的心理偏誤陷阱

除錯過程中，工程師經常會落入以下心理偏誤，導致在錯誤的方向上浪費大量時間：

心理偏誤 典型表現 對策
確認偏誤 只尋找支持自己假說的證據，忽略矛盾線索 主動嘗試「推翻」自己的假說
近因偏誤 「我剛改過這段代碼，所以問題一定在這」 使用 git diff 客觀比對變更
可用性偏誤 上次是記憶體洩漏，這次「肯定也是」 從現象重新推導，不依賴過往經驗
沉沒成本 「我已經在這個方向花了 2 小時，不想放棄」 設定時間盒（Time Box），超時即切換方向
警告 永遠不要在沒有記錄的情況下連續嘗試多種修復。如果你同時改了三個地方然後 Bug 消失了，你無法確定哪個修改才是真正的解方——這會埋下未來的隱患。一次只改一個變因，是科學除錯的鐵律。

🛠️ 2. 工具箱深度應用：做代碼的偵探

當你掌握了科學除錯的心法之後，下一步就是熟練運用專業的除錯工具。就像刑事鑑識人員不會只靠肉眼辦案，軟體工程師也不應該只依賴 console.log 來追蹤問題。本單元將帶你深入學習三大核心工具：條件斷點（Conditional Breakpoints）、堆疊追蹤（Stack Trace）與網路封包檢視器（Network Inspector）。

這些工具的共同價值在於：它們讓你能夠在不修改原始碼的前提下，深入觀察程式在執行期間的內部狀態。這就像是為你的代碼裝上了 X 光機——你可以透視每一個變數的數值、每一次函式呼叫的順序、以及每一個網路封包的內容。

條件斷點（Conditional Breakpoints）

普通斷點會在每次執行到該行時暫停程式，但在迴圈或高頻呼叫的場景中，這樣做會讓你淹沒在大量無關的停頓中。條件斷點允許你指定觸發條件，只有當條件成立時才會暫停，大幅提升除錯效率。

// 在 VS Code 或 Chrome DevTools 中設定條件斷點
// 右鍵點擊行號 → 「Add Conditional Breakpoint」

for (let i = 0; i < 1000; i++) {
  let value = sensor.read(i);
  processData(value); // 條件斷點：value < 0 || value > 255
}
備註 在 Chrome DevTools 中，你還可以使用「Logpoint」——它會在觸發時輸出訊息到 Console，但不會暫停程式執行。這在生產環境的即時偵錯中特別有用。

堆疊追蹤（Stack Trace）解讀

當程式拋出例外（Exception）時，堆疊追蹤會顯示從錯誤發生點到最外層呼叫者的完整函式呼叫鏈。學會閱讀堆疊追蹤，就等於掌握了定位「犯罪第一現場」的能力。

Uncaught TypeError: Cannot read property 'x' of undefined
  at parseCommand (comm_handler.js:42) ← 犯罪現場
  at onMessageReceived (ble_manager.js:118)
  at WebSocket.onmessage (transport.js:67) ← 事件源頭

閱讀堆疊追蹤的原則是由上往下：最上方是直接拋出錯誤的位置（犯罪現場），往下則是一路追溯到事件源頭（誰觸發了這個呼叫鏈）。通常，Bug 的根因不在最頂層，而是在中間某一層傳入了不正確的參數。

網路封包檢視器（Network Inspector）

在物聯網開發中，前端與韌體之間的通訊經常是 Bug 的重災區。瀏覽器的 Network 面板允許你即時檢視每一個 HTTP/WebSocket 請求的完整內容，包括請求標頭、Payload 與回應碼。

檢查項目 正常狀態 異常徵兆
HTTP 狀態碼 200 OK 404（路徑錯誤）、500（伺服器崩潰）
JSON 欄位名稱 前後端欄位一致 大小寫不一致（如 Speed vs speed）
Payload 大小 符合預期的 byte 數 異常膨脹（可能含未編碼的二進位資料）
請求頻率 穩定間隔 突然密集重試（可能是超時重連邏輯失控）
重要 在模組簡介中提到的「右轉 45 度停頓」幽靈 Bug，正是因為 JSON 欄位名稱 "Direction"（首字母大寫）與韌體端期待的 "direction"（全小寫）不一致，導致解析失敗並觸發了隱形的超時重試。只要在 Network 面板中比對一次送出的 Payload，即可在數秒內定位此問題。

Serial Monitor 與韌體端偵錯

對於 ESP32 等嵌入式裝置，Serial Monitor（序列埠監控器）是你觀察韌體內部行為的主要窗口。在 PlatformIO 中按下 Ctrl+Shift+M 即可開啟，搭配 Serial.println() 可輸出變數狀態與執行流程。進階做法是將 Serial 輸出格式化為 CSV，再用 Serial Plotter 繪製即時數據曲線，直觀地發現異常抖動或數值突變。

提示 結合前端 Network 面板與韌體端 Serial Monitor 進行「全鏈路比對」，是診斷通訊層 Bug 的終極手段。從前端發出的 JSON 封包，到韌體端接收到的原始字串，任何差異都可能是問題根源。

🛡️ 3. 防禦性重構：寫出「不壞」的代碼

除錯的最高境界不是「更快地修好 Bug」，而是「讓 Bug 根本無法發生」。防禦性程式設計（Defensive Programming）的核心理念是：假設所有外部輸入都是不可信的，假設所有非同步操作都可能失敗，然後在代碼層面建立層層防禦機制。這就像為你的程式碼穿上防彈衣——即便遭受攻擊，受損也會被侷限在可控範圍內。

本單元將教你三大防禦性重構策略：邊界驗證（Edge Case Guard）、異常捕獲（Try-Catch）與感知型報錯（Verbose UX），並以實際的物聯網控制場景作為範例。

策略一：邊界驗證（Edge Case Guard）

在嵌入式系統中，感測器可能回傳 NaN（因為斷線）、null（因為初始化未完成）或超出預期範圍的負值。如果這些異常值直接被送入 PID 運算或馬達驅動，輕則控制失效，重則硬體損壞。防禦性代碼必須在數據入口處主動攔截這些邊界情況。

// 防禦性邊界驗證：確保馬達指令永遠在安全範圍內
function safeMotorWrite(rawValue) {
  if (typeof rawValue !== 'number' || isNaN(rawValue)) {
    console.warn('非法馬達指令值：', rawValue);
    return 0; // 安全歸零
  }
  const clamped = Math.min(255, Math.max(0, Math.round(rawValue)));
  return clamped;
}

策略二：異常捕獲與優雅降級

try-catch 機制讓你的程式在遭遇不可預期的錯誤時，能夠優雅地降級（Graceful Degradation），而非直接崩潰。在控制系統中，這意味著即使通訊中斷或數據損毀，系統也應該安全停車而非失控暴衝。

try {
  const cmd = JSON.parse(rawPayload);
  const safeSpeed = safeMotorWrite(cmd.speed);
  sendToRobot(safeSpeed);
} catch (e) {
  emergencyStop(); // 安全停車
  showToast('⚠️ 偵測到異常指令：' + e.message);
}

策略三：感知型報錯（Verbose UX）

最糟糕的使用者體驗不是「看到錯誤訊息」，而是「什麼都沒發生」。當機器人停止回應時，操作者需要知道：是通訊斷了？是電池沒電了？還是韌體崩潰了？感知型報錯要求系統在每一種故障模式下，都能給予操作者具體的診斷引導。

故障模式 靜默失敗（壞做法） 感知型報錯（好做法）
通訊超時 車子不動，畫面無變化 UI 亮紅燈 + Toast：「藍牙連線逾時，嘗試重新連接中...」
JSON 解析失敗 undefined 靜默傳入下游 Console 警告 + 自動切換為安全模式
馬達值溢位 PWM 輸出異常數值 自動鉗位至 0–255 + Serial 日誌記錄溢位事件
感測器斷線 讀取到 NaN，PID 計算崩壞 偵測到 NaN 後切換為上一個有效值 + 警告閃爍
警告 在韌體端（如 include/hard_logic.h），防禦性邊界檢查必須在寫入硬體暫存器之前執行。一旦非法的 PWM 值被寫入馬達驅動 IC，可能導致硬體過熱甚至燒毀。這不是軟體層面的 Bug，而是可能造成實體損害的安全漏洞。

防禦性重構的核心原則

  • 永不信任外部輸入——所有來自網路、感測器或使用者的數據，都必須經過驗證才能進入核心邏輯。
  • 失敗時安全停止——系統遇到不可處理的錯誤時，必須進入已知的安全狀態（如停車），而非繼續帶病運行。
  • 錯誤可被觀測——每一個被捕獲的異常都應該留下日誌痕跡，方便事後分析根因。
  • 降級優於崩潰——即使部分功能失效，系統的核心安全功能必須維持運作。
提示 專業的程式碼就像防彈玻璃——即便遭受重擊，受損也會被侷限在可控範圍內。這種「故障隔離」的設計哲學，正是航太、醫療與自動駕駛領域的工程標準。

adv-14-unit-debugging-art (除錯藝術)

歡迎來到「除錯藝術」單元！在這個單元中，你將學習如何從「只是修好 Bug」轉變為建立一套嚴謹的 科學除錯流程 與 防禦性程式設計 (Defensive Design) 體系。這是在開發複雜系統時不可或缺的核心技能。

學習目標

  • 掌握科學定位 Bug 的方法：現象 ➡ 假說 ➡ 隔離驗證 ➡ 證偽/證實。
  • 學習使用專業除錯工具：代碼斷點、網路檢視器、抓包工具。
  • 實作防禦性重構，讓你的程式庫具備更高的韌性與免疫力。

任務說明

任務 1：命案現場：代碼陷阱偵查 (Crime Scene Forensics Lab)

在 src/main.ino 中有幾個精心設計的陷阱（如整數溢位、變數作用域問題）。

  • 目標：不要憑感覺修復，請使用「科學方法」定位故障點。
  • 要求：在 docs/Investigation_Log.md 中詳細紀錄你的除錯過程。重點在於：你觀察到什麼現象？你的假說是什麼？你是如何驗證或推翻該假說的？

任務 2：通訊黑盒診斷與抓包 (Full-Link Packet Sniffing)

診斷前後端通訊協議中的隱形錯誤。

  • 目標：掌握數據鏈路的觀測能力。
  • 要求：利用瀏覽器的 Network 面板或 WebSerial 工具進行封包比對，找出發送端的 JSON 字串與接收端的解析期待之間的落差。

任務 3：防禦性重構實戰 (Defensive Hardening Lab)

加固你的代碼，從根本上消滅 Bug 產生的土壤。

  • 目標：寫出具備「免疫力」的程式碼。
  • 要求：
    • 前端：實作 try-catch 捕捉異常，並加入 UI 層級的異常警告 (Toast)。
    • 韌體端：在 include/hard_logic.h 中實作安全防護閥，確保馬達指令永遠在安全範圍內（例如 0-255）。

程式庫結構

  • src/main.ino: 韌體主要邏輯與 Bug 修復。
  • include/hard_logic.h: 韌體端的防禦性邏輯加固。
  • web/: 控制介面，需實作具備容錯檢查的高強度 UI。
  • docs/Investigation_Log.md: 必填，詳列你的除錯日誌。
  • tests/: 壓力測試工具，用來檢核你加固後的代碼韌性。

繳交方式

  1. 完成所有程式碼修復與重構。
  2. 確保 docs/Investigation_Log.md 紀錄完整。
  3. 將所有更改 git commit 並 git push 到你的程式庫中。

1. 範本程式庫配置建議

除錯教學需要「模擬真實犯罪現場」，建議範本包含：

  • 📂 src/BuggyCore.ino：故障現場專案。預埋 3 個精心設計的 Bug：一個是「整數溢位 (Integer Overflow)」、一個是「變數作用域污染」、一個是「隱形通訊協議拼寫不一致」。

  • 📂 web/js/defensive_shield.js：防禦重構區域。學員需在此實作 try-catch 與 UI 層級的異常警告彈窗 (Toast)。
  • 📂 tests/stress_injector.js：壓力注入工具。自動發送 NaN、null 與非法字元，用來檢核學員加固代碼後的防禦力。

2. 作業任務部署細節

任務 1：命案現場：代碼陷阱偵查 (Crime Scene Forensics Lab)

  • 目標：練習「非經驗主義」的科學定位法。
  • Classroom 部署建議：
    • 故障重現紀錄：學員需紀錄在何種輸入下系統會崩潰（例如：當 PID Target 設定為 0 時是否發生除零錯誤？）。
    • 橡皮鴨報告遞交：在 README 中提交日誌。重點在於：你嘗試過哪些「錯誤的猜測」？你是如何透過二分搜尋法 (Binary Search) 縮小故障範圍的？
    • PR 核核：點評其找出的 Bug 是否準確命中根因，而非只是暫時修飾現象。

任務 2：通訊黑盒診斷與抓包 (Full-Link Packet Sniffing)

  • 目標：掌握「數據鏈路」的透明化觀測能力。
  • Classroom 部署建議：
    • 抓包證據收集：利用瀏覽器 Network 面板或 WebSerial 進行「封包比對」。學員需截圖展示發送端的 JSON 字串與接收端的解析期待，找出那個「隱形的拼寫錯誤」。
    • 故障隔離分析：判斷 Bug 是發生在「前端封包封裝」、「頻寬丟包」還是「韌體 JSON 段落解析」？
    • 提交結果：提交修復後的通訊解析代碼以及抓包證據圖。

任務 3：防禦性重構實戰 (Defensive Hardening Lab)

  • 目標：寫出具備「免疫力」的代碼，從根本上消滅 Bug 產生的土壤。
  • Classroom 部署建議：
    • 加固加載實作：重構 PID 與馬達寫入函式。 
      // 學生需實作：防禦性邏輯
try {
    if (isNaN(input)) throw "Input Error!";
    let safeVal = Math.min(255, Math.max(0, input));
    sendToRobot(safeVal);
} catch (e) {
    showToast("⚠️ 偵測到非法指令：" + e);
}
    • 感知型報錯 UX：在網頁端實作 Alert。當機器人偵測到異常重啟或數據損毀時，UI 面板需主動變成紅色警告色，而非讓操作者對著毫無反應的車子發呆。
    • 最終推送：完成具備「健壯門檻、主動申告」特質的全鏈路代碼優化。
