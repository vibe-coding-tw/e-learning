🏗️ 1. 三層神經架構：從螢幕到馬達

理解系統的全貌是工程師的核心競爭力。將無人車系統拆解為感知控制、無線傳輸、硬體執行三大層級，定義清晰的邊界。

Data Pathway Tiering

  • 感知/控制層 (Web Frontend)：負責 UI 事件捕捉與數據序列化 (JSON)。
  • 傳輸層 (Web Bluetooth)：跨平台通訊協議，負責封包的原子性傳輸。
  • 執行/反饋層 (ESP32 Firmware)：負責解析、邏輯判斷與底層 PWM 訊號產生。

📈 3. 性能與延遲：端到端測量

除錯不靠通靈，靠時間戳記 (Timestamp)。學會埋點測量「端到端延遲」，定位系統中的性能瓶頸：是瀏覽器太慢，還是序列化封包太大？

Tracing Protocol

  • 雙向埋點：在 Web 發送與 ESP 接收處同時記錄 millis()。
  • 壓力測試：提高發送頻率直到 ESP32 串列緩衝區 (Serial Buffer) 溢位。
  • 異常模擬：故意發送損毀 JSON，測試系統解析器的韌性 (Resilience)。

任務 1：數據流向分析儀 (Data Flow Trace Analyzer)

角色扮演：系統整合工程師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 端到端計時：在前端發送 (Serial.print) 前與 ESP32 接收 (Serial.read) 後分別加掛 timestamp。
  3. 精度校驗：發送如 0.854 這樣的浮點數，檢查 ESP32 映射後還原出的數值（如 217 PWM）誤差是否在 1% 以內。
  4. 報告撰寫：在 README 中彙整一份數據鏈路分析報告，詳細列出網頁端處理、藍牙傳輸、硬體解析各佔用的耗時比例。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 測量準確性：報告中的各階段耗時加總是否符合邏輯上的總延遲？
  • [ ] 對齊嚴謹度：是否能正確分析「Byte Order」或「緩衝區解析延遲」對數值的微小影響？

任務 2：數據延遲與瓶頸挑戰 (Latency & Bottleneck Challenge)

角色扮演：性能測試工程師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 封包極限測試：分別測試 16, 32, 64 bytes 的封包大小對傳輸可靠度的影響。
  2. 頻率變壓測試：持續提升發送頻率，找出系統產生「顯著丟包」或「明顯延遲堆積」的臨界點（如：25Hz 之後）。
  3. 趨勢圖產出：在 README 中附上一張簡單的性能折線圖（頻率 vs 丟包率）。
  4. 優化提案：基於數據，提出一個能兼顧「操縱手感」與「穩定性」的最佳通訊週期參數。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 數據分析力：報告中是否包含對「丟包」成因的定性分析（是 Radio 還是 Serial Buffer 滿了）？
  • [ ] 參數建議合理性：提出的參數優化方案是否具備實驗依據？

任務 3：系統異常模擬與防禦 (Anomaly Simulation & Defense)

角色扮演：安全性開發人員 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 損毀測試實作：修改前端代碼，故意在每 10 筆數據中混入一筆格式錯誤的亂碼。
  2. 防禦性解析器重構：重寫 ESP32 的 onReceive 解析邏輯。使用具備自動彈回功能 (Auto-Reset) 的狀態機，確保解析錯誤不會卡死主迴圈。
  3. 韌性壓力測試：在大量注入亂碼的情況下，觀察無人車是否依然能響應正常的控制指令（雖然可能略有跳動）。
  4. 提交 PR：提交你的「零卡頓防禦解析器」代碼與測試結果。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 解析器韌性檢核：在遇到損毀封包時，系統日誌是否能輸出警告而非直接 Panic 或當機？
  • [ ] 恢復速度：從異常狀態回到正常控制狀態的延遲是否低於 50ms？

GitHub Classroom 詳細操作指南

本單元教學重點：透過「數據流」建立全局觀感。讓學員學會像醫師一樣，利用追蹤工手診斷系統健康的任何微小不協調。

  1. 重點 1：端到端思維 - 弱化對單一功能的著迷，強化對整體鏈路 (Link) 的關注。
  2. 重點 2：量化與分析 - 工程師不說「好像有點慢」，只說「平均延遲增加了 12ms」。
  3. 重點 3：防禦性開發 - 所有的通訊都是不可靠的。引導學員建立「髒數據處理」的本能反應。
