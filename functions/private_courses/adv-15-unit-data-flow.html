🏗️ 1. 三層神經架構：從螢幕到馬達

理解系統全貌是架構師的核心競爭力。將自動駕駛系統拆解為感知控制、無線傳輸、硬體執行三大神經層級，定義清晰的層級協議邊界。

Data Pathway Tiering

  • 感知控制層 (Dashboard)：負責 UI 事件捕捉、遙測視覺化與控制指令序列化。
  • 通訊傳輸層 (Web Bluetooth)：跨平台 GATT 協議實施，負責封包的高效低延時傳輸。
  • 硬體執行層 (Firmware)：負責解析協議、狀態機管理與底層 PWM 物理訊號產生。

🔄 2. 數據變身術：序列化與動態映射

同一個控制概念在不同層級有完全不同的數位形態。學習定義具備容錯性的「通訊協議」，確保網頁端發出的浮點數能被 ESP32 精確映射為 8-bit 的物理功率。

📈 3. 性能指標：端到端延遲測量

工程師從不靠直覺除錯。學會使用時間戳記 (Timestamp) 埋點測量「全鏈路延遲」，精確定位瓶頸：是瀏覽器渲染太慢，還是固件端的序列化解析太過沉重？

End-to-End Latency Tracing

  • 雙向時間埋點：在 Web 發送與 ESP 接收處同步紀錄 millis()，量化 RTT 值。
  • 壓力限制測試：持續提升發送頻率直到觸發序列緩衝區 (Buffer) 溢位邊界。
  • 解析韌性 (Resilience)：故意注入損毀封包，測試系統解析器是否具備自動復位能力。

核心任務：全鏈路數據實證

利用測量工具驗證數據完整性，並在極限環境下實施性能壓力測試。透過 Git Workflow 建立具備工業強度的系統診斷報告。

🔍 60 min

【作業 15-1】

「數據鏈路追蹤」

在網頁發送端與固件接收端分別埋入時間戳記，量測一組 (x, y) 座標跨平台移動的確切耗時。產出具備精確數據支持的「系統全鏈路延時報告」。

📉 60 min

【作業 15-2】

「通訊瓶頸診斷」

執行高頻發送壓力測試，找出封包長度與控制即時性之間的最優平衡點。繪製「頻率 vs 丟包率」曲線，針對你的系統環境給出官方控制配置建議。

🛡️ 45 min

【作業 15-3】

「韌性通訊實作」

手動模擬損毀封包含亂碼注入。重構並實體化具備「錯誤隔離」能力的固件解析器，確保系統遇到非法通訊時能秒級自動復位而不當機，實現工業級韌性。

單元課程 進階 15 - 數據流分析

系統診斷實驗室：數據全鏈路追蹤與解析韌性

任務 1：端到端數據流追蹤分析 (E2E Data Flow Trace Lab)

角色扮演：系統效能整合專家 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 全鏈路埋點實施：在 dashboard.js 發送前與 vehicle_main.cpp 接收解析後各記錄一組基準時間。
  3. 映射精確度驗證：發送特定測試向量（如：0.500, -1.000），量測 ESP32 端產生的 PWM 數值是否嚴格對齊預期公式。
  4. Push Final：提交包含「全鏈路耗時分析圖」與「數據映射誤差表」的完整實驗日誌。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 測量嚴謹度：分析報告中的耗時結構是否清晰？（區分出傳輸耗時與各端的處理耗時）。
  • [ ] 對齊完整性：映射公式是否考慮了浮點數轉整數時的捨入 (Rounding) 偏差問題？

任務 2：通訊頻寬與延遲極限壓測 (Throughput & Latency Stress Lab)

角色扮演：通訊性能工程師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 變壓測試場景建立：動態調整 setInterval 頻率，逐步降低延遲，記錄系統首個封包丟失的時間點。
  2. 頻率/丟包曲線繪製：收集不同控制指令頻率下的成功接收率數據。
  3. 系統 SLA 定義：基於數據定義你的系統「最優控制頻率」其當下之總延遲（End-to-End Latency）。
  4. Push Final：推送具備「工業性能規格」價值的性能評測報告。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 性能洞察深度：能否指出系統主要的延時發生在哪個層級？（Radio 層、Serial 層或解析層）。
  • [ ] 建議科學性：給出的優化頻率參數是否能保證控制手感與連線穩定度的視覺平衡？

任務 3：髒數據注入與自動復位解析器 (Dirty Data Injection & Auto-Resettable Parser Lab)

角色扮演：安全與韌性開發工程師 | 難度：⭐️⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 混沌工程實踐：實做一段隨機產生損毀 JSON 封包的注入代碼。
  2. 固件解析器重構：重寫固件解析邏輯。實作「緩衝區同步點搜索」，確保在亂碼之後能立即找回下一組有效封包。
  3. 恢復時序測量：紀錄從注入亂碼到系統重新找回控制權的「平均復原時間」。
  4. Push Final：提交具備「極高通訊容錯率」的加固版固件代碼與壓力測試結果。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 解析器健壯性：在連續注入非法封包時，ESP32 是否依然能維持主迴圈運行且不發生看門狗 (WDT) 重啟？
  • [ ] 系統自癒力：遇到損毀數據後，系統是否真正落實了「安靜丟棄、迅速同步」的原則？

GitHub Classroom 詳細操作指南

本單元教學重點：透過「數據流」建立全棧 (Full-stack) 工程觀。讓學員學會像診斷醫師一樣看穿系統內部的每一毫秒數據脈動。

  1. 重點 1：端到端思維 - 引導學員排除「只看固件/只看網頁」的片面視野，轉向關注整體「連結 (Link)」的健康度。
  2. 重點 2：量化與性能監控 - 工程師不應說「好像慢了點」，要要求學員產出「平均延遲、RTT 值」等專業數值。
  3. 重點 3：通訊是不安全的設計預期 - 讓學員建立「防禦性解析」習慣。通訊亂碼、截斷是現實世界的常態，系統必須具備自癒力。
