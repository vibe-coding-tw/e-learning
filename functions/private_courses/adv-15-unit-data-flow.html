🏗️ 1. 三層神經架構：從螢幕到馬達

理解系統全貌是架構師的核心競爭力。將自動駕駛系統拆解為感知控制、無線傳輸、硬體執行三大神經層級，定義清晰的層級協議邊界。

Data Pathway Tiering

  • 感知控制層 (Dashboard)：負責 UI 事件捕捉、遙測視覺化與控制指令序列化。
  • 通訊傳輸層 (Web Bluetooth)：跨平台 GATT 協議實施，負責封包的高效低延時傳輸。
  • 硬體執行層 (Firmware)：負責解析協議、狀態機管理與底層 PWM 物理訊號產生。

🔄 2. 數據變身術：序列化與動態映射

同一個控制概念在不同層級有完全不同的數位形態。學習定義具備容錯性的「通訊協議」，確保網頁端發出的浮點數能被 ESP32 精確映射為 8-bit 的物理功率。

📈 3. 性能指標：端到端延遲測量

工程師從不靠直覺除錯。學會使用時間戳記 (Timestamp) 埋點測量「全鏈路延遲」，精確定位瓶頸：是瀏覽器渲染太慢，還是固件端的序列化解析太過沉重？

End-to-End Latency Tracing

  • 雙向時間埋點：在 Web 發送與 ESP 接收處同步紀錄 millis()，量化 RTT 值。
  • 壓力限制測試：持續提升發送頻率直到觸發序列緩衝區 (Buffer) 溢位邊界。
  • 解析韌性 (Resilience)：故意注入損毀封包，測試系統解析器是否具備自動復位能力。

單元課程 進階 15 - 數據流分析

數據流分析 (Data Flow Analysis)

「好的架構師，是能像觀測心電圖一樣，看穿數據流中每一毫秒跳動背後的工程邏輯。」

歡迎來到進階單元 15。在本單元中，你將不再只是開發功能，而是要像「系統診斷醫師」一樣，深入觀察數據在感知層、傳輸層到解析層之間的每一分傳遞、變形與損耗。你的目標是定義出系統的生命脈動，並在高壓與干擾環境下確保系統的絕對穩定。

🎯 學習目標

  1. 全鏈路觀測 (Full-Stack Observability)：實作高性能埋點工具，量化端到端的延遲 (RTT)。
  2. 性能極限定義：透過壓力測試找出系統的崩潰臨界點，並定義 SLA 規格。
  3. 系統韌性設計 (Resilience Design)：重構固件解析器，使其具備在髒數據環境下自動同步與復位的能力。

🛠️ 任務說明

任務 1：端到端數據流追蹤分析

  • 目標：找出數據在跨平台傳遞時「消失的毫秒數」。
  • 要求：
    • 在網頁前端點擊到馬達響應的完整路徑上，設定至少 3 個觀測點。
    • 量化並記錄各層級（感知、傳輸、解析）的精確耗時。
    • 產出：在 docs/LINK_AUDIT.md 中提交審計報告，並定位目前的性能瓶頸。

任務 2：通訊頻寬與延遲極限壓測

  • 目標：透過科學實驗定義系統的「官方規格」。
  • 要求：
    • 逐步提升發送頻率（從 10Hz 至 100Hz），記錄丟包率與 Serial 緩衝區狀態。
    • 產出：在 docs/Performance_Map.md 中繪製性能曲線圖，並在 README.md 的結案報告中給出具體的硬體配置建議。

任務 3：髒數據注入與自動復位解析器

  • 目標：在電磁干擾或亂碼注入下，確保系統「永不當機」。
  • 要求：
    • 重構 src/robust_parser.cpp，實作「同步標記搜索 (Sync Marker Search)」功能。
    • 使用混沌測試工具，驗證系統在封包損毀後能否在 50ms 內恢復控制。
    • 產出：提交加固後的代碼與測試日誌 tests/chaos_monkey.log。

1. 範本程式庫 (Template Repo) 配置建議

系統分析需要高精度的埋點工具與回報框架，建議範本包含：

  • 📂 src/latency_tracer.js：發送端高性能埋點工具。預置時間戳計算法，自動追蹤每一筆指令發出與回傳的精確耗時。
  • 📂 src/naive_parser.cpp：重構手術對象。一個「天真」的固件解析器（遇到亂碼會卡住或延遲），學員需將其升級為具備「自癒能力」的高級解析器。
  • 📂 docs/LINK_AUDIT.md：全鏈路審計報告。要求學員記錄：感知層耗時 | 傳輸層耗時 | 解析層耗時 | 最終執行延遲。
  • 📂 tests/chaos_injections.js：混沌工程注入器。這是一個自動化腳本，會隨機發送「損毀的 JSON」或「超長無效字串」，用來檢測系統的崩潰臨界點。

2. 作業任務部署細節

任務 1：端到端數據流追蹤分析 (E2E Data Flow Trace Lab)

  • 目標：看穿數據在跨平台傳遞時隱形的耗時分景。
  • Classroom 部署建議：
    • 數據路徑量測：學員需在網頁點擊到馬達轉動的完整路徑上設定 3 個以上的觀測點，量化總延遲 (RTT)。
    • 映射精確度驗證：比對網頁端 Normalized 資料 (01.0) 轉換為 C 語言 Uint8 (0255) 過程中的數據損耗。
    • 驗證要點：分析報告是否能清楚定位「哪一層」是目前的性能瓶頸？

任務 2：通訊頻寬與延遲極限壓測 (Throughput & Latency Stress Lab)

  • 目標：透過實驗找出系統的「崩潰臨界點」，並定義其 SLA 規格。
  • Classroom 部署建議：
    • 頻率/丟包曲線：逐步將發送頻率從 10Hz 拉升至 100Hz，紀錄系統開始出現丟包或 Serial 緩衝區溢位的臨界點。
    • 官方規格定義：學員需在 README 中定義：「在此硬體配置下，系統的最佳控制頻率為 XX Hz，延遲保證在 XX ms 內」。
    • PR 提交內容：提交具備「工業背書強度」的性能曲線圖與規格建議書。

任務 3：髒數據注入與自動復位解析器 (Dirty Data & Robust Parser Lab)

  • 目標：在現實世界的電磁干擾環境中，確保系統「永不當機」。
  • Classroom 部署建議：
    • 韌性重構：優化固件端解析器，使其具備「同步標記搜索 (Sync Marker Search)」能力。即使前一個封包損毀，也能在下一毫秒立即找回有效的數據開頭。
    • 復位速度挑戰：利用混沌工具注入亂碼，量測系統「恢復控制」所需的平均毫秒數。目標應小於 50ms。
    • Push Final：提交具備「高度通訊韌性、不因亂碼崩潰」的最終加固版解析代碼與測試 log。

3. 全鏈路系統工程師導師點評標準 (Full-Stack Observability Standards)

此單元的價值在於 「對全系統脈動的科學觀察力與抗壓設計」：

  • [ ] 診斷嚴謹度：報告是否揭示了延遲產生的物理原因（如：BLE 跳頻、Serial 波特率限制）？
  • [ ] 解析器防護力：面對連續的非法字串注入，ESP32 是否依然能穩定運行且不丟失後續的有效控制？
  • [ ] 規格承諾一致性：測量出的數據是否能在 PR 審查現場被導師複現？
