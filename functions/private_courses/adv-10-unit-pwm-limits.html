限制與飽和：
PWM 的物理極限

當數學計算遇上物理極限，如何保持幾何軌跡的完美？探索飽和處理與死區補償。

🛑 1. 什麼是 PWM 飽和？ (15 min)

物理世界是有天花板的。ESP32 的 8-bit PWM 上限是 255；當幾何公式計算出「需要 300 電力」時，直接裁剪（Clipping）會破壞左右輪的速度比例。

System Bottleneck

  • 比例失衡：原本 2:1 的速度比，因為一方飽和而變回 1.7:1，導致轉彎半徑偏離。
  • 指令溢位：未妥善處理的 0-255 映射會導致馬達驅動異常。

👻 2. 死區 (Dead Zone) 現象 (15 min)

並非 PWM > 1 馬達就會轉。受限於靜摩擦力，馬達在低速區間僅會發出嗡嗡聲。理解「有效作業區間」是實作絲滑緩啟動的關鍵。

⚖️ 3. 等比例縮放演算法

當最大輪需求超過 255，我們使用「縮放（Scaling）」而非「裁剪（Clipping）」。計算縮放係數 $Factor = 255 / \max(L, R)$，確保運動幾何關係不變。

Geometric Scaling

  • 維繫比例：即使速度變慢，轉彎半徑依然維持理論預期。
  • 平滑過渡：防止在高負擔動作下車體發生劇烈抖動或打滑。

實驗室：控制極限實務

處理硬體限制的標準工程方法。透過 Git Workflow 建立具備「飽和感知」能力的強健馬達驅動器。

💥 60 min

【作業 3-1】

「飽和失控器」

故意給予超出 255 的指令。使用 constrain() 裁剪並觀察軌跡變形。記錄幾何中心偏離理論圓周的路徑誤差。

🛡️ 60 min

【作業 3-2】

「比例保衛員」

撰寫 normalizePWM()。若任一輪需求大於 255，則同步按比例下拉。驗證在飽和狀態下，轉向半徑是否仍趨近理論預期。

💊 45 min

【作業 3-3】

「平滑補償器」

測量馬達 $PWM_{min}$。使用 map() 映射 0-255 至 $PWM_{min}$ 到 255。測試極低速（如 PWM=50）行駛的平順度與轉向精度。

GitHub Classroom 詳細操作指南

本單元教學重點：讓學生理解「軟體設計必須尊重物理現實」。當計算結果超出硬體極限，優先保護「幾何關係」。

  1. 重點 1：裁剪是惡魔 - 學生需親眼看見 Clipping 破壞控制穩定性的慘痛代價。
  2. 重點 2：不對稱死區 - 引導學員發現為什麼車子明明代碼一樣卻走歪：通常是左右輪死區電壓不同（摩擦力不對稱）。
  3. 重點 3：線性化映射 - 教導學員如何建立「虛擬的線性通道」，讓 PID 演算法在真實物理世界更穩定。
