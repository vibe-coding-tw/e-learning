🛑 1. 什麼是 PWM 飽和？ (15 min)

在差速驅動的運動學公式中，我們可以自由地計算出任意大小的左右輪速需求。 然而，物理世界有不可逾越的天花板：ESP32 的 8-bit PWM 解析度意味著輸出值的有效範圍僅為 0 到 255。 當幾何公式計算出「左輪需要 PWM = 300」時，系統必須做出抉擇——而這個抉擇的方式， 將決定你的車體是精確沿著圓弧行駛，還是偏離預期軌跡。

飽和裁剪（Clipping）的陷阱

最直覺的處理方式是使用 constrain() 函式，將超出範圍的值直接裁剪到 255。 這看似合理，實際上卻是幾何精度的隱形殺手。原因在於：裁剪會破壞左右輪的速度比例， 而差速驅動的轉彎半徑完全取決於這個比例。

重要 差速驅動的轉彎半徑公式 \(R = \frac{L}{2} \cdot \frac{V_L + V_R}{V_L - V_R}\) 完全由速度比值決定。 當 Clipping 改變了這個比值，轉彎半徑就會偏離計算的理論值，導致軌跡變形。

實例：Clipping 如何破壞軌跡

假設運動學公式計算出左輪需要 PWM = 300、右輪需要 PWM = 150（比例 2:1，對應特定轉彎半徑）。 以下比較兩種處理方式的結果：

處理方式 左輪 PWM 右輪 PWM 實際速比 對軌跡的影響
理論需求 300 150 2.00 : 1 理想圓弧
Clipping（裁剪） 255 150 1.70 : 1 半徑增大，圓弧變形
Scaling（縮放） 255 127.5 2.00 : 1 速度降低但半徑正確
// ❌ 錯誤做法：直接裁剪（破壞速比）
int pwmL = constrain(300, 0, 255); // → 255
int pwmR = constrain(150, 0, 255); // → 150
// 比例從 2:1 變成 1.7:1 → 軌跡偏離！

// ✅ 正確做法：等比例縮放（保持速比）
float maxVal = max(300, 150); // = 300
float factor = 255.0 / maxVal; // = 0.85
int pwmL = 300 * factor; // → 255
int pwmR = 150 * factor; // → 127 比例依然 ≈ 2:1
警告 飽和問題通常在「高速 + 小半徑轉彎」時最為嚴重——因為此時外側輪的速度需求最高。 如果你發現機器人在高速轉彎時軌跡突然「拉直」，幾乎可以確定是 Clipping 造成的比例失衡。
提示 記住這個核心原則：寧可降速，也不可破壞比例。 Scaling 讓車體「變慢但精確」，而 Clipping 讓車體「維持速度但失控」。在機器人控制中，精度永遠優先於速度。

👻 2. 死區 (Dead Zone) 現象 (15 min)

你可能以為只要 PWM 值大於 0，馬達就會開始轉動——但事實並非如此。 由於靜摩擦力（Static Friction）的存在，馬達需要一個最低的驅動電壓才能克服靜止慣性開始旋轉。 在這個臨界值以下的 PWM 區間，馬達只會發出嗡嗡聲卻不會轉動，這就是死區（Dead Zone）。

死區的物理成因

直流馬達的轉子需要產生足夠的電磁扭矩來克服以下阻力之後才會啟動：

  • 軸承摩擦力：轉軸與軸承之間的機械摩擦
  • 齒輪系靜摩擦：減速齒輪組的嚙合阻力
  • 負載慣性：車輪與車體的質量慣性矩
  • 輪胎與地面的接觸摩擦：尤其在粗糙表面上更為顯著

每台車的死區值都不同，即使是同一批次的馬達，左右兩側的死區也可能存在差異。 因此，死區補償必須透過逐台校準（Per-Unit Calibration）來完成。

警告 死區對差速驅動的影響在低速轉彎時最為嚴重。 如果內側輪的 PWM 落入死區而不轉，外側輪卻在轉，車體會變成意料之外的「單輪旋轉」—— 實際 ICC 位置與計算值完全不同，軌跡誤差可能高達數十公分。

測量你的馬達死區

死區測量是一個簡單但重要的校準步驟。對每顆馬達獨立測量，找出讓馬達「剛好開始穩定轉動」的最小 PWM 值：

1

從 PWM = 0 開始逐步遞增

每次增加 5，觀察馬達是否開始轉動。記錄首次穩定旋轉的 PWM 值。

2

對左右馬達分別測量

記錄 \(PWM_{min,L}\) 和 \(PWM_{min,R}\)。典型值介於 30–60 之間。

3

在不同表面重複測量

地毯的摩擦力較大，死區值通常會比磁磚高出 10–20。

線性映射補償

測量完死區後，我們使用 map() 函式建立一個虛擬的線性控制通道： 將軟體層的 0–255 指令映射到硬體層的 \(PWM_{min}\)–255 輸出， 確保軟體端的「0」代表停止、「1」就代表最低的實際轉速。

// 死區補償：線性映射
const int DEAD_ZONE_L = 42; // 左馬達實測死區
const int DEAD_ZONE_R = 38; // 右馬達實測死區

int compensate(int input, int deadZone) {
if (input == 0) return 0; // 0 就是停止
return map(input, 1, 255, deadZone, 255);
}

// 使用範例
int pwmL = compensate(50, DEAD_ZONE_L); // 50 → 83（跳過死區）
int pwmR = compensate(50, DEAD_ZONE_R); // 50 → 81
提示 將死區校準值存放在 CALIBRATION.md 或 config.h 中， 而不是寫死（Hard-code）在運動學函式裡。這樣每台車可以有自己的校準檔案， 程式碼不需要為每台車做修改。

⚖️ 3. 等比例縮放演算法

理解了飽和裁剪的危害和死區的成因之後，我們終於要來實作解決方案： 等比例縮放（Proportional Scaling）演算法。 核心思想非常簡單——當任一輪的 PWM 需求超過 255 時， 計算一個縮放係數 \(Factor = \frac{255}{\max(V_L, V_R)}\)， 並將兩輪同時乘以此係數，從而在降低速度的同時完美保持速度比例。

演算法步驟

1

計算原始輪速需求

透過 ICC 逆向運動學公式，得到左右輪的理想 PWM 值 \((V_L, V_R)\)。

2

檢測是否超出上限

取得兩輪的最大絕對值：maxVal = max(abs(V_L), abs(V_R))。如果 maxVal <= 255，無需縮放。

3

計算並套用縮放係數

縮放係數 \(f = 255 / maxVal\)。將兩輪同時乘以 \(f\)：V_L *= f; V_R *= f;

4

套用死區補償

縮放後的值可能落入死區。最後一步需通過死區映射確保馬達確實會轉動。

完整實作程式碼

struct NormalizedPWM { int left; int right; bool scaled; };

NormalizedPWM normalizePWM(float vL, float vR) {
NormalizedPWM result;
result.scaled = false;

float maxVal = max(abs(vL), abs(vR));
if (maxVal > 255) {
float factor = 255.0 / maxVal;
vL *= factor;
vR *= factor;
result.scaled = true;
}

result.left = (int)vL;
result.right = (int)vR;
return result;
}

驗證：縮放是否守住了比例？

以下表格展示了多組輸入經過等比例縮放後，速度比是否依然正確：

輸入 \(V_L\) 輸入 \(V_R\) 原始比例 縮放後 \(V_L\) 縮放後 \(V_R\) 縮放比例 比例守恆？
300 150 2.00 255 127 2.01
400 200 2.00 255 127 2.01
510 170 3.00 255 85 3.00
200 180 1.11 200 180 1.11 無需縮放
備註 微小的比例誤差（如 2.00 vs 2.01）是因為整數取整（truncation）造成的。 在實際應用中，這個精度等級完全可以接受。如果需要更高精度， 可以使用 round() 取代直接的整數截斷。

整合：縮放 + 死區補償的完整流程

在完整的馬達驅動流程中，等比例縮放與死區補償應按照正確的順序串聯： 先進行縮放以保護比例，再套用死區映射以消除靜摩擦。

// 完整的 PWM 輸出管線
void applyMotorOutput(float rawL, float rawR) {
// Step 1: 等比例縮放
NormalizedPWM n = normalizePWM(rawL, rawR);

// Step 2: 死區補償
int finalL = compensate(n.left, DEAD_ZONE_L);
int finalR = compensate(n.right, DEAD_ZONE_R);

// Step 3: 輸出至馬達
analogWrite(PIN_L, finalL);
analogWrite(PIN_R, finalR);
}
提示 記住核心原則：先縮放，再補償。如果順序相反（先補償死區再縮放）， 死區映射產生的偏移量也會被縮放，導致補償效果失真。 正確的管線順序是：運動學計算 → 等比例縮放 → 死區補償 → PWM 輸出。

進階單元 10：PWM 物理極限 (PWM Limits & Scaling)

歡迎來到「PWM 物理極限」單元！這是一個將純粹的數學模型對齊真實物理世界的重要實作環節。當軟體計算出的理想速度值超過了硬體能提供的極限（飽和），或是低於硬體啟動的門檻（死區），我們必須透過演算法介入，保護控制系統的幾何完整性與穩定度。

🎯 作業目標

  1. 理解物理限制：觀察並分析單純的數值裁剪 (Clipping) 會如何破壞雙輪速比與運動軌跡。
  2. 實作等比例縮放 (Scaling)：開發能維持速度幾何比例的演算法，確保車體即使降速也能維持正確轉向半徑。
  3. 馬達死區補償 (Dead-zone Compensation)：測量並補償馬達的靜摩擦死區，實現極低速下也能平順轉向。

📝 任務說明

任務 1：飽和軌跡觀測 (Saturation Analysis Lab)

  1. 實作一組會超出 PWM 255 極限的速度指令。
  2. 僅使用 constrain(val, 0, 255) 直接裁剪超出範圍的值。
  3. 觀察車體實際運動與預期軌跡的差異，並在報告中分析原因（為什麼高速圓周運動會變形？）。

任務 2：等比例縮放實作 (Scaling Logic Lab)

  1. 於 src/limit_logic.js (或 C++ 對應之 motor_limits.h) 實作等比例縮放演算法。
  2. 確保當任意輪速需求 > 255 時，按照相同比例降階雙輪速度。
    • 提示：取得最大值 maxVal，並計算縮放因子 Factor = 255 / maxVal。
  3. 自動評分：提交程式碼，我們將執行 tests/scaling.test.js 以驗證您的縮放邏輯是否能維持輸入之速度比。

任務 3：馬達死區補償與線性化

  1. 請開啟 CALIBRATION.md，實際測量並記錄您手上實體車的左右馬達啟動最小值 (minPWM)。
  2. 在您的程式碼中實作線性映射機制，將 0~255 的軟體指令映射至 minPWM~255 的硬體輸出。
  3. 展現低速控制能力，拍攝車輛於極低指令 (如 PWM=50) 時的平順轉向影片。

  • 📂 tests/ratio_integrity.test.js：自動化評分腳本。專門測試「比例守恆」——輸入超量需求如 $(300, 150)$，檢核輸出是否為 $(255, 127.5)$（維持 2:1），而非錯誤的裁剪結果 $(255, 150)$。

2. 作業任務部署細節

任務 1：飽和軌跡觀測 (Saturation Analysis Lab)

  • 目標：親眼目睹「數學正確但物理錯誤」導致的系統失控。
  • Classroom 部署建議：
    • 故障模擬檢核：學員需輸入超出 255 的幾何指令，並使用 constrain() 處理。
    • 軌跡誤差分析：要求學員錄影展示：為什麼在高速下圓周運動會「變形」？（原因：一輪飽和而另一輪尚未飽和，破壞了 ICC 的速比要求）。
    • README 紀錄：分析裁剪 (Clipping) 如何導致運動學模型在極限狀態下失效。

任務 2：等比例縮放實作 (Scaling Logic Lab)

  • 目標：維繫幾何比例優先的「安全降級」策略。
  • Classroom 部署建議：
    • 縮放演算法實作： 
      // 學生需實作：動態比例縮放
let maxVal = Math.max(vL, vR);
if (maxVal > 255) {
    let factor = 255 / maxVal;
    vL *= factor; vR *= factor;
}
    • 修正驗證：重新執行任務 1，證明雖然車速變慢了，但轉彎半徑依然精確維持在理論預期。
    • Autograding：透過多組超量數據，檢核 normalize() 函式是否確實守住了速度比例。

任務 3：馬達死區補償與線性化 (Dead-zone Compensation)

  • 目標：消除硬體靜摩擦力，實現「緩啟動」與「低速精確轉向」。
  • Classroom 部署建議：
    • 物理採樣要求：學員需測量並記錄左右輪各自的啟動臨界 PWM 值。
    • 映射函式實作：使用 map(val, 0, 255, minPWM, 255) 建立虛擬的線性控制通道。
    • 平滑度演示：錄製影片展示車體在 PWM=50（極低速）下，依然能毫無卡頓地平穩啟動並執行轉向。
    • Push Final：提交針對其個體車體完成校準的最終版馬達驅動模組。
