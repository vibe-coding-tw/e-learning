基礎 1 - 硬體連線

驅動程式與通訊埠原理

打通橋樑：理解 USB-to-UART、COM Port 與 Baud Rate。建立穩定的硬體開發基石。

🔌 1. 數位翻譯官：USB-to-UART 橋接晶片

通訊協定的衝突

電腦說的是 USB 語言（高速）；而 ESP32 說的是 UART 語言（序列）。兩者直接對話是無法溝通的。

💻 USB ↔️ 橋接晶片 ↔️ 🤖 UART

橋接晶片的角色

常見型號如 CP2102, CH340。負責將 USB 包還原為序列信號。

驅動程式的本質

給作業系統看的「說明書」。不安裝，系統就不知道如何操作橋接晶片。

🚪 2. 什麼是通訊埠 (COM Port)？

邏輯抽象化

安裝後，系統會將晶片「虛擬化」為一個 COM 埠。這就像為裝置開了一個專屬的「門牌號碼」。

⚠️ 唯一性與衝突

編號衝突或找不到門牌是失敗最常見原因。

// Terminal Check
> ls /dev/tty.*
/dev/tty.Bluetooth-In
/dev/tty.usbserial-0001
/dev/tty.wlan-debug

⏱️ 3. 序列通訊的核心參數：Baud Rate

波特率 (Baud Rate)：定義每秒傳輸 bit。雙方必須遵守的「溝通語速」。

9600 傳統速率。
115200 ESP32 標準速率。
921600 極速燒錄。

🚫 為何看見亂碼？

「語速不一，數據就會在解碼時崩潰，形成亂碼。」

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。

任務 2：Baud Rate 匹配實驗 (Baud Rate Synchronization)

角色扮演：通訊協定測試員 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 語速約定：代碼中使用 Serial.begin(115200)。
  2. 故意失步：修改 platformio.ini 的 monitor_speed 為 9600。
  3. 亂碼截圖：擷取輸出亂碼畫面，存入 assets。
  4. 同步修復：調回 115200。解釋同步通訊重要性。

🎯 教師點評標準 (Tutor Benchmarks)

任務 3：Loopback 自我對話測試 (Hardware Loopback Test)

角色扮演：硬體診斷工程師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 物理環回連接：使用杜邦線短路板上的 TX 與 RX。
  2. 回音測試：啟動 Monitor 輸入字符，驗證數據反射。
  3. 診斷結論撰寫：判斷 USB 轉串口鏈路是否健康。
  4. 最終 Commit：整理實驗圖片並 Push。

🎯 教師點評標準 (Tutor Benchmarks)
