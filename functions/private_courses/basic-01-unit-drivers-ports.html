基礎 1 - 硬體連線

驅動程式與通訊埠原理

打通電腦與晶片之間的溝通橋樑。理解 USB-to-UART、COM Port 與 Baud Rate 的奧秘。

🔌 1. 數位翻譯官：USB-to-UART 橋接晶片

通訊協定的衝突

電腦說的是 USB 語言（高速、複雜）；而 ESP32 核心說的是 UART 語言（簡單、序列）。兩者直接對話是無法溝通的。

💻 USB (高速) ↔️ 橋接晶片 (翻譯) ↔️ 🤖 UART (序列)

橋接晶片的角色

開發板上常見型號如 CP2102, CH340, 或 FT232，負責將 USB 包封裝與還原為序列信號。

驅動程式的本質

給作業系統看的「說明書」。如果沒安裝驅動，作業系統就不知道如何操作這顆橋接晶片。

門 2. 什麼是通訊埠 (COM Port)？

邏輯抽象化

當驅動正確安裝後，作業系統會將偵測到的晶片「虛擬化」為一個 COM 埠（Windows 為 COMx；Mac 為 /dev/tty...）。這就像為裝置開了一個專屬的「門牌號碼」。

⚠️ 唯一性與衝突

每個設備都有獨一無二的編號。編號衝突或找不到門牌是初學者上傳失敗最常見的原因。

// Terminal Check
> ls /dev/tty.*
/dev/tty.Bluetooth-In
/dev/tty.usbserial-0001
/dev/tty.wlan-debug

⏱️ 3. 序列通訊的核心參數：Baud Rate

波特率 (Baud Rate)：定義每秒傳輸多少 bit。這是通訊雙方必須遵守的「溝通語速」。

9600 傳統、慢速。
115200 ESP32 最常用的標準速率。
921600 極速，燒錄代碼常用。

🚫 為何看見亂碼？

「如果一方講太快 (115200)，另一方聽太慢 (9600)，數據就會在解碼時崩潰，形成亂碼。」

硬體通訊鏈路實驗室

任務 1：驅動與硬體偵錯 (Driver & Hardware Debugging)

角色扮演：部署工程師 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 至本地。
  2. 身份識別：在 README.md 貼上裝置管理員（Windows）或執行 ls /dev/tty.*（Mac）偵測到 ESP32 COM 埠的圖。
  3. 晶片型號標註：細看開發板上 USB 孔旁的小正方形晶片，指出型號（如 CP2102/CH340），並搜尋對應驅動。
  4. 排障筆記：若無法連線，記錄更換線材（數據 vs 充電）後的結果，並上傳至 Repo 的 docs/debug.log。

  • [ ] 排障意識：是否能主動區分線材造成的物理損壞與驅動造成的邏輯異常？

任務 2：Baud Rate 匹配實驗 (Baud Rate Synchronization)

角色扮演：通訊協定測試員 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 語速約定：在 Arduino 代碼中使用 Serial.begin(115200)。
  2. 故意失步實驗：修改 platformio.ini 的 monitor_speed 為常用的錯誤值（如 9600）。
  3. 亂碼截圖：擷取 Serial Monitor 輸出亂碼時的畫面，存入 assets/garbled.png。
  4. 同步修復：將速率調回 115200，截圖記錄恢復正常的輸出，並解釋同步通訊對於「起始位」辨識的重要性。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 邏輯鏈條完整性：代碼參數與配置文件是否能達成 100% 的吻合？
  • [ ] 文檔敘述：是否能在 README 中清晰說明 Baud Rate 錯誤為何會導致解碼崩潰？

任務 3：Loopback 自我對話測試 (Hardware Loopback Test)

角色扮演：硬體診斷工程師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 物理環回連接：使用杜邦線短路板上的 TX 與 RX。
  2. 回音測試：啟動 Serial Monitor 隨意輸入字符，驗證是否能立即在下方視窗看到「一模一樣」的數據反射。
  3. 診斷結論撰寫：根據實驗結果在 README 撰寫診斷，判斷 USB 轉串口鏈路是否健康。
  4. 最終 Commit：整理所有實驗圖片與報告，推送到作業倉庫。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 環回概念理解：是否能透過實驗理解「收發分離、物理共地」的全雙工特性？
  • [ ] 專業除錯技巧：是否掌握了在不寫程式的情況下，用「短路法」快速定位硬體故障點的專業技能？

GitHub Classroom 詳細操作指南

本單元教學重點：讓學生克服硬體開發的第一道高牆——「連不上電腦」。

  1. 第一步：基礎設施建設 - 透過驅動安裝實習，讓學生理解電腦如何與微控制器建立物理連結。
  2. 第二步：對抗盲目除錯 - 通過 Loopback 測試，教會學生「隔離問題 (Isolate the problem)」的專業工程方法。
  3. 第三步：標準化配置 - 養成在 `platformio.ini` 中標註通訊速率的好習慣，預防未來的低級 Bug。
