🔌 1. 數位翻譯官：USB-to-UART 橋接晶片

通訊協定的衝突

電腦說的是 USB 語言（高速）；而 ESP32 說的是 UART 語言（序列）。兩者直接對話是無法溝通的。

💻 USB ↔️ 橋接晶片 ↔️ 🤖 UART

橋接晶片的角色

常見型號如 CP2102, CH340。負責將 USB 包還原為序列信號。

驅動程式的本質

給作業系統看的「說明書」。不安裝，系統就不知道如何操作橋接晶片。

🚪 2. 什麼是通訊埠 (COM Port)？

邏輯抽象化

安裝後，系統會將晶片「虛擬化」為一個 COM 埠。這終端就像為裝置開了一個專屬的「門牌號碼」。

⚠️ 唯一性與衝突

編號衝突或找不到門牌是失敗最常見原因。

/dev/tty.usbserial-0001

⏱️ 3. 序列通訊的核心參數：Baud Rate

波特率 (Baud Rate)：定義每秒傳輸 bit。雙方必須遵守的「溝通語速」。

9600 傳統速率。
115200 ESP32 標準速率。
921600 極速燒錄。

環境通聯與偵錯

目標：連接開發板，識別其 USB 轉序列晶片與通訊埠 (COM Port)，並記錄於診斷文件中。

  • 將開發板透過 USB 數據線連接至電腦
  • 在裝置管理員 (Windows) 或終端機 (Mac: ls /dev/tty.*) 找到埠號
  • 識別並確認橋接晶片型號 (例如 CP2102 或 CH340)
  • 在 docs/Diagnostic.md 報告中記錄埠號與晶片型號
  • 截圖埠號偵測畫面，存為 assets/port_detected.png
繳交與驗證說明 確認學員能區分 localhost 與埠號的不同，並確保在 docs/Diagnostic.md 中確實記錄埠號。完成後請點擊此卡片繳交。
  作業 1-2
約 45 分鐘

亂碼解密與速率同步實驗

目標：體驗並解決序列通訊中常見的波特率 (Baud Rate) 不匹配導致的亂碼問題。

  • 使用 PlatformIO 開啟專案並燒錄測試程式
  • 開啟序列監視器 (Serial Monitor)，觀察並記錄預設 9600 產生的亂碼
  • 開啟 platformio.ini，尋找並修正 monitor_speed = 115200
  • 重新啟動序列監視器，驗證輸出文字清晰正常，不再是亂碼
  • 截圖正常接收資料的監視器畫面，存為 assets/serial_ok.png
波特率知識點 ESP32 的標準通訊語速是 115200 bps。如果電腦的接收語速不同步，解碼數據就會崩潰並產生亂碼 ▒▓▒░。
  作業 1-3
約 60 分鐘

Loopback 自我對話測試

目標：使用杜邦線短路 TX 與 RX 引腳進行硬體環回測試，診斷通訊晶片收發功能是否正常。

  • 使用杜邦線將板子上的 TX 與 RX 引腳直接短接/短路
  • 開啟序列監視器，在傳輸發送欄輸入任意英文字串
  • 發送訊息，確認監視器螢幕上即時顯示相同的「回音」字串
  • 在 docs/Diagnostic.md 中記錄你的測試結果與故障診斷結論
硬體環回測試原理 當 TX 直接連 RX 時，晶片自己發出的電訊號會立刻被自己接收回來。如果沒有回音，代表晶片或接線有故障，是序列排除問題的終極利器！

Unit: Drivers & Ports (驅動程式與通訊埠)

這是硬體開發的第一站：連線穩定度測試。在這個單元中，你將學會如何識別硬體埠號、處理波特率 (Baud Rate) 不匹配導致的亂碼，以及使用 Loopback 測試來排除硬體故障。

🎯 學習目標

  1. 正確安裝並識別 USB-to-Serial 驅動程式。
  2. 了解序列通訊中波特率 (Baud Rate) 同步的重要性。
  3. 掌握 Loopback (環回測試) 方法，獨立診斷通聯故障。

🛠 任務清單

任務 1：環境通聯與偵錯 (Driver & Hardware Debugging)

  • 步驟：
    1. 將開發板連接至電腦。
    2. 在裝置管理員 (Windows) 或終端機 (Mac/Linux: ls /dev/tty.*) 中找到對應的埠號。
    3. 確認橋接晶片型號 (CP2102, CH340 等)。
  • 驗證：在 docs/Diagnostic.md 中記錄你的埠號與晶片型號。

任務 2：亂碼解密實驗 (Baud Rate Synchronization Lab)

  • 步驟：
    1. 使用 PlatformIO 開啟專案並燒錄程式。
    2. 開啟序列監視器 (Serial Monitor)。
    3. 你應該會看到亂碼，因為 platformio.ini 中的 monitor_speed 被故意設為 9600，而程式碼使用的是 115200。
    4. 將 platformio.ini 中的 monitor_speed 修正為 115200。
  • 驗證：修正後截圖清晰的輸出畫面，存為 assets/serial_ok.png。

任務 3：Loopback 自我對話測試 (Hardware Loopback Test)

  • 步驟：
    1. 使用杜邦線將板子上的 TX 與 RX 引腳直接短路。
    2. 開啟序列監視器。
    3. 輸入文字並發送，觀察是否有「回音」顯示在螢幕上。
  • 驗證：在 docs/Diagnostic.md 中記錄測試結果與結論。

    • 📁 專案結構

    .
├── platformio.ini       # 核心：請在此修正速率設定 (115200)
├── src/main.cpp         # 測試：每秒發送一次信號的基礎程式
├── docs/Diagnostic.md   # 報告：填寫你的診斷過程與結果
├── assets/              # 存放截圖的目錄
└── README.md            # 本說明文件
