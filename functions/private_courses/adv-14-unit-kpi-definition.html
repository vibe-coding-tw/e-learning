🎯 1. 性能指標 (KPI) 三大維度

工程師不相信直覺，只相信證據。要評估一個控制系統的優劣，必須從時間、準繩度與穩定性三個面向進行精確量化。

System Benchmarking Dimensions

  • 時間性 (Temporal)：延遲 (RTT/Latency) 是核心。超過 100ms 的迴路反饋即視為不可接受。
  • 準確性 (Accuracy)：平均誤差值與抖動係數 (Jitter)。反映車輛行駛的軌跡平滑度。
  • 可靠性 (Reliability)：通訊封包丟失率與控制失效自動復歸效率。

⚖️ 2. 指標權衡與服務水準協議 (SLA)

在硬體資源受限的邊緣環境下，高更新率 (FPS) 與低延遲 (Latency) 往往是矛盾的。我們必須定義「複合效能分數」來衡量特定任務下的綜合表現。

System Optimization Equation

\[Score_{System} = \alpha \cdot FPS + \beta \cdot \frac{1000}{Latency_{ms}}\]

📈 3. 數據驅動優化 (Data-driven Dev)

真正的優化絕不憑感覺。學會「量化改進比率 (%)」，並透過對應的對比實驗證明技術改進在邊際極端情況下的穩定性提升。

Benchmarking Standard Operating Procedure

  • 基準建立 (Baseline)：在未優化前進行多樣本取樣，建立穩定的統計基準。
  • 變因控制 (Variable Control)：一次僅改動一個技術變量，確保優化效益可被追蹤溯源。
  • 收斂性驗證：確保優化後的性能在長時間運行下保持收斂，無內存洩漏或性能衰退。

核心任務：用數據書寫技術規格

量化系統效能，產出具備工程說服力的性能規格書。透過 Git Workflow 建立數據驅動的優化實踐，並證明你的系統實力。

【作業 14-1】

「實時性能監測器」

【作業 14-2】

「官方測試報告」

執行各場景下的壓力壓力測試。找出控制產生顯著「遲鈍感」的 KPI 臨界點。產出具備工業背書強度的 README 性能規格書，定義你的系統安全邊際。

⚡ 45 min

【作業 14-3】

「效能提升挑戰」

單元課程 進階 14 - 性能指標

KPI 實驗室：性能量化與邊際壓力報告

任務 1：專業級數位 KPI 儀表板 (Metric-Driven Dashboard Lab)

角色扮演：數據分析視覺化專家 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 遙測邏輯實作：在前端 Javascript 追蹤 Ping/Pong 時間差，並計算移動平均延遲 (Rolling Average Latency)。
  3. 自適應感知 UI：實作三色警報邏輯。當延遲 > 120ms 時，自動觸發「警告燈號」警示背景色。
  4. Push Final：提交具備「即時性能透視」功能的儀表板代碼與使用截圖。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 監測反饋度：顯示的指標是否能靈敏反映當前通訊品質？（如：當遮擋 BLE 訊號時是否有數值變化）。
  • [ ] 代碼健壯性：對於丟失的 Pong 封包是否存在超時處理機制，防止計算 RTT 時發生無窮大偏差？

任務 2：系統性能規格書 (System Spec Benchmarking Lab)

角色扮演：品質保證 (QA) 架構師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 全場景壓力測試：在不同距離、障礙物環境下各取樣 200 筆以上數據，計算 StdDev (標準差)。
  2. 失效邊界定義：紀錄系統在控制失效前的「臨界 KPI 數據組合」。
  3. 正式規格書彙編：在 README 中正式載錄性能指標：最大安全控制範圍、最優數據吞吐量。
  4. Push Final：推送具備「工業技術承諾」強度的完整報告文檔。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 測量嚴謹度：報告是否包含樣本量說明與測試情境描述，而非僅有單一結果？
  • [ ] 工程建議力：基於數據得出的「控制極限建議」是否具備指導初級操作員安全駕駛的價值？

任務 3：優化收益對比挑戰 (Optimization Contest Lab)

角色扮演：資深效能工程師 | 難度：⭐️⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 瓶頸擊破部署：實施一項優化（如：二進位封包打包），計算 RTT 改善比率。
  2. 提升公式導入：計算並以表格呈現優化前後的 KPI Delta \(\%\)。
  3. 權衡分析總結：在 PR 中反思：此優化是否對 PID 控制的指令即時性產生實質性幫助？
  4. Push Final：提交具備「數據驅動改進」意識的最終高效率代碼。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 測量真實性：報告中的優化收益數據是否能在測試環境中被複現？
  • [ ] 分析深度：是否能討論該優化動作對系統整體資源（如耗電、RAM）的副作用或潛在負擔？

GitHub Classroom 詳細操作指南

本單元教學重點：將「開發思維」提升為「性能工程思維」。讓學員學會量化自己的代碼產出。

  1. 重點 1：數據驅動 (Data-Driven) - 引導學員排除「我覺得比較順」這種主觀描述，要求使用 Latency、StdDev 與 RTT 等標準術語。
  2. 重點 2：資源權衡思考 - 討論優化方案 A 與 B 時，引導學員建立「性能 vs 複雜度」的工程權衡意識。
  3. 重點 3：SLA 技術承諾 - 讓學員理解工業產品必須具備「基準規格」。規格書就是工程師對產品穩定性的技術保證。
