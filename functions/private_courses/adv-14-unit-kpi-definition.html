🎯 1. 性能指標 (KPI) 三大維度

工程師不相信直覺，只相信證據。要評估一個控制系統的優劣，必須從時間、準繩度與穩定性三個面向進行精確量化。

System Benchmarking Dimensions

  • 時間性 (Temporal)：延遲 (RTT/Latency) 是核心。超過 100ms 的迴路反饋即視為不可接受。
  • 準確性 (Accuracy)：平均誤差值與抖動係數 (Jitter)。反映車輛行駛的軌跡平滑度。
  • 可靠性 (Reliability)：通訊封包丟失率與控制失效自動復歸效率。

⚖️ 2. 指標權衡與服務水準協議 (SLA)

在硬體資源受限的邊緣環境下，高更新率 (FPS) 與低延遲 (Latency) 往往是矛盾的。我們必須定義「複合效能分數」來衡量特定任務下的綜合表現。

System Optimization Equation

\[Score_{System} = \alpha \cdot FPS + \beta \cdot \frac{1000}{Latency_{ms}}\]

📈 3. 數據驅動優化 (Data-driven Dev)

真正的優化絕不憑感覺。學會「量化改進比率 (%)」，並透過對應的對比實驗證明技術改進在邊際極端情況下的穩定性提升。

Benchmarking Standard Operating Procedure

  • 基準建立 (Baseline)：在未優化前進行多樣本取樣，建立穩定的統計基準。
  • 變因控制 (Variable Control)：一次僅改動一個技術變量，確保優化效益可被追蹤溯源。
  • 收斂性驗證：確保優化後的性能在長時間運行下保持收斂，無內存洩漏或性能衰退。

單元課程 進階 14 - 性能指標

adv-14-unit-kpi-definition: 性能指標 KPI 定義

本單元將帶領你進入「商用級產品」開發的核心：性能量化與基準測試 (Benchmarking)。

作為一名工程師，你不能只憑感覺判斷系統「順不順」。你需要學會定義 關鍵績效指標 (KPI)，並透過數據來證明你的系統在不同環境下的可靠性與穩定性。掌握 KPI 定義與性能評鑑後，你將擁有了開發「自動駕駛安全協議」、「伺服器級別可靠性監控」或「高可用物聯網系統」必備的頂層工程師專業素養！

🎯 學習目標

  1. 實作實時監測：在儀表板中顯示關鍵性能數據（如 RTT 延遲）。
  2. 建立視覺預警：根據性能數據自動調整 UI 狀態，為操作者提供直觀回饋。
  3. 執行基準測試：科學化地測量系統在不同環境下的表現極限。
  4. 定義性能規格 (SLA)：編寫正式的系統性能規格書。

📂 程式庫結構

  • src/kpi_tracker.js：核心指標計算邏輯，你將在這裡實作延遲累加與數據分析。
  • src/telemetry_visualizer.js：遙測視覺化模組，負責計算 RTT 並更新 UI 狀態。
  • docs/Benchmark_Log.md：實驗數據紀錄表，用於詳細記錄測試環境與多樣本數據。
  • SPEC_SHEET.md：你最終需完成的「系統性能規格書 (SLA)」。
  • tests/latency_emulator.js：延遲模擬工具，可用於測試 UI 警示邏輯。

🚀 作業任務

任務 1：專業級數位 KPI 儀表板 (Metric-Driven Dashboard Lab)

建立系統透明度，讓操控者能在連線惡化前獲得視覺預警。

  • RTT 監測實作：在 src/telemetry_visualizer.js 中透過 Ping/Pong 時戳差計算實時延遲。
  • 自適應狀態感知：實作 UI 變色邏輯。
    • 🟢 綠色 (< 80ms)：連線優質，適合精確操控。
    • 🟡 黃色 (80-120ms)：連線延遲增加，建議降低操控強度。
    • 🔴 紅色 (> 120ms)：危險延遲，系統可能產生控制不可控。

任務 2：系統性能規格書極限測試 (System Spec Benchmarking Lab)

執行科學實驗，找出系統「控制失效」的臨界點。

  • 多場景取樣：在「室內空曠」、「隔牆連線」與「長距離」三種場景下各取樣至少 200 筆通訊數據。
  • 臨界點診斷：找出當 RTT 到達多少 ms 時，PID 循跡會開始產生不可控的震盪，定義該系統的「安全邊際」。
  • 規格書彙編：將測量結果正式載錄於 SPEC_SHEET.md。

任務 3：優化收益對比挑戰 (Performance Delta Benchmark Lab)

透過數據證明你的工程優化確實有效。

  • 優化實施：嘗試實施一項優化（如：優化封包結構或處理邏輯）。
  • ROI 計算：計算優化前後 KPI 的真實提升比率 ($\Delta %$)。
  • 整合報告：在 docs/Benchmark_Log.md 中提交具備數據支撐的優化決策報告。

🛠 如何開始

  1. 安裝依賴：npm install
  2. 運行測試驗證邏輯：npm test
  3. 在提交 PR 時，請附上不同延遲狀態下的 UI 截圖或錄影。

1. 範本程式庫 (Template 程式庫) 配置建議

性能評鑑需要標準化的分析模板與監測工具，建議範本包含：

  • 📂 src/telemetry_visualizer.js：前端指標計算模組。預留 calculateRTT() 與 updateTrafficLight() 的介面，讓學員實作延遲監測與視覺警示。
  • 📂 docs/BENCHMARK_REPORT.md：正式基準測試報告。提供表格模板，包含：測試環境（距離/障礙物）、樣本數、平均延遲、標準差 (Jitter) 等欄位。
  • 📂 tests/latency_emulator.js：延遲模擬工具。這是一個輔助腳本，能人為注入 200ms 的延遲，用來測試學員寫的「效能警示 UI」是否能正確變色。
  • 📂 SPEC_SHEET.md：系統性能規格書。學員最終需在此填入車輛的「官方參數」（例如：最大穩定連線距離、RTT 通訊極限值）。

2. 作業任務部署細節

任務 1：專業級數位 KPI 儀表板 (Metric-Driven Dashboard Lab)

  • 目標：建立系統透明度，讓操控者能在連線惡化前獲得視覺預警。
  • Classroom 部署建議：
    • RTT 監測實作：學員需在前端 JavaScript 透過 Ping/Pong 時戳差計算實時延遲。
    • 自適應狀態感知：實作 UI 變色邏輯。綠色 (< 80ms)、黃色 (80-120ms)、紅色 (> 120ms)。
    • PR 截圖驗證：提交 PR 時需附上不同延遲狀態下的 UI 截圖，證明監測邏輯與視覺回饋同步。

任務 2：系統性能規格書極限測試 (System Spec Benchmarking Lab)

  • 目標：執行科學實驗，定義「控制失效」的臨界數據點。
  • Classroom 部署建議：
    • 多場景取樣：學員需在「室內空曠」、「隔牆連線」與「長距離」三種場景下各取樣至少 200 筆通訊數據。
    • 臨界點診斷：找出當 RTT 到達多少 ms 時，PID 循跡會開始產生不可控的震盪，這就是該系統的「安全極限點」。
    • 規格書彙編：在 README 中正式載錄你的「Vibe-Car 性能規格」，這對於工業產品推廣至關重要。

任務 3：優化收益對比挑戰 (Performance Delta Benchmark Lab)

  • 目標：計算「工程改良」帶來的實質投資報酬率 (ROI)。
  • Classroom 部署建議：
    • 優化幅度計算：學員實施一項優化（如：改用二進位封包），計算優化前後 KPI 的真實提升比率 ($\Delta %$)。
    • 控制平穩度反思：說明延遲降低後，對車體「高速過彎」時的側向加速度穩定度有何實質幫助。
    • Push Final：提交具備「數據支撐優化決策」的最終整合報告。

3. 性能工程師導師點評標準 (KPI & Quality Standards)

此單元的價值在於 「對系統行為的數字掌控力與可靠性保證」：

  • [ ] 視覺導航力：前端儀表板的 KPI 顯示是否直觀，能讓一般操作者快速理解系統健康度？
