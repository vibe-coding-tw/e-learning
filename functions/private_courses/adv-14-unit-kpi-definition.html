歡迎來到本模組！本單元聚焦於 系統性能指標 (KPI) 定義。 「系統跑得順不順」不能憑感覺判斷——延遲多少毫秒算可接受？封包丟失率到什麼程度控制會失效？ 本模組將教你建立精確的量化指標體系，從 RTT 延遲、抖動係數到 SLA 服務水準定義，用數據而非直覺來評估並優化你的控制系統。

🎯 1. 性能指標 (KPI) 三大維度

工程師不相信直覺，只相信數據。當團隊在爭論系統「夠不夠快」時，如果沒有人拿得出精確的數字，這場討論就毫無意義。要科學地評估一個控制系統的優劣，必須從時間性（Temporal）、準確性（Accuracy）與可靠性（Reliability）三個維度進行量化分析。

這三大維度構成了系統性能評估的完整框架。每個維度都有對應的量化指標與量測方法，讓你能夠用數據取代感覺，用證據取代猜測。

維度一：時間性（Temporal）— 延遲與回應速度

在即時控制系統中，延遲（Latency）是最關鍵的 KPI。它直接決定了操作者的指令能否被即時執行。最常用的量測指標是 RTT（Round-Trip Time）——從前端發出指令到收到韌體回應的完整來回時間。

// RTT 量測的基本實作
const t_send = performance.now();
sendPingToDevice();

// 當收到 Pong 回應時
onPongReceived = () => {
  const rtt = performance.now() - t_send;
  rttHistory.push(rtt);
  updateDashboard(rtt);
};
延遲等級 RTT 範圍 控制體驗 建議行動
優秀 < 80 ms 操控順暢，適合精確循跡與高速過彎 正常操作
警戒 80 – 120 ms 可感知的延遲，轉向反應變鈍 降低操控強度，準備切換安全模式
危險 > 120 ms 控制明顯滯後，PID 震盪風險 自動切換為安全停車模式

維度二：準確性（Accuracy）— 誤差與抖動

即使延遲很低，如果數據傳輸的穩定性不足，控制品質依然會惡化。抖動係數（Jitter）衡量的是連續多次 RTT 量測之間的波動程度。高抖動意味著有時快有時慢，PID 控制器無法獲得穩定的回饋訊號，車輛軌跡就會出現不可預測的擺動。

備註 抖動係數的計算方式為連續 RTT 差值的平均絕對值：Jitter = Σ|RTT(i) - RTT(i-1)| / (n-1)。理想值應低於 15ms。當 Jitter 超過 30ms 時，即使平均 RTT 在安全範圍內，控制品質也會顯著下降。

維度三：可靠性（Reliability）— 封包丟失與自動復歸

在無線通訊環境中，封包丟失是無法完全避免的。可靠性維度關注的是：封包丟失率（Packet Loss Rate）有多高？以及當丟失發生時，系統能否自動偵測並復歸？

對於物聯網控制系統，可接受的封包丟失率通常要求低於 2%。超過 5% 時，即時控制將變得不可靠。系統應具備心跳偵測（Heartbeat）機制——如果連續 3 個心跳未收到回應，即判定為連線中斷並啟動安全程序。

警告 封包丟失率必須在「最嚴苛的環境」下量測，而非僅在理想條件下。室內空曠環境的丟失率可能僅 0.1%，但隔一面牆就可能飆升到 8%。產品規格書必須標註最差情況下的數值，這才是真正的安全邊際。

三大維度的整合視角

KPI 維度 核心指標 量測工具 合格基準
時間性 RTT 延遲（ms） Ping/Pong 時戳差 < 80 ms（均值）
準確性 Jitter 抖動（ms） 連續 RTT 差值分析 < 15 ms
可靠性 封包丟失率（%） 心跳偵測統計 < 2%
提示 在實作中，建議將這三大 KPI 整合到同一個即時儀表板上，搭配「紅黃綠」三色號誌燈，讓操控者在駕駛過程中能夠一眼掌握系統健康度。這正是作業 14-1 的核心目標。

⚖️ 2. 指標權衡與服務水準協議 (SLA)

在理想世界中，我們希望每個 KPI 都達到極致——延遲最低、準確度最高、可靠性 100%。但在硬體資源受限的邊緣運算環境下，這些目標之間往往存在不可調和的矛盾。例如，提高影像串流的更新率（FPS）會佔用更多通訊頻寬，導致控制指令的延遲增加。這就是為什麼我們需要定義服務水準協議（SLA, Service Level Agreement）——一份明確的性能規格書，定義系統在各種條件下「必須達到」和「容許偏離」的性能底線。

指標之間的權衡取捨（Trade-off）

工程設計的本質是在限制條件下做出最佳取捨。以下是物聯網控制系統中最常見的三組性能衝突：

衝突項 方向 A 方向 B 決策依據
頻寬分配 提高影像 FPS（視覺導航佳） 降低控制延遲（即時操控佳） 根據任務優先級動態分配
封包大小 完整 JSON（易除錯、可讀性佳） 二進位壓縮（頻寬佔用小） 開發階段用 JSON，上線切換二進位
重試策略 積極重試（可靠性高） 放棄舊封包（即時性高） 控制指令不重試，狀態回報可重試

複合效能分數（Composite Performance Score）

為了在單一數值中捕捉多維度的性能表現，我們可以定義一個加權複合分數。透過調整權重參數 \(\alpha\) 與 \(\beta\)，可以針對不同的任務場景偏重不同的性能面向：

\[Score_{System} = \alpha \cdot FPS + \beta \cdot \frac{1000}{Latency_{ms}}\]

其中 \(\alpha\) 代表視覺導航的權重，\(\beta\) 代表控制反饋速度的權重。例如，在循跡模式下，控制延遲比影像更重要，因此 \(\beta\) 應遠大於 \(\alpha\)；在遙控探索模式下，影像清晰度更關鍵，則 \(\alpha\) 權重提升。

備註 複合分數的權重不應憑感覺設定。正確做法是：先收集不同權重配置下的實際操控評分（由操作者主觀評價），再透過迴歸分析找出最佳權重組合。這就是「數據驅動決策」的精神。

撰寫服務水準協議（SLA）

一份合格的 SLA 必須包含以下四個要素：指標定義、合格閾值、量測條件與違約處置。以下是一份適用於物聯網控制系統的 SLA 範例：

SLA 條款 指標 合格閾值 量測條件
控制延遲 RTT 均值 ≤ 80 ms（P95 ≤ 120 ms） 室內空曠，距離 ≤ 5m
延遲穩定性 Jitter ≤ 15 ms 連續 200 筆取樣
通訊可靠性 封包丟失率 ≤ 2% 10 分鐘持續通訊
故障復歸 重連時間 ≤ 3 秒 模擬斷線後自動重連
重要 SLA 中的「量測條件」欄位至關重要。如果你在 1 公尺距離內測得 RTT 為 38ms，卻宣稱系統「延遲低於 40ms」而不標註測試條件，這就是工程上的不誠實。真正有價值的 SLA 必須明確記錄環境參數——距離、障礙物數量、環境電磁干擾程度等。

邊際壓力測試（Boundary Stress Test）

SLA 的制定不能只基於「最佳條件」下的數據。你需要在越來越嚴苛的條件下測試，找出系統的性能崩潰點——也就是 KPI 從「合格」急遽惡化到「不可用」的臨界距離或臨界負載。

提示 推薦測試三種場景：(1) 室內空曠（基準線）、(2) 隔一面混凝土牆、(3) 距離 10 公尺以上。你會發現 BLE 連線的 RTT 在穿牆後可能從 38ms 飆升到 195ms——這個數據差距，就是你系統的「安全邊際」。

📈 3. 數據驅動優化 (Data-driven Dev)

真正的工程優化絕不是憑感覺說「好像變快了」。數據驅動開發（Data-driven Development）要求你在每一次優化前後，都用精確的數據來量化改進幅度。這種方法論來自工業工程中的持續改善（Kaizen）理念——每一次微小的、可量化的改進，最終都會累積成巨大的性能飛躍。

本單元將教你建立標準化的基準測試流程（Benchmarking SOP），並學會計算優化的真實投資報酬率。

基準測試標準作業流程

任何優化工作開始之前，你必須先建立一個可信賴的性能基準線（Baseline）。以下三個步驟構成了標準化的基準測試流程：

1

建立基準（Establish Baseline）

在未進行任何優化的原始狀態下，收集至少 200 筆以上的性能數據。記錄平均值、中位數、標準差與第 95 百分位數（P95）。單一樣本的數據毫無統計意義。

2

控制變因（Control Variables）

一次只改動一個技術變量。例如，如果你同時更換了封包格式和通訊頻率，當性能改善時，你無法確定哪個變更才是真正的功臣。科學實驗的鐵則：單一變因。

3

收斂性驗證（Convergence Check）

優化後的性能數據必須在長時間運行下保持穩定。如果 RTT 在前 5 分鐘很低但之後逐漸升高，可能存在記憶體洩漏（Memory Leak）或緩衝區堆積。持續運行至少 10 分鐘來驗證收斂性。

計算優化的真實提升比率

優化效益的量化使用改進比率（Delta Percentage）公式：

\[\Delta\% = \frac{Metric_{before} - Metric_{after}}{Metric_{before}} \times 100\%\]

例如，如果優化前的平均 RTT 為 95ms，優化後降至 52ms，則改進比率為：

\[\Delta\% = \frac{95 - 52}{95} \times 100\% = 45.3\%\]

// 優化前後的基準對比範例
const baseline = { avgRTT: 95, jitter: 22, lossRate: 3.1 };
const optimized = { avgRTT: 52, jitter: 8, lossRate: 0.9 };

const delta = {};
for (const key of Object.keys(baseline)) {
  delta[key] = ((baseline[key] - optimized[key]) / baseline[key] * 100).toFixed(1);
}
// delta = { avgRTT: "45.3", jitter: "63.6", lossRate: "71.0" }

常見的優化方向

以下列出在物聯網控制系統中，效果最顯著的幾個優化方向及其預期效益：

優化方向 具體做法 預期 RTT 改進 風險
封包壓縮 將 JSON 格式改為二進位協議 -30% ~ -50% 除錯難度增加
頻率優化 降低非必要數據的傳輸頻率 -15% ~ -25% 影像流暢度可能下降
處理管線化 將序列處理改為非同步管線 -20% ~ -40% 需處理競態條件
連線參數調整 調整 BLE Connection Interval -10% ~ -30% 功耗可能增加
警告 「過度優化」（Over-optimization）同樣危險。如果系統已經穩定在 40ms RTT 而 SLA 要求 80ms，那麼為了再降 5ms 而大幅重構封包格式是不划算的——工程時間的投資報酬率（ROI）為負。優化應該聚焦於「離 SLA 閾值最近的瓶頸」。
提示 在提交優化報告時，不要只呈現「優化後的數字」——必須同時呈現基準線數據、優化後數據、改進比率以及測試環境參數。一份具備工業說服力的報告，讓讀者能夠獨立重現你的實驗。這正是「數據驅動」與「感覺驅動」之間的根本差異。

adv-14-unit-kpi-definition: 性能指標 KPI 定義

本單元將帶領你進入「商用級產品」開發的核心：性能量化與基準測試 (Benchmarking)。

作為一名工程師，你不能只憑感覺判斷系統「順不順」。你需要學會定義 關鍵績效指標 (KPI)，並透過數據來證明你的系統在不同環境下的可靠性與穩定性。掌握 KPI 定義與性能評鑑後，你將擁有了開發「自動駕駛安全協議」、「伺服器級別可靠性監控」或「高可用物聯網系統」必備的頂層工程師專業素養！

🎯 學習目標

  1. 實作實時監測：在儀表板中顯示關鍵性能數據（如 RTT 延遲）。
  2. 建立視覺預警：根據性能數據自動調整 UI 狀態，為操作者提供直觀回饋。
  3. 執行基準測試：科學化地測量系統在不同環境下的表現極限。
  4. 定義性能規格 (SLA)：編寫正式的系統性能規格書。

📂 程式庫結構

  • src/kpi_tracker.js：核心指標計算邏輯，你將在這裡實作延遲累加與數據分析。
  • src/telemetry_visualizer.js：遙測視覺化模組，負責計算 RTT 並更新 UI 狀態。

  • SPEC_SHEET.md：你最終需完成的「系統性能規格書 (SLA)」。
  • tests/latency_emulator.js：延遲模擬工具，可用於測試 UI 警示邏輯。

🚀 作業任務

任務 1：專業級數位 KPI 儀表板 (Metric-Driven Dashboard Lab)

建立系統透明度，讓操控者能在連線惡化前獲得視覺預警。

  • RTT 監測實作：在 src/telemetry_visualizer.js 中透過 Ping/Pong 時戳差計算實時延遲。
  • 自適應狀態感知：實作 UI 變色邏輯。
    • 🟢 綠色 (< 80ms)：連線優質，適合精確操控。
    • 🟡 黃色 (80-120ms)：連線延遲增加，建議降低操控強度。
    • 🔴 紅色 (> 120ms)：危險延遲，系統可能產生控制不可控。

任務 2：系統性能規格書極限測試 (System Spec Benchmarking Lab)

執行科學實驗，找出系統「控制失效」的臨界點。

  • 多場景取樣：在「室內空曠」、「隔牆連線」與「長距離」三種場景下各取樣至少 200 筆通訊數據。
  • 臨界點診斷：找出當 RTT 到達多少 ms 時，PID 循跡會開始產生不可控的震盪，定義該系統的「安全邊際」。
  • 規格書彙編：將測量結果正式載錄於 SPEC_SHEET.md。

任務 3：優化收益對比挑戰 (Performance Delta Benchmark Lab)

透過數據證明你的工程優化確實有效。

  • 優化實施：嘗試實施一項優化（如：優化封包結構或處理邏輯）。
  • ROI 計算：計算優化前後 KPI 的真實提升比率 ($\Delta %$)。
  • 整合報告：在 docs/Benchmark_Log.md 中提交具備數據支撐的優化決策報告。
