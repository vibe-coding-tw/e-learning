🎯 1. 關鍵性能指標 (KPI) 三大維度

工程師不相信直覺，只相信數據。要評估一個控制系統，必須從時間、準確性與可靠性三個面向進行量化。

System Benchmarking Dimensions

  • 時間性 (Temporal)：延遲 (Latency) 是最重要的指標。多過 100ms 的動作回饋即視為不可接受。
  • 準確性 (Accuracy)：平均誤差與抖動係數 (Jitter)。反映系統行駛的平穩度。
  • 可靠性 (Reliability)：任務完成率與斷線復歸效率。

⚖️ 2. 指標權衡與服務水準協議 (SLA)

📈 3. 數據驅動開發 (Data-driven Dev)

優化不是憑感覺。學會「量化優化效益」：計算優化前後指標的提升比率 (%)，並證明在邊際情況下的穩定性提升。

Benchmarking Protocol

  • 基準建立 (Baseline)：在未優化前進行三次完整的壓力測試。
  • 單一變因控制：一次只改動一個參數（如：JSON 鍵名縮減），確保因果關係明確。
  • 收斂性驗證：確保優化後的指標在長時間運行下依然保持穩定收斂。

KPI 實驗室：性能量化與極限測試

任務 1：「數位儀表板」實作 (Digital KPI Dashboard)

角色扮演：數據視覺化工程師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 核心遙測計算：在前端 Javascript 代碼中記錄 Ping 發送與 Pong 回傳的時間差，計算 RTT 並轉化為延遲 (ms)。
  3. 視覺狀態反饋：實作三色燈號邏輯。當平均延遲持續超過 150ms 時，介面背景自動切換為紅色模式提醒操作者。
  4. FPS 指標顯示：實作一個滑動窗口演算法，每秒計算一次接收到的數據量，顯示為「數據更新率 (Hz)」。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 監測時效度：顯示的指標是否與實際操作反映一致？（當遮住天線時，延遲燈號應有即時變化）。
  • [ ] 代碼邏輯性：延遲計算是否排除了重複發送或過期封包的干擾？

任務 2：「性能規格書」：壓力極限測試 (System Benchmarking)

角色扮演：基準測試專家 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 壓力實驗設計：分別在「近場正常控制」與「惡意干擾環境」下各進行 5 分鐘的性能取樣。
  2. 失效邊界判定：找出控制開始產生「顯著落後感」的 KPI 指標範圍（例如：封包丟失 > 15% 時）。
  3. 規格書撰寫：在 README 中正式彙編性能規格：包含最大控制距離（推估）、最優控制頻率其當下耗電狀況。
  4. 數據 Push：將實驗日誌與規格書報告推送至 GitHub Classroom。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 工程客觀性：報告中是否包含樣本數與測試變因的詳細紀錄，而非僅有單一結果？
  • [ ] 建議可行性：基於數據得出的「控制安全範圍建議」是否具備指導實際駕駛的價值？

任務 3：「數據驅動優化戰」 (Data-driven Optimization Contest)

角色扮演：性能優化大師 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 優化方案實作：針對一個特定瓶頸（如：JSON 解析慢）實作「精簡字串協議」。
  2. 對比測量：在相同物理條件下對比優化前後的 RTT。計算效能提升百分比 \(\Delta \% = \frac{T_{old} - T_{new}}{T_{old}} \times 100\)。
  3. 分析總結：在 PR 中詳細說明：此優化是否對 PID 穩定性產生了正面貢獻。
  4. 提交 PR：發起變更請求，展現數據驅動改進的具體成果。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 優化成效顯著性：報告中的優化數據是否能在測試中被複現？提升是否超過 10% 的測量誤差？
  • [ ] 工程權衡思考：是否能分析此優化是否增加了代碼複雜度或對其他資源（如快閃記憶體）造成了額外負擔？

GitHub Classroom 詳細操作指南

本單元教學重點：將「軟體開發」引導向「性能工程」。讓學員學會為自己的系統承諾性能指標。

  1. 重點 1：數據驅動 (Data-Driven) - 引導學員排除「感覺比較快」這種模糊描述，強制要求使用 Latency、FPS 與 Packet Loss 等標準術語。
  2. 重點 2：資源權衡 (Trade-off) - 討論方案 A vs 方案 B 時，建立「魚與熊掌」的思考框架：追求速度是否犧牲了穩定度？
  3. 重點 3：SLA 的重要性 - 讓學員體悟到工業產品必須具備「基準規格」。規格書就是你對使用者的技術承諾。
