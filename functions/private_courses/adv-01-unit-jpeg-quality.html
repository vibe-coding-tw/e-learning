🎨 1. JPEG 壓縮：丟棄人類看不見的資訊 (20 min)

有損壓縮 (Lossy Compression)：JPEG 的魔術在於它會丟棄人眼不敏感的高頻色彩資訊。這就是為什麼 JPEG 檔案重新儲存多次後會變得模糊。

量化 (Quantization)：這是控制品質的核心步驟。高品質（如 90）保留更多細節，檔案較大；低品質（如 10）壓縮率高，但會產生明顯的 8x8「方塊效應」。

⚖️ 2. 權衡：畫質 vs. 延遲 (25 min)

大小與速度的連鎖反應

高品質影像 $\rightarrow$ 檔案大 $\rightarrow$ 傳輸時間長 $\rightarrow$ 控制延遲增加。在即時遙控車的應用中，高延遲是致命的。

計算代價

在 ESP32 上，極高畫質的壓縮不僅消耗傳輸頻寬，也需要更多的運算時間與記憶體緩衝區來進行 JPEG 編碼。

🧠 預測思考

假設你的無線網路頻寬固定（例如 10 Mbps），如果你將 Quality 從 20 提高到 80，請預測你的 FPS（每秒幀數） 會發生什麼變化？

🤖 3. 視覺演算法的敏感度 (15 min)

並非所有演算法都需要最高畫質。選擇適當的 Q 值 (Quality) 可以節省大量頻寬而不影響功能。

🔑 關鍵概念：壓縮比 vs 頻寬

  • 顏色辨識 (Color Tracking)：通常對雜訊較不敏感。即使又糊又馬賽克，紅色還是紅色。
  • 形狀/邊緣辨識 (Shape/Edge Detection)：對雜訊非常敏感。方塊效應會創造出不存在的「假邊緣」，干擾判斷。
  • Q 值的物理意義：透過捨棄高頻資訊來換取傳輸速度。

Assignment: 影像品質參數的權衡 (adv-01-unit-jpeg-quality)

本單元探討在嵌入式視覺系統中，影像品質（JPEG Quality）、通訊頻寬與演算法穩定性之間的權衡關係。你將使用 ESP32-CAM 進行實驗，並透過數據分析找出最佳的系統參數。

任務目標

  1. 量化影響：了解 JPEG 品質參數如何影響影像檔案大小與傳輸幀率 (FPS)。
  2. 穩定性測試：觀察低品質影像對視覺演算法（如質心計算）產生的雜訊干擾。
  3. 動態策略：實作一個能根據場景需求（如運動狀態）自動切換品質的控制邏輯。

專案結構

  • 📂 firmware/: ESP32-CAM 韌體原始碼。
  • 📂 web/: 網頁端視覺處理邏輯 (vision-logic.js)。
  • 📂 analysis/: 實驗數據存放處。
    • results.csv: 記錄品質、大小與 FPS 的標定數據。
    • jitter-log.json: 記錄質心座標的抖動數據。

作業任務

任務 1：品質、大小與 FPS 關聯標定 (Scaling Lab)

  • 操作：手動調整 JPEG 品質參數（從 10 到 90），記錄對應的影像檔案大小與平均 FPS。
  • 繳交：將測量結果填入 analysis/results.csv。
  • 思考：在 README 中說明你觀察到的「邊際效益遞減」點在哪裡？

任務 2：演算法抗噪測試 (Centroid Jitter Lab)

  • 操作：在不同品質下執行質心偵測演算法。
  • 繳交：
    • 記錄 100 幀內的質心座標，存入 analysis/jitter-log.json。
    • 在 README 中上傳兩張對比圖：Q=90 與 Q=10 的質心標記圖，觀察低品質下的座標跳動。

任務 3：動態品質切換策略 (Adaptive Strategy)

  • 實作：在 web/vision-logic.js 中實作品質切換邏輯。
    • 策略：靜止時使用高品質 (Q=80) 以獲取細節；移動時（motorSpeed > 0）使用低品質 (Q=20) 以換取流暢度。
  • 挑戰：加入「抖動保護 (Debounce)」機制，避免品質切換過於頻繁。

繳交方式

  1. 完成程式碼實作與數據採集。
  2. 將實驗數據與對比圖更新至此 程式庫。
  3. git commit 並 git push 回 GitHub Classroom。

提示：沒有最好的參數，只有最適合場景的權衡。祝實驗順利！

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template 程式庫) 配置建議

進階單元的範本應更具「工具化」，建議包含：

  • 📂 firmware/：提供 ESP32-CAM 韌體，需包含一個可透過 REST API 動態修改 config.jpeg_quality 的介面。
  • 📂 analysis/results.csv：預置一個效能對照表，讓學員填入不同 Q 值下的 $FPS$ 與 $KB/Frame$ 數據。
  • 📂 web/vision-core.js：核心視覺處理腳本，將質心計算 (Centroid) 與品質切換邏輯留給學員實作。
  • 📂 .github/workflows/grading.yml：設置自動化檢查，驗證學員是否有按時回傳實驗數據。

2. 作業任務部署細節

任務 1：品質、大小與 FPS 關聯標定 (Scaling Lab)

  • 目標：量化影像品質對通訊頻寬的具體影響，尋找傳輸「拐點」。
  • Classroom 部署建議：
    • 自動化評分 (Autograding)：檢查 analysis/results.csv 是否已被正確填充且符合「品質上升 -> 大小上升 -> FPS 下降」的物理規律。
    • 挑戰項：要求學員在 README 中標出「邊際效益遞減」的 Q 值點（通常在 80 以上檔案激增但畫質提升有限）。

任務 2：演算法抗噪測試 (Centroid Jitter Lab)

  • 目標：觀察 JPEG 「方塊效應」對影像邊緣偵測的干擾。
  • Classroom 部署建議：
    • 驗證方式：學員需在 README 中上傳兩張對比圖：一張是 Q=90 時穩定的質心標記，另一張是 Q=10 時質心因雜訊產生的「座標跳動 (Jitter)」。
    • 數據提交：要求學員將 100 幀內的質心 X 座標偏差值存入 jitter-log.json 並上傳。導師核閱其偏差值是否在低畫質下顯著增加。

任務 3：動態品質切換策略 (Adaptive Strategy)

  • 目標：開發「情境感知」的畫質切換邏輯。
  • Classroom 部署建議：
    • 邏輯檢核： 
      // 核心邏輯：靜態細節優先，動態流暢優先
if (motorSpeed === 0) setQuality(80); 
else setQuality(20);
    • 驗證方式：導師在 PR 中核對此條件判斷是否正確實作，並確認學員是否考慮了切換時的「抖動保護 (Debounce)」，防止畫質頻繁閃爍。

3. 進階單元導師點評標準 (Advanced Benchmarks)

此階段評分應側重於 「工程數據的詮釋能力」：

  • [ ] 數據敏感度：學員是否能解釋為什麼 10 Mbps 的網路在 Q=90 時會導致影像嚴重延遲？
  • [ ] 雜訊處理思維：面對 Q=10 的馬賽克，學員是否提出使用「模糊濾波 (Blur)」來減少假邊緣？
  • [ ] 系統整合度：動態切換策略是否能與馬達控制信號無縫同步？
