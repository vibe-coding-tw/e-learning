簡介

在即時影像系統中，JPEG Quality 參數不只是「畫質好不好看」的問題——它直接牽動傳輸延遲、頻寬占用與視覺演算法的準確度。 把品質調太高，FPS 會掉到無法操控的程度；調太低，邊緣偵測演算法會被壓縮雜訊搞得判斷失準。 本模組將教你用數據量化這些權衡，並實作一套能根據場景自動切換品質的動態策略。

情境
你的遙控車專案終於能透過網頁看到即時影像了，但你發現一個惱人的問題：把 JPEG 品質設成 90 時畫面超清晰，車子卻因為延遲太高而頻繁撞牆；設成 10 時反應很靈敏，但車道線辨識演算法開始瘋狂誤判。到底該設多少？有沒有辦法讓系統自己判斷何時需要高畫質、何時該犧牲畫質換取速度？這就是本模組要解決的核心工程問題。

JPEG 壓縮：丟棄人類看不見的資訊

JPEG 是目前世界上使用最廣泛的影像壓縮格式，而它之所以能將圖片大小壓縮到原始資料的 1/10 甚至 1/50，核心秘密在於一個大膽的策略——有損壓縮 (Lossy Compression)。它會有選擇性地丟棄人類視覺系統不敏感的高頻色彩資訊，在「幾乎看不出差異」的前提下大幅縮小檔案大小。

JPEG 壓縮的四個步驟

理解 JPEG 的壓縮流程，能幫助你理解為什麼 Quality 參數會影響影像的方塊效應與檔案大小。JPEG 壓縮包含四個依序執行的步驟：

  1. 色彩空間轉換：將 RGB 像素轉換為 YCbCr 格式（Y = 亮度、Cb/Cr = 色度）。人眼對亮度變化遠比色彩變化敏感，因此色度分量可以被降採樣 (Chroma Subsampling) 至一半甚至四分之一的解析度。
  2. 離散餘弦變換 (DCT)：將影像切成 8×8 像素的小區塊，對每個區塊進行 DCT 變換，將空間域的像素值轉換為頻率域的係數。低頻係數代表區塊的平均色調，高頻係數代表細節與邊緣。
  3. 量化 (Quantization)：這是壓縮的核心步驟，也是 Quality 參數直接控制的環節。量化器會將 DCT 係數除以一個「量化矩陣」中的對應值，然後取整數。量化矩陣的值越大（Quality 越低），越多高頻係數會被歸零——資訊就此永久丟失。
  4. 熵編碼 (Entropy Coding)：對量化後的係數進行霍夫曼編碼 (Huffman Coding) 或算術編碼，進一步壓縮資料量。這一步是無損的。
注意 JPEG 的 Quality 參數（通常為 0～100 或 0～63）並不是線性的壓縮率控制。Quality 從 90 降到 80 可能只減少 20% 的檔案大小，但從 20 降到 10 卻可能造成畫質的劇烈崩壞。這種「邊際效益遞減」的特性，是後續實驗中你需要親手量化的重點。

量化矩陣：Quality 參數的幕後推手

每個 8×8 DCT 區塊經過變換後會產生 64 個頻率係數。量化矩陣也是一個 8×8 的表格，定義了每個頻率位置的「除數」。當 Quality 設得越低，量化矩陣中的除數就越大，意味著更多的 DCT 係數在除法後變成 0——這些歸零的係數就是被「丟棄」的資訊。

Quality 值 量化效果 檔案大小（VGA） 視覺品質
90 ～ 100 幾乎不量化，保留大量高頻 80 ～ 150 KB 肉眼無法區分原圖
60 ～ 80 適度量化，丟棄部分高頻 30 ～ 60 KB 略有模糊，可接受
30 ～ 50 較強量化，邊緣開始模糊 15 ～ 30 KB 可見方塊效應
10 ～ 20 激烈量化，大量高頻歸零 5 ～ 15 KB 嚴重方塊效應、色彩失真

8×8 方塊效應 (Block Artifact)

由於 JPEG 是以 8×8 像素為單位進行獨立壓縮的，當量化過於激烈時，相鄰區塊之間的色彩差異會變得明顯，形成肉眼可見的格狀邊界——這就是俗稱的「方塊效應」或「馬賽克」。這種人工雜訊對人類觀看來說只是「畫質差」，但對機器視覺演算法來說，它會產生大量不存在的假邊緣 (False Edges)，嚴重干擾邊緣偵測、輪廓追蹤與物件辨識。

// ESP-IDF 中設定 JPEG 品質的方式
camera_config_t config;
config.jpeg_quality = 12; // 範圍 0-63，數字越低品質越高

// 注意：ESP-IDF 的 jpeg_quality 與一般工具的 Q 值方向相反！
// jpeg_quality = 10 → 高品質 (約等於一般工具的 Q=85)
// jpeg_quality = 63 → 最低品質 (約等於一般工具的 Q=5)
警告 ESP-IDF 攝影機驅動中的 jpeg_quality 參數範圍是 0～63，且數值越低代表品質越高（與大多數影像編輯軟體的慣例相反）。這是一個極易混淆的陷阱。設成 10 是高品質，設成 50 反而是低品質。務必在程式碼中加上明確的註解以避免混淆。
提示 JPEG 壓縮是不可逆的——每次重新編碼都會進一步損失品質。這就是為什麼你在影像編輯軟體中對同一張 JPEG 反覆「開啟→儲存」後，畫面會逐漸變得模糊與失真。在 ESP32 的串流場景中，攝影機硬體直接輸出 JPEG 格式，所以每幀只經歷一次壓縮，不存在「多代損失」的問題。

權衡：畫質 vs. 延遲

在即時影像串流系統中，JPEG Quality 參數不只是「畫面好不好看」的美學選擇——它是一個涉及 畫質、檔案大小、傳輸延遲與 FPS（每秒幀數） 四者連鎖反應的工程決策。提高品質會讓畫面更清晰，但也會讓每一幀的檔案變大，導致傳輸時間拉長、FPS 下降，最終使遙控車的操控延遲變得無法接受。

連鎖反應模型

讓我們用一個具體的數字推演來理解這個連鎖反應。假設你的 ESP32-CAM 連接在一個 Wi-Fi 網路上，可用頻寬約為 10 Mbps（即 1.25 MB/s）：

jpeg_quality 約等於 Q 值 平均幀大小 (VGA) 理論最大 FPS 單幀傳輸延遲
10 ~85 ~80 KB ~15 FPS ~64 ms
20 ~60 ~35 KB ~35 FPS ~28 ms
40 ~35 ~15 KB ~83 FPS* ~12 ms
63 ~5 ~5 KB ~250 FPS* ~4 ms

* 實際 FPS 會受到攝影機擷取速度、JPEG 編碼時間與 TCP 協議開銷的限制，通常不會超過 30 FPS。但這個計算清楚展示了頻寬如何成為瓶頸。

注意 理論最大 FPS = 可用頻寬 ÷ 平均幀大小。例如 1.25 MB/s ÷ 80 KB ≈ 15 FPS。但在實際環境中，Wi-Fi 的有效吞吐量通常只有標稱值的 30%～60%（受干擾、協議開銷影響），因此實際 FPS 會更低。

編碼時間的隱藏成本

除了頻寬瓶頸之外，JPEG 編碼本身也需要 CPU 時間。在 ESP32-S3 上，壓縮一張 VGA 解析度的影像大約需要 15～30 ms（取決於品質設定與是否使用硬體 JPEG 編碼器）。這意味著即使頻寬無限大，單靠編碼延遲就已經將 FPS 上限壓到約 30～60 FPS。

更高的品質設定會讓量化步驟產生更多的非零 DCT 係數，進而增加熵編碼的工作量與輸出資料量。在某些極端情況下（如 jpeg_quality = 5），編碼時間可能比 jpeg_quality = 30 多出 50%。

端到端延遲的組成

對於遙控車這類「看到→反應→操控」的應用，延遲 (Latency) 是比 FPS 更關鍵的指標。完整的端到端延遲由以下部分組成：

延遲來源 典型值 受 Quality 影響？
攝影機曝光與讀出 10 ～ 33 ms
JPEG 編碼 15 ～ 30 ms 品質越高越慢
Wi-Fi 傳輸 4 ～ 80 ms 檔案越大越慢
瀏覽器解碼與渲染 2 ～ 5 ms
總計 31 ～ 148 ms
警告 人類的視覺反應時間約為 200 ms。如果端到端延遲超過 150 ms，操控者會明顯感受到「遙控車不聽話」的現象，因為車子在他做出操控動作之前的 150 ms 內已經移動了相當距離。對於高速移動的場景，延遲應盡量控制在 100 ms 以內。
提示 在實際測試中，你可以透過在每幀影像上疊加時間戳 (Timestamp) 的方式來精確測量端到端延遲。在 ESP32 端用 millis() 記錄擷取時間，在瀏覽器端用 JavaScript 記錄接收時間，兩者的差值即為近似的端到端延遲（需確保 ESP32 與電腦的時鐘已同步）。

視覺演算法的敏感度

並非所有電腦視覺演算法都需要最高畫質的影像。事實上，根據演算法的類型不同，JPEG 壓縮雜訊對其準確度的影響可以從「幾乎無感」到「完全失效」。理解不同演算法對壓縮品質的敏感度差異，是設計「動態品質切換策略」的理論基礎。

演算法敏感度分類

我們可以根據演算法依賴的影像特徵，將常見的視覺任務分為三個敏感度等級：

敏感度 演算法類型 依賴的影像特徵 建議最低 Q 值
低敏感 顏色追蹤 (Color Tracking) 像素的色相 (Hue) 值 Q ≈ 20（jpeg_quality ≈ 40）
低敏感 動態偵測 (Motion Detection) 前後幀像素差異的統計量 Q ≈ 15（jpeg_quality ≈ 45）
中敏感 質心追蹤 (Centroid Tracking) 目標區域的幾何中心 Q ≈ 40（jpeg_quality ≈ 25）
高敏感 邊緣偵測 (Canny / Sobel) 像素梯度的方向與強度 Q ≈ 60（jpeg_quality ≈ 15）
高敏感 特徵點匹配 (ORB / SIFT) 角點與紋理細節 Q ≈ 70（jpeg_quality ≈ 12）

為什麼顏色追蹤不怕壓縮？

顏色追蹤演算法的核心邏輯是：將影像從 RGB 轉換到 HSV 色彩空間，然後根據色相 (Hue) 值的閾值來判斷目標。即使 JPEG 壓縮造成了嚴重的方塊效應，每個 8×8 區塊內的平均色相仍然保持大致正確。一塊紅色區域即使變得模糊、出現馬賽克，它在 HSV 空間中仍然落在「紅色」的色相範圍內。

為什麼邊緣偵測極度敏感？

邊緣偵測演算法（如 Canny、Sobel）是透過計算相鄰像素之間的梯度來找出影像中的邊界。問題在於：JPEG 的方塊效應會在每個 8×8 區塊的邊界處製造出突然的亮度跳變，而邊緣偵測器會忠實地將這些跳變識別為「邊緣」——即使原始場景中根本不存在這些邊界。

// 質心追蹤的「抖動」問題示意（JavaScript 端）
function calculateCentroid(binaryMask) {
let sumX = 0, sumY = 0, count = 0;
for (let y = 0; y < height; y++) {
for (let x = 0; x < width; x++) {
if (binaryMask[y * width + x]) {
sumX += x; sumY += y; count++;
}
}
}
// 低品質影像中，方塊效應導致 binaryMask 的邊緣
// 每幀都不同，使得 centroidX/Y 產生「抖動」
return { x: sumX / count, y: sumY / count };
}
警告 在低品質影像中，即使目標物體完全靜止，質心座標也可能每幀都在跳動。這是因為方塊效應使得二值化遮罩 (Binary Mask) 的邊緣像素每幀都略有不同。如果你的控制邏輯直接使用原始質心座標來驅動馬達，車子會出現不可預測的抖動 (Jitter)。

動態品質切換策略

既然不同場景對畫質的需求不同，最佳策略不是固定一個 Quality 值，而是根據當前的系統狀態動態調整。以遙控車為例：

  • 車輛靜止時：使用高品質（jpeg_quality = 10～15），因為此時不需要低延遲，但可能需要高品質影像來執行精密的視覺辨識（如讀取路標文字）。
  • 車輛移動時：使用低品質（jpeg_quality = 30～40），優先確保低延遲與高 FPS，讓操控者能即時反應。此時畫面的運動模糊 (Motion Blur) 本身就會遮蓋壓縮雜訊。
  • 品質切換時加入 Debounce：避免在「剛好停下來又馬上啟動」的邊界情境中頻繁切換品質，造成畫面閃爍。
// 動態品質切換的核心邏輯 (JavaScript 端)
let lastSwitchTime = 0;
const DEBOUNCE_MS = 2000; // 切換後至少等 2 秒才能再切

function updateQuality(motorSpeed) {
const now = Date.now();
if (now - lastSwitchTime < DEBOUNCE_MS) return;

const newQ = (motorSpeed === 0) ? 10 : 35;
fetch(`/control?quality=${newQ}`);
lastSwitchTime = now;
}
提示 在作業中，你將需要使用 URL.createObjectURL(blob) 來在網頁上即時顯示從 ESP32 接收到的 JPEG Blob 影像。這個 API 會為記憶體中的二進位資料建立一個臨時 URL，可直接賦值給 <img> 標籤的 src 屬性。記得在每次更新前呼叫 URL.revokeObjectURL() 釋放前一張圖片的記憶體，否則會造成瀏覽器端的記憶體洩漏。

Assignment: 影像品質參數的權衡 (adv-01-unit-jpeg-quality)

本單元探討在嵌入式視覺系統中，影像品質（JPEG Quality）、通訊頻寬與演算法穩定性之間的權衡關係。你將使用 ESP32-CAM 進行實驗，並透過數據分析找出最佳的系統參數。

任務目標

  1. 量化影響：了解 JPEG 品質參數如何影響影像檔案大小與傳輸幀率 (FPS)。
  2. 穩定性測試：觀察低品質影像對視覺演算法（如質心計算）產生的雜訊干擾。
  3. 動態策略：實作一個能根據場景需求（如運動狀態）自動切換品質的控制邏輯。

專案結構

  • 📂 firmware/: ESP32-CAM 韌體原始碼。
  • 📂 web/: 網頁端視覺處理邏輯 (vision-logic.js)。
  • 📂 analysis/: 實驗數據存放處。
    • results.csv: 記錄品質、大小與 FPS 的標定數據。
    • jitter-log.json: 記錄質心座標的抖動數據。

作業任務

任務 1：品質、大小與 FPS 關聯標定 (Scaling Lab)

  • 操作：手動調整 JPEG 品質參數（從 10 到 90），記錄對應的影像檔案大小與平均 FPS。
  • 繳交：將測量結果填入 analysis/results.csv。
  • 思考：在 README 中說明你觀察到的「邊際效益遞減」點在哪裡？

任務 2：演算法抗噪測試 (Centroid Jitter Lab)

  • 操作：在不同品質下執行質心偵測演算法。
  • 繳交：
    • 記錄 100 幀內的質心座標，存入 analysis/jitter-log.json。
    • 在 README 中上傳兩張對比圖：Q=90 與 Q=10 的質心標記圖，觀察低品質下的座標跳動。

任務 3：動態品質切換策略 (Adaptive Strategy)

  • 實作：在 web/vision-logic.js 中實作品質切換邏輯。
    • 策略：靜止時使用高品質 (Q=80) 以獲取細節；移動時（motorSpeed > 0）使用低品質 (Q=20) 以換取流暢度。
  • 挑戰：加入「抖動保護 (Debounce)」機制，避免品質切換過於頻繁。
