ESP32 記憶體分區表

晶片藍圖：智慧裝置如何保持井然有序。規劃 Flash 空間，突破硬體預設限制。

🗺️ 為什麼需要分區表？

ESP32 的 4/16MB Flash 不是一個空箱，而是被精細劃分的「數位地產」。

數位地段規劃

Flash 就是一個大型倉儲。分區表清楚寫明：每個功能數據從哪個「偏移地址 (Offset)」開始，租用多長空間。

App 0 / App 1

存放程式碼。OTA 更新需要兩個對分空間。

NVS (Data)

存放 Wi-Fi 設定、參數等長度固定的數據。

Partition Types

otadata OTA 引導標記 (8KB)
nvs 非揮發性存儲 (16KB+)
spiffs 文件系統：存圖檔

🛠️ 預設 vs. 自定義分區

引入大型庫時，常遇 `Region overflowed`。這時 1MB 空間已不足，需重新規劃。

突破限制：partitions.csv

透過 CSV 文件，可縮減文件系統，將空間騰給 App，實現最大單體燒錄。

partitions.csv format
# Name, Type, SubType, Offset, Size, Flags
nvs, data, nvs, , 0x4000,
otadata, data, ota, , 0x2000,
factory, app, factory, , 2M,
storage, data, spiffs, , 1.5M,

課程單元 基礎 2 - 儲存架構

基礎 02：ESP32 儲存架構與分區規劃 (Partition Table & Persistence)

歡迎來到本單元！在本實驗中，你將學習如何從「黑盒開發」轉向「系統架構規劃」，學會精細管理 ESP32 的 Flash 資源。你將挑戰重新規劃分區表、實作 NVS 數據持久化，以及部署 LittleFS 檔案系統。

🎯 學習目標

  1. 分區規劃：學會使用 partitions.csv 重新分配 Flash 空間。
  2. 數據持久化：使用 NVS (Non-volatile Storage) 儲存裝置狀態。
  3. 檔案系統：實作 LittleFS 掛載並從 Flash 讀取網頁資源。

🚀 作業任務

任務 1：「擴容手術」自定義分區規劃

你的任務是修改 partitions.csv，打破預設限制，為 App 與 檔案系統騰出空間：

  • 目標 1：將 factory 分區（App 空間）擴大至 2MB。
  • 目標 2：將 storage 分區（LittleFS）規劃為 1.5MB。
  • 注意：必須確保所有分區地址符合 4KB (0x1000) 對齊，且各分區不可重疊。
  • 配置：在 platformio.ini 中手動添加 board_build.partitions = partitions.csv。

任務 2：「數據黑盒子」NVS 參數持久化

實作一個能「記住開機次數」的功能：

  • 使用 Preferences 函式庫。
  • 每次重新開機時，讀取並增加計數值，然後存回 NVS。
  • 驗證：拔掉電源 10 秒後重新通電，確認計數值是否能正確累加。

任務 3：「資源檔案庫」LittleFS 檔案部署

將 data/ 目錄下的 index.html 部署到 ESP32：

  1. 執行 PlatformIO 的 Upload Filesystem Image 任務。
  2. 在程式碼中掛載 LittleFS。
  3. 實作 Web Server 路由，當訪問 / 時，讀取並回傳 index.html 的內容。

📝 提交要求

  1. 程式碼：完成 src/main.cpp 中的邏輯。
  2. 分區表：提交修正後的 partitions.csv。
  3. 報告：在 docs/MemoryMap.md 中填寫你最終規劃的各分區 Offset 與 Size。
  4. 驗證：在 README.md (或提交訊息) 中說明你如何驗證 NVS 的持久化功能。

📁 程式庫結構

├── platformio.ini       # PlatformIO 設定檔
├── partitions.csv       # 分區表定義
├── src/main.cpp         # 程式碼實作區
├── data/                # 靜態網頁資源 (待上傳至 LittleFS)
├── docs/MemoryMap.md    # 分區地址核對報表
└── README.md            # 本作業說明

1. 範本程式庫 (Template Repo) 配置建議

分區作業的核心是「地址計算」與「資源隔離」，範本應提供一個「需要重新規劃」的儲存環境，建議包含：

  • 📂 partitions.csv：核心挑戰對象。提供一個遺漏地址或空間衝突的 CSV 範本，學員需依照規定的 2MB App 空間與 1.5MB 檔案空間進行「地址對齊」修正。
  • 📂 platformio.ini：引導學員手動添加 board_build.partitions = partitions.csv，使自定義配置生效。
  • 📂 data/index.html：資源庫。存放一個簡單的網頁檔，學員需學會如何將此目錄「打包上傳」至 LittleFS 分區。
  • 📂 src/main.cpp：存儲實作區。預設空的 NVS 與 LittleFS 掛載框架，學員需寫出能「記住開機次數」與「從 Flash 讀取網頁」的代碼。
  • 📂 docs/MEMORY_MAP.md：存儲地圖。學員需在此填寫最終規劃的各分區 Offset (起始地址) 與 Size，建立對 4KB 對齊的物理直覺。

2. 作業任務部署細節

任務 1：「擴容手術」自定義分區規劃 (Flash Expansion Surgery Lab)

  • 目標：打破 1MB 的預設限制，為未來複雜的 AI 或 Web 專案騰出空間。
  • Classroom 部署建議：
    • 規劃挑戰：學員需在 CSV 中將 factory 分區擴大至 2MB。
    • 地址計算：正確計算下一個分區（如 nvs 或 storage）的 Offset，防止空間重疊 (Overlap)盤。
    • 驗證要點：提交燒錄時的終端機截圖。證明 Firmware 已成功依照自定義分區表進行部署。

任務 2：「數據黑盒子」NVS 參數持久化 (NVS Persistence Lab)

  • 目標：掌握 Key-Value 存儲，讓裝置在斷電後仍能像擁有「長期記憶」般記住設定。
  • Classroom 部署建議：
    • 持久化實作：使用 Preferences 函式庫，實作一個能自動記錄「馬達最後位置」或「開機累計次數」的功能。
    • 邏輯強健性：學員需處理「初次使用」時 NVS 為空的預設值返回邏輯。
    • 驗證要點：在 README 中說明：拔掉電源 10 秒後重新通電，數據是否如預期被正確還原？

任務 3：「資源檔案庫」LittleFS 檔案部署 (LittleFS Asset Deployment Lab)

  • 目標：實作「動靜分離」。將 UI 與程式碼隔離，建立專業的嵌入式 Web 開發結構。
  • Classroom 部署建議：
    • 檔案上傳實戰：學員需執行 PlatformIO 的 Upload Filesystem Image 任務，將 data/ 目錄寫入 Flash。
    • 路徑映射：在代碼中掛載 LittleFS，並寫出一個簡單的 Web 路由，讓瀏覽器請求 /index 時，能直接從 Flash 返回檔案內容。
    • 提交結果：展示瀏覽器成功渲染出存儲於 LittleFS 中 HTML 頁面的畫面。

3. 硬體系統架構師點評標準 (Architecture & Storage Standards)

此單元的價值在於 「對儲存空間與數據生命週期的精細化管理」：

  • [ ] 地址對齊 (Alignment)：分區地址是否皆為 4KB (0x1000) 的倍數？（這是 ESP32 的硬體底層要求）。
  • [ ] 資源隔離思維：是否理解 SPIFFS/LittleFS 與 NVS 應用場景的差異？（大檔案用 FS，小數據用 NVS）。
  • [ ] 部署穩定性：是否成功在 platformio.ini 中正確宣告分區表路徑？
