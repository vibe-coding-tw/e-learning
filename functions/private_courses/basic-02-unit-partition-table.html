ESP32 記憶體分區表

晶片藍圖：智慧裝置如何保持井然有序。規劃 Flash 空間，突破硬體預設限制。

🗺️ 為什麼需要分區表？

ESP32 的 4/16MB Flash 不是一個空箱，而是被精細劃分的「數位地產」。

數位地段規劃

Flash 就是一個大型倉儲。分區表清楚寫明：每個功能數據從哪個「偏移地址 (Offset)」開始，租用多長空間。

App 0 / App 1

存放程式碼。OTA 更新需要兩個對分空間。

NVS (Data)

存放 Wi-Fi 設定、參數等長度固定的數據。

Partition Types

otadata OTA 引導標記 (8KB)
nvs 非揮發性存儲 (16KB+)
spiffs 文件系統：存圖檔

🛠️ 預設 vs. 自定義分區

引入大型庫時，常遇 `Region overflowed`。這時 1MB 空間已不足，需重新規劃。

突破限制：partitions.csv

透過 CSV 文件，可縮減文件系統，將空間騰給 App，實現最大單體燒錄。

partitions.csv format
# Name, Type, SubType, Offset, Size, Flags
nvs, data, nvs, , 0x4000,
otadata, data, ota, , 0x2000,
factory, app, factory, , 2M,
storage, data, spiffs, , 1.5M,

Flash 映射與永續存儲實驗室：分區規劃、NVS 與 LittleFS 實戰

任務 1：「擴容手術」自定義 partitions.csv (Flash Expansion Surgery)

角色扮演：韌體架構師 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫教 Clone 到本地。
  2. 分區規劃實做：建立 partitions.csv，擴大 factory 並對齊地址。
  3. 配置連結：在 platformio.ini 中使用 board_build.partitions。
  4. 驗證繳交：在 README.md 貼上 Flash 分區映射截圖。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 語法準確性：CSV 語法是否符合規範且地址無衝突？
  • [ ] 空間邊界分配：是否理解 4KB 對齊。規劃出儲存邊界？

任務 2：「數據黑盒子」NVS 參數持久化 (Data Blackbox: NVS Persistence)

角色扮演：嵌入式工程師 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 持久化開發：實作「開機計數器」與「記住檔位」功能。
  2. API 操作驗證：正確調用 begin()、putInt()、getInt()。
  3. 實機斷電測試：斷電後重新通電，記錄數據是否被正確還原。
  4. 技術分享：寫下關於 NVS 磨損均衡 (Wear Leveling) 特性說明。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 邏輯健壯性：是否處理了「初次讀取」的預設值返回？
  • [ ] 持久化成功：斷電重啟後，計數器是否連續遞增？

任務 3：「資源檔案庫」LittleFS 檔案部署 (Resource Vault: LittleFS)

角色扮演：全端 IoT 開發者 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 資源製作：將 HTML/CSS 放入 data 目錄。
  2. 鏡像打包：執行 Upload Filesystem Image 映射至 storage。
  3. 掛載存取：實作 LittleFS.begin()，在 Web Server 中動態讀取返回。
  4. 最終 Commit：整理專案模組並 Push 提交。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 系統掛載完整度：日誌是否正确印出 LittleFS 容量訊息？
  • [ ] 路徑響應正確性：能否成功顯示 Flash 中的靜態 HTML？

GitHub Classroom 詳細操作指南

本單元教學重點：掌握儲存的「實體層級」管理，消除對燒錄的恐懼。掌握持久化核心邏輯。

  1. 重點 1：土地重劃 - 將 Flash 變成可量化的地址區塊，具備規劃大專案能力。

  3. 重點 3：解耦設計 - 透過 LittleFS 示範 UI 與邏輯實體隔離，優化效率。
