基礎 2 - 儲存架構

ESP32 記憶體分區表

晶片的秘密藍圖：你的智慧裝置如何保持井然有序。學習如何規劃 Flash 空間，突破硬體預設限制。

🗺️ 為什麼需要分區表？ (20 min)

ESP32 的 4MB/16MB Flash 不是一個混沌的大容器，而是被精細劃分的「數位地產」。

數位地段規劃

想像 Flash 是一個大型倉儲。分區表就是一份土地權狀，清楚寫明了：每個功能的數據從哪個「偏移地址 (Offset)」開始，租用了多長的空間。

App 0 / App 1

存放執行的程式碼。OTA 更新需要兩個對應分區。

NVS (Data)

存放 Wi-Fi 設定、參數等長度固定的 Key-Value 數據。

Partition Types

otadata OTA 引導標記 (8KB)
nvs 非揮發性存儲 (16KB+)
spiffs 文件系統：存網頁/圖檔

🛠️ 預設 vs. 自定義分區 (25 min)

當你在程式中引入了大型的 AI 庫、顯示驅動時，常會遇到 `Region overflowed`。這時，預設的 1MB 程式空間已經無法負荷。

突破限制：partitions.csv

透過一個 CSV 文件，你可以自由縮減文件系統空間，將其騰挪給 App 分區，實現最大 4MB 的單體燒錄。

partitions.csv format
# Name, Type, SubType, Offset, Size, Flags
nvs, data, nvs, , 0x4000,
otadata, data, ota, , 0x2000,
factory, app, factory, , 2M,
storage, data, spiffs, , 1.5M,

📝 空間預算挑戰

假設你的 Flash 總量是 4MB (4096KB)。

若 App 佔用了 2MB，NVS 分區佔用了 32KB，Bootloader 佔用了 64KB...

🔍 思考題：
你的 LittleFS 分區最大能規劃到多少 KB？如果還要預留給 OTA 備援分區，這空間還夠嗎？

* Hint: All offsets must be aligned to 4KB blocks.

💾 持久化存儲的決擇 (15 min)

🔑

NVS (Non-Volatile Storage)

專為 「小而碎」 的參數設計。Key-Value 結構，適合儲存 PID 常數、使用者暱稱或最後一次的操作記錄。

Preferences.h 4KB Aligned

LittleFS / SPIFFS

正式的文件系統。適合存放 「大而靜態」 的資源：如 HTML 遙控頁面、CSS 樣式表或預覽 JPG 圖檔。

LittleFS.h Dir Tree Structure

任務 1：「擴容手術」自定義 partitions.csv (Flash Expansion Surgery)

角色扮演：韌體架構師 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 分區規劃實做：在專案根目錄建立 partitions.csv，重新劃分 Flash 空間，將 factory 分區擴大至 2MB 並相應對齊 Offset 地址。
  3. 配置連結：在 platformio.ini 中使用 board_build.partitions 指向該 CSV 檔案。
  4. 驗證繳交：成功燒錄後，在 README.md 貼上 PlatformIO 終端機顯示的 Flash 分區映射截圖。

  • [ ] 空間邊界分配：是否理解 4KB 對齊原則，並成功規劃出滿足專案需求的儲存邊界？

任務 2：「數據黑盒子」NVS 參數持久化 (Data Blackbox: NVS Persistence)

角色扮演：嵌入式工程師 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 持久化開發：實作一個使用 Preferences 庫的「開機計數器」以及一個用來記住「上次操作檔位」的功能。
  2. API 操作驗證：正確調用 begin()、putInt()、getInt() 並確保在適當時機關閉 Preferences 會話。
  3. 實機斷電測試：將開發板完全斷電後重新通電，記錄數據是否被正確還原。
  4. 技術分享：在 README 中寫一段關於 NVS 為何具有「磨損均衡 (Wear Leveling)」特性的簡短說明。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 邏輯健壯性：代碼是否處理了「初次讀取（Key 不存在）」時的預設值返回邏輯？
  • [ ] 持久化成功：斷電重啟後，計數器是否能連續遞增而非從 1 開始？

任務 3：「資源檔案庫」LittleFS 檔案部署 (Resource Vault: LittleFS)

角色扮演：全端 IoT 開發者 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 靜態資源製作：將遙控介面 HTML 與 CSS 放入專案的 data 目錄下。
  2. 鏡像打包與燒錄：執行 PlatformIO 的 Upload Filesystem Image 任務將 data 目錄映射至 storage 分區。
  3. 掛載存取開發：實作 LittleFS.begin() 初始化，並在 Web Server 中動態讀取此分區內的網頁內容返回給瀏覽器。
  4. 最終 Commit：整理包含 partitions.csv、data 目錄與完整存取代碼的 Repo 並 Push 提交。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 系統掛載完整度：日誌是否正確印出 LittleFS 的總容量與已使用空間百分比？
  • [ ] 路徑響應正確性：是否能透過 Web 路由成功顯示存放在 Flash 中的靜態 HTML？

GitHub Classroom 詳細操作指南

本單元教學重點：透過分區表掌握硬體儲存的「實體層級」管理，消除對 Flash 燒錄的恐懼感。

  1. 第一 Point：土地重劃 - 將 Flash 從黑盒子變成可量化的地址區塊，讓學生具備規劃大型專案的能力。
  2. 第二 Point：持久化思維 - 從 `volatile`（揮發）轉向 `persistent`（持久），是開發真實世界 IoT 產品的必經之路。
  3. 第三 Point：解耦設計 - 透過 LittleFS，示範如何將「UI 樣式」與「控制邏輯」在硬體層級進行實體隔離，優化更新效率。
