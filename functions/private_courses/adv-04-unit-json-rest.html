雲端數據實驗室：JSON 標準化與 RESTful API 實戰

任務 1：標準化封包產生器 (Standardized Payload Generator)

角色扮演：後端 API 開發者 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 數據建模：設計一個包含 device_info, sensors, status 的巢狀 JSON 物件模板。
  3. 接口實作：在 ESP32 上實作 GET /api/telemetry 路由，將感測器數據動態序列化後回傳。
  4. Push Final：提交具備「高層次結構清晰」且符合標準 Web 規範的數據源接口。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 語義化命名：JSON 中的 Key 是否具備高度可讀性？（例如 temp_c 而非 t1）
  • [ ] 層級合理性：物件嵌套是否逻辑一致？相關聯的數據是否被正確歸類在同一個 Parent 物件下？

任務 2：指令接收器 (POST Command Receiver)

角色扮演：系統整合工程師 | 難度：⭐️⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 路由配置：實作 POST /api/control 接口，接收來自網頁端的 JSON 格式操控指令。
  2. 解析與執行：使用 ArduinoJson 解析 cmd 與 val，並即時轉換為對應的硬體控制信號。
  3. 異常防禦：針對格式錯誤或內容損毀的 JSON Payload，實作 400 Bad Request 的錯誤攔截機制。
  4. Push Final：提交具備「高度魯棒性」且具備錯誤處理回饋的指令接收模組。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 防崩潰校驗：在接收異常 JSON 時，系統是否能穩定返回錯誤碼而非導致硬體重新啟動？
  • [ ] 通訊對話感：Response 中是否包含了執行結果的狀態碼與具體的原因描述？

任務 3：API 效能與壓力測試 (Payload Benchmark)

角色扮演：雲端效能優化工程師 | 難度：⭐️⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 封包瘦身：對比「冗長語義」與「極致精簡」兩種不同 Key 長度的封包，測量其序列化耗時與體積差異。
  2. 高頻壓力測試：模擬每秒 10 次以上的並發請求，紀錄 ESP32 的 CPU 負載與 API 回應延遲變化。
  3. 優化報告：在 README 中提出在有限頻寬情境下，對 JSON 結構設計的折衷與優化建議。
  4. Push Final：提交經過「量化分析」且兼顧性能與可維護性的 API 最終版本。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 數據權衡能力：報告中是否能清晰指出「Key 長度」對序列化速度的具體影響？
  • [ ] 架構思考度：是否考慮到了縮寫 Key 對後端資料庫索引與前端維護的易讀性損失？

GitHub Classroom 詳細操作指南

本單元教學重點：數據與接口的標準化。掌握 RESTful 設計美學與 HTTP 狀態碼的禮儀。

  1. 重點 1：資源導向路徑 (Resource-Oriented) - 強調路徑應代表「名詞」，而行為應代表「動詞」（GET/POST），打破以往用路徑寫邏輯的習慣。
  2. 重點 2：防禦性的反序列化 - 透過 3-2 任務引入校驗機制，讓學生看見在開放式網路中，不信任任何外部輸入的重要性。
  3. 重點 3：邊緣運算的計算開銷 - 透過壓力測試，讓學生實際體驗在 ESP32 這類微型裝置上，複雜的 JSON 處理對 CPU 與延遲的影響。
