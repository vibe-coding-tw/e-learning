簡介

影像串流不只有一種做法——MJPEG 和 H.264 代表著兩種截然不同的壓縮哲學，各有其適用的場景與代價。 選錯編碼方式，輕則浪費頻寬，重則導致無法接受的延遲或畫面殘影。 本模組將帶你理解兩種串流架構的核心差異，讓你在設計物聯網影像系統時能做出有根據的技術決策。

情境
你被指派為一個智慧監控專案選擇串流方案。主管要求「高畫質、低延遲、省頻寬」，但你很快發現這三者不可能同時達成。用 MJPEG 延遲極低但吃掉了大量頻寬；換成 H.264 頻寬省了十倍，卻在網路不穩時出現詭異的馬賽克殘影。你需要真正理解 I-Frame、P-Frame 的運作原理，才能向團隊解釋為什麼「沒有完美方案，只有最適合場景的權衡」。

MJPEG：簡單暴力的圖片連播

MJPEG（Motion JPEG）是最直覺的影像串流方式：將攝影機捕捉到的每一幀畫面都獨立壓縮為一張 JPEG 圖片，然後透過 HTTP 的 multipart/x-mixed-replace 協定，以「接力」的方式不斷將新圖片推送到客戶端。你可以把它想像成一本「翻頁書」——每頁都是完整的圖，快速翻動就形成了影片效果。

這種壓縮方式在技術上稱為畫面內壓縮（Intra-frame Compression），意味著每一幀的壓縮只參考自己——不會去「偷看」前一幀或後一幀的資訊。這帶來了極大的簡潔性：任何一幀都能獨立解碼，不存在幀與幀之間的依賴關係。

MJPEG 的工作流程

在 ESP32-CAM 的典型實作中，MJPEG 串流的整個生命週期可以拆解為三個階段：

1

擷取與壓縮

攝影機感測器（如 OV2640）擷取原始影像後，由晶片內建的 JPEG 硬體編碼器將畫面壓縮為 JPEG 格式。Quality 參數（10–63）決定壓縮強度。

2

封裝與傳送

壓縮後的 JPEG 資料以 HTTP multipart boundary 分隔，透過 Wi-Fi 以 TCP 連線送出。每一幀都帶有 Content-Type: image/jpeg 與 Content-Length 標頭。

3

接收與顯示

瀏覽器的 <img> 標籤原生支援 MJPEG 串流——只需將 src 指向串流 URL，瀏覽器會自動刷新顯示每一幀新圖片。

ESP32 上的 MJPEG 串流程式碼

以下是 ESP32-CAM 建立 MJPEG 串流端點的核心邏輯：

// 串流處理函式：持續推送 JPEG 幀 esp_err_t stream_handler(httpd_req_t *req) { camera_fb_t *fb = NULL; httpd_resp_set_type(req, "multipart/x-mixed-replace;boundary=frame"); while (true) { fb = esp_camera_fb_get(); // 取得一幀 JPEG size_t hlen = snprintf(buf, 128, "--frame\r\nContent-Type: image/jpeg\r\n" "Content-Length: %u\r\n\r\n", fb->len); httpd_resp_send_chunk(req, buf, hlen); httpd_resp_send_chunk(req, (char *)fb->buf, fb->len); esp_camera_fb_return(fb); } }
提示 ESP32-CAM 的 OV2640 感測器內建 JPEG 硬體編碼器，因此 MJPEG 串流幾乎不消耗 CPU 算力。這是 MJPEG 在資源受限的嵌入式設備上如此受歡迎的核心原因。

MJPEG 的優勢與限制

特性 表現 說明
編碼複雜度 極低 僅需硬體 JPEG 編碼，ESP32 即可輕鬆處理
延遲 極低 無幀間依賴，不需緩衝區累積，編碼即送出
隨機存取 完美 每幀獨立，可隨時跳轉或截圖，無需等待關鍵幀
頻寬效率 極差 不利用時間冗餘，每幀完整傳送所有像素資訊
丟包影響 輕微 最多掉一幀，下一幀立即恢復正常顯示
警告 在 VGA（640×480）解析度、Quality=20 的設定下，單幀 JPEG 約為 30–50 KB。以 15 FPS 計算，MJPEG 串流需要約 3.6–6.0 Mbps 的頻寬——這已經接近 ESP32 在 2.4 GHz Wi-Fi 下的實際傳輸上限。

丟包時的行為特徵

MJPEG 在網路不穩時的行為非常好預測：由於每一幀都是獨立的完整圖片，當網路發生壅塞時，結果是整幀丟失（掉幀）而非畫面撕裂或殘影。觀看體驗是「畫面凍結一瞬間，然後直接跳到最新畫面」，不會出現 H.264 那種方塊殘影（Artifacts）。這種「乾淨的失敗模式」是 MJPEG 在即時遙控場景中被偏好的重要原因。

反過來說，MJPEG 最大的致命傷在於頻寬浪費。以一個監控畫面為例：90% 的時間裡背景是靜止的，但 MJPEG 仍然會對每一幀完整重複編碼這些不變的像素。相較之下，H.264 的時間壓縮只需要傳送「變化的部分」，頻寬效率高出 10 倍以上。

備註 在下一節的實戰挑戰中，你將實際測量不同 Quality 設定下的單幀大小，並計算在現有頻寬下 FPS 的理論上限。理解 MJPEG 的「暴力美學」，有助於你在後續評估 H.264 時做出更有依據的比較。

H.264 (AVC)：聰明的變化預測

H.264（又稱 AVC，Advanced Video Coding）採用了完全不同於 MJPEG 的壓縮哲學：不傳送整張圖片，只傳送「跟上一幀比起來，哪裡發生了變化」。這種技術稱為幀間壓縮（Inter-frame Compression），它利用了影片中一個關鍵的物理事實——連續兩幀之間，通常有 90% 以上的像素是相同的。

透過只編碼「差異」而非「全部」，H.264 可以將頻寬消耗壓縮到 MJPEG 的 1/10 甚至更低。但這種聰明的壓縮方式也帶來了代價：更高的計算需求、更複雜的解碼流程，以及一旦有幀遺失，錯誤會沿時間軸向後擴散。

幀類型（Frame Types）

H.264 編碼器會將影片的每一幀標記為三種類型之一，每種類型承擔不同的角色：

幀類型 全名 壓縮方式 大小 說明
I-Frame Intra-coded Frame 獨立壓縮（類似 JPEG） 最大 完整畫面，是所有預測的「基石」，又稱關鍵幀（Keyframe）
P-Frame Predicted Frame 參考前一幀的差異 中等 只記錄「跟前一幀相比，哪些區塊移動了、哪些色彩變了」
B-Frame Bi-directional Frame 同時參考前後幀 最小 壓縮率最高，但需要緩衝未來的幀，增加解碼延遲
備註 在 ESP32 等嵌入式設備的即時串流場景中，通常不會使用 B-Frame，因為 B-Frame 需要等待後續幀才能解碼，這會增加無法接受的延遲。ESP32 的 H.264 編碼器通常只產生 I-Frame 與 P-Frame 的組合。

GOP 結構與運動補償

一組連續的幀從 I-Frame 開始，到下一個 I-Frame 之前結束，這組幀的集合稱為GOP（Group of Pictures）。典型的 GOP 結構如：I-P-P-P-P-P-P-P-P-P-I，其中 GOP 長度為 10，表示每 10 幀插入一個關鍵幀。

// ESP32 H.264 GOP 設定範例 typedef struct { int gop_size; // GOP 長度（兩個 I-Frame 之間的幀數） int target_bitrate; // 目標位元率 (bps) int fps; // 輸出幀率 } h264_config_t; // GOP=10：每 10 幀送一個 I-Frame // GOP 越長 → 壓縮率越高 → 但丟包恢復越慢 h264_config_t cfg = { .gop_size = 10, .target_bitrate = 500000, .fps = 15 };

運動補償（Motion Compensation）是 P-Frame 壓縮的核心機制。編碼器將畫面切成 16×16 像素的巨集區塊（Macroblock），然後分析每個區塊在前後幀之間的位移。例如，一輛車向右移動了 5 個像素，編碼器只需記錄「把這個區塊向右平移 5 像素」這條運動向量（Motion Vector），而不是重新傳送整個車輛的像素資料。

H.264 vs MJPEG 效能對比

H.264 (AVC)
  • 頻寬消耗僅為 MJPEG 的 1/10
  • 適合頻寬受限的長時間監控
  • 丟包時出現方塊殘影，需等待下個 I-Frame 恢復
  • 編碼需要較多 CPU/硬體資源
MJPEG
  • 頻寬消耗為 H.264 的 10 倍以上
  • 適合短距離、高頻寬的即時遙控
  • 丟包時只掉一幀，下一幀立即正常
  • 編碼極簡單，ESP32 硬體即可處理
重要：馬賽克殘影的成因 當網路丟失了一個 P-Frame，解碼器缺少了「差異資料」，卻仍然以前一幀為基礎繼續預測後續幀。這導致畫面中出現殘留的方塊（Macroblock Artifacts），直到下一個 I-Frame 抵達才能完全「重置」畫面。GOP 設定越長（I-Frame 間隔越大），殘影持續的時間就越久。

延遲 (Latency) 的組成

在即時影像應用中，「延遲」是比「畫質」更致命的問題。你的遙控車畫面解析度再高，如果延遲超過 200 毫秒，操控者看到障礙物時車子早已撞上去了。理解延遲的組成，才能知道該從哪裡「砍」延遲、該犧牲什麼來換取即時性。

端到端延遲的三大組成

從攝影機捕捉到光線，到使用者在螢幕上看到畫面，整段路徑的總延遲稱為端到端延遲（End-to-End Latency）。它由三個階段疊加而成：

延遲階段 MJPEG 典型值 H.264 典型值 主要影響因素
編碼延遲
感測器擷取 → 壓縮完成		 10–30 ms 30–100 ms 編碼器硬體效能、幀大小、是否使用 B-Frame
傳輸延遲
封包從裝置到客戶端		 5–50 ms 5–50 ms 網路品質（Ping）、封包大小、壅塞程度
解碼 / 緩衝延遲
接收端解壓 → 顯示		 5–15 ms 50–200 ms 解碼器緩衝區大小、GOP 長度、播放器策略
端到端總計 20–95 ms 85–350 ms —
重要 H.264 的解碼延遲遠高於 MJPEG，因為解碼器需要緩衝多個幀才能開始解碼 P-Frame。特別是使用 B-Frame 時，解碼器必須等待「未來的幀」到達後才能回頭解碼，這會額外增加至少一個幀間隔（約 33–66 ms）的延遲。

為什麼 MJPEG 天生低延遲？

MJPEG 的每一幀都是獨立可解碼的——接收端拿到一幀 JPEG，立刻就能解碼顯示，不需要等待前面或後面的幀。沒有幀間依賴、沒有解碼緩衝區，這使得 MJPEG 的端到端延遲可以壓到 50 ms 以下，接近人類感知的即時極限。

// MJPEG 延遲路徑（極簡） 感測器擷取(10ms) → JPEG編碼(15ms) → TCP傳送(20ms) → 直接顯示(5ms) // 總計：約 50 ms // H.264 延遲路徑（含緩衝） 感測器擷取(10ms) → H.264編碼(50ms) → TCP傳送(20ms) → 緩衝2幀(66ms) → 解碼(20ms) // 總計：約 166 ms

場景化延遲需求分析

不同的應用場景對延遲的容忍度截然不同：

應用場景 可接受延遲 推薦編碼 決策依據
遙控賽車 / FPV 飛行 < 100 ms MJPEG 操控者需即時反應，延遲 = 碰撞風險
遠端手術 / 精密操作 < 50 ms MJPEG（低解析度） 毫秒級延遲攸關生命安全
居家安防監控 < 2000 ms H.264 頻寬與儲存成本優先，延遲容忍度高
自動駕駛視覺避障 < 30 ms 無壓縮 / MJPEG 影像處理在本地端完成，不經網路傳輸
提示 判斷「該選 MJPEG 還是 H.264」的簡易法則：如果延遲會造成物理傷害或操控失敗（如碰撞），選擇 MJPEG 並犧牲頻寬；如果系統只需要事後回放或長時間監控，選擇 H.264 並節省頻寬與儲存空間。

測量端到端延遲的方法

在實戰挑戰中，你將使用 tools/latency_meter.html——一個顯示高速計數器的網頁。方法是將攝影機對準運行中的計時器螢幕，然後同時拍下「原始螢幕上的數字」與「串流畫面上的數字」。兩者的差值就是端到端延遲。

備註 記住：「沒有完美方案，只有最適合場景的權衡。」MJPEG 與 H.264 各有不可調和的物理限制，你的工程價值在於能根據需求做出有根據的技術選擇——這正是本模組希望你帶走的核心能力。

    4. 在快速甩動相機時，描述為什麼 H.264 會出現「方塊殘影 (P-Frame Artifacts)」。
    5. 提交證據：上傳一張帶有明顯編碼殘影的截圖 (建議命名為 latency-proof.jpg)。

任務 3：適應性編碼策略 (Adaptive Strategy Design)

  • 目標：根據業務場景（即時遙控 vs. 錄影存證）設計動態切換策略。
  • 要求： 在 src/codec_switcher.cpp 中實作以下狀態機邏輯：
    • 遠端遙控模式 (REMOTE_CONTROL)：切換為 MJPEG (優先保證延遲低於 100ms)。
    • 儲存模式 (STORAGE_MODE)：切換為 H.264 (優先保證節省 SD 卡空間)。
