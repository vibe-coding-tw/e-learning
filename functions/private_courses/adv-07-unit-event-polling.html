簡介

本單元探討前端與硬體通訊中兩種最基本的資料同步模式：事件驅動（Event-Driven）與輪詢（Polling）。理解這兩種架構的差異，能幫助你在延遲敏感度、電力消耗與伺服器負載之間做出正確取捨，並為不同應用場景選用最合適的即時更新策略。

情境
你負責一個工廠產線的即時告警系統：當機台異常時必須在 500 毫秒內通知操作員。起初你使用每秒一次的輪詢來檢查狀態，卻發現反應太慢且浪費頻寬；改用 WebSocket 事件推送後速度達標，但連線中斷的處理又成為新挑戰。這兩種模式各有何優劣？如何設計 fallback 機制？本單元將為你逐一解析。

觀念拆解：外送員與餐廳的溝通

在軟體與硬體系統中，「如何得知狀態變化」是一個核心問題。無論是前端網頁需要即時更新資料，還是嵌入式裝置需要偵測感測器數值，都離不開兩種基本模式：輪詢（Polling）與事件驅動（Event-Driven）。這兩種模式的差異，直接決定了系統的效能、功耗與回應速度。

情境比喻：外送訂單追蹤
想像你訂了一份外送午餐。輪詢模式就像你每隔 30 秒就打一通電話給餐廳：「餐點好了嗎？」——大部分時候得到的回答都是「還沒」，但你依然不停地打。這不僅佔用你的時間，也讓餐廳員工疲於應付。

事件驅動模式則完全不同：你告訴餐廳「好了請傳簡訊給我」，然後安心做自己的事。當餐點備妥的那一刻，你立即收到通知並前去取餐——零浪費、零延遲。

輪詢（Polling）的運作原理

輪詢是一種主動查詢機制：程式透過定時迴圈，反覆向目標詢問「有沒有新資料？」。這種方式邏輯簡單、容易實作，但代價是 CPU 必須持續執行檢查，即使大部分時間什麼事都沒發生。這種空轉（Busy-waiting）在資源有限的嵌入式系統上尤為致命。

// 輪詢模式：每 500ms 檢查一次輸入框的值
setInterval(() => {
  const val = document.getElementById('password').value;
  if (val.length >= 8) {
    document.getElementById('submitBtn').disabled = false;
  }
}, 500);

事件驅動（Event-Driven）的運作原理

事件驅動是一種被動等待機制：程式預先註冊一個「回調函式（Callback）」，當目標事件真正發生時，系統自動呼叫該函式。在事件未發生期間，CPU 處於閒置狀態，不執行任何無謂的檢查。這種模式更節能、回應更即時，但程式流程較不直觀，需要理解非同步程式設計的概念。

// 事件驅動模式：僅在使用者實際輸入時才觸發
document.getElementById('password')
  .addEventListener('input', (e) => {
    if (e.target.value.length >= 8) {
      document.getElementById('submitBtn').disabled = false;
    }
  });

兩種模式的比較

比較項目 輪詢（Polling） 事件驅動（Event-Driven）
觸發方式 主動定時查詢 被動等待通知
CPU 佔用 持續佔用（即使無事發生） 僅事件發生時佔用
回應延遲 最大延遲 = 輪詢間隔時間 接近零延遲（即時回調）
實作難度 簡單直觀（線性邏輯） 較複雜（需理解回調與非同步）
功耗影響 高（持續運算） 低（閒置時可進入省電模式）
適用場景 需定頻取樣的感測器數據 按鈕點擊、連線狀態變化、中斷
注意 輪詢並非總是劣於事件驅動。在某些場景中，輪詢反而是更好的選擇——例如搖桿座標需要以固定頻率（如 20Hz）送出時，使用 setInterval 定時發送可以避免高頻事件「打爆」藍牙協議棧。實務上，許多系統會採用混合架構：用事件即時更新緩衝區，再用輪詢定頻發送。

瀏覽器中的事件與循環

JavaScript 是一門單執行緒（Single-threaded）語言，這意味著同一時間只能執行一段程式碼。那麼，瀏覽器是如何同時處理使用者點擊、網路請求、計時器回調等多個任務的呢？答案就是 Event Loop（事件循環）——它是瀏覽器非同步運行的核心引擎。

Event Loop 的運作機制

Event Loop 持續監控三個關鍵區域，並決定哪段程式碼應該在什麼時候執行：

  • Call Stack（呼叫堆疊）：目前正在執行的函式。JavaScript 引擎會逐一從堆疊頂端取出函式執行，直到堆疊清空。
  • Task Queue（任務佇列）：又稱 Callback Queue 或 Macrotask Queue。setTimeout、setInterval、DOM 事件回調等完成後，會被排入此佇列。Event Loop 在 Call Stack 清空後，從這裡取出下一個任務執行。
  • Microtask Queue（微任務佇列）：Promise.then()、MutationObserver 等產生的回調會進入此佇列。微任務的優先級高於一般任務——每當 Call Stack 清空時，Event Loop 會先清空所有微任務，才會處理下一個 Task Queue 中的任務。
運作流程
Event Loop 的核心循環可簡化為：① 執行 Call Stack 中的所有同步程式碼 → ② 清空 Microtask Queue → ③ 從 Task Queue 取出一個回調執行 → ④ 若需要，執行瀏覽器渲染（repaint）→ 回到 ①。這個循環永不停止，因此稱為「事件循環」。

使用 addEventListener 監聽事件

在 IoT 儀表板中，藍牙連線狀態的改變（連線成功、斷線）是典型的事件驅動場景。我們使用 addEventListener 註冊回調函式，當事件發生時由瀏覽器將回調推入 Task Queue，等待 Event Loop 排程執行。

// 監聽 BLE 裝置斷線事件
bleDevice.addEventListener('gattserverdisconnected', (event) => {
  console.warn('⚠️ 藍牙裝置已斷線！', event.target.name);
  updateUI('disconnected');
  attemptReconnect();
});

// 監聽按鈕點擊事件
document.getElementById('connectBtn')
  .addEventListener('click', async () => {
    const device = await navigator.bluetooth.requestDevice({
      filters: [{ services: ['battery_service'] }]
    });
    console.log('已選擇裝置：', device.name);
  });

使用 setInterval 實作輪詢

對於需要定頻取樣的資料——例如搖桿座標——事件驅動並不合適（mousemove 每秒可能觸發數百次，遠超 BLE 的傳輸能力）。此時使用 setInterval 以固定頻率讀取並發送數據，是更穩定的做法。

// 輪詢模式：每 50ms (20Hz) 讀取搖桿座標並發送至 ESP32
let latestX = 0, latestY = 0;

// 事件層：快速更新座標到緩衝區
joystick.addEventListener('mousemove', (e) => {
  latestX = e.offsetX;
  latestY = e.offsetY;
});

// 輪詢層：以固定頻率發送最新座標
const pollId = setInterval(() => {
  sendBLE({ x: latestX, y: latestY });
}, 50);
選擇指南 在 IoT 前端開發中，選擇輪詢還是事件驅動取決於資料的產生特性：離散且不可預測的事件（按鈕點擊、藍牙斷線）→ 用 addEventListener；連續且需要定頻取樣的資料（搖桿座標、感測器數值）→ 用 setInterval 輪詢或混合架構。
資源清理 務必在元件卸載或頁面關閉時清除計時器與事件監聽器，否則將造成記憶體洩漏（Memory Leak）。使用 clearInterval(pollId) 停止輪詢，使用 removeEventListener 移除事件監聽。在 BLE 斷線後也應清除所有相關的 Interval，避免對已關閉的連線持續發送指令。

硬體端的中斷 (Hardware Interrupts)

在嵌入式系統中，「中斷（Interrupt）」是比輪詢更底層、更即時的事件驅動機制。當 ESP32 的某個 GPIO 引腳狀態發生變化（例如碰撞感測器被觸發），硬體會強制暫停目前正在執行的程式碼，立即跳轉到預先註冊的中斷服務程式（ISR, Interrupt Service Routine）。這個過程不需要等待主程式完成當前迴圈，延遲通常在微秒（μs）等級。

ESP32 上的 attachInterrupt

在 Arduino / PlatformIO 框架下，我們使用 attachInterrupt() 函式將某個 GPIO 引腳與一個 ISR 綁定。當該引腳偵測到指定的電位變化（上升緣 RISING、下降緣 FALLING、或任意變化 CHANGE）時，ISR 會被立即執行。

// ESP32 C++ - 碰撞感測器中斷停止馬達
#define BUMP_PIN 14
#define MOTOR_PIN 25

volatile bool emergencyStop = false;

// ISR：必須極短、不可使用 Serial.print
void IRAM_ATTR onBump() {
  emergencyStop = true;
  digitalWrite(MOTOR_PIN, LOW); // 立即切斷馬達
}

void setup() {
  pinMode(BUMP_PIN, INPUT_PULLUP);
  pinMode(MOTOR_PIN, OUTPUT);
  attachInterrupt(digitalPinToInterrupt(BUMP_PIN), onBump, FALLING);
}

void loop() {
  if (emergencyStop) {
    Serial.println("緊急停止已觸發！");
    while(true) { delay(1000); } // 停機等待人工處理
  }
  // 正常馬達控制邏輯...
}
ISR 限制 中斷服務程式（ISR）必須盡可能精簡。在 ESP32 的 ISR 中，禁止使用 Serial.print()、delay()、malloc() 等耗時或需要中斷的函式。ISR 只應設定旗標（flag）或操作 GPIO，複雜的後續處理應回到 loop() 中執行。使用 volatile 關鍵字宣告與 ISR 共享的變數，避免編譯器最佳化導致主程式讀不到更新值。

瀏覽器端的「軟體中斷」：gattserverdisconnected

在前端（網頁）控制藍牙裝置時，同樣需要事件驅動的安全機制。Web Bluetooth API 提供的 gattserverdisconnected 事件就扮演著「軟體中斷」的角色：一旦藍牙連線意外中斷，瀏覽器會立即觸發此事件，讓你的程式碼可以在第一時間執行安全措施。

// JavaScript - 藍牙斷線緊急停止
let motorSpeed = 0;
let pollInterval = null;

function onDisconnected(event) {
  console.error('🚨 BLE 裝置斷線！立即停止馬達');
  motorSpeed = 0; // 馬達速度歸零
  clearInterval(pollInterval); // 停止座標輪詢
  updateUI('emergency-stop');
}

bleDevice.addEventListener(
  'gattserverdisconnected',
  onDisconnected
);
安全關鍵 在安全性攸關（Safety-critical）的系統中，絕對不能使用輪詢來偵測緊急狀況。如果你以每秒一次的頻率輪詢連線狀態，在最壞情況下，車輛可能在斷線後繼續全速行駛長達 1 秒鐘——對於一台速度為 1 m/s 的小車，這代表它會失控衝出 1 公尺。事件驅動（中斷）讓回應時間降至毫秒甚至微秒等級，是安全機制的唯一可靠選擇。

輪詢安全 vs. 中斷安全：延遲風險比較

比較項目 輪詢式安全偵測 中斷式安全偵測
偵測機制 主迴圈中定時檢查 isConnected() 硬體中斷 / gattserverdisconnected 事件
最大回應延遲 等於輪詢間隔（如 100ms ~ 1000ms） 微秒等級（硬體）/ 毫秒等級（軟體）
主程式阻塞時 可能無限延遲（檢查被跳過） 仍可觸發（硬體中斷獨立於主程式）
CPU 空閒時的行為 持續消耗 CPU 資源執行檢查 CPU 可進入省電模式，中斷喚醒
失控風險 高——間隔期間車輛持續運行 極低——事件發生即刻回應
適用場景 非關鍵性的狀態監控 安全停車、碰撞偵測、斷線保護

理解硬體中斷與事件驅動的核心思想後，你會發現：事件驅動不僅是為了效能，更是為了安全。在下一個單元的實作挑戰中，你將親手實作這些概念——從效能對比實驗到搖桿混合架構，再到緊急停止系統的建置。

任務 1：從輪詢到事件監聽

目標：透過實驗親眼看見「被動監聽」如何大幅節省 CPU 資源。

  1. 在 js/polling_bench.js 中，用 setInterval 實作定時輪詢一個輸入框的值。
  2. 在 js/event_bench.js 中，用 addEventListener('input', ...) 實作事件監聽版本。
  3. 使用 Chrome DevTools → Performance 標籤，分別錄製兩套邏輯在「使用者無操作」時的 CPU 佔用截圖。
  4. 將截圖放入 performance_data/ 資料夾，並在本 README 底部填寫你的觀察結論。

任務 2：混合架構搖桿（Hybrid Joystick）

目標：解決「高頻事件雨」打爆藍牙協議棧（Write Busy 錯誤）的問題。

在 js/hybrid_joystick.js 中實作以下混合架構：

// 事件層：快速更新緩衝區（不直接傳送）
joystick.on('move', (pos) => {
  latestPos = pos; // TODO: 更新緩衝區
});

// 輪詢層：以固定頻率（20Hz）定時發送
setInterval(() => {
  ble.send(latestPos); // TODO: 傳送最新座標
}, 50);

驗收標準（Autograding）：模擬每秒 500 次移動事件，你的 BLE 發送頻率必須穩定維持在 20Hz，且不得丟失最後一組座標。

任務 3：安全緊急停止（Emergency Fail-safe）

目標：理解為何安全機制必須掛載在「事件中斷層」，而非普通輪詢迴圈。

在 js/safety_logic.js 中實作 gattserverdisconnected 事件監聽：

  • 一旦藍牙連線中斷，必須立即將馬達速度清零。
  • 思考題（請在 README 心得區回答）：若改用「每秒輪詢一次連線狀態」的方式實作安全停車，最壞的情況下車子可能發生什麼？

驗收標準（Autograding）：測試程式會發送一個模擬斷連事件，motorSpeed 必須在事件觸發後瞬間歸零（< 5ms）。

任務 1：從輪詢到監聽 (From Polling to Event)

  • 目標：解決「高頻事件雨」導致的藍牙協議棧堵塞（Write Busy）。

      // 學生需實作：Event 更新 Buffer, Polling 定時發送
