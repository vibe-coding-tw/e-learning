🍕 1. 觀念拆解：外送員與餐廳的溝通 (25 min)

輪詢像是不停打電話問「好了嗎？」，而事件驅動則是等對方主動傳簡訊通知。這在嵌入式系統中直接影響到 CPU 的佔用率。

Polling

主動詢問、邏輯簡單，但極度浪費資源且具備固定延遲。

Event-Driven

被動中斷、零閒置浪費，反應即時但程式流程較跳躍。

🚨 3. 硬體端的中斷 (Hardware Interrupts)

當網頁傳送「緊急停止」或車子撞到東西時，ESP32 如何利用 attachInterrupt 立即跳出主程式進行處理？這是安全性通訊的最終防線。

Fail-safe Logic

  • 致命延遲：輪詢 100ms 代表你可能有 100ms 的空白期無法煞車。
  • 優先級調色：中斷處理器 (ISR) 會強制暫停 CPU 目前的工作。
  • 連線監聽：瀏覽器 gattserverdisconnected 的零秒感應能力。

ADV-07｜事件驅動 vs 輪詢 (Event-Driven vs Polling)

🎯 學習目標

完成本次作業後，你將能夠：

  1. 解釋輪詢（Polling）與事件驅動（Event-Driven）的根本差異，並用 CPU 佔用數據支持你的論點。
  2. 實作混合架構（Event Buffer + Polling Dispatch），解決高頻事件與低速傳輸通道之間的矛盾。
  3. 建立基於事件中斷的安全停止機制（Fail-safe），理解為何它在安全工程中不可被輪詢取代。

📝 任務說明

任務 1：從輪詢到事件監聽

目標：透過實驗親眼看見「被動監聽」如何大幅節省 CPU 資源。

  1. 在 js/polling_bench.js 中，用 setInterval 實作定時輪詢一個輸入框的值。
  2. 在 js/event_bench.js 中，用 addEventListener('input', ...) 實作事件監聽版本。
  3. 使用 Chrome DevTools → Performance 標籤，分別錄製兩套邏輯在「使用者無操作」時的 CPU 佔用截圖。
  4. 將截圖放入 performance_data/ 資料夾，並在本 README 底部填寫你的觀察結論。

任務 2：混合架構搖桿（Hybrid Joystick）

目標：解決「高頻事件雨」打爆藍牙協議棧（Write Busy 錯誤）的問題。

在 js/hybrid_joystick.js 中實作以下混合架構：

// 事件層：快速更新緩衝區（不直接傳送）
joystick.on('move', (pos) => {
  latestPos = pos; // TODO: 更新緩衝區
});

// 輪詢層：以固定頻率（20Hz）定時發送
setInterval(() => {
  ble.send(latestPos); // TODO: 傳送最新座標
}, 50);

驗收標準（Autograding）：模擬每秒 500 次移動事件，你的 BLE 發送頻率必須穩定維持在 20Hz，且不得丟失最後一組座標。

任務 3：安全緊急停止（Emergency Fail-safe）

目標：理解為何安全機制必須掛載在「事件中斷層」，而非普通輪詢迴圈。

在 js/safety_logic.js 中實作 gattserverdisconnected 事件監聽：

  • 一旦藍牙連線中斷，必須立即將馬達速度清零。
  • 思考題（請在 README 心得區回答）：若改用「每秒輪詢一次連線狀態」的方式實作安全停車，最壞的情況下車子可能發生什麼？

驗收標準（Autograding）：測試程式會發送一個模擬斷連事件，motorSpeed 必須在事件觸發後瞬間歸零（< 5ms）。

2. 作業任務部署細節

任務 1：從輪詢到監聽 (From Polling to Event)

  • 目標：透過實驗體悟「被動監聽」如何節省 CPU 資源。
  • Classroom 部署建議：
    • 效能實驗要求：學員需分別執行「定時輪詢輸入框」與「監聽 input 事件」兩套邏輯。
    • 數據提交：學員必須在 README 中貼出兩者的 CPU 佔用對比圖。這能讓學員親眼看見：輪詢在使用者「什麼都沒做」時依然在燃燒 CPU 資源。
    • PR 深度核核：點評其對「空轉 (Busy-waiting)」概念的解釋。

任務 2：「手感微調」：搖桿的自適應輪詢 (Hybrid Joystick Logic)

  • 目標：解決「高頻事件雨」導致的藍牙協議棧堵塞（Write Busy）。
  • Classroom 部署建議：
    • 混合架構檢核： 
      // 學生需實作：Event 更新 Buffer, Polling 定時發送
joystick.on('move', (pos) => { latestPos = pos; }); // 事件回調（極快）
setInterval(() => { ble.send(latestPos); }, 50);   // 定時發送（20Hz 定頻）
    • Autograding：模擬極高頻率的移動（每秒 500 次），檢核學員的 BLE 指令發送頻率是否穩定維持在 20Hz，且沒有發生丟失最後一組座標的邏輯錯誤。
    • 性能回饋：這能教會學員如何處理「數據產生速度 > 數據傳輸速度」的工程矛盾。

任務 3：安全緊急停止系統 (Emergency Fail-safe)

  • 目標：理解中斷 (Interrupt) 思想在安全性工程中的不可替代性。
  • Classroom 部署建議：
    • 安全邏輯檢核：學員需實作 gattserverdisconnected 事件。一旦觸發，必須清空馬達變數。
    • 對比挑戰：要求學員思考：如果用「每秒檢查一次連線」的輪詢法來做安全停車，會發生什麼危險？（答案：車子可能已經衝出桌面 1 公尺後才發現斷線）。
    • Autograding：發送一個模擬斷連事件，檢測 motorSpeed 是否「瞬間」變為 0。

3. 硬體架構導師點評標準 (Systems Architecture Benchmarks)

此單元的價值在於 「對非同步控制與資源管理的直覺」：

  • [ ] 資源利用率意識：是否理解「不必要的回調就是浪費」？代碼中是否有適當的 Cleanup（如：clearInterval）？
  • [ ] 通訊流暢度 (Throttling)：當快速操作時，藍牙連線是否依然保持穩定而不報錯？
  • [ ] ** Fail-safe 思維**：安全保護機制是否掛載在最優先级的「事件中斷層」而非普通的應用迴圈中？
