事件驅動 vs 輪詢
Event-Driven Architecture

優化數據存取效能：理解中斷 (Interrupt) 與輪詢 (Polling) 的差異，打造高效的即時監控系統。

🍕 1. 觀念拆解：外送員與餐廳的溝通 (25 min)

輪詢像是不停打電話問「好了嗎？」，而事件驅動則是等對方主動傳簡訊通知。這在嵌入式系統中直接影響到 CPU 的佔用率。

Polling

主動詢問、邏輯簡單，但極度浪費資源且具備固定延遲。

Event-Driven

被動中斷、零閒置浪費，反應即時但程式流程較跳躍。

🔄 2. 瀏覽器中的事件與循環 (25 min)

理解 addEventListener 如何監聽藍牙斷線與按鈕點擊。在 JavaScript 中，Event Loop 決定了回調函式的執行時機，而輪詢則常透過 setInterval 實作。

Scenario Selection

  • 搖桿控制：適合輪詢 (定時更新座標)。
  • 連線成功/失敗：適合事件 (異步回調)。

🚨 3. 硬體端的中斷 (Hardware Interrupts)

當網頁傳送「緊急停止」或車子撞到東西時，ESP32 如何利用 attachInterrupt 立即跳出主程式進行處理？這是安全性通訊的最終防線。

Fail-safe Logic

  • 致命延遲：輪詢 100ms 代表你可能有 100ms 的空白期無法煞車。
  • 優先級調色：中斷處理器 (ISR) 會強制暫停 CPU 目前的工作。
  • 連線監聽：瀏覽器 gattserverdisconnected 的零秒感應能力。

Emergency Vector

STOP
