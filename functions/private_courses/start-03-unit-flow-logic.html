核心技術：頻率與流量控制

Module 2: Core Principles
🔁

1. 非同步迴圈 (setInterval)

瀏覽器的事件 (如 scroll 或 drag) 觸發頻率極高。為了穩定控制，我們不直接使用事件驅動，而是開啟一個 setInterval 定時器。

timer = setInterval(() => {
  sendCmd('forward');
}, 50); // 每 50ms 發送一次
🖐️

2. 防抖與節流

為什麼不能每 1ms 就傳一次？Wi-Fi 頻寬有限，過快的指令會導致 ESP32 處理不及而「當機」或延遲 (Lag)。

最佳實踐：
控制頻率在 20Hz - 50Hz 之間 (即每 20ms-50ms 一次) 最為流暢且穩定。
💓

3. 心跳機制

如果網路突然斷線，車子收不到「停止」指令怎麼辦？這就是「死人開關 (Dead Man's Switch)」的重要性。

安全邏輯：若 ESP32 超過 500ms 未收到新指令，應自動切斷馬達電源。

指令流邏輯 — 單次觸發與重複發送

歡迎來到 指令流邏輯 (Temporal Logic & Safety) 單元！

在這個單元中，你將學習如何將「單次的點擊指令」轉化為「連續的動態節奏」。這對於開發高頻控制系統（如賽道競技、機械臂控制或自動化終端）至關重要。你將掌握 setInterval 的精準管理、多按鍵狀態聚合，以及最重要的「失效安全 (Fail-safe)」機制。

🎯 學習目標

  • 掌握 setInterval 與 clearInterval 的生命週期管理。
  • 實作「狀態機」概念處理多按鍵同時輸入的衝突。
  • 建立安全防護意識，實作「死人開關 (Dead Man's Switch)」機制。

✍️ 作業任務 (Assignment Tasks)

請按照以下步驟在 js/logic.js 中完成開發：

1. 實作指令流控制

  • 完成 startFlow 與 stopFlow 函式。
  • 關鍵要點：確保在連點按鈕或快速按鍵時，不會產生定時器堆疊（Timer Leak）。在啟動新的定時器前，必須確保舊的已被清除。

2. 實作多鍵並發邏輯

  • 使用 keysActive 物件記錄所有按鍵的狀態。
  • 確保當同時按下多個按鍵（例如：前進 + 右轉）時，日誌能正確顯示組合後的指令（如：ArrowUp + ArrowRight）。

3. 實作失效安全機制 (Dead Man's Switch)

  • 監聽 window.blur (失去焦點) 或 visibilitychange (切換分頁) 事件。
  • 當使用者離開網頁視窗時，系統必須立即執行 emergencyStop，停止所有指令並送出 STOP 命令。

針對 start-03-unit-flow-logic (指令流邏輯：單次觸發與重複發送) 單元，這是學員從「靜態指令」進化到「動態節奏」的進階課程。它要求學員具備 時序邏輯 (Temporal Logic) 與 失效安全 (Fail-safe) 的意識，學習如何利用 setInterval 構建穩定的控制循環，處理多按鍵衝突，並實作極其重要的「死人開關 (Dead Man's Switch)」安全機制。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template Repo) 配置建議

控制流作業的核心在於「時序精準度、狀態一致性與安全防護」，範本應包含指令流偵錯工具與事件監控框架，建議包含：

  • 📂 index.html：指令流可視化監控器。預設包含一個「事件日誌 (Event Logger)」，學員在開發時能即時看到 setInterval 觸發的頻率與具體送出的指令。
  • 📂 logic.js：核心控制邏輯區。學員在此實作定時器啟動/停止、多鍵狀態聚合與安全監聽器。預留 // TODO 引導學員處理定時器記憶體洩漏問題。
  • 📂 README.md：系統穩定性分析錄。提供模板要求學員紀錄：在不同的 interval（如 10ms vs 100ms）下，機器人的反應與穩定性有何差異？
  • 📂 tests/：邏輯邊界測試。使用 Actions 或腳本自動驗證：當視窗失去焦點 (Blur) 時，系統是否真的送出了 STOP 指令？

2. 作業任務部署細節

任務 1：指令發送「甜蜜點」實驗 (Interval Tuning Lab)

  • 目標：尋找效率與穩定的平衡點。掌握非同步迴圈的正確生命週期管理（預防重複啟動定時器）。
  • Classroom 部署建議：
    • 定時器實作挑戰：撰寫一個穩健的 startFlow 函式，確保在連發過程中，不會因為重複點擊而產生定時器堆疊。
    • 頻率測試實驗：在 README 中詳細紀錄 10ms、50ms、100ms 下的操作感，並說明為什麼過快的頻率（如 1ms）會導致硬體端潰散。
    • 驗證要點：資源管理審核。學員是否在啟動前檢查了 flowInterval 是否已存在？選定的頻率是否能兼顧順暢與安全？

任務 2：多按鍵同時輸入衝突解決 (Multi-Key Logic Lab)

  • 目標：實作「平行指令處理」。確保當使用者同時按下「前進」與「左轉」時，系統能產生合併指令，而非讓機器人在兩者間跳動或鎖死。
  • Classroom 部署建議：
    • 狀態機設計實作：使用一個物件（如 keysActive）記錄所有按鍵的真假值，並在每個循環中根據該物件內容合成最終 Payload。
    • 複雜狀態驗證：模擬快速交替按鍵的操作，驗證是否有按鍵被「黏死」或狀態未正確重置的現象。
    • 驗證要點：組合指令的輸出是否符合物理直覺？代碼邏輯是否清晰且易於擴展到更多按鍵？

任務 3：斷訊安全停機機制 (Dead Man's Switch Lab)

  • 目標：實作「失效安全」。確保在極端狀況下（網頁當機、切換分頁、用戶分心），機器人具備自動歸零、防衝撞的自我保護能力。
  • Classroom 部署建議：
    • 危險事件監聽實作：監聽 window.blur 或 visibilitychange 事件，並在觸發時立即執行 stopFlow 並送出 STOP 命令。
    • 模擬失控測試：在 README 中錄製或截圖證明：當按住按鈕時切換到別的視窗，系統日誌是否顯示了自動停機。
    • 驗證要點：安全機制的優先權是否足夠高？是否徹底清除了所有掛載的定時器以釋放系統資源？

3. 時序健壯性與安全意識點評標準 (Temporal Robustness & Safety Standards)

此單元的價值在於 「對非同步流程的絕對掌控力與對防護邏輯的工程嚴謹力」：

  • [ ] 資源釋放嚴謹度：定時器是否在所有預期路徑（放開按鍵、安全停機）中都被正確 clearInterval？
  • [ ] 狀態聚合準確性：多鍵並發時，生成的指令是否能精準反映最後一個綜合的操作意圖？
  • [ ] 安全防護覆蓋率：除了正常的 mouseup，是否考慮到了滑鼠滑出視窗、視窗失去焦點等異常邊界情境？
