Developer Notes

UUID Configuration

  • Service: 6E400001-B5A3-F393-E0A9-E50E24DCCA9E
  • Char (SSID): 6E400002... (Write)
  • Char (Pwd): 6E400003... (Write)

Key Logic

TextEncoder is used to convert strings to Uint8Array before writing to characteristics.

Device auto-restarts upon receiving the password (implied commit).

  • [ ] 魯棒性：當藍牙意外中斷時，網頁是否能正確引導使用者重新連線？

任務 2：編碼偵錯 - UTF-8 協議與字元檢查

角色扮演：通訊協定工程師 | 難度：⭐️⭐️ | 估計時長：40 min

🛠️ 執行技術流程 (Git Workflow)

  1. 觀察編碼：使用 console.log 觀察 TextEncoder 對包含空格或特殊符號的 SSID 的編碼結果。
  2. 長度檢查：在前端實作字元長度限制，防止傳送超過 ESP32 緩衝區大小的密碼。
  3. 異常捕捉：使用 try...catch 攔截藍牙寫入時的異常，並將錯誤代碼顯示在網頁控制台上。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 除錯效率：是否能透過日誌精確診斷傳輸失敗的原因？

任務 3：永恆連結 - NVS 記憶與自動重連驗證

角色扮演：韌體維運工程師 | 難度：⭐️⭐️⭐️ | 估計時長：30 min

🛠️ 執行技術流程 (Git Workflow)

  1. 斷電測試：將設定好的 WiFi 裝置拔除電源再重新插上，驗證是否能自動連回網路。
  2. 持久性檢查：確認 SSID/Password 是否正確存入 ESP32 的 NVS (Non-volatile storage) 分區。
  3. 多次覆寫實驗：測試快速連續更改 WiFi 設定，觀察 NVS 存取是否發生衝突或損毀。

本單元聚焦在 Web Bluetooth + ESP32 WiFi 配網 的完整流程。請依序完成以下任務，並將成果提交到您的 GitHub Classroom repository。

任務 1：連線狀態與 UX 回饋

  • 實作明確的藍牙連線狀態顯示（連線中、已連線、已中斷）。

  • 提交內容：除錯紀錄（console output）與錯誤處理邏輯說明。

任務 3：NVS 記憶與重啟驗證

  • 驗證 WiFi 設定可被持久化儲存，重啟後可自動連線。
  • 進行多次覆寫測試，確認配置流程在連續操作下仍穩定。
  • 提交內容：重啟測試結果與穩定性觀察結論。

03-unit-wifi-setup (WiFi 組態設定 - BLE 傳輸) 單元，這是一個將「網頁前端技術」應用於「嵌入式系統配網」的實戰單元。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template 程式庫) 配置建議

由於本單元的核心在於 Web Bluetooth (BLE) 的非同步操作，建議範本包含：

  • 📂 wifi-config.html：提供穩定的 UI 框架，但在 connectBtn 與 sendBtn 的 addEventListener 中留下關鍵的非同步邏輯 // TODO。
  • 📂 firmware/esp32-wifi-ble.ino：提供對應的 ESP32 韌體程式碼。這對學生理解「寫入特徵值 (Characteristic)」後硬體如何接收並重啟至關重要。
  • 📂 test/validation.js：預置簡單的 JS 測試，驗證 SSID 與密碼的長度過濾邏輯。

2. 作業任務部署細節

任務 1：行動優先 - UX 反饋與連線狀態優化 (UX/UI Mastery)

  • 目標：提升藍牙連線時的透明度，解決使用者不知是否連線成功的痛點。
  • Classroom 部署建議：
    • 要求：實作 gattserverdisconnected 事件監聽。
    • 自動評分 (Autograding)：檢查 wifi-config.html 中是否包含 animate-pulse (Tailwind) 或類似的 CSS 動畫類名，用於顯示連線中的狀態。
    • 手動核閱：導師在 PR 中檢查學生是否正確重置了「傳送按鈕」的啟用/禁用狀態。

任務 2：編碼偵錯 - UTF-8 協議與異常捕捉 (Protocol Debugging)

  • 目標：確保資料能正確轉換為 Byte Stream 並處理傳輸錯誤。
  • Classroom 部署建議：
    • 驗證方式：學生需在 console.log 記錄 TextEncoder().encode(ssid) 的輸出。
    • Autograding：撰寫一個測試腳本，模擬呼叫學生的傳送函數，並檢查是否由 try...catch 捕捉到無效的藍牙特徵值寫入錯誤（例如模擬 Characteristic not found）。

任務 3：永恆連結 - 硬體連線與 NVS 驗證 (Hardware Integration)

  • 目標：確保持久化儲存 (Non-volatile storage) 運作正常。
  • Classroom 部署建議：
    • 證據提交：學生需將 ESP32 成功連線後的「序列埠日誌 (Serial Monitor Log)」內容複製到 serial-output.log。
    • 核核重點：日誌中必須出現 WiFi Connected! IP: 192.168.x.x 以及 NVS Commit Success 等關鍵字，證明資訊已正確存入硬體快閃記憶體。

3. 給導師的翻轉教室實施建議

  • 技術挑戰提醒：
    • 部分 iOS 使用者可能找不到設備，需提醒下載 Bluefy 瀏覽器。
    • 部分 Android 裝置有 MTU (Maximum Transmission Unit) 限制，若學生輸入超長 SSID 導致截斷，這是絕佳的通訊協議教學機會。
  • 評分指標 (Rubric)：
    • [ ] 非同步掌握度：是否正確使用 await 處理藍牙寫入？
    • [ ] 防呆機制：空值或超長字元是否在前端就被攔截？
    • [ ] 狀態機邏輯：UI 是否能根據「未連線、連線中、已連線、傳送中、成功」等狀態精確切換？

這種部署方式能讓學生理解，前端開發不僅是頁面排版，更是控制設備的核心介面，從而建立深厚的跨領域技術素養。
