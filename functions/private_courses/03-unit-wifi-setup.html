WiFi 組態設定 - BLE 傳輸操作

任務 1：行動優先 - UX 反饋與連線狀態優化

角色扮演：產品設計師 (UX Designer) | 難度：⭐️⭐️ | 估計時長：30 min

🛠️ 執行技術流程 (Git Workflow)

  1. 優化狀態視景：在網頁前端實作連線中的動畫反饋（例如：使用 Tailwind 的 animate-pulse）。
  2. 連線成功回饋：當裝置成功接收 WiFi 資訊後，自動改變網頁背景顏色或顯示明顯的成功 Badge。
  3. 斷線處理：實作 gattserverdisconnected 事件監聽，並在斷線時自動重置 UI 狀態。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 直覺性：使用者是否能從 UI 立即得知當前與硬體的連線狀態？
  • [ ] 魯棒性：當藍牙意外中斷時，網頁是否能正確引導使用者重新連線？

任務 2：編碼偵錯 - UTF-8 協議與字元檢查

角色扮演：通訊協定工程師 | 難度：⭐️⭐️ | 估計時長：40 min

🛠️ 執行技術流程 (Git Workflow)

  1. 觀察編碼：使用 console.log 觀察 TextEncoder 對包含空格或特殊符號的 SSID 的編碼結果。
  2. 長度檢查：在前端實作字元長度限制，防止傳送超過 ESP32 緩衝區大小的密碼。
  3. 異常捕捉：使用 try...catch 攔截藍牙寫入時的異常，並將錯誤代碼顯示在網頁控制台上。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 協議正確性：是否理解 JS 陣列緩衝區 (Uint8Array) 如何轉換為 BLE 的 Byte Stream？
  • [ ] 除錯效率：是否能透過日誌精確診斷傳輸失敗的原因？

任務 3：永恆連結 - NVS 記憶與自動重連驗證

角色扮演：韌體維運工程師 | 難度：⭐️⭐️⭐️ | 估計時長：30 min

🛠️ 執行技術流程 (Git Workflow)

  1. 斷電測試：將設定好的 WiFi 裝置拔除電源再重新插上，驗證是否能自動連回網路。
  2. 持久性檢查：確認 SSID/Password 是否正確存入 ESP32 的 NVS (Non-volatile storage) 分區。
  3. 多次覆寫實驗：測試快速連續更改 WiFi 設定，觀察 NVS 存取是否發生衝突或損毀。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 硬體穩定性：裝置重啟後是否具備自主恢復連網的能力？
  • [ ] NVS 運作原理：是否理解 WiFi 資訊在硬體層級是如何被持久化儲存的？

WiFi 配置教學引導指南

教學重點：

本單元專注於「配網體驗」。重點在於讓學生理解 Web Bluetooth 如何扮演配置面板的角色。評分時應特別注意學生對 NVS 持久化儲存與非同步 JavaScript 操作的掌握度。

翻轉教室的實施策略

階段 學生行動 (Git Workflow) 教師/平台角色
課前 (預習) 學習 Web Bluetooth 的基本連接流程 (Request Device)。 Dashboard：確認學生是否能成功搜尋到開發板。
課中 (實作) 實作配網狀態機，處理連線中、成功、失敗的三種 UI 反饋。 助教角色：解決部分 Android 裝置 MTU 限制導致的字串截斷問題。
課後 (精進) 優化自動重新連線邏輯，並撰寫穩定性測試心得。 Feedback PR：核發評語，確認其理解異步開發的錯誤捕捉。
