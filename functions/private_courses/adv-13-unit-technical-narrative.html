進階 13 - 技術敘事

視覺敘事：
如何展示技術專案

技術再強，沒人看懂也是白搭。學習用視覺與故事為代碼注入靈魂，打造令人驚艷的作品集。

📖 1. 英雄之旅：技術敘事的骨架

別只說 "I made a car"，試著講一個關於「克服極限」的故事。面試官只有 30 秒能被你吸引，你需要一套具備邏輯與張力的敘事框架。

Narrative Structure

  • 挑戰 (The Problem)：為什麼要做這個專案？它解決了什麼現實痛點。
  • 掙扎 (The Struggle)：開發中遇到了什麼障礙（延遲、崩潰、雜訊）？
  • 突破 (The Solution)：你用了什麼黑科技（PID, FSM, Protocol）解決它。

🎨 2. 邏輯視覺化：讓代碼「被看見」

看不見的代碼很難被外部欣賞。利用流程圖、架構圖與子母畫面展示，將抽象的邏輯轉化為直觀的「視覺證據」，增強說服力。

Visual Assets Checklist

  • 指令/數據封包格式圖 (Protocol Specs)
  • PID 誤差修正實測波形 (Waveform Proof)

🎬 3. 實機演示與電梯簡報

沒影片就沒真相 (Pics or it didn't happen)。錄製子母畫面 (PiP) 同步展示代碼邏輯與硬體動態，展現系統的毫秒級響應實力。

Master Demo Strategies

  • 子母畫面 (PiP)：左側放 Dashboard，右側放車子實際動態，證明低延遲控制。
  • 極限挑戰演示：拍攝馬達精確停靠或高速過彎的慢動作鏡頭，展現穩定度。
  • 30 秒原則：別錄太長！在影片前 10 秒內就要讓觀眾看到最炸裂的功能。

核心任務：說一個令人驚艷的技術故事

打造你的專業技術作品集。將這十二週的專案成果轉化為一眼就能吸引開發者、評審或面試官的視覺資產。透過 Git Workflow 建立高質量的專案文檔與 Demo。

🧭 60 min

【作業 13-1】

「系統藍圖繪製」

整理系統模組架構。使用 Mermaid 或方塊圖標註前後端、藍牙與馬達控制層的數據流向。將代碼邏輯轉化為清晰、具備工業感的視覺藍圖。

📉 60 min

【作業 13-2】

「性能證據採集」

定義專案的核心里程碑。採集 Dashboard 上的即時波形 GIF 或對比截圖。在 README 中用數據證明你的系統優化（如延遲削減、更新率提升）成效。

🎬 45 min

【作業 13-3】

「技術電梯簡報」

錄製包含 PiP 的 Demo 影片。結合技術口述與實機演示，在 60 秒內清楚介紹專案背景、遭遇的技術挑戰以及你採用的關鍵解決策略。

單元課程 進階 13 - 技術敘事

技術劇院：專業作品集與 Demo 敘事

任務 1：專業級系統架構圖 (Professional Architecture Lab)

角色扮演：技術文案架構師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 分層建模：將專案拆解為「實體層」、「韌體層」、「通訊層」與「前端層」。
  3. 視覺實現：使用 Mermaid 或繪圖工具完成架構圖。標註出每一層的數據傳遞格式 (JSON/Binary)。
  4. Push Final：將圖表嵌入 README 首頁並提交。確保這是讀者理解專案的「快速道路」。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 精確度：架構圖是否能準確反映硬體介面 (PWM/I2C) 與軟體層級的關聯？
  • [ ] 一致性：圖表中的命名與實際代碼中的模組劃分是否完全一致？

任務 2：動態性能證據集 (Dynamic Performance Portfolio)

角色扮演：性能數據分析師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 採集證據：採集 Dashboard 上的即時波形 GIF，展示 PID 的收斂或協議優化後的更新率。
  2. 對比實驗：在 README 中列出「優化前」與「優化後」的延遲毫秒值對比表。
  3. 視覺優化：為你的波形圖加入標記，指出哪一塊是「過彎震盪」，哪一塊是「平滑修正」。
  4. Push Final：提交具備「視覺說服力」的全方位專案證據集。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 證據可信度：收集的 GIF 或截圖是否清晰標註了測試條件與結果數據？
  • [ ] 表現力：README 的排版是否能讓觀看者快速抓住該專案最亮眼的技術規格？

任務 3：30 秒電梯簡報錄製 (Elevator Pitch Video Lab)

角色扮演：技術產品經理 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 腳本撰寫：採用「挑戰、掙扎、突破」模型撰寫 30-60 秒的技術亮點介紹文。
  2. 同步演示錄影：錄製子母畫面 (PiP) 影片。確保左側 UI 的數據變動與右側車子的動態完全同步。
  3. 成果發布：將 Demo 影片上傳或嵌入 README 最顯眼處，展現低延遲、高強度的控制實力。
  4. Push Final：完成作品集最終推送，結束單元任務。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 傳達力：口述內容是否緊扣專案的核心技術突破？有無展現出工程師的解決問題特質？
  • [ ] 展示完整度：Demo 影片是否涵蓋了平穩行駛與異常處理 (Fail-safe) 的雙重展示？

GitHub Classroom 詳細操作指南

本單元教學重點：培養「技術敘事力」。教導學員如何將艱澀的代碼轉化為外行也懂、內行驚嘆的視覺作品集。

  1. 重點 1：架構圖是第一語言 - 引導學員理解一張高質量的架構圖比 100 行註解更能說明代碼的素質。
  2. 重點 2：數據的視覺衝擊 - 強調「證據導向」：不要只說快，要展示「延遲縮短 30%」並輔以波形 GIF。
  3. 重點 3：作品集即資產 - 這是課程的總結，提醒學員這份 README 是他進入科技產業或名門大學的重要技術背書。
