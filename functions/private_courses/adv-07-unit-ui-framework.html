什麼是元件化設計 (Component-Based Design)？

元件化設計 (Component-Based Design) 是現代前端開發的基石思維。它的核心理念是：將複雜的使用者介面拆解為一系列獨立、可複用的元件，每個元件封裝自己的 HTML 結構、CSS 樣式與 JavaScript 邏輯。這就像用樂高積木蓋房子——每一塊積木都有明確的形狀與用途，組合在一起便能構成複雜的建築。

在物聯網 (IoT) 儀表板的情境中，你可能需要同時顯示馬達轉速儀表、電池電量條、藍牙連線狀態指示燈、搖桿控制面板等多種 UI 區塊。如果所有邏輯都寫在同一份檔案中，任何一處小改動都可能引發連鎖錯誤。元件化設計正是解決這類「義大利麵程式碼 (Spaghetti Code)」問題的關鍵方法論。

情境
想像你接手了一份超過 2000 行的 index.html 檔案。裡面混雜著燈光開關的 DOM 操作、空調溫度滑桿的事件監聽、窗簾百分比的計算邏輯，以及能源用量圖表的繪圖程式碼。當你想修改溫度滑桿的樣式時，卻意外觸發了燈光開關的 bug。這就是「低內聚、高耦合」程式碼的典型災難場景。元件化設計的目標，就是讓你再也不必面對這種困境。

Atomic Design：從原子到頁面

Brad Frost 提出的 Atomic Design 方法論，為元件化設計提供了一套清晰的層級架構。它借用化學的概念，將 UI 元件從最小單位（原子）逐步組合為完整頁面。這套分類方式特別適合物聯網儀表板的開發——你可以從最基本的按鈕與指示燈開始，一路組裝到完整的遙控介面。

層級 定義 IoT 儀表板範例
原子 (Atoms) 最小的 UI 單位，無法再拆分。 按鈕、LED 指示燈、數值標籤、輸入框
分子 (Molecules) 數個原子組成的功能群組。 搜尋列（輸入框＋按鈕）、帶標籤的滑桿
組織體 (Organisms) 由分子和原子構成的複雜 UI 區塊。 導覽列、遙測儀表盤面板、藍牙通訊控制面板
模板 (Templates) 定義頁面的骨架佈局，尚無真實資料。 儀表板的三欄 Grid 結構、行動版單欄排版
頁面 (Pages) 模板填入真實內容後的最終呈現。 即時馬達遙控介面、設備監控首頁

封裝：元件的自給自足

一個良好的元件必須做到封裝 (Encapsulation)：它的內部實作細節對外部完全隱藏。外部程式碼不需要知道元件內部使用了哪些 DOM 節點或 CSS 類別，只需透過定義好的介面（Props 與 Events）與之互動。以下是一個簡化的 BaseComponent 基類範例，展示元件封裝的基本骨架：

class BaseComponent {
  constructor(container, props = {}) {
    this.container = container;
    this.props = { ...props };
    this.render();
  }

  // 子類別必須覆寫此方法
  render() {
    throw new Error('render() must be implemented');
  }

  // 更新 Props 並重新渲染
  updateProps(newProps) {
    this.props = { ...this.props, ...newProps };
    this.render();
  }

  // 發送自定義事件
  emit(eventName, detail) {
    this.container.dispatchEvent(
      new CustomEvent(eventName, { detail, bubbles: true })
    );
  }
}

透過這個基類，所有子元件（如 Gauge、Joystick）只需繼承 BaseComponent 並實作自己的 render() 方法，即可獲得統一的 Props 管理與事件發送能力。這確保了整個元件體系的一致性與可預測性。

高內聚、低耦合 (High Cohesion, Low Coupling) 這是軟體工程中最重要的設計原則之一。高內聚意味著一個元件內部的所有程式碼都緊密服務於同一個目的（例如 Gauge.js 只負責數值顯示）；低耦合意味著元件之間的依賴關係降到最低，修改一個元件不會連帶影響其他元件。當你的每個元件都達到這兩項指標時，整個系統將變得容易理解、測試與維護。

資料傳遞：Props 與 Events

當介面被拆分為獨立的元件後，一個關鍵問題隨之而來：元件之間如何溝通？答案是遵循「單向資料流 (Unidirectional Data Flow)」原則——資料永遠只朝一個方向流動。父元件透過 Props 將資料向下傳遞給子元件，而子元件則透過 Events 將使用者互動或狀態變化向上通知父元件。

這個模式可以簡單記為：「Props 向下流、Events 向上冒」。它確保了資料來源的唯一性——任何一筆資料的「真實來源 (Single Source of Truth)」永遠只存在於父元件中，子元件僅負責接收並呈現。這種架構讓你的應用程式在規模擴大時依然容易推理與除錯。

特性 Props（屬性） Events（事件）
方向 父元件 → 子元件（向下） 子元件 → 父元件（向上）
用途 注入資料與配置（數值、標籤、顏色等） 通知互動結果（座標變更、按鈕點擊等）
可修改性 子元件不可直接修改 Props 父元件決定如何回應事件
IoT 範例 <Gauge value={85} label="電池" /> joystick:move → { x: 0.5, y: -0.3 }
類比 父母給孩子的指示與資源 孩子向父母回報現場狀況

透過 Props 驅動元件渲染

以一個數值儀表 Gauge 元件為例，父元件在初始化時傳入 label、value 與 color 三個 Props。當感測器回傳新數據時，父元件呼叫 updateProps() 即可自動觸發重新渲染，完全不需要手動操作 DOM：

// 父元件：初始化 Gauge 並傳入 Props
const batteryGauge = new Gauge(
  document.getElementById('battery-slot'),
  { label: '電池電量', value: 85, color: 'green' }
);

// 當感測器回傳新數據時，更新 Props
batteryGauge.updateProps({ value: 15, color: 'red' });
// Gauge 自動重新渲染，顯示警示紅色

透過 CustomEvent 發送事件

當子元件偵測到使用者互動（例如搖桿被拖動），它不應該直接呼叫業務邏輯，而是透過 CustomEvent 將資訊「冒泡」給父元件，由父元件決定後續行為。以下是搖桿元件發射事件的範例：

// Joystick.js — 子元件內部
handleMove(x, y) {
  this.emit('joystick:move', { x, y });
}

// app.js — 父元件監聯事件
joystickEl.addEventListener('joystick:move', (e) => {
  const { x, y } = e.detail;
  ble.send(packCommand(x, y)); // 業務邏輯只在父元件中
});
重要：禁止在子元件中放置業務邏輯 永遠不要在子元件（如 Joystick.js）內部撰寫 ble.send()、API 呼叫或資料庫寫入等業務邏輯。子元件的職責僅限於「視覺呈現」與「使用者互動偵測」。所有業務決策都應由父元件（或應用層）在接收到事件後處理。違反此原則會導致元件無法在不同專案中複用，並使測試變得困難。
提示：善用 CustomEvent 實現鬆耦合架構 CustomEvent 是瀏覽器原生的事件機制，不需要引入任何第三方函式庫。透過 bubbles: true 設定，事件會沿 DOM 樹向上冒泡，父元件可以在任意層級攔截處理。這種模式與 React 的回呼 Props 和 Vue 的 $emit 在概念上完全一致，掌握它等於掌握了所有主流框架的通訊基礎。

響應式佈局與遙測視覺化

物聯網儀表板必須在各種螢幕尺寸上都能正常運作——從實驗室的 27 吋桌上型螢幕，到口袋中的手機。CSS Grid 與 Flexbox 是實現響應式佈局的兩大核心工具。CSS Grid 負責定義整體的二維網格結構（列與欄），而 Flexbox 則適合處理單一方向的排列（例如按鈕列或導覽欄）。兩者搭配使用，可以輕鬆打造適配多種裝置的專業級儀表板。

除了佈局，「數據驅動 (Data-Driven)」的 UI 思維同樣關鍵。傳統做法中，開發者手動透過 document.getElementById() 逐一操作 DOM 節點來更新畫面。而在元件化架構中，我們追求的是讓 UI 成為資料狀態的鏡像——當狀態改變時，介面自動重新渲染，無需手動干預。

使用 CSS Grid 建構儀表板佈局

以下範例展示如何使用 CSS Grid 搭配 @media 查詢，讓同一組元件在桌機上顯示為三欄佈局，在平板上切換為兩欄，而在手機上自動堆疊為單欄：

/* dashboard.css */
.dashboard-grid {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
  gap: 16px;
  padding: 20px;
}

/* 平板：兩欄 */
@media (max-width: 1024px) {
  .dashboard-grid {
    grid-template-columns: repeat(2, 1fr);
  }
}

/* 手機：單欄堆疊 */
@media (max-width: 600px) {
  .dashboard-grid {
    grid-template-columns: 1fr;
  }
}
斷點名稱 寬度範圍 Grid 欄數 適用裝置
Desktop > 1024px 3 欄 桌上型電腦、大螢幕
Tablet 601px – 1024px 2 欄 平板、小型筆電
Mobile ≤ 600px 1 欄 手機

狀態驅動的 UI 切換

在 IoT 應用中，裝置的連線狀態會直接影響介面的視覺呈現。一種常見的做法是：當藍牙連線成功時，為 <body> 加上 .is-connected 類別，然後透過 CSS 變數讓所有子元件自動從「離線灰階」切換為「上線彩色」主題。這就是所謂的「狀態驅動 UI (State-Driven UI)」——UI 的外觀完全由資料狀態決定。

/* 預設離線狀態：灰階主題 */
:root {
  --gauge-color: #8a8886;
  --panel-bg: #f3f2f1;
  --status-text: 'Disconnected';
}

/* 連線成功：彩色主題 */
body.is-connected {
  --gauge-color: #107c10;
  --panel-bg: #dff6dd;
  --status-text: 'Connected';
}

/* JavaScript：切換連線狀態 */
document.body.classList.toggle('is-connected', isConnected);
元件生命週期：掛載與卸載 每個元件都有生命週期——它在何時被「掛載 (Mount)」到 DOM 中，又在何時被「卸載 (Unmount)」移除。理解生命週期至關重要，因為它決定了你何時應該初始化資源（例如建立 BLE 連線監聽器），以及何時必須清理資源以防止記憶體洩漏。在 React 中這對應 useEffect 的清理函數，在原生 JavaScript 中則需要你手動管理。
警告：務必在元件卸載時清除事件監聽器 如果你在元件掛載時註冊了 BLE 回呼函數（例如 ble.on('data', this.handleData)），卻忘記在元件卸載時呼叫 ble.off('data', this.handleData) 來移除它，這個監聽器會持續存在於記憶體中。每次元件重新掛載都會新增一個監聽器，最終導致嚴重的記憶體洩漏與重複執行問題。請務必在 destroy() 或 unmount() 方法中清理所有外部資源。

任務 1：通用原子組件 Gauge.js

目標：透過 Props 驅動視覺呈現，落實宣告式 UI 思維。

需求：

  • 繼承 BaseComponent，實作 render() 方法

導師指南：adv-07-unit-ui-framework (UI 框架與元件化設計)

針對 adv-07-unit-ui-framework (UI 框架與元件化設計) 單元，這是一個從「面段式編程」轉向「物件導向/組件化開發」的關鍵轉折點。它模仿了 React/Vue 等主流前端框架的核心哲學：封裝、複用、狀態驅動。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template Repo) 配置建議

理解組件化架構需要清晰的目錄規範與「基底類別」，建議範本包含：

  • 📂 components/BaseComponent.js：組件基類。定義統一的 render() 與 updateProps() 介面，讓學員繼承並擴展。
  • 📂 components/Gauge.js：待實作的數據顯示原子組件。
  • 📂 components/Joystick.js：待封裝的互動控制組件。
  • 📂 css/modules/：模組化樣式目錄。鼓勵學員將樣式按組件分類，避免樣式污染 (Global CSS Pollution)。
  • 📂 tests/component.test.js：自動化評分脚本。專門測試組件的「隔離性」——檢核改變一個組件的 Props 是否會錯誤地影響到另一個同類型組件。

任務 1：通用原子組件 (Atomic Data Component)

  • 目標：學會透過 Props 驅動視覺呈現，落實「數據為王」的宣告式 UI 思維。
  • Classroom 部署建議：
    • 渲染邏輯檢核： 
      // 學生需實作：根據 Props 反映狀態
render() {
    this.container.innerHTML = `<h3>${this.props.label}</h3>`;
    if (this.props.value < 20) this.container.classList.add('bg-red-500');
}
    • Autograding：初始化三個 Gauge 元件，檢核它們是否能同時顯示「不同」的數值與標籤。
    • 視覺驗證：要求提交截圖：當電池組件數值改為 15 時，該組件是否自動變為高品質的警示色。

任務 2：事件發射器 (Joystick Event Emitter)

  • 目標：實作強解耦 (Decoupling) 架構。組件只負責交互，業務邏輯交由父層處理。
  • Classroom 部署建議：
    • 關注點分離檢核：學員必須將 Canvas 繪圖邏輯封裝在組件內，並透過 CustomEvent 向外拋出座標資訊。

此單元的價值在於 「代碼的組織能力與可維護性」：

  • [ ] 封裝程度 (Encapsulation)：組件是否具備獨立的邏輯？如果換一個專案，這個 Gauge.js 是否能直接複製貼上後正常運作？
  • [ ] 通訊規範度：父子組件傳值是否嚴格遵守 Props In / Events Out 的原則？
