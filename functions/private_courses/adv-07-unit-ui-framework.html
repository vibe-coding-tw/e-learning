🧱 1. 什麼是元件化設計 (Component-Based Design)？ (25 min)

將複雜的網頁視為樂高積木。每個元件（如「馬達狀態列」）應完全封裝自己的 HTML 結構、CSS 樣式與狀態邏輯，實現「高內聚、低耦合」。

Design Pattern

  • 原子元件 (Atoms)：按鈕、輸入框、標籤。
  • 組件 (Organisms)：導覽欄、遙測儀表盤、通訊面板。

⚡ 3. 響應式佈局與遙測視覺化

使用 CSS Grid 與 Flexbox 打造「數據驅動 (Data-Driven)」的儀表板。當資料狀態 (State) 變更時，介面應自動響應，無需手動操作 DOM 節點。

Layout Efficiency

  • 單向更新流：告別 document.getElementById，擁抱狀態同步。
  • 響應式網格 (Responsive Grid)：在大螢幕顯示全儀表，在手機端自動切換為縱向控制欄。
  • 元件生命週期：理解元件何時「掛載 (Mount)」與「卸載」，防止藍牙回調造成的內存洩漏。

本單元的核心目標是從「流程式撰寫」轉向組件化開發思維。你將實作一套模仿 React/Vue 核心哲學的 UI 元件系統：封裝、複用、狀態驅動。

✅ 作業任務

任務 1：通用原子組件 Gauge.js

目標：透過 Props 驅動視覺呈現，落實宣告式 UI 思維。

需求：

  • 繼承 BaseComponent，實作 render() 方法
  • 根據 this.props.label 與 this.props.value 更新 DOM
  • 當 value < 20 時，自動套用警示樣式（例如背景變紅）
  • 多個 Gauge 實例之間狀態必須完全獨立

驗收方式：

  • [ ] 初始化三個 Gauge，各自顯示不同數值與標籤
  • [ ] 截圖：電池組件數值設為 15 時，該組件出現警示色

任務 2：搖桿事件發射器 Joystick.js

目標：實作強解耦架構。組件只負責交互，業務邏輯交由父層處理。

需求：

  • 在 Joystick.js 內封裝 Canvas 繪圖邏輯
  • 透過 CustomEvent 向外拋出座標資訊（事件名稱：joystick:move）
  • 禁止在 Joystick.js 內撰寫任何通訊代碼（如 ble.send()）

驗收方式：

  • [ ] 在 index.html 或 app.js（父層）監聽 joystick:move 事件並印出座標日誌
  • [ ] Code Review：Joystick.js 內不含任何業務邏輯

任務 3：響應式儀表板 dashboard.css

目標：利用 CSS Grid 與狀態標籤，實現多端適配的專業儀表板。

需求：

  • 使用 CSS Grid 完成儀表板佈局
  • 加入 @media 查詢：手機直式 1 欄、桌機橫式 3 欄
  • 藍牙連線成功時，為 body 加上 .is-connected 類別
  • 透過 CSS 變數讓所有組件自動切換主題（灰階 → 彩色）

驗收方式：

  • [ ] 截圖（手機直式排版）並貼至本 README 下方
  • [ ] 截圖（桌機橫式排版）並貼至本 README 下方

導師指南：adv-07-unit-ui-framework (UI 框架與元件化設計)

針對 adv-07-unit-ui-framework (UI 框架與元件化設計) 單元，這是一個從「面段式編程」轉向「物件導向/組件化開發」的關鍵轉折點。它模仿了 React/Vue 等主流前端框架的核心哲學：封裝、複用、狀態驅動。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template Repo) 配置建議

理解組件化架構需要清晰的目錄規範與「基底類別」，建議範本包含：

  • 📂 components/BaseComponent.js：組件基類。定義統一的 render() 與 updateProps() 介面，讓學員繼承並擴展。
  • 📂 components/Gauge.js：待實作的數據顯示原子組件。
  • 📂 components/Joystick.js：待封裝的互動控制組件。
  • 📂 css/modules/：模組化樣式目錄。鼓勵學員將樣式按組件分類，避免樣式污染 (Global CSS Pollution)。
  • 📂 tests/component.test.js：自動化評分脚本。專門測試組件的「隔離性」——檢核改變一個組件的 Props 是否會錯誤地影響到另一個同類型組件。

2. 作業任務部署細節

任務 1：通用原子組件 (Atomic Data Component)

  • 目標：學會透過 Props 驅動視覺呈現，落實「數據為王」的宣告式 UI 思維。
  • Classroom 部署建議：
    • 渲染邏輯檢核： 
      // 學生需實作：根據 Props 反映狀態
render() {
    this.container.innerHTML = `<h3>${this.props.label}</h3>`;
    if (this.props.value < 20) this.container.classList.add('bg-red-500');
}
    • Autograding：初始化三個 Gauge 元件，檢核它們是否能同時顯示「不同」的數值與標籤。
    • 視覺驗證：要求提交截圖：當電池組件數值改為 15 時，該組件是否自動變為高品質的警示色。

任務 2：事件發射器 (Joystick Event Emitter)

  • 目標：實作強解耦 (Decoupling) 架構。組件只負責交互，業務邏輯交由父層處理。
  • Classroom 部署建議：
    • 關注點分離檢核：學員必須將 Canvas 繪圖邏輯封裝在組件內，並透過 CustomEvent 向外拋出座標資訊。
    • PR 深度核核：這是一個陷阱題。如果導師在 Joystick.js 檔案中發現了 ble.send() 這種通訊代碼，則評定為「違反架構原則」需要重構。
    • 驗證方式：在 app.js (父層) 中監聽 joystick:move 事件並印出日誌。

任務 3：響應式佈局重構 (Responsive Dashboard Refactor)

  • 目標：利用 CSS Grid 與狀態標籤實現具備「專業感」的多端適配儀表板。
  • Classroom 部署建議：
    • 佈局技術檢核：學員需在 dashboard.css 中實作 @media 查詢，調整 Grid 欄位數量。
    • 狀態驅動佈局：當藍牙連線成功時，學員需給 body 加上 .is-connected，並讓所有子組件透過 CSS 變數自動切換主題（如：從灰階變為流光彩繪）。
    • 數據提交：要求在 README 中分別貼出「手機直式」與「電腦橫式」的儀表板排版結果。

3. 前端架構導師點評標準 (UI Architecture Benchmarks)

此單元的價值在於 「代碼的組織能力與可維護性」：

  • [ ] 封裝程度 (Encapsulation)：組件是否具備獨立的邏輯？如果換一個專案，這個 Gauge.js 是否能直接複製貼上後正常運作？
  • [ ] 通訊規範度：父子組件傳值是否嚴格遵守 Props In / Events Out 的原則？
  • [ ] 視覺執行力：響應式斷點 (Breakpoints) 設置是否合理？在 4K 螢幕與手機小螢幕上是否都能優雅閱讀遙測數據？
