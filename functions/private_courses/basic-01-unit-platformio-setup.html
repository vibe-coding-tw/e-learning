基礎 1 - 開發環境

VS Code + PlatformIO 環境搭建

從 Arduino IDE 畢業，邁向專業開發之路。IntelliSense、版本控制、多環境管理，一次滿足。

🚀 為什麼捨棄 Arduino IDE？ (15 min)

PlatformIO 讓你的開發體驗從「寫作業」進化為「做工程」。

✨

代碼補完 (IntelliSense)

自動校正拼字。輸入 `Serial.` 就會跳出所有可用函式，減少 Typo (拼寫錯誤)，寫程式不再依賴記憶。

📦

版本管理

專案級函式庫鎖定。避免發生「今天能跑，明天更新就壞掉」的悲劇，確保專案長期維護的穩定度。

🌍

多環境編譯

同一個專案，一套代碼。換個配置就能分別部署到不同開發板或不同上傳速率，靈活應對各種場景。

🛠️ 安裝與初始化 (25 min)

1

安裝 VS Code 本體

下載並安裝 Visual Studio Code。這是目前最受歡迎的通用型 IDE。

2

載入 PlatformIO IDE 套件

在 Extensions 中搜尋 "PlatformIO"。安裝後看到左側出現「外星人圖標 👽」即代表架設成功。

3

認識 platformio.ini

這是專案的「大腦」，管理板型、驅動速率與函式庫依賴。

Configuration Example
[env:esp32dev]
platform = espressif32
board = esp32dev
framework = arduino
monitor_speed = 115200
upload_speed = 921600

📂 專案結構巡禮 (20 min)

src/main.cpp

程式碼主戰場。注意：手動包含 #include <Arduino.h> 是 PlatformIO 的基本功。

lib/

存放本地私有函式庫。當程式碼超過 500 行時，建議開始拆解模組至此。

include/

放全域 Header (.h) 檔案。這與 src 配合建立嚴謹的 C++ 專案結構。

.pio/

編譯暫存區。Git 提交時應忽略此目錄，避免上傳無效的大型二進位檔。

💡 第一次編譯通常會需要下載 Toolchain，需保持網路通暢。

核心挑戰：從玩家變專家

建立專業的開發流，掌握當代嵌入式軟體管理的精髓。

🚢 60 min

【作業 1-1】

「首次編譯與上傳」

建立新專案並對齊開發板型號。將傳統 Arduino 代碼移入 main.cpp 並加上必要的標頭檔，完成首次燒錄驗證。

📚 60 min

【作業 1-2】

「自動化依賴管理」

練習透過設定檔安裝 Library。在 platformio.ini 加入 lib_deps 以引入 ArduinoJson，體驗 PIO 的強大同步力。

🎛️ 45 min

【作業 1-3】

「多環境上傳與調速」

設定 monitor_speed 解決顯示亂碼。嘗試建立不同 [env] 段落，區分 Debug 與 Release 的上傳速度配置。

VS Code 工程化開發環境實驗室

任務 1：新環境的首航 (First Voyage in PlatformIO)

角色扮演：韌體開發實習生 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 專案初始化：在 VS Code 中點選 PIO Home，建立全新的專案，選擇 ESP32 Dev Module 與 Arduino 框架。
  3. 代碼移植與修正：將傳統的 .ino 範例碼搬移到 src/main.cpp，手動在最上方加上 #include <Arduino.h>，解決 PlatformIO 的編譯要求。
  4. 驗證繳交：成功燒錄後，在 README.md 貼上 IntelliSense 正常運作（無紅字警告）的代碼截圖。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 編譯成功率：PIO 終端機是否顯示綠色的 [SUCCESS] 狀態並正確識別 COM 埠？
  • [ ] 專案結構掌握：是否理解 src/main.cpp 是唯一的主程式入口？

任務 2：函式庫管家 (Library Manager Implementation)

角色扮演：軟體套件管理員 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 依賴聲明實作：開啟專案根目錄的 platformio.ini，在 lib_deps 欄位加入 bblanchon/ArduinoJson @ ^6.21.0。
  2. 自動化安裝驗證：觸發重新編譯，觀察 PIO 如何自動從網路抓取函式庫並存放在專案目錄內。
  3. 技術報告：在 README 中比較 PIO 管家模式與傳統 Arduino「手動下載 zip」在專案協作上的優劣。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 配置導向思維：是否掌握了透過配置文件（而非 UI 點擊）來管理依賴的核心開發規範？
  • [ ] 版本控制意識：是否能說明 ^ 符號在語義化版本控制中的意義？

任務 3：多環境並行配置 (Multi-Environment Configuration)

角色扮演：自動化工程師 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 通訊參數優化：在 platformio.ini 中手動設定 monitor_speed = 115200 解決 Serial Monitor 的亂碼問題。
  2. 環境 Profile 分流：建立 [env:release] 段落，設定 upload_speed = 921600 以體會極速燒錄。
  3. 切換演練：練習使用 VS Code 下方狀態列環境選單，分別針對不同 Profile 進行編譯與上傳。
  4. 最終 Commit：整理優化後的 platformio.ini 配置文件並 Push 至 GitHub。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 環境抽象化理解：是否能透過實驗感受不同上傳速度配置對開發效率的具體影響？
  • [ ] 排障能力：是否能根據設備輸出自動調整相應的波特率參數以恢復數據可讀性？

GitHub Classroom 詳細操作指南

本單元教學重點：引導學員建立「代碼即配置」的專業工程思維。

  1. 第一 step：告別小打小鬧 - 強調 VS Code 強大的查閱與修補能力，讓學生體驗生產力工具的威力。
  2. 第二 step：依賴隔離 - 解釋為什麼「全域安裝」函式庫是個壞主意，並引導 PIO 的局部依賴管理。
  3. 第三 step：工作流自動化 - 透過 `platformio.ini` 的極速上傳設定，展現對開發流程進行微調與優化的可能性。
