從物件到 JSON

為什麼我們需要 JSON？學習如何將搖桿座標與按鈕狀態，轉換為 ESP32 能聽懂的通用語言。

🧪
1. 課前體驗：為什麼是 JSON？

想像你要傳送「向左轉，速度 100」給無人車。如果只是單純拼接字串，接收端會很難解析。 JSON 因應而生，它像是一個標準化的包裹，讓發送端 (Web) 與接收端 (ESP32) 都能精準理解。

⛔ 如果用字串拼接...

"L-100-MOVE" ← 順序錯了就完了！

✅ 使用 JSON 標準格式

{"dir": "L", "speed": 100} ← 清晰明確

2. 核心原理：JSON.stringify 與 ESP32 解析

JavaScript 端：序列化

JSON.stringify() 是我們最強大的助手。但要注意，它會過濾掉它不認識的東西！

console.js 🔴 Live Demo
const
data
= {
  speed: 100,
  func: () => {}, // 🙈 會消失
  val: undefined // 🙈 會消失
};

JSON.stringify(data)

↳ "{"speed":100}"

ESP32 端：ArduinoJson

微控制器沒有無限的記憶體。使用 StaticJsonDocument 來預先配置固定的記憶體空間，避免記憶體碎片化。

C++ / Arduino
String input ➔ "{\"cmd\":\"move\"}"
Deserialization ➔
StaticJsonDocument<200> doc;
deserializeJson(doc, input);
const char* cmd = doc["cmd"];

💡 key-Value 設計思維

為什麼不直接傳送 "前進"？因為中文編碼複雜且佔用頻寬。使用 {"cmd": "move", "val": 255} 能兼顧人類可讀性與機器解析效率。

目標： 原始封包太大了！請將 Key Name 簡化以減少字元數。

Length: 40

JSON 封包設計師

任務 1：把控制指令轉成標準 JSON (Serialize Command)

角色扮演：物聯網開發者 | 難度：⭐️⭐️ | 估計時長：40 min

🛠️ 執行技術流程 (Git Workflow)

  1. 接受作業：建立專屬 GitHub Classroom 倉庫，確認範本含有資料物件與輸出位置。
  2. 建立指令物件：定義包含方向、速度、時間戳等欄位的控制資料。
  3. 轉成 JSON：使用 JSON.stringify() 產生標準字串，並驗證輸出格式是否正確。
  4. 同步版本：將函式與測試結果寫進 README 後再 commit / push。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 格式正確：輸出的 JSON 是否符合標準語法，而不是手刻字串拼接？
  • [ ] 欄位完整：方向、速度與時間資訊是否清楚存在？
  • [ ] 序列化理解：是否知道哪些資料型別會在 stringify 時被忽略？

任務 2：幫封包瘦身以節省頻寬 (Compact Payload)

角色扮演：韌體通訊工程師 | 難度：⭐️⭐️ | 估計時長：40 min

🛠️ 執行技術流程 (Git Workflow)

  1. 設計縮寫：把長欄位改成短鍵名，例如 command → c、velocity → v。
  2. 比較長度：量測精簡前後字節數，確認確實有顯著縮短。
  3. 維持可解碼性：確保 ESP32 或接收端仍能按約定讀回正確資料。
  4. 提交協議：把縮寫對照表與示例封包一起推送到 GitHub。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 瘦身效果：封包長度是否真的有縮短，而不是只換名稱？
  • [ ] 映射清楚：縮寫與原始欄位意義是否交代清楚？
  • [ ] 可維護性：協議精簡後，是否仍保有團隊可讀性？

任務 3：規劃可擴充的巢狀控制資料 (Nested State)

角色扮演：嵌入式系統架構師 | 難度：⭐️⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 拆分資料責任：將搖桿、按鈕、系統狀態等資訊整理成清楚的巢狀結構。
  2. 控制深度：避免過深巢狀，讓資料既清楚又容易在 MCU 端解析。
  3. 驗證案例：用不同操作組合測試資料輸出是否穩定且可預測。
  4. 提交方案：把資料 schema、測試案例與範例輸出一併提交。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 結構合理：資料分層是否容易在前後端與 ESP32 間對齊？
  • [ ] 擴充彈性：之後新增欄位時，是否不需要整體推翻原結構？
  • [ ] 測試完整：是否有實例證明多種控制狀態都能正確組合？

數據格式化與 JSON 協議 教學引導指南

第一步：準備範本倉庫 (Template Repo)

  1. 上傳基礎專案框架與 protocol-helper.js 註解框架，並勾選 "Template repository"。

第二步：建立 Classroom 作業

  1. 建立作業並連結範本，啟用 "Enable feedback pull requests" 以便檢視協議設計。

  1. 學生領取作業，定義 JSON 協議邏輯並實作數據縮減（瘦身）策略。

  1. 進入 Feedback PR，針對學生縮寫 Key 的邏輯嚴謹度、巢狀結構的合理性給予評價。

B. 參考範例代碼 (Reference Example Code)

protocol-helper.js (封裝與縮寫)

// 基礎封裝 (Task 1)
function createPacket(btnId) {
    const data = {
        btn: btnId,
        ts: Date.now()
    };
    return JSON.stringify(data);
}

// 封包瘦身 (Task 2)
// 使用單字元 Key 可節省約 50% 頻寬
function createOptimizedPacket(cmd, val) {
    return JSON.stringify({ c: cmd, v: val });
}

// 巢狀結構 (Task 3)
const joyData = {
    joy: { x: 10, y: -5 },
    btn: { a: 1 }
};

C. 自動評分 (Autograding) 的設計

  • 字節檢查： 使用 Actions 檢查產生的 JSON 字串長度（優化挑戰要求 < 20 bytes）。
  • 語法檢查： 驗證學生產出的 JSON 是否包含正確的嵌套 Key (joy/btn)。
