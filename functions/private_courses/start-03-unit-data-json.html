數據格式化：
從物件到 JSON

為什麼我們需要 JSON？學習如何將搖桿座標與按鈕狀態，轉換為 ESP32 能聽懂的通用語言。

🧪
1. 課前體驗：為什麼是 JSON？

想像你要傳送「向左轉，速度 100」給無人車。如果只是單純拼接字串，接收端會很難解析。 JSON 因應而生，它像是一個標準化的包裹，讓發送端 (Web) 與接收端 (ESP32) 都能精準理解。

⛔ 如果用字串拼接...

"L-100-MOVE" ← 順序錯了就完了！

✅ 使用 JSON 標準格式

{"dir": "L", "speed": 100} ← 清晰明確

2. 核心原理：JSON.stringify 與 ESP32 解析

JavaScript 端：序列化

JSON.stringify() 是我們最強大的助手。但要注意，它會過濾掉它不認識的東西！

console.js 🔴 Live Demo
const
data
= {
  speed: 100,
  func: () => {}, // 🙈 會消失
  val: undefined // 🙈 會消失
};

JSON.stringify(data)

↳ "{"speed":100}"

ESP32 端：ArduinoJson

微控制器沒有無限的記憶體。使用 StaticJsonDocument 來預先配置固定的記憶體空間，避免記憶體碎片化。

C++ / Arduino
String input ➔ "{\"cmd\":\"move\"}"
Deserialization ➔
StaticJsonDocument<200> doc;
deserializeJson(doc, input);
const char* cmd = doc["cmd"];

💡 key-Value 設計思維

為什麼不直接傳送 "前進"？因為中文編碼複雜且佔用頻寬。使用 {"cmd": "move", "val": 255} 能兼顧人類可讀性與機器解析效率。

start-03-unit-data-json (數據格式化：從物件到 JSON)

🎯 作業目標

掌握 JSON 數據的序列化、通訊協定設計以及封包優化（瘦身）。在物聯網與前端開發中，「機機對話 (Machine-to-Machine)」的效率至關重要。本單元將帶領你學習如何設計專業的 JSON 通訊格式，並在有限的傳輸頻寬中極大化資訊含量。

🚀 如何開始

  1. 開啟 index.html：這是你的數據終端監控介面，預設包含一個「封包監視器」。
  2. 編輯 protocol-helper.js：你將在此實作核心的數據封裝邏輯。

📝 學生作業任務 (Student Assignment)

請依序完成以下任務，並在 protocol-helper.js 中實作對應函式：

1. 基礎序列化練習 (Basic Serialization)

  • 目標：將 JavaScript 物件轉化為標準 JSON 字串。掌握時間戳記 (Timestamp) 在控制封包中的角色。
  • 要求：實作 createPacket(btnId)，回傳包含指令 ID 與 Date.now() 的 JSON 字串。
  • 重點：嚴禁使用字串拼接，必須使用 JSON.stringify()。

2. 封包瘦身挑戰 (Compact Payload)

  • 目標：培養「頻寬意識」。理解通訊協定中的 Overhead，學習透過短 Key (Short Keys) 提升傳輸效率。
  • 要求：將原本冗長的 Key（如 command）改為單字元（如 c）。
  • 實驗：量測並紀錄優化前後的字串 Byte 數變化。

3. 巢狀結構設計 (Nested State Machine)

  • 目標：設計「有層次」的專業通訊。學習整合來自不同模組（搖桿 vs 按鈕）的狀態。
  • 要求：設計一個二層結構（如 {"joy":{x,y}, "btn":{a,b}}），方便後端或硬體（ESP32）分類提取。

📖 通訊協議字典 (Protocol Dictionary)

請在此紀錄你設計的縮寫對應表，這對團隊開發與硬體解析非常重要：

原始 Key 縮寫 Key 說明
command c 指令代碼
value v 數值 (0-255)
timestamp t 時間戳記 (Unix Epoch)

1. 範本程式庫 (Template Repo) 配置建議

JSON 數據作業的核心在於「序列化正確性、層級設計與傳輸優化」，範本應包含連線協定框架與格式檢查，建議包含：

  • 📂 index.html：數據終端監控介面。預設包含一個「封包監視器」，學員在開發時能即時看到 JSON.stringify() 轉化後的原始字串。
  • 📂 protocol-helper.js：封裝器實作區。學員在此實作 createPacket()、createCompactPacket() 等核心函式。預留 // TODO 引導學員處理時序與巢狀結構。
  • 📂 README.md：通訊協議說明書 (JSON Spec)。提供模板要求學員填寫協議字典（例如：c 代表 Command, v 代表 Value），並紀錄瘦身前後的 Byte 數差異。
  • 📂 tests/：封包規格測試。使用 Actions 或小型腳本自動驗證 JSON 語法是否合法，以及欄位名稱是否符合「瘦身挑戰」的字節限制。

2. 作業任務部署細節

任務 1：基礎指令序列化 (Serialize Command Lab)

  • 目標：學會使用 JSON.stringify() 將 JavaScript 物件標準化。掌握時間戳記 (Timestamp) 在控制封包中的角色。
  • Classroom 部署建議：
    • 指令封裝實作：建立 createPacket(btnId)，回傳包含指令 ID 與 Date.now() 的 JSON 字串。
    • 標準化驗證：嚴禁使用字串拼接。學員必須展示出對 JSON 語法（雙引號、大毀號）的正確理解。
    • 驗證要點：序列化完整性審核。生成的 JSON 是否能被 JSON.parse() 成功還原？時間戳是否能正常更新？

任務 2：封包瘦身優化 (Compact Payload Lab)

  • 目標：培養「頻寬意識」。理解通訊協定中的 Overhead，學習透過短 Key (Short Keys) 在不丟失資訊的前提下極大化傳輸效率。
  • Classroom 部署建議：
    • 協定精簡實作：將原本冗長的 Key (如 velocity) 改為單字元 (如 v)。
    • 長度對比實驗：量測並紀錄優化前後的字串長度變化（例如：從 40 bytes 縮短至 15 bytes）。
    • 驗證要點：瘦身後的封包是否依然具備可讀性（對應表是否詳盡）？長度是否真的達到了低於 20 位元的優化目標？

任務 3：擴充式巢狀結構設計 (Nested State Machine Lab)

  • 目標：設計「有層次」的專業通訊。學習如何在單一封包中同時處理來自不同模組（搖桿 vs 按鈕）的狀態。
  • Classroom 部署建議：
    • 巢狀物件佈局：設計一個結構（如 {"joy":{x,y}, "btn":{a,b}}），方便 ESP32 在解析時能分類提取。
    • 解析難度控制：提醒學員層級不宜過深，以免增加微處理器的解析負擔。
    • 驗證要點：資料分區是否邏輯清晰？能否同步提交多個輸入裝置的合成狀態而不產生衝突？

3. 通訊標準化與傳輸效能點評標準 (Standardization & Performance Standards)

此單元的價值在於 「對數組語彙的規範力與對頻寬資源的精算力」：

  • [ ] 語法合規性：代碼是否正確調用了 JSON 原生 API，避免了不穩定的手動拼接？
  • [ ] 封包經濟性：是否具備「用最少位元傳遞最大資訊量」的設計意識？
  • [ ] 結構可擴充性：設計的巢狀 Schema 在未來新增感測器數據時，是否具備良好的相容性？
