讀取、寫入與通知

📖 1. 讀取 (Read)：查詢

定義：客戶端主動索取數據。

Pull Mechanism

手機詢問時才回傳。節能但實時性低。

適合內容

韌體版本、設備 ID

頻率建議

按需發起

⚙️ Firmware Setup 

pChar->setValue("1.0.0");
// Device waits for
// Remote client to read.

✍️ 2. 寫入 (Write)：下令

手機推送資料。設計遙控指令的核心。

🛡️ With Response (Safety)

安全握手。收到後回傳 ACK。保證指令傳達。

⚡ No Response (Command)

戰鬥高速。只管送，不等待。適合搖桿座標。

ON_WRITE CALLBACK 
void onWrite(BLECharacteristic *pC) {
  std::string val = pC->getValue();
  if (val == "GO") {
    // Trigger Motor logic immediately
    drive(255);
  }
}

🔔 3. 通知 (Notify)：推播

訂閱機制：手機訂閱。數據變動即主動推播。

即時監控

  • 超音波距離
  • 陀螺儀角度
  • 主動警報
  • 循跡誤差值

Efficiency King

"手機不再需要輪詢，電力消耗將大幅下降。"

pCharacteristic->notify();

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 混合配置：建立學號 (Read)、LED (Write) 與秒計數 (Notify)。
  3. 互動測試：控制開關，觀測秒計數自動跳動。
  4. 最終 Commit：整理對應表至 README。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 屬性精確：正確區分通訊權限？
  • [ ] 推播有效：Notify 主動發起而非手機輪詢？

任務 2：「延遲挑戰」：安全 vs. 快速寫入 (Latency Duel)

角色扮演：分析師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 壓力測試：手機每秒送 50 筆座標。
  2. 模式組測：切換 WRITE 與 WRITE_NR 觀察延遲。
  3. 心得報告：說明 ACK 在高頻場景影響。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 數據對比：量化描述高頻下模式穩定度？
  • [ ] 決策力：理解控制場景選用 NR 合適性。

任務 3：「智慧推播」：條件式 Notify 優化 (Smart Notify)

角色扮演：設計師 | 難度：⭐️⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 閾值實作：變動超過 10% 才 notify。
  2. 封包監測：觀察靜置與搖動狀態頻率。
  3. 效益分析：推導省電、寬度占用貢獻。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 演算法正確：有效排除雜訊且不遺漏重大變動？
  • [ ] 優化意識：展現智慧節能通訊設計。

GitHub Classroom 詳細操作指南

本單元教學重點：從「可通訊」到「高效通訊」的配置。

  1. 重點 1：適才適所 - 根據類型選擇 Read 或 Notify。
  2. 重點 2：延遲權衡 - 感受到 ACK 在高頻下的效能瓶頸。
  3. 重點 3：變動智慧 - 只傳遞意義數據，達成節能。
