✍️ 2. 寫入 (Write)：下令

手機推送資料。設計遙控指令的核心。

🛡️ With Response (Safety)

安全握手。收到後回傳 ACK。保證指令傳達。

⚡ No Response (Command)

戰鬥高速。只管送，不等待。適合搖桿座標。

ON_WRITE CALLBACK 
void onWrite(BLECharacteristic *pC) {
  std::string val = pC->getValue();
  if (val == "GO") {
    // Trigger Motor logic immediately
    drive(255);
  }
}

課程單元 基礎 5 - 藍牙進階

BLE 讀取、寫入與通知 (Properties & Smart Notify)

歡迎來到 basic-05-unit-ble-properties 單元！在本課程中，你將學習如何從一名「能通訊」的開發者進階為「通訊架構設計師」。

本單元的重點在於學習 GATT 屬性 (Properties) 的選擇，並透過 「智慧推播 (Smart Notify)」 實作工業級的節能數據傳輸策略。

🎯 學習目標

  1. 掌握 GATT 權限配置：實作 READ、WRITE 與 NOTIFY 三種核心屬性。
  2. 優化通訊延遲：體會「帶回應寫入 (With Response)」與「不帶回應寫入 (No Response)」的性能差異。
  3. 實作智慧推播：開發閾值過濾演算法 (Threshold Filter)，達成節能並減少無線電頻段佔用。

🛠️ 任務說明

任務 1：混合儀表板實務 (Hybrid Dashboard)

你需要在 src/main.cpp 中實作三個具備不同屬性的特徵值：

  • ID 特徵值 (READ)：顯示你的學號。
  • 執行器特徵值 (WRITE)：接收來自手機的控制指令。
  • 感測器特徵值 (NOTIFY)：自動推播模擬感測器數值。

提示：務必正確配置 CCCDescriptor (0x2902)，否則手機端將無法接收通知。

任務 2：延遲挑戰 (Latency Duel)

測試並比較「帶回應寫入」與「不帶回應寫入」的差異。

  • 觀測在壓力測試下，手機端發送指令是否會因為等待 ACK 而造成阻塞？
  • 請將實驗數據記錄於 docs/Latency_Study.md。

任務 3：智慧推播優化 (Smart Notify)

實作「按需傳輸」邏輯。

  • 規則：原本每 100ms 盲目推播一次數值，請改為「當前數值與上一次發送值差異超過 10%」才執行 notify()。
  • 目標：在數值變動劇烈時維持即時性，在數值靜止時節省電力。
  • 挑戰：請正確處理 lastSentValue 的更新邏輯，避免「慢速飄移 (Drift)」被略過。

📁 程式庫結構

  • 📂 src/main.cpp：核心程式實作區。
  • 📂 include/gatt_map.h：定義 Service 與 Characteristic 的 UUID。
  • 📂 docs/Latency_Study.md：實驗報告與心得填寫。
  • 📂 assets/packet_log.txt：智慧推播開啟前後的數據量對比紀錄。

針對 basic-05-unit-ble-properties (BLE 讀取、寫入與通知) 單元，這是學員從「能通訊」進階到「通訊架構設計師」的關鍵課程。它要求學員具備 通訊協議優化 (Protocol Optimization) 的思維，學習如何針對不同特性的數據（如靜態的版本號 vs. 高頻的感測器流）選擇最合適的 GATT 屬性，並透過 「條件式推播 (Smart Notify)」 實現工業級的節能效果。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template 程式庫) 配置建議

BLE 屬性作業的核心在於「權限配置」與「數據流設計」，範本應包含標準的 GATT 服務架構，建議包含：

  • 📂 src/main.cpp：特徵值屬性實作區。學員需在此實作三個具備不同屬性的特徵值：ID_Characteristic (READ), Actuator_Characteristic (WRITE), Sensor_Characteristic (NOTIFY)。
  • 📂 include/ble_protocol.h：協議定義中心。供學員定義自定義 UUID，並配置 CCCDescriptor (0x2902) 以支援移動端的訂閱功能。
  • 📂 README.md：效能報告與對比。提供表格範本讓學員紀錄：「Write 有無回應 (ACK) 的延遲差異」以及「智慧推播前後的封包數量統計」。
  • 📂 tests/latency_test.py (加分項)：提供一個簡單的 Python/Node.js 腳本，協助學員進行每秒 50 次的高頻指令壓力測試，量化各屬性的性能極限。

2. 作業任務部署細節

任務 1：「混合儀表板」屬性交互實務 (Hybrid Dashboard Lab)

  • 目標：掌握 GATT 服務與特徵值的讀、寫、通知三種核心權限配置，建立「數據權限」意識。
  • Classroom 部署建議：
    • 三位一配置：建立學號 (Read-only)、LED 控制 (Write-only) 與系統運行秒數 (Notify-only) 三個特徵值。
    • 訂閱驗證：學員需在手機 APP (nRF Connect) 中成功開啟 Notify 訂閱，並看到秒數自動跳動，而非透過手動下拉更新。
    • 驗證要點：README 中需列出三個特徵值的 UUID 與對應的屬性標誌 (Flags)。

任務 2：「延遲挑戰」安全 vs. 快速寫入 (Latency Duel Lab)

  • 目標：體會「帶回應寫入 (With Response)」與「不帶回應寫入 (No Response)」在實時控制（如遙控車搖桿）中的效能差異。
  • Classroom 部署建議：
    • 壓力測試壓力感：模擬每秒發送 50 組座標指令。
    • 延遲與阻塞分析：觀測在「With Response」模式下，手機端是否因等待 ACK 而導致發送阻塞？切換到「No Response」後延遲是否顯著下降？
    • 驗證要點：在報告中說明：對於「即時遙控」與「韌體更新」，分別應選用哪種寫入模式？

任務 3：「智慧推播」條件式 Notify 優化 (Smart Notify Lab)

  • 目標：實作閾值過濾演算法 (Threshold Filter)，實現「按需傳輸」，這是低功耗藍牙設備生存的靈魂。
  • Classroom 部署建議：
    • 變動觸發機制：原本每 100ms 盲目推播一次模擬值 (Analog Value) 改為「當前數值與上一次發送值差異超過 10%」才執行 notify()。
    • 節能數據量化：記錄在數值靜止時與劇烈變動時，每分鐘發送的封包數量差異。
    • 驗證要點：關鍵技術審查。學員是否正確儲存了 lastSentValue？若僅與 previousSample 比較會造成「慢速飄移 (Drift)」被略過的錯誤，這是一個專業的工程細節點評。

3. 通訊架構與效能優化點評標準 (Communication Architecture Standards)

此單元的價值在於 「對通訊帶寬的尊重與對數據類型精確匹配的能力」：

  • [ ] CCCDescriptor 配置完整性：是否正確配置了 0x2902 描述符？（這是 Notify 運作的工業標準）。
  • [ ] 場景匹配精確度：是否理解「不具備回應的寫入 (Write NR)」在高頻控制中的絕對優勢？
  • [ ] 智慧推播演算法：閾值過濾邏輯是否嚴謹？是否成功優化了無線電頻道的佔用？
