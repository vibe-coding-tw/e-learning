簡介

GATT 特徵值的三種核心屬性——Read、Write、Notify——決定了數據的流向與觸發方式。 本單元將帶你理解「主動查詢」與「被動推播」的效能差異，學會為不同類型的數據（靜態 ID、即時指令、高頻感測） 配置最合適的屬性，並實作智慧推播演算法來大幅節省無線電頻寬與電力消耗。

情境
你的無人車遙測系統每 100ms 盲目推播一次感測器數值，即使車輛靜止不動，手機仍瘋狂接收封包，電池在半小時內耗盡。 同時，搖桿控制因為使用了「帶回應寫入」而出現明顯延遲——你需要為不同場景選擇正確的 BLE 屬性策略。

讀取 (Read)：主動查詢機制

Read 是 BLE 中最基本的數據獲取方式——由 Client（手機）主動向 Server（ESP32）發起請求， Server 收到後回傳當前的 Characteristic 值。這是一種典型的「拉取（Pull）」模式， 只有在 Client 詢問時才會產生數據傳輸。

Read 操作是同步且有確認的：Client 發送 Read Request，Server 回傳 Read Response， 整個過程在一次連線事件中完成。這保證了 Client 一定能獲得數據，但也意味著每次查詢都需要一個完整的來回。

Read 的適用場景

Read 屬性最適合以下類型的數據：

數據類型 特性 範例
靜態識別資訊 幾乎不變，查詢一次即可 韌體版本、設備序號、學號
低頻狀態 偶爾變動，無即時性需求 電池電量（每分鐘查一次足矣）
配置參數 用戶按需查閱當前設定 當前運作模式、閾值設定
備註 Read 操作不會產生任何「推播」行為——如果 Server 端的值發生了變化，Client 不會自動收到通知，必須再次主動發起 Read 才能獲得新值。若需要即時更新，應使用 Notify 屬性。

ESP32 實作 Read 特徵值

實作 Read 特徵值非常簡單——你只需要建立帶有 PROPERTY_READ 的 Characteristic， 並透過 setValue() 設定其值。當手機發起 Read 請求時，BLE 協定棧會自動回傳該值：

// 建立 Read-only 特徵值（例如：設備學號）
BLECharacteristic *pIDChar = pService->createCharacteristic(
  ID_CHAR_UUID,
  BLECharacteristic::PROPERTY_READ
);

// 設定固定值（學號）
pIDChar->setValue("Student-07");

// 動態值範例：回傳 Uptime 秒數
BLECharacteristic *pUptimeChar = pService->createCharacteristic(
  UPTIME_CHAR_UUID,
  BLECharacteristic::PROPERTY_READ
);

// 在 loop() 中持續更新值
void loop() {
  String uptime = String(millis() / 1000);
  pUptimeChar->setValue(uptime.c_str());
  delay(1000);
}

Read vs. Notify：何時用哪個？

初學者常見的疑問是：既然 Notify 可以主動推播，為何還需要 Read？答案在於效率與場景匹配：

  • 如果 Client 只需要偶爾查詢一次（如開機讀取版本號），使用 Read 比維持 Notify 訂閱更節省資源。
  • Read 不需要 BLE2902 描述符，實作更簡單。
  • Read 保證每次都能取得最新值，而 Notify 可能因為錯過封包而遺漏更新。
提示 最佳實踐是為同一個感測 Characteristic 同時配置 Read + Notify 屬性。平時由 Notify 推播即時數據，但 Client 也能隨時 Read 獲取當前值——這在 App 重連或初始化時特別有用。

寫入 (Write)：控制指令下達

Write 是 Client（手機）向 Server（ESP32）傳送數據的方式——在遙控車應用中， 這就是手機下達「前進」「停止」「轉向」等控制指令的核心機制。Write 屬性是實現「遠端控制」的基礎。

BLE 規範定義了兩種寫入模式，它們在可靠性與延遲之間存在取捨，開發者必須根據應用場景選擇最合適的模式。

兩種寫入模式比較

模式 Write With Response Write Without Response (Write Command)
確認機制 Server 回傳 ACK 確認收到 Fire-and-forget，不等確認
可靠性 高（保證送達） 較低（可能丟失）
延遲 較高（需等待 ACK） 極低（發出即完成）
吞吐量 受限（每次需 2 個封包） 高（每次僅 1 個封包）
適用場景 韌體更新、關鍵模式切換 搖桿座標、連續動作指令
警告 對於即時遙控（如搖桿操控無人車），若誤用 Write With Response，每秒只能發送約 10–15 次指令（因為每次都要等 ACK）。改用 Write Without Response 後，可輕鬆達到每秒 50+ 次的更新頻率。

onWrite 回呼實作

當手機端寫入數據時，ESP32 透過 BLECharacteristicCallbacks 的 onWrite 方法接收指令。 你需要在回呼中解析收到的 Byte Array，並轉換為有意義的動作：

class MotorCallbacks : public BLECharacteristicCallbacks {
  void onWrite(BLECharacteristic *pCharacteristic) {
    std::string value = pCharacteristic->getValue();

    if (value.length() > 0) {
      if (value == "GO") {
        digitalWrite(LED_PIN, HIGH);
        motorDrive(255);
      } else if (value == "STOP") {
        digitalWrite(LED_PIN, LOW);
        motorDrive(0);
      }
    }
  }
};

// 掛載回呼
pMotorChar->setCallbacks(new MotorCallbacks());

數據編解碼注意事項

BLE 傳輸的本質是原始位元組陣列（Byte Array）。當手機端發送字串 "GO" 時， ESP32 收到的是 ASCII 編碼 [0x47, 0x4F]。在處理數據時需注意：

  • 使用 getValue() 取得的是 std::string，可直接與字串比較
  • 若傳送數值型數據（如速度 0–255），建議直接傳送單一 Byte 而非字串，節省封包空間
  • 務必檢查 value.length() 避免處理空資料造成未定義行為
提示 設計指令協議時，建議保持簡潔明確。例如使用固定長度的命令格式 [CMD_ID][PARAM1][PARAM2]，這樣解析邏輯更穩健，也更容易擴充新指令。

通知 (Notify)：即時推播機制

Notify 是 BLE 中最強大的數據傳輸方式——由 Server（ESP32）在數據變化時主動推播給 Client（手機）， 不需要 Client 反覆詢問。這是一種典型的「推送（Push）」模式，也是實現即時遙測監控的靈魂機制。

與 Read 的「手機問一次、設備答一次」不同，Notify 讓手機訂閱（Subscribe）特定 Characteristic 後， 只要 Server 端的值發生變化，就會自動將新值推送到手機——無需手機再次發起請求。

Notify 的運作流程

1

Client 訂閱通知

手機向 BLE2902 描述符寫入 0x0001，表示「我要訂閱此 Characteristic 的通知」。在 nRF Connect 中，這就是點擊向下箭頭圖示。

2

Server 更新值並推播

ESP32 透過 setValue() 更新 Characteristic 的值，然後呼叫 notify() 將新值推送給已訂閱的 Client。

3

Client 接收更新

手機的 App 在回呼函數中收到新值，即時更新 UI 顯示。整個過程無需手機輪詢。

BLE2902 描述符：訂閱的鑰匙

BLE2902 是Client Characteristic Configuration Descriptor (CCCD) 的簡稱， 它是 Notify/Indicate 機制的核心基礎設施。沒有掛載 BLE2902 的 Characteristic，即使設定了 NOTIFY 屬性，手機也無法訂閱它。

// 建立 Notify 特徵值
BLECharacteristic *pSensorChar = pService->createCharacteristic(
  SENSOR_CHAR_UUID,
  BLECharacteristic::PROPERTY_NOTIFY
);

// 關鍵！必須掛載 BLE2902 描述符
pSensorChar->addDescriptor(new BLE2902());

// 在 loop() 中定時推播感測數據
void loop() {
  if (deviceConnected) {
    int sensorValue = analogRead(SENSOR_PIN);
    String data = String(sensorValue);
    pSensorChar->setValue(data.c_str());
    pSensorChar->notify();
    delay(500); // 每 500ms 推播一次
  }
}
警告 忘記加入 pCharacteristic->addDescriptor(new BLE2902()) 是 BLE 開發中排名第一的常見錯誤。症狀是：nRF Connect 中 Notify 按鈕為灰色（不可點擊），或點擊後無任何反應。請務必在每個 Notify/Indicate 特徵值上都掛載此描述符。

智慧推播：閾值過濾策略

盲目地以固定間隔推播數據（如每 100ms 發送一次）會造成大量無效傳輸——當感測器數值沒有變化時，重複發送相同的值只是浪費電力與頻寬。 工業級的做法是實作閾值過濾（Threshold Filter）：

int lastSentValue = 0;
const float THRESHOLD = 0.10; // 變動 10% 才推播

void loop() {
  int current = analogRead(SENSOR_PIN);
  float change = abs(current - lastSentValue) / (float)max(lastSentValue, 1);

  if (change >= THRESHOLD) {
    pSensorChar->setValue(String(current).c_str());
    pSensorChar->notify();
    lastSentValue = current; // 更新基準值
  }
  delay(100);
}
提示 注意 lastSentValue 必須記錄的是上一次實際發送出去的值，而非上一次取樣的值。若改用「與上一次取樣比較」， 會產生「慢速飄移（Drift）」問題——每次微小變動都不超過閾值，但累積起來可能偏離很遠卻從未通知 Client。

BLE 讀取、寫入與通知 (Properties & Smart Notify)

歡迎來到 basic-05-unit-ble-properties 單元！在本課程中，你將學習如何從一名「能通訊」的開發者進階為「通訊架構設計師」。

🎯 學習目標

  1. 掌握 GATT 權限配置：實作 READ、WRITE 與 NOTIFY 三種核心屬性。
  2. 優化通訊延遲：體會「帶回應寫入 (With Response)」與「不帶回應寫入 (No Response)」的性能差異。
  3. 實作智慧推播：開發閾值過濾演算法 (Threshold Filter)，達成節能並減少無線電頻段佔用。

🛠️ 任務說明

任務 1：混合儀表板實務 (Hybrid Dashboard)

你需要在 src/main.cpp 中實作三個具備不同屬性的特徵值：

  • ID 特徵值 (READ)：顯示你的學號。
  • 執行器特徵值 (WRITE)：接收來自手機的控制指令。
  • 感測器特徵值 (NOTIFY)：自動推播模擬感測器數值。

提示：務必正確配置 CCCDescriptor (0x2902)，否則手機端將無法接收通知。

任務 2：延遲挑戰 (Latency Duel)

測試並比較「帶回應寫入」與「不帶回應寫入」的差異。

  • 觀測在壓力測試下，手機端發送指令是否會因為等待 ACK 而造成阻塞？
  • 請將實驗數據記錄於 docs/Latency_Study.md。

任務 3：智慧推播優化 (Smart Notify)

實作「按需傳輸」邏輯。

  • 規則：原本每 100ms 盲目推播一次數值，請改為「當前數值與上一次發送值差異超過 10%」才執行 notify()。
  • 目標：在數值變動劇烈時維持即時性，在數值靜止時節省電力。
  • 挑戰：請正確處理 lastSentValue 的更新邏輯，避免「慢速飄移 (Drift)」被略過。

  • Classroom 部署建議：
    • 三位一配置：建立學號 (Read-only)、LED 控制 (Write-only) 與系統運行秒數 (Notify-only) 三個特徵值。
    • 訂閱驗證：學員需在手機 APP (nRF Connect) 中成功開啟 Notify 訂閱，並看到秒數自動跳動，而非透過手動下拉更新。
    • 驗證要點：README 中需列出三個特徵值的 UUID 與對應的屬性標誌 (Flags)。

任務 2：「延遲挑戰」安全 vs. 快速寫入 (Latency Duel Lab)

  • 目標：體會「帶回應寫入 (With Response)」與「不帶回應寫入 (No Response)」在實時控制（如遙控車搖桿）中的效能差異。
  • Classroom 部署建議：
    • 壓力測試壓力感：模擬每秒發送 50 組座標指令。
    • 延遲與阻塞分析：觀測在「With Response」模式下，手機端是否因等待 ACK 而導致發送阻塞？切換到「No Response」後延遲是否顯著下降？
    • 驗證要點：在報告中說明：對於「即時遙控」與「韌體更新」，分別應選用哪種寫入模式？

任務 3：「智慧推播」條件式 Notify 優化 (Smart Notify Lab)

  • 目標：實作閾值過濾演算法 (Threshold Filter)，實現「按需傳輸」，這是低功耗藍牙設備生存的靈魂。
  • Classroom 部署建議：
    • 變動觸發機制：原本每 100ms 盲目推播一次模擬值 (Analog Value) 改為「當前數值與上一次發送值差異超過 10%」才執行 notify()。
    • 節能數據量化：記錄在數值靜止時與劇烈變動時，每分鐘發送的封包數量差異。
    • 驗證要點：關鍵技術審查。學員是否正確儲存了 lastSentValue？若僅與 previousSample 比較會造成「慢速飄移 (Drift)」被略過的錯誤，這是一個專業的工程細節點評。

3. 通訊架構與效能優化點評標準 (Communication Architecture Standards)

此單元的價值在於 「對通訊帶寬的尊重與對數據類型精確匹配的能力」：

  • [ ] CCCDescriptor 配置完整性：是否正確配置了 0x2902 描述符？（這是 Notify 運作的工業標準）。
  • [ ] 場景匹配精確度：是否理解「不具備回應的寫入 (Write NR)」在高頻控制中的絕對優勢？
  • [ ] 智慧推播演算法：閾值過濾邏輯是否嚴謹？是否成功優化了無線電頻道的佔用？
