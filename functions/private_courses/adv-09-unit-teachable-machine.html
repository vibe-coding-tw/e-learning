核心實驗室：建模與指令整合

透過 Git Workflow 建立手勢識別模型。學習如何在真實場景中對抗誤判，並將 AI 的判斷結果無縫轉化為機器人指令。

🏢 60 min

【作業 2-1】

「場景鲁棒性加強」

手邊採集 3 種手勢。特別建立「人影噪音」類別。在教室內進行模型驗證，確保在有人走動的情況下，置信度依然穩定保持在臨界點以上。

🛰️ 60 min

【作業 2-2】

「模型橋接轉發」

在 BLE 專案中導入 tf.js 權重模型。撰寫轉發邏輯：唯有當特定標籤機率 $> 0.9$ 且連續出現 3 幀時，才正式對硬體發出控制包。

⚖️ 45 min

【作業 2-3】

「失效保護機制」

模擬極端光線場景。實作 lowConfidenceStop 邏輯：當所有標籤機率均低於 0.5 時，強制推入安全狀態。量化分析從辨識到執行指令的端到端延遲。

單元課程 進階 9 - 模型實踐

建模實驗室：從數據標記到自動化駕駛

任務 1：自定義手勢建模 (Custom Gesture Training)

角色扮演：AI 資料訓練師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 標記採集：定義「Background」、「Left」、「Right」、「Stop」四個類別。每個類別至少採集 100 個樣本。
  3. 模型優化：測試不同的紀元 (Epochs)。在 README 中截圖 Loss 曲線圖。
  4. 鏈接分發：導出 Teachable Machine 雲端模型連結，並寫入 README 備查。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 資料庫多樣性：是否有在採集時改變光線與手勢距離？背景是否包含可能干擾的靜止物體？
  • [ ] 過擬合檢查：手勢背景是否過於單一？（避免 AI 認背景不認手）。

任務 2：Web AI 驅動程式 (AI-BLE Component)

角色扮演：視覺邏輯開發者 | 難度：⭐️⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 權重加載：實作 tmImage.load() 載入任務一導出的模型。
  2. 非同步預測：在 js/ai_logic.js 建立預測循環。設置 Prediction Throttle 避免過度佔用 CPU。
  3. 信心閘門 (Confidence Gate)：實作一個邏輯，只有當平均置信度持盈 3 幀且大於 0.9 時才發送 BLE 封包。
  4. Push Final：提交具備 AI 動態控制能力的前端遙控介面。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 執行效能：網頁預覽在加載模型後，FPS 是否依然流暢？是否有做適當的連發頻率限制？
  • [ ] 控制穩定性：指令發送是否具備「防閃爍 (Debounce)」保護，避免車體頻繁一頓一頓的動作？

任務 3：全系統負載測試 (System Stress Analysis)

角色扮演：安全測試工程師 | 難度：⭐️⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 主動干擾測試：請同學在鏡頭後晃動。觀察 AI 是否會產生誤觸發。
  2. 失效安全性 (Fail-safe)：實作當 tm_result.length < 0.5 時的主動停機。
  3. 延遲量化：紀錄從「手勢發生」到「車體連動」的體感總延遲時間。寫入報告分析延遲分佈。
  4. Push Final：提交具備工業級安全考量的 AI 控制系統。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 安全意識：是否實作了針對「未知狀態」的主動防護措施？
  • [ ] 性能洞察力：報告中對「推論延遲」與「通訊延遲」是否有清晰的區分與紀錄？

GitHub Classroom 詳細操作指南

本單元教學重點：讓學生跨越「代碼控制」進入「模型驅動」。強調機器學習是基於機率的面試，而非基於邏輯的斷言。

  1. 重點 1：垃圾進，垃圾出 (GIGO) - 向學員證明採集資料時的一時偷懶，會換來實測時的無限痛苦。
  2. 重點 2：從雲端到瀏覽器 - 解釋 tf.js 如何將原本巨大的深度學習模型序列化為網頁可解讀的權重文件。
  3. 重點 3：信心控制論 - 教導學員：好的工程師不只看類別，更看「機率」。99% 的信心值得信賴，51% 的信心必須停車。
