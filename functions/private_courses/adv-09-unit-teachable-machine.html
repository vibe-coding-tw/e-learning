快速建模 (Teachable Machine) 作業指南

歡迎來到進階 09 單元：快速建模！本單元將帶領你跨越「寫代碼控制」，進入「模型驅動控制」的領域。透過機器學習中的 「垃圾進，垃圾出 (GIGO)」 原理，你將學會如何訓練一個有用的模型，並實作嚴謹的「置信度閘門」來確保硬體安全。

作業目標

  1. 掌握數據標記：了解數據標記的多樣性與模型優化。
  2. 實現穩定控制：實作穩定且低延遲的 AI 轉導控制邏輯。
  3. 設計安全邊界：針對未知情境實作失效保護（Fail-safe）機制。

任務說明

任務 1：自定義手勢建模 (Custom Gesture Training)

  1. 建模規範：請在 Teachable Machine 採集四個類別：「Background (背景)」、「Left」、「Right」、「Stop」。
  2. 超參數紀錄：請在下方「學習紀錄」區塊中截圖並貼上訓練後的 Loss 曲線，並記錄你所使用的 Epochs 數量。
  3. 數據提交：請導出模型，並將模型權重文件（例如 model.json, weights.bin 等）放入專案相對應的資料夾中。

任務 2：AI-BLE 驅動程式 (AI-BLE Component)

  1. 置信度閘門實作：在程式中實作高信心連發保護。只有當某個類別的預測機率 大於 0.9 且連續出現 3 幀 時，才發送對應的藍牙指令至機器人。這能有效過濾雜訊，避免機器人錯誤動作。
  2. 推論速率限制 (Throttling)：請設定並限制你的推論頻率。建議控制在 10 ~ 15 FPS，以避免推論次數過高導致電腦卡頓。

任務 3：失效保護與壓力測試 (Fail-safe Strategy)

  1. 失效安全性實作：請實作安全保護邏輯：當所有標籤的最高機率都低於 0.5 時（代表 AI 看不懂現在的情況），系統必須自動發送 STOP 指令。
  2. 性能報告：請在下方「學習紀錄」區塊量化記錄從「手部動作發生」到「車體反應」的總體延遲，並簡單分析其中「推論時間」與「通訊時間」的佔比。

針對 adv-09-unit-teachable-machine (快速建模) 單元，這是一個讓學員跨越「寫代碼控制」進入「模型驅動控制」的里程碑。它強調了機器學習中的 「垃圾進，垃圾出 (GIGO)」 原理，並要求學員實作嚴謹的「置信度閘門」來確保硬體安全。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template) 配置建議

AI 整合需要處理非同步載入模型與連續推論流，建議範本包含：

  • 📂 js/ai_controller.js：AI 驅動核心。預置 TensorFlow.js 與 Teachable Machine 函式庫加載腳本，留下 tmImage.load() 與 predict() 的 TODO。
  • 📂 js/confidence_gate.js：穩定性過濾組件。預留一個狀態緩衝區邏輯，讓學員實作「必須連續 3 幀滿足高機率才觸發」的防閃爍邏輯。
  • 📂 style/dashboard.css：監測 UI。提供置信度即時亮條 (Progress Bar) 的樣式，讓學員能同步在畫面上看到 AI 的判斷信心。
  • 📂 tests/stability.test.js：自動化評分脚本。專門測試「防誤觸邏輯」——模擬一串不穩定的預測結果（如：0.9, 0.2, 0.9, 0.9），檢核學員的系統是否能正確過濾掉中間的雜訊而不發送錯誤指令。

2. 作業任務部署細節

任務 1：自定義手勢建模 (Custom Gesture Training)

  • 目標：掌握數據標記的多樣性與模型優化。
  • Classroom 部署建議：
    • 建模規範：學員必須在 Teachable Machine 採集四個類別：「Background (背景)」、「Left」、「Right」、「Stop」。
    • 超參數紀錄：要求學員在 README.md 中截圖訓練後的 Loss 曲線，並記錄使用的 Epochs 數量。
    • 數據提交：需提交導出的模型連結或模型權重文件 (model.json, weights.bin)。

任務 2：AI-BLE 驅動程式 (AI-BLE Component)

  • 目標：實現穩定且低延遲的 AI 轉導控制邏輯。
  • Classroom 部署建議：
    • 置信度閘門實作： 
      // 學生需實作：高信心連發保護
if (prediction[i].probability > 0.9) {
    count++;
    if (count >= 3) sendBLECommand(prediction[i].className);
}
    • 推論速率限制 (Throttling)：要求學員設定推論頻率。如果每秒推論 60 次會導致電腦卡頓，應學會限制在 10 ~ 15 FPS。
    • Autograding：透過 Jest 模擬 AI 回傳標籤，檢核 BLE_Write 函式是否在滿足連續幀條件後才被呼叫。

任務 3：失效保護與壓力測試 (Fail-safe Strategy)

  • 目標：針對「未知情境」設計系統的安全邊界。
  • Classroom 部署建議：
    • 失效安全性實作：學員需實作：當所有標籤的最高機率都低於 0.5 時（代表 AI 看不懂現在的情況），必須自動發送 STOP 指令。
    • 性能報告：學員需量化記錄從「手部動作發生」到「車體反應」的總體延遲，並分析其中「推論時間」與「通訊時間」的佔比。
    • PR 深度核核：點評其對「人影噪音」分類的處理。如果背景有人走動，模型是否會產生嚴重的誤判？

3. AI 實踐工程導師點評標準 (AI Engineering Benchmarks)

此單元的價值在於 「對機率性預測的控制與調優能力」：

  • [ ] 模型魯棒性 (Robustness)：在光線變化或背景干擾下，模型是否仍能維持高信心？
  • [ ] 穩定性邏輯 (Stability)：是否實作了「緩衝幀」機制？是否有處理指令過度發送（連發頻率）的問題？
  • [ ] 安全防護意識：是否理解「低信心狀態」的危險性並實作了主動停機？
