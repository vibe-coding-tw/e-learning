機器學習三部曲

所有監督式機器學習（Supervised Learning）專案都遵循相同的三階段流程：收集資料、訓練模型、部署推論。Google 的 Teachable Machine 將這三個階段全部封裝在瀏覽器介面中，讓你無需撰寫任何 Python 或安裝 GPU 驅動，就能完成一個可用的分類模型。

第一步：資料收集（Data Collection）

機器學習的第一定律是 GIGO（Garbage In, Garbage Out）——模型的表現上限取決於訓練資料的品質。在 Teachable Machine 中，你需要為每個要辨識的類別提供足夠數量的樣本。

  • 類別設計：除了目標類別（如「左轉」、「右轉」、「停止」），必須建立一個「背景 / 無動作」類別。這讓模型學會「什麼都不是」的狀態，大幅降低誤觸發率。
  • 樣本數量：每個類別建議至少 50~200 個樣本。影像類型可直接用 Webcam 連續拍攝，音訊類型則錄製 2 秒短片段。
  • 多樣性：確保樣本涵蓋不同角度、光線、距離與背景條件，避免模型學到與目標無關的環境特徵（如特定牆壁顏色）。

第二步：模型訓練（Training）

點擊「訓練模型」後，Teachable Machine 會在你的瀏覽器中使用 TensorFlow.js 進行本地訓練。你可以調整以下超參數：

超參數 說明 建議值
Epochs 模型遍歷全部訓練資料的次數。太少學不夠，太多會過擬合。 50~100
Batch Size 每次權重更新使用的樣本數。較大值訓練更穩定，但佔更多記憶體。 16~32
Learning Rate 權重更新步幅。太大會震盪不收斂，太小收斂過慢。 0.001 (預設)
備註 Teachable Machine 使用遷移學習（Transfer Learning）技術：它不從零訓練 CNN，而是在預訓練的 MobileNet 基礎上微調最後幾層。這就是為什麼只需要少量樣本（50 張）和短暫訓練（數十秒）就能獲得不錯結果的原因。

第三步：即時預覽與驗證

訓練完成後，Teachable Machine 會自動開啟即時預覽面板，顯示各類別的信心分數（Confidence Score，0~1 之間的機率值）。你應該測試以下情境來評估模型品質：

  1. 正常條件下，目標類別的信心分數是否穩定高於 0.9？
  2. 切換類別時，分數轉換是否迅速且乾淨（無殘留高分）？
  3. 在「無動作」狀態下，背景類別是否能穩定拿到最高分？
  4. 極端條件（遮蔽、遠距離、噪音）下表現如何？
提示 如果預覽時發現特定類別經常被誤判，不要急著增加 Epochs——更有效的做法是回到第一步，針對容易混淆的場景增加更多樣化的樣本。資料品質的改善永遠比調參數來得有效。

影像品質與多樣性

機器學習模型的泛化能力（Generalization）取決於訓練資料能否涵蓋部署環境中的各種變異。一個在「書桌前方、日光燈下」訓練的手勢辨識模型，搬到「教室後方、窗戶逆光」環境時可能完全失靈——因為模型學到的不是「手的形狀」，而是「特定光影模式」。

影響模型魯棒性的關鍵因素

變異因素 影響說明 資料收集建議
光線條件 亮度與色溫改變會劇烈影響像素值分布 在日光、燈光、陰影下各拍一組
拍攝角度 正面、側面、俯角看到的物體輪廓完全不同 每個類別至少涵蓋 3 種角度
背景干擾 模型可能學到背景特徵而非目標物體 在不同背景（桌面、地板、走廊）下收集
目標距離 遠近不同會改變物體在畫面中的比例 近距（30cm）、中距（1m）、遠距（2m）
操作者差異 不同人的手型、膚色、動作幅度皆不同 請多位同學分別提供樣本

資料增強（Data Augmentation）

當你無法在所有條件下親自拍攝時，資料增強是一種透過程式自動產生變異樣本的技術。Teachable Machine 內部已自動應用部分增強策略，但理解原理有助於你在更進階的 TensorFlow 專案中手動配置：

  • 隨機翻轉（Flip）：水平鏡像，讓模型不偏好特定方向
  • 隨機旋轉（Rotation）：±15° 範圍內旋轉，模擬相機傾斜
  • 亮度/對比度擾動：模擬不同光源環境
  • 隨機裁切（Crop）：讓模型適應目標不在正中央的情況
注意 並非所有增強都適合你的場景。對於手勢辨識，水平翻轉可能有害——「左轉」手勢翻轉後變成「右轉」，如果不加區分地增強會嚴重混淆模型。永遠根據你的具體任務語意來選擇增強策略。

「背景噪音」類別的設計哲學

在部署環境中，模型面對的絕大多數時間都是「什麼都沒發生」的狀態。如果訓練時沒有足夠的背景樣本，模型會傾向於「硬選」一個目標類別，導致持續的誤觸發（False Positive）。

建議背景類別的樣本數量至少等於所有目標類別的總和，且涵蓋：空無一人的場景、有人走動但無做手勢、手放在桌上不動、以及各種與目標動作相似但不完全一致的「干擾動作」。

提示 驗證模型時，用「混淆矩陣（Confusion Matrix）」來分析哪些類別最容易互相混淆。如果「左轉」經常被誤判為「背景」，代表「左轉」的樣本不夠多樣化或數量不足；如果「背景」經常被判為「停止」，則需要增加更多日常場景的背景樣本。

實戰整合與導出

Teachable Machine 訓練完成的模型可以匯出為 TensorFlow.js 格式，直接在瀏覽器中執行推論——無需後端伺服器。匯出的檔案包含三部分：model.json（模型結構定義）、metadata.json（類別標籤映射）、weights.bin（量化後的權重資料）。這意味著你的整個 AI 推論管線可以完全在前端 JavaScript 中運作。

載入模型與啟動推論

Teachable Machine 提供了專用的 JavaScript 函式庫 @teachablemachine/image，簡化了模型載入與相機整合的流程：

// 載入 Teachable Machine 匯出的影像模型
const URL = './my-model/';
const modelURL = URL + 'model.json';
const metadataURL = URL + 'metadata.json';

const model = await tmImage.load(modelURL, metadataURL);
const maxPredictions = model.getTotalClasses();

// 持續推論迴圈
async function predict() {
  const prediction = await model.predict(webcamCanvas);
  for (let i = 0; i < maxPredictions; i++) {
    console.log(prediction[i].className, prediction[i].probability);
  }
  requestAnimationFrame(predict);
}

置信度閘門：防止誤觸發

AI 模型的輸出是機率性的——它不會說「絕對是左轉」，而是說「左轉的機率 0.87、右轉 0.05、停止 0.03、背景 0.05」。在將預測結果轉為硬體控制指令時，必須實作置信度閘門（Confidence Gate）邏輯，避免在低信心或不穩定的預測下誤發指令：

// 連續幀穩定性檢查：同一類別連續 3 幀 > 0.9 才發送
let stableCount = 0;
let lastLabel = '';

function processPrediction(predictions) {
  const top = predictions.reduce((a, b) =>
    a.probability > b.probability ? a : b);

  if (top.probability > 0.9 && top.className === lastLabel) {
    stableCount++;
    if (stableCount >= 3) sendCommand(top.className);
  } else {
    stableCount = (top.className === lastLabel) ? stableCount : 0;
    lastLabel = top.className;
  }
}

推論速率控制

如果使用 requestAnimationFrame 驅動推論迴圈，在 60Hz 螢幕上每秒會推論 60 次。對於 Teachable Machine 的輕量模型，單次推論約耗時 15~30ms，勉強跟得上——但這會佔用大量 CPU/GPU 資源，導致 UI 其他部分卡頓。建議將推論頻率限制在 10~15 FPS：

// 限制推論頻率為 ~12 FPS
const INFERENCE_INTERVAL = 80; // ms
setInterval(async () => {
  const predictions = await model.predict(canvas);
  processPrediction(predictions);
}, INFERENCE_INTERVAL);
重要 當所有類別的最高機率都低於 0.5 時，代表模型「看不懂」當前狀態。此時絕對不能隨意選擇一個類別——正確的做法是觸發失效保護（Fail-safe）機制，自動發送停止指令，確保硬體不會在 AI 困惑時做出危險動作。

快速建模 (Teachable Machine) 作業指南

歡迎來到進階 09 單元：快速建模！本單元將帶領你跨越「寫代碼控制」，進入「模型驅動控制」的領域。透過機器學習中的 「垃圾進，垃圾出 (GIGO)」 原理，你將學會如何訓練一個有用的模型，並實作嚴謹的「置信度閘門」來確保硬體安全。

作業目標

  1. 掌握數據標記：了解數據標記的多樣性與模型優化。
  2. 實現穩定控制：實作穩定且低延遲的 AI 轉導控制邏輯。
  3. 設計安全邊界：針對未知情境實作失效保護（Fail-safe）機制。

任務說明

任務 1：自定義手勢建模 (Custom Gesture Training)

  1. 建模規範：請在 Teachable Machine 採集四個類別：「Background (背景)」、「Left」、「Right」、「Stop」。
  2. 超參數紀錄：請在下方「學習紀錄」區塊中截圖並貼上訓練後的 Loss 曲線，並記錄你所使用的 Epochs 數量。
  3. 數據提交：請導出模型，並將模型權重文件（例如 model.json, weights.bin 等）放入專案相對應的資料夾中。

任務 2：AI-BLE 驅動程式 (AI-BLE Component)

  1. 置信度閘門實作：在程式中實作高信心連發保護。只有當某個類別的預測機率 大於 0.9 且連續出現 3 幀 時，才發送對應的藍牙指令至機器人。這能有效過濾雜訊，避免機器人錯誤動作。
  2. 推論速率限制 (Throttling)：請設定並限制你的推論頻率。建議控制在 10 ~ 15 FPS，以避免推論次數過高導致電腦卡頓。

任務 3：失效保護與壓力測試 (Fail-safe Strategy)

  1. 失效安全性實作：請實作安全保護邏輯：當所有標籤的最高機率都低於 0.5 時（代表 AI 看不懂現在的情況），系統必須自動發送 STOP 指令。
  2. 性能報告：請在下方「學習紀錄」區塊量化記錄從「手部動作發生」到「車體反應」的總體延遲，並簡單分析其中「推論時間」與「通訊時間」的佔比。

1. 範本程式庫 (Template) 配置建議

AI 整合需要處理非同步載入模型與連續推論流，建議範本包含：

  • 📂 js/ai_controller.js：AI 驅動核心。預置 TensorFlow.js 與 Teachable Machine 函式庫加載腳本，留下 tmImage.load() 與 predict() 的 TODO。
  • 📂 js/confidence_gate.js：穩定性過濾組件。預留一個狀態緩衝區邏輯，讓學員實作「必須連續 3 幀滿足高機率才觸發」的防閃爍邏輯。
  • 📂 style/dashboard.css：監測 UI。提供置信度即時亮條 (Progress Bar) 的樣式，讓學員能同步在畫面上看到 AI 的判斷信心。
  • 📂 tests/stability.test.js：自動化評分脚本。專門測試「防誤觸邏輯」——模擬一串不穩定的預測結果（如：0.9, 0.2, 0.9, 0.9），檢核學員的系統是否能正確過濾掉中間的雜訊而不發送錯誤指令。

2. 作業任務部署細節

任務 1：自定義手勢建模 (Custom Gesture Training)

  • 目標：掌握數據標記的多樣性與模型優化。
  • Classroom 部署建議：
    • 建模規範：學員必須在 Teachable Machine 採集四個類別：「Background (背景)」、「Left」、「Right」、「Stop」。
    • 超參數紀錄：要求學員在 README.md 中截圖訓練後的 Loss 曲線，並記錄使用的 Epochs 數量。
    • 數據提交：需提交導出的模型連結或模型權重文件 (model.json, weights.bin)。

任務 2：AI-BLE 驅動程式 (AI-BLE Component)

  • 目標：實現穩定且低延遲的 AI 轉導控制邏輯。
  • Classroom 部署建議：
    • 置信度閘門實作： 
      // 學生需實作：高信心連發保護
if (prediction[i].probability > 0.9) {
    count++;
    if (count >= 3) sendBLECommand(prediction[i].className);
}
    • 推論速率限制 (Throttling)：要求學員設定推論頻率。如果每秒推論 60 次會導致電腦卡頓，應學會限制在 10 ~ 15 FPS。
    • Autograding：透過 Jest 模擬 AI 回傳標籤，檢核 BLE_Write 函式是否在滿足連續幀條件後才被呼叫。

任務 3：失效保護與壓力測試 (Fail-safe Strategy)

  • 目標：針對「未知情境」設計系統的安全邊界。
  • Classroom 部署建議：
    • 失效安全性實作：學員需實作：當所有標籤的最高機率都低於 0.5 時（代表 AI 看不懂現在的情況），必須自動發送 STOP 指令。
    • 性能報告：學員需量化記錄從「手部動作發生」到「車體反應」的總體延遲，並分析其中「推論時間」與「通訊時間」的佔比。
    • PR 深度核核：點評其對「人影噪音」分類的處理。如果背景有人走動，模型是否會產生嚴重的誤判？

3. AI 實踐工程導師點評標準 (AI Engineering Benchmarks)

此單元的價值在於 「對機率性預測的控制與調優能力」：

  • [ ] 模型魯棒性 (Robustness)：在光線變化或背景干擾下，模型是否仍能維持高信心？
  • [ ] 穩定性邏輯 (Stability)：是否實作了「緩衝幀」機制？是否有處理指令過度發送（連發頻率）的問題？
  • [ ] 安全防護意識：是否理解「低信心狀態」的危險性並實作了主動停機？
