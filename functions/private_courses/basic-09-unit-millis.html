😈 1. 為什麼 delay() 是邪惡的？

執行 delay(1000) 時，處理器全停。像等紅燈閉眼，無視緊急狀況。

The Impact

  • 指令無視：CPU 忙著 delay，你的按鈕指令被丟棄。
  • 連線中斷：超時導致反應遲鈍甚至斷線。

Single Thread Problem

"Doing Absolutely Nothing but Waiting."

⏱️ 2. millis() 碼錶邏輯

不要叫電腦「等待」。要它「檢查時間差是否足夠」。

Non-blocking Formula

\[now - prev \ge Interval\]

Variable: unsigned long

MILLIS PATTERN (C++) 
unsigned long last = 0;
void loop() {
  unsigned long n = millis();
  if (n - last >= 1000) {
    last = n; // Task!
  }
  // Free to handle other tasks...
}

擺脫 delay() — 使用 millis() 實現非阻塞邏輯

🎯 學習目標

在本單元中，你將學習如何從「線性思維」跨入「多工系統」。透過掌握 millis() 與非阻塞邏輯，你將學會讓機器人同時處理多個任務，解決因 delay() 導致的指令遺失或系統延遲問題。

🛠 任務說明

任務 1：心跳與指令並行 (Heartbeat & Commands Parallel Lab)

  • 目標：移除所有的 delay()，讓 LED 閃爍（系統心跳）與使用者指令接收互不干擾。
  • 實作要點：
    • 將 src/main.cpp 中的 LED 閃爍邏輯從 delay(500) 重構為基於 millis() 的時間檢查邏輯。
    • 嚴格要求：除了 setup() 外，loop() 中禁止出現任何 delay() 字樣。

任務 2：多重計時器管理 (Multi-Tasking Timers Lab)

  • 目標：管理多個獨立的時間戳記，讓不同頻率的任務（如：10Hz 測距、2Hz 電量回報）穩定運行。
  • 實作要點：
    • 在 include/scheduler.h 中定義任務調度結構（如果適用）。
    • 使用 unsigned long 並採用「減法判定」(now - last >= interval) 以確保溢位安全。

任務 3：反應速度壓力測試 (Response Speed & Safety Lab)

  • 目標：實作超時強迫停機邏輯，並量化非阻塞邏輯帶來的效能提升。
  • 實作要點：
    • 實作 Watchdog 邏輯：若超過 1.5 秒未收到遙控指令，系統必須立即停機。
    • 在 docs/Benchmark.md（或指定位置）中記錄改用 millis() 前後的反應時間差異。

📂 程式庫結構

  • src/main.cpp: 核心程式碼實作區。
  • include/: 標頭檔目錄。
  • docs/: 實驗報告與相關文件。

📝 繳交規範

2. 作業任務部署細節

任務 1：心跳與指令並行 (Heartbeat & Commands Parallel Lab)

  • 目標：根除 delay()，讓系統心跳（LED 閃爍）與使用者指令（網路通訊）互不干擾。
  • Classroom 部署建議：
    • 阻塞清除術：將 LED 閃爍邏輯從 delay(500) 重構為 if (now - last >= 500)。
    • 響應測評：在燈號持續閃爍時，嘗試連續發送 10 個控制指令。
    • 驗證要點：關鍵技術審核。在所有代碼中搜尋 delay 字樣。除了 setup 以外，loop 裡面出現 delay 即視為失敗。確認閃燈過程中指令反應毫無遲滯。

任務 2：多重計時器管理 (Multi-Tasking Timers Lab)

  • 目標：掌握獨立時間戳管理，學習如何讓不同頻率的任務（如：10Hz 測距、2Hz 回傳電壓）穩定運行而不產生頻率偏移。
  • Classroom 部署建議：
    • 分時調度實作：配置多組 previousMillis，確保兩項任務以各自的間隔運行。
    • 溢位安全演算：核心關鍵點。務必使用「減法判定」(now - last >= interval)，確保在 unsigned long 溢位（約 50 天一次）時邏輯依然正確。
    • 驗證要點：檢視代碼是否使用了獨立的 last_time 變數？任務執行週期是否符合預期（例如：每 10 次測距才回報一次電量）？

任務 3：反應速度壓力測試 (Response Speed & Safety Lab)

  • 目標：透過量化實驗證明非阻塞邏輯在安全性上的巨大優勢，並實作超時強迫停機邏輯。
  • Classroom 部署建議：
    • 量化對照實驗：在 README 中記錄：當 loop 中有 delay(1000) 時，按下剎車後的反應時間；與改用 millis() 後的反應時間。
    • 安全守護 (Watchdog)：實作一個獨立的計時器。若超過 1.5 秒未接收到遙控指令，系統必須立即強迫停機，且此邏輯不能被閃燈任務干擾。
    • 驗證要點：實驗日誌是否展示了非阻塞邏輯對剎車距離的顯著改善？失效停機邏輯是否能在閃燈運行的同時可靠發揮作用？
