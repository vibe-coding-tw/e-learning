擺脫 delay()：
millis() 與非阻塞邏輯

解鎖 ESP32 的並行潛力，讓機器人不再因為「等待」而錯失關鍵指令。

😈 1. 為什麼 delay() 是邪惡的？

執行 delay(1000) 時，處理器會完全停下來。這就像你在等紅綠燈時閉上眼睛，完全無視周遭發生的緊急狀況。

The Impact

  • 指令無視：若無人車快撞牆，你的按鈕指令會因為 CPU 还忙著 delay 而被丟棄。
  • 連線中斷：緩衝區溢位或超時會導致遙控器反應遲鈍甚至斷線。

Single Thread Problem

"CPU is Doing Absolutely Nothing but Waiting."

⏱️ 2. millis() 的本質：碼錶邏輯

不要叫電腦「等待 1 秒」，而是要它「檢查現在時間，減去上次執行的時間，是否大於 1 秒」。

Non-blocking Formula

\[T_{now} - T_{prev} \ge Interval\]

Variable type: unsigned long

MILLIS PATTERN (C++) 
unsigned long last = 0;
void loop() {
  unsigned long now = millis();
  if (now - last >= 1000) {
    last = now;
    // Task Executed!
  }
  // Free to handle other tasks instantly...
}

🤝 3. 合作式多工：任務並行

每個任務都只佔用處理器極短的時間（檢查一下時間，不該做就跳過），這讓多個任務看起來像是「同時運行」。

Cooperative Loop

  • 高頻任務：10ms 讀取一次感測器與控制指令。
  • 低頻任務：500ms 回傳一次電量資料。
  • 失效保護：偵測通訊超時，若 2s 沒訊號則強制剎車。

GitHub Classroom 詳細操作指南

本單元教學重點：掌握合作式多工思維與即時任務調度。

  1. 重點 1：封印 delay() - 讓學員理解在通訊系統中，任何毫秒級的延遲都會導致 CPU 進入「盲聽」狀態，造成操控事故。
  2. 重點 2：減法判定的數學原理 - 務必詳細解釋為何必須使用 now - last >= interval 而不能用加法，這關乎系統的長期運行穩定性。
  3. 重點 3：優先級與時間片 - 引導學員思考：接收控制指令的任務優先級永遠應該高於傳送狀態資料的低頻任務。
