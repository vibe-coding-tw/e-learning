為什麼需要流控？

初學者常有一個直覺迷思：「傳送越多資料，控制就越精準」。事實恰好相反——在無線通訊環境中，未經節制的高頻資料傳輸會迅速填滿通道緩衝區，導致排隊延遲（Queuing Delay）急劇上升。當你的搖桿每 10ms 送出一筆位置資料，但 BLE 連線每 30ms 才能消化一筆時，後方的封包開始堆積，使用者感受到的是操控「越來越遲鈍」。

頻寬瓶頸與壅塞理論

任何通訊通道都有其理論最大吞吐量（Throughput）。以 BLE 4.2 為例，在 ATT MTU 為 23 bytes、連線間隔（Connection Interval）為 7.5ms 的最佳條件下，實際有效吞吐量約為 305 kbps。然而現實環境中因封包重傳、協議開銷與干擾，實測吞吐量常常只有理論值的 30~50%。

當傳入速率（Input Rate）持續超過通道消化速率（Drain Rate），緩衝區使用率就會線性增長。根據利特爾定律（Little's Law）：

$L = \lambda \cdot W$

其中 $L$ 是佇列中的平均封包數、$\lambda$ 是封包到達率、$W$ 是平均等待時間。當 $\lambda$ 增加而服務速率不變，$W$ 必然上升——這就是「傳越多，延遲越高」的數學證明。

注意 ESP32 的 BLE 堆疊（NimBLE / Bluedroid）內部維護一個有限的傳送佇列（通常 8~16 筆）。當佇列滿載時，後續的 write() 呼叫會被直接丟棄或阻塞，導致資料遺失或程式凍結。流控不是「錦上添花」，而是系統穩定的基本要求。

前端事件風暴的真實案例

瀏覽器中的 UI 事件觸發頻率遠超一般預期：

事件來源 典型觸發頻率 5 秒內產生的事件數
input (Slider 拖動) 60~120 次/秒 300~600 筆
mousemove 60~144 次/秒 300~720 筆
keydown (長按) 30 次/秒 150 筆
WebSocket 感測器推送 50 次/秒 250 筆

如果每一筆事件都直接觸發 BLE 寫入，通道會在數百毫秒內壅塞。流控技術的核心目標就是：在不犧牲操控即時性的前提下，將傳輸頻率降至通道能舒適消化的範圍。

備註 在 AI 語音辨識場景中，流控同樣重要。如果 WebSocket 每 20ms 推送一段 MFCC 特徵向量到前端做即時分類顯示，但 UI 渲染來不及消化，就會出現掉幀甚至記憶體耗盡。本單元的技術同樣適用於音訊串流的流量管理。

兩大技術武器

流控領域有兩種經典策略：防抖（Debounce）與節流（Throttle）。雖然兩者都用於限制函式的執行頻率，但它們的觸發時機與適用場景截然不同。選擇錯誤的策略不僅無法解決問題，甚至可能引發更嚴重的操控異常。

防抖 Debounce：「等到停下來才執行」

防抖的核心邏輯是：每次事件觸發時重置計時器，只有當事件停止觸發超過指定等待時間後，才真正執行一次函式。適用於「使用者完成操作後才需要回應」的場景，例如搜尋框輸入、視窗大小調整、參數設定確認等。

// 防抖：trailing edge（預設行為）
function debounce(fn, wait) {
  let timer = null;
  return (...args) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), wait);
  };
}

節流 Throttle：「固定頻率放行」

節流的核心邏輯是：無論事件觸發多頻繁，函式在每個 wait 時間間隔內最多只執行一次。這確保了穩定且可預測的輸出頻率，非常適合需要持續性回饋的場景，如搖桿位置回報、即時繪圖更新、感測器資料顯示等。

// 節流：基於時間戳記的實作
function throttle(fn, wait) {
  let lastTime = 0;
  return (...args) => {
    const now = Date.now();
    if (now - lastTime >= wait) {
      lastTime = now;
      fn(...args);
    }
  };
}

對照比較：如何選擇？

面向 防抖 (Debounce) 節流 (Throttle)
觸發時機 事件停止後延遲執行 固定間隔定時執行
適用場景 搜尋輸入、設定確認、按鍵防彈跳 搖桿控制、滾動更新、即時繪圖
輸出特性 只保留「最終值」 等間距採樣，保留過程
IoT 控制適用性 適合狀態切換指令 適合連續位置控制
提示 對於遙控車的搖桿控制，Throttle 是唯一正確的選擇。使用 Debounce 會導致使用者在持續拖動搖桿時完全沒有回應（因為事件一直在觸發，計時器不斷重置），直到手指鬆開才送出一筆——這對即時操控是災難性的。

高階流控優化

基礎的 Throttle/Debounce 僅從「時間維度」限制傳輸頻率。然而在真實的物聯網控制系統中，更精細的流控策略需要結合資料本身的語意來判斷「這筆資料是否值得傳送」。本節介紹三種進階技術：死區過濾、變動門檻與 Leading Edge 執行。

死區（Deadzone）過濾

物理搖桿在「中心位置」時，由於機械公差與 ADC 雜訊，讀數往往會在 $\pm 2\sim5\%$ 之間微幅抖動。如果不加過濾，這些抖動會持續觸發馬達微小電流，產生「嗡嗡聲」且消耗電力。死區邏輯將中心附近的數值強制歸零：

// 死區過濾：將中心 ±10% 範圍歸零
function applyDeadzone(value, threshold = 0.1) {
  if (Math.abs(value) < threshold) return 0;
  // 重新映射：讓 threshold~1.0 映射到 0~1.0
  const sign = Math.sign(value);
  return sign * (Math.abs(value) - threshold) / (1 - threshold);
}

變動門檻（Delta Threshold）

即使搖桿已離開死區，如果連續兩次讀數的差異極小（例如從 0.51 變為 0.52），傳送這筆更新對馬達行為幾乎沒有影響。設定一個最小變動量 $\epsilon$，只有當 $|V_{now} - V_{last\_sent}| > \epsilon$ 時才觸發傳送：

let lastSentValue = 0;
const EPSILON = 0.05; // 5% 變動門檻

function shouldSend(currentValue) {
  if (Math.abs(currentValue - lastSentValue) > EPSILON) {
    lastSentValue = currentValue;
    return true;
  }
  return false;
}

Leading Edge 執行

標準 Throttle 在第一次觸發時就立刻執行（Leading Edge），但標準 Debounce 則必須等待靜默期結束才執行（Trailing Edge）。對於遙控場景，Leading Edge 是必須的——使用者按下按鈕的那一瞬間就期望看到回應，任何啟動延遲都會破壞「手感」。

// 帶 leading 的 Debounce：首次觸發立即執行
function debounce(fn, wait, { leading = false } = {}) {
  let timer = null;
  let canLead = true;
  return (...args) => {
    if (leading && canLead) { fn(...args); canLead = false; }
    clearTimeout(timer);
    timer = setTimeout(() => { canLead = true; }, wait);
  };
}

組合策略：實戰中的最佳實踐

在生產級 IoT 控制系統中，通常會將以上三種技術串聯使用：

  1. 原始感測值先經過死區過濾，消除靜止噪音
  2. 再通過變動門檻，確保只有有意義的變化才進入傳輸管線
  3. 最後套用Throttle（帶 Leading Edge），保證輸出頻率不超過通道能力

這種多層過濾策略通常能將封包數削減 70~85%，同時維持操控的即時感與流暢度。

重要 流控參數的選擇是安全性與效率的權衡。如果 Throttle 間隔設為 500ms（每秒只傳 2 包），雖能極度省流量，但對於避障場景可能致命——偵測到障礙物後 500ms 才能送出煞車指令，這段延遲足以讓車體撞上障礙物。建議控制類指令的 Throttle 間隔不超過 100ms。

作業 9：資料緩衝與流控 (Data Buffering and Flow Control)

這是一個從「能跑就好」進展到「工程級優化」的核心作業。透過本次作業，你將建立「流量衛生 (Flow Hygiene)」觀念：理解傳送 「過多且重複」 的數據不僅浪費頻寬，更會引發緩衝區排隊導致嚴重的操控延遲。

作業目標

  1. 精準節流 (UI Event Throttling Lab)：解決「高頻事件」引發的通訊塞車現象，實作基於時間戳記或計時器的節流機制。
  2. 流暢防抖與死區設計 (Smart Debounce & Deadzone)：消除「手動噪音」並實作具有爆發力的即時響應。
  3. 系統端到端性能診斷 (Performance Profiling Report)：學會使用專業工具（如 Chrome DevTools Performance 面版）診斷控制系統的瓶頸。

作業需求

請在專案的 js/utils_logic.js 中實作以下流控制函式：

1. throttle(fn, wait)

限制函式在 wait 毫秒內最多只能被觸發一次。確保即使高速滑動 UI 控件，通訊頻率也能被精確控制。

2. debounce(fn, wait, options)

實作防抖機制。若包含 leading: true 選項，必須在第一次觸發時立即執行，隨後在 wait 毫秒內忽略後續重複觸發。這有助於消除物理噪音並保持立即響應的「手感」。

3. 死區過濾 (Deadzone)

在你的控制器邏輯中，加入物理死區判斷，過濾掉接近 0 的微小波動（例如：當絕對值小於 0.1 時，將其視為 0）。

    • Autograding：透過模擬事件流，檢核學員實作的函式執行次數是否在容許誤差範圍內。

任務 2：「流暢防抖」與死區設計 (Smart Debounce & Deadzone)

  • 目標：消除「手動噪音」並實作具有爆發力的即時響應。
  • Classroom 部署建議：
    • 響應邏輯檢核：學員必須實作 leading: true 特性的防抖（初次觸發立即執行）。
    • 死區過濾 (Deadzone) 實作： 
      // 學生需實作：物理死區過濾
if (Math.abs(value) < 0.1) value = 0; // 靜止狀態下過濾微小抖動
    • PR 深度核核：點評其對「為什麼遙控器需要 Leading Edge 防抖」的理解（答案：防止手感產生初次點擊的啟動延遲）。

任務 3：系統端到端性能診斷 (Performance Profiling Report)

  • 目標：學會使用專業工具診斷控制系統的瓶頸。
  • Classroom 部署建議：
    • 工具使用要求：學員需開啟 Chrome DevTools 的 Performance 面版，錄製一段操作過程。
    • 延遲分析報告：在 README.md 中貼上火焰圖 (Flame Chart) 截圖，標示出事件處理 (Scripting) 與封包傳送的預計耗時。
    • 理論反思：引導學員討論：如果我們將節流時間設為 500ms（每秒只傳 2 包），雖極度省流量，但會對「安全性」造成什麼致命傷？（答案：操作反應變得極慢，無法避障）。
