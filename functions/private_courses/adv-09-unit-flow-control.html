🚧 1. 為什麼需要流控？ (20 min)

在即時控制中「傳越多越準」是誤區。未經過濾的高頻訊號會堵塞 ESP32 的藍牙緩衝區，導致嚴重的封包排隊 (Queuing Delay)，讓你的操控產生顯著的「延遲感」。

Congestion Theory

  • 頻寬瓶頸：無線連結的吞吐量有限，必須精打細算。
  • 資料噪音：手指的生理顫抖不需要傳送給馬達，這只會浪費資源。

🧠 3. 高階流控優化

除了時間限制，我們還能透過數據本身進行過濾。設置變化閾值 (Threshold) 與死區 (Deadzone)，消除無效的微小抖動，確保只有「有意義的改變」才會被傳輸。

Efficiency Engineering

  • Deadzone (死區)：忽略中心點 $\pm 10\%$ 的極小位移，防止馬達空轉嗡鳴。
  • Threshold (變動門檻)：只有當數值變動 $|V_{now} - V_{last}| > \epsilon$ 時才發送。
  • Leading Execution：確保流控下，第一個動作能被立即執行（零延遲起步）。

作業 9：資料緩衝與流控 (Data Buffering and Flow Control)

這是一個從「能跑就好」進展到「工程級優化」的核心作業。透過本次作業，你將建立「流量衛生 (Flow Hygiene)」觀念：理解傳送 「過多且重複」 的數據不僅浪費頻寬，更會引發緩衝區排隊導致嚴重的操控延遲。

作業目標

  1. 精準節流 (UI Event Throttling Lab)：解決「高頻事件」引發的通訊塞車現象，實作基於時間戳記或計時器的節流機制。
  2. 流暢防抖與死區設計 (Smart Debounce & Deadzone)：消除「手動噪音」並實作具有爆發力的即時響應。
  3. 系統端到端性能診斷 (Performance Profiling Report)：學會使用專業工具（如 Chrome DevTools Performance 面版）診斷控制系統的瓶頸。

作業需求

請在專案的 js/utils_logic.js 中實作以下流控制函式：

1. throttle(fn, wait)

限制函式在 wait 毫秒內最多只能被觸發一次。確保即使高速滑動 UI 控件，通訊頻率也能被精確控制。

2. debounce(fn, wait, options)

實作防抖機制。若包含 leading: true 選項，必須在第一次觸發時立即執行，隨後在 wait 毫秒內忽略後續重複觸發。這有助於消除物理噪音並保持立即響應的「手感」。

3. 死區過濾 (Deadzone)

在你的控制器邏輯中，加入物理死區判斷，過濾掉接近 0 的微小波動（例如：當絕對值小於 0.1 時，將其視為 0）。

測試與評分

我們已經準備了自動化測試來檢核你的實作：

  • 執行 npm test 確認是否通過所有測試。
  • 測試包含執行頻率的檢核，確保 throttle 和 debounce 的時間間隔精準且穩定。

提交方式

將所有修改推送到你的 GitHub 程式庫中，系統將自動進行評分 (Autograding)。 確保 npm test 在 GitHub Actions 中能順利通過。

進階思考 (Reflection)
