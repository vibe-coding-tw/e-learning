什麼是 AI 代理人？
概念介紹 ⏱ 10 分鐘

傳統的 GitHub Copilot 是一個「預測下一個字元」的工具。而 AI Agent（AI 代理人） 是一個能夠接收高階目標，自行分解任務、呼叫工具、執行操作、觀察結果並反覆修正，直到完成任務的自主系統。

傳統 AI 助手 vs. AI 代理人

特性 傳統 Copilot（AI 助手） Copilot Agent Mode（AI 代理人）
工作範圍 單一檔案、當前游標 ✓ 整個 Workspace
執行能力 只寫程式碼建議 ✓ 能執行終端機指令、跑測試
錯誤修復 等你發現錯誤 ✓ 看見報錯後自動修復
主動性 被動補全 ✓ 主動規劃方案（Plan First）
跨工具整合 不支援 ✓ 透過 MCP 存取外部工具

自主回饋迴圈（Agentic Loop）

AI 代理人的核心機制是一個不斷循環的自我改進迴圈：

🎯 接收目標
→
📋 制定計畫
→
⚡ 執行操作
→
👁 觀察結果
→
🔄 修正重試
💡 開發者角色的轉變
在 2026 年，優秀的工程師不再是「最能寫程式碼的人」，而是最能引導 AI Agent 完成複雜任務的「代理人主管（Agent Director）」。你的核心技能是：精確描述目標、審查計畫、在關鍵節點介入修正。
Copilot Agent Mode 深度實戰
操作技能 ⏱ 15 分鐘

啟動 Agent Mode

在 VS Code 中啟動 Copilot Agent Mode 只需三個步驟：

  1. 1
    開啟 Copilot Chat
    使用快捷鍵 Ctrl+Alt+I（Mac: ⌘⌥I）開啟 Copilot Chat 視窗。
  2. 2
    切換到 Agent 模式
    在 Chat 頂部的下拉選單中，從 「Ask」 切換到 「Agent」 模式。你會看到模式指示器變化，表示 Agent 現在能存取你的整個 Workspace。
  3. 3
    下達高階目標
    輸入任務描述。不需要告訴它「怎麼做」，只需要說「做什麼」。例如：「幫我把這個專案的 REST API 改成 GraphQL，並確保所有測試通過。」
📝 Agent 會先給你計畫
Agent 不會立刻修改任何檔案。它會先產出一份執行計畫（Plan），列出所有步驟，等待你確認後才開始執行。這是你作為「指揮官」的關鍵介入時機——審查計畫、刪除不需要的步驟、調整優先順序。

MCP：賦予 Agent「手和腳」

MCP (Model Context Protocol) 是一個開放標準，讓 AI Agent 能夠連接外部工具與私有數據，突破模型知識的邊界。

🔌 常見 MCP 應用場景
  • ✅ 自動查詢 Jira / GitHub Issues
  • ✅ 即時搜尋 Google 外部資訊
  • ✅ 讀取本地私有資料庫
  • ✅ 呼叫公司內部 API
VS Code settings.json
// 啟用 MCP 功能
"chat.mcp.enabled": true

AGENTS.md：定義 AI 的工作 DNA

在專案根目錄建立 AGENTS.md，Agent 會自動讀取並遵循其中的規則，確保團隊協作中 AI 行為一致。

# AGENTS.md - AI 代理人工作規範 ## 程式碼規範 - 所有 API 必須使用 TypeScript 定義介面 - 所有組件使用 Tailwind CSS 進行樣式設定 - 測試使用 Jest，覆蓋率需大於 80% ## 命名規範 - 函數使用 camelCase - 所有導出函數必須包含 JSDoc 註解 ## 禁止行為 - 禁止使用 console.log（請改用 logger 模組） - 禁止硬編碼任何敏感資訊或金鑰
💡 AGENTS.md 的適用範圍
AGENTS.md 適用於 OpenAI Codex CLI 等 Agent。VS Code Copilot 則讀取 .github/copilot-instructions.md。兩者格式相似，根據你的主要工具選擇對應文件即可。

網頁版代理人平台
平台實戰 ⏱ 15 分鐘

不需要安裝任何軟體，只用瀏覽器就能完成完整的 AI 代理開發。GitHub 生態系提供了三個各有定位的雲端代理平台。

🚀 GitHub Copilot Workspace

目前自動化程度最高的 Web 代理環境。以「任務 (Task)」為中心而非「檔案」，從 Issue 到 PR 全自動閉環。

📋
規劃 (Plan)
Agent 讀取整個 Repo，針對你的需求寫出可編輯的實施計畫，供你審查與修改。
⚡
執行 (Implement)
確認計畫後，在雲端環境自動修改多個檔案，整個過程可即時追蹤。
✅
驗證 (Verify)
自動執行測試，失敗時 Agent 讀取錯誤訊息並自行修復，直到通過。
🔗 核心工作流：From Issue to PR
任務完成後，Workspace 直接幫你開啟 Pull Request，PR 描述中會附上 Agent 的執行計畫與修改摘要。前往：copilot-workspace.github.next.com

其他 Web 代理平台

💬
GitHub.com Copilot Chat
Web 對話
直接在 Repo 頁面使用。按 Ctrl+B 啟動，支援 Tool Calls 讓 Agent 搜尋程式碼、分析結構。
💻
github.dev / Codespaces
雲端 IDE
按 .（句點）進入輕量編輯器；Codespaces 提供完整終端機與 Agent Mode 環境。

⚙️ GitHub Agentic Workflows

把 AI Agent 嵌入 CI/CD 流程，讓它成為自動運行的「幽靈工程師」。指派 Issue 給 @copilot，它自動開分支、寫程式碼並提交 PR。

# .github/workflows/ai-agent.yml # 當 Issue 加上 'ai-fix' 標籤時自動啟動 Agent on: issues: types: [labeled] jobs: agentic-fix: if: github.event.label.name == 'ai-fix' runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - name: Run Copilot Agent run: copilot-agent --issue ${{ github.event.issue.number }}

Vibe Coding 工具旗艦店
工具介紹 ⏱ 15 分鐘

Vibe Coding 指的是以自然語言描述意圖，讓 AI 自主完成大部分程式碼的開發方式。不同工具有不同定位，選對工具是關鍵。

⚡
Replit Agent
全棧自動化
Web 端的全棧開發代理。你描述一個功能需求，Agent 自動建立資料夾結構、安裝套件、撰寫前後端程式碼，並在即時預覽中執行。無需本地環境，適合快速驗證想法。
🎯 最適合：後端 API、全棧原型、腳本自動化
💜
Lovable.dev
設計優先
設計優先的 Vibe Coding 平台。用自然語言描述 UI 元件，生成精美的 React 組件。支援視覺點擊編輯，修改樣式像操作 Figma 一樣直覺。背後整合 Supabase 作為後端，一鍵啟動全棧應用。
🎯 最適合：Landing Page、SaaS 儀表板、組件庫
🖥️
Cursor / Windsurf
AI 原生 IDE
以 VS Code 為基礎的 AI 原生代碼編輯器。Cursor 以 Claude 為核心，Tab 預測你的下一步編輯，Composer 模式處理多檔案任務。Windsurf 的 Cascade 模式擁有深度脈絡感知，適合複雜代碼庫。
🎯 最適合：複雜代碼庫重構、多檔案修改、本地開發
🤖
Claude Code
終端機代理
Anthropic 出品的終端機 AI 代理。在專案根目錄執行 claude，描述任務後它能讀取整個代碼庫、執行 bash 命令、修改檔案。特別擅長安全稽核、架構重構等需要深度理解的複雜任務。
🎯 最適合：安全掃描、大型重構、代碼分析報告
⚠️ 選工具的思考框架
  • 需要快速建 UI 介面？→ Lovable.dev
  • 需要完整後端 API 與全棧原型？→ Replit Agent
  • 在既有大型專案中工作？→ Cursor / Windsurf
  • 需要安全審計或深度架構分析？→ Claude Code
Antigravity：Vibe Team 多 Agent 協作
進階概念 ⏱ 10 分鐘

如果說單個 AI Agent 是「Vibe Coding」，那麼 Antigravity 就是「Vibe Team」——一個能協調多個 AI 子代理並行工作的多 Agent 編排平台。

Antigravity 架構

一個主代理（Orchestrator）統籌，多個專門子代理並行執行各自領域的任務：

🎯 Orchestrator Agent
↙ ↓ ↘
🎨
UI Agent
設計與前端元件
⚙️
Backend Agent
API 與資料庫
🧪
Test Agent
自動整合測試
# Antigravity 工作流示例 orchestrator: "建立一個電商後台管理系統" # 並行啟動三個子代理 parallel: - agent: ui-agent task: "生成產品管理 Dashboard" - agent: backend-agent task: "實作 Product CRUD API" - agent: test-agent task: "為所有 API 端點撰寫整合測試" # 彙整並自動部署 after: deploy_to_production

CI/CD：Vibe Coding 的品質保障

在 Vibe Coding 工作流中，CI/CD 不只是「自動跑測試」，更是多個 Agent 並行成果的最終整合點。

💡 為什麼 Vibe Coding 更需要 CI/CD？
  • 速度：Vibe Coding 的頻繁迭代需要自動化部署管道
  • 品質保障：自動測試防止 AI 產生的回歸錯誤
  • 可追溯性：每次 commit 對應一個可驗證的部署版本
  • 多 Agent 整合：CI/CD 是各子代理成果的最終驗證關卡
⚠️ 多 Agent 協作的注意事項
多個 Agent 並行工作可能產生衝突（如同時修改同一個介面定義）。Orchestrator 的核心職責是定義清晰的介面邊界（Interface Boundaries），並在整合時解決衝突。建議在開始前先讓 Orchestrator 輸出完整的模組分工說明。
