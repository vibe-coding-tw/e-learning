Vibe 時代的提交規範

在 AI 協助開發的時代，一次性提交（複製貼上）會導致學習過程不對稱。我們建議採用 「微提交 (Micro-commits)」 策略：

範例提交訊息：
• git commit -m "[AI-Generated] 實作 BLE 連線基礎邏輯"
• git commit -m "[Human-Modified] 修正 MTU 限制下的分段傳輸 Bug"
• git commit -m "[Review-Only] 完成單元測試驗證"

💡 每一筆 Commit 都是你的開發思考痕跡。

自動評分 (Autograding) 邏輯

系統會模擬真實的產品驗收，對你的程式碼進行功能測試。

✅
通過測試
邏輯符合預期，滿足產品規格。
❌
測試失敗
點擊紅叉查看報錯，修正後再次 Push。

核心教學策略：從「寫代碼」轉向「架構設計」

Prompt 審查作業

除了提交代碼，你也需要記錄與 AI 的對話過程。我們關注的是你如何精確描述需求，以及如何糾正 AI 的「幻覺」。這才是 Vibe Coding 真正的基本功。

規格說明書 (Spec) 的定義權

訓練撰寫高品質的 Markdown 文檔，定義介面、資料結構與流程。在 2026 年，定義問題的能力遠比寫出語法更重要。

GitHub Classroom 工作流

任務 1：接受邀請與首次推送 (The Lifecycle Loop)

角色扮演：實習開發者 | 難度：⭐️ | 估計時長：15 min

🛠️ 執行技術流程 (Git Workflow)

  1. 接受挑戰：點擊老師提供的 GitHub Classroom 邀請連結。
  2. Clone 倉庫：將建立好的倉庫 git clone 到你的開發環境。
  3. 驗證推送：修改 README.md，增加一行「我已完成工作流設定」，並執行 git push。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 連線成功：GitHub Actions 是否成功觸發？
  • [ ] 權限驗證：是否能正確在 Classroom Dashboard 看到學員的名單？

任務 2：微提交實踐 (Micro-Commit Mastery)

角色扮演：資深工程師 (Senior) | 難度：⭐️⭐️ | 估計時長：30 min

🛠️ 執行技術流程 (Git Workflow)

  1. 邏輯實作：在一個小型專案中，分別完成「骨架定義」與「內容填充」。
  2. 提交規範：為這兩個動作分別建立 Commit：
    • feat: Define structural components
    • docs: Add descriptive content
  3. 同步提交：以此展現你的開發並非大量複製貼上，而是有層次的構建過程。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] Commit 質量：提交訊息是否清晰描述了變動內容？
  • [ ] 細粒度控管：是否成功將工作拆分為獨立的邏輯單元？

任務 3：紅燈轉綠燈 (Autograding Fix)

角色扮演：測試工程師 (QA) | 難度：⭐️⭐️⭐️ | 估計時長：40 min

🛠️ 執行技術流程 (Git Workflow)

  1. 故意失敗：推送一段不符合 assignment_test.js 規範的代碼（故意讓測試失敗）。
  2. 查閱日誌：在 GitHub 介面的 "Actions" 分頁中找出報錯原因。
  3. 修正與通過：根據錯誤訊息修正代碼再次推送，直到看到 ✅ 指標。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 修復紀錄：是否有從失敗到成功的 Git Commit 紀錄？
  • [ ] 測試覆蓋：最終是否通過了所有的自動評分測試案例？
