簡介

真實世界是類比的——溫度、光線、搖桿角度都是連續變化的物理量，但微控制器只懂 0 與 1。 ADC（類比數位轉換器）就是這座橋：它把 0～3.3V 的電壓轉換成數位數值，而「解析度」決定了這個轉換有多精細。 搞懂 ADC 的位元數、參考電壓與數值映射邏輯，你才能準確地將感測器的原始讀數變成有意義的工程單位。

情境
你用 ESP32 讀取一顆光敏電阻的數值，想根據亮度自動開關 LED。程式寫好了，analogRead() 回傳的值在 0～4095 之間跳動， 但你卻不確定「2048」到底代表幾伏特？為什麼同一顆電阻在 Arduino Uno 上讀到 512，在 ESP32 上卻是 2048？ 這些疑問的答案都藏在 ADC 解析度與電壓映射的數學裡。

類比與數位的橋樑：什麼是 ADC？

世界是類比 (Analog) 的——溫度的高低、光線的強弱、聲音的大小，這些物理量都是連續且平滑地變化。然而，微控制器的核心是數位處理器，它只理解由 0 與 1 組成的離散數值。那麼，我們如何讓一個只懂「開」和「關」的晶片，去感知連續變化的真實世界呢？

答案就是 ADC（Analog-to-Digital Converter，類比數位轉換器）。ADC 就像一把「數位量尺」，它會在某個瞬間對輸入的類比電壓進行取樣，然後將該電壓值量化為一個整數。這個整數的大小取決於兩個關鍵參數：參考電壓（Reference Voltage）與解析度（Resolution）。

參考電壓決定了量尺的上限。ESP32 的 ADC 參考電壓預設為 3.3V，代表它能量測 0V 到 3.3V 之間的電壓。而解析度則決定了這把量尺被切成多少格——格數越多，量測越精細。

什麼是「位元數」與「解析度」？

ADC 的解析度以「位元數（bit）」表示。N 位元的 ADC 能將電壓範圍切分成 2N 個等級。以下是常見微控制器的 ADC 解析度比較：

開發板 ADC 位元數 數值範圍 最小可辨識電壓
Arduino Uno (ATmega328P) 10-bit 0 ~ 1,023 5V ÷ 1024 ≈ 4.88 mV
ESP32 12-bit 0 ~ 4,095 3.3V ÷ 4096 ≈ 0.81 mV
STM32 (部分型號) 16-bit 0 ~ 65,535 3.3V ÷ 65536 ≈ 0.05 mV
備註 ESP32 的 ADC 預設為 12-bit 解析度，但你也可以透過 analogReadResolution() 函式將其調整為 9-bit、10-bit 或 11-bit。降低解析度可以加快轉換速度，但會犧牲精度。在大多數應用中，保持預設的 12-bit 是最佳選擇。

電壓與數位讀值的換算

理解了位元數之後，你就能計算出任何 ADC 讀值對應的實際電壓。公式非常簡單：

電壓 (V) = ADC 讀值 × (參考電壓 ÷ 2N)

以 ESP32 為例，若 analogRead() 回傳 2048，代表的電壓為：2048 × (3.3 ÷ 4096) = 1.65V，恰好是 3.3V 的一半。同理，讀值 4095 理論上對應 3.3V，而讀值 0 則對應 0V。

int adcValue = analogRead(34);
float voltage = adcValue * (3.3 / 4096.0);
Serial.print("電壓: ");
Serial.print(voltage, 2);
Serial.println(" V");
提示 為什麼同一顆光敏電阻在 Arduino Uno 上讀到 512，在 ESP32 上卻是 2048？因為兩者解析度不同！512 ÷ 1024 = 0.5，而 2048 ÷ 4096 = 0.5——它們都代表參考電壓的 50%，只是刻度尺精細度不同而已。

數據映射：Map 函式的數學魔法

感測器讀到的原始 ADC 值（0–4095）對人類或周邊裝置來說毫無意義。馬達驅動器需要的是 0–255 的 PWM 佔空比；LCD 螢幕想顯示的是 0%–100% 的百分比；無人車的轉向邏輯需要的是 -90° 到 +90° 的角度值。數據映射（Mapping）就是把一組數值範圍按比例轉換到另一組數值範圍的過程。

Arduino 生態系提供了一個現成的函式 map()，讓你不需要自己動手算數學。但理解它背後的數學原理，是真正掌握數據處理的關鍵。

線性變換公式

map() 的本質是一條線性比例方程式。給定一個輸入值 x，從輸入範圍 [Inmin, Inmax] 映射到輸出範圍 [Outmin, Outmax]，公式如下：

y = (x − Inmin) × (Outmax − Outmin) ÷ (Inmax − Inmin) + Outmin

這個公式做了三件事：先將輸入值歸零（減去起點），再按比例縮放（乘以目標範圍 ÷ 原始範圍），最後加上輸出的偏移量。讓我們以實際範例來說明。

常見的映射場景

應用場景 輸入範圍 輸出範圍 程式碼
12-bit ADC → 8-bit PWM 0 ~ 4095 0 ~ 255 map(val, 0, 4095, 0, 255)
電池電壓 → 百分比 2800 ~ 4000 0 ~ 100 map(val, 2800, 4000, 0, 100)
搖桿 → 轉向角度 0 ~ 4095 -90 ~ +90 map(val, 0, 4095, -90, 90)
反向映射（越轉越暗） 0 ~ 4095 255 ~ 0 map(val, 0, 4095, 255, 0)

Arduino map() 實作範例

以下是一個將旋鈕讀值轉換為 LED 亮度的完整範例。旋鈕連接到 GPIO 34（ADC1 通道），LED 連接到 GPIO 16（PWM 輸出）：

const int POT_PIN = 34;
const int LED_PIN = 16;

void setup() {
  pinMode(LED_PIN, OUTPUT);
  Serial.begin(115200);
}

void loop() {
  int raw = analogRead(POT_PIN);
  int pwm = map(raw, 0, 4095, 0, 255);
  analogWrite(LED_PIN, pwm);
  delay(50);
}
注意 Arduino 的 map() 函式只進行整數運算，不支援浮點數。若你需要精確到小數的映射結果（例如電壓值），應自己用浮點數公式計算：float v = (float)(raw - inMin) * (outMax - outMin) / (inMax - inMin) + outMin;
提示 map() 不會自動限制輸出範圍。若輸入值超出預設範圍，輸出也會超出。你可以搭配 constrain() 函式來限制輸出：int safe = constrain(map(val, 0, 4095, 0, 255), 0, 255);

ESP32 ADC 的真實面貌

理論上，ADC 應該將 0V 到 3.3V 的電壓完美地對應到 0 到 4095 的數位值。但在真實世界中，ESP32 的 ADC 存在幾個你必須了解的「不完美」——忽略它們可能導致感測器數據嚴重失真，讓你的專案在關鍵時刻出錯。

非線性「死區」現象

ESP32 的 ADC 在電壓極低（接近 0V）與極高（接近 3.3V）的區間會表現出明顯的非線性行為。當實際電壓低於約 0.1V 時，ADC 讀值可能已經「黏」在 0 不動了；當電壓高於約 3.2V 時，讀值則提前飽和到 4095。這意味著你實際可用的「線性區」大約在 0.1V ~ 3.2V 之間。

這個特性在精密量測時影響尤為顯著。例如，你若想用 ADC 偵測電池是否已經完全放電（接近 0V），可能會發現讀值在 0.1V 以下就不再變化了。解決方案之一是使用分壓電路將感測器的電壓範圍縮限在 ADC 的線性區內。

注意 絕對不要將超過 3.3V 的電壓直接接到 ESP32 的 ADC 引腳！ESP32 不像 Arduino Uno 那樣有 5V 容忍度，超壓可能會永久損壞晶片內部的 I/O 電路。若你需要量測 5V 訊號，必須先用分壓電阻將電壓降至 3.3V 以下。

ADC2 與 Wi-Fi 的衝突

ESP32 擁有兩組 ADC 模組：ADC1（GPIO 32–39）與 ADC2（GPIO 0, 2, 4, 12–15, 25–27）。ADC1 是專屬的類比輸入通道，任何時候都能使用。然而，ADC2 的控制權會被 Wi-Fi 驅動程式搶走——只要 Wi-Fi 功能處於啟動狀態（即使只是 Wi-Fi.begin() 還沒連上），ADC2 通道就完全無法使用。

ADC 模組 可用引腳 Wi-Fi 開啟時 建議用途
ADC1 GPIO 32, 33, 34, 35, 36, 39 正常可用 感測器、搖桿等主要類比輸入
ADC2 GPIO 0, 2, 4, 12–15, 25–27 無法使用 僅在不需要 Wi-Fi 時使用
重要 若你的專案需要同時使用 Wi-Fi 與類比感測器，必須將所有感測器接在 ADC1 的引腳上（GPIO 32–39）。這是 ESP32 IoT 開發中最常被忽略的陷阱之一。

消除讀值抖動：移動平均濾波器

即使你的電路完全正確，ADC 讀值仍會有 ±10 到 ±30 的微小跳動。這是因為 ESP32 內部的電源雜訊、環境電磁干擾以及 ADC 本身的量化誤差所致。對於需要穩定輸出的應用（如馬達控制或顯示面板），你需要用軟體濾波來平滑數據。

最簡單且有效的方式是移動平均濾波器（Moving Average Filter）：維護一個固定長度的陣列，每次新增一筆讀值、丟棄最舊的一筆，然後取整個陣列的平均值作為輸出。

const int WINDOW = 10;
int readings[WINDOW];
int index = 0;
long total = 0;

int smoothRead(int pin) {
  total -= readings[index];
  readings[index] = analogRead(pin);
  total += readings[index];
  index = (index + 1) % WINDOW;
  return total / WINDOW;
}
提示 窗口大小（WINDOW）越大，輸出越穩定，但反應速度也越慢。一般感測器應用建議使用 10 次取樣；對於快速變化的搖桿輸入，可降為 4–5 次以保持即時性。

    • 反向邏輯挑戰：實作「越轉越暗」控制，學員需將 0–4095 映射為 255–0（PWM 佔空比）。
    • 雜訊抑制實作：實作一個 10 次採樣取平均的濾波器。
    • 提交結果：在 Serial Plotter 中對比原始數據與濾波數據。要求：濾波後的曲線必須顯著平穩。

任務 3：「精準轉向模擬器」死區補償 (Steering Simulator & Dead-band Lab)

  • 目標：為自動駕駛或機器人建立具備「容錯能力」的控制邏輯。
  • Classroom 部署建議：
    • 轉角映射：將讀值轉換為 -90° 至 +90° 的轉向角，並在 Serial 中輸出。
    • 死區 (Dead-zone) 補償：設定中心點 ±5 度的死區。當旋鈕保持在中位附近時，穩定輸出 0° 防止馬達微顫。
    • 差速邏輯計算：根據轉角，輸出兩台馬達的速差比例（例如：左轉時，左輪 80%，右輪 100%）。

3. 量化感知專家導師點評標準 (Sensing & Signal Processing Standards)

此單元的價值在於 「對原始數據的批判性處理與算法優化意識」：

  • [ ] 映射準確度：map() 函式的參數設定是否考慮了 12-bit (4095) 的邊界？
  • [ ] 濾波效率：採樣平均邏輯是否簡潔？是否有成功抑制高頻跳動感？
  • [ ] UX 思考深度：死區 (Dead-zone) 的設定是否合理地解決了「實體手感」與「數位控制」的摩擦？
