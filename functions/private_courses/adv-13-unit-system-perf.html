🔍 1. 偵測效能黑洞 (Bottlenecks)

控制系統的反應速度取決於整條鏈路中最慢的那一環。 即使你的 PID 演算法再精妙、馬達再強勁，如果主迴圈中有一行被遺忘的 Serial.print() 佔據了每次循環 12ms 中的 10ms，整台車的操控體驗就會像在水中移動一樣遲鈍。 性能優化的第一步，永遠是用數據找出瓶頸，而非憑直覺猜測。

嵌入式系統的性能瓶頸通常隱藏在三個層面：通訊層（藍牙頻寬與封包延遲）、 運算層（CPU 被阻塞式 I/O 佔用）、以及渲染層（前端 DOM 操作過於頻繁）。 本單元將教你使用 micros() 函數對 ESP32 主迴圈進行精確的時間埋點， 找出真正的「效能黑洞」。

使用 micros() 進行效能剖析

micros() 回傳自 ESP32 開機以來的微秒數，精度遠高於 millis()。 透過在主迴圈的各個關鍵區段前後插入時間戳記，你可以精確計算每個模組的執行耗時， 並算出其佔總循環時間的百分比。

void loop() {
  unsigned long t0 = micros();

  readSensors();
  unsigned long t1 = micros();

  processCommand();
  unsigned long t2 = micros();

  updateMotors();
  unsigned long t3 = micros();

  sendTelemetry(); // 包含 Serial.print — 常見瓶頸！
  unsigned long t4 = micros();

  // 每 100 次循環輸出一次性能報告
  static int count = 0;
  if (++count % 100 == 0) {
    Serial.printf("Sensor:%lu Cmd:%lu Motor:%lu Tele:%lu\n",
      t1-t0, t2-t1, t3-t2, t4-t3);
  }
}
常見陷阱 Serial.print() 在 115200 baud 下傳輸 100 個字元需要約 8.7ms—— 這在一個理想循環時間只有 10ms 的控制系統中，幾乎佔用了 87% 的 CPU 時間！ 解法是將 Serial 輸出改為條件式（如每 N 次才輸出一次）， 或直接透過 WebSocket 傳送遙測數據來取代 Serial。

常見瓶頸排行榜

瓶頸來源典型耗時佔比影響優化策略
Serial.print() (阻塞式)5–12 ms50–85%條件式輸出 / 改用非阻塞 UART
analogRead() (多通道)1–3 ms10–30%降低取樣頻率 / 使用 DMA
JSON 序列化0.5–2 ms5–20%改用二進位精簡協議
PID 計算10–50 μs<1%通常不是瓶頸
WiFi/BLE 事件處理0–5 ms不定使用回呼 (Callback) 而非輪詢
阿姆達爾定律 (Amdahl's Law) 優化應優先針對佔比最大的瓶頸。如果 Serial 佔了 85% 的時間， 即使你把 PID 計算速度提升 10 倍，整體效能也只會改善不到 1%。 先量測，再優化——這是性能工程的鐵律。

🧱 2. 數據壓縮：少即是多的智慧

無線傳輸的頻寬是有限且珍貴的資源。在 BLE (Bluetooth Low Energy) 環境中， 單次封包的有效載荷 (Payload) 通常只有 20–244 位元組。 如果你仍然使用像 {"velocity":100,"angle":45,"brake":false} 這樣的冗長 JSON 格式， 每個封包就要消耗 40+ 位元組——相當於把寶貴的頻寬浪費在大量的引號、冒號和鍵名上。

數據壓縮的核心哲學是「少即是多」： 透過設計更精簡的自定義協議，將相同的語意資訊用更少的位元組表達， 不僅能降低單次傳輸延遲，更能在相同時間內傳送更多筆指令，直接提升系統的控制更新率 (Hz)。

協議演進：從 JSON 到極簡格式

以下展示了三種不同壓縮層級的協議設計，以及它們對系統性能的影響：

協議類型封包範例大小壓縮率解析複雜度
原始 JSON {"velocity":100,"angle":45} 31 bytes — 中（需 JSON 解析器）
短字串協議 v100a45 7 bytes 77% 低（字串切割）
二進位協議 0x64 0x2D (2 bytes) 2 bytes 94% 極低（直接記憶體映射）

實作：短字串協議的編碼與解碼

短字串協議在壓縮率與可讀性之間取得了良好的平衡，非常適合教學與除錯場景：

// ESP32 端：解析短字串協議 "v100a45"
void parseCompactCmd(String msg) {
  int vIdx = msg.indexOf('v');
  int aIdx = msg.indexOf('a');
  if (vIdx < 0 || aIdx < 0) return;

  int vel = msg.substring(vIdx+1, aIdx).toInt();
  int ang = msg.substring(aIdx+1).toInt();

  vel = constrain(vel, 0, 255);
  ang = constrain(ang, -90, 90);
  applyControl(vel, ang);
}
// JavaScript 端：編碼短字串協議
function encodeCmd(velocity, angle) {
  return `v${velocity}a${angle}`;
}
二進位協議的取捨 雖然二進位協議的壓縮率最高，但它犧牲了人類可讀性。 在除錯階段，你無法直接用 Serial Monitor 查看封包內容。 建議在開發初期使用短字串協議，待系統穩定後再切換到二進位格式以獲取極致性能。
量化驗證 切換協議後，務必量測「數據刷新率 (Hz)」的實際提升幅度。 典型結果是從 JSON 的 10–15 Hz 提升到短字串的 40–60 Hz， 甚至二進位的 80–120 Hz。這個數據將是你性能報告中最有說服力的證據。

⚡ 3. 前端極速渲染：告別控制卡頓

當後端（ESP32）與通訊協議都已經過優化，最後一個可能拖慢系統的環節就是前端渲染。 如果你的 Dashboard 每收到一筆數據就直接操作 DOM——更新文字、重繪 Canvas、插入日誌—— 瀏覽器很快就會因為過度重繪 (Repaint) 與重排 (Reflow) 而卡頓。 當數據流達到每秒 50–100 筆時，未經優化的前端甚至會拖累整個控制鏈路。

高效能前端渲染的關鍵在於三個策略： 使用 requestAnimationFrame 同步螢幕刷新率、 採用離屏 Canvas 預渲染靜態圖層、以及 利用循環緩衝區管理數據流。 這三者結合，能確保即便數據洪流來襲，儀表板依然如絲般順滑。

requestAnimationFrame：與瀏覽器同步

requestAnimationFrame(callback) 會在瀏覽器下一次重繪之前調用你的回呼函數， 頻率通常為 60 次/秒（即 60 FPS）。相比直接用 setInterval 來驅動渲染， rAF 有兩個核心優勢：它會自動匹配螢幕刷新率，避免無謂的計算； 且在頁面不可見時會自動暫停，節省 CPU 資源。

let dataBuffer = [];
let needsRedraw = false;

// 數據接收：只更新 buffer，標記需要重繪
ws.onmessage = (e) => {
  dataBuffer.push(JSON.parse(e.data));
  if (dataBuffer.length > 200) dataBuffer.shift();
  needsRedraw = true;
};

// 渲染迴圈：僅在有新數據時才重繪
function renderLoop() {
  if (needsRedraw) {
    drawWaveform(dataBuffer);
    needsRedraw = false;
  }
  requestAnimationFrame(renderLoop);
}
requestAnimationFrame(renderLoop);

離屏 Canvas：預渲染靜態圖層

儀表板上的許多元素（如刻度盤背景、座標軸、標籤文字）是靜態的，不隨數據變動。 如果每一幀都重新繪製這些元素，就是在浪費 GPU 運算。 解法是使用 Offscreen Canvas 預先繪製靜態圖層， 然後在每一幀的渲染中只需用 drawImage() 把它「貼」上去，再疊加動態波形。

三種渲染策略的效能比較

渲染策略數據流 50 Hz數據流 100 HzCPU 佔用適用場景
直接 DOM 操作卡頓明顯近乎凍結極高低頻文字更新
Canvas + rAF流暢 (55+ FPS)偶有掉幀中等中頻圖表繪製
Offscreen + rAF + Buffer穩定 60 FPS穩定 60 FPS低高頻即時儀表板
Chrome DevTools 性能診斷 打開 Chrome DevTools 的 Performance 面板，錄製 5 秒的操作過程。 在火焰圖 (Flame Chart) 中觀察 Scripting（JavaScript 執行）佔比是否超過每幀 10ms。 如果超過，就需要進一步優化渲染邏輯。目標是讓每幀的總處理時間保持在 16.67ms 以內（60 FPS 的預算）。
垃圾回收 (GC) 的影響 頻繁建立與銷毀 JavaScript 物件會觸發瀏覽器的垃圾回收，造成不定期的微型凍結。 使用預分配的循環陣列 (Ring Buffer) 而非 push/shift， 可以顯著減少記憶體分配壓力，讓波形繪製更加穩定。

作業單元 13：系統性能分析 (System Performance Profiling)

歡迎來到 系統性能分析 挑戰。在這個單元中，你將學習如何診斷並優化嵌入式系統、通訊協議以及 Web 前端渲染的性能瓶頸。

🎯 學習目標

  1. 系統耗時診斷：使用嵌入式性能工具找出系統中的「性能黑洞」。
  2. 通訊協議優化：設計緊湊的二進位協議，壓榨無線頻寬的極限。
  3. 前端渲染加速：實作 Offscreen Canvas 與 requestAnimationFrame，確保高頻數據流下的流暢體驗。

🚀 任務說明

任務 1：系統耗時診斷 (System Profiling)

  • 在 src/main.ino 的主迴圈中使用 micros() 進行時間標記。
  • 定位出執行最耗時的模組（例如過度的 Serial.print 或頻繁的感測器讀取）。
  • 優化程式碼並記錄優化前後的 Loop Time (ms)。

任務 2：通訊協議瘦身 (Protocol Compression)

  • 將原本基於 JSON 的指令重構為緊湊的二進位或短字串格式（參考 include/protocol_v2.h）。
  • 驗證數據刷新率（Hz）是否顯著提升。
  • 在 web/ 目錄中實作相對應的高效解碼器。

任務 3：Canvas 渲染加速 (UI Rendering Optimization)

  • 使用 Chrome DevTools 的 Performance 面板診斷渲染卡頓。
  • 實作離屏緩衝 (Offscreen Canvas) 技術。
  • 確保儀表板即便在數據流高達 100 筆/秒時，依然能穩定維持在 60 FPS。

📂 程式庫結構

  • include/：包含性能監控與協議定義標頭檔。
  • src/：嵌入式端主程式與性能分析工具。
  • web/：具備渲染優化特性的前端儀表板。
  • docs/：請在此放置你的性能分析報告。

📝 繳交要求

  1. 程式碼：確保所有優化邏輯已實作並通過測試。
  2. 性能報告：在 docs/profiling_report.md 中詳細列出：
    • 優化前的瓶頸分析。
    • 所採取的優化手段。
    • 優化後的數據对比 (Loop Time, FPS, Throughput)。
  3. PR 說明：提交 PR 時，請附上性能提升的截圖或螢幕錄影。

掌握性能優化，是你邁向「全疊堆高級開發者」的關鍵一步。加油！

針對 adv-13-unit-system-perf (系統性能分析) 單元，這是邁向「全疊堆高級開發者」的最終磨練。它不只要求程式能跑，更要求程式「極致高效」。這涉及到從嵌入式 CPU 的微秒級埋點量測，到通訊協議的二進位壓縮，再到瀏覽器 GPU 的 Canvas 渲染加速。

1. 範本程式庫配置建議

性能優化需要精確的對比數據與分析工具，建議範本包含：

  • 📂 src/SystemProfiler.cpp：性能量測工具。預置多個計時節點，能自動計算每個模組（如 PID、感測器讀取）的平均執行時間。
  • 📂 include/protocol_v2.h：高效協議定義。提供一個框架，讓學員將原本的 JSON 指令重構為緊湊的二進位或短字串格式。

  • 📂 PERFORMANCE_STATS.md：性能對比白皮書。要求學員在優化前後，紀錄「Loop Time (ms)」、「通訊頻率 (Hz)」與「UI 幀率 (FPS)」。

2. 作業任務部署細節

任務 1：系統耗時診斷 (System Profiling Lab)

  • 目標：學會使用嵌入式性能工具找出躲在主快取中的「性能黑洞」。
  • Classroom 部署建議：
    • 模組耗時計數：學員需在主迴圈內使用 micros() 進行時間戳記埋點。
    • 瓶頸定位報告：在 README 中列出排行榜。通常會發現 Serial.print() 或過度頻繁的 AnalogRead 是最大瓶頸。
    • PR 提交內容：提交一份包含分析數據的 PR。解釋為什麼某個區塊佔用了 70% 的 CPU 時間。

任務 2：通訊協議瘦身術 (Protocol Compression Lab)

  • 目標：透過設計極簡协议，在有限的無線頻寬下實現更高的反饋頻率。
  • Classroom 部署建議：
    • 協議重構實作：將原本肥大的 {"speed":100,"angle":45} 壓縮為緊湊協議（如 v100a45）。
    • 傳輸吞吐量驗證：要求學員對比在相同環境下，優化後的數據刷新率是否提升了 3-5 倍。
    • 解碼器開發：同步開發 Javascript 端的高效解碼函數，確保通訊鏈路的全線路通暢。

任務 3：Canvas 渲染加速挑戰 (High-performance UI Rendering)

  • 目標：優化前端顯示，確保在高頻數據流衝擊下操控手感依然順滑。
  • Classroom 部署建議：
    • 渲染卡頓診斷：使用 Chrome DevTools 錄製 Performance Profile。
    • 非同步渲染實作：實作 requestAnimationFrame (rAF) 同步技術。
    • 離屏緩衝技術：繪製靜態背景與標籤於 Offscreen Canvas，動態波形僅重繪變動部分，大幅降低 GPU 負載。
    • Final Video：展示即便數據流高達 每秒 100 筆，儀表板也毫無延遲、流暢運行的最終成果。

3. 系統架構師導師點評標準 (Full-Stack Performance Benchmarks)

此單元的價值在於 「對全系統資源極限的掌控力」：

  • [ ] 數據說服力：優化前後的數據是否有顯著提升 (如 Loop 時間下降、FPS 穩定在 60)？
  • [ ] 協議設計精妙度：自定義協議是否兼顧了「節省頻寬」與「易於解析」的平衡？
  • [ ] 架構優雅性：優化手段是否引入了過多複雜度？還是透過更聰明的設計達成高效？
