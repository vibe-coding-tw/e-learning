打破速度的天花板：從毫秒計較到數據瘦身，壓榨系統的最後一滴性能。

🔍 1. 偵測效能黑洞 (Bottlenecks)

系統的速度取決於「最慢」的那一環。即使馬達再強，如果感測器讀取或數據傳輸卡住，車子表現就會像「Lag」一樣遲鈍。

The Three Efficiency Layers

  • 通訊瓶頸：藍牙頻寬不足或封包內容過於肥大。
  • 運算瓶頸：ESP32 主迴圈內含有阻塞代碼（如 Serial 輸出過多）。
  • 渲染瓶頸：瀏覽器端過度重繪 (Repaint) 導致指令堆疊。

🧱 2. 數據壓縮：少即是多的智慧

藍牙 MTU 封包長度有限。透過自定義精簡協議（如將 JSON 縮減為二進位），能直接將系統更新率提昇 3-5 倍。

⚡ 3. 前端極速渲染：告別 UI 卡頓

瀏覽器在處理大量即時波形時容易因 Canvas 頻繁重繪而當機。掌握緩衝策略與 Frame Rate 限制，確保操控手感如絲般順滑。

Optimization Toolkit

  • 离屏渲染 (Offscreen)：將靜態格線預先繪製在隱藏畫布，減少重複運算。
  • 畫面節流 (Throttling)：使用 requestAnimationFrame 鎖定 60fps，防止 CPU 爆增。
  • 記憶體管理：預先建立 Buffer 陣列重複使用，避免頻繁觸發 GC 導致的小停頓。

任務 1：「瓶頸偵探」：控制循環性能分析 (Bottleneck Detective: Controller Profiling)

角色扮演：系統性能分析師 | 難度：⭐️⭐️ | 估計時長：50 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 性能埋點：在 loop() 的各關鍵節點（讀感測器、PID、BLE 打包）埋入 micros() 計時器。
  3. 壓力監測：透過序列埠印出每項任務的執行微秒數。分析在哪種感測器配置下系統延遲最高。
  4. 診斷日誌：在 README 中列出效能對比表，並指出一項顯著增加 Loop 負擔的「無效代碼」行為。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 數據顆粒度：統計數據是否精確到微秒 (\(\mu s\)) 級別，且能區分不同的運算模組？
  • [ ] 瓶頸識別能力：README 是否成功指出導致控制循環降低的真正原因（如 Serial 頻率過高或複雜判斷式）？

任務 2：「指令瘦身」：極簡通訊協議優化 (Payload Slimming: Protocol Optimization)

角色扮演：通訊協議專家 | 難度：⭐️⭐️⭐️ | 估計時長：70 min

🛠️ 執行技術流程 (Git Workflow)

  1. 協議重構：放棄傳統 JSON 傳輸。設計一組高密度的字元協議（如 "v150a20" 代表速度與轉角）。
  2. 前端解碼實作：在 Dashboard 代碼中實作解析 Split 邏輯，從簡短字串中還原遙測數據。
  3. 吞吐量壓測：將 ESP32 的發送頻率提升至每秒 100 次，觀察與原本 JSON 格式相比的封包丟失紀錄。
  4. PR 提交：推送到 GitHub 並在代碼註解中說明此協議設計如何節省藍牙傳輸的 MTU 空間。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 壓縮效率：封包長度是否相比原始版本減少了 60% 以上？
  • [ ] 可靠性驗證：在高速傳輸下，前端 Dashboard 指令解析是否依然 100% 正確無訛？

任務 3：「前端渲染加速」：Canvas 性能優化 (Rendering Acceleration)

角色扮演：前端優化大師 | 難度：⭐️⭐️ | 估計時長：40 min

🛠️ 執行技術流程 (Git Workflow)

  1. 性能錄製：利用 Chrome DevTools 的 Performance 面板，記錄數據湧入時主執行緒的 Scripting 佔比。
  2. 繪圖邏輯優化：實作 requestAnimationFrame() 以對齊螢幕刷新率。避免在每一筆數據到達時都觸發重繪。
  3. 離屏緩衝策略：將不變動的 UI（如：背景網格、座標軸）移至個別畫布或預繪製，大幅降低每一幀的運算。
  4. 對比 Push：將優化前後的 DevTools 截圖附在 docs 資料夾中併入 Commit。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 幀率穩定性：優化後的介面在高頻數據更新下，是否仍能維持穩定的 60FPS 渲染效果？
  • [ ] 資源消耗改善：CPU 佔用率（尤其是渲染主執行緒）是否有明顯下降趨勢？

GitHub Classroom 詳細操作指南

本單元教學重點：在 IoT 系統中，如何建立「全鏈路優化」的思維，從硬體到雲端的每一環節壓榨效能。

  1. 重點 1：數據驅動的優化 - 強調「先測量、後優化」。透過計時器埋點，讓學生學會用數據講話，而非憑直覺刪減代碼。
  2. 重點 2：從文字到位元 (Bits) - 引導學員理解協議設計的藝術。雖然 JSON 易讀，但在極限頻寬下，自定義二進位或緊湊協議才是工業標準。
  3. 重點 3：感知性能 (Perceived Performance) - 指導學員理解前端渲染優化的價值：流暢的 UI 能大幅提升使用者的操控信任感。
