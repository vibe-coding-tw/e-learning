🔍 1. 偵測效能黑洞 (Bottlenecks)

控制系統的速度取決於「最慢」的那一環。即使馬達再快，如果感測器讀取或數據傳輸卡住，控制表現就會像「Lag」一樣遲鈍。

The Three Efficiency Layers

  • 通訊瓶頸 (Latency)：藍牙頻寬不足或封包內容過於肥大導致延遲。
  • 運算瓶頸 (CPU)：主迴圈內含有過多阻塞式 I/O 或複雜數學運算。
  • 渲染瓶頸 (Frontend)：瀏覽器過度重繪與 DOM 更新導致指令響應緩慢。

⚡ 3. 前端極速渲染：告別控制卡頓

瀏覽器在處理海量即時遙測數據時，容易因 Canvas 頻繁刷新而阻塞。透過渲染緩衝 (Buffering) 與 Frame Rate 管理，確保操控手感如絲般順滑。

Advanced Rendering Strategy

  • 離屏渲染 (Offscreen Draw)：將靜態圖層預先繪製，減少重複渲染運算負荷。
  • 請求幀節流 (rAF)：同步螢幕刷新率，防止過度計算造成的 CPU 資源浪費。
  • Buffer 管理：預建循環隊列 Buffer，優化記憶體分配並減少 GC 引起的抖動。

作業單元 13：系統性能分析 (System Performance Profiling)

歡迎來到 系統性能分析 挑戰。在這個單元中，你將學習如何診斷並優化嵌入式系統、通訊協議以及 Web 前端渲染的性能瓶頸。

🎯 學習目標

  1. 系統耗時診斷：使用嵌入式性能工具找出系統中的「性能黑洞」。
  2. 通訊協議優化：設計緊湊的二進位協議，壓榨無線頻寬的極限。
  3. 前端渲染加速：實作 Offscreen Canvas 與 requestAnimationFrame，確保高頻數據流下的流暢體驗。

🚀 任務說明

任務 1：系統耗時診斷 (System Profiling)

  • 在 src/main.ino 的主迴圈中使用 micros() 進行時間標記。
  • 定位出執行最耗時的模組（例如過度的 Serial.print 或頻繁的感測器讀取）。
  • 優化程式碼並記錄優化前後的 Loop Time (ms)。

任務 2：通訊協議瘦身 (Protocol Compression)

  • 將原本基於 JSON 的指令重構為緊湊的二進位或短字串格式（參考 include/protocol_v2.h）。
  • 驗證數據刷新率（Hz）是否顯著提升。
  • 在 web/ 目錄中實作相對應的高效解碼器。

任務 3：Canvas 渲染加速 (UI Rendering Optimization)

  • 使用 Chrome DevTools 的 Performance 面板診斷渲染卡頓。
  • 實作離屏緩衝 (Offscreen Canvas) 技術。
  • 確保儀表板即便在數據流高達 100 筆/秒時，依然能穩定維持在 60 FPS。

📂 程式庫結構

  • include/：包含性能監控與協議定義標頭檔。
  • src/：嵌入式端主程式與性能分析工具。
  • web/：具備渲染優化特性的前端儀表板。
  • docs/：請在此放置你的性能分析報告。

📝 繳交要求

  1. 程式碼：確保所有優化邏輯已實作並通過測試。
  2. 性能報告：在 docs/profiling_report.md 中詳細列出：
    • 優化前的瓶頸分析。
    • 所採取的優化手段。
    • 優化後的數據对比 (Loop Time, FPS, Throughput)。
  3. PR 說明：提交 PR 時，請附上性能提升的截圖或螢幕錄影。

掌握性能優化，是你邁向「全疊堆高級開發者」的關鍵一步。加油！

針對 adv-13-unit-system-perf (系統性能分析) 單元，這是邁向「全疊堆高級開發者」的最終磨練。它不只要求程式能跑，更要求程式「極致高效」。這涉及到從嵌入式 CPU 的微秒級埋點量測，到通訊協議的二進位壓縮，再到瀏覽器 GPU 的 Canvas 渲染加速。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫配置建議

性能優化需要精確的對比數據與分析工具，建議範本包含：

  • 📂 src/SystemProfiler.cpp：性能量測工具。預置多個計時節點，能自動計算每個模組（如 PID、感測器讀取）的平均執行時間。
  • 📂 include/protocol_v2.h：高效協議定義。提供一個框架，讓學員將原本的 JSON 指令重構為緊湊的二進位或短字串格式。
  • 📂 web/js/high_speed_canvas.js：空白的渲染優化檔案。學員需在此實作離屏渲染與 requestAnimationFrame 邏輯。
  • 📂 PERFORMANCE_STATS.md：性能對比白皮書。要求學員在優化前後，紀錄「Loop Time (ms)」、「通訊頻率 (Hz)」與「UI 幀率 (FPS)」。

2. 作業任務部署細節

任務 1：系統耗時診斷 (System Profiling Lab)

  • 目標：學會使用嵌入式性能工具找出躲在主快取中的「性能黑洞」。
  • Classroom 部署建議：
    • 模組耗時計數：學員需在主迴圈內使用 micros() 進行時間戳記埋點。
    • 瓶頸定位報告：在 README 中列出排行榜。通常會發現 Serial.print() 或過度頻繁的 AnalogRead 是最大瓶頸。
    • PR 提交內容：提交一份包含分析數據的 PR。解釋為什麼某個區塊佔用了 70% 的 CPU 時間。

任務 2：通訊協議瘦身術 (Protocol Compression Lab)

  • 目標：透過設計極簡协议，在有限的無線頻寬下實現更高的反饋頻率。
  • Classroom 部署建議：
    • 協議重構實作：將原本肥大的 {"speed":100,"angle":45} 壓縮為緊湊協議（如 v100a45）。
    • 傳輸吞吐量驗證：要求學員對比在相同環境下，優化後的數據刷新率是否提升了 3-5 倍。
    • 解碼器開發：同步開發 Javascript 端的高效解碼函數，確保通訊鏈路的全線路通暢。

任務 3：Canvas 渲染加速挑戰 (High-performance UI Rendering)

  • 目標：優化前端顯示，確保在高頻數據流衝擊下操控手感依然順滑。
  • Classroom 部署建議：
    • 渲染卡頓診斷：使用 Chrome DevTools 錄製 Performance Profile。
    • 非同步渲染實作：實作 requestAnimationFrame (rAF) 同步技術。
    • 離屏緩衝技術：繪製靜態背景與標籤於 Offscreen Canvas，動態波形僅重繪變動部分，大幅降低 GPU 負載。
    • Final Video：展示即便數據流高達 每秒 100 筆，儀表板也毫無延遲、流暢運行的最終成果。

3. 系統架構師導師點評標準 (Full-Stack Performance Benchmarks)

此單元的價值在於 「對全系統資源極限的掌控力」：

  • [ ] 數據說服力：優化前後的數據是否有顯著提升 (如 Loop 時間下降、FPS 穩定在 60)？
  • [ ] 協議設計精妙度：自定義協議是否兼顧了「節省頻寬」與「易於解析」的平衡？
  • [ ] 架構優雅性：優化手段是否引入了過多複雜度？還是透過更聰明的設計達成高效？
