Web Bluetooth 是一個強大的 API，它允許網頁直接控制實體設備。為了防止惡意網站偷偷連線你的設備，瀏覽器強制要求網站必須在 HTTPS 環境下才能執行。

Web BLE 三大安全鐵律

Module 2: Core Principles
🔐

1. 環境限制

除了 localhost 用於開發測試外，所有上線的 Web App 必須部署在具備 SSL 憑證 (HTTPS) 的伺服器上。

  • ✔ localhost
  • ✔ 127.0.0.1
  • ✔ https://...
  • ✘ http://192.168.x.x
👆

2. 使用者手動觸發

瀏覽器禁止網頁在背景「自動」掃描。所有的連線請求必須由使用者明確的點擊 (User Gesture) 觸發。

btn.addEventListener('click', function() {
  // 必須在這裡呼叫
  navigator.bluetooth.requestDevice(...)
});
🔎

3. 設備過濾與隱私

網頁不能掃描「所有」設備。必須明確宣告要存取的 filters 或 optionalServices，否則瀏覽器會拒絕執行。

隱私保護：
這防止了廣告商透過掃描你身邊的藍牙設備 (如智慧手環) 來追蹤你的位置。

Web BLE 安全機制與權限規範 (Web BLE Security & Permissions)

歡迎來到 Start 02 Unit: BLE Security！ 在這個單元中，我們將學習 Web Bluetooth API 在瀏覽器沙箱中的各項限制（如 HTTPS、User Gesture 與設備過濾），並學習如何在多人高干擾環境下保護使用者的通訊主權，進行防禦性編程與精準過濾。

🎯 作業目標 (Assignment Objectives)

本作業分為三個主要任務，幫助你掌握 Web BLE 的安全機制：

任務 1：「授權失敗」模擬與排除

  • 實作例外處理：使用 try...catch 包裹 requestDevice()，捕捉並處理不同的 Web BLE 錯誤碼（例如 NotFoundError 或是 SecurityError）。
  • 確保介面友善：當使用者拒絕授權或藍牙關閉時，網頁必須顯示「人類語言」的提示，而非在背景靜默失敗。

任務 2：多裝置環境下的「精準過濾」

  • 設定過濾條件：利用 filters (例如 namePrefix 或 services UUID) 取代 acceptAllDevices: true，確保只掃描並連線到你的目標設備。
  • 防止數位誤入：在多人同時操作的環境下，你的程式碼應該只顯示你專屬的設備，降低連線錯誤的風險。

任務 3：跨平台權限大調查

  • 多裝置與跨平台測試：使用不同的設備（Android、iOS、macOS、Windows 等）測試你的網頁。
  • 分析權限差異：觀察各個平台在請求藍牙與定位權限時的不同行為，將觀察到的現象與藍牙配對對話框的截圖記錄在你的報告中。

  1. app.js 與 index.html：

  2. screenshots/ 資料夾：

2. 作業任務部署細節

任務 1：「授權失敗」模擬與排除 (Permission Error Handling Lab)

  • 目標：實施防禦性編程。確保當使用者拒絕授權或藍牙關閉時，網頁能給出正確的「人類語言」提示，而非靜默失敗。
  • Classroom 部署建議：
    • 例外處理實作：包裹 requestDevice()。針對特定的錯誤類型（如 NotFoundError 代表取消、SecurityError 代表非主動觸發）撰寫對應的 UI 提示。
    • 取消實驗重現：手動點擊連線後按下「取消」，驗證頁面是否能顯示「請選擇設備以繼續」。
    • 驗證要點：關鍵健壯性審核。代碼是否能區分不同的 Web BLE 錯誤碼？UI 回饋是否能引導使用者下一步該怎麼做？

任務 2：多裝置環境下的「精準過濾」 (Precision Filtering Lab)

  • 目標：實踐隱私最小化原則。在多台機器人同時廣播的環境下，透過過濾條件確保只連線到目標設備，防止「數位誤入」。
  • Classroom 部署建議：
    • 過濾字典設計：利用 namePrefix (如 Car-) 或特定的 services UUID 來過濾列表。
    • 多人情境模擬：全班同時開啟藍牙，學員需修改代碼使其「只顯示」標有自己編號的機器人。
    • 驗證要點：學員是否理解了 acceptAllDevices: true 的潛在風險？過濾器的實作是否能有效減少清單噪音？

任務 3：跨平台權限大調查 (Cross-platform Audit Lab)

  • 目標：拓寬工程視野。理解 Web 標準在不同作業系統（Android vs iOS vs MacOS）與瀏覽器核心下的實作差異。
  • Classroom 部署建議：
    • 多裝置對比：使用不同品牌的手機或電腦造訪專案，截下各平台的「設備選擇器」樣式。
    • 權限路徑分析：在 README 中紀錄哪些平台要求先開啟「定位」，哪些平台強制要求 HTTPS。
    • 驗證要點：提交的報告中是否有明確的平台差異觀察（如：iOS 必須透過專用 App 才能開啟 Web BLE）？

3. 安全合規與異常處理點評標準 (Security Compliance & Handling Standards)

此單元的價值在於 「對 Web 安全約束的尊重與對用戶隱私操作的細膩處理」：

  • [ ] 沙箱理解深度：是否能正確解釋為什麼 Web BLE 禁止在背景自動連線（User Gesture 限制）？
  • [ ] 代碼健壯度：當硬體藍牙無預警關閉時，網頁前端是否具備自檢與自癒（給予重試指引）的能力？
  • [ ] 相容性敏感度：是否對「 Web BLE 在 iOS 上的局限性」有明確的認知與技術對策？
