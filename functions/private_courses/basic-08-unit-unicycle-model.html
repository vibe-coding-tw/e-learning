簡介

搖桿給出的是「前進速度 v」與「轉向角速度 ω」，但雙馬達差速車需要的是左右輪各自的轉速。 Unicycle Model 正是這座橋樑——用一條簡潔的線性公式，將 (v, ω) 轉換為 (v_L, v_R)， 讓操控者只需思考「方向」與「速度」，不必手動分配左右馬達出力。

情境

什麼是 Unicycle Model？

在機器人學中，Unicycle Model（獨輪車模型）是一種經典的運動學抽象—— 它將一台擁有兩個馬達的差速車簡化為一個在平面上移動的「質點」。這個質點只需要兩個參數就能完整描述其運動狀態： 線速度 \(v\)（前進或後退的速度）和 角速度 \(\omega\)（旋轉的速度）。

為什麼要做這層抽象？因為人類的直覺是「我要往哪走」和「我要怎麼轉」， 而不是「左輪要轉多快」和「右輪要轉多快」。Unicycle Model 讓操控者只需思考方向與速度， 底層的左右馬達分配交給數學公式自動完成。

兩個核心參數

參數 符號 物理意義 搖桿對應 正值方向
線速度 \(v\) 機器人整體前進/後退速度 搖桿 Y 軸 向前為正
角速度 \(\omega\) 機器人旋轉/轉向速度 搖桿 X 軸 向右為正（順時針）

在歸一化的搖桿系統中，\(v\) 和 \(\omega\) 的範圍都是 [-1.0, 1.0]。 搖桿推到最上方時 \(v = 1.0\)（全速前進），推到最右邊時 \(\omega = 1.0\)（全速右轉）。 這兩個值的組合描述了機器人所有可能的運動狀態——直線、轉彎、原地旋轉。

備註 Unicycle Model 的名稱來自於「獨輪車」——它假設機器人只有一個接觸點與地面互動， 就像騎獨輪車的人一樣。這種簡化讓運動學公式非常優雅，但要記住它是一個近似模型， 實際的雙輪差速車在高速轉彎時會有滑動等非理想行為。

從搖桿到運動意圖

搖桿的二維座標天然對應了 Unicycle Model 的兩個參數：Y 軸控制「去哪」（前進/後退）， X 軸控制「怎麼轉」（左轉/右轉）。這種映射讓操控非常直覺——推搖桿的方向就是車子要去的方向。 接下來的問題是：如何把 \((v, \omega)\) 這對高階指令，拆解成左右馬達各自的轉速？ 這就是下一節「差速驅動與混控公式」的核心課題。

提示 在實作中，建議先將搖桿的 X/Y 原始像素偏移做歸一化（上一個單元的技術）， 再直接作為 \(\omega\) 和 \(v\) 使用。這樣整條數據處理管線就是： 原始像素 → 歸一化 → (v, ω) → 左右馬達轉速。

差速驅動與混控公式

差速驅動 (Differential Drive) 是雙馬達機器人最常見的驅動方式：左右兩輪各自由獨立的馬達控制， 透過兩輪之間的速度差來實現轉向。當左右輪轉速相同時，車子直行； 當左輪比右輪快時，車子向右偏轉；反之則向左。

Unicycle Model 的核心公式將 \((v, \omega)\) 轉換為左右馬達的個別轉速：

\[L = v - \omega\] \[R = v + \omega\]

其中 \(L\) 是左輪轉速，\(R\) 是右輪轉速，\(v\) 是線速度（前進/後退）， \(\omega\) 是角速度（轉向）。當 \(\omega > 0\)（向右轉）時，右輪減速、左輪加速， 產生向右的轉彎效果。

典型輸入與輸出對照

下表展示了不同搖桿指令下，混控公式的運算結果：

操作意圖 \(v\) \(\omega\) 左輪 \(L\) 右輪 \(R\) 車輛行為
全速前進 1.0 0.0 1.0 1.0 直線前進
全速後退 -1.0 0.0 -1.0 -1.0 直線倒退
原地右旋 0.0 1.0 -1.0 1.0 坦克式順時針旋轉
原地左旋 0.0 -1.0 1.0 -1.0 坦克式逆時針旋轉
前進 + 右轉 0.8 0.3 0.5 1.1 前進中的右弧線（注意溢出）
警告 注意最後一列：當 \(v = 0.8\) 且 \(\omega = 0.3\) 時，右輪算出 1.1， 超出了馬達的最大轉速範圍 [-1.0, 1.0]。這就是「飽和溢出」問題， 下一節將介紹如何用等比例縮放來解決。

程式碼實作

以下是混控公式的基礎 C++ 實作，適用於 ESP32 開發板：

// Unicycle Model 混控器
void applyMix(float v, float omega) {
  float left  = v - omega;
  float right = v + omega;

  // 將 [-1.0, 1.0] 映射到 PWM [0, 255] + 方向
  motorL.setSpeed(left  * 255);
  motorR.setSpeed(right * 255);
}
提示 驗證混控邏輯最簡單的方法是測試「原地旋轉」：輸入 \(v = 0,\;\omega \neq 0\)， 檢查左右輪的轉速是否大小相等、方向相反。如果不對稱，說明公式或馬達接線有誤。

飽和處理：限幅思維

上一節的混控公式 \(L = v - \omega\)、\(R = v + \omega\) 有一個隱藏的陷阱： 當線速度和角速度都很大時（例如全速前進同時急轉彎），計算出的左右輪轉速可能超過馬達的物理極限 [-1.0, 1.0]。這就是飽和溢出 (Saturation Overflow) 問題。

為什麼不能直接裁切？

最直覺的做法是把超出範圍的值「裁切」到 ±1.0（Clipping）。但這會破壞左右輪之間的速度比例， 導致轉向意圖喪失。以下例子說明了這個問題：

處理方式 原始 L 原始 R 處理後 L 處理後 R 轉向比 (R-L)
無處理 0.5 1.3 — — 0.8
直接裁切 0.5 1.3 0.5 1.0 0.5 (被壓縮!)
等比例縮放 0.5 1.3 0.385 1.0 0.615 (比例保留)

直接裁切讓轉向比從 0.8 被壓縮到 0.5——車子在全速行駛時「轉不動」了。 這就是所謂的控制鎖死 (Control Lock)，在實際操控中會讓駕駛者感覺車子不聽使喚。

重要 等比例縮放的核心理念是：寧可犧牲整體速度，也要保留轉向意圖。 這是一個專業的工程取捨——在極限動力下，「能轉彎」比「全速直行」更重要。

等比例縮放演算法

演算法的關鍵在於使用 max(|L|, |R|, 1.0) 作為縮放分母：

  1. 計算原始的 \(L = v - \omega\) 和 \(R = v + \omega\)
  2. 找出兩者絕對值的最大值：scale = max(|L|, |R|, 1.0)
  3. 如果 scale > 1.0（有溢出），將 L 和 R 同時除以 scale
  4. 如果 scale ≤ 1.0（無溢出），保持原值不變
// 帶飽和保護的混控器
void mixWithSaturation(float v, float omega) {
  float left  = v - omega;
  float right = v + omega;

  // 等比例縮放：保護轉向比例
  float scale = max(max(fabs(left), fabs(right)), 1.0f);
  left  /= scale;
  right /= scale;

  motorL.setSpeed(left  * 255);
  motorR.setSpeed(right * 255);
}
提示 分母中的 1.0 確保了在沒有溢出時不會進行不必要的縮放。 這是一個精巧的設計：max(|L|, |R|, 1.0) 在正常情況下等於 1.0（不改變數值）， 只有在溢出時才大於 1.0（啟動縮放）。

差速轉換數學 (Unicycle Model)

本單元的目標是學習如何將機器人的抽象運動指令（線速度 $v$ 與角速度 $\omega$）轉換為實體的左、右馬達輸出。這是一個關鍵的技術里程碑，你將學習如何處理硬體限制並守護操縱意圖。

任務目標

任務 1：基礎差速混合器實作 (Kinematics Mixer)

  • 目標：實作 Unicycle Model 到差速傳動的幾何映射。
  • 實作檔案：src/mix_engine.cpp
  • 公式：
    • $L = v - \omega$
    • $R = v + \omega$
  • 驗證：確保當 $\omega > 0$（向右轉）時，右輪轉速降低，左輪轉速增加。

任務 2：飽和度處理與比例縮放 (Saturation Scaling)

  • 目標：在馬達輸出達到上限時，透過等比例縮放來保護轉向意圖。
  • 挑戰：當 $v + \omega > 1.0$ 時，直接裁切 (Clipping) 會導致轉向失效。
  • 解決方案：實作 output = target / max(|L|, |R|, 1.0)。

任務 3：轉向敏感度調教 (Dynamic Sensitivity)

  • 目標：引入轉向係數 $K$，根據物理環境調整操縱感。
  • 公式：$\omega' = K \cdot \omega$

  2. 打開 src/mix_engine.cpp。
  3. 根據 include/mixer_def.h 中的定義完成 mix_unicycle 函式。
  4. 運行測試以驗證你的實作： 
    pio test -e native
