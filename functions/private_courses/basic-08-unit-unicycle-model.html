基礎 8 - 運動學轉換

差速轉換數學：
Unicycle Model

將抽象的速度與轉向指令，精準解構成左右馬達的動力輸出來源。

🚲 1. 什麼是 Unicycle Model？

將複雜的機器人簡化為一個運動點。我們不直接控制馬達轉速，而是控制兩個宏觀參數：

v 線速度 (Linear)

代表整體前進/後退速度。在搖桿上通常對應 Y 軸。

ω 角速度 (Angular)

代表自轉或轉向速度。在搖桿上通常對應 X 軸。

Velocity Decomposition

"這是建立直覺操控的基礎：將『想去哪裡』與『怎麼轉』解耦。"

MotorMix = f(v, ω)

⚙️ 2. 差速驅動與混控公式

左右兩輪獨立驅動。要讓車轉向，必須在左輪與右輪之間產生「速度差」。

\[L = v - \omega\]
Left Motor
\[R = v + \omega\]
Right Motor
MIXER LOGIC (C++) 
void applyMix(float v, float w) {
  float left = v - w;
  float right = v + w;
  motorL.setSpeed(left * 255);
  motorR.setSpeed(right * 255);
}

🛑 3. 飽和處理：工程師的限幅思維

若 $v=1.0, \omega=0.5$，算出的 $R = 1.5$。但馬達最大只有 1.0 (255 PWM)，若直接裁切 (Clipping) 會導致車輪鎖死轉向失靈。

Normalizing Mixers

  • 等比例縮放：當數值溢出時，將左右馬達同步按比例縮小，確保轉向比 (ω) 被保留。
  • 防裁切機制：這讓車子在全速行駛時仍能做出順滑的蛇行轉向。
  • 飽和度判定：使用 max(|L|, |R|, 1.0) 作為縮放的分母。

Robust Control

"比例大於絕對值。在極限輸出下保留轉向意圖，是控制理論的核心優雅。"

任務 1：基礎差速混合機實作 (Kinematics Mixer)

角色扮演：軟體運動學工程師 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 線性混控器編寫：在驅動代碼中實作 v, ω 轉換公式：L = v - ω 與 R = v + ω。
  3. 原地自旋校準：輸入 v=0, ω=0.5，觀察車子是否能精準原地空轉。若左輪前進右輪後退，則方向正確。
  4. 物理直覺同步：驗證搖桿往右推時，右輪轉速應低於左輪。Push 具備正確運動學邏輯的代碼。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 自旋對稱性：原地旋轉測試中，左右輪的 PWM 數值是否嚴格對應且方向完全相反？

任務 2：飽和度處理與比例縮放 (Saturation Scaling)

角色扮演：控制系統優化專家 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 飽和極限診斷：全速前進時嘗試急轉向，記錄單純數值截斷 (Clipping) 造成的控制失效現象。
  2. 縮放演算法開發：實作等比例縮放邏輯：當 max(|L|, |R|) > 1.0 時，將兩輪轉速同步縮放，保留轉向比。
  3. 高速穩定度路測：在極限速度 (v=100%) 場景下執行蛇行，驗證轉向精確度是否顯著提升。
  4. 模塊化交付：將具備限幅保護能力的高階 MotorMixer 類別推送到 GitHub 倉庫。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 代碼穩健度：縮放邏輯是否處理了極小分母的異常值，且運算流程具備優良的即時性？

任務 3：轉向敏感度調教 (Dynamic Sensitivity)

角色扮演：賽道調校教練 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 權重係數引入：在混控公式中加入 Sensitivity (K) 參數：R = v + (K * ω)。
  2. 環境適配性測試：在不同摩擦係數的地板上測試不同 K 值對彎道響應的影響。
  3. 實驗日誌彙整：尋找並記錄能最穩、最快通過 S 彎賽道的黃金 K 值。
  4. 正式提交成果：將包含調教參數設定與實驗數據心得的代碼 Push 至 GitHub PR。
