簡介

傳統控制只看「車頭正下方」，一旦速度提升，系統來不及反應就衝出彎道。 本模組將帶你理解預覽控制的核心理念：用空間換取時間， 學會透過掃描影像遠處的目標點來預判彎道走向，實現高速下依然平滑流暢的自動導航。

情境
你的循跡車在低速時表現完美，但一加速就開始劇烈蛇行，甚至在急彎處直接衝出賽道。 原因在於：系統只根據「此刻的偏差」做修正，等到發現彎道時已經來不及了。 如果能像賽車手一樣「看向遠方的彎心」提前轉向，震盪問題就能迎刃而解—— 這就是預覽控制的威力。

為什麼要「往遠處看」？

在前面的單元中，我們學會了在影像底部附近計算質心並產生轉向指令。這套「應對式控制（Reactive Control）」在低速行駛時表現良好，但當車速提升後，一個嚴重的問題會浮現：系統延遲（System Latency）讓車輛來不及反應。

系統延遲的來源

從攝影機擷取畫面到舵機完成轉向，中間經歷了多個延遲環節：

延遲環節典型耗時說明
影像擷取~33 ms（30 fps）攝影機曝光與資料傳輸
影像處理~10-20 ms灰階轉換、二值化、質心計算
控制計算~5 ms誤差計算與角度映射
舵機響應~50-100 msPWM 訊號到機械轉動的物理延遲
總計~100-160 ms約 0.1 ~ 0.16 秒

以時速 1 m/s 計算，0.15 秒的延遲意味著車輛在「看到彎道」到「完成轉向」之間已經前進了 15 公分。在窄賽道上，這個距離足以讓車輛衝出跑道。

注意 即使你的程式處理速度極快，舵機的機械響應延遲是物理限制，無法透過軟體優化來消除。預覽控制的核心理念就是：既然無法消除延遲，那就提前預判來補償它。

預覽控制的核心理念

預覽控制的靈感來自人類駕駛行為。經驗豐富的賽車手不會盯著引擎蓋前方看——他們的目光永遠鎖定在彎道的出口。透過觀察遠方的道路走向，大腦有充裕的時間提前計算轉向角度，產生平滑而精準的入彎動作。

在影像處理中，「往遠處看」意味著：在影像的上方區域（較小的 Y 值）而非底部區域計算質心。影像透視投影的特性決定了——畫面上方對應的是現實世界中較遠的位置。

提示 你可以做一個簡單的實驗來感受預覽控制的效果：先用 $Y=220$（車頭正下方）計算質心並驅動舵機，再切換到 $Y=120$（畫面中段）。你會發現後者的轉向明顯更早、更柔和，蛇行震盪也大幅減少。
警告 預覽太遠也有風險——畫面頂部可能包含其他賽道、觀眾席或雜訊，質心會被這些無關目標拉偏。預覽距離的選擇需要在「提前量」與「準確度」之間取得平衡，這正是下一節的主題。

預覽點 (Look-ahead Point) 的數學選取

預覽控制的關鍵參數是預覽點的 Y 軸高度（$Y_{target}$）。在攝影機的透視投影中，Y 值越小代表現實中越遠的位置。選擇適當的 $Y_{target}$，就是在決定「車輛要看多遠的未來」。

多層次掃描線架構

最佳實務不是只看單一高度的質心，而是同時在影像的近、中、遠三個高度各設一條掃描線，分別計算質心。三條線的質心差異能揭示賽道的彎曲方向與程度：

掃描層建議 Y 值對應距離用途
近層（Near）$Y = 200$車頭前方 ~5 cm精確定位當前位置
中層（Mid）$Y = 120$前方 ~15 cm主要導航參考（推薦預設）
遠層（Far）$Y = 60$前方 ~30 cm彎道預判與提前修正
// 多層次質心提取
const SCAN_LINES = [200, 120, 60]; // 近、中、遠
const centroids = SCAN_LINES.map(y => {
  let sumX = 0, count = 0;
  for (let x = 0; x < width; x++)
    if (isBlack(x, y)) { sumX += x; count++; }
  return count > 0 ? sumX / count : null;
});

自適應預覽距離

固定的預覽高度無法同時滿足低速與高速的需求。更進階的做法是讓預覽高度隨車速動態調整：

$$Y_{target} = \text{map}(speed, 0, V_{max}, Y_{near}, Y_{far})$$

速度越快，$Y_{target}$ 越小（看越遠），給系統更多的反應時間。速度越慢，$Y_{target}$ 越大（看越近），提供更精確的即時定位。

// 自適應預覽高度：速度越快，看越遠
function updateLookahead(currentSpeed) {
  const Y_NEAR = 200; // 最近
  const Y_FAR = 50; // 最遠
  const MAX_SPEED = 100;
  let targetY = Y_NEAR - (currentSpeed / MAX_SPEED) * (Y_NEAR - Y_FAR);
  targetY = Math.max(Y_FAR, Math.min(Y_NEAR, targetY));
  return Math.round(targetY);
}
注意 自適應預覽高度必須設有上下限防護。$Y_{target}$ 不應小於 40（過遠的區域容易看到賽道以外的雜訊），也不應大於 220（過近的位置已失去預覽效果）。
提示 在 Canvas 上用三種顏色（紅、綠、藍）分別繪製近、中、遠三層的質心位置，可以直觀觀察入彎時「遠處質心先偏移」的現象。這是驗證預覽控制是否正確運作的最佳視覺化手段。

純追蹤 (Pure Pursuit) 概念與切彎問題

純追蹤演算法（Pure Pursuit Algorithm）是自動駕駛領域中最經典的路徑追蹤方法之一。它的核心思想極其直覺：在車輛前方選定一個目標點（Look-ahead Point），然後計算出一段能讓車輛精確通過該點的圓弧路徑，再將圓弧的曲率轉換為舵機轉向角。

純追蹤的轉向公式

假設車輛的軸距為 $L$，目標點相對於車頭的距離為 $l_d$（預覽距離），目標點偏離車頭方向的角度為 $\alpha$，則所需的轉向角為：

$$\delta = \arctan\left(\frac{2L \sin \alpha}{l_d}\right)$$

在簡化的循跡場景中，$\alpha$ 可以用質心誤差近似計算：alpha = atan2(error, lookaheadDistance)。這個公式的美妙之處在於——它會根據預覽距離自動調節轉向的「積極程度」。

截彎取直（Cutting Corners）問題

預覽控制的一大陷阱是：當預覽距離設定過長時，車輛在接近彎道時就「看到」了彎道出口，於是直接朝出口方向前進，走了一條切過彎心內側的捷徑。雖然這在某些情境下反而更快，但在需要嚴格沿線行駛的場景中，這會導致車輛偏離賽道。

預覽距離 $l_d$轉向行為問題
過短（如 $Y=220$）急轉急停，頻繁修正蛇行震盪、超調（Overshoot）
適中（如 $Y=120$）平滑入彎，貼合賽道（理想狀態）
過長（如 $Y=40$）提前轉向，走捷徑截彎取直、偏離賽道線

$K_p$ 與預覽高度的連動調校

解決截彎取直的方法不是一味縮短預覽距離，而是同時調整比例增益 $K_p$。當預覽距離增加時，應降低 $K_p$ 以抑制轉向幅度；反之，近距離預覽時可以使用較高的 $K_p$ 提升響應速度。

// 預覽控制與 P-Control 整合
const lookaheadY = updateLookahead(currentSpeed);
const centroidX = getCentroidAtY(binaryImage, lookaheadY);

if (centroidX !== null) {
  const error = centroidX - 160;
  // Kp 隨預覽距離反向調整
  const Kp = map(lookaheadY, 200, 50, 0.25, 0.10);
  const steer = 90 + error * Kp;
  servo.write(Math.max(60, Math.min(120, steer)));
}
警告 當預覽目標點突然消失（例如直道進入彎道的瞬間），質心可能產生大幅跳變。此時若直接將跳變值傳給舵機，會導致機械暴力抽動。建議加入變化率限制（Rate Limiting）：每幀的角度變化不超過 5°，確保轉向始終平滑。
提示 在 Canvas 上繪製一條從車頭中心（畫面底部中央）到預覽質心的連接線，可以直觀觀察車輛的「視線方向」。入彎時這條線會提前偏轉，讓你確認預覽控制是否正確地在「看向彎心」。

Tutor Guide｜adv-06-unit-look-ahead（預覽控制觀念）

針對 adv-06-unit-look-ahead (預覽控制觀念) 單元，這是一個將「應對式控制」提升到「預測性控制 (Predictive Control)」的高度進階課。它解決了高速巡航下的震盪問題，是自動駕駛導航理論的核心。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template Repo) 配置建議

預覽控制高度依賴「空間換取時間」的邏輯與「視覺化除錯」，建議範本包含：

  • 📂 js/multi_roi_tracker.js：多層次追蹤引擎，提供解析影像緩衝區的基底，留下定義不同 $Y$ 軸掃描高度（如 Y=200, 120, 40）與計算質心的 TODO。
  • 📂 js/adaptive_navigation.js：自適應導航模組。預置速度連動函式骨架，讓學員實作 $Y_{target} = f(Speed)$ 的映射邏輯。
  • 📂 ui/navigation_overlay.js：負責在 Canvas 上繪製不同高度的預覽點、偵測線與預期路徑軌跡（Trajectory）。
  • 📂 tests/predictive_lead.test.js：自動化評分脚本。模擬一個 45 度彎道，檢核學員實作的「遠處預覽點」是否正確位於「近處點」的偏向方位。

2. 作業任務部署細節

任務 1：實作「多層次 ROI」檢測器 (Multi-level ROI Detector)

  • 目標：透過多點取樣，在空間維度上建立「時間序列」的預判能力。
  • Classroom 部署建議：
    • 視覺化檢核：學員需在 Canvas 上用紅、綠、藍三種顏色標註近、中、遠三層的質心。這能讓工程師一目瞭然：入彎時是否是「遠處先偏」。
    • Autograding：餵入一張彎道影像，檢核程式是否回傳了三個不同的 $X$ 座標值。如果三者相同，代表學員未正確切分掃描區間。

任務 2：自適應預覽距離 (Adaptive Look-ahead Setup)

  • 目標：解決不同車速下控制動態特性的不匹配問題。
  • Classroom 部署建議：
    • 動態演算法檢核： 
      // 學生需實作：動態調整視線高度
function updateLookahead(currentSpeed) {
    // 速度愈快，Y 座標愈小（看愈遠）
    targetY = map(currentSpeed, 0, 100, 200, 50);
}
    • 實驗紀錄：學員需在 BENCHMARK.md 中記錄：在高速直線行駛時，提高預覽高度是否能顯著減少馬達的「蛇行頻率」。
    • PR 深度核核：導師點評其對「預覽高度上限」的防禦性處理（防止看太遠看到賽道外的干擾）。

任務 3：預覽整合與「截彎取直」除錯 (Look-ahead Integration)

  • 目標：調校一套具備「專業賽車手」質感的入彎路徑。
  • Classroom 部署建議：
    • 整合邏輯檢核：學員需將「遠方預覽點」的 Error 作為 P 控制器的主要輸入，並解決「太早切彎導致撞內牆」的問題（即：平衡預覽高度與 $K_p$ 係數）。
    • 驗證方式：提交一段車輛過 90 度彎道的連續測試影片。
    • Tutor View (PR 審核)：觀察學員是否加入了「平滑過渡」邏輯。當視線點突然發生大位移時，輸出指令是否具備足夠的阻尼（Damping）防止馬達暴力抽動？

3. 導航工程導師點評標準 (Predictive Navigation Benchmarks)

此單元的價值在於 「對系統延遲的預補償思維」：

  • [ ] 視覺化除錯完備度：Canvas 上是否有清晰的輔助線與點位標註？（這是複雜控制系統除錯的生命線）。
  • [ ] 自適應邏輯合理性：設定的視線距離是否能隨速度平滑變化，有無「突變斷點」？
  • [ ] 路徑優雅度 (Driving Quality)：車輛入彎是否柔和？是否能精確貼合彎心的理想軌跡（Racing Line）？
