🔭 1. 為什麼要「往遠處看」？ (20 min)

感測器擷取、AI 計算到馬達轉向都有延遲。如果只看車頭正下方，高速時會衝出彎道。預覽遠方可以讓轉向動作更提早、更柔和。

Control Benefits

  • 震盪抑制：解決蛇行震盪問題。
  • 預判過彎：在入彎前就開始切換轉向舵角。

🔄 3. 純追蹤 (Pure Pursuit) 概念與切彎問題

預覽過遠會導致「截彎取直 (Cutting Corners)」現象。學習如何控制預覽長度，在「過早轉彎」與「來不及轉彎」之間取得平衡。

Path Geometry Control

  • 切西瓜問題：視線過遠，車輛會提早切入彎道內側。
  • 追蹤延遲：視線過近，高速下會產生嚴重的超調 (Overshoot)。
  • 幾何補償：透過 $K_p$ 係數與 Y 軸高度的連動優化。

Steering Correction

$\delta = \arctan(\frac{2L \sin \alpha}{l_d})$

adv-06｜預覽控制觀念 — 看得更遠，走得更穩

這個程式庫是你的練習場地。你將學會讓車輛「看遠方來控制當下」，解決高速循跡時的蛇行問題。

🎯 本單元目標

傳統的應對式控制（Reactive Control）只看「當下在哪裡」。本單元要帶你進階到 預測性控制（Predictive Control）：

「在彎道出現前，就已經開始轉向。」

你將實作一套多層次預覽系統，讓機器人在高速移動時也能保持穩定、流暢的路徑追蹤。

✅ 作業任務

任務 1：實作多層次 ROI 質心提取

檔案：js/multi_layer_tracker.js

目標：在影像的三個不同 Y 軸高度（近、中、遠）各取一條掃描線，分別計算質心 X 座標。

驗收標準：

  • [ ] 三個掃描高度的質心 X 座標彼此不同（若完全相同，代表切分邏輯有誤）
  • [ ] Canvas 上以 🔴紅（近）、🟢綠（中）、🔵藍（遠）三色標記各層質心
  • [ ] 通過 tests/predictive_lead.test.js 中的自動評分測試

提示：看看 this.scanLinesY 陣列，從這三個 Y 值出發，針對每條掃描線提取亮度加權的質心。

任務 2：實作自適應預覽距離

檔案：js/adaptive_engine.js

目標：讓預覽高度隨著車速動態調整。速度越快，預覽點看得越遠（Y 座標越小）。

核心邏輯（需你完成）：

function updateLookahead(currentSpeed) {
    // 速度愈快，Y 座標愈小（看愈遠）
    // 請將 currentSpeed 映射到合理的 Y 範圍

}

驗收標準：

  • [ ] targetY 隨速度連續變化，無突變斷點
  • [ ] 設有「預覽高度上限」防禦邏輯（避免看太遠偵測到賽道外的雜訊）

Tutor Guide｜adv-06-unit-look-ahead（預覽控制觀念）

針對 adv-06-unit-look-ahead (預覽控制觀念) 單元，這是一個將「應對式控制」提升到「預測性控制 (Predictive Control)」的高度進階課。它解決了高速巡航下的震盪問題，是自動駕駛導航理論的核心。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template Repo) 配置建議

預覽控制高度依賴「空間換取時間」的邏輯與「視覺化除錯」，建議範本包含：

  • 📂 js/multi_roi_tracker.js：多層次追蹤引擎，提供解析影像緩衝區的基底，留下定義不同 $Y$ 軸掃描高度（如 Y=200, 120, 40）與計算質心的 TODO。
  • 📂 js/adaptive_navigation.js：自適應導航模組。預置速度連動函式骨架，讓學員實作 $Y_{target} = f(Speed)$ 的映射邏輯。
  • 📂 ui/navigation_overlay.js：負責在 Canvas 上繪製不同高度的預覽點、偵測線與預期路徑軌跡（Trajectory）。
  • 📂 tests/predictive_lead.test.js：自動化評分脚本。模擬一個 45 度彎道，檢核學員實作的「遠處預覽點」是否正確位於「近處點」的偏向方位。

2. 作業任務部署細節

任務 1：實作「多層次 ROI」檢測器 (Multi-level ROI Detector)

  • 目標：透過多點取樣，在空間維度上建立「時間序列」的預判能力。
  • Classroom 部署建議：
    • 視覺化檢核：學員需在 Canvas 上用紅、綠、藍三種顏色標註近、中、遠三層的質心。這能讓工程師一目瞭然：入彎時是否是「遠處先偏」。
    • Autograding：餵入一張彎道影像，檢核程式是否回傳了三個不同的 $X$ 座標值。如果三者相同，代表學員未正確切分掃描區間。

任務 2：自適應預覽距離 (Adaptive Look-ahead Setup)

  • 目標：解決不同車速下控制動態特性的不匹配問題。
  • Classroom 部署建議：
    • 動態演算法檢核： 
      // 學生需實作：動態調整視線高度
function updateLookahead(currentSpeed) {
    // 速度愈快，Y 座標愈小（看愈遠）
    targetY = map(currentSpeed, 0, 100, 200, 50);
}
    • 實驗紀錄：學員需在 BENCHMARK.md 中記錄：在高速直線行駛時，提高預覽高度是否能顯著減少馬達的「蛇行頻率」。
    • PR 深度核核：導師點評其對「預覽高度上限」的防禦性處理（防止看太遠看到賽道外的干擾）。

任務 3：預覽整合與「截彎取直」除錯 (Look-ahead Integration)

  • 目標：調校一套具備「專業賽車手」質感的入彎路徑。
  • Classroom 部署建議：
    • 整合邏輯檢核：學員需將「遠方預覽點」的 Error 作為 P 控制器的主要輸入，並解決「太早切彎導致撞內牆」的問題（即：平衡預覽高度與 $K_p$ 係數）。
    • 驗證方式：提交一段車輛過 90 度彎道的連續測試影片。
    • Tutor View (PR 審核)：觀察學員是否加入了「平滑過渡」邏輯。當視線點突然發生大位移時，輸出指令是否具備足夠的阻尼（Damping）防止馬達暴力抽動？

3. 導航工程導師點評標準 (Predictive Navigation Benchmarks)

此單元的價值在於 「對系統延遲的預補償思維」：

  • [ ] 視覺化除錯完備度：Canvas 上是否有清晰的輔助線與點位標註？（這是複雜控制系統除錯的生命線）。
  • [ ] 自適應邏輯合理性：設定的視線距離是否能隨速度平滑變化，有無「突變斷點」？
  • [ ] 路徑優雅度 (Driving Quality)：車輛入彎是否柔和？是否能精確貼合彎心的理想軌跡（Racing Line）？
