BLE 與 Web Bluetooth API

什麼是 BLE（Bluetooth Low Energy）？

BLE 是藍牙 4.0 以後引入的低功耗通訊協定。與傳統藍牙（Classic Bluetooth）相比，BLE 專為短距離、小資料量、低功耗的物聯網場景設計。它不需要「配對」程序即可快速建立連線，非常適合感測器資料傳輸或裝置設定等場景。

BLE vs 傳統藍牙 vs WiFi 比較

特性 BLE 傳統藍牙 WiFi
功耗 極低（mA 等級） 中等 高
傳輸速率 ~1 Mbps ~3 Mbps ~100+ Mbps
連線建立時間 數毫秒 數秒 數秒
典型用途 感測器、設定配網 音訊串流、檔案傳輸 網路連線、大量資料
是否需配對 不需要 需要 需要密碼

GATT 架構（Generic Attribute Profile）

BLE 裝置透過 GATT 架構組織資料。你可以把它想像成一個「資料夾結構」：

1

Profile（設定檔）

整個裝置的最上層定義，描述裝置的用途。

2

Service（服務）

一組相關功能的集合，每個 Service 有一個唯一的 UUID。例如：WiFi 配網服務。

3

Characteristic（特徵值）

Service 下的實際資料欄位。每個 Characteristic 也有自己的 UUID，可設為可讀 / 可寫 / 可通知。

TIP 類比思考：Service 就像一個「資料夾」，Characteristic 就像資料夾裡的「檔案」。你可以往 Characteristic 裡寫入資料（Write）、讀取資料（Read）或訂閱變更通知（Notify）。

Web Bluetooth API 連線流程

Web Bluetooth 讓瀏覽器能直接與 BLE 裝置通訊。整個連線流程是一連串非同步操作：

1

requestDevice — 掃描並選擇裝置

navigator.bluetooth.requestDevice({ filters: [...] }) — 彈出裝置選擇器，使用者點選後回傳 BluetoothDevice 物件。

2

connect — 連線 GATT Server

device.gatt.connect() — 與裝置建立 GATT 連線，回傳 BluetoothRemoteGATTServer。

3

getPrimaryService — 取得服務

server.getPrimaryService(SERVICE_UUID) — 透過 UUID 找到目標 Service。

4

getCharacteristic — 取得特徵值

service.getCharacteristic(CHAR_UUID) — 在 Service 下找到目標 Characteristic。

5

writeValue — 寫入資料

characteristic.writeValue(encodedData) — 將位元組陣列寫入 Characteristic。

WARNING Web Bluetooth 需要安全環境（HTTPS 或 localhost），且 requestDevice 必須由使用者操作（如點擊按鈕）觸發。一般的 Line 或 Facebook 內建瀏覽器不支援 Web Bluetooth，iOS 用戶請使用 Bluefy 瀏覽器。

NVS 與資料流

什麼是 NVS（Non-Volatile Storage）？

NVS 是 ESP32 上的一塊持久化儲存區域，存放於 SPI Flash 的特定分區中。「Non-Volatile（非揮發性）」意味著即使裝置斷電，資料也不會消失。這正是 WiFi 憑證能夠「記住」的原因。

TIP 你可以把 NVS 想像成嵌入式系統的「localStorage」——鍵值對（Key-Value）結構，斷電不遺失。ESP-IDF 提供 nvs_set_str() 和 nvs_get_str() 等 API 來操作它。

NVS 的關鍵特性

特性 說明
儲存格式 Key-Value 鍵值對，支援字串、整數、Binary Blob
容量 預設約 16~24 KB（可在 partition table 中調整）
磨損均衡 內建 Wear Leveling，延長 Flash 壽命
Namespace 可用命名空間隔離不同模組的資料
斷電安全 寫入後立即持久化，不受意外斷電影響

WiFi 憑證如何跨電源週期存活？

當 ESP32 收到 WiFi SSID 和密碼後，韌體會將這些資料寫入 NVS。下次裝置重新上電時，韌體會先從 NVS 讀取已儲存的 WiFi 資訊，如果存在就自動嘗試連線——完全不需要重新透過 BLE 配網。

完整資料流：從瀏覽器到 WiFi 連線

1

瀏覽器：使用者輸入 SSID / 密碼

前端 JavaScript 取得表單值，使用 TextEncoder 編碼為 Uint8Array。

2

BLE 傳輸：寫入 Characteristic

呼叫 characteristic.writeValue(encoded)，位元組經由 BLE 空中介面傳送至 ESP32。

3

ESP32 韌體：接收並解碼

韌體的 Write Callback 被觸發，從 BLE Buffer 讀出位元組並還原為字串。

4

NVS：持久化寫入

韌體呼叫 nvs_set_str() 將 SSID 與密碼寫入 Flash，再 nvs_commit() 確認。

5

WiFi 連線：自動啟動

裝置重啟後，從 NVS 讀取憑證，呼叫 esp_wifi_connect() 自動連上路由器。

WARNING NVS 中的 WiFi 密碼以明文儲存。在量產產品中應使用 ESP-IDF 的 NVS Encryption 功能加密整個 NVS 分區，或在應用層自行加密敏感資料後再存入。

TextEncoder 與 UUID

為什麼需要 TextEncoder？

BLE Characteristic 只接受位元組陣列（ArrayBuffer / DataView / TypedArray），而 JavaScript 的字串是 UTF-16 編碼的高階物件。TextEncoder 負責將字串轉換為 UTF-8 位元組序列，產生一個 Uint8Array 供 BLE 寫入使用。

1

建立 Encoder

const encoder = new TextEncoder(); — 預設產生 UTF-8 編碼器。

2

編碼字串

const bytes = encoder.encode("MyWiFi"); — 回傳 Uint8Array [77, 121, 87, 105, 70, 105]。

3

寫入 BLE

characteristic.writeValue(bytes); — 位元組經由空中介面傳送。

NOTE UTF-8 對 ASCII 字元（英文、數字）每字元只佔 1 byte，但中文字佔 3 bytes。如果 SSID 包含中文（如「我的網路」），編碼後長度會超出預期。BLE 單次寫入上限通常為 MTU - 3 bytes（預設 MTU = 23，可用 20 bytes）。

BLE UUID 結構

每個 BLE Service 和 Characteristic 都透過 UUID（Universally Unique Identifier）來識別。BLE 定義兩種格式：

類型 格式 範例
16-bit UUID SIG 標準服務 0x180A（Device Info）
128-bit UUID 自定義服務 6e400001-b5a3-f393-e0a9-e50e24dcca9e

Nordic UART Service（NUS）UUID

本專案使用的是 Nordic Semiconductor 定義的 UART Service（NUS），這是 BLE 串口模擬的事實標準：

角色 UUID 屬性
Service 6E400001-B5A3-F393-E0A9-E50E24DCCA9E —
RX Characteristic（寫 SSID） 6E400002-B5A3-F393-E0A9-E50E24DCCA9E Write
TX Characteristic（寫密碼） 6E400003-B5A3-F393-E0A9-E50E24DCCA9E Write
TIP UUID 中只有最後四碼不同（0001、0002、0003），前綴一致。這是 NUS 的命名慣例——只改變 Service/Characteristic 編號。在 Web Bluetooth API 中，UUID 必須全部使用小寫。

非同步錯誤處理

BLE 操作全部是非同步的（回傳 Promise），任何一步都可能失敗。良好的錯誤處理是穩定配網工具的關鍵：

1

使用者取消選擇

NotFoundError — 使用者按了「取消」或裝置不在範圍內。應提示重試。

2

安全環境限制

SecurityError — 頁面不在 HTTPS 或 localhost。應顯示環境需求提示。

3

連線中斷

NetworkError — 裝置超出範圍或關機。監聽 gattserverdisconnected 事件來重置 UI。

4

寫入失敗

InvalidStateError — Characteristic 不可寫或連線已失效。應在 try...catch 中捕捉並回報。

WARNING 永遠在 async 函數中使用 try...catch 封裝 BLE 操作。未捕捉的 Promise rejection 會導致無聲失敗——使用者看不到任何反饋，卻不知道配網未成功。

  • 目標：確保資料能正確轉換為 Byte Stream 並處理傳輸錯誤。
  • Classroom 部署建議：
    • 驗證方式：學生需在 console.log 記錄 TextEncoder().encode(ssid) 的輸出。
    • Autograding：撰寫一個測試腳本，模擬呼叫學生的傳送函數，並檢查是否由 try...catch 捕捉到無效的藍牙特徵值寫入錯誤（例如模擬 Characteristic not found）。

任務 3：永恆連結 - 硬體連線與 NVS 驗證 (Hardware Integration)

  • 目標：確保持久化儲存 (Non-volatile storage) 運作正常。
  • Classroom 部署建議：
    • 證據提交：學生需將 ESP32 成功連線後的「序列埠日誌 (Serial Monitor Log)」內容複製到 serial-output.log。
    • 核核重點：日誌中必須出現 WiFi Connected! IP: 192.168.x.x 以及 NVS Commit Success 等關鍵字，證明資訊已正確存入硬體快閃記憶體。

3. 給導師的翻轉教室實施建議

  • 技術挑戰提醒：
    • 部分 iOS 使用者可能找不到設備，需提醒下載 Bluefy 瀏覽器。
    • 部分 Android 裝置有 MTU (Maximum Transmission Unit) 限制，若學生輸入超長 SSID 導致截斷，這是絕佳的通訊協議教學機會。
  • 評分指標 (Rubric)：
    • [ ] 非同步掌握度：是否正確使用 await 處理藍牙寫入？
    • [ ] 防呆機制：空值或超長字元是否在前端就被攔截？
    • [ ] 狀態機邏輯：UI 是否能根據「未連線、連線中、已連線、傳送中、成功」等狀態精確切換？

這種部署方式能讓學生理解，前端開發不僅是頁面排版，更是控制設備的核心介面，從而建立深厚的跨領域技術素養。
