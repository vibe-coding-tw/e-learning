簡介

按鈕看似簡單——按下去就是 LOW，放開就是 HIGH——但現實遠比這複雜。 沒有上拉電阻，GPIO 腳位在按鈕未按時會處於「浮動」狀態，讀到的值完全隨機；而機械按鈕在按壓瞬間會產生數十次的金屬彈跳，讓微控制器誤以為你連按了好幾下。 本單元教你用硬體（電阻）與軟體（時間過濾）兩道防線，讓每一次按壓都被正確且唯一地識別。

情境
你做了一個計數器：每按一次按鈕，LCD 上的數字加 1。但實際測試時，輕輕按一下數字卻跳了 3～5 次。 你確認程式邏輯沒問題——問題出在硬體層：按鈕的金屬觸片在接觸瞬間會快速彈跳數十次，每次彈跳都被 GPIO 讀為一次狀態變化。 這就是為什麼你需要「防彈跳」技術，無論是外接電容還是軟體延遲濾波，都能讓一次物理按壓只產生一次有效事件。

消失的邏輯：為什麼需要上拉電阻？

在電子世界中，「沒有接線」不代表讀值是「零」。當你把一個 GPIO 引腳設為輸入模式，卻不連接任何電路時，這支引腳就處於浮空狀態（Floating State）。浮空的引腳就像一根暴露在空氣中的天線，會不斷拾取環境中的電磁雜訊——你的手靠近它、附近的馬達旋轉、甚至日光燈的電源頻率，都會讓它的讀值在 HIGH 和 LOW 之間瘋狂跳動。

這不是程式的 Bug，而是物理法則。在數位電路中，每一個輸入引腳都必須被明確地連接到一個確定的電位——要嘛是 HIGH（3.3V），要嘛是 LOW（GND）。浮空是工程師最不想看到的狀態，因為它代表「不確定性」，而不確定性在控制系統中是致命的。

什麼是上拉電阻？

上拉電阻（Pull-up Resistor）是一顆連接在引腳與電源（VCC / 3.3V）之間的電阻。它的作用是在按鈕未被按下時，將引腳「拉」到穩定的 HIGH 狀態。當按鈕被按下，引腳透過按鈕直接接地（GND），因為接地路徑的電阻遠低於上拉電阻，所以引腳被拉到 LOW。這就形成了一個明確的邏輯：未按 = HIGH，按下 = LOW。

相反地，下拉電阻（Pull-down Resistor）是連接在引腳與 GND 之間。未按時引腳為 LOW，按下後透過電源路徑讀為 HIGH。兩種方式都能消除浮空問題。

備註 使用上拉電阻時，按鈕的邏輯是反向的（Active Low / 負邏輯）：按下按鈕讀到 LOW，放開時讀到 HIGH。初學者常因此困惑，但這是業界最常見的按鈕接法，因為它最穩定且最省線路。

ESP32 的內部上拉電阻

好消息是：你不一定需要外接一顆實體電阻。ESP32（以及大多數現代微控制器）在晶片內部已經整合了上拉電阻（約 45kΩ）。你只需在程式中將引腳設為 INPUT_PULLUP 模式，內部上拉電阻就會自動啟用：

const int BUTTON_PIN = 14;

void setup() {
  pinMode(BUTTON_PIN, INPUT_PULLUP);
  Serial.begin(115200);
}

void loop() {
  int state = digitalRead(BUTTON_PIN);
  Serial.println(state); // 未按: 1, 按下: 0
  delay(100);
}

INPUT vs INPUT_PULLUP 比較

模式 引腳未連接時 按鈕未按時 按鈕按下時 適用情境
INPUT 浮空（不確定） 需外接電阻 取決於電路 有外部上/下拉電阻時
INPUT_PULLUP 穩定 HIGH HIGH (3.3V) LOW (GND) 按鈕、開關（推薦）
提示 ESP32 的 GPIO 34–39 是僅限輸入引腳，它們沒有內部上拉電阻。若你將按鈕接在這些引腳上，必須外接一顆 10kΩ 的上拉（或下拉）電阻，否則仍會出現浮空問題。

物理現實：機械彈跳 (Debounce)

解決了浮空問題之後，你滿懷信心地寫了一個按鈕計數器：每按一下，數字加 1。程式邏輯完美——但實測時，輕輕按一下數字卻跳了 3 到 8 次。這不是程式的錯，而是機械彈跳（Mechanical Bouncing）的物理現象。

按鈕的金屬觸片並不是瞬間從「斷開」變成「接通」。在按壓的那一瞬間（通常持續 1～50 毫秒），觸片會像彈簧一樣快速地彈跳數次，產生一連串的開合訊號。用示波器觀察，波形看起來就像這樣：

// 示波器觀察到的訊號時序（理想 vs 真實）

理想: 1111111110000000000
真實: 111111010110010000000000
              ^^^^^^^^
              彈跳區間（約 5~50ms）

在那段「彈跳區間」裡，微控制器以微秒級的速度不斷讀取引腳狀態，每次狀態從 HIGH 跳到 LOW 或從 LOW 跳到 HIGH，都會被程式判定為一次「事件」。這就是為什麼一次物理按壓會產生多次計數的原因。

誰也會抖動？

機械彈跳不是按鈕的專利。所有具有機械接觸點的元件都會產生彈跳，只是程度與持續時間不同：

元件類型 典型彈跳時間 常見應用
觸控按鈕 (Tact Switch) 5 ~ 20 ms 面板操作、模式切換
極限開關 (Limit Switch) 1 ~ 10 ms 3D 列印機歸零、機械臂到位偵測
磁簧開關 (Reed Switch) 0.5 ~ 5 ms 門窗感測器、轉速計
繼電器 (Relay) 1 ~ 5 ms 電源控制、工業自動化

硬體去彈跳 vs 軟體去彈跳

去彈跳有兩種主要途徑：硬體方式（在按鈕兩端並聯一顆 0.1μF 的電容，利用 RC 充放電延遲來「吸收」彈跳）和軟體方式（在程式中用時間過濾邏輯忽略彈跳期間的訊號變化）。

備註 硬體去彈跳（電容濾波）的優點是完全不佔用 CPU 時間，缺點是需要額外的元件與電路空間。軟體去彈跳不需任何額外硬體，但需要良好的程式設計。在學習階段，我們以軟體方式為主——它更靈活，也能讓你深刻理解「時間過濾」的演算法思維。
注意 不要忽視機械彈跳！在工業場景中，一次按壓被判讀為多次觸發，可能導致繼電器誤切、計數器失準、甚至機器人做出危險動作。去彈跳不是可選的優化，而是必要的安全措施。

軟體去抖：時間過濾術

軟體去彈跳的核心思想非常直覺：當偵測到按鈕狀態改變時，不要立即信任它，而是等待一段時間（通常 20~50 毫秒），確認狀態穩定不再跳動後，才判定為一次有效的按壓事件。

這個演算法可以拆解為三個步驟：

1

邊緣偵測（Edge Detection）

在每次 loop() 迴圈中讀取按鈕當前狀態，並與「上一次確認的穩定狀態」比較。如果不同，代表可能發生了按壓或釋放事件——但也可能只是彈跳雜訊。

2

等待穩定（Debounce Delay）

記錄狀態首次改變的時間戳。在接下來的 20~50 毫秒內，持續讀取引腳狀態但不做任何動作。這段等待時間會讓所有的機械彈跳自然消退。

3

確認有效（Double Confirm）

等待時間結束後，再次讀取引腳狀態。若狀態確實與之前不同，才正式更新為新的穩定狀態，並觸發對應的事件處理。

為什麼不用 delay()？

很多初學者的第一反應是：「偵測到按壓後，直接 delay(50) 等一下不就好了？」這樣做確實能解決彈跳問題，但會帶來一個嚴重的副作用——程式在 delay() 期間什麼都不能做。如果你的系統同時需要控制馬達、讀取感測器、或更新顯示器，delay() 會讓這些背景任務全部停擺。

專業的做法是使用 millis() 函式進行非阻塞式計時（Non-blocking Timing）。millis() 會回傳自程式啟動以來經過的毫秒數，你只要記錄「上次狀態改變的時間」，然後在每次 loop() 中檢查是否已經過了足夠的等待時間即可，完全不會阻塞其他程式邏輯。

完整的非阻塞去彈跳程式碼

const int BTN = 14;
const int LED = 2;
const unsigned long DEBOUNCE_MS = 50;

int lastStable = HIGH;
int lastReading = HIGH;
unsigned long lastChangeTime = 0;
bool ledOn = false;

void loop() {
  int reading = digitalRead(BTN);
  if (reading != lastReading) {
    lastChangeTime = millis();
  }
  if (millis() - lastChangeTime > DEBOUNCE_MS) {
    if (reading != lastStable) {
      lastStable = reading;
      if (lastStable == LOW) {
        ledOn = !ledOn;
        digitalWrite(LED, ledOn);
      }
    }
  }
  lastReading = reading;
}
提示 DEBOUNCE_MS 的最佳值取決於你使用的按鈕品質。廉價的觸控按鈕可能需要 50ms，高品質的工業開關可能只需 10ms。你可以透過實驗（記錄一次按壓產生的彈跳次數與持續時間）來找到最佳值。

delay() vs millis() 比較

特性 delay() 阻塞式 millis() 非阻塞式
等待期間 CPU 狀態 完全停止，不執行任何程式 持續運行 loop()，可處理其他任務
適合多工系統 不適合 完美支援
程式碼複雜度 簡單（一行即可） 需要額外的狀態變數
專業等級 教學示範用 工業級生產品質
重要 在專業韌體開發中，全程禁止在 loop() 中使用 delay() 是一條鐵律。當你的系統需要同時處理按鈕偵測、馬達控制、感測器讀取與無線通訊時，任何一個 delay() 都會讓其他任務的反應時間劣化。從現在開始，養成使用 millis() 的習慣。

數位控制：上拉電阻與按鈕去彈跳 (Digital Control: Pull-up Resistors & Button Debouncing)

課程目標

在電子電路中，機械開關（如按鈕）並非完美的。當你按下或放開按鈕時，接觸點會在毫秒等級的時間內發生多次微小的彈跳（Bouncing），這會導致微控制器誤以為你快速按下了多次按鈕。本單元將帶領你透過軟體演算法「馴服」這些彈跳，並學習專業韌體開發必備的 非阻塞（Non-blocking） 程式設計技巧。

核心技能

  • 理解浮空狀態 (Floating State)：學習為何數位輸入需要上拉或下拉電阻來確保邏輯確定性。
  • 物理特性量化：觀察並記錄按鈕按下時產生的機械彈跳現象。
  • 非阻塞計時 (Non-blocking Timing)：掌握 millis() 取代 delay()，讓程式在等待的同時仍能執行其他任務。

任務說明

任務 1：天線實驗 - 體會浮空狀態 (Floating State Lab)

  1. 將 Arduino 的一個數位引腳（預設為 BUTTON_PIN）設定為 INPUT 模式，且不連接任何電子零件（懸空）。
  2. 開啟 Serial Plotter，觀察當你的手靠近或觸摸引腳時，波形如何像天線一樣吸收雜訊並產生亂跳。
  3. 驗證穩定性：將模式改為 INPUT_PULLUP，再次觀察波形是否回歸穩定的 HIGH（高電位）。
  4. 紀錄：在 docs/Oscilloscope.md 中描述「浮空引腳」的行為及其對數位系統的危害。

任務 2：計數器挑戰 - 量化機械彈跳 (Bouncing Capture Lab)

  1. 實作一個基礎計數器：每偵測到按鈕按下（LOW），pressCount 加 1。
  2. 觀察連發：嘗試輕按一次按鈕，記錄 Serial Monitor 顯示的計數值增加了多少（通常會超過 1）。
  3. 數據蒐集：分別用「輕按」、「正常按」、「用力重按」三種方式進行測試，觀察彈跳現象的差異。
  4. 報告：將實驗數據填入 docs/Oscilloscope.md 的表格中。

任務 3：專業級去彈跳演算法 (Pro Debounce & Millis Lab)

  1. 實作目標：使用 millis() 實作狀態機去彈跳邏輯。
  2. 嚴格要求：全程嚴禁使用 delay()。
  3. 功能實作：
    • 精準計數：確保不管按鈕如何彈跳，每次完整的按壓動作只觸發一次 pressCount++。
    • 狀態切換 (Toggle)：實作按一下亮燈、再按一下熄燈的功能。
    • 並行任務：在 loop() 中加入一個每 2 秒印出一次 "System Heartbeat" 的功能，驗證按鈕偵測不會阻塞背景任務。

檔案結構

  • src/main.cpp：你的程式實作核心。
  • include/debouncer.h：定義去彈跳的時間閾值（例如 50ms）與引腳編號。
  • docs/Oscilloscope.md：物理實驗報告，需填寫你的觀察結果。

1. 範本程式庫 配置建議

去彈跳作業的核心在於「時間過濾」與「狀態跟蹤」，範本應包含實驗性的檢測代碼，建議包含：

  • 📂 src/main.cpp：算法實作核心。提供一個會產生「連發錯誤」的基礎計數器代碼，學員需在此實作基於 millis() 的工業級去彈跳邏輯，確保按壓一次僅觸發一次計數。
  • 📂 include/debounce_config.h：定義時間閾值。學員需在此填入實驗測得的最佳 DEBOUNCE_TIME (例如 50ms)，體會不同機械結構對彈跳時間的差異。
  • 📂 README.md：物理實驗報表。提供表格讓學員記錄：「未去抖前的連發次數」 vs 「去抖後的精準度」。並描述「天線實驗」觀察到的浮空雜訊。
  • 📂 docs/TIMING_DIAGRAM.md：時序邏輯區。學員需在此說明為何使用 millis() 而非 delay()，並畫出一個簡單的狀態切換流程圖。

2. 作業任務部署細節

任務 1：「天線實驗」體會浮空狀態 (Floating State Lab)

  • 目標：親眼見證數位引腳在「未定義電位」時的混亂，建立必備的上拉/下拉直覺。
  • Classroom 部署建議：
    • 雜訊捕捉：引導學員將引腳設為 INPUT 且懸空，手部靠近觀察 Serial Plotter 的混亂跳動。
    • 穩寫驗證：開啟 INPUT_PULLUP 模式，記錄波形如何平滑回歸穩定的 HIGH。
    • 驗證要點：README 中需描述「浮空引腳」如何像天線一樣吸收環境電磁干擾，並解釋為何這在數位系統中是致命的。

任務 2：「計數器挑戰」量化機械彈跳 (Bouncing Capture Lab)

  • 目標：透過代碼量化肉眼看不見的毫秒級開合，理解硬體不完美的代價。
  • Classroom 部署建議：
    • 連發捕捉：實作一個靈敏的計數器（每偵測到 LOW 就 count++）。學員需記錄：輕按一下按鈕，實際產生的數據「連發」倍數。
    • 數據蒐集：分別用「輕按」、「重按」、「快按」三種方式記錄彈跳次數。
    • 提交結果：在 README 中彙整彈跳數據表，證明機械接觸點在物理層面的不穩定。

任務 3：「工業級去彈跳」非阻塞演算法 (Pro Debounce & Millis Lab)

  • 目標：掌握 millis() 非阻塞控制，這是開發複雜系統（如一邊跑馬達、一邊偵測按鈕）的基石。
  • Classroom 部署建議：
    • 無延遲去抖：強制要求移除所有 delay()。學員需實作「狀態對比」 logic：只有當電位穩定維持超過 N 毫秒，才判定為有效觸發。
    • 模式切換功能：實作一個 Toggle 功能（按一下亮燈，再按一下熄燈）。
    • 壓力測試：在 loop() 中加入一個模擬的背景任務（例如每秒印出一次訊息），驗證按鈕偵測不會阻塞背景任務的執行。

3. 韌體穩定性專家導師點評標準 (Firmware Robustness & Timing Standards)

此單元的價值在於 「對時間維度的精確掌控與非阻塞程式設計的運用」：

  • [ ] 時間管理正確性：是否完全移除了 delay()？改用 millis() 進行無感去抖？
  • [ ] 狀態邏輯嚴密性：是否正確處理了「前次狀態」與「當前狀態」的比對？
  • [ ] 物理解釋力：README 是否準確描述了負邏輯 (Active Low) 的工作原理？
