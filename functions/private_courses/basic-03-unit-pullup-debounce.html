基礎 3 - 訊號穩定性

數位控制：
上拉與按鈕抖動

學會馴服不穩定的物理訊號。從浮空狀態到毫秒級的彈跳，讓你的程式像工業級設備一樣穩固。

📡 1. 消失的邏輯：為什麼需要上拉電阻？

在電子數位世界中，「沒有接線」並不代表讀值是「零」。

浮空狀態 (Floating)

當 GPIO 未與任何明確電壓相連時，它就像天線一樣會吸收環境中的雜訊與電磁波。讀取值會在 0 與 1 之間瘋狂跳動，導致極高的系統不確定性。

解決方案：INPUT_PULLUP

透過軟體配置內部的上拉電阻，將引腳穩定拉至 3.3V (HIGH)。當物理按鈕按下時才將其拉至接地 (LOW)。

pinMode(PIN, INPUT_PULLUP);

🧠 Logic Snapshot

  • ✓ 未按下時：讀取值為 HIGH (1)
  • ✓ 按下按鈕時：讀取值為 LOW (0)
  • 💡 這就是標準的「負邏輯」配置。

📉 2. 物理現實：機械彈跳 (Debounce)

按鈕的內部的金屬片接觸的一瞬間，並非完美的「0 到 1」，而是會在極短的時間內（數個毫秒）產生密集的、猶如火花般的快速開合。

除了常見的微控按鈕，所有具備「機械接觸點」的感測器都需要進行去抖處理。

⏱️ 3. 軟體去抖：時間過濾術

我們不能改變物理特性，但可以透過演算法來「等待穩定」。這在專業開發中至關重要。

1

Edge Catching：偵測到電位第一次改變的瞬間。

2

Wait Epoch：忽略接下來 20~50 ms 內的所有波動。

3

Double Confirm：再次確認狀態。若穩定，才判定為有效操作。

Non-Blocking Only

為什麼不建議用 delay()？

在無人車開發中，系統必須 100% 同時監聽遙控指令與處理馬達。任何一丁點的阻塞 (blocking) 都會造成車輛反應遲鈍甚至斷線。

最佳實踐

使用 millis() 建立非阻塞(Non-blocking)計時狀態機是所有高級韌體的必修課。

GitHub Classroom 詳細操作指南

本單元教學重點：將「機械彈跳」這一物理現實問題，轉化為「非阻塞程式設計」的實習契機。

  1. Point 1：浮空與確定性 - 強調在數位系統中，任何一個 bit 都必須有明確的 0 或 1，不存在「中間地帶」。
  2. Point 2：硬體缺陷的補償 - 引導學員理解：完美的程式往往是為了相容不完美的硬體。
  3. Point 3：邁向 Non-blocking - 藉由去彈跳需求，正式引入 millis() 代替 delay()，這是後續開發複雜車輛控制的基石。
