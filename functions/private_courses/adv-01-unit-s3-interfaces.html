簡介

當你想在 ESP32 上跑攝影機或 AI 模型時，第一個撞上的牆往往不是程式邏輯，而是記憶體不夠用。 ESP32-S3 的 PSRAM 與 DMA 機制正是為此而生——理解它們的運作原理，才能讓你的邊緣運算專案從「勉強能動」進化為「穩定高效」。 本模組將帶你深入 S3 的記憶體分層架構與感測器介面，為後續的影像串流與 AI 推論打下硬體基礎。

ESP32-S3：為 AI 與影像而生

ESP32-S3 是樂鑫 (Espressif) 針對 AIoT（人工智慧物聯網） 場景所設計的旗艦級微控制器。與前代 ESP32 相比，S3 最關鍵的升級在於：它內建了專為向量運算設計的硬體加速指令集，並原生支援高速外部記憶體與攝影機介面。這使得它不再只是一顆「連上 Wi-Fi 的 MCU」，而是一台真正能在邊緣端執行影像辨識與 AI 推論的微型電腦。

雙核心 Xtensa LX7 架構

ESP32-S3 搭載兩顆 Xtensa LX7 核心，主頻最高可達 240 MHz。在典型的攝影機串流應用中，開發者通常會將兩顆核心做 任務分工：一顆核心專責處理 Wi-Fi 協議棧與 HTTP 串流推送，另一顆核心則負責影像擷取與 AI 推論。這種分工模式能避免無線通訊中斷導致的幀延遲，是即時影像系統穩定運作的關鍵。

更重要的是，LX7 核心內建了 向量擴展指令 (PIE — Processor Instruction Extensions)，可在單一指令週期內完成多筆乘法-累加運算 (MAC)。對於卷積神經網路 (CNN) 中大量的矩陣乘法，這能帶來 2～3 倍的加速效果，讓 S3 能在不依賴外部加速晶片的情況下執行輕量級 AI 模型。

注意 ESP32-S3 的向量指令主要針對 8-bit 與 16-bit 整數運算做了優化。若你的 AI 模型使用 float32 權重，需先透過 TensorFlow Lite 的量化工具轉換為 int8 格式，才能真正發揮硬體加速的效果。

感測器介面：DVP 與 SCCB

ESP32-S3 透過 DVP（Digital Video Port） 介面與攝影機模組通訊。DVP 是一種並列式 (Parallel) 資料傳輸協定，使用 8 至 16 條資料線同時傳輸像素，搭配 PCLK（像素時鐘）、VSYNC（垂直同步）與 HREF（水平參考）三條控制線來界定每一幀的邊界。

攝影機模組的暫存器設定（如解析度、曝光值、白平衡）則透過 SCCB（Serial Camera Control Bus） 協定進行控制。SCCB 本質上是 I2C 的子集，使用兩條線（SIOD 資料線與 SIOC 時鐘線）與感測器溝通。以常見的 OV2640 感測器為例，其 SCCB 從裝置位址為 0x30。

// 典型的 ESP32-S3 攝影機腳位配置 (以 ESP32-S3-CAM 為例)
#define CAM_PIN_PWDN -1 // 電源控制 (部分模組無此腳位)
#define CAM_PIN_XCLK 15 // 主時鐘輸出 (通常設為 20 MHz)
#define CAM_PIN_SIOD 4 // SCCB 資料線 (I2C SDA)
#define CAM_PIN_SIOC 5 // SCCB 時鐘線 (I2C SCL)

#define CAM_PIN_D7 16 // 資料線 D7 (MSB)
#define CAM_PIN_D6 17
#define CAM_PIN_D5 18
#define CAM_PIN_D4 12
#define CAM_PIN_D3 10
#define CAM_PIN_D2 8
#define CAM_PIN_D1 9
#define CAM_PIN_D0 11 // 資料線 D0 (LSB)

#define CAM_PIN_VSYNC 6 // 垂直同步信號
#define CAM_PIN_HREF 7 // 水平參考信號
#define CAM_PIN_PCLK 13 // 像素時鐘
提示 在選購 ESP32-S3 開發板時，注意確認板載攝影機連接器的腳位定義。不同廠商的開發板（如 Freenove、LILYGO、Seeed XIAO）使用的 GPIO 映射可能完全不同。務必查閱對應開發板的電路圖 (Schematic)，而非直接套用網路範例程式的腳位定義。

腳位的分類：固定 vs. 彈性

ESP32-S3 的 GPIO 可透過 IO MUX 與 GPIO Matrix 機制靈活映射功能。然而在攝影機應用中，以下腳位通常是固定占用的：DVP 的 8 條資料線（D0–D7）、PCLK、VSYNC、HREF，以及 SCCB 的 SIOD/SIOC 共 13 支腳位。加上 XCLK 主時鐘輸出，總計需要 14 支 GPIO。

剩餘的 GPIO 則可彈性用於其他外接裝置，例如：透過 I2C 連接 IMU 慣性感測器（陀螺儀/加速度計）、透過 SPI 驅動 TFT 螢幕、或使用 UART 與 GPS 模組通訊。在設計無人車或機器人系統時，合理規劃這些「可用腳位」是硬體架構設計的核心挑戰之一。

記憶體魔術：什麼是 PSRAM？

在嵌入式影像處理中，記憶體是最珍貴的資源。ESP32-S3 的內部 SRAM 僅有約 512 KB，扣除系統核心、Wi-Fi 協議棧與 FreeRTOS 所需的空間後，真正可供應用程式使用的可能只剩 200～300 KB。然而，一張未壓縮的 VGA 彩色影像（640×480 像素，RGB565 格式）就需要約 614 KB——這已經超出了可用的內部記憶體。PSRAM 正是解決這個瓶頸的關鍵技術。

內部 SRAM 的物理極限

讓我們用簡單的數學來理解這個問題。以下公式計算一張未壓縮影像佔用的記憶體大小：

Buffer 大小 = 寬度 × 高度 × 每像素位元組數

解析度 像素數 RGB565 大小 內部 SRAM 能否容納？
QQVGA (160×120) 19,200 ~38 KB 可以
QVGA (320×240) 76,800 ~150 KB 勉強
VGA (640×480) 307,200 ~614 KB 不可能
HD (1280×720) 921,600 ~1.8 MB 完全不可能

從表格可以清楚看出，如果想要在 VGA 以上的解析度工作（這是大多數影像辨識的基本需求），就必須使用外部記憶體。

PSRAM：外掛的記憶體魔法

PSRAM（Pseudo Static RAM，虛擬靜態隨機存取記憶體） 是一種透過串列介面掛載在 MCU 外部的記憶體晶片。它的內部結構其實是 DRAM（動態記憶體），但封裝了自動刷新電路，對外表現得像 SRAM 一樣簡單易用——因此得名「Pseudo（虛擬的）」Static RAM。

ESP32-S3 支援透過 Octal SPI (OPI) 介面連接最高 16 MB 的 PSRAM。OPI 使用 8 條資料線並列傳輸，時鐘頻率可達 80 MHz，理論頻寬約為 80 MB/s。這比前代 ESP32 的 4-bit SPI PSRAM 快了數倍。

警告 PSRAM 雖然空間大，但存取速度約為內部 SRAM 的 1/5 ～ 1/10。因此，頻繁存取的小型資料（如迴圈計數器、區域變數）應放在內部 SRAM，而大型且循序存取的資料（如影像緩衝區、AI 模型權重）才適合放在 PSRAM。搞錯配置策略會嚴重拖慢程式效能。

記憶體配置 API 對比

ESP-IDF 提供了多種記憶體配置函數，你必須根據資料的特性選擇正確的 API。標準的 malloc() 只會從內部 SRAM 分配空間，當空間不足時會直接返回 NULL，即使 PSRAM 還有好幾 MB 的空間也不會去使用。

// ❌ 錯誤示範：malloc() 無法分配 PSRAM
uint8_t* buf = (uint8_t*)malloc(614400); // VGA RGB565 需要 ~600KB
// buf == NULL (內部 SRAM 空間不足)

// ✅ 正確做法：使用 heap_caps_malloc 指定 SPIRAM
uint8_t* buf = (uint8_t*)heap_caps_malloc(614400, MALLOC_CAP_SPIRAM);
if (buf == NULL) {
ESP_LOGE(TAG, "PSRAM 配置失敗！請確認 PSRAM 已啟用");
}

// ✅ 簡便寫法：ps_malloc() 是 heap_caps_malloc 的封裝
uint8_t* buf2 = (uint8_t*)ps_malloc(614400);
提示 在分配大型緩衝區之前，務必先呼叫 heap_caps_get_free_size(MALLOC_CAP_SPIRAM) 檢查剩餘可用空間。這個習慣能避免因記憶體不足而導致的 panic 重啟——這是 ESP32 攝影機專案中最常見的錯誤之一。

SRAM 與 PSRAM 的分工策略

理想的記憶體分配策略是根據「存取頻率」與「資料大小」來決定存放位置：

資料類型 建議位置 理由
堆疊 (Stack)、區域變數 內部 SRAM 極高頻率的隨機存取
FreeRTOS Task 堆疊 內部 SRAM 任務切換需要極低延遲
影像幀緩衝區 (Frame Buffer) PSRAM 數百 KB 的大型循序存取
AI 模型權重 (.tflite) PSRAM 一次載入、循序讀取

高速傳輸的幕後推手：DMA

在攝影機串流應用中，影像感測器每秒會產生數 MB 的原始像素資料。如果每一個位元組都需要 CPU 親自搬運（從周邊暫存器讀出，再寫入記憶體），CPU 將被完全佔滿，毫無餘裕處理 Wi-Fi 通訊、馬達控制或 AI 推論。DMA（Direct Memory Access，直接記憶體存取） 就是為了解決這個瓶頸而生的硬體機制。

DMA 的運作原理

DMA 控制器是一個獨立於 CPU 的硬體單元，能在不佔用 CPU 的情況下自動將資料從一個記憶體位址搬運到另一個位址，或從周邊設備搬到記憶體。CPU 只需要下達一次搬運指令（設定來源位址、目標位址與長度），DMA 控制器就會接手完成整個傳輸，傳輸結束後再透過中斷通知 CPU。

在 ESP32-S3 的攝影機驅動中，DMA 的工作流程如下：

  1. 攝影機感測器透過 DVP 介面，在每個 PCLK 上升緣輸出一個像素。
  2. DMA 控制器自動將像素資料搬入預先配置的環形緩衝區 (Ring Buffer)。
  3. 當一整幀影像的 VSYNC 信號觸發時，DMA 通知 CPU：「一幀完整影像已就緒」。
  4. CPU 此時才介入，將幀資料進行 JPEG 壓縮或傳送至網路。
注意 ESP32-S3 的 DMA 控制器使用鏈結描述符 (Linked List Descriptors) 來管理多段不連續的記憶體區塊。ESP-IDF 的攝影機驅動會自動建立這些描述符鏈——你通常不需要手動操作，但理解這個機制有助於排查「幀損毀」或「DMA 溢出」的錯誤。

有無 DMA 的效能對比

以下數據模擬了 VGA 解析度（640×480）、RGB565 格式、30 FPS 的影像傳輸場景，比較 CPU 手動搬運（Polling）與 DMA 自動搬運的差異：

指標 CPU Polling 模式 DMA 模式
每幀資料搬運時間 ~8 ms（CPU 全速搬運） ~8 ms（DMA 硬體搬運）
搬運期間 CPU 占用率 100% < 1%
同時可執行其他任務 不行 Wi-Fi / AI / 控制邏輯
30 FPS 下的每秒 CPU 成本 ~240 ms（佔去一顆核心的 24%） ~0.3 ms（僅中斷處理）

可以看出，DMA 不是讓搬運速度變快，而是讓 CPU 完全解放。在 30 FPS 的串流中，CPU Polling 模式每秒浪費約 240 ms 在純粹的資料搬運上；而 DMA 模式下，這些時間可以用來執行 AI 推論或處理控制邏輯。

警告 DMA 緩衝區必須分配在內部 SRAM 的 DMA 可存取區域。並非所有記憶體位址都支援 DMA 存取——特別是 PSRAM 的部分區段可能不在 DMA 可達範圍內。使用 heap_caps_malloc(size, MALLOC_CAP_DMA) 確保分配的記憶體符合 DMA 對齊與位址限制。

匯流排競爭：DMA 與 PSRAM 的潛在衝突

ESP32-S3 的 DMA 控制器與 PSRAM 控制器共享部分內部匯流排頻寬。當攝影機 DMA 正在高速搬運像素資料的同時，如果 CPU 也正在大量讀寫 PSRAM（例如執行 AI 推論），兩者可能產生匯流排競爭，導致影像出現雜訊或 I2C 通訊逾時。

這種現象在 PSRAM 設定為 80 MHz Octal SPI 時特別明顯——因為更高的時鐘頻率意味著匯流排被佔用的時間更密集。一種常見的解決方案是將 PSRAM 頻率降至 40 MHz，犧牲部分頻寬來換取更乾淨的影像信號與更穩定的 I2C 通訊。

// platformio.ini 中的關鍵配置
[env:esp32-s3-cam]
platform = espressif32
board = esp32-s3-devkitc-1
framework = arduino
board_build.arduino.memory_type = qio_opi ; OPI PSRAM 模式
build_flags =
-DBOARD_HAS_PSRAM
-DCONFIG_SPIRAM_SPEED_80M ; 或 40M 以降低匯流排干擾
提示 如果你在高解析度串流時發現影像偶爾出現橫向的彩色雜訊線條（通常是紫色或綠色的），這很可能是 DMA 與 PSRAM 匯流排競爭所造成的。嘗試降低 PSRAM 頻率或減少解析度，觀察問題是否改善。這是在作業「感測器與 PSRAM 聯動除錯」中需要深入探索的議題。

本程式庫提供 ESP32-S3 核心開發練習，重點在於硬體資源調度與底層記憶體管理。

課程目標

  • 掌握 ESP32-S3 的記憶體分層架構（SRAM vs PSRAM）。
  • 實作高效能的影像緩衝區管理技術。
  • 觀察高速匯流排競爭對訊號品質的影響。

作業任務

任務 1：PSRAM 初始化與壓力測試 (Memory Stress Test)

  • 內容：實作 heap_caps_malloc 指令申請 PSRAM，並撰寫計時器測量寫入速度。
  • 目標：理解為何在邊緣 AI 運算中需要 PSRAM 以及其效能物理極限。
  • 提交：將數據紀錄於 docs/hardware_bench.md 中。

任務 2：影像緩衝區管理 - ROI 提取 (Pointer Arithmetic Lab)

  • 內容：在 src/roi_math.h 中實作指針運算邏輯。
  • 目標：學會如何在不進行記憶體拷貝的情況下，利用步幅 (Stride) 存取影像特定區域 (ROI)。

任務 3：感測器與 PSRAM 聯動除錯 (Signal Integrity Lab)

  • 內容：觀察 PSRAM 在 80MHz 與 40MHz 下，對感測器 I2C 訊號或影像雜訊的影響。
  • 目標：培養對硬體系統級干擾的偵錯與權衡能力。

專案結構

  • platformio.ini: 核心硬體配置 (啟用 PSRAM 支援)。
  • src/: 核心程式碼，包含壓測腳本與指針運算。
  • docs/: 存放實驗報告 hardware_bench.md。

評分標準

  1. 內存安全：是否正確檢查 Heap 容量。
  2. 指針嚴謹度：ROI 計算是否考慮了對齊與越界。
  3. 實驗完整度：報告中是否如實紀錄物理現象並給出合理解釋。

本課程為邊緣 AI 系列單元之一。

針對 adv-01-unit-s3-interfaces (ESP32-S3 感測器介面與 PSRAM) 單元，這是一個深入探討「硬體資源調度」與「底層記憶體管理」的課程，是實施邊緣 AI 運算的物理基礎。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本倉庫 (Template 程式庫) 配置建議

進階硬體單元的範本應模擬一個「小型作業系統」的環境，建議包含：

  • 📂 platformio.ini：預置關鍵的編譯選項，如 -DBOARD_HAS_PSRAM，並正確設定 memory_type（如 qio_opi），確保學員能直接調用 PSRAM。
  • 📂 src/mem_lab.cpp：提供一個測試框架，內含精確到微秒 ($\mu s$) 的計時器，用於測量 SRAM 與 PSRAM 的效能差異。
  • 📂 src/roi_processor.h：定義影像處理結構體，將「指針運算 (Pointer Math)」邏輯留給學員補完。
  • 📂 docs/hardware_report.md：預置結構化的實驗報告，用於紀錄不同時鐘頻率下的訊號表現。

2. 作業任務部署細節

任務 1：PSRAM 初始化與壓力測試 (Memory Stress Test)

  • 目標：理解 ESP32-S3 的記憶體分層架構，掌握特殊分配指令。
  • Classroom 部署建議：
    • 關鍵代碼檢核： 
      // 學生需實作：使用正確的 Capability 標籤申請 PSRAM
uint8_t* buf = (uint8_t*) heap_caps_malloc(1024*1024, MALLOC_CAP_SPIRAM);
    • 數據提交：學員需在 hardware_report.md 中填入百萬次單次 1MB 的寫入時間。導師核閱其數據是否展現出「SRAM 快、PSRAM 慢」的物理特徵。

任務 2：影像緩衝區管理 - ROI 提取 (Pointer Arithmetic Lab)

  • 目標：在不複製數據的情況下，利用指針偏移技術存取影像特定區域。
  • Classroom 部署建議：
    • 驗證方式：學員需實作一個函數，計算給定 $(x, y)$ 座標後，在寬度為 640 的 RGB565 Buffer 中的指針地址。
    • Autograding：撰寫一個模擬單元測試，提供一組 mock 的 Buffer 並檢核學員提取的 ROI 數值是否與預期一致。這能強化學生對「步幅 (Stride)」與「像素格式」的理解。

任務 3：感測器與 PSRAM 聯動除錯 (Signal Integrity Lab)

  • 目標：觀察高速匯流排競爭對類比訊號（影像雜訊）的影響。
  • Classroom 部署建議：
    • 證據提交：學員需將 PSRAM 設定在 80MHz (Octal SPI) 下產生的影像雜訊截圖上傳。
    • 分析任務：在報告中描述當 PSRAM 全速讀寫時，I2C 感測器（如陀螺儀）是否出現數據丟失或延遲現象。這能培養學員對「硬體系統級除錯」的直覺。

3. 進階硬體單元導師點評標準 (Advanced Benchmarks)

此單元的價值在於 「對硬體物理邊界的認知」：

  • [ ] 內存安全意識：學員在申請大容量 Buffer 前是否檢查了 heap_caps_get_free_size？
  • [ ] 指針嚴謹度：ROI 計算是否考慮了邊界對齊（Alignment）與越界檢查？
  • [ ] 資源權衡能力：是否能解釋為什麼在某些場景下需要手動調降 PSRAM 頻率以獲得更乾淨的影像？
