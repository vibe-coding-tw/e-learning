🤖 1. 什麼是 FSM (有限狀態機)？

一個系統在任何時間點，都處於「有限個狀態」中的其中一個。這能讓系統行為從混亂變得「可預測」。

State

「正在做什麼」。如：前進、待機、急停。

Event

「發生了什麼」。如：按下按鈕、偵測障礙。

💻 3. 程式實作：switch...case 與 enum

擺脫魔術數字。使用 enum 定義語義化狀態，並用 switch 攤平多層嵌套邏輯。

Clean Architecture

  • 消滅 Magic Numbers：用 IDLE 代替 0，代碼即文檔。
  • 高擴充性：新增工作模式只需在 switch 中增加一個 case。
  • 事件驅動：由條件觸發轉移，而非程式流隨機跳轉。
enum State { IDLE, AUTO };
State s = IDLE;

switch(s) {
  case IDLE:
    if(btn) s = AUTO;
    break;
  case AUTO:
    motorOn();
    break;
}

核心任務：設計機器人智慧大腦

透過 FSM 架構，賦予機器人在複雜任務與異常環境間冷靜切換的能力。

🕹️ 60 min

【作業 10-1】

「三模切換骨幹」

實作 IDLE、REMOTE、AUTO 狀態機。透過按鈕進行有序切換，並以 RGB LED 同步呈現當前系統模式。

「避障行為原子化」

將自動避障細分為 SEARCH 到 TURN 的有序子狀態機。結合 millis() 實現非阻塞的動作時間窗口管理。

🆘 45 min

【作業 10-3】

「搶佔式安全防線」

新增 EMERGENCY 狀態。實作斷線或硬體異常時的瞬間轉移邏輯，並設計嚴謹的防誤觸救援解鎖機制。

有限狀態機基礎實驗室

任務 1：基礎控制狀態機 (Core FSM Architecture)

角色扮演：韌體架構工程師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 狀態定義：使用 enum 定義三核心狀態：IDLE (待機)、REMOTE (手控)、AUTO (自律)。
  3. 中轉邏輯實作：構建 switch...case 核心，根據按鈕事件與通訊報文進行狀態轉移判定。
  4. 視覺反饋實施：根據當前 State 控制 RGB LED（如：待機綠、手控藍、自律紅），實現狀態的可視化診斷。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 命名規範：是否完全捨棄了「魔術數字」，改採具備高可讀性的 enum 命名狀態？
  • [ ] 邏輯一致性：狀態切換時，硬體端的物理動作與 LED 燈號是否與內部軟體模式完全同步？

任務 2：避障邏輯狀態機 (Obstacle Logic Sub-FSM)

角色扮演：自動駕駛演算法工程師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 子狀態細分：將 AUTO 模式細化為 SEARCH -> DETECTED -> BACKING -> TURN 序列。
  2. 非阻塞計時轉場：利用 millis() 設定轉發計時器，實現如「後退 500ms 後自動轉場至轉向模式」的非阻塞邏輯。
  3. 邏輯攤平重構：移除所有深層嵌套的 if-else，改由事件驅動狀態機進行扁平化處理。
  4. 閉環實測：驗證車體在突發障礙下，是否能順暢執行整套「察覺、閃避、重啟搜尋」的閉環行為。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 計時精準度：狀態間的轉換延遲是否準確執行，且未對系統主循環 (Loop) 造成任何阻塞掛起？
  • [ ] 狀態覆蓋率：避障邏輯是否周全考慮了所有轉場邊界，不存在導致車輛「原地死鎖」的邏輯空洞？

任務 3：異常救援模式：最高優先權 (Emergency Recovery & Failsafe)

角色扮演：安全系統工程師 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 救援模式接入：新增 EMERGENCY_STOP 狀態，並在 loop() 最頂部實施最高搶佔優先權判定。
  2. 失效即轉移：若偵測到藍牙心跳丟失、電池電量過低或緊急虛擬鈕觸發，立即強行關閉任何運轉任務並保速鎖死。
  3. 解鎖序列設計：實作防誤觸解鎖邏輯（如：特定指令序列或控制端長按 3s），確保現場安全後方可回歸 IDLE。
  4. 報告提交：評估緊急狀態的響應時間，並將具備安全防護邏輯的代碼 Push 至 GitHub PR。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 搶佔實效性：在任何高頻運算模式下（如全速 AUTO），緊急觸發是否能瞬間（<20ms）接管硬體控制權？
  • [ ] 安全嚴密性：解鎖機制是否能有效防止在未排除硬體故障的情況下，系統意外重新啟動？

GitHub Classroom 詳細操作指南

本單元教學重點：從「功能導向」進化至「系統狀態」導向的系統建模。

  1. 重點 1：邏輯的可視化 - 引導學員不只是盲目編碼，而是先繪製狀態轉移圖，釐清行為邊界與事件觸發關係。
  2. 重點 2：switch-case 的優雅性 - 強調 FSM 是解決代碼「義大利麵化 (Spaghetti Code)」的最佳武器。
  3. 重點 3：防禦性救援機制 - 灌輸工業級系統設計思維：在任何不可控異常發生時，系統必須能進入穩定的、已知的低權限安全狀態。
