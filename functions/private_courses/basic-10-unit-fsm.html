💻 3. switch...case 與 enum

擺脫魔術數字。使用 enum 定義語義。

Clean Architecture

  • 消滅 Magic Numbers：用 IDLE 代替 0。
  • 高擴充性：新增模式只需在 switch 增加 case。
  • 事件驅動：由條件觸發轉移。
enum State { IDLE, AUTO };
State s = IDLE;

switch(s) {
  case IDLE:
    if(btn) s = AUTO;
    break;
  case AUTO:
    motorOn();
    break;
}

課程單元 基礎 10 - 邏輯架構

作業單元 10：有限狀態機 (FSM) — 機器人大腦的邏輯架構

歡迎來到本程式庫！本單元的目標是學習如何從單純的「功能堆疊」進化到「系統架構設計」。你將使用 有限狀態機 (Finite State Machine, FSM) 的概念，為機器人構建一個穩定、可預測且具備異常處理能力的邏輯架構。

🎯 學習目標

  • 掌握 確定性運算 (Deterministic Computing) 的思維。
  • 學習將混亂的 if-else 嵌套結構重構成清晰的 switch-case 狀態轉移邏輯。
  • 實作具備最高優先權的「強制救援」邏輯，確保系統安全性。

🛠️ 任務內容

任務 1：基礎控制狀態機 (Core FSM Architecture)

  • 目標：建立三模式切換骨幹。
  • 實作內容：
    • 定義 IDLE (紅燈)、REMOTE (藍燈)、AUTO (綠燈) 三種狀態。
    • 按下 按鍵 A 切換至下一個狀態。
    • 按下 按鍵 B 無論目前為何種狀態，立即回歸 IDLE（除非在 EMERGENCY 狀態）。
  • 驗證：確保使用 enum 定義狀態，代碼中不出現魔術數字。

任務 2：避障邏輯子狀態機 (Obstacle Logic Sub-FSM)

  • 目標：在 AUTO 模式內實作非阻塞的子狀態管理。
  • 實作內容：
    • 定義子狀態：SEARCH (掃描)、BACKUP (後退)、TURN (轉向)。
    • 當偵測到障礙物時，觸發轉移：SEARCH -> BACKUP (持續 500ms) -> TURN (持續 300ms) -> SEARCH。
  • 驗證：確保使用 millis() 進行計時，不使用 delay()，以維持系統的響應性。

任務 3：緊急救援模式 (Emergency Failsafe)

  • 目標：打造最高優先權的安全防線。
  • 實作內容：
    • 新增 EMERGENCY 狀態（特徵：紅燈閃爍、馬達停止）。
    • 模擬異常觸發（如感測器超時或特定錯誤訊號），系統必須能瞬間跳轉至 EMERGENCY。
  • 驗證：確認救援模式能接管所有其他狀態，且回應迅速。

📂 程式庫結構說明

  • src/main.cpp：主程式邏輯與狀態機調度區。
  • include/FSM_Definitions.h：定義所有的 enum 狀態類型。
  • docs/state_map.md：邏輯架構圖，建議使用 Mermaid 語法繪製你的狀態轉移流程。

🚀 如何開始

  1. 複製 (Clone) 此程式庫至本地。
  2. 使用 Visual Studio Code 開啟專案（需安裝 PlatformIO 擴充功能）。
  3. 在 include/FSM_Definitions.h 中定義你的狀態。
  4. 在 src/main.cpp 的 handleStateTransitions() 與 executeStateLogic() 中實作邏輯。
  5. 點擊下方工具列的 PlatformIO: Build 進行編譯，PlatformIO: Upload 進行上傳。

📝 繳交要求

  • [ ] 代碼需結構清晰，邏輯嚴密。
  • [ ] 所有的狀態轉移必須符合預期，無死鎖。
  • [ ] 提交前請確保 docs/state_map.md 已更新，反映你的最終邏輯設計。

2. 作業任務部署細節

任務 1：基礎控制狀態機 (Core FSM Architecture Lab)

  • 目標：建立三模式切換骨幹，掌握「語義化編程」以提升代碼的可讀性與穩定性。
  • Classroom 部署建議：
    • 三模架構實作：定義 IDLE (紅燈)、REMOTE (藍燈)、AUTO (綠燈)。
    • 轉移觸發器：實作按下按鍵 A 切換至次一狀態，按下按鍵 B 回歸 IDLE。
    • 驗證要點：關鍵工程審核。學員是否使用了 enum？代碼中是否還存在 state == 1 這種數字判斷？（應使用 state == REMOTE）。

任務 2：避障邏輯狀態機 (Obstacle Logic Sub-FSM Lab)

  • 目標：在 AUTO 模式內實作「子狀態機 (Sub-FSM)」，學習如何在非阻塞架構下管理具備時間窗口的有序動作。
  • Classroom 部署建議：
    • 行為原子化：定義子狀態 SEARCH (掃描障礙)、BACKUP (後退 500ms)、TURN (轉向 300ms)。
    • 精密轉場：結合 millis()。例如：當進入 BACKUP 時記錄起點，時間到自動轉移至 TURN。
    • 驗證要點：避障動作是否流暢？是否因使用 delay 而導致狀態切換過程中無法取消任務？（應維持 Loop 的非阻塞運行）。

任務 3：異常救援模式：最高優先權 (Emergency Failsafe Lab)

  • 目標：打造「搶佔式 (Preemptive)」安全防線。當偵測到極端異常時，系統必須能跨越目前所有狀態，強制進入安全鎖定。
  • Classroom 部署建議：
    • 救援轉移實作：新增 EMERGENCY 狀態（特徵：紅燈全快閃、馬達鎖死）。
    • 異常捕捉器：例如監視通訊心跳，若 2 秒無訊號，無論目前處於 AUTO 還是 REMOTE，必須瞬間轉移至 EMERGENCY。
    • 解鎖機制設計：實作嚴謹的解鎖序列。
    • 驗證要點：救援模式是否具備最高執行權限？其轉移響應是否在 20ms 以內？

3. 邏輯嚴密性與確定性架構點評標準 (Logic Architecture & Deterministic Standards)

此單元的價值在於 「對系統狀態的穷舉能力與對行為邊界條件的嚴格定義」：

  • [ ] 狀態窮舉性：是否考慮了所有可能的狀態變化？是否有狀態「死鎖」的情況？
  • [ ] 代碼架構優雅度：是否完全消滅了麵條式的 if-else 嵌套，改由清晰的 switch-case 驅動？
  • [ ] 失效安全性 (Failsafe)：緊急狀態是否能在任何時刻可靠接管控制權？
