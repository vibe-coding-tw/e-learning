簡介

當裝置的行為隨「目前處於什麼狀態」而改變時，你需要的就是有限狀態機（FSM）。 FSM 把複雜的流程拆解成若干明確的狀態與轉移條件，讓程式碼結構清晰、易於擴充與除錯。 無論是遙控車的「待機 → 行駛 → 停車」還是連線流程的「掃描 → 連線 → 已連線」，FSM 都是最自然的建模工具。

情境
你的車子有四種模式：待機、手動遙控、自動循線、緊急停止。 沒有 FSM 時，程式裡散布著大量 if-else 巢狀判斷，新增一個模式就要到處修改、容易遺漏。 改用 FSM 後，每個狀態獨立封裝、轉移條件集中管理——新增模式只需定義一個新狀態與其轉移規則。

什麼是 FSM (有限狀態機)？

有限狀態機（Finite State Machine, FSM）是一種數學計算模型，用來描述系統在有限數量的狀態之間如何根據輸入事件進行轉移。在嵌入式系統開發中，FSM 是最基礎也最強大的架構設計工具——它讓你的程式碼從「隨機反應」進化到「確定性運算」。

一台遙控車可能同時面對多種需求：遙控行駛、自動循線、緊急停止。若沒有 FSM，你可能會寫出層層嵌套的 if-else 判斷來處理這些情境，導致程式碼難以閱讀、容易出錯、擴充困難。FSM 提供了一套結構化的方法，將每種「行為模式」定義為獨立的狀態，再透過明確的事件觸發轉移——讓系統行為完全可預測。

FSM 的三大核心元素

每一個有限狀態機都由三個基本元素組成：

元素 定義 車載範例
狀態 (State) 系統當前「正在做什麼」的描述 IDLE（待機）、REMOTE（遙控）、AUTO（自動）
事件 (Event) 觸發狀態變更的外部輸入或條件 按下按鍵 A、偵測到障礙物、通訊斷線
轉移 (Transition) 從一個狀態切換到另一個狀態的動作 IDLE → 按鍵 A → REMOTE
備註 「有限」的意義在於：系統的所有可能行為都可以被窮舉。這代表你可以在設計階段就驗證所有邊界情況，確保沒有「未定義行為」導致系統當機或失控。

確定性運算 (Deterministic Computing)

FSM 的最大優勢在於確定性：給定相同的當前狀態和相同的輸入事件，系統永遠會轉移到相同的下一個狀態。這意味著你的機器人行為完全可重現、可測試、可驗證——這在安全性要求高的場景（如自動駕駛、工業控制）中至關重要。

// 確定性行為範例：相同輸入 → 相同結果
if (currentState == IDLE && buttonA_pressed) {
    currentState = REMOTE; // 永遠是這個結果
}

// 非確定性行為（不好的寫法）：結果取決於隱藏條件
if (buttonA_pressed && !someFlag && timer > 500) {
    mode = 2; // 2 是什麼？哪個模式？
}

為什麼不用 if-else？

當系統只有兩三個判斷條件時，if-else 夠用。但當你的車子有四種模式、每個模式有三四種退出條件，再加上錯誤處理邏輯，巢狀的 if-else 會迅速變成「義大利麵條碼（Spaghetti Code）」——難以追蹤程式流向，新增一個模式就要修改十幾處，漏改一處就是一個 bug。

提示 FSM 的核心價值不只是「程式碼更乾淨」，而是讓你能在動筆寫程式之前，就用狀態圖窮舉所有系統行為——找出潛在的死鎖或漏洞。這是真正的「設計先行」工程思維。

狀態轉移圖

狀態轉移圖（State Transition Diagram）是 FSM 的視覺化表示法。它使用圓圈代表狀態、箭頭代表轉移、箭頭上的標籤代表觸發事件。在你開始撰寫任何一行程式碼之前，先畫出狀態轉移圖，是避免邏輯漏洞最有效的方法。

一張完整的狀態轉移圖能讓你在設計階段就回答以下問題：系統有幾種可能的狀態？每個狀態在什麼條件下會轉移到哪裡？是否存在「死鎖」——某個狀態永遠無法離開？是否存在「未覆蓋的邊界」——某個事件在某個狀態下沒有被處理？

狀態圖的組成元素

符號 意義 繪製規則
○ 圓圈 一個狀態 內部標注狀態名稱，如 IDLE
→ 箭頭 一個轉移 從來源狀態指向目標狀態
箭頭上標籤 觸發條件（事件） 標注什麼條件會觸發此轉移
◉ 雙圈 / 粗框 初始狀態 系統上電後的起始點
✖ 叉號圈 終止狀態（選用） 系統完全關閉（嵌入式少用）

範例：遙控車三模系統

以本課程的遙控車為例，我們定義三種運行模式與一個安全模式。使用 Mermaid 語法可以這樣描述：

// Mermaid 狀態圖語法
stateDiagram-v2
    [*] --> IDLE
    IDLE --> REMOTE : 按鍵A
    REMOTE --> AUTO : 按鍵A
    AUTO --> IDLE : 按鍵A
    REMOTE --> IDLE : 按鍵B
    AUTO --> IDLE : 按鍵B

    %% 緊急狀態可從任何地方進入
    IDLE --> EMERGENCY : 異常偵測
    REMOTE --> EMERGENCY : 異常偵測
    AUTO --> EMERGENCY : 異常偵測
    EMERGENCY --> IDLE : 解鎖序列完成
警告 繪製狀態圖時，務必檢查每個狀態是否都有「離開的路徑」。如果某個狀態只有進入箭頭、沒有離開箭頭，代表系統可能陷入死鎖（Deadlock）。唯一允許的例外是有意設計的終止狀態。

設計流程

1

列舉所有狀態

把系統所有可能的運行模式寫下來，包括錯誤處理與安全模式。

2

定義所有事件

列出所有可能的輸入：按鈕、感測器訊號、計時器超時、通訊事件等。

3

填寫轉移表

對每一對「狀態 × 事件」組合，決定下一個狀態。空白格代表「忽略該事件」或「保持原狀態」。

4

驗證安全性

確認 EMERGENCY 可以從任何狀態進入，且每個正常狀態至少有一條通往 IDLE 的路徑。

提示 建議將狀態圖存放在專案的 docs/state_map.md 中，使用 Mermaid 語法撰寫。GitHub 能直接渲染 Mermaid 圖表，讓團隊成員能在 PR 審查時一目了然地理解你的邏輯設計。

switch...case 與 enum

在 C/C++ 中實作 FSM，最經典也最清晰的結構就是 enum 搭配 switch...case。enum 讓每個狀態擁有語義化的名稱，switch...case 則提供了一對一對應的分支結構——每個 case 就是一個狀態的行為定義。

為什麼需要 enum？消滅魔術數字

所謂「魔術數字（Magic Number）」是指程式碼中出現的裸數字常數，例如 if (state == 2)。三個月後你再看這行程式碼，還記得 2 代表什麼嗎？使用 enum 定義狀態，可以讓程式碼自我說明：

// ❌ 不好的寫法：魔術數字
int state = 0;
if (state == 0) { /* 待機？*/ }
if (state == 1) { /* 遙控？*/ }
if (state == 2) { /* 自動？*/ }

// ✅ 正確寫法：語義化 enum
enum SystemState {
    IDLE, // 待機模式 — 紅燈
    REMOTE, // 遙控模式 — 藍燈
    AUTO, // 自動模式 — 綠燈
    EMERGENCY // 緊急停止 — 紅燈閃爍
};

SystemState currentState = IDLE;

switch...case 實作狀態調度

switch...case 結構天然適合 FSM：每個 case 對應一個狀態，內部處理該狀態的行為與轉移條件。相比多層 if-else，switch 更扁平、更容易添加新狀態。

void loop() {
    switch (currentState) {
        case IDLE:
            setLED(RED);
            if (buttonA()) currentState = REMOTE;
            break;

        case REMOTE:
            setLED(BLUE);
            handleRemoteControl();
            if (buttonA()) currentState = AUTO;
            if (buttonB()) currentState = IDLE;
            break;

        case AUTO:
            setLED(GREEN);
            runLineFollowing();
            if (buttonA()) currentState = IDLE;
            if (buttonB()) currentState = IDLE;
            break;

        case EMERGENCY:
            flashLED(RED);
            stopAllMotors();
            if (unlockSequenceComplete()) currentState = IDLE;
            break;
    }

    // 全域檢查：緊急條件優先於一切
    if (detectAnomaly() && currentState != EMERGENCY) {
        currentState = EMERGENCY;
    }
}
重要 永遠不要忘記 break！在 C/C++ 中，switch 語句如果省略 break，程式會「穿透（fall-through）」到下一個 case，導致多個狀態的程式碼被同時執行——這是 FSM 中最常見的低階 bug。

架構優勢對比

面向 if-else 巢狀 enum + switch
可讀性 隨狀態數量指數下降 每個 case 獨立清晰
擴充性 新增模式需修改多處 新增一個 case 即可
除錯 難以追蹤當前分支 印出 currentState 即知
編譯器輔助 無法檢查遺漏 可警告未處理的 enum 值
最佳實踐 將所有 enum 定義集中放在 include/FSM_Definitions.h 標頭檔中。這不僅讓狀態定義一目了然，也方便多個 .cpp 檔案共用相同的狀態定義，避免重複宣告造成的不一致。

作業單元 10：有限狀態機 (FSM) — 機器人大腦的邏輯架構

歡迎來到本程式庫！本單元的目標是學習如何從單純的「功能堆疊」進化到「系統架構設計」。你將使用 有限狀態機 (Finite State Machine, FSM) 的概念，為機器人構建一個穩定、可預測且具備異常處理能力的邏輯架構。

🎯 學習目標

  • 掌握 確定性運算 (Deterministic Computing) 的思維。
  • 學習將混亂的 if-else 嵌套結構重構成清晰的 switch-case 狀態轉移邏輯。
  • 實作具備最高優先權的「強制救援」邏輯，確保系統安全性。

🛠️ 任務內容

任務 1：基礎控制狀態機 (Core FSM Architecture)

  • 目標：建立三模式切換骨幹。
  • 實作內容：
    • 定義 IDLE (紅燈)、REMOTE (藍燈)、AUTO (綠燈) 三種狀態。
    • 按下 按鍵 A 切換至下一個狀態。
    • 按下 按鍵 B 無論目前為何種狀態，立即回歸 IDLE（除非在 EMERGENCY 狀態）。
  • 驗證：確保使用 enum 定義狀態，代碼中不出現魔術數字。

任務 2：避障邏輯子狀態機 (Obstacle Logic Sub-FSM)

  • 目標：在 AUTO 模式內實作非阻塞的子狀態管理。
  • 實作內容：
    • 定義子狀態：SEARCH (掃描)、BACKUP (後退)、TURN (轉向)。
    • 當偵測到障礙物時，觸發轉移：SEARCH -> BACKUP (持續 500ms) -> TURN (持續 300ms) -> SEARCH。
  • 驗證：確保使用 millis() 進行計時，不使用 delay()，以維持系統的響應性。

任務 3：緊急救援模式 (Emergency Failsafe)

  • 目標：打造最高優先權的安全防線。
  • 實作內容：
    • 新增 EMERGENCY 狀態（特徵：紅燈閃爍、馬達停止）。
    • 模擬異常觸發（如感測器超時或特定錯誤訊號），系統必須能瞬間跳轉至 EMERGENCY。
  • 驗證：確認救援模式能接管所有其他狀態，且回應迅速。

📂 程式庫結構說明

  • src/main.cpp：主程式邏輯與狀態機調度區。
  • include/FSM_Definitions.h：定義所有的 enum 狀態類型。
  • docs/state_map.md：邏輯架構圖，建議使用 Mermaid 語法繪製你的狀態轉移流程。

🚀 如何開始

  1. 複製 (Clone) 此程式庫至本地。
  2. 使用 Visual Studio Code 開啟專案（需安裝 PlatformIO 擴充功能）。
  3. 在 include/FSM_Definitions.h 中定義你的狀態。
  4. 在 src/main.cpp 的 handleStateTransitions() 與 executeStateLogic() 中實作邏輯。
  5. 點擊下方工具列的 PlatformIO: Build 進行編譯，PlatformIO: Upload 進行上傳。

FSM 作業的核心在於「語義化狀態管理」與「邏輯地圖」，範本應包含狀態結構定義與狀態圖範本，建議包含：

  • 📂 src/main.cpp：狀態機調度區。學員在此定義 enum State 並構建核心的 switch(currentState) 結構，處理不同模式間的轉換。
  • 📂 docs/STATE_DIAGRAM.md：邏輯架構圖。要求學員使用 Mermaid 語法或流程圖工具繪製其狀態轉移邏輯，包含轉移條件（Event）。
  • 📂 README.md：狀態驗證日誌。提供表格讓學員記錄：從「待機 (IDLE)」切換到「遙控 (REMOTE)」的觸發條件，以及 RGB LED 的對應顏色反饋。
  • 📂 include/FSM_Definitions.h：狀態語義標頭檔。強制要求學員在此集中定義所有狀態，杜絕魔術數字 (Magic Numbers)。

2. 作業任務部署細節

任務 1：基礎控制狀態機 (Core FSM Architecture Lab)

  • 目標：建立三模式切換骨幹，掌握「語義化編程」以提升代碼的可讀性與穩定性。
  • Classroom 部署建議：
    • 三模架構實作：定義 IDLE (紅燈)、REMOTE (藍燈)、AUTO (綠燈)。
    • 轉移觸發器：實作按下按鍵 A 切換至次一狀態，按下按鍵 B 回歸 IDLE。
    • 驗證要點：關鍵工程審核。學員是否使用了 enum？代碼中是否還存在 state == 1 這種數字判斷？（應使用 state == REMOTE）。

任務 2：避障邏輯狀態機 (Obstacle Logic Sub-FSM Lab)

  • 目標：在 AUTO 模式內實作「子狀態機 (Sub-FSM)」，學習如何在非阻塞架構下管理具備時間窗口的有序動作。
  • Classroom 部署建議：
    • 行為原子化：定義子狀態 SEARCH (掃描障礙)、BACKUP (後退 500ms)、TURN (轉向 300ms)。
    • 精密轉場：結合 millis()。例如：當進入 BACKUP 時記錄起點，時間到自動轉移至 TURN。
    • 驗證要點：避障動作是否流暢？是否因使用 delay 而導致狀態切換過程中無法取消任務？（應維持 Loop 的非阻塞運行）。

任務 3：異常救援模式：最高優先權 (Emergency Failsafe Lab)

  • 目標：打造「搶佔式 (Preemptive)」安全防線。當偵測到極端異常時，系統必須能跨越目前所有狀態，強制進入安全鎖定。
  • Classroom 部署建議：
    • 救援轉移實作：新增 EMERGENCY 狀態（特徵：紅燈全快閃、馬達鎖死）。
    • 異常捕捉器：例如監視通訊心跳，若 2 秒無訊號，無論目前處於 AUTO 還是 REMOTE，必須瞬間轉移至 EMERGENCY。
    • 解鎖機制設計：實作嚴謹的解鎖序列。
    • 驗證要點：救援模式是否具備最高執行權限？其轉移響應是否在 20ms 以內？

3. 邏輯嚴密性與確定性架構點評標準 (Logic Architecture & Deterministic Standards)

此單元的價值在於 「對系統狀態的穷舉能力與對行為邊界條件的嚴格定義」：

  • [ ] 狀態窮舉性：是否考慮了所有可能的狀態變化？是否有狀態「死鎖」的情況？
  • [ ] 代碼架構優雅度：是否完全消滅了麵條式的 if-else 嵌套，改由清晰的 switch-case 驅動？
  • [ ] 失效安全性 (Failsafe)：緊急狀態是否能在任何時刻可靠接管控制權？
