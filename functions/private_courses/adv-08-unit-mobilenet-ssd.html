簡介

本單元介紹兩個在邊緣裝置上廣泛使用的深度學習架構：MobileNet 與 SSD（Single Shot Detector）。MobileNet 透過深度可分離卷積大幅減少運算量，而 SSD 則以單次前向傳播完成物件偵測與定位。理解這兩者的設計取捨，能幫助你在嵌入式硬體的運算限制下，做出模型選擇與效能調校的正確決策。

情境
你想讓自走車在行駛中即時辨識路上的障礙物類型（行人、車輛、路錐）。傳統的 YOLO 模型雖然精準，但在低功耗微控制器上每幀需要數秒才能完成推論，完全無法滿足即時需求。換用 MobileNet-SSD 後推論速度提升十倍以上，但精度有所下降。這之間的取捨如何量化？何時該犧牲精度換取速度？本單元將為你解析模型架構背後的設計邏輯。

MobileNet：神經網路的瘦身專家

MobileNet 是 Google 於 2017 年提出的輕量化卷積神經網路架構，專為行動裝置與嵌入式設備設計。其核心創新是深度可分離卷積（Depthwise Separable Convolution），將標準卷積運算拆解為兩個步驟，大幅降低計算量與模型參數數量，同時僅損失極少的辨識精度。

深度可分離卷積原理

傳統卷積同時進行「空間過濾」與「通道融合」，計算成本為 \(D_K \times D_K \times M \times N\)。深度可分離卷積將其拆為兩步：

  1. Depthwise Convolution：對每個輸入通道獨立套用一個 \(D_K \times D_K\) 的空間濾波器。成本：\(D_K \times D_K \times M\)。
  2. Pointwise Convolution：使用 \(1 \times 1\) 卷積核將所有通道的特徵融合為 N 個輸出通道。成本：\(M \times N\)。
運算量對比 以 3×3 卷積核、輸入 64 通道、輸出 128 通道為例：標準卷積需要 3×3×64×128 = 73,728 次乘法；深度可分離卷積僅需 3×3×64 + 64×128 = 8,768 次乘法，縮減至 1/8。

MobileNet 的調節旋鈕

MobileNet 提供兩個超參數讓工程師根據硬體限制調整模型：

參數功能效果典型值
Width Multiplier (α)縮放每層的通道數α=0.5 → 參數量降為 1/40.25, 0.5, 0.75, 1.0
Resolution Multiplier (ρ)縮放輸入影像解析度ρ=0.5 → 計算量降為 1/4224, 192, 160, 128

在 TensorFlow.js 中載入 MobileNet

// 使用 TensorFlow.js 載入預訓練的 MobileNet
import * as mobilenet from '@tensorflow-models/mobilenet';

const model = await mobilenet.load({
version: 2, // MobileNet V2
alpha: 0.5, // 輕量模式，適合嵌入式
});

const predictions = await model.classify(imageElement);
// 回傳: [{className: "cup", probability: 0.87}, ...]
效能提醒 首次推論（Cold Start）通常比後續推論慢 3–5 倍，因為需要編譯 GPU Shader 與分配記憶體。在效能測試中，應先執行 2–3 次「預熱」推論，再開始計時。
實務建議 MobileNet 僅能進行「分類」（這張圖是什麼），無法告訴你物體在畫面中的位置。若需要同時取得位置資訊（Bounding Box），則需要搭配 SSD 偵測框架——這正是下一節的主題。

SSD：一眼看穿全局

SSD（Single Shot MultiBox Detector）是一種「一階段」物件偵測架構。與 R-CNN 系列的「兩階段」方法不同，SSD 在單次前向傳播中同時完成物體定位與分類，無需耗時的區域提議（Region Proposal）步驟。這使得 SSD 的推論速度足以應對即時自動駕駛的需求。

SSD 的核心設計

SSD 的偵測流程可分為三個關鍵階段：

1

Backbone 特徵提取

使用 MobileNet 或 VGG 作為骨幹網路，從輸入影像中提取多尺度特徵圖（Feature Maps）。

2

Anchor Box 預測

在每個特徵圖的每個位置上，預設多組不同長寬比的「錨框（Anchor Boxes）」。模型預測每個錨框的偏移量與類別機率。

3

NMS 後處理

非極大值抑制（Non-Maximum Suppression）過濾掉重疊度過高的冗餘偵測框，只保留信心分數最高的結果。

偵測結果的資料格式

SSD 輸出的每個偵測結果包含以下欄位：

// SSD 偵測結果格式
{
label: "person", // 物體類別名稱
confidence: 0.92, // 信心分數 (0~1)
bbox: [0.12, 0.25, 0.45, 0.88] // [x1, y1, x2, y2] 正規化座標
}

// 轉換為像素座標（以 320×240 為例）
const pixelBox = {
x1: Math.round(0.12 * 320), // 38
y1: Math.round(0.25 * 240), // 60
x2: Math.round(0.45 * 320), // 144
y2: Math.round(0.88 * 240), // 211
};
重要 模型回傳的 Bounding Box 座標通常是正規化值（0~1），必須乘以實際影像寬高才能得到像素座標。忘記這一步是初學者最常見的錯誤——會導致偵測框縮在畫面左上角的一個小區域內。

MobileNet-SSD 的組合優勢

將 MobileNet 作為 SSD 的 Backbone，形成 MobileNet-SSD 組合，兼具輕量化與偵測能力。這是目前嵌入式即時物件偵測的主流方案：

  • 模型大小僅約 6–23 MB（依 α 值而定），可載入 ESP32-S3 的 PSRAM
  • 在瀏覽器中透過 WebGL 加速，可達 20–30 FPS
  • 支援 COCO 資料集的 90 類物體偵測（行人、車輛、動物等）
提示 在自走車應用中，我們通常只關心 1–3 種類別（如 person、car、cone）。可以在後處理階段過濾掉不相關的偵測結果，減少下游控制邏輯的複雜度。

權衡之美：FPS 與 mAP

在嵌入式 AI 系統中，模型的評估不能只看精度——推論速度同樣是決定系統可用性的關鍵指標。一個 mAP 高達 95% 但每幀需要 2 秒的模型，在自動駕駛場景中毫無價值。工程師必須學會在「看得準」與「看得快」之間找到最佳平衡點。

核心效能指標

指標全名定義自駕車需求
FPSFrames Per Second每秒可處理的影像幀數≥ 10 FPS（最低安全閾值）
mAPmean Average Precision所有類別平均精度的均值≥ 50%（可接受）
Latency推論延遲單幀從輸入到輸出的時間≤ 100ms
Model Size模型檔案大小權重檔容量≤ 20 MB（PSRAM 限制）

信心門檻（Confidence Threshold）的調校

信心門檻決定了模型輸出中哪些偵測結果被保留。這是精度與召回率之間最直接的權衡旋鈕：

  • 門檻過高（如 0.9）：只保留「非常確定」的結果。減少誤判（False Positive），但會漏掉真實目標（False Negative）。車體可能在障礙物面前毫無反應。
  • 門檻過低（如 0.3）：保留更多偵測結果。不容易漏失目標，但會產生大量虛假偵測（雜訊），導致控制系統頻繁跳動。
最佳實務 在自走車避障場景中，建議起始門檻設定為 0.6。對於「行人」類別可適度降低至 0.5（寧可誤停也不可撞人），對於「路錐」等靜態物體可提高至 0.7。

硬體加速策略

相同的模型在不同的運算後端上，速度差異可達 10 倍以上：

// 指定 TensorFlow.js 運算後端
await tf.setBackend('webgl'); // GPU 加速 → ~25 FPS
// await tf.setBackend('cpu'); // 純 CPU → ~3 FPS

// 效能量測方法
const t0 = performance.now();
const results = await model.detect(canvas);
const inferenceMs = performance.now() - t0;
const fps = Math.round(1000 / inferenceMs);
安全考量 若 FPS 低於 10，代表系統在車體移動 100ms 內無法更新決策。以車速 1 m/s 計算，這意味著車體在每次決策之間已前進 10 cm——對近距離障礙物而言可能來不及煞車。低於此閾值時，應自動降低車速或切換為保守模式。

從偵測結果到控制指令

偵測到的 Bounding Box 本身就包含豐富的空間資訊，可直接轉換為控制依據：

  • Box 中心 X 座標 → 目標物的水平位置 → 轉向誤差
  • Box 寬度 → 目標物的視覺大小 → 距離估算（寬度越大 = 越近）
  • 類別標籤 → 決策優先級（person > obstacle > cone）
備註 MobileNet-SSD 的偵測結果可能在連續幀之間「閃爍」（Flickering）——某一幀偵測到物體，下一幀又消失。在控制邏輯中應加入計數器或移動平均濾波器，確保至少連續 3 幀偵測到同一物體才觸發反應。

    • 防呆處理：實作平滑器或計數器處理「偵測閃爍 (Flickering)」，防止單一幀的誤判導致車體急剎。
