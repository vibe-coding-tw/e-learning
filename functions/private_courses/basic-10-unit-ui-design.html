簡介

當你的裝置具備多種運行模式（手動、自動、校正…），使用者需要一個直覺的介面來切換與監控狀態。 好的控制介面不只是「按鈕能按」，還要即時反映裝置狀態、防止誤操作、並在不同模式間提供清晰的視覺回饋。 本單元教你設計一套與 FSM 連動的網頁控制面板，讓前端 UI 與硬體狀態始終同步。

情境
你的車子有「手動遙控」和「自動循線」兩種模式，網頁上需要一個切換開關。 但如果切換瞬間車子正在高速行駛，直接跳到自動模式可能造成危險。 良好的 UI 設計會在切換前顯示確認提示、切換中顯示過渡動畫、切換後更新所有相關控件的可用狀態——讓使用者隨時知道車子「現在在做什麼」。

模式切換的 UI 邏輯

當你的機器人具備多種運行模式（手動遙控、自動循線、緊急停止），使用者需要一個直覺且防錯的介面來切換。UI 組件的選擇不是美學問題——它直接決定使用者能否在壓力下（車子正高速行駛）迅速、正確地操作。

模式切換介面的核心設計原則是「互斥性可視化」：系統在任何時刻只能處於一種模式，UI 必須清楚地表達「現在是哪個模式」以及「其他模式是什麼」。這與 FSM 的確定性概念一致——一個狀態，一種行為。

UI 組件選型

不同的操作場景適合不同的 UI 組件。以下是常見的模式切換組件比較：

組件類型 適用場景 優點 缺點
分段控制 (Segmented) 2–4 個互斥模式 所有選項一目了然，高對比高亮 佔用橫向空間較大
切換開關 (Toggle) 單一功能啟用/停用 極省空間，操作直覺 不適合三選一以上的情境
下拉選單 (Dropdown) 5 個以上模式 節省空間，支援多選項 需額外點擊展開，不直覺
大型按鈕組 (Button Group) 觸控螢幕 / 移動裝置 觸控友善，不易誤按 佔用大量畫面空間
提示 對於遙控車的三模式系統（Manual / Auto / E-Stop），推薦使用分段控制組件。它讓所有模式平行陳列，使用者在晃動環境中也能一眼看出當前狀態並快速切換。

CSS 狀態類別實作

使用 CSS 類別來管理模式的視覺狀態，是最乾淨也最易維護的做法。每個模式對應一個狀態類別，透過 JavaScript 動態切換：

/* CSS：模式狀態類別 */
.mode-btn {
    padding: 12px 24px;
    border: 2px solid #e1dfdd;
    background: #fff;
    color: #605e5c;
    transition: all 0.2s ease;
}

.mode-btn.active-manual {
    background: #0078d4;
    border-color: #0078d4;
    color: #fff;
    box-shadow: 0 0 12px rgba(0, 120, 212, 0.4);
}

.mode-btn.active-auto {
    background: #107c10;
    border-color: #107c10;
    color: #fff;
    box-shadow: 0 0 12px rgba(16, 124, 16, 0.4);
}

/* 自動模式下鎖死搖桿區域 */
.joystick-zone.locked {
    pointer-events: none;
    opacity: 0.3;
    filter: grayscale(1);
}

權限隔離：鎖死衝突控件

當系統進入自動駕駛模式時，手動遙控的搖桿區域必須被邏輯鎖定——不只是視覺上變灰，而是真正地以 pointer-events: none 阻止任何觸碰事件。這防止了使用者在自動模式下不小心碰到搖桿而發送衝突指令。

警告 僅使用 opacity: 0.3 做視覺提示是不夠的！必須同時加上 pointer-events: none 才能真正阻止互動。在觸控螢幕上，使用者可能在專注觀察車輛時無意間碰到灰色區域。

佈局與觸控安全

遙控車的操作場景通常不是在安靜的辦公桌前——使用者可能正站在戶外、手持手機、看著遠處的車輛。在這種晃動且注意力分散的環境下，UI 佈局的設計直接影響操作安全性。一個按鈕太小、太靠近搖桿、或沒有足夠的間距，都可能在關鍵時刻導致誤操作。

觸控安全設計的核心思想來自工業控制系統的 HMI（Human-Machine Interface）標準：關鍵操作必須有物理隔離、尺寸足夠大、且具備防誤觸機制。

觸控目標尺寸規範

根據各大平台的 HIG（Human Interface Guidelines），觸控按鈕的最小尺寸如下：

平台標準 最小觸控目標 推薦間距
Apple HIG 44 × 44 pt 8 pt
Material Design 48 × 48 dp 8 dp
WCAG 2.5.5 (AAA) 44 × 44 CSS px —
遙控車建議 56 × 56 dp 以上 16 dp 以上
備註 在晃動環境中操作時，使用者的手指精確度會大幅下降。建議將遙控車控制面板的關鍵按鈕尺寸設為 56dp 以上，並確保相鄰按鈕之間有至少 16dp 的空白區域。

佈局安全原則

以下是控制面板佈局的五大安全原則：

  • 空間隔離：E-Stop 按鈕必須遠離日常操作區域（搖桿、模式切換），避免手指滑動時誤觸。
  • 功能分區：將介面分為「控制區」（搖桿、方向鍵）和「系統區」（模式切換、E-Stop），以不同背景色區分。
  • 上下文鎖定：自動模式下，手動控制區必須完全鎖死（pointer-events: none）。
  • 危險操作確認：E-Stop 之外的關鍵操作（如切換模式），建議在高速行駛時加上確認對話框。
  • 單手可達：最常用的控制元素應放在拇指自然觸及的範圍內。
/* 佈局範例：功能分區 CSS */
.control-panel {
    display: grid;
    grid-template-rows: auto 1fr auto;
    height: 100vh;
    gap: 16px;
    padding: 16px;
}

/* 頂部：系統狀態與模式切換 */
.system-zone {
    background: #f3f2f1;
    border-radius: 8px;
    padding: 12px;
}

/* 中部：搖桿操控區 */
.joystick-zone {
    display: flex;
    align-items: center;
    justify-content: center;
    min-height: 200px;
}

/* 底部：E-Stop 獨立區域 — 遠離搖桿 */
.emergency-zone {
    display: flex;
    justify-content: center;
    padding: 24px 0;
    border-top: 2px solid #d13438;
}
警告 絕對不要將 E-Stop 按鈕放在搖桿旁邊！在緊張的遙控場景中，使用者的手指可能因為緊張而滑動，若 E-Stop 離搖桿太近，可能在不該停的時候意外觸發緊急停止——或者更糟，在該停的時候因為害怕誤觸而猶豫不按。

雙向同步與警報

控制面板的最終目標是成為硬體的「忠實鏡像」——它不能有自己的「意見」，只能反映裝置的真實物理狀態。這需要雙向狀態同步：硬體定期發送心跳報文告知當前狀態，面板持續監聽並自動校正顯示。當出現嚴重異常時，面板還必須透過多感官（視覺 + 觸覺）警報來確保使用者注意到。

心跳監聽與自動校正

硬體每隔固定時間（建議 200ms）主動推送當前狀態的 NOTIFY 封包。前端收到後，比對面板顯示是否一致。若不一致（例如硬體因故障跳回 IDLE，但面板還顯示 AUTO），UI 必須立即自動修正，不需使用者介入。

// 心跳監聽與自動校正邏輯
function onHeartbeat(packet) {
    const hardwareMode = packet.current_mode;
    const displayedMode = getDisplayedMode();

    if (hardwareMode !== displayedMode) {
        console.warn(`狀態不一致！面板:${displayedMode} 硬體:${hardwareMode}`);
        forceUpdateUI(hardwareMode); // 強制校正
    }

    // 重設斷線計時器
    resetDisconnectTimer();
}

// 超過 2 秒無心跳 → 顯示斷線警報
function onDisconnectTimeout() {
    showConnectionLostOverlay();
    lockAllControls();
}

E-Stop 緊急停機介面

E-Stop（Emergency Stop）是控制面板中優先權最高的元素。當觸發 E-Stop 時，系統必須透過多維度感官警報確保使用者意識到緊急狀態：

  • 視覺警報：滿版紅色半透明遮罩覆蓋整個面板，封鎖所有正常控制路徑。
  • 觸覺回饋：呼叫 navigator.vibrate([200, 100, 200]) 產生緊急脈衝震動。
  • 文字提示：遮罩上顯示大字「緊急停止已觸發」與解鎖指示。
// E-Stop 觸發邏輯
function triggerEStop() {
    // 1. 發送緊急停止指令（最高優先級）
    sendCriticalCommand({ cmd: 'EMERGENCY_STOP' });

    // 2. 視覺警報：紅色全域遮罩
    document.getElementById('estop-overlay').style.display = 'flex';

    // 3. 觸覺回饋：震動警報脈衝
    if (navigator.vibrate) {
        navigator.vibrate([200, 100, 200, 100, 200]);
    }

    // 4. 鎖定所有控制元素
    lockAllControls();
}
重要 E-Stop 的解除必須設計防誤觸機制。推薦實作「長按 2 秒」解鎖：使用者必須持續按住解鎖按鈕超過 2000ms 才能撤銷緊急停止。嚴禁單擊解除——防止使用者在慌亂中反覆點擊而意外恢復。

長按解鎖實作

let holdTimer = null;
const UNLOCK_DURATION = 2000; // 2 秒

unlockBtn.addEventListener('pointerdown', () => {
    holdTimer = setTimeout(() => {
        releaseEStop();
        navigator.vibrate && navigator.vibrate(50); // 確認回饋
    }, UNLOCK_DURATION);
    startProgressRing(); // 顯示環形倒數
});

unlockBtn.addEventListener('pointerup', () => {
    clearTimeout(holdTimer);
    resetProgressRing(); // 中途放手 → 取消
});
提示 長按過程中建議顯示一個環形進度條（Progress Ring），讓使用者清楚知道「還需要按多久」。這比純粹等待 2 秒不給任何回饋的體驗好得多——使用者能確信操作正在進行中。

basic-10-unit-ui-design (網頁端模式控制介面設計：打造戰術面板)

歡迎來到本單元的實作練習！在這裡，你將學習如何將後端邏輯具象化為專業的工業控制面板。這不僅僅是畫出漂亮的介面，更重要的是學習 使用者體驗 (UX) 與 交互安全性 的設計。

🎯 學習目標

  1. 視覺層級與權限隔離：透過 CSS 狀態切換，清晰區分系統模式，並防止人為誤觸。
  2. 物理現實同步 (State Sync)：打破「樂觀 UI」假象，確保面板永遠反映機器人的真實物理狀態。
  3. 多維度警報系統：利用視覺（遮罩）與觸覺（Web Vibrate API）傳達緊急狀態。

📂 程式庫結構

  • src/UI_Controller.js：處理按鈕點擊、CSS 切換與震動邏輯。
  • src/Sync_Observer.js：監聽硬體狀態並強制修正 DOM。
  • docs/UX_Rationale.md：關於安全佈局與視覺導引的設計決策解說。
  • index.html：戰術面板的結構定義。

🚀 任務說明

任務 1：戰術面板狀態切換 (Tactical Panel & Authority Lab)

  • 實作重點：設計具備高對比效果的模式選單（Manual / Pilot / E-Stop）。
  • 權限隔離：當進入 Pilot (自動駕駛) 模式時，必須將手動遙控區（如搖桿、微調鍵）鎖定（pointer-events: none），防止誤操作。

任務 2：雙向狀態同步 (Bi-directional State Sync Lab)

  • 實作重點：每 200ms 解析一次「虛擬硬體」回傳的狀態。
  • 強制一致性：若硬體狀態與網頁顯示不符（例如硬體因故跳回手動），網頁 UI 必須立即自動彈回正確位置。

任務 3：「E-Stop」緊急停機與警報 (Haptic & Visual Alert Lab)

  • 視覺警報：按下 E-Stop 後，顯示滿版紅色半透明遮罩。
  • 觸覺回饋：調用 navigator.vibrate([200, 100, 200]) 產生警報脈衝。
  • 安全解鎖：實作「長按 2 秒」才能撤銷 E-Stop 狀態的機制，嚴禁單擊解除。

1. 範本程式庫 (Template Repo) 配置建議

UI 設計作業的核心在於「交互安全性」與「狀態可視化」，範本應包含前端框架與通訊介面，建議包含：

  • 📂 src/PanelControls.js：交互邏輯實作區。學員在此實作按鈕點擊後的 CSS 類別切換，並調用 Web Vibrate API 實現物理報警。
  • 📂 src/SyncEngine.js：DOM 同步引擎。學員在此實作監聽來自硬體的心跳報文，並強制將網頁 UI 修正為與實體一致的狀態。
  • 📂 README.md：UX 設計論證與測評。提供表格讓學員記錄：為什麼選擇目前的配色方案？在自動駕駛模式下，如何確保搖桿不會被誤觸？
  • 📂 docs/UI_STATES.md：介面狀態對照表。要求學員定義不同模式（Manual、Pilot、E-Stop）對應的視覺風格（光效、顏色、遮罩）。

2. 作業任務部署細節

任務 1：戰術面板：CSS 狀態切換實作 (Tactical Panel & Authority Lab)

  • 目標：透過視覺層級區分系統狀態，並實施「功能權限隔離」以提升物理安全性。
  • Classroom 部署建議：
    • 三模分段選單：設計具備高對比高亮效果的模式切換鈕。
    • 控制權隔離：核心關鍵點。當系統切換至 Pilot (自動駕駛) 時，手動遙控區（搖桿、微調鍵）必須變為灰色背景且 pointer-events: none 鎖死交互。
    • 驗證要點：視覺上是否一眼就能看出目前的模式？自動駕駛模式下，是否真的無法透過網頁遙控干擾機器人？

任務 2：雙向狀態同步與校正 (Bi-directional State Sync Lab)

  • 目標：打破「樂觀 UI」的假象。確保面板反映的是機器人的「物理現實」，而非使用者的「點擊紀錄」。
  • Classroom 部署建議：
    • 真值追蹤器：實作每 200ms 解析一次硬體回傳的 current_mode。
    • 強制一致性：若用戶點了「自動」，但硬體因電力不足跳回「手動」，網頁按鈕必須在收到心跳後立即自動彈回「手動」位置。
    • 驗證要點：重啟機器人後，網頁面板是否能自動恢復到與機器人一致的顯示狀態？「眼見是否即真實」？

任務 3：「E-Stop」緊急停機與警報 (Haptic & Visual Alert Lab)

  • 目標：打造最高優先權的安全屏障。透過多維度（視覺、觸覺）的反饋告知使用者緊急狀態，並設計預防性的解鎖流程。
  • Classroom 部署建議：
    • 全域緊急遮罩：按下 E-Stop 後，畫面上方出現滿版紅色半透明遮罩，封鎖所有正常的控制路徑。
    • 震動報警實作：調用 navigator.vibrate([200, 100, 200]) 產生緊急物理脈衝。
    • 安全長按解鎖：技術細節。嚴禁單擊解除。必須實作「長按 2 秒」才能撤銷 E-Stop 狀態。
    • 驗證要點：遮罩是否優先於所有 UI 元件？震動回饋是否顯著提升了警報感知？長按機制是否能有效防止誤觸？
