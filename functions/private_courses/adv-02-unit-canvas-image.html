Canvas 畫布：數位繪圖的物理基礎

HTML5 <canvas> 元素是瀏覽器提供的像素級繪圖介面。與 <img> 標籤被動顯示圖片不同，Canvas 讓你能在 JavaScript 中主動控制每一個像素的繪製——畫線、填色、貼圖、甚至逐像素修改色彩值。對物聯網影像應用而言，Canvas 是你在前端實現即時視覺處理（如疊加準心線、標記追蹤框、計算像素統計值）的核心工具。

Canvas 座標系統

Canvas 的座標系統與傳統數學的笛卡兒座標系有一個關鍵差異：原點 (0, 0) 位在畫布的左上角，x 軸向右為正，y 軸向下為正。這套座標系統與螢幕像素的排列方式一致——螢幕從左上角開始逐行掃描，因此 y 值越大表示越靠近畫面底部。

座標系統 原點位置 x 軸方向 y 軸方向 使用場景
Web Canvas / 螢幕座標 左上角 → 向右 ↓ 向下 瀏覽器、影像處理、遊戲開發
笛卡兒座標（數學） 左下角（或中心） → 向右 ↑ 向上 數學繪圖、物理模擬
警告 座標系統混淆是初學者最常犯的錯誤。如果你的準心線畫在「上下顛倒」的位置，幾乎可以確定是 y 軸方向搞反了。在 Canvas 中，y = 0 是畫面頂端，y = height 是畫面底端。

取得 2D 繪圖上下文

Canvas 元素本身只是一塊「空白畫布」，所有繪圖操作都必須透過繪圖上下文（Context）來執行。你可以把 Context 想像成你的畫筆——沒有畫筆，畫布再大也畫不出任何東西。

// 步驟 1：取得 Canvas 元素 const canvas = document.getElementById('myCanvas'); // 步驟 2：取得 2D 繪圖上下文（你的「畫筆」） const ctx = canvas.getContext('2d'); // 步驟 3：使用畫筆繪製圖形 ctx.fillStyle = '#0078d4'; ctx.fillRect(10, 10, 100, 50); // x, y, width, height

將影像繪製到 Canvas

drawImage() 是 Canvas 最常用的 API 之一。它可以將 <img>、<video>、甚至另一個 <canvas> 的內容繪製到當前畫布上。在物聯網影像應用中，你通常會把 MJPEG 串流的每一幀「畫」到 Canvas 上，然後再進行後續的像素處理。

// 將 <img> 元素的內容繪製到 Canvas const img = document.getElementById('stream-img'); ctx.drawImage(img, 0, 0, canvas.width, canvas.height); // 將 Blob 轉換為影像再繪製 const blob = await fetch('/stream').then(r => r.blob()); const bitmap = await createImageBitmap(blob); ctx.drawImage(bitmap, 0, 0);
提示 使用 createImageBitmap() 搭配 Blob 是效能最佳的做法——它在背景執行緒解碼影像，不會阻塞主執行緒的 UI 渲染。相比之下，透過 new Image() 搭配 img.src = URL.createObjectURL(blob) 的做法會在主執行緒解碼，可能造成畫面卡頓。

使用 requestAnimationFrame 實現連續繪製

當你需要以 30 或 60 FPS 的速率持續更新 Canvas（例如即時顯示串流影像並疊加 UI 標註），不應該使用 setInterval，而應該使用 requestAnimationFrame。後者會自動與瀏覽器的重繪週期同步，避免不必要的 CPU 消耗，並在頁面不可見時自動暫停。

function renderLoop() { ctx.drawImage(videoElement, 0, 0); // 繪製最新幀 ctx.strokeStyle = '#00ff00'; ctx.strokeRect(100, 80, 120, 80); // 疊加追蹤框 requestAnimationFrame(renderLoop); // 排程下一幀 } renderLoop();
備註 Canvas 的繪圖結果是即時光柵化的——一旦像素被畫上去，你無法「選取並移動」它。如果要移動一個矩形框，你必須先清除整個畫布（或該區域），重新繪製影像底圖，再把框畫在新位置。這就是為什麼連續動畫需要在每一幀「清除→重繪」的循環。

ImageData：揭開像素的秘密

當你在 Canvas 上繪製了一張影像後，這些像素資料就儲存在 Canvas 的內部緩衝區中。透過 getImageData() API，你可以將這些像素「取出」到 JavaScript 的記憶體中，直接讀取或修改每一個像素的色彩值。這是實作灰階濾鏡、色彩偵測、特徵提取等功能的基礎。

Uint8ClampedArray：像素的一維排列

getImageData() 回傳的 ImageData 物件包含一個 Uint8ClampedArray 型別的 data 屬性。這是一個超長的一維陣列，將二維影像的所有像素「攤平」成一條直線。每 4 個連續的元素代表一個像素的 R（紅）、G（綠）、B（藍）、A（透明度）四個分量，每個值的範圍是 0–255。

// 取得 Canvas 上的像素資料 const imageData = ctx.getImageData(0, 0, canvas.width, canvas.height); const data = imageData.data; // Uint8ClampedArray // 陣列結構：[R0, G0, B0, A0, R1, G1, B1, A1, R2, ...] // 第 0 個像素: data[0]=R, data[1]=G, data[2]=B, data[3]=A // 第 1 個像素: data[4]=R, data[5]=G, data[6]=B, data[7]=A // 陣列總長度 = 寬 × 高 × 4 console.log(data.length); // 320 × 240 × 4 = 307,200
分量 索引偏移 範圍 說明
R（紅色） i + 0 0 – 255 0 = 無紅色，255 = 最大紅色
G（綠色） i + 1 0 – 255 0 = 無綠色，255 = 最大綠色
B（藍色） i + 2 0 – 255 0 = 無藍色，255 = 最大藍色
A（透明度） i + 3 0 – 255 0 = 完全透明，255 = 完全不透明

索引公式：從 (x, y) 定位像素

影像是二維的（有 x 和 y 座標），但 data 陣列是一維的。要找到座標 (x, y) 的像素在陣列中的起始位置，你需要使用以下公式：

// 核心公式：(x, y) 座標 → 一維陣列索引 const index = (y * width + x) * 4; // 讀取該像素的 RGBA 值 const r = data[index + 0]; const g = data[index + 1]; const b = data[index + 2]; const a = data[index + 3]; // 範例：取得座標 (160, 120) 的像素（畫面中央） // index = (120 × 320 + 160) × 4 = 154,880
提示 公式中 × 4 的原因是每個像素佔用 4 個陣列位置（R、G、B、A）。忘記乘以 4 是最常見的 Bug——你會讀到隔壁像素的色彩分量，導致「顏色偏移」的詭異現象。

修改像素並寫回 Canvas

修改 data 陣列中的值後，使用 putImageData() 將修改後的像素寫回 Canvas。這是實作所有像素級濾鏡的核心流程：取出 → 修改 → 寫回。

// 完整流程：取出 → 修改 → 寫回 const imageData = ctx.getImageData(0, 0, w, h); const data = imageData.data; // 將座標 (10, 20) 的像素改成紅色 const idx = (20 * w + 10) * 4; data[idx] = 255; // R data[idx + 1] = 0; // G data[idx + 2] = 0; // B data[idx + 3] = 255; // A（不透明） ctx.putImageData(imageData, 0, 0); // 寫回畫布
警告 getImageData() 與 putImageData() 會進行大量的記憶體複製操作。在 320×240 的解析度下，每次呼叫都要搬移 307,200 個數字。如果你每幀都做全畫面的 getImageData，在高 FPS 下可能成為效能瓶頸。後續單元會介紹如何用「局部取出（ROI 提取）」來降低開銷。

濾波與變換的數學邏輯

現在你已經知道如何存取每個像素的 RGBA 值，接下來要學習如何有目的地修改這些數值來實現各種視覺效果。無論是灰階化、水平翻轉還是色彩偵測，每一種影像處理操作的背後都是一條明確的數學公式。

灰階化 (Grayscale Conversion)

將彩色影像轉換為灰階，是最基礎的像素操作。原理是將每個像素的 R、G、B 三個色彩分量取平均值，然後將平均值同時寫回 R、G、B（保持 Alpha 不變）。當三個分量相同時，人眼感知到的就是灰色。

// 灰階化：(R+G+B)/3 的簡單平均法 const imageData = ctx.getImageData(0, 0, w, h); const data = imageData.data; for (let i = 0; i < data.length; i += 4) { const avg = (data[i] + data[i+1] + data[i+2]) / 3; data[i] = avg; // R data[i + 1] = avg; // G data[i + 2] = avg; // B // data[i + 3] 保持不變（Alpha） } ctx.putImageData(imageData, 0, 0);
備註 簡單平均法 (R+G+B)/3 在學習階段足夠使用。但在專業影像處理中，常使用亮度加權公式：0.299×R + 0.587×G + 0.114×B，因為人眼對綠色最敏感、紅色次之、藍色最不敏感。加權後的灰階影像在視覺上更接近人眼的真實感受。

水平翻轉 (Mirror / Flip)

ESP32-CAM 的前置鏡頭（自拍模式）拍出的畫面是左右相反的——你向左移動，畫面中的你卻向右移動。這在遙控場景中會造成致命的操控錯誤。水平翻轉可以修正這個問題，原理是將每一行的像素左右對調。

// 水平翻轉：逐行交換對稱位置的像素 for (let y = 0; y < h; y++) { for (let x = 0; x < w / 2; x++) { const left = (y * w + x) * 4; const right = (y * w + (w - 1 - x)) * 4; // 交換 R, G, B, A 四個分量 for (let c = 0; c < 4; c++) { const tmp = data[left + c]; data[left + c] = data[right + c]; data[right + c] = tmp; } } }

圖層分離：為什麼需要兩個 Canvas？

在工業級影像處理系統中，「原始影像」和「UI 標註」必須嚴格分離到不同的 Canvas 圖層。如果你把準心線直接畫在原始影像的 Canvas 上，當你下次呼叫 getImageData() 時，取出的資料會包含準心線的像素——這會污染後續的色彩偵測或 AI 分析。

圖層 Canvas ID 用途 操作
底層：影像層 #video-canvas 存放原始影像資料 drawImage、getImageData（濾鏡處理）
頂層：標註層 #overlay-canvas 繪製 UI 元素（準心、框線、文字） strokeRect、moveTo / lineTo、fillText
// 雙層 Canvas 的 HTML 結構 <div style="position:relative"> <canvas id="video-canvas"></canvas> <canvas id="overlay-canvas" style="position:absolute; top:0; left:0"> </canvas> </div>
警告 圖層分離不只是「好習慣」——它是架構正確性的保證。在你的遙控車專案中，如果 AI 色彩偵測模組把「準心的綠色線條」誤判為追蹤目標，整個系統的行為都會出錯。分離圖層讓影像處理管線只看到純淨的感測器資料。

為什麼不用 CSS 濾鏡？

CSS 提供了 filter: grayscale(100%) 等便捷的濾鏡效果，但它們只是視覺上的變化——你無法從 CSS 濾鏡中取得處理後的像素數據。當你需要「計算畫面中紅色區塊的質心座標」或「判斷某個像素是否符合特定色彩閾值」時，只有透過 ImageData 的像素級操作才能取得這些數值資訊。

提示 在效能敏感的場景中，可以結合兩種方式：用 CSS 濾鏡處理純視覺效果（如亮度調整），用 ImageData 處理需要計算結果的操作（如色彩偵測）。這樣可以減少 getImageData 的呼叫次數，提升整體 FPS。

Assignment: HTML5 Canvas 影像處理與即時標註系統

1. 任務目標

本單元旨在引導學員掌握 Web 前端底層影像處理的核心技術。你將學習如何直接操控 HTML5 Canvas 的像素數據，並實現即時的濾鏡效果與視覺標註系統。

2. 核心技術點

  • 像素操作：使用 getImageData() 與 putImageData() 進行高效像素處理。
  • 濾鏡算法：實作灰階 (Grayscale)、鏡像 (Mirror)、反轉 (Invert) 等經典演算法。
  • 圖層解耦：學習將「原始影像層」與「UI/標註層」分離，避免數據污染。
  • ROI 提取：實作區域感興趣區域 (Region of Interest) 的截取與數據轉換。

3. 專案結構

  • index.html: 主頁面結構，包含視訊與雙層畫布。
  • src/filters.js: 存放你的像素處理演算法（需實作部分）。
  • src/interaction.js: 負責 UI 標註與滑鼠互動邏輯。
  • tests/: 自動化測試腳本，用於驗證你的演算法是否正確。

4. 開始作業

  1. 安裝依賴： 
    npm install
  2. 啟動開發伺服器： 
    npm run serve
  3. 實作任務：
    • 打開 src/filters.js 並根據註釋實作 applyGrayscale 與 applyMirror 函式。
    • 在 src/interaction.js 中實作座標映射邏輯，確保準星能準確鎖定像素。
  4. 執行測試： 
    npm test

5. 提交要求

  • 確保所有測試項目通過。
  • 將你的代碼推送到 GitHub 程式庫。
  • 在 PR 中附上你實作的濾鏡效果截圖。

  • 📂 tests/vision_test.js：使用單元測試檢核學員的濾鏡公式。例如：輸入紅色 (255, 0, 0)，檢查灰階化後是否正確輸出約 (85, 85, 85)。
  • 📂 .github/workflows/ui_snapper.yml：設置一個簡單的工作流，自動截取學員提交後的網頁畫面並貼在 PR 評論中，方便導師一眼看出濾鏡與準星是否正常。

2. 作業任務部署細節

任務 1：即時濾鏡開發師 (Filter Development Lab)

  • 目標：掌握 Uint8ClampedArray 的一維索引規律。
  • Classroom 部署建議：
    • 核心代碼檢核： 
      // 學生需實作：(R+G+B)/3 的灰階變換
for (let i = 0; i < data.length; i += 4) {
    let avg = (data[i] + data[i+1] + data[i+2]) / 3;
    data[i] = data[i+1] = data[i+2] = avg; // 固定 Alpha 不動
}
    • Autograding：透過 headless browser 測試，讀取 Canvas 的數據值，驗證像素變換是否精確符合數學公式。

任務 2：視覺準星與動態標註 (Visual Overlay System)

  • 目標：練習座標空間映射 (Coordinate Mapping) 與圖層解耦思維。
  • Classroom 部署建議：
    • 核核重點：導師應檢查學員是否將 UI 繪製在「正確的畫布層級」。如果是直接繪製在處理過後的影像層，會導致後續 AI 偵測時「誤把準星當特徵」，這是常見的工程失誤。
    • 互動性驗證：要求學員實作即時顯示移動座標旁的像素 $R, G, B$ 原生數據，這能驗證學員是否有成功「反推」回一維陣列的正確索引。

任務 3：鏡像修正與 ROI 提取 (Mirroring & ROI)

  • 目標：練習空間幾何變換與局部數據封裝。
  • Classroom 部署建議：
    • 空間變換檢核：實作水平鏡像。這需要雙重 $for$ 迴圈或 $O(N)$ 的陣列搬移，這能測試學員對「記憶體配置」與「座標對稱性」的理解。
    • ROI 提交：要求學員將提取出的 $100 \times 100$ 局部特徵區塊（ROI），使用 canvas.toDataURL() 轉為 Base64 格式並印在 Console。這能模擬未來將「特徵小圖」上傳至雲端 AI 的真實場景。

3. 前端影像導師點評標準 (Frontend Vision Benchmarks)

此單元的價值在於 「對影像主權的操控」：

  • [ ] 效能韌性：在開啟濾鏡的狀態下，FPS 是否仍能維持在「準時重繪 (30-60 FPS)」？
  • [ ] 索引精確度：濾鏡處理是否發生「顏色偏移」或「邊界溢漏」（例如：不小心修改了最後一列像素）？
  • [ ] 架構優雅度：是否理解「分離圖層 (Layer Separation)」的重要性？
