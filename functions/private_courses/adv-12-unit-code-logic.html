🔄 1. 有限狀態機 (FSM)：決策的骨架

一台循跡機器人的行為遠比「沿著線走」複雜。它需要在不同情境下採取完全不同的策略：正常循跡時全速追蹤、脫線時旋轉搜尋、收到停止指令時安全減速。有限狀態機 (Finite State Machine, FSM) 正是管理這些複雜行為模式的最佳架構。

FSM 的核心思想是：系統在任何時刻只能處於一個明確定義的狀態 (State)，並且只有在特定事件 (Event) 發生時，才會切換到另一個狀態。這種「分而治之」的設計方式，讓每個狀態內的邏輯簡單且可預測。

FSM 的三大元素

元素 定義 循跡車範例
狀態 (States) 系統當下的「身份」，決定執行哪套邏輯 IDLE、TRACKING、SEARCHING、EMERGENCY_STOP
轉換 (Transitions) 觸發狀態切換的條件或事件 「感測器全白」→ 進入 SEARCHING
行為 (Actions) 在特定狀態下執行的控制邏輯 TRACKING 狀態下執行 PID 循跡演算法

程式碼實作：switch-case 骨架

// 定義狀態列舉
enum RobotState {
  STATE_IDLE,         // 靜止等待指令
  STATE_TRACKING,     // 正常循跡中
  STATE_SEARCHING,    // 脫線搜尋中
  STATE_EMERGENCY_STOP // 緊急停止
};

RobotState currentState = STATE_IDLE;

void updateFSM() {
  switch (currentState) {
    case STATE_IDLE:
      motorStop();
      if (startCommandReceived())
        currentState = STATE_TRACKING;
      break;

    case STATE_TRACKING:
      runPID();            // 執行 PID 控制
      if (allSensorsWhite())
        currentState = STATE_SEARCHING;
      break;

    case STATE_SEARCHING:
      spinToFindLine();    // 旋轉搜尋
      if (lineDetected())
        currentState = STATE_TRACKING;
      if (searchTimeout())
        currentState = STATE_EMERGENCY_STOP;
      break;
  }
}

狀態轉換圖

清晰的狀態轉換圖是 FSM 設計的第一步。在寫任何程式碼之前，你應該先畫出：

// 狀態轉換圖 (State Transition Diagram)
 
    [IDLE]
      │ 收到啟動指令
      ▼
  [TRACKING] ◀──── 偵測到線
      │                ▲
      │ 感測器全白     │
      ▼                │
  [SEARCHING] ────────┘
      │ 超時未找到
      ▼
  [EMERGENCY_STOP]
設計原則 好的 FSM 設計應遵循三個原則：(1) 每個狀態的行為單一且明確；(2) 轉換條件互斥且完備（不會有兩個條件同時成立）；(3) 每個狀態都必須有離開的路徑（避免死鎖）。
進階提示 當狀態從 SEARCHING 切回 TRACKING 時，記得清除 PID 的積分項。否則搜尋期間累積的積分值會導致重新循跡時產生猛烈過衝。這種「狀態切換時的初始化」是 FSM 設計中最容易被忽略的細節。

♾️ 2. 非阻塞迴圈：維持大腦的清醒

在嵌入式控制系統中，有一條鐵律：永遠不要使用 delay()。當你呼叫 delay(500) 時，CPU 完全停止工作 500 毫秒——在這半秒內，感測器的讀數被忽略、PID 無法更新、通訊中斷也無法偵測。對於一台高速循跡車來說，0.5 秒的盲目足以讓它衝出賽道。

delay() 的致命問題

PID 控制器的精確度直接取決於控制迴圈的更新頻率。典型的循跡車需要 100Hz (每 10ms 更新一次) 的控制頻率。任何超過 10ms 的阻塞都會破壞控制品質：

阻塞時間 對 PID 的影響 物理後果
0 ms (理想) dt 穩定，微分與積分計算精確 平穩追蹤軌跡
50 ms dt 抖動，D 項產生假脈衝 馬達偶爾抽搐
500 ms 半秒無更新，積分大幅偏離 車輛直接衝出彎道
關鍵觀念 delay() 不只是「等一下」那麼簡單——它讓整個系統陷入失聰、失明、失控的狀態。專業的控制專案中，程式碼審查的第一件事就是搜尋 delay 關鍵字：如果找到任何一個，就是不合格。

解法：millis() 時間戳計時器

取代 delay() 的方法是使用 millis() 建立非阻塞計時器。原理：記錄上次執行的時間戳，每次迴圈檢查「時間到了嗎？」——如果到了就執行，沒到就跳過。CPU 永遠不會被卡住。

// 非阻塞多工架構
unsigned long lastPID    = 0;
unsigned long lastBlink  = 0;
unsigned long lastReport = 0;

void loop() {
  unsigned long now = millis();

  // 任務 1：PID 控制 (每 10ms = 100Hz)
  if (now - lastPID >= 10) {
    lastPID = now;
    readSensors();
    computePID();
    driveMotors();
  }

  // 任務 2：狀態燈閃爍 (每 500ms)
  if (now - lastBlink >= 500) {
    lastBlink = now;
    toggleLED();
  }

  // 任務 3：藍牙遙測 (每 200ms)
  if (now - lastReport >= 200) {
    lastReport = now;
    sendTelemetry();
  }
}

確保 dt 的穩定性

PID 的積分項和微分項都依賴精確的 \(\Delta t\)。如果控制迴圈的間隔不穩定，計算結果就會出錯。使用 millis() 計時器可以保證 dt 近乎恆定：

// 精確計算 dt
unsigned long now = millis();
float dt = (now - lastPID) / 1000.0;  // 轉為秒
lastPID = now;

// 安全檢查：防止 dt 為零（連續兩次呼叫間隔太短）
if (dt <= 0) return;
效能驗證法 要確認你的非阻塞架構是否達標，可以在 Serial Plotter 中繪製每次 loop() 的實際間隔。理想情況下應該是一條平直的水平線（如 10ms）。如果出現尖峰抖動，代表某處存在隱藏的阻塞操作（常見原因：Serial.print 過多、WiFi 阻塞、I2C 超時未設定）。

🛡️ 3. 防錯邏輯與系統魯棒性

一個好的控制程式，20% 處理正常路徑，80% 處理出錯後的安全退路。在真實物理世界中，感測器會故障、通訊會中斷、馬達會卡死、電壓會驟降。如果程式碼沒有為這些「意外」做好準備，輕則車輛失控，重則硬體燒毀。

這一節介紹三種最關鍵的防護機制：數值鉗位 (Clamping)、心跳包監測 (Heartbeat)、堵轉保護 (Stall Guard)。它們共同構成嵌入式控制系統的「安全防護網」。

防護機制一覽

防護機制 防禦對象 觸發條件 保護動作
數值鉗位 PID 計算溢位 output 超出 [0, 255] 範圍 強制截斷至安全範圍
心跳包監測 通訊中斷 超過 2 秒未收到指令 立即進入 EMERGENCY_STOP
堵轉偵測 馬達過載/卡死 PWM > 100 且編碼器無變化達 3 秒 切斷馬達電力

實作一：數值鉗位 (Output Clamping)

PID 的計算結果可能是任何數值——正的、負的、極大的。但 PWM 硬體只接受 0~255。如果不做限制，負值可能導致整數下溢變成 255（全速倒車），極大值可能導致未定義行為。

// 安全鉗位：確保輸出永遠在合法範圍內
int output = Kp*error + Ki*integral + Kd*derivative;

// 方法一：使用 constrain()
output = constrain(output, 0, 255);

// 方法二：手動實作（更清楚）
if (output > 255) output = 255;
if (output < 0)   output = 0;

analogWrite(MOTOR_PIN, output);
整數溢位陷阱 在 Arduino 中，如果 PID 的計算結果是 int 型態且值為 -10，直接傳入 analogWrite() 時會被解讀為 unsigned byte = 246（近乎全速）。這是嵌入式開發中最常見也最危險的 Bug 之一。永遠在傳入驅動函數之前做鉗位。

實作二：心跳包監測 (Heartbeat Monitor)

如果你的機器人透過藍牙或 WiFi 接收控制指令，當通訊中斷時，馬達會以最後收到的指令持續運轉——可能是全速前進。心跳機制的解法：要求外部每隔固定時間發送一個「我還在」的信號。

unsigned long lastHeartbeat = 0;
const unsigned long HEARTBEAT_TIMEOUT = 2000; // 2 秒

void loop() {
  // 收到任何有效指令時，重置心跳計時
  if (commandReceived()) {
    lastHeartbeat = millis();
    processCommand();
  }

  // 檢查心跳是否超時
  if (millis() - lastHeartbeat > HEARTBEAT_TIMEOUT) {
    currentState = STATE_EMERGENCY_STOP;
    motorStop();           // 立即停機！
  }
}

實作三：堵轉保護 (Stall Guard)

當輪子被障礙物卡住時，馬達電流會急劇上升（因為轉子無法旋轉，等效為短路）。如果不及時斷電，馬達線圈會過熱燒毀。堵轉偵測的邏輯：高 PWM 輸出 + 編碼器無變化 = 堵轉。

unsigned long stallStartTime = 0;
bool stallDetected = false;

void checkStall(int pwm, long encoderTicks) {
  static long lastTicks = 0;

  if (pwm > 100 && encoderTicks == lastTicks) {
    if (stallStartTime == 0)
      stallStartTime = millis();
    if (millis() - stallStartTime > 3000) {
      motorStop();        // 緊急斷電！
      stallDetected = true;
    }
  } else {
    stallStartTime = 0;  // 正常運轉，重置
  }
  lastTicks = encoderTicks;
}
工程哲學 專業的嵌入式系統開發遵循「防禦性程式設計 (Defensive Programming)」原則：假設每一個外部輸入都可能是錯誤的、每一個硬體元件都可能在任何時刻故障。你的程式碼必須在最壞情況下依然能安全停機，而不是撞毀或燒毀。

adv-12-unit-code-logic: 程式路徑邏輯與系統魯棒性

歡迎來到 進階 12 單元！在本單元中，我們將學習如何將簡單的機器人行為轉化為專業級的控制系統。這不僅僅是關於「讓車子動起來」，更是關於如何構建一個 可靠、強韌且具備自我保護能力 的系統。

🎯 學習目標

  1. 有限狀態機 (FSM)：學習如何使用狀態機管理複雜的行為（如：靜止、循跡、搜尋、緊急停止）。
  2. 非阻塞 (Non-blocking) 程式設計：徹底擺脫 delay()，使用 millis() 實現精確的多工處理。
  3. 系統魯棒性 (Robustness)：實作安全保護機制，確保系統在異常情況下（如通訊中斷、馬達堵轉）能安全停止。

📂 程式庫結構

  • include/fsm_states.h: 定義核心狀態與系統資料結構。
  • src/ControlBrain.cpp: 你的主要開發區域，在此實作狀態轉換與控制邏輯。
  • include/system_safety.h: 配置安全閾值與常數。
  • docs/Safety_Logic.md: 安全設計說明文件。

🚀 任務清單

任務 1：狀態機框架實作

在 src/ControlBrain.cpp 中完成 updateFSM() 函數。你需要根據當前感測器狀態與時間戳，決定何時在 TRACKING、SEARCHING 與 EMERGENCY_STOP 之間切換。

任務 2：非阻塞時序優化

檢查程式碼中是否有任何 delay() 調用。將其全部替換為基於 millis() 的邏輯，以確保控制迴圈能維持在 100Hz (10ms) 的穩定頻率。

任務 3：實作失效保護 (Fail-Safe)

實作以下安全機制：

  • 心跳包監測：若超過 2 秒未收到指令，系統必須進入 EMERGENCY_STOP。
  • 堵轉保護：若 PWM 輸出很高但編碼器數值無變化，必須立即斷電。

檢核清單

  • [ ] 程式碼中完全沒有 delay()。
  • [ ] 狀態機能正確處理「全白脫線」並進入搜尋模式。
  • [ ] 斷開模擬通訊時，車輛能在 2 秒內停止。
  • [ ] 通過 npm test 自動化測試。

這是一個「系統架構師」的起點。記住：好的程式碼，20% 處理正常路徑，80% 處理出錯後的安全退路。

針對 adv-12-unit-code-logic (程式路徑邏輯) 單元，這是控制系統從「簡單腳本」轉向「專業工程」的關鍵飛躍。它要求學員具備 系統架構思維，能透過 有限狀態機 (FSM) 管理複雜行為，並利用 非阻塞 (Non-blocking) 技術實現即時多工處理。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template Repo) 配置建議

高效能邏輯需要清晰的框架支撐，建議範本包含：

  • 📂 src/FSM_Core.cpp：狀態機骨架。預置 switch-case 結構，讓學員填入 IDLE（靜止）、TRACKING（循跡）與 SEARCHING（失線搜尋）的具體動作。
  • 📂 include/system_safety.h：安全性配置檔。定義 MAX_PWM、STALL_THRESHOLD 等常數，引導學員建立數據防護層。
  • 📂 task_scheduler.ino：任務調度模板。示範如何使用 millis() 建立 100Hz 的精確 PID 週期，並併行處理非即時任務（如指示燈閃爍）。
  • 📂 LOGIC_DIAGRAM.md：邏輯設計圖。要求學員在寫代碼前，先用 Mermaid 語法或文字描述繪製出狀態轉移的觸發條件。

2. 作業任務部署細節

任務 1：循跡狀態機設計 (State Machine Logic Plan)

  • 目標：將複雜任務「分而治之 (Divide and Conquer)」，解決「全白脫線」等異常場景。
  • Classroom 部署建議：
    • 場景建模設計：學員需在 README 中定義：當感測器「全白」時，狀態機應切換至哪一種搜尋策略（如：旋轉找線 vs. 倒退回找）？
    • 行為封裝實作：在 FSM 框架中實作不同模式下的 PID 參數切換（例如：直線高速 PID vs. 彎道精準 PID）。
    • PR 審視點：檢核狀態轉移邏輯是否嚴謹？是否有處理從「暫停」到「啟動」的平滑過渡（防抖與積分項清除）？

任務 2：非阻塞併行時序實作 (Async Timing Implementation)

  • 目標：徹底消除 delay()，維持大腦在執行 PID 時依然能隨時響應。
  • Classroom 部署建議：
    • 代碼去延遲化：學員需將所有 delay(500) 改寫為基於 millis() 的時間戳邏輯。
    • 併行效能觀測：實作一段代碼，在 Serial Plotter 繪製 Loop_Interval。若線條呈現平直的 10ms，則代表多工處理達標；若發生劇烈抖動 (Jitter)，則代表某個功能出現了阻塞。
    • Autograding：使用靜態掃描，若代碼中出現 delay 關鍵字則自動扣分（或警告）。

任務 3：系統魯棒性防線 (Robustness & Fail-Safe Engineering)

  • 目標：實現「安全性 (Safety)」重於「功能性 (Functionality)」的專業工程理念。
  • Classroom 部署建議：
    • 心跳包與失效保護：實作通訊心跳 (Heartbeat)。若 2 秒未收到網頁指令，車輛必須立即 SAFE STOP。
    • 馬達堵轉偵測 (Stall Guard)：學員需撰寫邏輯：如果 PWM 輸出 > 100 且 編碼器 Ticks 數連續 3 秒無變化，必須切斷輸出以防燒毀馬達。
    • 極限 Clamping 測試：模擬極大誤差輸入，檢核 PID 輸出是否被精確鎖定在 [0, 255] 區間。
    • Final Video：提交影片演示一個「意外情境」（如斷開通訊），證明系統能自行進入安全模式。

3. 系統架構師導師點評標準 (System Architecture Benchmarks)

此單元的價值在於 「程式碼的強韌度與可維護性」：

  • [ ] 架構解耦性：狀態邏輯是否與馬達驅動底層完全分離？
  • [ ] 即時響應率：控制迴圈是否有足夠的頻寬去處理突發的中斷？
  • [ ] 異常覆蓋程度：學員是否考慮到了「感測器故障」或「電量不足」時的緩解措施？
