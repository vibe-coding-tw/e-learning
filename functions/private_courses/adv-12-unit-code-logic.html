🔄 1. 有限狀態機 (FSM)：決策的骨架

無人車的大腦不是單一線程跑到底。它會根據目前的模式（如：靜止 Idle / 循跡 Tracking / 搜尋 Search）採取完全不同的計算策略。

FSM Core Elements

  • 狀態 (States)：系統當下的身份。如「搜尋循跡線中」。
  • 轉換 (Transitions)：觸發身份切換的事件。如「感測器全白」。
  • 行為 (Actions)：在特定狀態下執行的具體控制邏輯。

🛡️ 3. 防錯邏輯與系統魯棒性

工業級程式必須考量「驚喜」。數據溢位、感測器誤報、通訊中斷，都必須具備相應的失效保護 (Fail-Safe) 解決方案。

Robustness Architecture

  • 數值鉗位 (Clamping)：確保 PWM 永不超出邊際 \[0, 255\]，防止溢位導致的倒車災難。
  • 心跳包 (Heartbeat)：若一段時間未收到外部指令，系統應立刻進入 SAFE STOP。
  • 堵轉偵測 (Stall Guard)：感測馬達過載狀態，及時切斷電力以防電機燒毀。

控制大腦實驗室：高性能邏輯與異常處理

任務 1：循跡狀態機規劃 (State Machine Logic Plan)

角色扮演：系統架構分析師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 行為建模：用偽代碼寫下當「感測器突然全白」或「感測器全黑」時，系統應如何切換狀態？
  3. 模式轉換：規劃手動遙控模式與 PID 自動模式的「熱切換 (Hot Swap)」邏輯。
  4. Push Final：提交具備「狀態轉移地圖」與「行為偽文本」的架構專案。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 邏輯覆蓋率：偽代碼是否處理了「異常脫線」後的搜尋策略？
  • [ ] 轉換合理性：狀態機是否存在死循環或無法到達的狀態分支？

任務 2：非阻塞併行時序實作 (Asynchronous Timing Implementation)

角色扮演：實時系統工程師 | 難度：⭐️⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 迴圈重構：全面移除 delay()，改用時鐘閥門控制 PID 採樣率。
  2. 任務併行：在每一幀控制中，併行插入數據藍牙發送與視覺指示燈閃爍。
  3. 抖動測試：在 Serial Plotter 中記錄每一次 PID 計算的時間間隔 \(\Delta t\)，觀測其均勻度。
  4. Push Final：提交具備「準時序、無阻塞」特質的高性能控制代碼。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 代碼純淨度：全文是否徹底移除了會導致阻塞的指令？
  • [ ] 計算定標性：PID 的積分與微分項是否正確使用了實測的 \(\Delta t\) 而非固定常數？

任務 3：系統魯棒性防線 (Robustness Engineering)

角色扮演：系統安全性驗證專家 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 數據完整性保護：實作輸入 Clamping，保證 PID 計算值永不溢位。
  2. 異常探針部署：實作「馬達堵轉偵測」：若有輸出但編碼器讀值未變達 3 秒，強制切斷馬達輸出。
  3. 故障模式分析：在 README 中模擬 3 種失效場景，並用代碼證明系統能安全停機。
  4. Push Final：提交具備「工業級安全防護」的強韌型控制引擎專案。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 安全性評分：在模擬意外（如斷訊、卡死）時，代碼是否能有效防止設備受損？
  • [ ] 魯棒性思維：對界外值 (Outlier) 與硬體極限的考量是否足夠周全？
