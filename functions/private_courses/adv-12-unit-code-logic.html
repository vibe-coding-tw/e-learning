🔄 1. 有限狀態機 (FSM)：決策的骨架

無人車不是在一行程式跑到底。它會根據目前的模式（Idle / Run / Search）採取不同的計算策略。

FSM Core Elements

  • 狀態 (States)：系統當下的身份。如「搜尋循跡線中」。
  • 轉換 (Transitions)：觸發身份切換的事件。如「感測器全白」。
  • 行為 (Actions)：在特定狀態下執行的邏輯。

♾️ 2. 非阻塞迴圈：維持大腦的清醒

嚴禁在 PID 迴圈中使用 delay()。每一毫秒的阻塞，感測器都是盲目的，這足以讓高速行駛的車子衝出跑道。

🛡️ 3. 防錯邏輯與系統魯棒性

工業級程式必須考量「意外」。通訊中斷、數據溢位、感測器誤報，都必須具備相應的保護機制。

Robustness Checklist

  • 數值鉗位 (Clamping)：確保 PWM 永不超出 [0, 255]，防止因計算溢位導致的倒車。
  • 斷連保護 (Heartbeat)：若一段時間未收到藍牙指令，系統應立刻進入 SAFE STOP。
  • 卡死判定 (Stall Guard)：若馬達全速運轉但位置未更新，應切斷電力防止電機過熱燒毀。

Safety Shield

任務 1：「偽代碼建築師」：循跡狀態邏輯規劃 (Pseudocode Architect)

角色扮演：架構設計師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 邏輯梳理：用中文撰寫 PID 循跡的主迴圈架構：如何初始化誤差、何時更新積分、何時應急停止。
  3. 特殊狀態處理：定義「全白（脫線）」時的分支行為：是左右搜尋還是原地打轉？將此邏輯寫入狀態機分支中。
  4. 時序規劃報告：在 README 中說明為何不能頻繁讀取序列埠數據（如每一行 loop 都 print），分析對控制時間戳的影響。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 狀態轉換完整性：偽代碼是否具備應對外界感測變化的「非線性」決策能力（而不僅是簡單的 if-else）？
  • [ ] 關鍵時鐘理解：能否正確定位 PID 更新點必須處於「定時器」或「準確時間檢測」之下？

任務 2：「非阻塞任務器」：millis() 與 micros() 實戰 (Non-Blocking Multi-tasking)

角色扮演：代碼優化師 | 難度：⭐️⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 消除阻塞：移除所有的 delay()。實作 if (millis() - last_time > T) 結構來執行週期性任務。
  2. 高速控制迴圈：設定一個 10ms (100Hz) 的控制視窗。在該視窗內精確執行一次 PID 運算。
  3. 混合通訊壓測：在後台背景每隔 500ms 同步一次狀態數據至藍牙，驗證是否會導致主控制迴圈的「抖動 (Jitter)」。
  4. 結果 Push：將優化後的無阻塞主迴圈代碼推送至 GitHub。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 併行處理效率：在維持 PID 高頻更新的同時，副任務（如 LED 顯示、通訊）是否能順暢執行？
  • [ ] 控制週期穩定性：透過 Serial Plotter 輸出的執行時間戳，觀察 \(\Delta t\) 的波動是否控制在 10% 以內？

任務 3：「防錯邏輯保護」：系統魯棒性設計 (Robustness & Safeguard)

角色扮演：魯棒性工程師 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 範圍安全鉗位：實作 map() 與 constrain() 組合，保證馬達控制量永不溢位或出現非法負值。
  2. 看門狗與心跳檢測：實作一套機制，若連續 2 秒未從 App 收到校準指令，自動切換至安全模式或停機。
  3. 自我診斷邏輯：偵測「堵轉（感測器有差額但轉速為 0）」。若持續 3 秒，則發出警報並斷電。
  4. Push PR：提交具備工業防錯思維的最終版代碼支撐。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 軟體防禦意識：代碼中是否包含對感測器「突變值」或「無效值 (NaN)」的排除邏輯？
  • [ ] 安全性測試合格：在模擬通訊線路中斷時，系統是否能按照預期安全停擺，無危害行人行為。
