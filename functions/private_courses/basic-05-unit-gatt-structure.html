揭開藍牙通訊的面紗：
GATT 協議與數據交換

掌握 BLE 的通訊階層，設計出邏輯嚴密、分類清晰且高效的通訊協議地圖。

🏛️ 1. 藍牙的階層美學：GATT 結構

GATT (Generic Attribute Profile) 定義了設備如何交換數據。你可以把它想像成一個組織嚴密的數位圖書館：

📦 Profile

設備角色的完整定義。例如：「一台全功能的無人車」。

📂 Service

功能的邏輯分類。如：「動力控制」、「傳感數據」。

📄 Characteristic

具體的數據節點。如：「即時轉速」、「電池剩餘電量」。

🧠 Coding Tip

好的架構始於分類。將實時感測（超音波）與指令接收（馬達）分開放在不同 Service，能大幅簡化 App 端的邏輯。

🆔 2. 身份識別：UUID 的規格

為了確保全球設備不衝突，每個 Service 與 Characteristic 都必須擁有專屬的 128-bit 生存碼。

16-bit Standard

由藍牙聯盟預先定義的功能。例如心跳服務 (0x180D)。

128-bit Custom

開發者自行生成的隨機 ID。例如我們自製的無人車服務。

🔄 3. 數據交換的四大流派

📥

Read

由手機發問。適合穩定數值（如韌體版本）。

📤

Write

手機下令。用於馬達控制或 LED 切換。

🔔

Notify

設備主動推播。最高效的即時遙測方式。

✅

Indicate

附帶確認的通知。更安全但通訊較慢。

核心任務：打造你的 BLE 伺服器

從定義階層 Service 開始，到實現雙向、具備可靠性的通訊對話鏈。

🏛️ 60 min

【作業 5-1】

「多 Service 模型實作」

在 ESP32 上實作動力 (Motor) 與遙測 (Tele) 兩個獨立 Service。並透過 UUID 管理與 nRF Connect 驗證呈現的樹狀階層。

💬 45 min

【作業 5-2】

「指令解析與狀態回傳」

實作 Write 指令解析 (GO/STOP) 與 Read 運行秒數回傳。建立字串與 Byte 陣列間穩健的編解碼格式。

🚨 45 min

【作業 5-3】

「高頻率自動推播系統」

配置 BLE2902 描述符並開啟 Notify 推送。每 500ms 主動更新感測值，並與 Read 模式進行流暢度對比。

GATT 協議與階層架構實驗室

任務 1：「無人車通訊地圖」：GATT 服務建模 (GATT Modeling)

角色扮演：藍牙系統架構師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 多級架構編碼：在 ESP32 上實作動力 Service (UUID_A) 與遙測 Service (UUID_B)。
  3. 特徵值掛載：分別為兩個服務掛載對應的 Characteristic。確保 UUID 使用 128-bit 隨機生成避免衝突。
  4. 階層驗證：手機使用 nRF Connect 掃描後截圖，確認 Service 樹狀圖顯示正確。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] UUID 唯一性：是否採用隨機生成的 128-bit ID 而非重複使用預設 ID？
  • [ ] 建模合理性：動力與遙測功能是否被清晰解耦到不同的服務資料夾中？

任務 2：「雙向數據流」：字元與位元轉換 (Stream & Format)

角色扮演：韌體開發員 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 指令解析實作：開發 onWrite 回呼，偵測到 "GO" 則亮 LED。
  2. 狀態回饋實作：開發受動讀取特徵值。當被 Read 時回傳目前的累積秒數。
  3. 編解碼處理：在程式中正確控制 setValue() 與 getValue() 的數據類型轉換。
  4. 最終提交：將具備雙向握手能力的代碼 Push 進 PR。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 控制穩健度：手機能否 100% 成功觸發 LED 狀態切換而無遺落？
  • [ ] 格式對等性：手機讀取到的數據格式是否正確（非亂碼）且數值邏輯合理？

任務 3：「主動遙測器」：Descriptor 與 Notify 實作 (Active Telemetry)

角色扮演：資料遙測工程師 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 描述符注入：實作 BLE2902 Client Characteristic Configuration Descriptor。
  2. 異步推送邏輯：在 loop 中每 500ms 定時調用 pCharacteristic->notify() 向手機推播虛擬遙測值。
  3. 效能分析撰寫：在 README 中分析「主動推送」相對於「被動請求」在即時導航應用中的技術優勢。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 機制配置度：是否正確配置 Descriptor 以允許手機端開啟通知開關？
  • [ ] 通訊效率感：數據流在手機端的更新是否展現出比 Read 更好的連貫性？
