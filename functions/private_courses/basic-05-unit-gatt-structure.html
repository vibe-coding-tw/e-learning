簡介

GATT（通用屬性協定）定義了 BLE 設備之間交換數據的階層結構：Profile、Service、Characteristic。 本單元將帶你學會如何將無人車的動力控制與感測遙測拆分為獨立的 Service， 並透過 UUID 標識與 Notify / Read / Write 機制，建立一套邏輯嚴密的通訊藍圖。

藍牙的階層美學：GATT 結構

GATT（Generic Attribute Profile，通用屬性協定）是 BLE 設備間交換數據的核心框架。它定義了一套嚴謹的階層式資料模型， 讓任何 BLE 裝置都能以統一格式公開其功能與數據。你可以把 GATT 想像成一座結構嚴密的圖書館——每一層都有明確的職責與命名規範。

當手機與 ESP32 建立連線後，手機會執行一次服務發現（Service Discovery）程序，自動列舉出 ESP32 上所有的 Service 與 Characteristic。 這個過程完全由 BLE 協定棧自動完成，開發者只需在韌體中正確定義 GATT 結構即可。

三層階層架構

GATT 採用 Profile → Service → Characteristic 的三層樹狀結構。每一層的職責如下：

層級 角色 類比 無人車範例
Profile 定義設備的整體角色與能力組合 一本書的「類別」 無人車通訊協議
Service 將功能邏輯分組，每個 Service 代表一個獨立功能模組 書中的「章節」 動力服務、遙測服務
Characteristic 實際承載數據的最小單位，擁有值（Value）與屬性（Properties） 章節中的「段落」 馬達轉速、電池電量
備註 在實務中，Profile 通常是概念性的分類（如 Heart Rate Profile），程式碼中你主要操作的是 Service 與 Characteristic。一個 BLE Server 可以同時擁有多個 Service。

功能解耦設計

好的 GATT 架構設計遵循功能解耦原則——不同類型的功能應拆分到獨立的 Service 中。以無人車為例：

  • 動力服務（Drive Service）：包含馬達控制指令、轉向角度等 Characteristic
  • 遙測服務（Telemetry Service）：包含電池電量、超音波距離、陀螺儀角度等 Characteristic
  • 系統服務（System Service）：包含韌體版本、設備名稱等 Characteristic

這種設計讓手機 App 可以選擇性地訂閱感興趣的服務，也方便日後獨立擴充功能而不影響其他模組。

ESP32 程式碼範例

以下展示如何在 ESP32 上建立包含兩個 Service 的 GATT 結構：

// 建立 BLE Server
BLEServer *pServer = BLEDevice::createServer();

// 建立動力服務
BLEService *pDriveService = pServer->createService(DRIVE_SERVICE_UUID);

// 在動力服務下掛載馬達控制特徵值
BLECharacteristic *pMotorChar = pDriveService->createCharacteristic(
  MOTOR_CHAR_UUID,
  BLECharacteristic::PROPERTY_WRITE
);

// 建立遙測服務
BLEService *pTelemetryService = pServer->createService(TELEMETRY_SERVICE_UUID);

// 在遙測服務下掛載電池特徵值
BLECharacteristic *pBatteryChar = pTelemetryService->createCharacteristic(
  BATTERY_CHAR_UUID,
  BLECharacteristic::PROPERTY_READ | BLECharacteristic::PROPERTY_NOTIFY
);
提示 使用 nRF Connect App 連線到你的 ESP32 後，可以清楚看到完整的 GATT 樹狀結構。建議在開發過程中隨時使用此工具驗證你的 Service 與 Characteristic 是否正確掛載。

身份識別：UUID 規格

在 BLE 的世界中，每一個 Service 與 Characteristic 都需要一個唯一的識別碼——UUID（Universally Unique Identifier）。 UUID 就像是每個數據節點的「身份證號碼」，讓手機端能準確找到並操作目標特徵值，不會因為名稱重複而產生混淆。

BLE 規範支援兩種 UUID 格式：16-bit 標準 UUID 與 128-bit 自定義 UUID。選擇哪種格式取決於你的應用場景。

16-bit 標準 UUID

藍牙技術聯盟（Bluetooth SIG）預先定義了大量標準化的 16-bit UUID，涵蓋常見的功能，例如心率監測、電池電量、環境感測等。 使用標準 UUID 的好處是互通性（Interoperability）——任何支援該標準的 App 或設備都能自動識別並正確解析你的數據。

常見的 16-bit 標準 UUID：

UUID (Hex) Service 名稱 用途
0x180D Heart Rate 心跳感測器
0x180F Battery Service 電池電量回報
0x181A Environmental Sensing 溫度、濕度等環境數據
0x1800 Generic Access 設備名稱與外觀
備註 16-bit UUID 實際上是 128-bit 的縮寫形式。完整格式為 0000XXXX-0000-1000-8000-00805F9B34FB，其中 XXXX 就是 16-bit 值。

128-bit 自定義 UUID

當你開發的功能沒有對應的標準定義時（例如自製無人車的動力控制），就必須使用 128-bit UUID。這是一串由隨機演算法產生的全球唯一識別碼， 格式為 xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx（32 位十六進位字元，以連字號分隔為 5 組）。

產生 128-bit UUID 的方式：

  • 線上工具：uuidgenerator.net
  • 命令列：Linux/macOS 執行 uuidgen
  • Python：import uuid; print(uuid.uuid4())

UUID 定義最佳實踐

建議將所有 UUID 集中定義在一個標頭檔中，方便管理與維護：

// include/uuid_defs.h
#define DRIVE_SERVICE_UUID "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
#define MOTOR_CHAR_UUID    "a1b2c3d4-e5f6-7890-abcd-ef1234567891"
#define SPEED_CHAR_UUID   "a1b2c3d4-e5f6-7890-abcd-ef1234567892"

#define TELEM_SERVICE_UUID "b2c3d4e5-f6a7-8901-bcde-f12345678900"
#define BATTERY_CHAR_UUID  "b2c3d4e5-f6a7-8901-bcde-f12345678901"
警告 絕對不要手動「編造」UUID（如 "12345678-1234-1234-1234-123456789012"）。手動編造的 UUID 在全球範圍極可能與他人衝突，必須使用隨機生成工具確保唯一性。同一個專案中，每個 Service 和 Characteristic 的 UUID 也不可重複。

數據交換流派

GATT Characteristic 的屬性（Properties）決定了數據的流向與觸發方式。BLE 規範定義了四種主要的數據交換模式， 每種模式適合不同的應用場景。作為系統架構師，你必須為每個 Characteristic 選擇最合適的屬性組合。

四種核心交換模式

屬性 方向 觸發者 確認機制 適用場景
Read Server → Client Client（手機）主動發起 有（Request/Response） 韌體版本、設備 ID、靜態設定
Write Client → Server Client（手機）主動發起 可選（With/Without Response） 馬達指令、LED 控制、模式切換
Notify Server → Client Server（ESP32）主動推播 無（Fire-and-forget） 高頻感測數據、即時遙測
Indicate Server → Client Server（ESP32）主動推播 有（需等待 Client ACK） 關鍵警報、韌體更新確認

Notify vs. Indicate 的權衡

Notify 與 Indicate 都是 Server 主動推播數據的方式，但兩者有關鍵差異：

  • Notify：發送後不等待確認，吞吐量高、延遲低，但可能丟包。適合高頻感測數據——即使偶爾丟失一筆讀數，下一筆很快就會到來。
  • Indicate：發送後必須等待 Client 回傳確認（ACK），保證每一筆數據都被接收。適合關鍵事件（如緊急斷電確認），但頻率受限。
提示 在無人車專案中，超音波距離感測推薦使用 Notify（高頻、允許偶爾丟包），而「緊急斷電確認」則應使用 Indicate（必須保證指令已被執行）。

屬性組合設計

一個 Characteristic 可以同時擁有多種屬性。例如，你可以建立一個同時支援 Read 和 Notify 的感測特徵值——平時由 Server 主動推播（Notify）， 但 Client 也可以隨時主動查詢當前值（Read）：

BLECharacteristic *pSensorChar = pService->createCharacteristic(
  SENSOR_UUID,
  BLECharacteristic::PROPERTY_READ |
  BLECharacteristic::PROPERTY_NOTIFY
);

// Notify 需要掛載 BLE2902 描述符
pSensorChar->addDescriptor(new BLE2902());
警告 任何使用 Notify 或 Indicate 屬性的 Characteristic，都必須掛載 BLE2902（Client Characteristic Configuration Descriptor）。 少了這個描述符，手機端將無法訂閱通知，這是最常見的 BLE 開發錯誤之一。

實務設計思考

設計 GATT 屬性時，請思考以下問題：這筆數據的時效性如何？是否需要可靠傳輸？更新頻率多高？ 根據這三個維度，你就能為每個 Characteristic 選出最適合的屬性組合，打造出既高效又節能的通訊架構。

BLE 核心架構：GATT 協議與階層 (basic-05-unit-gatt-structure)

歡迎來到 BLE GATT 架構實作單元！在本課程中，你將從單一功能開發進化為通訊系統架構師。你將學習如何將複雜的硬體功能（動力、感測、資訊）邏輯性地解耦，建立層級分明的 GATT 服務樹。

🎯 學習目標

  1. GATT 建模：學會設計 Service 與 Characteristic 的階層關係。
  2. UUID 管理：掌握 128-bit 隨機 UUID 的配置與應用。
  3. 數據交互：實作 onWrite 指令解析與 Read 狀態回傳。
  4. 主動推播：配置 BLE2902 描述符並實作 Notify 通訊機制。

🚀 任務說明

任務 1：「無人車通訊地圖」GATT 服務建模

  • 目標：建立兩個獨立的 Service 以實現功能解耦。
    • 動力服務 (Drive Service)：負責控制車輛移動。
    • 遙測服務 (Telemetry Service)：負責監測設備狀態。
  • 要求：為每個 Service 與 Characteristic 生成唯一的 128-bit UUID。

任務 2：「雙向數據流」指令與狀態轉換

  • 目標：實現韌體與手機端的雙向數據交換。
  • 要求：
    • 實作 onWrite：接收手機端傳送的 "GO" 指令並觸發 LED。
    • 實作 Read：當手機讀取時，回傳設備啟動後的累計秒數 (Uptime)。

任務 3：「主動遙測器」Notify 實作

  • 目標：掌握 BLE 的「非同步推播」機制。
  • 要求：
    • 為遙測特徵值正確掛載 BLE2902 描述符。
    • 每 500ms 自動推送一次感測數據（可先使用虛擬數值）。

📂 程式庫結構

  • src/main.cpp：BLE Server 核心邏輯與服務樹實作。
  • include/uuid_defs.h：所有 128-bit UUID 的定義中心。
  • docs/GATT_Map.md：在此紀錄你的 GATT 樹狀架構規劃。

📝 繳交規範

  1. 完成代碼：確保代碼能通過編譯並在 ESP32 上正確運行。
  2. 通訊驗證：使用 nRF Connect 掃描並截圖你的 GATT 服務樹，貼至 docs/GATT_Map.md。
  3. Push 更新：完成後將代碼 Push 回 GitHub 程式庫以完成作業。

    • 狀態回饋 (Read)：當手機發起 Read 請求時，回傳設備自啟動以來的累積秒數。
    • 封裝穩健性：處理資料轉換中的常見錯誤（如 Byte 陣列長度判斷）。
    • 驗證要點：提交一份測試紀錄，證明能從手機端成功「下令」與「查詢」。

任務 3：「主動遙測器」Descriptor 與 Notify 實作 (Active Telemetry Lab)

  • 目標：掌握 BLE 特有的「推播 (Push)」機制，學習配置 BLE2902 描述符，這是專業藍牙開發的標配。
  • Classroom 部署建議：
    • 描述符注入：核心技術考點。學員必須正確配置 pCharacteristic->addDescriptor(new BLE2902())。
    • 定時推播：撰寫一個定時器（或在 Loop 中），每 500ms 自動推送一組虛擬感測值給手機。
    • 效能對比：在報告中說明：為何「主動推播 (Notify)」比「輪詢讀取 (Read)」在監控感測器時更具優勢？
    • 最終 Commit：封裝完整的遙測系統代碼並 Push。

3. 通訊建模與階層規劃點評標準 (Architecture & Protocol Standards)

此單元的價值在於 「對通訊系統的階層化思維與對工業標準（如 2902）的遵循」：

  • [ ] GATT 建模完整性：服務與特徵值的分類是否符合邏輯上的功能解耦？
  • [ ] 2902 描述符配置：是否正確配置了描述符以支持 Notify 權限？（這是最常見的失敗點，需重點確認）。
  • [ ] 編解碼穩健性：讀寫過程中的資料類型轉換（getValue(), setValue()）是否處理得當？
