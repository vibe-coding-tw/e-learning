簡介

ESP32 同時支援 AP（存取點）與 STA（站台）兩種 Wi-Fi 角色，甚至能雙模並行。 理解兩者的運作差異，是決定你的裝置「自建網路」還是「加入既有網路」的關鍵第一步， 也直接影響後續 Web Server 和 OTA 更新的網路拓撲設計。

情境
你正在設計一台遙控車：在家裡有路由器時，希望手機透過區網連線操控； 但帶到戶外沒有 Wi-Fi 時，裝置需要自己開一個熱點讓手機直連。 這兩種需求分別對應 STA 與 AP 模式——本單元將帶你搞清楚何時用哪一種、以及如何切換。

AP 模式：自立門戶

當 ESP32 以 AP（Access Point，存取點）模式運作時，它會自己建立一個獨立的 Wi-Fi 網路， 就像一台迷你無線路由器。任何裝置（手機、平板、電腦）都可以搜尋到這個網路並連入。 在此模式下，ESP32 扮演「基地台」角色——它是網路的中心節點，負責分配 IP 位址給連入的客戶端。

AP 模式最大的優勢是不依賴任何外部網路基礎設施。無論你身處有路由器的客廳， 還是完全沒有 Wi-Fi 的戶外空地，ESP32 都能自行建立通訊通道。這使得 AP 模式非常適合 「直連控制」場景——例如在公園裡用手機遙控車子，或在沒有網路的工廠環境中進行設備配對。

啟動 Soft-AP 的核心 API

ESP32 的 Arduino 框架提供了 WiFi.softAP() 函式來啟動軟體存取點。 以下是完整的初始化流程，包含 SSID 命名、密碼保護與基本網路配置：

#include <WiFi.h>

const char* ssid = "MyCar_01";
const char* password = "12345678";

void setup() {
  Serial.begin(115200);

  // 啟動 Soft-AP：頻道 1、不隱藏 SSID、最多 4 台連線
  WiFi.softAP(ssid, password, 1, 0, 4);

  IPAddress IP = WiFi.softAPIP();
  Serial.print("AP IP: ");
  Serial.println(IP);  // 預設輸出 192.168.4.1
}

softAP() 參數一覽

WiFi.softAP() 的完整簽名包含五個參數，每個都有預設值。 了解這些參數能幫助你針對不同應用場景進行精確調校：

參數型別說明預設值
ssidconst char*網路名稱，最長 32 字元（必填）
passwordconst char*密碼（至少 8 字元），傳 NULL 為開放網路NULL
channelintWi-Fi 頻道（1–13），避免與周圍路由器撞頻1
ssid_hiddenint設為 1 則隱藏 SSID（需手動輸入名稱才能連入）0
max_connectionint最大同時連線裝置數（ESP32 硬體上限為 10）4
注意 AP 模式下 ESP32 預設無法連上網際網路。連入 AP 的裝置只能與 ESP32 本身通訊， 無法瀏覽網頁或使用雲端服務。若需要同時存取外部網路，必須啟用 AP + STA 雙模式（第四單元將詳細說明）。

DHCP 與 IP 分配機制

ESP32 在 AP 模式下會自動啟動內建的 DHCP 伺服器，為每一台連入的客戶端自動分配 IP 位址。 預設的子網路配置如下：

  • ESP32（閘道器）：192.168.4.1
  • 第一台連入裝置：192.168.4.2
  • 子網路遮罩：255.255.255.0

你可以透過 WiFi.softAPConfig(local_ip, gateway, subnet) 自訂 IP 網段。 例如將 ESP32 設定為 10.0.0.1，這在需要避免與其他設備網段衝突時特別有用。 搭配 WiFi.softAPgetStationNum() 可隨時查詢目前連入人數，適合製作連線狀態儀表板。

提示 為你的 AP 取一個有辨識度的 SSID（如 VibeCar_張三），在多台 ESP32 同時運作的教室環境中， 可以避免同學之間互相連到錯誤的裝置。建議在 SSID 中加入設備編號或姓名。

STA 模式：加入家族

當 ESP32 以 STA（Station，站台）模式運作時，它會「加入」一個現有的 Wi-Fi 網路， 就像你的手機連上家裡的路由器一樣。ESP32 不再是網路的中心，而是區域網路中的一個節點（Client）， 由路由器負責分配 IP 位址並轉發封包。

STA 模式的核心優勢在於同時存取區網與網際網路。ESP32 不僅能被同一區網內的手機控制， 還能主動與外部伺服器通訊——上傳感測器數據到雲端、接收 OTA 韌體更新、或與其他物聯網設備協作。 這讓 STA 模式成為大多數室內物聯網應用的首選。

建立 STA 連線

使用 WiFi.begin() 即可啟動 STA 連線。但在生產環境中， 你絕不應該用 while 迴圈加 delay() 來等待連線完成—— 這會阻塞整個 loop()，導致馬達控制和感測器讀取全部停擺。 正確的做法是使用事件驅動（Event-Driven）模式。

#include <WiFi.h>

const char* ssid = "MyRouter";
const char* password = "router_password";
unsigned long lastReconnect = 0;

void onWiFiEvent(WiFiEvent_t event) {
  switch (event) {
    case ARDUINO_EVENT_WIFI_STA_GOT_IP:
      Serial.print("已連線！IP: ");
      Serial.println(WiFi.localIP());
      break;
    case ARDUINO_EVENT_WIFI_STA_DISCONNECTED:
      Serial.println("斷線！等待重連...");
      break;
  }
}

void setup() {
  Serial.begin(115200);
  WiFi.onEvent(onWiFiEvent);
  WiFi.begin(ssid, password);
}

void loop() {
  // 非阻塞重連：每 5 秒嘗試一次
  if (WiFi.status() != WL_CONNECTED
      && millis() - lastReconnect > 5000) {
    WiFi.reconnect();
    lastReconnect = millis();
  }
  // 馬達控制與感測器讀取正常運行，不受連線狀態影響
}
重要 禁止在 loop() 中使用 while (!WiFi.isConnected()) { delay(500); } 等待連線。這種阻塞式寫法會讓整個系統停擺——馬達不會轉、感測器不會讀、LED 不會閃。 務必使用 millis() 搭配非阻塞邏輯來實現重連。

常見 Wi-Fi 事件

WiFi.onEvent() 可以攔截多種連線狀態變化。 以下是 STA 模式中最常使用的事件，理解它們是建立穩定連線的基礎：

事件常數觸發時機典型處理
WIFI_STA_STARTSTA 模式啟動記錄日誌
WIFI_STA_CONNECTED已連上 AP（但尚未取得 IP）顯示「連線中…」
WIFI_STA_GOT_IPDHCP 已分配 IP 位址啟動 Web Server
WIFI_STA_DISCONNECTED連線中斷（AP 關閉或訊號消失）觸發重連計時器
WIFI_STA_LOST_IPIP 租約過期強制重新連線
提示 使用 WiFi.RSSI() 可以取得目前的訊號強度（單位為 dBm）。一般而言， -30 dBm 為極佳，-70 dBm 以下則可能導致頻繁斷線。 你可以將 RSSI 數值顯示在 Web 儀表板上，幫助使用者判斷是否需要調整路由器位置。

混合備援：產品級配網

在真實產品中，ESP32 不可能永遠只用 AP 或只用 STA 單一模式。使用者家裡有 Wi-Fi 時， 設備應自動連上路由器（STA）獲得最佳體驗；但如果帶到戶外或路由器故障時，設備必須能 自動切換到 AP 模式，讓使用者依然能夠控制。這種「雙模自動切換」就是產品級配網（Provisioning）的核心概念。

ESP32 原生支援 WIFI_AP_STA 雙模並行——也就是同時當存取點又當站台。 但更常見的做法是根據連線狀態動態切換模式，搭配 LED 指示燈讓使用者一眼就能辨識目前狀態。

配網備援流程

以下是一套經過實戰驗證的配網備援（Provisioning Fallback）流程：

1

嘗試 STA 連線

開機後讀取儲存的 SSID/密碼，呼叫 WiFi.begin() 嘗試連線。設定逾時上限（如 15 秒）。

2

逾時判定

若在逾時內未取得 IP，判定 STA 連線失敗。記錄失敗原因至 Serial 日誌。

3

降級啟動 AP

自動切換至 WiFi.softAP()，開放管理介面。使用者可透過 192.168.4.1 重新設定 Wi-Fi 憑證。

4

背景掃描與自動恢復

在 AP 模式下，定期以 WiFi.scanNetworks() 掃描周圍環境。一旦偵測到目標 SSID，自動切回 STA 模式。

狀態機程式碼範例

enum WiFiState { TRYING_STA, CONNECTED_STA, FALLBACK_AP };
WiFiState wifiState = TRYING_STA;
unsigned long staTimeout = 15000;
unsigned long startTime;

void setup() {
  Serial.begin(115200);
  WiFi.begin("MyRouter", "password");
  startTime = millis();
}

void loop() {
  switch (wifiState) {
    case TRYING_STA:
      if (WiFi.status() == WL_CONNECTED) {
        wifiState = CONNECTED_STA;
      } else if (millis() - startTime > staTimeout) {
        WiFi.disconnect();
        WiFi.softAP("MyCar_Setup", "12345678");
        wifiState = FALLBACK_AP;
      }
      break;
    case FALLBACK_AP:
      // 定期掃描，嘗試恢復 STA
      break;
  }
}

LED 狀態指示設計

好的產品不應該讓使用者猜測「設備現在到底在幹嘛」。透過 LED 閃爍頻率， 使用者可以一眼判斷 ESP32 目前處於哪種通訊狀態：

LED 行為代表狀態說明
慢閃（每 1 秒一次）STA 連線中正在嘗試連接路由器
恆亮STA 已連線成功加入區域網路
快閃（每 200ms 一次）AP 模式啟動已降級，等待使用者連入設定
熄滅Wi-Fi 已關閉省電模式或尚未初始化
注意 切換模式時務必先呼叫 WiFi.disconnect() 或 WiFi.softAPdisconnect() 關閉當前模式， 再啟動新模式。若不清理舊連線狀態，ESP32 可能出現記憶體洩漏或連線異常。 建議在每次切換時加入 delay(100) 讓底層 TCP/IP 堆疊完成清理。

掌握 AP + STA 雙模切換後，你的裝置就具備了「永不失聯」的通訊韌性。 這正是商用物聯網產品（如智慧家電、工業感測器）的基本要求—— 使用者不應該因為路由器重開機就需要拔掉設備電源重插。

ESP32 Wi-Fi 雙模式 (AP vs. STA) - 學生作業指南

課程目標

掌握 ESP32 的雙重通訊身分，學習如何處理不穩定的網路環境，並利用 事件驅動 (Event-Driven) 模式實作具備「自我修復能力 (Self-Healing)」與「配網備援 (Provisioning Fallback)」的強韌韌體架構。

作業任務

任務 1：「我的領地」AP 模式與 SSID 配置 (Access Point Lab)

  • 目標：建立私有網路控制段，理解 DHCP IP 分配機制，確保在無外部路由環境下依然具備控制權。
  • 實作要求：
    • 在 src/main.cpp 中實作 WiFi.softAP()，設定個性化的 SSID（如 MyCar_ID）與加密密碼。
    • 驗證：手機能連入並獲取 IP (通常為 192.168.4.x)，且能成功開啟 ESP32 託管的簡單網頁。

任務 2：「智慧尋路」STA 模式與斷線重連 (Smart Reconnect Lab)

  • 目標：掌握非同步事件監讀，學習如何在不干擾機器人主線程運行的情況下，自動修復網路連線。
  • 實作要求：
    • 實作 WiFi.onEvent() 攔截 ARDUINO_EVENT_WIFI_STA_DISCONNECTED。
    • 實作自動重連邏輯（如：每 5 秒嘗試一次）。
    • 關鍵限制：禁止 在 loop() 函數中使用 delay() 等待連線。
    • 數據提交：在 docs/Recovery_Log.md 中記錄關掉路由器後，設備恢復連線所需的平均時間。

任務 3：「終極備援」AP + STA 自動切換 (Provisioning Fallback Lab)

  • 目標：打造「永不失聯」的產品級韌體架構。當 STA 無法加入網路時，自動降級為 AP 模式供使用者重新設定，避免設備「變磚」。
  • 實作要求：
    • 開機先嘗試 STA 連線。若失敗，自動啟動 AP 模式。
    • UI 指示設計：使用 LED 閃爍頻率標示模式（慢閃 = STA 連線中，快閃 = 已降級為 AP 模式）。
    • 恢復機制：在 AP 模式下，若偵測到環境中預設路由出現，應能自動切回 STA 模式。
    • 驗證要點：模擬路由器關機情境，確認設備是否能在 30 秒內自動開啟 AP 訊號供緊急存取。

  • 📂 README.md：連線事件紀錄表。提供表格讓學員記錄：斷線發生時間、偵測到事件的時間，以及自動恢復連線的時間（毫秒級）。
  • 📂 docs/STATE_DIAGRAM.md：狀態轉移圖。要求學員以 Mermaid 語法或手繪截圖繪製其「備援切換邏輯」。

2. 作業任務部署細節

任務 1：「我的領地」AP 模式與 SSID 配置 (Access Point Lab)

  • 目標：學會建立私有網路控制段，理解 DHCP IP 分配機制，確保在無外部路由環境下依然具備控制權。
  • Classroom 部署建議：
    • 領地宣告：實作 WiFi.softAP()，設定個性化的 SSID（如 MyCar_ID）與加密密碼。
    • 連通驗證：手機能連入並獲取 IP (通常為 192.168.4.x)，且能成功開啟 ESP32 託管的簡單網頁。
    • 驗證要點：是否能正確取得 IP？AP 模式下的網頁反應是否迅速？

任務 2：「智慧尋路」STA 模式與斷線重連 (Smart Reconnect Lab)

  • 目標：掌握非同步事件監讀，學習如何在不干擾機器人主線程運行的情況下，自動修復網路連線。
  • Classroom 部署建議：
    • 事件監聽：實作 WiFi.onEvent() 攔截 ARDUINO_EVENT_WIFI_STA_DISCONNECTED。
    • 背景修復：實作自動重連邏輯（如：每 5 秒嘗試一次）。
    • 驗證要點：關鍵工程審核。學員是否在 loop() 函數中使用 delay() 等待連線？（這是錯誤作法）。應在背景事件中自動修復。
    • 數據提交：在 README 中記錄關掉路由器後，設備恢復連線所需的平均時間。

任務 3：「終極備援」AP + STA 自動切換 (Provisioning Fallback Lab)

  • 目標：打造「永不失聯」的產品級韌體架構。當 STA 無法加入網路時，自動降級為 AP 模式供使用者重新設定，避免設備「變磚」。
  • Classroom 部署建議：
    • 優先權邏輯：開機先嘗試 STA 連線 N 次。若失敗，自動啟動 AP 模式。
    • UI 指示設計：使用 LED 閃爍頻率標示模式（慢閃 = STA 連線中，快閃 = 已降級為 AP 模式）。
    • 恢復機制：在 AP 模式下，若偵測到環境中預設路由出現，是否能自動切回 STA 模式？
    • 驗證要點：模擬路由器關機情境，確認設備是否能在 30 秒內自動開啟 AP 訊號供緊急存取。

此單元的價值在於 「對通訊狀態的敏銳感知與對系統可用性的極致追求」：

  • [ ] 備援機制完備性：降級為 AP 後，使用者是否能透過預設 IP 重新進入管理介面？
