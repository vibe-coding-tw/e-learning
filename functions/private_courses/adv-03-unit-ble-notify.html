BLE Notify 機制深論
「拉」到「推」

🔄 1. 觀念翻轉：「我要數據」到「有數據通知我」 (25 min)

📥 Read (拉 - Pull)

客戶端（網頁）定時發送請求問：「現在座標是多少？」。

代價：即使數據沒變，也要消耗無線電頻寬，產生大量無效通訊。

📤 Notify (推 - Push)

客戶端先「訂閱 (Subscribe)」某個特徵值，當數值改變時，ESP32 主動推送封包。

優勢：低延遲、省電、僅在必要時占用頻寬。

🤔 思考任務

思考為什麼「按鈕按下」或「感測器警報」絕對應該使用 Notify 而 Read？

⚙️ 2. CCCD 與 MTU 的核心限制 (20 min)

CCCD (Client Characteristic Configuration Descriptor)

Notify 不是預設開啟的。網頁端必須寫入一個特殊的描述符（CCCD），告訴 ESP32：「我已經準備好接收通知了」。沒有這個「握手」，Notify 就不會運作。

MTU (Maximum Transmission Unit)

單個 Notify 封包的大小限制（預設通常為 20-23 Bytes），這決定了我們 JSON 數據打包的上限。如要傳大數據，就必須「切片」。

⚡ 3. 事件驅動的程式流程 (15 min)

阻監聽：觀察 Web Bluetooth API 中的 characteristicvaluechanged 事件。這與傳統的 setInterval 獲取數據有何本質上的不同？

🚀 預習任務：代碼觀察 

characteristic.startNotifications().then(_ => {
  characteristic.addEventListener('characteristicvaluechanged', handleNotifications);
});

當 ESP32 端呼叫 notify() 時，JS 端的呼函式會「立即」被喚醒。

任務 2：「變動觸發」優化實驗 (Adaptive Trigger Lab)

角色扮演：韌體優化工程師 | 難度：⭐️⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 閾值邏輯：在 ESP32 端實作 diff > Threshold 判斷，避免發送重複或極微小的數值變化。
  2. 流量監測：觀察並紀錄在「靜止」與「劇烈變動」場景下，藍牙封包發送頻率的差異。
  3. 穩定性測試：確保在快速變化的極端情況下，系統依然能準確捕捉到關鍵轉折。
  4. Push Final：提交具備「自適應推送策略」的高效能通訊韌體。

任務 3：大數據分段與流控 (MTU Fragmentation Lab)

角色扮演：通訊協議開發者 | 難度：⭐️⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 字串碎形化：發送長度超過 100 Bytes 的 JSON 字串，挑戰 MTU 實體限制。
  2. 序列化發送：將長字串分割成片段，並依序調用 Notify 函式，確保數據封裝完整。
  3. 緩衝重組：在前端實作 Buffer 合併機制，直到收到結束符才進行 JSON 解。
  4. Push Final：提交具備「長報文分段傳輸」能力的穩定通訊構。
