簡介

如果你的網頁每 100 毫秒就發一次 BLE Read 去問「現在感測器數據是多少？」，不僅浪費無線電資源，還會產生不可預期的延遲。 BLE Notify 機制讓裝置在數據變化時主動推送，實現真正的事件驅動通訊——低延遲、省電、零輪詢開銷。 本模組將帶你理解 Notify 的訂閱流程（CCCD）與 MTU 限制，掌握建構即時響應系統的核心通訊模式。

情境
你的無人車網頁控制台需要即時顯示陀螺儀數據。最初你用 setInterval 每 200ms 呼叫一次 BLE Read，結果畫面更新又慢又不穩定，而且手機電量消耗飛快。學長告訴你：「用 Notify 啊，讓 ESP32 主動推給你。」但你寫了 startNotifications() 卻什麼事都沒發生——原來是少了 CCCD 訂閱這一步「握手」。本模組將幫你徹底搞懂 Notify 的完整生命週期。

觀念翻轉：「我要數據」到「有數據通知我」

在 BLE 通訊中，數據的傳遞方式可以分為兩種截然不同的模型：Read（拉取）與 Notify（推送）。這兩者的差異不僅影響延遲與功耗，更決定了你的物聯網應用架構是「笨重的輪詢」還是「優雅的事件驅動」。

Read（拉取模型）：我要數據

在 Read 模式下，客戶端（網頁）必須主動發起請求才能取得感測器數據。典型做法是使用 setInterval() 定時呼叫 characteristic.readValue()——即使感測器數據毫無變化，每次都會佔用一次完整的 BLE 請求-回應交易（Request-Response Transaction）。

// ❌ Read 模式：每 200ms 輪詢一次
setInterval(async () => {
  const val = await characteristic.readValue();
  updateUI(val); // 即使數據沒變也要處理
}, 200);

Notify（推送模型）：有數據通知我

在 Notify 模式下，客戶端只需訂閱一次，之後由 ESP32 在數據變化時主動推送封包。網頁端透過事件監聽器（Event Listener）接收更新，完全無需輪詢。這種模式消除了無效通訊，實現了真正的事件驅動架構。

// ✅ Notify 模式：訂閱後被動接收
await characteristic.startNotifications();
characteristic.addEventListener(
  'characteristicvaluechanged',
  (event) => updateUI(event.target.value)
);

效能對比

比較項目 Read（拉取） Notify（推送）
數據取得方式 客戶端定時主動請求 伺服端主動推送
延遲 取決於輪詢間隔（如 200ms） 接近即時（< 30ms）
功耗 高（持續無線電交易） 低（僅變化時傳送）
無效通訊 大量（數據未變仍要請求） 零（無變化 = 無封包）
BLE 頻寬佔用 每次請求 = 2 個封包（Req + Rsp） 每次通知 = 1 個封包
適合場景 低頻、按需讀取（如設定值） 高頻即時數據（如感測器串流）
警告 使用 setInterval + readValue() 來模擬即時更新，會在手機端造成嚴重的電池消耗。實測顯示，以 100ms 間隔持續輪詢 BLE Characteristic，手機藍牙模組功耗可達純 Notify 模式的 5~8 倍。
提示 任何「瞬時事件」（如按鈕按下、碰撞偵測、感測器警報）都應該使用 Notify，因為 Read 模式的輪詢間隔會產生不可預期的延遲，可能錯過短暫的事件觸發窗口。

CCCD 與 MTU 的核心限制

Notify 機制並不是「開箱即用」的——它需要客戶端主動完成一個訂閱握手流程。這個握手的核心就是 CCCD（Client Characteristic Configuration Descriptor）。同時，每個 Notify 封包的大小也受 MTU 限制，這決定了單次推送能攜帶多少數據。

CCCD 是什麼？

CCCD 是 BLE GATT 規範中定義的一個特殊描述符（Descriptor），UUID 為 0x2902。它附屬在每個支援 Notify 的 Characteristic 底下，用來記錄客戶端的「訂閱偏好」。只有當客戶端向 CCCD 寫入正確的值後，伺服器端（ESP32）才會開始向該客戶端發送通知。

CCCD 值（16-bit） 二進位 含義
0x0000 00000000 00000000 關閉通知與指示（預設值）
0x0001 00000000 00000001 啟用 Notification（不需確認）
0x0002 00000000 00000010 啟用 Indication（需要確認回覆）
提示 Notify 與 Indicate 的差異在於：Notify 是「發射後不管」（Fire-and-Forget），伺服器不會等待客戶端回覆確認；而 Indicate 則要求客戶端回傳 ACK 確認收到。對於高頻感測器數據，Notify 更適合；對於關鍵控制指令，Indicate 更可靠。

訂閱握手流程

以下是 Web Bluetooth 啟用 Notify 的完整步驟：

1

取得 Characteristic 參考

透過 service.getCharacteristic(uuid) 取得目標特徵值物件。

2

呼叫 startNotifications()

此方法會自動向 CCCD 寫入 0x0001，完成訂閱握手。這是關鍵的「啟動開關」。

3

註冊事件監聽器

透過 addEventListener('characteristicvaluechanged', handler) 接收後續的推送數據。

// 完整的 Notify 訂閱流程
const characteristic = await service.getCharacteristic(TX_UUID);

// 步驟 1: 啟動通知（自動寫入 CCCD = 0x0001）
await characteristic.startNotifications();

// 步驟 2: 註冊事件監聽器
characteristic.addEventListener(
  'characteristicvaluechanged',
  handleNotification
);
警告 如果你只呼叫了 addEventListener() 而忘記呼叫 startNotifications()，ESP32 端的 notify() 會被靜默忽略——因為 CCCD 仍為 0x0000（關閉狀態），伺服器不會向該客戶端發送任何通知封包。

MTU 對 Notify 封包的限制

每個 Notify 封包的最大數據量同樣受 ATT MTU 限制。在預設 MTU = 23 Bytes 的環境下，單次 Notify 最多只能攜帶 20 Bytes 的有效數據（扣除 3 Bytes ATT 標頭）。如果 ESP32 端呼叫 notify() 發送超過此限制的數據，超出部分將被靜默截斷。

備註 若要透過 Notify 推送大型 JSON 數據，必須在 ESP32 韌體端實作分段發送邏輯——將長字串切成多個 Notify 封包依序發出，並在前端的事件處理器中進行緩衝區重組。此流程與前一單元的 MTU 分段傳輸概念相同。

事件驅動的程式流程

BLE Notify 的核心價值在於將程式架構從「定時輪詢」轉變為事件驅動（Event-Driven）。在這種模式下，你的程式碼不再主動詢問數據，而是註冊一個回呼函式（Callback），當 ESP32 端觸發 notify() 時，瀏覽器會自動喚醒你的處理函式。這與 DOM 事件（如 click、input）的運作方式完全一致。

characteristicvaluechanged 事件

Web Bluetooth API 提供了 characteristicvaluechanged 事件，每當 ESP32 端透過 notify() 推送新數據時，此事件就會被觸發。事件物件的 event.target.value 屬性包含接收到的原始位元組數據（型別為 DataView）。

完整的事件處理流程

// 步驟 1: 啟動 Notify 並註冊監聽器
async function subscribeToSensor(characteristic) {
  await characteristic.startNotifications();
  characteristic.addEventListener(
    'characteristicvaluechanged',
    handleNotification
  );
}

// 步驟 2: 處理收到的 Notify 數據
function handleNotification(event) {
  const decoder = new TextDecoder();
  const text = decoder.decode(event.target.value);
  console.log('收到推送:', text);
}

DataView 與文字解碼

event.target.value 回傳的是 DataView 物件（原始位元組緩衝區），而非直接的字串。你需要使用 TextDecoder 將其轉換為可讀的 UTF-8 文字。如果 ESP32 端發送的是 JSON 字串，解碼後就可以直接傳入 JSON.parse()。

DataView 方法 用途 適用場景
TextDecoder.decode() 將位元組轉為 UTF-8 字串 接收 JSON 文字數據
getUint8(offset) 讀取指定位置的單一位元組 接收二進位感測器數據
getInt16(offset, true) 讀取 16-bit 有號整數（Little-Endian） 讀取加速度計原始值
警告 如果 ESP32 端以高頻率發送 Notify（例如每 20ms 一次），前端的事件處理器也會以同樣的頻率被觸發。如果 handleNotification 內包含耗時的 DOM 操作或 JSON.parse()，可能導致介面渲染卡頓。建議使用 requestAnimationFrame 或節流（Throttle）機制來控制 UI 更新頻率。

資源清理：避免記憶體洩漏

當使用者離開頁面或中斷 BLE 連線時，務必清除事件監聽器並停止通知訂閱，否則會造成記憶體洩漏或幽靈回呼（Ghost Callback）：

async function cleanup(characteristic) {
  characteristic.removeEventListener(
    'characteristicvaluechanged',
    handleNotification
  );
  await characteristic.stopNotifications();
}
提示 良好的資源管理習慣：在呼叫 startNotifications() 之前，先檢查 characteristic.properties.notify 是否為 true。如果該 Characteristic 不支援 Notify 屬性，呼叫將會拋出錯誤。這種防禦性檢查展現了專業的函式庫開發思維。

adv-03-unit-ble-notify: 從「拉」到「推」的進化

本單元聚焦於 BLE 的 Notify (通知) 機制。你將學習如何從傳統的輪詢 (Polling/Read) 模式，轉化為更高效、更及時的事件驅動 (Event-Driven) 推送模式。這對於開發低延遲、省電的 IoT 系統至關重要。

🎯 學習目標

  1. 掌握 CCCD 訂閱機制：實作前端網頁與 ESP32 之間的通知握手。
  2. 變動觸發優化：撰寫韌體邏輯，僅在數據發生顯著變化時才發送更新，降低通訊開銷。
  3. MTU 分段與流控：處理超過物理限制的大型 JSON 數據推送，實作前端重組邏輯。

  • [ ] 邊界安全檢查：對於超過 MTU 的 JSON 片段，重組邏輯是否考慮了連線突然中斷後 Buffer 的自動清除機制？
