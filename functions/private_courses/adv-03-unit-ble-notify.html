🔄 1. 觀念翻轉：「我要數據」到「有數據通知我」 (25 min)

📥 Read (拉 - Pull)

客戶端（網頁）定時發送請求問：「現在座標是多少？」。

代價：即使數據沒變，也要消耗無線電頻寬，產生大量無效通訊。

📤 Notify (推 - Push)

客戶端先「訂閱 (Subscribe)」某個特徵值，當數值改變時，ESP32 主動推送封包。

優勢：低延遲、省電、僅在必要時占用頻寬。

思考為什麼「按鈕按下」或「感測器警報」絕對應該使用 Notify 而 Read？

⚙️ 2. CCCD 與 MTU 的核心限制 (20 min)

CCCD (Client Characteristic Configuration Descriptor)

Notify 不是預設開啟的。網頁端必須寫入一個特殊的描述符（CCCD），告訴 ESP32：「我已經準備好接收通知了」。沒有這個「握手」，Notify 就不會運作。

MTU (Maximum Transmission Unit)

單個 Notify 封包的大小限制（預設通常為 20-23 Bytes），這決定了我們 JSON 數據打包的上限。如要傳大數據，就必須「切片」。

🎯 學習目標

  • 繪製 BLE 通訊中的客戶端與服務端互動流程圖。
  • 理解 Client 訂閱 CCCD 的完整過程。

⚡ 3. 事件驅動的程式流程 (15 min)

阻監聽：觀察 Web Bluetooth API 中的 characteristicvaluechanged 事件。這與傳統的 setInterval 獲取數據有何本質上的不同？

🚀 預習任務：代碼觀察 

characteristic.startNotifications().then(_ => {
  characteristic.addEventListener('characteristicvaluechanged', handleNotifications);

當 ESP32 端呼叫 notify() 時，JS 端的呼函式會「立即」被喚醒。

adv-03-unit-ble-notify: 從「拉」到「推」的進化

本單元聚焦於 BLE 的 Notify (通知) 機制。你將學習如何從傳統的輪詢 (Polling/Read) 模式，轉化為更高效、更及時的事件驅動 (Event-Driven) 推送模式。這對於開發低延遲、省電的 IoT 系統至關重要。

🎯 學習目標

  1. 掌握 CCCD 訂閱機制：實作前端網頁與 ESP32 之間的通知握手。
  2. 變動觸發優化：撰寫韌體邏輯，僅在數據發生顯著變化時才發送更新，降低通訊開銷。
  3. MTU 分段與流控：處理超過物理限制的大型 JSON 數據推送，實作前端重組邏輯。

🛠 任務說明

任務 1：心跳偵測系統 (Heartbeat Notify Lab)

  • 目標：建立穩定且正確的 BLE 通知訂閱流程。
  • 關鍵點：在 src/ble_app.js 中實作 startNotifications() 並正確處理 characteristicvaluechanged 事件。

任務 2：變動觸發優化 (Adaptive Trigger Lab)

  • 目標：透過數學規律過濾不必要的數據推送。
  • 關鍵點：修改 firmware/sensor_threshold.ino，實作閾值 (Threshold) 判定邏輯。只有當數值位移超過設定範圍時，才觸發 notify()。

任務 3：大數據分段與流控 (MTU Fragmentation Lab)

  • 目標：突破 MTU 限制，實現完整 JSON 數據的可靠推送。
  • 關鍵點：在 src/buffer_manager.js 中實作數據重組邏輯，並在 sensor_threshold.ino 中分段發送長字串。

📁 程式庫結構

  • src/ble_app.js: 前端主要邏輯，負責連線與訂閱。
  • src/buffer_manager.js: 負責處理長數據的分段與重組。
  • firmware/sensor_threshold.ino: ESP32 韌體，包含感測器模擬與推送邏輯。
  • docs/traffic_results.md: 效能分析報告，請在此填寫你的實驗數據。

🚀 如何開始

  1. 複製此 程式庫 到你的本地環境。
  2. 使用 PlatformIO 或 Arduino IDE 上傳 firmware/ 下的程式碼到 ESP32。
  3. 開啟 index.html (建議使用 Live Server) 進行測試。
  4. 完成各項任務後，將代碼推送到你的 程式庫。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template Repo) 配置建議

Notify 的開發需要「訂閱」與「監聽」的精準配合，建議範本包含：

  • 📂 src/notify-listener.js：前端 JS 檔案，提供 Web Bluetooth 的連線骨架，重點留下 characteristic.startNotifications() 與事件監聽器的 TODO 區塊。
  • 📂 firmware/sensor-notify.ino：ESP32 韌體程式碼，預置模擬感測器讀取邏輯，並預留 notify() 觸發條件的實作空間。
  • 📂 docs/performance_stats.csv：效能紀錄表。學員需填入在不同閾值 (Threshold) 設定下，一分鐘內產生的封包總數，這能讓學員直觀感受「流量優化」的效果。
  • 📂 tests/event_check.test.js：單元測試腳本。模擬 BLENotify 事件，驗證前端是否能正確解析 DataView（例如：位元組偏移量是否計算正確）。

2. 作業任務部署細節

任務 1：心跳偵測系統 (Heartbeat Notify Lab)

  • 目標：成功建立 CCCD 訂閱機制，掌握事件觸發流程。
  • Classroom 部署建議：
    • 驗證方式：學員需實作網頁端的異步訂閱邏輯。導師應檢查代碼中是否包含了正確的事件移除處理（防止記憶體洩漏）。
    • Autograding：透過自動化測試檢測網頁是否發出了 startNotifications 請求。

任務 2：變動觸發優化 (Adaptive Trigger Lab)

  • 目標：利用數學規律降低通訊開銷，落實「數據即資源」的環保通訊思維。
  • Classroom 部署建議：
    • 核心代碼檢核： 
      // 學生需實作：僅在數值位移超過 5 的時候才調用 Notify
if (abs(currentVal - lastSentVal) > 5) {
    characteristic->setValue(currentVal);
    characteristic->notify();
    lastSentVal = currentVal;
}
    • 數據提交：學員需在 performance_stats.csv 中提交對比數據，證明在靜止狀態下，優化後的封包量比定時 Read 降低了 80% 以上。

任務 3：大數據分段與流控 (MTU Fragmentation Lab)

  • 目標：突破物理載體限制，實現原子化的 JSON 推送。
  • Classroom 部署建議：
    • 重組邏輯檢核：學員需在前端 JS 實作 Buffer 緩衝區。導師點評重點在於「終止符處理」：當收到含有 \n 的片段時，是否能正確觸發 JSON.parse？
    • 魯棒性驗證：要求學員實作一個 Try-Catch 機制，當接收到「不完整或錯誤」的推送數據時，系統應能自動清空 Buffer 重新等待，而不是直接讓網頁當機。

3. 事件驅動導師點評標準 (Event-Driven Benchmarks)

此單元的價值在於 「對通訊效率與系統魯棒性的權衡」：

  • [ ] 非對稱更新處理：前端 UI 是否能流暢反應突發的高頻通知（如：馬達全速轉動時的編碼器回傳）而不產生渲染卡頓？
  • [ ] 訂閱握手嚴謹度：代碼中是否判斷了 characteristic.properties.notify 屬性？（這展現了專業庫的開發思維）。
  • [ ] 邊界安全檢查：對於超過 MTU 的 JSON 片段，重組邏輯是否考慮了連線突然中斷後 Buffer 的自動清除機制？
