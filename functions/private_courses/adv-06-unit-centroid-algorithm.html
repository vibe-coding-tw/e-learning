什麼是影像質心？

二值化後的影像告訴我們「哪裡有目標像素」，但控制系統需要的是一個精確的座標數字。影像質心（Centroid）就是將分散在二維空間中的成千上萬個目標像素，「壓縮」成一個單一座標點 $(\bar{x}, \bar{y})$ 的數學操作。這個座標代表了目標區域的幾何平衡中心，是驅動舵機轉向的最關鍵輸入。

質心的數學定義

質心座標的計算公式非常直覺——它就是所有目標像素座標的算術平均：

$$\bar{x} = \frac{\sum_{i=1}^{n} x_i}{n}, \quad \bar{y} = \frac{\sum_{i=1}^{n} y_i}{n}$$

其中 $n$ 是目標像素的總數，$(x_i, y_i)$ 是第 $i$ 個目標像素的座標。以一張 $320 \times 240$ 的影像為例，如果二值化後有 500 個黑色像素，質心就是這 500 個像素座標的平均值。

注意 質心計算的本質是一次空間降維：將二維的像素分佈（$320 \times 240 = 76,800$ 個數據點）壓縮為一組 $(x, y)$ 座標。這種降維讓控制系統能夠以極低的計算成本做出即時的轉向決策。

物理直覺：平衡點的概念

質心的概念來自物理學的重心（Center of Mass）。想像將一塊形狀不規則的薄板放在手指上——能讓它保持水平平衡的那個點，就是質心。同理，影像質心就是目標像素「質量分佈」的平衡點。如果賽道線偏左，質心也會偏左；如果賽道線在正中央，質心就恰好在畫面中心。

實作考量：除以零的保護

在實際應用中，有一個極其重要但容易被忽略的邊界情況：當畫面中完全沒有目標像素時（例如車輛衝出賽道、或白色背景完全遮蓋賽道線），$n = 0$，此時公式中的除法會導致程式崩潰。

// 質心計算的基本框架（含除以零保護）
let sumX = 0, sumY = 0, count = 0;
for (let y = 0; y < height; y++)
  for (let x = 0; x < width; x++)
    if (isBlack(x, y)) { sumX += x; sumY += y; count++; }

if (count > 0) {
  centroidX = sumX / count;
  centroidY = sumY / count;
} else {
  // 目標遺失：維持上一幀座標或回到畫面中心
  centroidX = width / 2;
}
警告 if (count > 0) 這行看似簡單，卻是區分「能寫公式的學生」與「具備防禦性編程意識的工程師」的關鍵。在自動評分中，此保護邏輯是必測項目。

從二值化到質心：數據流

質心計算的完整數據流是一條清晰的管線（Pipeline）：從原始影像出發，經過灰階轉換、二值化、形態學處理，最終萃取出一組座標數字。理解這條管線中每個階段的輸入與輸出格式，是正確實作演算法的前提。

數據管線總覽

階段輸入輸出數據規模
1. 灰階轉換RGB 三通道像素陣列單通道灰階陣列$320 \times 240 \times 3$ → $320 \times 240 \times 1$
2. 二值化灰階值 $[0, 255]$二元值 $\{0, 255\}$資訊量大幅壓縮
3. 形態學處理含噪二值影像乾淨二值影像像素數量不變，品質提升
4. 質心計算所有目標像素的 $(x, y)$單一座標 $(\bar{x}, \bar{y})$$n$ 個點 → 1 個點

累加器模式（Accumulator Pattern）

質心計算的程式邏輯遵循經典的累加器模式：宣告三個變數 sumX、sumY、count，初始值皆為 0。接著用雙重迴圈逐一掃描每個像素，當發現目標像素時，將其座標累加到對應的變數中。迴圈結束後，除以計數即可得到質心。

// 累加器模式的完整實作
function computeCentroid(binaryData, width, height) {
  let sumX = 0, sumY = 0, count = 0;

  for (let y = 0; y < height; y++) {
    for (let x = 0; x < width; x++) {
      const idx = (y * width + x) * 4;
      if (binaryData[idx] === 0) { // 黑色 = 目標
        sumX += x; sumY += y; count++;
      }
    }
  }

  return count > 0
    ? { x: sumX / count, y: sumY / count, pixels: count }
    : null; // 無目標時回傳 null
}
注意 累加器變數必須在每一幀開始時重置為 0。若忘記重置，sumX 和 sumY 會持續累積前幾幀的數據，導致質心座標完全失準。這是初學者最常犯的邏輯錯誤。

效能考量：行掃描 vs. 全幀掃描

在循跡應用中，通常只需要質心的水平座標 $\bar{x}$（用來判斷賽道線偏左或偏右）。此時可以只掃描影像的特定行（例如 $Y = 200$ 附近），而不需要遍歷全部 76,800 個像素。這種行掃描（Line Scan）策略能將計算量從 $O(W \times H)$ 降低到 $O(W)$，大幅提升即時性。

提示 函式回傳的 pixels（目標像素計數）是一項重要的附帶資訊。當 pixels 數量過少時，代表目標可能已丟失或被噪點取代，此時質心座標的可靠度極低，不應直接用於控制。

ROI、誤差與控制連動

質心座標本身只是一組數字，它必須經過「誤差計算」與「角度映射」兩步轉換，才能變成舵機能執行的控制指令。同時，透過感興趣區域（Region of Interest, ROI）的裁切，可以排除畫面中無關區域的干擾，讓質心計算更加精準。

ROI 裁切：聚焦關鍵區域

一張 $320 \times 240$ 的影像中，上半部通常是天花板或遠方背景，對循跡毫無幫助，反而可能引入噪點。透過 ROI 裁切，我們只對影像的下半部（例如 $Y = 160$ 到 $Y = 240$ 之間）進行質心計算，大幅減少計算量與噪點干擾。

// 只掃描 ROI 區域內的像素
const ROI_START_Y = 160;
const ROI_END_Y = 240;

for (let y = ROI_START_Y; y < ROI_END_Y; y++)
  for (let x = 0; x < width; x++)
    if (isBlack(x, y)) { sumX += x; count++; }

誤差計算：偏差量化

誤差（Error）是質心水平座標與畫面中心之間的差值。它代表了「目標偏離正前方的程度」。畫面寬度為 320 像素時，中心點為 160：

$$e = \bar{x} - 160$$

  • $e > 0$：目標在畫面右側 → 車輛需要右轉
  • $e < 0$：目標在畫面左側 → 車輛需要左轉
  • $e = 0$：目標正好在正前方 → 直行

線性映射：從像素誤差到舵機角度

舵機的轉向角度通常限制在 $[60°, 120°]$ 範圍內（其中 $90°$ 為直行）。我們需要將誤差值 $e \in [-160, 160]$ 線性映射到角度區間：

$$\theta = 90 + \frac{e}{160} \times 30$$

// 誤差 → 舵機角度的線性映射
function mapToServo(error) {
  let angle = 90 + (error / 160) * 30;
  // 數值限幅：防止舵機超出安全範圍
  angle = Math.max(60, Math.min(120, angle));
  return Math.round(angle);
}
質心位置 $\bar{x}$誤差 $e$映射角度 $\theta$車輛行為
0（最左側）$-160$$60°$全力左轉
80$-80$$75°$中度左轉
160（正中央）$0$$90°$直行
240$+80$$105°$中度右轉
320（最右側）$+160$$120°$全力右轉
警告 Math.max(60, Math.min(120, angle)) 這行數值限幅（Clipping）邏輯至關重要。若映射結果超出 $[60°, 120°]$，過大的角度會導致舵機齒輪卡死甚至燒毀。這是保護硬體安全的最後一道防線。
提示 在低通濾波尚未加入前，質心每幀可能跳動數十像素。可使用加權移動平均 filteredX = newX * 0.7 + prevX * 0.3 來平滑質心軌跡，抑制舵機的高頻抖動。

任務 1：質心追蹤器 — 從像素到座標

目標：在 js/accumulators.js 中實作像素遍歷與質心計算邏輯。

需求：

  • 對影像所有像素進行雙重迴圈遍歷

任務 2：質心穩定化過濾器

目標：在 js/smooth_filters.js 中實作低通濾波，消除質心座標的抖動。

需求：

  • 實作加權移動平均公式：filteredX = (newX * 0.7) + (prevX * 0.3)
  • 設定 MIN_PIXELS 面積門檻，過濾散點雜訊
  • 在 README 末尾附上截圖，對比加入濾波前後質心軌跡的差異

評分重點：測試將送入含雜訊影像，驗證質心是否能鎖定主要目標而不被雜訊拉偏。

任務 3：質心誤差與轉向角度映射

目標：在 js/servo_mapper.js 中將質心位置轉換為舵機控制角度。

需求：

  • 計算質心與畫面中心的誤差值：error = centroidX - 160
  • 將誤差線性映射至轉向角度區間 $[60°, 120°]$
  • 必須實作數值限幅 (Clipping)，確保輸出角度永遠不超出安全範圍

評分重點：給定位於畫面最左側的質心，輸出角度必須精確等於 $60°$。

  • 📂 tests/centroid_precision.test.js：自動化評分脚本。餵入標準幾何圖形（如：正方形、長方形），檢核學員計算出的質心座標 $( \bar{x}, \bar{y} )$ 是否精確命中幾何中心。

2. 作業任務部署細節

任務 1：「質心追蹤器」：從像素到座標 (Build Centroid Tracker)

  • 目標：透過加權平均法 (Weighted Average) 實現空間維度的降維提取。
  • Classroom 部署建議：
    • 核心代碼檢核： 
      // 學生需實作：累加與除法
for (let y = 0; y < height; y++) {
    for (let x = 0; x < width; x++) {
        if (isBlack(x, y)) { sumX += x; sumY += y; count++; }
    }
}
if (count > 0) { centroidX = sumX / count; }
    • Autograding：餵入一個完全空白的影像，檢核學員是否實作了「除以零保護」。這能有效區分「只會寫公式」與「具備防禦性編程意識」的學生。

任務 2：「質心穩定化」過濾器 (Centroid Stabilization)

  • 目標：引入數位濾波概念，解決物理感測器常見的「數據抖動 (Jitter)」問題。
  • Classroom 部署建議：
    • 濾波演算法檢核：要求學員實作加權移動平均： filteredX = (newX * 0.7) + (prevX * 0.3)。
    • 面積門檻測試：設定 MIN_PIXELS 過濾器。餵入帶有少量散點雜訊的影像，檢核質心準星是否能保持在主體目標上而不被雜訊「拉走」。
    • 實驗紀錄：學員需在 README 中對比加入濾波前後，質心座標隨時間變化的波形曲線（截圖）。

任務 3：質心誤差與轉向控制連動 (Centroid-to-Control Linkage)

  • 目標：完成視覺回饋控制 (Visual Feedback Control) 的最後一哩路。
  • Classroom 部署建議：
    • 地圖映射邏輯： 
      // 學生需實作：線性映射
let error = centroidX - 160; 
let steerAngle = map(error, -160, 160, 60, 120);
    • Autograding：給定一個位於畫面最左側的質心，檢核輸出的轉向角是否精確等於 $60^\circ$。
    • Tutor View (PR 審核)：導師点評其對「數值限幅 (Clipping)」的處理。若算出的角度超過 $[60, 120]$，代碼是否能自動截斷，以防止舵機過載燒毀？

3. 演算法導師點評標準 (Algorithm Benchmarks)
