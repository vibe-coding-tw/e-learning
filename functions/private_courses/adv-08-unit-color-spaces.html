🌈 1. RGB：光的加法與顯示器的邏輯 (20 min)

RGB (Red, Green, Blue) 是最直覺的色彩空間，直接對應顯示器硬體。然而，它最大的限制是「亮度與顏色耦合」，這對電腦視覺識別來說非常致命。

Coupling Limitation

  • 環境變數干擾：當燈光變暗，R、G、B 數值會同時等比例下降。
  • 硬體依賴：不同攝像頭感測到的 RGB 原色分布可能完全不同。

🎞️ 3. YUV/YCbCr：影像傳輸與壓縮

YUV 將影像拆分為亮度 (Y) 與色度 (UV)。由於人類視覺對明暗極其敏感，但對顏色細節相對遲鈍，這成為 JPEG 壓縮與 ESP32 攝像頭輸出的核心協議。

Compression Logistics

  • Chroma Subsampling：捨棄部分 UV 數據（如 4:2:2），節省 33% 頻寬而不失真感官體驗。
  • Y-Channel Focus：利用亮度頻道進行高效率的邊緣檢測 (Edge Detection)。
  • RGB565 Optimization：理解 16-bit 像素如何在 MCU 記憶體中節省每一分 Byte。

Adv-08: 顏色空間與機器視覺 (Color Spaces & Computer Vision)

歡迎來到 adv-08-unit-color-spaces 單元！這是從「看到圖片」進步到「分析光譜數據」的關鍵階段。理解 RGB、HSV 與 YUV 的數學本質，是打造具備環境魯棒性 (Robustness) 的自駕系統感知核心。

🎯 學習目標

本專案旨在透過實作，讓你深入理解並應用不同的顏色空間：

  1. 色彩獵人 (HSV Real-time Filtering Lab)：實作即時 HSV 過濾器，體悟 HSV 在光影變化下相對於 RGB 的穩定性優勢。
  2. 記憶體瘦身計畫 (Memory Optimization Lab)：實作 RGB565 編碼，掌握嵌入式系統中資源受限下的數據壓縮思維。
  3. YUV 亮度自適應 (Auto-Exposure Leveling)：利用 YUV 的亮色分離特性，實作自動補償演算法。

🚀 作業任務 (Assignments)

  • 繳交要求：完成程式碼後，請在本 README 下方提供截圖，展示如何透過一組參數，同時鎖定明亮區與陰影區的紅球特徵。

2. 作業任務部署細節

任務 1：「色彩獵人」：實作即時 HSV 過濾器 (HSV Real-time Filtering Lab)

  • 目標：體悟 HSV 在光影變化下相對於 RGB 的穩定性優勢。
  • Classroom 部署建議：
    • 過濾邏輯檢核： 
      // 學生需實作：HSV 範圍判定邏輯
const isMatch = (H >= minH && H <= maxH) && (S >= minS && S <= maxS) && (V >= minV && V <= maxV);
pixel = isMatch ? 255 : 0; // 二值化：目標為白，背景為黑
    • Autograding：給定一個「陰影下的紅像素」，檢核學員的過濾範圍是否能正確將其識別為 Target。
    • 截圖提交：學員需在 README 中展示如何透過一組參數，同時鎖定明亮區與陰影區的紅球特徵。

任務 2：RGB565 記憶體瘦身計畫 (Memory Optimization Lab)

  • 目標：掌握嵌入式系統中資源受限下的數據壓縮思維。
  • Classroom 部署建議：
    • 位元運算檢核： 
      // 學生需實作：8-8-8 轉 5-6-5
uint16_t rgb565 = ((r & 0xF8) << 8) | ((g & 0xFC) << 3) | (b >> 3);
    • 優化報告：學員需在 README.md 中計算：在 30FPS 的 QVGA 影像流中，從 24-bit 降至 16-bit 每秒能節省多少 MB 的頻寬。
    • PR 深度核核：點評其對「位元運算 (Bitwise)」的理解度。是否理解為什麼 G 頻道分配了 6 個位元？（人眼對綠色解析度最高）。

任務 3：YUV 亮度自適應 (Auto-Exposure Leveling)

  • 目標：利用 YUV 的亮色分離特性，實作自動補償演算法。
  • Classroom 部署建議：
    • 自適應邏輯檢核：提取 Y 頻道，計算全局平均亮度。若過低，則自動按比例調高 V-Value 或 Y-Value 的輸出。
    • 魯棒性測試：學員需錄製一段畫面。展示當環境突然變暗時，原有的追蹤輪廓是否能透過「自適應補償」維持穩定，而不發生丟失。
    • 溢位檢查：檢核代碼中是否正確處理了補償後的數值上限（必須 Math.min(val, 255) 防止報錯）。
