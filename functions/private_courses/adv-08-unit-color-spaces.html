RGB：光的加法與顯示器的邏輯

RGB（Red, Green, Blue）是數位影像中最基礎也最普遍的色彩表示法。每一個像素由紅、綠、藍三個通道組成，每個通道以 8 位元（0–255）儲存強度值。三色混合後產生人眼可見的色彩——這正是「加法混色」的原理。所有 LCD 螢幕、攝影機感測器與影像檔案格式都以 RGB 為基礎架構。

加法混色原理

與繪畫顏料的「減法混色」不同，光源遵循加法混色：紅光 + 綠光 = 黃色；紅光 + 藍光 = 洋紅色；三色全開 = 白色；三色全關 = 黑色。每個像素佔用 3 Bytes（24 bits），因此一張 320×240 的影像需要 320 × 240 × 3 = 230,400 Bytes ≈ 225 KB 的原始記憶體。

備註 在 ESP32-CAM 模組中，受限於 PSRAM 容量（通常 4 MB），實際運作時多使用 RGB565（16-bit）或 YUV422 格式以節省記憶體，而非完整的 RGB888。

RGB 的耦合問題

RGB 最大的限制在於亮度與色彩高度耦合。當環境光線改變時，三個通道的數值會同時變化。例如，一顆紅色球在強光下可能是 (220, 30, 25)，但在陰影中變成 (90, 12, 10)。若以固定閾值過濾「紅色」，系統將在光線變化時完全失效。

光照條件紅球 R 值紅球 G 值紅球 B 值辨識結果
強光直射2203025 成功
室內日光燈1802218 成功

像素的位元結構

理解像素在記憶體中的排列方式，對嵌入式系統開發至關重要。以下是一個像素值 (200, 100, 50) 在 RGB888 格式下的二進位表示：

// RGB888 像素結構（每通道 8 bits，共 24 bits）
R: 11001000 // 200
G: 01100100 // 100
B: 00110010 // 50

// 在 frame buffer 中的排列（Big Endian）
Memory: [0xC8][0x64][0x32] → 1 pixel = 3 bytes
重要 不同硬體平台的通道排列順序可能不同。OpenCV 使用 BGR 順序，而 HTML Canvas 與大多數攝影機感測器使用 RGB 順序。混淆排列順序是嵌入式視覺開發中最常見的 Bug 來源之一。

為什麼需要其他色彩空間？

RGB 雖然直覺，但其「三維耦合」特性使得色彩過濾極為困難。在電腦視覺應用中，我們需要一種能夠將色相（Hue）從亮度中解耦的表示法，才能在不同光照條件下穩定辨識物體顏色。這正是下一單元 HSV 色彩空間要解決的核心問題。

提示 在後續的實作作業中，你將親身體驗：同樣的紅球追蹤任務，在 RGB 空間需要針對每種光照條件設定不同閾值，而在 HSV 空間只需一組參數即可涵蓋所有情境。

HSV/HSL：色彩處理的利器

HSV（Hue, Saturation, Value）色彩空間將顏色資訊拆解為三個獨立維度：色相（顏色種類）、飽和度（顏色純度）與明度（亮暗程度）。這種圓柱座標系的設計，完美解決了 RGB 中亮度與色彩耦合的問題，使得色彩過濾在不同光照條件下依然穩定可靠。

HSV 三通道解析

通道物理意義數值範圍OpenCV 範圍
H（Hue）色相角度，決定「是什麼顏色」0°–360°0–179
S（Saturation）飽和度，顏色的鮮豔程度0%–100%0–255
V（Value）明度，顏色的明暗程度0%–100%0–255
重要 OpenCV 中 H 通道的範圍是 0–179（而非 0–360），因為使用 8-bit 無法儲存超過 255 的值，所以角度被除以 2。在撰寫過濾邏輯時必須注意此差異，否則過濾範圍會完全錯誤。

RGB 到 HSV 的轉換公式

轉換的核心步驟為：先找出 RGB 中的最大值與最小值，計算差值 (Delta)，再依據哪個通道為最大值來決定 H 的象限。轉換公式如下：

// RGB → HSV 轉換核心邏輯
let max = Math.max(r, g, b);
let min = Math.min(r, g, b);
let delta = max - min;

// V (明度) = 最大通道值 / 255
let V = max / 255;

// S (飽和度) = delta / max
let S = (max === 0) ? 0 : delta / max;

// H (色相) 依據最大通道決定象限
if (delta === 0) H = 0;
else if (max === r) H = 60 * ((g - b) / delta % 6);
else if (max === g) H = 60 * ((b - r) / delta + 2);
else H = 60 * ((r - g) / delta + 4);

HSV 在視覺追蹤中的優勢

在追蹤任務中，我們最關心的是「目標物是什麼顏色」。HSV 的 H 通道恰好只回答這個問題，不受光照影響。只要設定合理的 S 與 V 下限（排除過暗或過灰的區域），即可在各種環境下穩定鎖定目標。

實務技巧 紅色在 HSV 色環中位於 0° 附近，但因為是圓形分布，紅色會「跨越」0°/360° 的邊界。過濾紅色時需使用兩段範圍：H < 10 OR H > 170（OpenCV 尺度）。這是初學者最常犯的錯誤。

HSV 與 HSL 的差異

HSL（Hue, Saturation, Lightness）與 HSV 類似，但 L 通道的計算方式不同：L = (max + min) / 2。HSL 中純白色的 L = 1、S = 0，而 HSV 中純白色 V = 1、S = 0。在電腦視覺領域，HSV 遠比 HSL 常用，因為 V 通道更直覺地反映了感測器接收到的光強度。

備註 OpenCV 的 cv2.cvtColor(img, cv2.COLOR_BGR2HSV) 是最常用的轉換函式。在 JavaScript 環境中（如 ESP32 的 Web 控制介面），你需要手動實作上述轉換邏輯。

YUV/YCbCr：影像傳輸與壓縮

YUV 色彩空間源自類比電視時代，其核心設計理念是利用人類視覺系統的特性：人眼對亮度的敏感度遠高於對色彩的敏感度。因此，YUV 將影像分離為亮度通道（Y）與兩個色差通道（U/Cb 與 V/Cr），允許對色差通道進行更激進的壓縮而不被察覺。

YUV 通道定義

通道全名物理意義應用價值
YLuminance（亮度）灰階影像，描述明暗邊緣檢測、灰階處理
U (Cb)Blue-difference Chrominance藍色色差色彩還原
V (Cr)Red-difference Chrominance紅色色差色彩還原

RGB 與 YUV 的轉換公式

BT.601 標準定義的轉換矩陣如下（適用於 SD 視訊）：

// RGB → YUV (BT.601 標準)
Y = 0.299 × R + 0.587 × G + 0.114 × B
U = -0.169 × R - 0.331 × G + 0.500 × B + 128
V = 0.500 × R - 0.419 × G - 0.081 × B + 128

// 注意：綠色對 Y 通道的貢獻最大 (0.587)
// 因為人眼對綠色光最為敏感

色度次取樣（Chroma Subsampling）

由於人眼對色差資訊的空間解析度需求較低，我們可以用較少的取樣點來記錄 U 和 V 通道。常見的次取樣格式如下：

  • 4:4:4 — 無壓縮，每個像素都有完整的 YUV 資料。
  • 4:2:2 — 水平方向每 2 個像素共享一組 UV，節省 33% 頻寬。
  • 4:2:0 — 水平與垂直方向各每 2 個像素共享 UV，節省 50% 頻寬。JPEG 與 H.264 的預設格式。
備註 ESP32-CAM 的 OV2640 感測器原生輸出格式為 YUV422。若需要 RGB 資料進行色彩分析，必須執行格式轉換，這會額外消耗 CPU 時間。在設計系統架構時，應考慮是否能直接在 YUV 空間完成所需處理。

RGB565：嵌入式系統的折衷方案

RGB565 是另一種節省記憶體的策略，使用 16 bits 取代 24 bits 儲存一個像素：紅色 5 bits、綠色 6 bits、藍色 5 bits。綠色分配較多位元是因為人眼對綠色最敏感。此格式在 QVGA（320×240）解析度下，每幀僅需 150 KB，相比 RGB888 的 225 KB 節省了 33%。

// RGB888 → RGB565 位元運算
uint16_t rgb565 =
((r & 0xF8) << 8) | // 取 R 的高 5 位，左移至 [15:11]
((g & 0xFC) << 3) | // 取 G 的高 6 位，左移至 [10:5]
((b >> 3)); // 取 B 的高 5 位，置於 [4:0]
注意 RGB565 會產生可見的色彩斷層（Banding），尤其在漸層色區域。對於需要精確色彩判斷的視覺追蹤任務，建議在 HSV 轉換前先還原為完整的 RGB888 精度。

  • 📂 tests/bit_logic.test.js：自動化評分脚本。餵入特定 RGB 數值，檢核產出的 RGB565 (16-bit) 與 HSV 數值是否符合數學預期。

2. 作業任務部署細節

任務 1：「色彩獵人」：實作即時 HSV 過濾器 (HSV Real-time Filtering Lab)

  • 目標：體悟 HSV 在光影變化下相對於 RGB 的穩定性優勢。
  • Classroom 部署建議：
    • 過濾邏輯檢核： 
      // 學生需實作：HSV 範圍判定邏輯
const isMatch = (H >= minH && H <= maxH) && (S >= minS && S <= maxS) && (V >= minV && V <= maxV);
pixel = isMatch ? 255 : 0; // 二值化：目標為白，背景為黑
    • Autograding：給定一個「陰影下的紅像素」，檢核學員的過濾範圍是否能正確將其識別為 Target。
    • 截圖提交：學員需在 README 中展示如何透過一組參數，同時鎖定明亮區與陰影區的紅球特徵。

任務 2：RGB565 記憶體瘦身計畫 (Memory Optimization Lab)

  • 目標：掌握嵌入式系統中資源受限下的數據壓縮思維。
  • Classroom 部署建議：
    • 位元運算檢核： 
      // 學生需實作：8-8-8 轉 5-6-5
uint16_t rgb565 = ((r & 0xF8) << 8) | ((g & 0xFC) << 3) | (b >> 3);
    • 優化報告：學員需在 README.md 中計算：在 30FPS 的 QVGA 影像流中，從 24-bit 降至 16-bit 每秒能節省多少 MB 的頻寬。
    • PR 深度核核：點評其對「位元運算 (Bitwise)」的理解度。是否理解為什麼 G 頻道分配了 6 個位元？（人眼對綠色解析度最高）。

任務 3：YUV 亮度自適應 (Auto-Exposure Leveling)

  • 目標：利用 YUV 的亮色分離特性，實作自動補償演算法。
  • Classroom 部署建議：
    • 自適應邏輯檢核：提取 Y 頻道，計算全局平均亮度。若過低，則自動按比例調高 V-Value 或 Y-Value 的輸出。
    • 魯棒性測試：學員需錄製一段畫面。展示當環境突然變暗時，原有的追蹤輪廓是否能透過「自適應補償」維持穩定，而不發生丟失。
    • 溢位檢查：檢核代碼中是否正確處理了補償後的數值上限（必須 Math.min(val, 255) 防止報錯）。
