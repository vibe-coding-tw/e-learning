基礎 1 - 核心架構

ESP32 架構與開發板導覽

深入了解你的數位大腦：雙核、Wi-Fi、藍牙與豐富外設。這不僅僅是一塊電路板，這是你通往物聯網世界的鑰匙。

⚡ 為什麼選擇 ESP32？ (15 min)

相較於傳統 Micro-controllers，ESP32 是一個質的飛躍。

🧠

雙核心 (Dual-Core)

高達 240MHz 的運算速度。雙核架構允許一個核心處理連網通訊，另一個核心專注執行控制邏輯，效率加倍。

📡

無線連線能力

內建 Wi-Fi 與 雙模藍牙 (BLE + BT)。這是它成為物聯網神器與 Vibe Coding 平台核心的關鍵。

🔌

豐富的外設

內建觸控感應、霍爾傳感器、ADC/DAC 以及多組序列接口。這讓它能輕鬆驅動馬達、讀取感測器。

🔬 開發板外觀大解剖 (20 min)

ESP-WROOM-32
CE FCC
1

ESP-WROOM-32 模組

金屬屏蔽罩下藏著 CPU、SRAM 與 Flash，負責保護精密晶片不受電磁干擾。

2

CP2102/CH340 橋接晶片

數位翻譯官，負責將 USB 訊號轉換為 UART 序列訊號與電腦通訊。

3

LDO 電壓調節器

將 USB 的 5V 穩定降至 3.3V。請注意，過高的電壓會直接燒毀核心。

4

EN 與 BOOT 系統按鈕

EN 為重啟鈕；BOOT 為模式鈕。按住 BOOT 再上傳代碼是解決燒錄失敗的常見技巧。

💻 軟體環境預熱 (25 min)

在開始編碼前，我們必須確保電腦能夠識別這個「外星裝置」。

1. 安裝專業 IDE

強烈推薦 VS Code + PlatformIO。它提供自動補全、函式跳轉與更強大的專案管理能力。

2. 識別通訊埠 (COM Port)

確保安裝了 CP210x 或 CH340 驅動程式。沒有它，你的電腦就只是在對著一塊磚頭說話。

新手防狼手冊

  • 拒絕純充電線： 必須使用具備「數據傳輸」能力的線材。

  • 物理保護： 插拔時請捏住接頭，避免拉斷脆弱的 USB 焊點。

💡 Tip: 如果不亮燈，請先檢查電源指示燈 LED。

核心挑戰：對話的第一步

驗證你的開發環境，並與晶片建立初步連結。

💡 45 min

【作業 1-1】

「摩斯密碼閃爍」

成功安裝環境並燒錄 Blink 程式。進階挑戰：修改 delay 邏輯讓 LED 閃出「SOS」摩斯密碼訊號。

🕵️ 60 min

【作業 1-2】

「晶片身分證偵查」

撰寫 API 查詢代碼，從 Serial Monitor 提取晶片 Cores、Flash 與 MAC Address，建立對資源邊界的認知。

🌡️ 45 min

【作業 1-3】

「啟動日誌診斷」

分析開機第一秒的 Bootloader 日誌。執行高強度運算並監測晶片物理溫升，理解軟硬體之間的能量關聯。

ESP32 SoC 生態實驗室

任務 1：硬體連通性與燒錄測試 (Hardware Connectivity & Flashing)

角色扮演：韌體工程師 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 環境建立：完成 VS Code + PlatformIO 安裝，建立新專案並通過編譯。
  2. 物理閃爍：實作一個 LED 閃爍程式。進階任務：修改 delay() 邏輯，讓板載 LED 閃爍出「SOS」摩斯密碼 (三短、三長、三短)。
  3. 驗證提交：在 README.md 貼上 PlatformIO 上傳成功 (SUCCESS) 的終端機內容與 LED 閃爍照片。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 引腳配置正確性：是否能正確找出板載 LED 的 GPIO 引腳（通常在 GPIO 2）？
  • [ ] 燒錄時機掌握：是否能在燒錄超時前，正確操作 BOOT 鈕進入燒錄模式？

任務 2：串口通訊的偵探遊戲 (Serial Communication Detective)

角色扮演：系統分析師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 系統 API 調用：引入 esp_system.h，調用 esp_chip_info() 獲取晶片核心數與修訂版本。
  2. 數位指紋提取：透過 ESP.getEfuseMac() 打印晶片的唯一 MAC Address。
  3. 觀察亂碼特徵：故意調低 Serial Monitor 的 Baud Rate，截圖記錄並描述數據崩潰的視覺型態，分析「失步」原因。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 數據獲取完整性：README 中是否包含明確的晶片核心數與 Flash 容量報表？
  • [ ] 同步原理理解：能否正確解釋為什麼「發送端 115200 / 接收端 9600」會導致無法閱讀的亂碼？

任務 3：電源與發熱量監測 (Power & Thermal Monitoring)

角色扮演：硬體測試專家 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 冷啟動日誌分析：監看按下 EN 鍵後第一毫秒產生的 Bootloader 日誌，識別出當前的 Flash Mode (DIO/QIO) 參數。
  2. 負載溫升感測：執行包含 WiFi 掃描 (WiFi.scanNetworks()) 的死循環程式，用手指輕壓金屬屏蔽罩，記錄運作前後的溫差感受。
  3. 元件佈局圖解：在 Repo 中上傳一張開發板照片，並用圈圖工具標示出 LDO 降壓元件與 RESET/BOOT 按鈕的物理位置。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 日誌解讀深度：能否從開機日誌中識別出 CPU 啟動頻率與 Flash 加載訊息？
  • [ ] 供電安全意識：是否理解 Brownout Detector 觸發的原因通常與 USB 供電電流不足有關？

GitHub Classroom 詳細操作指南

本單元教學重點：讓學生與晶片進行「第一次親密接觸」，建立抽象代碼與物理硬體間的感知。

  1. 第一步：基礎燒錄 - 確保所有學員能在 30 分鐘內解決驅動、線材與 COM 埠識別問題。
  2. 第二步：資源邊界探索 - 通過讀取 Chip Info，讓學生理解即便外觀相似，每顆 SoC 的核心與 Flash 可能大不相同。
  3. 第三步：供電與穩定性 - 藉由 Bootloader 日誌與溫升實驗，埋下未來除錯「穩定性」問題的種子。
