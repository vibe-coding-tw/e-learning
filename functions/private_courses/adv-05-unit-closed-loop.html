視覺閉迴路控制理論

🔗 1. 閉迴路控制的核心架構 (25 min)

開迴路 (Open-Loop)

只給指令（如：前進 5 秒），不看結果。容易受地面摩擦力、電池電量影響而偏離。

閉迴路 (Closed-Loop)

不斷觀察結果（影像位置），並根據結果調整指令，具備自動補償能力。

🏗️ 控制鏈元件 (Control Chain)

  • 控制器 (Controller)：網頁中的控制邏輯 (P-Control)。
  • 執行器 (Actuator)：ESP32 與馬達。
  • 感測器 (Sensor)：攝影機 (Camera)。
  • 回饋 (Feedback)：即時計算出的質心誤差。

⏱️ 2. 系統延遲 (Latency) 的影響 (20 min)

視覺延遲：影像傳輸、解析 JSON、計算質心都需要時間。

連鎖後果

如果延遲太高，車子修正的是「過去的誤差」，這會導致嚴重的震盪甚至失控（Over-correction）。就像在延遲嚴重的視訊通話中，兩個人總是同時說話又同時停下。

  • ✓ 降低影像解析度 (減少傳輸量)
  • ✓ 提升藍牙傳輸 MTU (增加吞吐量)
  • ✓ 使用跳行掃描演算法 (減少運算量)

🎯 3. 設定點 (Set-point) 與偏差 (Deviation) (15 min)

  • ● 設定點 (Set-point)：我們希望物體出現在影像中的目標位置（通常是中心 $x = 160$）。
  • ● 偏差 (Deviation)：當前偵測位置與設定點之間的動態距離。
  • 💡 預習任務：為什麼閉迴路系統可以「自動容忍」馬達左右出力不均的問題？

PD 控制的物理意義

Kp (比例) 決定目前的轉向強度。Kp 越大，修正越果斷。
Kd (微分) 預測誤差變化趨勢，抑制過度震盪（就像是阻尼）。
線上任務：觀察「追蹤震盪」案例影片，分析為何 $K_p$ 過大會導致車輛在線上蛇行。

進階課程 5 - 單元 3 視覺閉迴路控制理論

實驗五：視覺閉迴路控制實作 (Closed-Loop Control Theory)

本單元將學習如何將「視覺感知」、「數學量化」與「物理執行」串聯，建立一個完整的自動化閉迴路系統。你將實作 P (Proportional) 控制器，並透過實驗理解系統穩定性與參數調校的關係。

🎯 學習目標

  1. 實作比例控制 (P-Control) 與輸出限幅 (Saturation)。
  2. 開發 Failsafe 機制，確保影像遺失時的硬體安全。
  3. 透過「階躍響應」實驗量化控制性能，學會參數調校的工程方法。

🛠️ 作業任務

任務 1：實作 P 控制邏輯

請修改 src/pid_logic.js：

  • 在 compute(error) 方法中實作：output = error * Kp。
  • 實作 Saturation (飽和限制)：確保輸出數值保持在 minOutput 與 maxOutput 之間，防止馬達指令超出硬體極限。

任務 2：實作 Safety Guard (Failsafe)

請修改 src/safety_guard.js：

  • 在 update(targetDetected) 方法中實作判定邏輯。
  • 如果連續遺失目標的幀數 (framesSinceLastTarget) 超過 timeoutFrames，則必須觸發 triggerFailsafe()。
  • 此機制是為了防止在視覺盲區時車輛仍持續「盲衝」。

任務 3：參數調校與實驗紀錄

進行實際測試後，請在 docs/tuning_log.md 紀錄不同 $K_p$ 參數下的系統表現：

  • 紀錄系統的 上升時間 (Rise Time) 與 過衝 (Overshoot)。
  • 找出讓系統產生持續震盪（蛇行）的臨界 $K_p$ 值。
  • 回答文件中的分析問題。

📁 程式庫結構

  • src/pid_logic.js: 核心控制器邏輯 (需實作)。
  • src/safety_guard.js: 安全保護模組 (需實作)。
  • docs/tuning_log.md: 實驗數據紀錄表。
  • tests/: 自動化評分測試腳本。

🚀 提交要求

  1. 確保所有 npm test 測試項目通過。
  2. 完成 docs/tuning_log.md 中的實驗數據紀錄與問題回答。
  3. 將代碼 push 回此程式庫即可完成繳交。

[!TIP] 當 $K_p$ 調得過大時，系統會產生劇烈震盪；調得太小時，反應會變得遲鈍。嘗試尋找兩者之間的動態平衡！

閉迴路系統需要精密的「時序控制」與「狀態管理」，建議範本包含：

  • 📂 src/safety_failsafe.js：安全模組，提供幀率計數器，留下「視覺遺失判定與自動停車」邏輯。

  • 📂 tests/loop_stability.test.js：自動化評分脚本。模擬持續性的偏差輸入，檢核控制輸出是否能保持線性且符合安全限幅 (Clipping)。

2. 作業任務部署細節

任務 1：視覺追蹤閉迴路整合實作 (Vision Feedback Integration)

  • 目標：打通數據鏈路，實現首個「感知－決策－執行」的自動化閉環。
  • Classroom 部署建議：
    • 核心代碼檢核： 
      // 學生需實作：將誤差轉化為馬達脈衝輸出
let output = error * Kp; 
motor.sendPayload({ steer: output });
    • 整合指標：要求學員確保 Bluetooth 的指令發送頻率不超過影像 FPS，防止「指令堆疊」造成的控制滯後。
    • 驗證方式：學員需提交一段影片，展示車輛能隨著目標物（如：手中的亮色物體）移動而自動擺頭或轉向。

任務 2：階躍響應 (Step Response) 紀錄實驗

  • 目標：科學量化控制性能，理解「動態平衡」的權衡。
  • Classroom 部署建議：
    • 實驗要求：在標定的環境下，瞬間移走目標物產生的「階躍偏差」。
    • 性能紀錄：學員需在 README 中填寫：
      • 上升時間 (Rise Time)：從 0 追到 90% 偏差所需的時間。
      • 過衝 (Overshoot)：修正是否過頭？
    • 調校任務：試著將 $K_p$ 調大直到系統產生「蛇行震盪」，並紀錄這個臨界值。這能讓學員深刻理解系統不穩定性的物理來源。

任務 3：斷點與恢復：Failsafe 機制測試

  • 目標：培養工業級的安全性開發習慣，防止「視覺盲衝」。
  • Classroom 部署建議：
    • 失效邏輯檢核： 
      // 學生需實作：超時判定
if (framesSinceLastTarget > 10) {
    sendEmergencyStop();
}
    • Autograding：透過腳本模擬影像串流突然中斷，檢核學員的代碼是否能在 500ms 內發出「停止指令」。
    • 恢復測試：模擬目標重新出現，檢查系統是否能自動恢復追蹤，而不需要重整網頁。

3. 控制工程導師點評標準 (Control System Benchmarks)

此單元的價值在於 「對動態穩定性的直覺與量化分析」：

  • [ ] 時延控制意識：是否理解「傳輸延遲」是造成震盪的主因？在代碼中是否有減少非必要數據上傳的優化？
  • [ ] 代碼完整性：控制器邏輯是否處理了「飽和 (Saturation)」問題？（即：計算出的轉角不能超過硬體極限）。
  • [ ] 故障安全思維：README 是否討論了「失去視覺」時的最優策略？（停止 vs. 慢速原地旋轉搜尋）。
