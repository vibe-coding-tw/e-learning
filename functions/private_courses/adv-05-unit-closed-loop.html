閉迴路控制的核心架構

在自動化控制中，系統可以分為兩大類：開迴路 (Open-Loop) 與閉迴路 (Closed-Loop)。 兩者的根本差異在於——系統是否會「看」自己的執行結果並據此修正。開迴路就像閉著眼睛走路， 只靠預設指令前進；閉迴路則像睜著眼睛走路，每一步都根據所見調整方向。

開迴路 vs. 閉迴路

特性 開迴路 (Open-Loop) 閉迴路 (Closed-Loop)
回饋機制 無。執行預設指令後不檢查結果。 有。持續監測輸出並動態調整。
典型範例 「前進 5 秒後停止」 「持續追蹤黑線直到對齊中心」
抗干擾能力 差。地面摩擦力、電池電量變化直接導致偏離。 強。能自動補償外界干擾。
複雜度 低。無需感測器與運算。 較高。需要感測器、計算邏輯與即時通訊。

控制迴路的四大元件

一個完整的視覺閉迴路系統由四個關鍵元件組成，數據在這四個節點之間不斷循環流動：

1

感測器 (Sensor) — 攝影機

擷取環境影像，將物理世界的光學資訊轉換為數位像素矩陣。這是閉迴路的「眼睛」。

2

控制器 (Controller) — 網頁端 P-Control 邏輯

接收誤差訊號，透過比例控制公式 output = Kp × error 計算出修正量。這是閉迴路的「大腦」。

3

執行器 (Actuator) — ESP32 與馬達

接收控制指令（透過 BLE），驅動馬達產生物理運動。這是閉迴路的「手腳」。

4

回饋 (Feedback) — 質心誤差訊號

車輛移動後，攝影機重新擷取影像並計算新的質心誤差，形成完整的回饋迴路。

備註 「迴路」一詞的精髓在於：數據永不停止地循環。每一幀影像產生一個新的誤差值， 控制器據此發出新的指令，馬達執行後改變車輛位置，攝影機再次觀測——這個循環以每秒數十次的頻率運作， 直到系統被關閉或目標消失。

比例控制 (P-Control) 的基本概念

在本課程中，我們使用最基本也最直觀的控制策略——比例控制 (Proportional Control)。 核心公式只有一行：

// P 控制器核心公式
let output = Kp * error;

// Kp = 比例增益 (Proportional Gain)
// error = 正規化後的視覺誤差 [-1.0, 1.0]
// output = 送往馬達的轉向指令

Kp 的大小決定了系統的「反應靈敏度」：Kp 越大，同樣的誤差會產生越強的修正力道； 但若 Kp 過大，系統會過度修正，導致車輛左右蛇行震盪。 如何找到最佳的 Kp 值，就是控制工程中參數調校 (Tuning) 的核心任務。

重要 控制器的輸出必須加入飽和限制 (Saturation)：確保計算值不超過馬達的物理極限。 例如馬達的 PWM 範圍為 0–255，則輸出必須被限制（Clamp）在此範圍內，否則可能損壞硬體。

系統延遲 (Latency) 的影響

閉迴路控制的效能取決於「迴路轉一圈要多久」。從攝影機擷取影像、傳輸數據、計算質心與誤差、 發送 BLE 指令到馬達執行——每一個環節都會引入延遲 (Latency)。 當總延遲過高時，控制器修正的是「過去的誤差」而非「當前的誤差」， 這會導致系統產生嚴重的震盪 (Oscillation) 甚至完全失控。

這就像在一場嚴重延遲的視訊通話中，兩個人總是同時說話又同時停下——因為雙方看到的都是對方幾秒前的狀態， 無法根據當前狀態做出正確回應。對自動駕駛而言，延遲的後果更為嚴重：車輛可能在轉彎時「遲到修正」而衝出賽道。

延遲預算 (Latency Budget)

一個典型的視覺閉迴路系統，其延遲可以拆解為以下幾個階段：

階段 來源 典型延遲 優化策略
影像擷取 攝影機曝光 + 編碼 30–100 ms 降低解析度（如 160×120）、使用 JPEG 壓縮
數據傳輸 Wi-Fi / BLE 無線傳輸 20–80 ms 提高 MTU、減少封包大小、使用二進位協定
影像處理 二值化 + 質心計算 5–20 ms 跳行掃描、ROI 裁切、避免全幀遍歷
控制計算 P-Control 運算 < 1 ms 運算量極小，通常不是瓶頸
指令傳輸 BLE Write 到 ESP32 15–40 ms 確保指令頻率不超過影像 FPS
關鍵 總迴路延遲應控制在 200 ms 以內（即控制頻率 ≥ 5 Hz）。 當延遲超過 300 ms 時，比例控制器的修正幾乎必然過時，系統將進入「修正→過衝→反向修正→再過衝」的震盪循環。

延遲導致過度修正的機制

假設控制迴路延遲為 200 ms，而車輛正以穩定速度行駛。當車偏向右邊時：

  1. 攝影機在 T=0 ms 捕捉到偏右影像
  2. 經過 200 ms 的處理與傳輸，馬達在 T=200 ms 收到「左轉」指令
  3. 但在這 200 ms 內，車輛可能已經因慣性繼續偏右甚至已經自然回正
  4. 此時「左轉」指令反而造成了過度修正 (Over-correction)
// 延遲越高，Kp 必須越小，否則過度修正
// 典型經驗法則：
// 延遲 < 100ms → Kp 可達 1.0–2.0
// 延遲 100–200ms → Kp 建議 0.3–0.8
// 延遲 > 300ms → Kp 須低於 0.3，否則必然震盪

降低延遲的實務策略

  • 降低影像解析度：從 320×240 降至 160×120，傳輸量減少 75%，處理速度提升 4 倍。
  • 跳行掃描 (Row Skipping)：只掃描影像的偶數行（或特定 ROI 區域），運算量直接減半。
  • 提升 BLE MTU：將藍牙最大傳輸單元從預設的 23 byte 提升至 247 byte，減少分包次數。
  • 指令頻率限制：確保 BLE 指令發送頻率不超過影像 FPS，防止「指令堆疊」造成控制滯後。
提示 在作業中，你可以透過在控制迴路中插入 console.log(Date.now()) 來測量實際的迴路延遲。 記錄「影像抵達時間」與「BLE 指令送出時間」的差值，就是你系統的端到端延遲。

設定點 (Set-point) 與偏差 (Deviation)

在控制理論中，設定點 (Set-point) 是系統的「目標狀態」——我們希望被控變量達到的理想值。 對循跡車而言，設定點就是「目標線出現在影像正中央」，即質心 X 座標等於影像中心（例如 x = 160）。 偏差 (Deviation) 則是當前偵測值與設定點之間的動態距離——也就是我們在前一個模組中學到的「正規化誤差」。

閉迴路系統的核心目標只有一個：持續將偏差驅動為零。 當偏差為正，控制器輸出右轉修正；當偏差為負，輸出左轉修正；當偏差趨近零，修正力道自然消失。 這個自動調節過程不需要預先知道路徑形狀，也能自動補償馬達出力不均、地面摩擦力差異等外部干擾—— 這就是閉迴路控制的強大之處。

P 控制器的參數調校

比例增益 Kp 是 P 控制器唯一需要調校的參數，但它的影響極為深遠。 以下表格展示了不同 Kp 值對系統行為的影響：

Kp 範圍 系統表現 物理現象
過小（< 0.2） 修正力道不足，偏差無法歸零 車輛持續偏離賽道中心，過彎時容易衝出
適中（0.3–0.8） 快速收斂，輕微過衝後穩定 車輛平穩追蹤賽道，過彎流暢
過大（> 1.5） 嚴重過衝，持續震盪 車輛劇烈蛇行，如同醉酒駕駛
臨界值 等幅震盪，永不收斂 車輛在賽道上持續左右擺盪，幅度不減

從 P 到 PD：加入微分項

純 P 控制器有一個固有缺陷：它只看「當前的偏差大小」，不考慮「偏差的變化趨勢」。 微分項 (D-term) 的加入解決了這個問題：它監測誤差的變化速率，提前預測偏差走向， 就像是為系統裝上了「阻尼器」——當誤差正在快速減小時（車輛正在回正），微分項會自動減弱修正力道，防止過衝。

// PD 控制器實作
let previousError = 0;

function pdControl(error, Kp, Kd) {
  let derivative = error - previousError;
  previousError = error;

  let output = (Kp * error) + (Kd * derivative);

  // 飽和限制 (Saturation)
  output = Math.max(-1, Math.min(1, output));

  return output;
}
備註 Kp（比例項）決定修正的「力道」——偏差越大，修正越強。 Kd（微分項）決定修正的「阻尼」——誤差變化越快，抑制力越強。 在本課程的作業中，你將先實作純 P 控制器，體驗震盪問題後再嘗試加入 D 項來改善穩定性。

Failsafe 安全機制

真實世界中，攝影機可能被遮擋、BLE 可能斷線、目標可能完全消失。 如果系統在「看不見」時仍然繼續執行最後一次的控制指令，車輛將失控「盲衝」—— 這就是為什麼每一個閉迴路系統都必須內建 Failsafe（故障安全）機制。

// Failsafe：影像遺失超時保護
let framesSinceTarget = 0;
const TIMEOUT_FRAMES = 10; // 連續 10 幀未偵測到目標

function safetyCheck(targetDetected) {
  if (targetDetected) {
    framesSinceTarget = 0;
    return true; // 系統安全，繼續控制
  }

  framesSinceTarget++;
  if (framesSinceTarget > TIMEOUT_FRAMES) {
    sendEmergencyStop(); // 緊急煞車！
    return false;
  }
  return true;
}
關鍵 Failsafe 不是「附加功能」，而是閉迴路系統的必要組成部分。 在工業機器人與自動駕駛領域，任何缺乏失效保護的控制系統都無法通過安全認證。 作業中你將實作超時判定邏輯，並模擬影像中斷場景來驗證 Failsafe 是否能在 500ms 內觸發停車。

  • 📂 src/safety_failsafe.js：安全模組，提供幀率計數器，留下「視覺遺失判定與自動停車」邏輯。
  • 📂 REPORT.md：預置「參數調校紀錄表」。學員需紀錄在不同比例增益 ($K_p$) 下，系統的「震盪幅度」與「收斂時間」。
  • 📂 tests/loop_stability.test.js：自動化評分脚本。模擬持續性的偏差輸入，檢核控制輸出是否能保持線性且符合安全限幅 (Clipping)。

2. 作業任務部署細節

任務 1：視覺追蹤閉迴路整合實作 (Vision Feedback Integration)

  • 目標：打通數據鏈路，實現首個「感知－決策－執行」的自動化閉環。
  • Classroom 部署建議：
    • 核心代碼檢核： 
      // 學生需實作：將誤差轉化為馬達脈衝輸出
let output = error * Kp; 
motor.sendPayload({ steer: output });
    • 整合指標：要求學員確保 Bluetooth 的指令發送頻率不超過影像 FPS，防止「指令堆疊」造成的控制滯後。
    • 驗證方式：學員需提交一段影片，展示車輛能隨著目標物（如：手中的亮色物體）移動而自動擺頭或轉向。

任務 2：階躍響應 (Step Response) 紀錄實驗

  • 目標：科學量化控制性能，理解「動態平衡」的權衡。
  • Classroom 部署建議：
    • 實驗要求：在標定的環境下，瞬間移走目標物產生的「階躍偏差」。
    • 性能紀錄：學員需在 README 中填寫：
      • 上升時間 (Rise Time)：從 0 追到 90% 偏差所需的時間。
      • 過衝 (Overshoot)：修正是否過頭？
    • 調校任務：試著將 $K_p$ 調大直到系統產生「蛇行震盪」，並紀錄這個臨界值。這能讓學員深刻理解系統不穩定性的物理來源。

任務 3：斷點與恢復：Failsafe 機制測試

  • 目標：培養工業級的安全性開發習慣，防止「視覺盲衝」。
  • Classroom 部署建議：
    • 失效邏輯檢核： 
      // 學生需實作：超時判定
if (framesSinceLastTarget > 10) {
    sendEmergencyStop();
}
    • Autograding：透過腳本模擬影像串流突然中斷，檢核學員的代碼是否能在 500ms 內發出「停止指令」。
    • 恢復測試：模擬目標重新出現，檢查系統是否能自動恢復追蹤，而不需要重整網頁。

3. 控制工程導師點評標準 (Control System Benchmarks)

此單元的價值在於 「對動態穩定性的直覺與量化分析」：

  • [ ] 時延控制意識：是否理解「傳輸延遲」是造成震盪的主因？在代碼中是否有減少非必要數據上傳的優化？
  • [ ] 代碼完整性：控制器邏輯是否處理了「飽和 (Saturation)」問題？（即：計算出的轉角不能超過硬體極限）。
  • [ ] 故障安全思維：README 是否討論了「失去視覺」時的最優策略？（停止 vs. 慢速原地旋轉搜尋）。
