簡介

OTA 讓你能遠端更新韌體，但如果缺乏安全防護，任何同網段的人都能推送惡意程式到你的裝置上。 從密碼驗證、版本號比對到韌體簽章校驗，每一層防線都在阻止「未經授權的程式碼」進入你的晶片。 本單元將教你如何為 OTA 通道加上「鎖」，確保只有合法的更新才能通過。

情境
一間新創公司部署了 200 台智慧門鎖，全部開放了 OTA 埠方便維護。某天凌晨，一名攻擊者在同一 Wi-Fi 網段掃描到這些裝置， 用預設的空密碼推送了一個「開門」韌體——所有門鎖瞬間失效。事後檢討才發現：OTA 端點既沒有密碼保護，也沒有韌體版本驗證。 這場災難本可透過幾行安全設定完全避免。

為什麼需要 OTA 安全機制？

OTA 功能為你的裝置打開了一扇「遠端替換大腦」的大門。這扇門帶來了巨大的便利性——你可以在千里之外修復 Bug、推送新功能。但如果這扇門沒有「鎖」，任何有心人士都能從同一個 Wi-Fi 網路中推送惡意韌體，完全接管你的裝置。

在生產環境中，OTA 安全不是可選項目，而是絕對必要的基礎設施。一個沒有防護的 OTA 端口，其危險程度等同於把 SSH root 密碼設為空值然後開放到公網上。

三大安全威脅

要設計有效的防護機制，首先需要理解攻擊者可能利用哪些漏洞。以下是 OTA 系統面臨的三大核心威脅：

威脅類型 攻擊方式 可能危害 防禦機制
中間人攻擊 (MITM) 攻擊者攔截 OTA 傳輸通道，替換為惡意韌體 裝置被完全接管、資料外洩、物理損害 HTTPS 加密 + 韌體數位簽章
版本降級 (Rollback) 攻擊者故意推送含有已知漏洞的舊版韌體 重新暴露已修補的安全漏洞 語義化版本比對 + Anti-rollback
硬體型號不匹配 將 A 型號的韌體誤刷到 B 型號的硬體 GPIO 設定錯誤可能損壞硬體元件 晶片型號鎖定 + 韌體標頭檢查

威脅一：中間人攻擊

中間人攻擊（Man-in-the-Middle, MITM）是最嚴重的威脅。攻擊者不需要知道你的 OTA 密碼，只需在裝置與更新伺服器之間的網路路徑上進行監聽與竄改。在區域網路環境中，這可以透過 ARP Spoofing 輕易實現。

想像一個場景：你的智慧門鎖透過 HTTP（非加密）接收 OTA 更新。攻擊者在同一 Wi-Fi 網段中攔截了更新請求，將你的「修復 Bug」韌體替換為「永遠開門」的惡意版本。由於沒有數位簽章驗證，門鎖無法分辨這個韌體是否來自合法來源。

重要 在涉及人身安全的應用（門鎖、車輛控制、醫療設備）中，OTA 通道必須使用 TLS 加密，且韌體必須經過數位簽章驗證。這不是建議，而是安全合規的基本要求。

威脅二：版本降級攻擊

假設你在 v1.2.0 修補了一個關鍵的緩衝區溢位漏洞。如果攻擊者能讓裝置「更新」回 v1.1.0，這個漏洞就會重新暴露。更狡猾的是，降級攻擊可能不容易被察覺——裝置看起來運作正常，但安全防線已被悄悄撕開。

防禦方法很直觀：在接受更新前，比對新韌體的版本號與目前版本。若新版本號 ≤ 現有版本號，立即拒絕更新。ESP-IDF 原生提供 Anti-rollback 功能，可在 Bootloader 層級強制執行此策略。

威脅三：硬體不匹配

ESP32 系列包含多種晶片變體：ESP32、ESP32-S2、ESP32-S3、ESP32-C3 等。它們的 GPIO 配置、周邊裝置、記憶體架構都不同。若將為 ESP32-S3 編譯的韌體刷入 ESP32-C3，輕則程式無法運行，重則因為 GPIO 電壓設定錯誤而燒毀外接元件。

提示 使用 ESP.getChipModel() 可在運行時取得晶片型號字串（如 "ESP32-S3"）。將此資訊與韌體標頭中的目標型號進行比對，就能在寫入 Flash 之前阻止不相容的更新。

在接下來的單元中，我們將逐一實作這三種防護機制的程式碼，從最基礎的密碼保護開始，逐步建立起完整的安全防線。

密碼保護：HTTP 基本驗證

HTTP Basic Authentication 是為 OTA 端點建立存取控制的最直接方法。它利用 HTTP 協議內建的認證機制，讓瀏覽器在使用者嘗試存取受保護頁面時自動彈出登入對話框。雖然這不是最安全的認證方式，但對於區域網路內的 IoT 裝置而言，它提供了足夠的防護層級。

認證流程解析

HTTP Basic Auth 的運作機制基於「挑戰-回應」（Challenge-Response）模型。整個流程完全由 HTTP 標頭驅動，不需要額外的登入頁面或 Session 管理：

1

客戶端發送請求

使用者在瀏覽器輸入 http://esp32.local/update，瀏覽器發送一個不帶認證資訊的 HTTP GET 請求。

2

伺服器回傳 401 挑戰

ESP32 的 WebServer 檢測到請求中沒有 Authorization 標頭，回傳 HTTP 401 狀態碼，並附帶 WWW-Authenticate: Basic realm="OTA" 標頭。

3

瀏覽器彈出登入框

瀏覽器收到 401 回應後，自動顯示原生的使用者名稱/密碼輸入對話框。使用者輸入認證資訊後，瀏覽器將其以 Base64 編碼放入 Authorization 標頭，重新發送請求。

4

伺服器驗證並放行

ESP32 解碼 Authorization 標頭，與儲存在 NVS 中的帳密比對。若正確，回傳 OTA 更新頁面；若錯誤，再次回傳 401。

ESP32 實作程式碼

在 ESP32 的 WebServer 框架中，實作 Basic Auth 只需要兩個 API 呼叫：server.authenticate() 進行比對，server.requestAuthentication() 發送 401 挑戰。

#include <WebServer.h>
#include <Preferences.h>

WebServer server(80);
Preferences prefs;
String ota_user, ota_pass;

void setup() {
  prefs.begin("ota-auth", false);

  // 從 NVS 讀取帳密（首次使用時設定預設值）
  ota_user = prefs.getString("username", "admin");
  ota_pass = prefs.getString("password", "changeme");

  server.on("/update", HTTP_GET, []() {
    // 驗證身份：失敗則要求登入
    if (!server.authenticate(
        ota_user.c_str(), ota_pass.c_str())) {
      return server.requestAuthentication();
    }
    server.send(200, "text/html", update_html);
  });

  server.begin();
}
警告 絕對不要將密碼硬編碼（Hardcode）在原始碼中。原始碼可能被上傳到 GitHub、被反編譯，或者在團隊成員間流通。正確的做法是將密碼存儲在 NVS 中，並在裝置首次啟動時由管理員設定。

使用 NVS 持久化認證資訊

將帳密存儲在 NVS 的好處是：每台裝置可以擁有獨立的認證資訊、密碼可以在不重新燒錄韌體的情況下被更新、且即使裝置重啟密碼也不會遺失。你可以建立一個專用的「管理介面」讓使用者修改密碼：

// 密碼變更路由（同樣需要先通過認證）
server.on("/change-password", HTTP_POST, []() {
  if (!server.authenticate(ota_user.c_str(), ota_pass.c_str())) {
    return server.requestAuthentication();
  }
  String newPass = server.arg("new_password");
  if (newPass.length() >= 8) {
    prefs.putString("password", newPass);
    ota_pass = newPass;
    server.send(200, "text/plain", "密碼已更新");
  }
});
提示 HTTP Basic Auth 搭配 HTTPS 才能提供完整防護。在僅使用 HTTP 的區域網路環境中，密碼雖以 Base64 編碼傳輸（可被輕易解碼），但至少能阻止「隨意掃描就能更新」的無差別攻擊。對於教學與原型開發階段，這已經是足夠的起步防護。

版本檢查與中繼資料

密碼保護解決了「誰可以更新」的問題，但還有另一個同樣重要的問題需要處理：「這個韌體應該被更新嗎？」版本檢查機制能在韌體寫入 Flash 之前，先驗證它的版本號、目標型號等中繼資料（Metadata），主動攔截不合規的更新請求。

這就是「在上傳前就知道會失敗」的防禦哲學——與其等到韌體寫入一半才發現問題，不如在第一個 byte 抵達時就做出判斷。

語義化版本控制 (Semantic Versioning)

語義化版本（SemVer）使用 MAJOR.MINOR.PATCH 三段式格式來表達韌體的演進程度。在 OTA 安全的語境中，版本號的比對規則很明確：只接受版本號嚴格大於目前版本的更新。

版本段 含義 更新範例
MAJOR 不相容的 API 變更或架構重構 1.x.x → 2.0.0（通訊協議變更）
MINOR 新增功能，保持向下相容 1.2.x → 1.3.0（新增感測器支援）
PATCH Bug 修復，無新功能 1.2.3 → 1.2.4（修復溫度偏移）

在韌體中，版本號通常定義在獨立的標頭檔案中，方便管理與引用：

// include/version.h
#define FW_VERSION_MAJOR  1
#define FW_VERSION_MINOR  2
#define FW_VERSION_PATCH  4
#define FW_VERSION_STRING "1.2.4"
#define HW_TARGET         "ESP32-S3"

實作版本比對邏輯

版本比對的核心邏輯是：將三段版本號各自轉為整數，從 MAJOR 到 PATCH 依序比較。以下函式會在 OTA 上傳的初期階段被呼叫：

// 比對版本號：回傳 true 表示允許更新
bool isVersionNewer(int newMaj, int newMin, int newPat) {
  if (newMaj != FW_VERSION_MAJOR)
    return newMaj > FW_VERSION_MAJOR;
  if (newMin != FW_VERSION_MINOR)
    return newMin > FW_VERSION_MINOR;
  return newPat > FW_VERSION_PATCH;
}

// 在上傳處理中使用
if (!isVersionNewer(newMaj, newMin, newPat)) {
  Serial.println("拒絕：韌體版本過舊");
  Update.abort();
  server.send(403, "text/plain",
    "Error: 版本降級被拒絕");
  return;
}
注意 Update.abort() 必須在 Update.end() 之前呼叫。一旦 end() 被執行且返回成功，分區切換就已經完成，此時再呼叫 abort() 為時已晚。最佳實作是在收到韌體的前幾個封包時就完成所有預檢。

硬體型號鎖定

除了版本號，韌體標頭中還應包含目標硬體型號資訊。ESP32 系列有多種變體，它們的指令集、GPIO 腳位、記憶體配置都不同。將錯誤型號的韌體刷入裝置可能造成不可逆的損害。

// 檢查硬體型號是否匹配
String chipModel = ESP.getChipModel();

if (chipModel != String(HW_TARGET)) {
  Serial.printf("型號不匹配！本機: %s, 韌體目標: %s\n",
    chipModel.c_str(), HW_TARGET);
  Update.abort();
  server.send(403, "text/plain",
    "Error: 硬體型號不匹配");
  return;
}

防禦層級總結

將所有安全機制組合在一起，OTA 更新會經過層層檢查才能最終寫入 Flash：

1

身份驗證（密碼）

攔截未經授權的使用者 → HTTP 401

2

版本檢查（Anti-rollback）

拒絕降級更新 → HTTP 403 + Update.abort()

3

硬體綁定（Chip Model）

阻止不相容韌體 → HTTP 403 + Update.abort()

4

完整性校驗（Checksum）

偵測傳輸損壞 → Update.end() 返回 false

提示 在前端 UI 中，當更新被攔截時，應提供清晰的錯誤訊息告訴使用者原因（如「版本過舊」或「型號不匹配」），而非僅顯示一個通用的「更新失敗」。好的錯誤提示能大幅減少技術支援的負擔。

OTA 安全性 (密碼保護與版本檢查)

本單元將帶領你將物聯網裝置從「實驗室雛型」提升到「商業級產品」。你將學習如何透過身份驗證、語義化版本控制與硬體型號鎖定，為你的裝置建立起完善的安全防護。

🎯 學習目標

  1. 實作 HTTP Basic Authentication，確保只有授權使用者能發起更新。
  2. 整合 NVS (Non-Volatile Storage)，讓管理密碼具備重啟後的持久性。
  3. 實作 版本檢查 (Anti-rollback)，防止系統被惡意降級。
  4. 實作 硬體型號鎖定 (Hardware Affinity)，避免誤刷不相容的韌體。

🚀 任務說明

任務 1：「數位守門員」實作 OTA 密碼鎖

  • 目標：在 /update 路由實作身份驗證。
  • 關鍵技術：使用 Preferences 函式庫儲存密碼，並調用 server.authenticate() 進行攔截。
  • 驗證方式：開啟 /update 時，瀏覽器應彈出登入視窗。

任務 2：「防呆機制」版本檢查與降級攔截

  • 目標：確保上傳的韌體版本必須高於目前版本。
  • 關鍵技術：在 Update.onStart 時解析韌體標頭，若 New_Version < Current_Version 則調用 Update.abort()。
  • 驗證方式：嘗試上傳舊版本韌體，系統應能正確攔截並報錯。

任務 3：「硬體適配」晶片型號鎖定

  • 目標：防止韌體被刷入不匹配的硬體架構（例如將 ESP32-S3 韌體刷入 ESP32-C3）。
  • 關鍵技術：利用 ESP.getChipModel() 與 version.h 中的預定義型號進行比對。
  • 驗證方式：修改型號後嘗試上傳，系統應拒絕更新。

📂 程式庫結構

  • src/main.cpp：主要的安全邏輯實作位置。
  • include/version.h：定義版本號與目標硬體型號。

1. 範本程式庫 (Template 程式庫) 配置建議

安全作業的核心在於「攔截非法請求」，範本應包含完整的安全框架測試環境，建議包含：

  • 📂 src/main.cpp：安全邏輯實作對象。提供一個具備基礎 OTA 但完全不設防的代碼，學員需在此實作 server.authenticate() 攔截器與版本比對邏輯。
  • 📂 include/version_config.h：定義當前韌體的 APP_VERSION (例如 1.0.1) 與 HARDWARE_TARGET (例如 ESP32_S3)。
  • 📂 data/access_denied.html：設計一個專業的「拒絕訪問」頁面，當學員模擬攻擊失敗時顯示。
  • 📂 docs/SECURITY_AUDIT.md：安全審計清單。學員需在此記錄三項測試結果：
    1. 輸入錯誤密碼是否能成功攔截？
    2. 上傳舊版本 (.bin) 是否會自動觸發 Update.abort()？
    3. 若偵測到硬體型號不符，系統是否能停止更新？

2. 作業任務部署細節

任務 1：「數位守門員」實作 OTA 密碼鎖 (Digital Gatekeeper Lab)

  • 目標：實作 HTTP Basic Authentication，並利用 NVS (Non-Volatile Storage) 讓管理密碼具備重啟後的持久性。
  • Classroom 部署建議：
    • 攔截器實作：學員需在 /update 路由入口處插入驗證 logic，未登入者應收到 401 錯誤並彈出登入視窗。
    • NVS 整合：利用 Preferences 函式庫，學員需實作一個功能：第一次開機時設定密碼，之後所有更新請求都必須匹配該密碼。
    • 驗證要點：提交一張截圖，展示瀏覽器彈出「需要使用者名稱與密碼」的標準認證對話框。

任務 2：「防呆機制」版本檢查與降級攔截 (Integrity Guard: Version Pre-check Lab)

  • 目標：解析二進位檔標頭資訊，實作「拒絕降級 (Anti-rollback)」策略，確保系統版本永遠向上提升。
  • Classroom 部署建議：
    • 標頭解析實作：學員需學會在 Update.onStart 時解析上傳流的前 1K Bytes，提取其中的版本字串。
    • 邏輯裁決：若 New_Version < Current_Version，必須立即調用 Update.abort() 並回傳錯誤訊息。
    • 技術說明：在 README 中解釋，為什麼在生產環境中，「防止版本退回」比「新增功能」更重要。

任務 3：「硬體適配」晶片型號鎖定 (Hardware Affinity: Architecture Lock Lab)

  • 目標：利用 SoC 內部指紋，防止「誤刷」導致的硬體毀損（如將 S3 韌體刷入 C3）。
  • Classroom 部署建議：
    • 指紋提取：調用 ESP.getChipModel() 讀取當前硬體架構。
    • 嚴格綁定：在更新標頭中加入型號檢查位元。若不符，立即強制關閉連線。
    • 穩定性測試：學員需證明：即便持有密碼且版本號正確，但若編譯目標架構不符，系統仍能守住最後一線防波堤。

3. 物聯網安全官點評標準 (IoT Security & Audit Standards)

此單元的價值在於 「對異常路徑 (Negative Paths) 的處理能力」：

  • [ ] 認證嚴謹度：帳號密碼是否硬編碼 (Hardcoded) 在代碼中？（若是，則應扣分，引導使用 NVS）。
  • [ ] 版本管理邏輯：是否能正確解讀語義化版本 (Semantic Versioning) 並執行正確的攔截判斷？
  • [ ] 安全訊息提示：當更新被攔截時，前端介面是否能提供清晰的錯誤原因（如：Error 101 - Version Too Old）？
