基礎 2 - 安全防線

OTA 安全性：密碼與版本檢查

守護你的硬體防線：防止非法篡改與錯誤更新。學習如何建立數位守門員，確保韌體更新的安全與穩定。

⚠️ 為什麼需要 OTA 安全機制？ (20 min)

沒有保護的 OTA，就像是把家門大開並在門口放了台「全自動換鎖機」。

🔒

攔截與偽造 (MITM)

理解「中間人攻擊」。若無加密，攻擊者可上傳惡意代碼接管無人車，甚至造成物理上的安全危害。

📉

版本倒退風險 (Rollback)

將舊版具有 Bug 的韌體重新刷入，可能導致硬體參數失效或穩定性崩潰。好的系統必須「拒絕降級」。

🎭

硬體規格不匹配

誤將高階 S3 的代碼刷入低階 C3？系統可能直接斷氣。更新前必須確認設備的身分證 (Chip Model)。

🔐 密碼保護：HTTP 基本驗證 (20 min)

HTTP Basic Auth 流程

  • 1

    瀏覽器發送更新請求，ESP32 檢查標頭 (Auth Header) 中是否包含正確資訊。

  • 2

    若無，回傳 401 Unauthorized 並觸發瀏覽器原生的登入視窗。

  • 3

    密碼應從 NVS (非揮發性存儲) 讀取，以確保每個人的設備都能擁有獨特的密碼。

auth-logic-snippet

網頁路徑攔截器

server.on("/update", HTTP_GET, []() {
  // 檢查 Preferences 庫中的帳密
  if (!server.authenticate(username, password)) {
    return server.requestAuthentication();
  }
  server.sendContent(index_html);
});

🏷️ 版本檢查與中繼資料 (20 min)

「在上傳前就知道會失敗」是專業開發者與業餘愛好者的差別。

📦

語義化版本

理解 MAJOR.MINOR.PATCH 定義。確保更新方向正確。

🏷️

韌體標頭解析

讀取 BIN 檔前 1K Bytes，提取版本與目標型號。

🛡️

主動攔截 (Abort)

在寫入 Flash 之前調用 `Update.abort()` 以除後患。

實戰任務：OTA 安全實驗室

打造具備專業防護等級的韌體更新系統。

🔐 45 min

【作業 3-1】

「身份驗證與持久化」

為更新介面增加密碼防護，並練習將管理員帳密存放於 NVS 非揮發性存儲中，實現密碼的「永久記憶」。

🤖 60 min

【作業 3-2】

「防止降級與標頭檢查」

撰寫預檢邏輯。在上傳過程中先檢查版本號，若偵測到版本過舊，立即主動叫停更新流程。

🆔 45 min

【作業 3-3】

「晶片型號交叉鎖定」

獲取 Chip Model。實作代碼校驗，當上傳的韌體目標與實體硬體架構不符時，鎖定更新功能以防硬體損壞。

安全 OTA 防禦實驗室 (Security Vault)

任務 1：「數位守門員」實作 OTA 密碼鎖 (Digital Gatekeeper: Password Protection)

角色扮演：安全工程師 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 至本地開發。
  2. 身份驗證實作：為 /update 路由增加 HTTP Basic Auth。
  3. 參數持久化：使用 Preferences 庫將 OTA 帳號與密碼儲存於 NVS，而非 Hardcode。
  4. 防禦機制演練：實作錯誤計數器。在 README 中貼上「連續密碼錯誤被拒絕」的瀏覽器截圖。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 驗證嚴謹度：是否能正確攔截未經授權的更新請求並彈出 Auth 視窗？
  • [ ] 持久化邏輯：斷電重啟後，自定義的密碼是否依然有效？

任務 2：「防呆機制」韌體版本檢查 (Integrity Guard: Version Checking)

角色扮演：韌體校驗官 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 版本標記預設：在專案中定義 v1.0.0 旗標。
  2. 標頭預檢實作：在更新伺服器處理函式中，先讀取 BIN 檔前段解析出新版本號。
  3. 攔截與中止：若 New Version <= Current Version，立即調用 Update.abort() 並發送「禁止降級」警告。
  4. 文檔敘述：在 README 中闡述為什麼工業設備必須拒絕版本回滾 (Rollback) 以防禦舊有漏洞。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 預檢準確性：能否在 Flash 寫入動作前就識別出版本衝突並發出中止指令？
  • [ ] 用戶告知質量：前端介面能否清晰解析並顯示板端發出的版本異常代碼？

任務 3：「硬體適配」硬體 ID 驗證 (Hardware Affinity: ID Validation)

角色扮演：系統整合專家 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 指紋識別：調用 ESP.getChipModel() 提取實體處理器資訊。
  2. 配對校驗邏輯：在 OTA 訊息中包含架構名稱。開發一段代碼：若架構 (如 S3 vs C3) 不符，立即鎖定更新流程。
  3. 壓力測試報告：嘗試上傳錯誤晶片型號的韌體，截圖證明系統已成功識別並拒絕燒錄。
  4. 最終提交：整併所有安全模塊並 Push 最終 Commit 到 GitHub。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 硬體綁定能力：是否正確拒絕了非法硬體架構的韌體更新請求？
  • [ ] 安全架構思維：README 中是否體現了「誤傳韌體可能導致硬體損毀」的安全防範意識？

GitHub Classroom 詳細操作指南

本單元教學重點：在學會無線更新後，進一步建立「全棧安全」與「抗攻擊」工程能力。

  1. 第一 Point：保護物理設施 - 強調駭客接管 IoT 裝置的代價不只是資料遺失，更可能是實體損壞。
  2. 第二 Point：無感化驗證 - 展示如何利用 NVS 存儲帳密，讓學員體驗到產品化流程中的「個人化配置」概念。
  3. 第三 Point：流程穩健性 - 經由版本號校驗實驗，引導學生思考 CI/CD 流程中「版本管理」與「硬體驗收」的嚴謹性。
