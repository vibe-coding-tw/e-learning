簡介

本單元深入探討 影像座標系的定義方式與追蹤誤差的量化計算。在視覺追蹤系統中，攝影機畫面的像素座標必須被轉換為有意義的偏差量，才能驅動馬達進行修正。你將學會如何定義畫面中心、計算目標偏移量，以及將像素誤差正規化為控制訊號所需的比例值。

情境
你的自走車搭載了一顆 320×240 像素的攝影機，程式已能偵測到前方目標物的中心點座標。但座標 (185, 130) 到底代表目標偏左還是偏右？偏了多少？這個數字要怎麼換算成馬達應該轉動的速度？如果不建立明確的座標系與誤差公式，控制邏輯就無從下手。本單元將為你打下這個計算基礎。

影像座標系：從 (0,0) 開始

在數位影像中，座標系的定義與傳統數學座標系完全相反。影像的原點 \((0, 0)\) 位於左上角，X 軸向右遞增，Y 軸向下遞增。這是因為攝影機逐行掃描像素的方式——從左上角開始，一行一行往下讀取。理解此座標系統是建構視覺控制回饋的第一步。

影像座標系 vs. 數學座標系

特性影像座標系傳統數學座標系
原點位置左上角左下角（或中心）
X 軸方向向右為正向右為正
Y 軸方向向下為正向上為正
單位像素 (px)任意實數
範圍[0, W-1] × [0, H-1](-∞, +∞)

定義畫面中心

在視覺追蹤中，我們關心的是目標物相對於畫面中心的偏差。畫面中心即為攝影機光軸的投影點，也對應車體的正前方。對於解析度為 \(W \times H\) 的影像，幾何中心座標為：

// 計算畫面幾何中心
const centerX = Math.floor(width / 2); // 320/2 = 160
const centerY = Math.floor(height / 2); // 240/2 = 120

// 對於 QVGA (320×240) 的攝影機：
// 中心 = (160, 120)
// 左上角 = (0, 0)，右下角 = (319, 239)

從像素座標到偏差量

當影像辨識演算法偵測到目標物後，會回傳其質心座標 \((target_x, target_y)\)。我們需要計算目標物與畫面中心的偏差（即「誤差」）：

  • 水平誤差：\(error_x = target_x - center_x\)。正值代表目標在畫面右側，負值代表在左側。
  • 垂直誤差：\(error_y = target_y - center_y\)。正值代表目標在畫面下方，負值代表在上方。
備註 在水平追蹤的自走車應用中，通常只使用 X 軸方向的誤差來控制左右轉向。Y 軸誤差可用於估算目標物的遠近距離（目標越遠，在畫面中越偏上方）。
實務技巧 在 Canvas 或 OpenCV 中繪製十字準星標示畫面中心，是最基本的除錯工具。當你看到目標物的標記點剛好位於十字交叉處時，代表誤差為零，車體正對目標。

誤差 (Error) 定義與正規化

原始像素誤差（例如「偏右 45 像素」）是與攝影機解析度綁定的絕對值。如果你將攝影機從 320×240 換成 640×480，同樣的物理偏移會產生不同的像素誤差值，導致控制參數必須重新調校。正規化（Normalization）解決了這個問題——將誤差映射到固定的 \([-1.0, +1.0]\) 區間，使控制邏輯不再依賴特定硬體。

正規化公式

正規化的核心公式非常簡潔。以 X 軸為例：

// 正規化誤差公式
normalizedError = (targetX - centerX) / centerX;

// 結果範圍：
// 目標在最左邊 (targetX = 0) → error = -1.0
// 目標在正中央 (targetX = W/2) → error = 0.0
// 目標在最右邊 (targetX = W-1) → error ≈ +1.0

跨解析度的一致性驗證

正規化最大的好處是：無論攝影機解析度為何，相同的物理偏移量會產生相同的正規化誤差值。以下示例展示一個目標物位於畫面左側 1/4 處時的計算結果：

攝影機解析度目標 X 座標中心 X像素誤差正規化誤差
160 × 1204080-40 px-0.50
320 × 24080160-80 px-0.50
640 × 480160320-160 px-0.50
1280 × 720320640-320 px-0.50
設計優勢 正規化後，你的控制參數（如 \(K_p\) 增益）只需調校一次，即可在不同解析度的攝影機上通用。這大幅降低了硬體升級時的軟體維護成本。

誤差的物理意義

正規化誤差的正負號與絕對值分別傳達不同的控制訊息：

  • 正負號（方向）：決定馬達應該向左還是向右修正。正值 → 目標在右方 → 車體需右轉。
  • 絕對值（強度）：表示偏差程度。\(|error| = 1.0\) 代表目標已到達畫面邊緣，需要最大修正力道。\(|error| = 0.1\) 代表目標幾乎對準中心，僅需微量修正。
注意 當目標物完全離開畫面（偵測失敗）時，不應產生誤差值。此時應執行「失去目標」的保護邏輯（如停止或原地旋轉搜尋），而非輸出一個虛假的誤差。

程式實作範例

function getNormalizedError(targetX, canvasWidth) {
const center = canvasWidth / 2;
const error = (targetX - center) / center;
return Math.max(-1.0, Math.min(1.0, error)); // 限幅保護
}

誤差與控制的聯覺 (Mapping)

正規化誤差本身只是一個數字，要讓車體「動起來」，我們需要將誤差映射（Map）為馬達的控制指令。這個映射過程就是自動控制的核心：誤差越大，修正力道越強；誤差為零時，系統維持直行。

線性映射：從誤差到轉向

最直覺的映射方式是線性比例：將正規化誤差乘以一個增益常數 \(K_p\)，得到馬達的差速指令。

// 基礎線性映射
const Kp = 150; // 比例增益
const steering = normalizedError * Kp;

// 差速驅動：左右馬達速度
const baseSpeed = 200;
let leftMotor = baseSpeed + steering;
let rightMotor = baseSpeed - steering;

// 限幅保護：PWM 範圍 [0, 255]
leftMotor = Math.max(0, Math.min(255, leftMotor));
rightMotor = Math.max(0, Math.min(255, rightMotor));

映射邏輯解讀

正規化誤差物理意義steering 值 (Kp=150)馬達行為
-1.0目標在最左邊-150大幅左轉（左輪減速、右輪加速）
-0.3目標略偏左-45微幅左修正
0.0目標在正中央0直行
+0.5目標偏右+75明顯右轉
+1.0目標在最右邊+150大幅右轉

死區（Dead Zone）設計

真實感測器會產生雜訊，即使目標物靜止不動，偵測到的座標也會在幾個像素內跳動。如果不處理這些微小的誤差波動，馬達會不斷進行無意義的微修正，產生可聽見的「嗡嗡」聲且浪費電力。解決方案是設定死區——當誤差的絕對值低於某個閾值時，視為零誤差。

// 死區處理
const DEAD_ZONE = 0.05; // 忽略 ±5% 以內的誤差

function applyDeadZone(error) {
if (Math.abs(error) < DEAD_ZONE) return 0;
return error;
}
備註 增益 \(K_p\) 的選擇是一門藝術：過小會導致修正動作太慢，車體在彎道中衝出軌跡；過大則會導致車體劇烈擺盪（震盪）。最佳值需要透過實驗調校——這正是下一個模組「P 控制策略」的核心主題。
重要 限幅處理（Clamping / Saturation）是工程必要步驟。若 PWM 值超過 255 或低於 0，可能導致硬體驅動晶片行為異常。永遠在輸出端加上 Math.max(0, Math.min(255, value)) 保護。

    • 延遲討論：引導學員在 PR 中討論：為什麼移動平均視窗不能開得太大（例如 50 幀）？（答案：會產生不可接受的反應遲滯）。

3. 控制工程導師點評標準 (Control Ops Benchmarks)

此單元的價值在於 「數據的標準化與系統的即時性」：

  • [ ] 誤差量化正確性：正規化後的數值是否具備清晰的物理意義？（0 是否真的代表正中央？負號是否代表左側？）。
  • [ ] 抗噪優化度：開啟濾波後，控制曲線是否明顯變「絲滑」？
  • [ ] 參數敏感度認識：是否理解 $K_p$ 增益對「反應速度 vs 穩定性」的權衡關係。
