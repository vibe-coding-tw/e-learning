// 監聽 XMLHttpRequest 上傳事件
xhr.upload.addEventListener('progress', function(e) {
  if (e.lengthComputable) {
    const percent = Math.round((e.loaded / e.total) * 100);
    progressBar.style.width = percent + '%';
    statusText.innerText = `已完成: ${percent}%`;
  }
});

課程單元 基礎 2 - 雲端更新

實驗單元：Arduino OTA 與 Web OTA 原理

本單元將帶領你從「手動燒錄」進化到「物聯網自動化維護」。你將學習如何實作無線韌體更新 (OTA)，建立具備進度監測的前端介面，並理解硬體層級的 A/B 分區回滾安全機制。

🎯 學習目標

  1. 掌握 ESP32 分區表 (Partition Table) 的配置與原理。
  2. 實作 Web-based OTA 更新功能。
  3. 開發具備進度條 (Progress Bar) 的使用者介面。
  4. 理解並測試 A/B 分區的容錯與回滾機制。

🛠️ 任務說明

任務 1：分區表配置 (Partition Configuration)

為了支援 OTA，ESP32 需要至少兩個應用程式分區 (app0, app1)。

  • 修改 platformio.ini，指定使用自定義的分區表文件。
  • 編輯 ota_split.csv，正確劃分 app0, app1, otadata 與 nvs 空間。

任務 2：Web OTA 核心實作

在 src/main.cpp 中實作 WebServer 的 /update 路由處理。

  • 使用 Update.begin(), Update.write(), 與 Update.end() 處理上傳的二進位檔案。
  • 確保更新完成後設備能自動重啟。

任務 3：進度條視覺化介面

編輯 data/index.html，為 OTA 上傳頁面加入視覺化反饋。

  • 利用 XMLHttpRequest 的 progress 事件監控上傳進度。
  • 實作動態 CSS 進度條，並在更新成功後引導使用者重啟或自動跳轉。

任務 4：故障與回滾測試

模擬真實環境中可能發生的意外中斷。

  • 在上傳過程中（約 50%）中斷電源或網路。
  • 驗證設備是否能自動回滾至原本健康的版本。
  • 完成 docs/Rollback.md 中的實驗報告。

1. 範本程式庫 (Template 程式庫) 配置建議

OTA 實作涉及「分區佈局」與「前後端協作」，範本應包含完整的系統架構，建議包含：

  • 📂 platformio.ini：配置練習核心。學員需在此設定 partitions = ota_2mb.csv，引導其手動配置非標準的分區空間。
  • 📂 ota_2mb.csv：分區定義表。預設為空的或錯誤的分區表，學員需依照硬體 Flash 大小（如 4MB）劃分出 app0, app1 與 otadata 空間。
  • 📂 data/update.html：前端實戰區。提供一個基礎的上傳表單，學員需在此補全 JavaScript 邏輯，實作具備動態 CSS 的「更新進度條」。
  • 📂 src/main.cpp：伺服器框架區。內置 WebServer 框架，但 /update 的處理函式中預留 TODO，學員需實作二進位流寫入 (Update.write) 的核心對話。
  • 📂 docs/Rollback.md：容錯實驗日誌。學員需模擬「斷電/斷網」情境，並記錄 A/B 分區如何發揮保護作用防止變磚。

2. 作業任務部署細節

任務 1：「無線進化」基礎 Web OTA (Web-Based Wireless Update Lab)

  • 目標：打通電腦與 ESP32 的無線通訊路徑，徹底擺脫 USB 數據線。
  • Classroom 部署建議：
    • 路由實作建構：學員需實作 Update.begin(), Update.write(), 與 Update.end() 這套三位一體的核心指令。
    • 連通性驗證：要求學員在 esp32.local/update 介面成功上傳一段修改後的 Blink 程式。
    • 驗證要點：README 中需展示上傳成功後，設備自動重啟並進入新程式的日誌。

任務 2：「更新進度條」視覺化設計 (OTA Progress & UX Design Lab)

  • 目標：透過 JavaScript 監控二進位流傳輸狀態，建立專業的用戶反饋體驗。
  • Classroom 部署建議：
    • 事件監聽實作：使用 XMLHttpRequest 的 progress 事件，學員需計算 (e.loaded / e.total) 並實時更新 CSS Width。
    • 狀態引導設計：上傳完成後需觸發「系統重啟中...」的提示頁面，並在 3 秒後自動跳轉。
    • PR 提交內容：提交一段螢幕錄影或 GIF，展示進度條從 0% 到 100% 的平滑移動過程。

任務 3：「安全防線」故障與回滾測試 (Rollback & Security Audit Lab)

  • 目標：掌握 A/B 分區的底層邏輯，建立「硬體永遠能自癒」的專業工程觀念。
  • Classroom 部署建議：
    • 故障模擬挑戰：要求學員在上傳進度達到 50% 時故意拔掉開發板電源。
    • 恢復力檢測：學員需重啟設備，並證明 ESP32 能自動切換回原本健康的分區 (Rollback)，而非停在損毀的分區中。
    • 技術原理分析：在報告中解釋 otadata 分區在這次回滾實驗中扮演的角色。

3. 物聯網部署專家導師點評標準 (IoT Deployment & Reliability Standards)

此單元的價值在於 「對大型系統佈署可靠性與前端交互的反饋意識」：
