簡介

當你的 IoT 裝置部署到天花板、工廠產線或戶外田間，每次修 Bug 都要拆下來接線燒錄，不僅耗時更可能損壞接頭。 OTA（Over-The-Air）讓韌體能透過 Wi-Fi 無線推送更新，搭配雙分區切換機制，即使更新中途斷電也不會「變磚」。 理解 OTA 原理，是讓你的產品從實驗室原型走向真實部署的關鍵一步。

OTA 的核心：雙分區機制

OTA（Over-The-Air）更新的核心挑戰在於：如何在不中斷裝置正常運作的前提下，安全地將新韌體寫入 Flash 記憶體？答案就是雙分區（A/B Partition）機制。這套設計靈感來自伺服器世界的藍綠部署（Blue-Green Deployment），確保即使更新過程中發生斷電或網路中斷，裝置仍能回到上一個正常運作的韌體版本。

在傳統的單分區架構中，新韌體會直接覆蓋舊韌體。一旦寫入過程中發生異常（例如斷電、封包遺失），Flash 中的資料就會處於「半新半舊」的損壞狀態，裝置將無法正常啟動——這就是俗稱的「變磚」（Bricking）。雙分區機制徹底消除了這個風險。

A/B 分區的運作原理

ESP32 的 Flash 被劃分為兩個大小相等的應用程式分區：app0（又稱 OTA_0）和 app1（又稱 OTA_1）。在任何時刻，只有一個分區是「活躍」的（正在執行的韌體所在），另一個則是「備用」的（等待接收新韌體）。OTA 更新流程不會觸碰正在運行的分區，而是將新韌體寫入備用分區。

1

接收韌體資料

裝置透過 Wi-Fi 接收新的韌體二進位流（.bin 檔案），並將資料逐塊寫入備用分區。此時活躍分區的程式仍在正常執行。

2

驗證完整性

所有資料寫入完畢後，系統會驗證備用分區中的韌體完整性（Checksum 或 SHA-256 校驗）。若驗證失敗，備用分區標記為無效，裝置繼續使用原韌體。

3

切換啟動分區

驗證成功後，系統更新 otadata 分區中的引導標記，將「下次啟動分區」指向剛寫入的備用分區，然後執行重啟。

4

啟動新韌體

重啟後，Bootloader 讀取 otadata 中的標記，從新分區載入韌體。原本的活躍分區現在成為備用分區，角色互換完成。

otadata 分區的關鍵角色

otadata 是一個僅 8KB 的微型分區，但它是整個 OTA 機制的「指揮中心」。它記錄了哪個應用分區應該被啟動、上次更新是否成功，以及是否需要回滾。ESP32 的 Bootloader 在每次開機時都會先讀取 otadata，再決定載入哪個分區的韌體。

注意 若新韌體啟動後未呼叫 esp_ota_mark_app_valid_cancel_rollback() 確認自身健康，系統會在下次重啟時自動回滾到舊分區。這是 ESP-IDF 內建的「自動回滾」安全網。

為什麼需要 4MB 以上的 Flash？

以典型的 OTA 分區配置為例，一個支援雙分區 OTA 的 ESP32 裝置需要以下最低空間分配：

分區名稱 類型 建議大小 用途
nvs data 16 KB Wi-Fi 設定、使用者參數
otadata data 8 KB OTA 引導標記與回滾狀態
app0 (OTA_0) app 1.5 MB 主應用程式分區 A
app1 (OTA_1) app 1.5 MB 主應用程式分區 B
spiffs data 896 KB 網頁資源 / 設定檔

從上表可以看出，光是兩個應用分區就需要 3MB 空間。加上 NVS、otadata 和檔案系統，4MB 是支援完整 Web OTA 功能的最低門檻。若你的韌體包含 BLE + Wi-Fi + Web Server 等大型功能，建議選用 8MB 或 16MB Flash 的開發板。

提示 在 PlatformIO 中，你可以透過 board_build.partitions = partitions_two_ota.csv 指定使用雙 OTA 分區表。ESP-IDF 預設提供多種分區表範本，位於 $IDF_PATH/components/partition_table/ 目錄下。

Web OTA 的傳輸流程

Web OTA 是一種透過 HTTP 協議實現韌體更新的方法。使用者只需開啟瀏覽器、選擇 .bin 韌體檔案、按下上傳按鈕，ESP32 就能自動接收並寫入新韌體。這個流程涉及前端（瀏覽器 JavaScript）與後端（ESP32 WebServer）的緊密協作。

相較於 Arduino OTA（使用 UDP 廣播與專用協議），Web OTA 的優勢在於：不需要安裝任何開發工具，任何有瀏覽器的設備（手機、平板、筆電）都能執行更新。這使得 Web OTA 成為生產環境中最常採用的遠端更新方案。

完整傳輸流程解析

Web OTA 的資料流可以分為三個階段：前端封裝、網路傳輸、後端寫入。每個階段都有對應的技術實作重點。

1

前端：封裝二進位資料

使用者透過 <input type="file"> 選取 .bin 檔案後，JavaScript 將檔案包裝進 FormData 物件，並透過 XMLHttpRequest 以 HTTP POST 方法發送至 ESP32 的 /update 路由。

2

傳輸：串流式分塊上傳

由於韌體檔案通常為 1～2MB，瀏覽器不會一次送出全部資料，而是將檔案分割成多個 HTTP chunk 逐步傳輸。ESP32 的 WebServer 在接收每個 chunk 時立即調用回呼函式處理。

3

後端：逐塊寫入 Flash

ESP32 在 /update 路由的 handler 中使用 Update 類別的三段式 API：Update.begin() 開始更新、Update.write() 逐塊寫入、Update.end() 完成並驗證。

4

完成：重啟切換分區

寫入完成後，ESP32 回傳成功回應給瀏覽器，然後呼叫 ESP.restart() 重新啟動，Bootloader 自動載入新寫入的分區。

ESP32 後端核心程式碼

以下是 /update 路由的核心處理邏輯。此 handler 會在每個資料區塊到達時被呼叫：

// /update POST handler — 處理韌體二進位流
server.on("/update", HTTP_POST,
  // 回應函式：上傳完成後執行
  [](AsyncWebServerRequest *request) {
    request->send(200, "text/plain",
      Update.hasError() ? "FAIL" : "OK");
    ESP.restart();
  },
  // 上傳處理函式：逐塊接收資料
  [](AsyncWebServerRequest *request, String filename,
     size_t index, uint8_t *data, size_t len, bool final) {
    if (index == 0) {
      Update.begin(UPDATE_SIZE_UNKNOWN);
    }
    Update.write(data, len);
    if (final) {
      Update.end(true);
    }
  }
);
警告 Update.begin() 的參數若傳入 UPDATE_SIZE_UNKNOWN，系統會使用整個備用分區的空間。若你已知韌體大小，建議傳入精確的 byte 數以啟用大小檢查，避免寫入超出分區範圍的資料。

前端進度監聽實作

使用者體驗的關鍵在於即時回饋。透過 XMLHttpRequest 的 upload.onprogress 事件，前端可以精確計算已上傳的百分比並更新 UI：

const xhr = new XMLHttpRequest();
const formData = new FormData();
formData.append('firmware', fileInput.files[0]);

// 監聽上傳進度事件
xhr.upload.addEventListener('progress', function(e) {
  if (e.lengthComputable) {
    const percent = Math.round((e.loaded / e.total) * 100);
    progressBar.style.width = percent + '%';
    statusText.innerText = `上傳中: ${percent}%`;
  }
});

// 監聽完成事件
xhr.onload = function() {
  statusText.innerText = '更新成功！裝置重啟中...';
  setTimeout(() => location.reload(), 5000);
};

xhr.open('POST', '/update');
xhr.send(formData);
提示 更新完成後，ESP32 會重啟並可能獲得新的 IP 位址。建議在前端設計中加入 mDNS 名稱（如 esp32.local）作為重新導向目標，而非使用硬編碼的 IP 位址。

安全與驗證的護城河

開放 OTA 端口等同於在無線網路上公開了一扇「可以替換裝置大腦」的門。若缺乏適當的安全防護，任何在同一網段的攻擊者都能推送惡意韌體，接管你的裝置。在生產環境中，OTA 安全機制不是「加分項」，而是絕對必要的基礎設施。

一個完善的 OTA 安全體系通常包含多層防護：網路層的存取控制、應用層的身份驗證、韌體層的完整性校驗。每一層都能獨立阻擋特定類型的攻擊，層層疊加形成縱深防禦（Defense in Depth）。

mDNS 本地服務發現

mDNS（Multicast DNS）讓你能使用人類可讀的名稱（如 esp32.local）來存取裝置，取代難以記憶且可能動態變化的 IP 位址。雖然 mDNS 本身不是安全機制，但它是建立可靠 OTA 工作流程的基礎設施——你需要一個穩定的方式來「找到」你的裝置。

// 在 setup() 中初始化 mDNS
#include <ESPmDNS.h>

void setup() {
  WiFi.begin(ssid, password);
  while (WiFi.status() != WL_CONNECTED) delay(500);

  // 註冊 mDNS 名稱：sensor-01.local
  if (MDNS.begin("sensor-01")) {
    MDNS.addService("http", "tcp", 80);
  }
}

HTTP Basic Authentication

HTTP Basic Auth 是最基礎的存取控制機制。當使用者嘗試進入 /update 頁面時，ESP32 會檢查 HTTP Header 中是否包含正確的帳號密碼。若驗證失敗，瀏覽器會彈出登入對話框，要求使用者提供認證資訊。

在 ESP32 的 WebServer 框架中，實作 Basic Auth 只需幾行程式碼：

const char* ota_username = "admin";
const char* ota_password = "mySecurePass123";

server.on("/update", HTTP_GET, []() {
  if (!server.authenticate(ota_username, ota_password)) {
    return server.requestAuthentication();
  }
  server.send(200, "text/html", update_page_html);
});
警告 HTTP Basic Auth 的帳密是以 Base64 編碼（非加密）傳輸的。在區域網路內這提供了基本防護，但若需要在公開網路上操作 OTA，應搭配 HTTPS（TLS）以防止密碼被竊聽。

韌體完整性校驗

即使通過了身份驗證，仍需確保上傳的韌體檔案本身是完整且未被篡改的。ESP32 的 Update 類別內建了基礎的校驗機制：在 Update.end() 時會驗證已寫入資料的 MD5 雜湊值。你也可以額外實作更嚴格的驗證：

驗證層級 方法 防護目標
基礎 Update 內建 MD5 校驗 傳輸過程中的資料損壞
進階 SHA-256 雜湊比對 檔案被惡意替換
企業級 RSA/ECDSA 數位簽章 韌體來源偽造（中間人攻擊）
注意 ESP-IDF 原生支援 Secure Boot v2，可在 Bootloader 層級驗證韌體的數位簽章。啟用後，只有使用你的私鑰簽署的韌體才能被載入執行。這是商業產品的最高安全標準。

在本課程的實作中，我們將聚焦於 HTTP Basic Auth 與版本檢查這兩個最實用的防護層。在後續的進階課程中，你將學習如何整合 TLS 加密與韌體簽章，建立完整的企業級安全鏈。

實驗單元：Arduino OTA 與 Web OTA 原理

本單元將帶領你從「手動燒錄」進化到「物聯網自動化維護」。你將學習如何實作無線韌體更新 (OTA)，建立具備進度監測的前端介面，並理解硬體層級的 A/B 分區回滾安全機制。

🎯 學習目標

  1. 掌握 ESP32 分區表 (Partition Table) 的配置與原理。
  2. 實作 Web-based OTA 更新功能。
  3. 開發具備進度條 (Progress Bar) 的使用者介面。
  4. 理解並測試 A/B 分區的容錯與回滾機制。

🛠️ 任務說明

任務 1：分區表配置 (Partition Configuration)

為了支援 OTA，ESP32 需要至少兩個應用程式分區 (app0, app1)。

  • 修改 platformio.ini，指定使用自定義的分區表文件。
  • 編輯 ota_split.csv，正確劃分 app0, app1, otadata 與 nvs 空間。

任務 2：Web OTA 核心實作

在 src/main.cpp 中實作 WebServer 的 /update 路由處理。

  • 使用 Update.begin(), Update.write(), 與 Update.end() 處理上傳的二進位檔案。
  • 確保更新完成後設備能自動重啟。

任務 3：進度條視覺化介面

編輯 data/index.html，為 OTA 上傳頁面加入視覺化反饋。

  • 利用 XMLHttpRequest 的 progress 事件監控上傳進度。
  • 實作動態 CSS 進度條，並在更新成功後引導使用者重啟或自動跳轉。

任務 4：故障與回滾測試

模擬真實環境中可能發生的意外中斷。

  • 在上傳過程中（約 50%）中斷電源或網路。
  • 驗證設備是否能自動回滾至原本健康的版本。
  • 完成 docs/Rollback.md 中的實驗報告。

2. 作業任務部署細節

任務 1：「無線進化」基礎 Web OTA (Web-Based Wireless Update Lab)

  • Classroom 部署建議：
    • 路由實作建構：學員需實作 Update.begin(), Update.write(), 與 Update.end() 這套三位一體的核心指令。
    • 連通性驗證：要求學員在 esp32.local/update 介面成功上傳一段修改後的 Blink 程式。
    • 驗證要點：README 中需展示上傳成功後，設備自動重啟並進入新程式的日誌。

任務 2：「更新進度條」視覺化設計 (OTA Progress & UX Design Lab)

  • 目標：透過 JavaScript 監控二進位流傳輸狀態，建立專業的用戶反饋體驗。
  • Classroom 部署建議：
    • 事件監聽實作：使用 XMLHttpRequest 的 progress 事件，學員需計算 (e.loaded / e.total) 並實時更新 CSS Width。
    • 狀態引導設計：上傳完成後需觸發「系統重啟中...」的提示頁面，並在 3 秒後自動跳轉。
    • PR 提交內容：提交一段螢幕錄影或 GIF，展示進度條從 0% 到 100% 的平滑移動過程。

任務 3：「安全防線」故障與回滾測試 (Rollback & Security Audit Lab)

  • 目標：掌握 A/B 分區的底層邏輯，建立「硬體永遠能自癒」的專業工程觀念。
  • Classroom 部署建議：
    • 故障模擬挑戰：要求學員在上傳進度達到 50% 時故意拔掉開發板電源。
    • 恢復力檢測：學員需重啟設備，並證明 ESP32 能自動切換回原本健康的分區 (Rollback)，而非停在損毀的分區中。
    • 技術原理分析：在報告中解釋 otadata 分區在這次回滾實驗中扮演的角色。
