🤝 1. 非同步握手：從 Promise 鏈到 Async

藍牙連線是一個充滿不確定性且緩慢的過程。使用現代語法 async/await 代替繁瑣的 .then() 地獄，讓複雜的硬體連線邏輯具備極高的高可讀性。

  • 訂閱推送 (Notify)：開啟底層自動通知，實現零延遲的非阻塞式數據回填。

進階課程 15 - 單元 1 Web Bluetooth 非同步與連線優化

Web Bluetooth 非同步連線與自癒機制 (adv-15-unit-ble-async)

本單元是通訊底層技術的進階挑戰。你將學習如何將「非同步編程 (Async/Await)」與「實體硬體通訊」深度結合，解決無線通訊中不可避免的 斷線、延遲與併發衝突 問題，打造工業級穩定的「自癒式控制鏈」。

🎯 學習目標

  1. 非同步管道重構：將複雜的 Callback 鏈轉化為線性、易讀的 async/await 代碼。
  2. 自癒式防禦機制：實作指數退避 (Exponential Backoff) 算法，確保設備斷線後能自動回復連線。
  3. 串流流控優化：透過指令節流 (Throttle) 與併發隔離，維持高頻通訊下的系統流暢度。

🛠 任務說明

任務 1：線性連線序列重構 (Async Pipeline Reconstruction)

  • 目標：重構 src/ble_manager.js。
  • 內容：
    • 將原本充滿 .then() 的連線邏輯（裝置選取 ➡ GATT 連結 ➡ 特徵訂閱）改造成現代化的 async/await 序列。
    • 異常防禦：必須處理 DOMException（例如使用者取消選取裝置），確保 UI 能顯示友善提示。
  • 驗證：代碼是否具備高度可讀性？錯誤處理是否覆蓋了所有階段？

任務 2：自癒式自動重連 (Self-Healing Redundancy)

  • 目標：實作斷線自動恢復機制。
  • 內容：
    • 在 src/lib/auto_heal.js 中實作 exponentialBackoff() 邏輯。
    • 監聽 gattserverdisconnected 事件。
    • 策略：第 1 次失敗等 1s，第 2 次 2s，第 3 次 4s... 依此類推，避免在訊號不穩時過度佔用資源。
  • 驗證：手動關閉電源後，系統是否能在電源恢復後「悄無聲息」地自動找回數據流？

任務 3：並行流控優化 (Duplex Stream Optimization)

  • 目標：解決「高頻指令發送」導致的通訊死結。
  • 內容：
    • 指令節流 (Throttle)：在指令發送處實作 50ms 節流。
    • 併發隔離：確保數據 Notify 接收與指令寫入不會互相阻塞。
    • 性能記錄：在 README.md 或指定報告中記錄 Chrome DevTools 的 Performance Profile。
  • 驗證：在大規模控制寫入時，頁面是否仍能維持 60 FPS 的渲染效能？

📂 程式庫結構

  • src/ble_manager.js：核心 BLE 管理類別（待重構）。
  • src/lib/auto_heal.js：自動重連策略實作區。
  • docs/STRESS_REPORT.md：壓力測試報告（需記錄自癒恢復時間與重試次數）。
  • tests/connection.spec.js：非同步健全性測試。

  • 📂 src/lib/auto_heal.js：重連邏輯區。預留 exponentialBackoff() 函式骨架，讓學員實作重試次數與延遲時間的乘法遞增邏輯。

2. 作業任務部署細節

任務 1：線性連線序列重構 (Async Pipeline Reconstruction Lab)

  • 目標：將混亂的 Callback 鏈轉化為直覺、可維護的線性代碼。
  • Classroom 部署建議：
    • 連線序列優化：學員需將「裝置選取 ➡ GATT 連結 ➡ 特徵訂閱」重構為單一的 await 序列。
    • 異常防禦實作：特別要求處理 DOMException（當使用者關閉藍牙選取視窗時），確保 UI 顯示友善提示而非沈默崩潰。
    • 驗證要點：代碼是否具備高度可讀性？錯誤處理是否徹底覆蓋了所有非同步階段？

任務 2：自癒式自動重連 (Self-Healing Redundancy Lab)

  • 目標：實踐「硬體隨時會斷線」的防禦性機制。
  • Classroom 部署建議：
    • 斷線截獲：掛載 gattserverdisconnected 事件監聽。
    • 指數退避算法：實施重連策略。第 1 次失敗等 1s，第 2 次 2s，第 3 次 4s... 避免在訊號不穩時過度佔用 CPU 資源（避免 Spectrum Drowning）。
    • Demo 影片提交：要求學員上傳影片：手動關閉機器人電源，數秒後恢復，系統是否能自動找回數據流而無需重新點擊按鈕或刷新頁面。

任務 3：並行流控優化戰 (Duplex Stream Optimization Lab)

  • 目標：解決「高頻指令發送」導致「遙測數據卡頓」的併發衝撞。
  • Classroom 部署建議：
    • 指令節流 (Throttle) 部署：在發送指令處實作 50ms 節流。避免 BLE TX 緩衝區溢位導致的通訊死結。
    • 併發處理隔離：確保數據 Notify 接收與指令寫入使用獨立的微任務隊列 (Microtask Queues)，維持介面渲染的 60 FPS 流暢度。
    • 性能證據：在 README 中提交 Chrome DevTools 的 Performance Profile，證明在大規模控制寫入時，頁面沒有產生 Long Tasks (延時任務)。
