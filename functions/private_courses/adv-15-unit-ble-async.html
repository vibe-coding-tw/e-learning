🤝 1. 非同步握手：從 Promise 鏈到 Async

在 Web Bluetooth API 中，每一步操作——從請求裝置、建立 GATT 連線到訂閱特徵通知——都是非同步的 I/O 操作。傳統的 .then() 鏈式寫法在面對多階段連線管道時，會迅速堆疊成難以維護的「回呼地獄 (Callback Hell)」。現代 JavaScript 的 async/await 語法讓我們能以線性、同步般的方式撰寫非同步邏輯，大幅提升程式碼的可讀性與錯誤處理能力。

為什麼 BLE 連線必須是非同步的？

藍牙低功耗 (BLE) 的連線過程涉及多個硬體層級的交握：射頻掃描、GAP 廣播解析、GATT 服務發現、特徵值訂閱等。每一步都依賴無線通訊的物理回應，耗時從數十毫秒到數秒不等。如果使用同步阻塞式呼叫，整個瀏覽器介面將在連線期間完全凍結，使用者無法進行任何互動。

async/await 的核心價值在於：它讓非同步程式碼看起來像同步程式碼一樣線性執行，同時保持了 JavaScript 事件迴圈的非阻塞特性。搭配 try/catch 結構，我們可以優雅地捕捉連線過程中任何階段的異常。

備註 Web Bluetooth API 目前僅在 Chrome、Edge 等 Chromium 核心瀏覽器中獲得支援，且必須在安全環境 (HTTPS 或 localhost) 下使用。使用者必須透過手勢觸發（如點擊按鈕）才能啟動裝置掃描，這是瀏覽器安全策略的強制要求。

連線管道的三階段序列

完整的 BLE 連線管道可拆分為三個嚴格依序執行的階段。每個階段都必須等待前一階段成功完成後才能繼續：

階段API 方法功能說明常見錯誤
1. 裝置選取 navigator.bluetooth.requestDevice() 顯示系統配對視窗，依據 UUID 或名稱過濾目標裝置 使用者取消選取 → NotFoundError
2. GATT 連結 device.gatt.connect() 建立底層 ATT 連線，獲取 GATT Server 物件 裝置離線 → NetworkError
3. 特徵訂閱 characteristic.startNotifications() 啟用 BLE Notify 機制，裝置端主動推送數據 UUID 不符 → NotFoundError

使用 async/await 重構連線管道

以下範例展示了如何將三階段連線管道重構為線性化的 async/await 程式碼。每個 await 暫停點都代表一次等待硬體回應的時機：

async function connectBLE() {
  try {
    // 階段 1：使用 Service UUID 過濾並選取目標裝置
    const device = await navigator.bluetooth.requestDevice({
      filters: [{ services: ['0000ffe0-0000-1000-8000-00805f9b34fb'] }]
    });

    // 階段 2：建立 GATT 連線並取得 Primary Service
    const server = await device.gatt.connect();
    const service = await server.getPrimaryService(SERVICE_UUID);
    const char = await service.getCharacteristic(CHAR_UUID);

    // 階段 3：啟用 Notify，開始接收遙測數據流
    await char.startNotifications();
    char.addEventListener('characteristicvaluechanged', onTelemetry);

  } catch (err) {
    if (err.name === 'NotFoundError') {
      showToast('使用者取消了裝置選取');
    } else {
      showToast(`連線失敗：${err.message}`);
    }
  }
}
提示 在 catch 區塊中，務必區分 NotFoundError（使用者主動取消）與 NetworkError（裝置離線）。前者是預期行為，應顯示友善提示而非錯誤訊息；後者則應觸發重連邏輯。

Promise 鏈 vs. Async/Await 對比

以下對比展示了相同連線邏輯在兩種寫法下的差異。隨著連線階段增加，.then() 鏈的巢狀層級呈指數增長，而 async/await 始終保持線性結構：

.then() 鏈式寫法
回呼函式逐層巢狀，錯誤處理分散在每一個 .catch() 中，邏輯流難以追蹤。當需要在中間階段存取前幾階段的變數時，必須透過外部閉包或額外的變數傳遞，大幅降低可維護性。
async/await 線性寫法
所有變數（device、server、service）都在同一個函式作用域內可用，單一 try/catch 統一處理所有階段的異常，程式碼如同閱讀操作手冊般清晰直觀。
警告 requestDevice() 必須由使用者手勢（click、touch）直接觸發。在非手勢驅動的上下文中呼叫——例如 setTimeout 回呼、DOMContentLoaded 事件或 WebSocket 訊息處理器——會被瀏覽器安全策略阻擋並拋出 SecurityError。

🛡️ 2. 自癒式架構：指數退避重連機制

在真實環境中，BLE 連線隨時可能因為訊號干擾、裝置斷電、超出有效距離或使用者意外操作而中斷。一個生產級的控制系統不能要求使用者手動刷新頁面重新連線——它必須具備自癒能力：在連線中斷後自動偵測、等待適當時機、然後悄無聲息地恢復數據流。

監聽 GATT 斷線事件

Web Bluetooth API 提供了 gattserverdisconnected 事件，當底層 GATT 連線因任何原因中斷時（包括裝置端主動斷開、射頻干擾、超距等），此事件會被觸發。這是實作自動重連的入口點：

// 掛載斷線監聽器（在首次連線成功後立即設定）
device.addEventListener('gattserverdisconnected', () => {
  console.warn('GATT 連線已中斷，啟動自動重連...');
  updateUI('disconnected');
  exponentialReconnect(device);
});
警告 gattserverdisconnected 事件在使用者主動呼叫 device.gatt.disconnect() 時也會觸發。在重連邏輯中必須區分「意外斷線」與「使用者主動斷線」。建議設置一個 userInitiatedDisconnect 旗標，在主動斷線前將其設為 true，避免觸發不必要的重連。

指數退避演算法 (Exponential Backoff)

當連線中斷後，最直觀的做法是立即嘗試重連。但如果中斷原因是環境干擾或裝置斷電，立即重試幾乎必定失敗，而持續的重試嘗試會佔用 BLE 射頻資源（Spectrum Drowning），干擾周圍其他 BLE 裝置。指數退避策略讓每次失敗後的等待時間呈倍數增長，在「快速恢復」與「資源節約」之間取得平衡：

第 2 次2,000 ms1000 × 213.0s

第 4 次8,000 ms1000 × 2315.0s

完整重連實作

async function exponentialReconnect(device, maxRetries = 5) {
  const BASE_DELAY = 1000;
  const MAX_DELAY = 16000;

  for (let attempt = 0; attempt < maxRetries; attempt++) {
    const delay = Math.min(BASE_DELAY * 2 ** attempt, MAX_DELAY);
    const jitter = Math.random() * delay * 0.2;
    await sleep(delay + jitter);

    try {
      await device.gatt.connect();
      await resubscribeNotifications(device);
      console.log(`重連成功 (第 ${attempt + 1} 次嘗試)`);
      return;
    } catch (err) {
      console.warn(`重連失敗 #${attempt + 1}：${err.message}`);
    }
  }
  showToast('裝置無法連線，請手動重試');
}
提示 加入 Jitter（隨機抖動）是工業實踐的最佳做法。當多台裝置同時斷線後嘗試重連（例如基地台重啟），如果所有裝置都在完全相同的時刻發起重連請求，會造成「驚群效應 (Thundering Herd)」。加入 ±20% 的隨機延遲可有效分散重連請求，降低射頻碰撞機率。

重連後的狀態恢復

成功重連 GATT 後，必須重新訂閱 Notify 特徵並恢復所有事件監聽器。這是因為 BLE 的 CCCD (Client Characteristic Configuration Descriptor) 設定在斷線後會被裝置端清除。resubscribeNotifications() 函式需要重新執行 getPrimaryService()、getCharacteristic() 和 startNotifications() 的完整序列。

重要 自動重連僅適用於先前已通過 requestDevice() 配對的裝置。Web Bluetooth 的安全模型不允許在背景中自動掃描新裝置。因此請務必在首次連線時保存 device 物件的參考，供後續重連使用。

⚡ 3. 非同步並行：雙向流控處理

在控制系統中，數據流是雙向的：瀏覽器需要高頻地向 ESP32 發送控制指令（Write），同時也需要接收來自裝置的遙測數據（Notify）。這兩條數據通道共享同一條 BLE 連線，如果不加以管理，高頻的 Write 操作可能佔滿傳輸佇列，導致 Notify 回傳延遲甚至遺失。實現穩健的雙向流控 (Duplex Flow Control) 是打造流暢控制體驗的最後一塊拼圖。

Write 與 Notify 的衝突本質

BLE GATT 的底層使用同一條 L2CAP 通道承載所有操作。Write（主機→周邊）和 Notify（周邊→主機）在連線事件 (Connection Event) 中交替執行。當 Write 請求過於密集時，連線事件的時間槽會被 Write 佔滿，Notify 封包只能排隊等待下一輪連線事件。

操作類型方向GATT 方法確認機制適用場景
Write with Response 主機 → 周邊 writeValueWithResponse() 等待 ATT 確認 關鍵指令（煞車、模式切換）
Write without Response 主機 → 周邊 writeValueWithoutResponse() 無（Fire & Forget） 高頻搖桿數據
Notify 周邊 → 主機 startNotifications() 無 遙測資料、感測器數值
備註 對於高頻搖桿控制指令，建議使用 writeValueWithoutResponse()。此方法跳過 ATT 層的確認握手，在單一連線事件中可發送更多封包。代價是無法確認裝置是否收到——但對於每秒 20-50 次的搖桿更新而言，偶爾遺失一筆指令幾乎不會被感知。

智慧節流 (Throttling) 機制

搖桿的 pointermove 事件觸發頻率可達 60-120Hz，但 BLE 的有效傳輸頻率通常只有 20-50Hz。直接將每一次搖桿事件都轉為 Write 請求，會迅速塞滿 BLE 的傳輸佇列。節流機制的核心思想是：在固定時間窗口內只發送最新一筆數據，丟棄中間的過時數據。

let lastSend = 0;
const THROTTLE_MS = 30; // ~33Hz 發送頻率

function onJoystickMove(x, y) {
  const now = performance.now();
  if (now - lastSend < THROTTLE_MS) return;
  lastSend = now;

  const packet = packCommand(x, y);
  txChar.writeValueWithoutResponse(packet)
    .catch(err => console.warn('Write 失敗:', err));
}

併行初始化：Promise.all 加速握手

當 BLE 連線成功後，通常需要依序取得多個 Service 和 Characteristic。但如果這些操作之間沒有依賴關係，可以使用 Promise.all() 將它們並行化，大幅縮短初始化時間：

const [txChar, rxChar] = await Promise.all([
  service.getCharacteristic(TX_UUID),
  service.getCharacteristic(RX_UUID),
]);
await rxChar.startNotifications();
rxChar.addEventListener('characteristicvaluechanged', onTelemetry);
提示 將 Write 特徵 (TX) 與 Notify 特徵 (RX) 分成兩個不同的 GATT Characteristic，是實現穩健雙向流控的設計最佳實踐。兩個特徵可共用同一個 Service UUID，但使用不同的 Characteristic UUID。這種分離設計讓韌體端也能獨立管理收發佇列。
警告 Web Bluetooth 的 writeValueWithoutResponse() 內部仍有流量控制——當 BLE 控制器的發送佇列已滿時，此 Promise 會暫時掛起直到佇列有空位。如果在上一次 Write 完成前就發起新的 Write，可能導致 GATT operation already in progress 錯誤。建議使用序列化佇列或確保每次 Write 完成後才發送下一次。
