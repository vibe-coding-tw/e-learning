封包裡裝了什麼？

每當瀏覽器向 ESP32 發出一個控制指令，底層實際上是透過一封封 HTTP 報文（Message）在傳遞。 HTTP 報文分為請求（Request）與回應（Response）兩種。 理解報文的結構，是你調試「為什麼指令沒收到」或「回應格式不正確」的第一步。

HTTP 請求（Request）結構

一個完整的 HTTP 請求由三部分組成：請求行（Request Line）、 標頭（Headers）、以及可選的主體（Body）。 以下是一個控制 ESP32 馬達的典型 POST 請求：

POST /api/command HTTP/1.1
Host: 192.168.4.1
Content-Type: application/json
Content-Length: 30

{ "direction": "F", "speed": 200 }

請求行中的 POST 是 HTTP 方法（Method），/api/command 是資源路徑， HTTP/1.1 是協定版本。標頭區段提供了元資料——告訴伺服器「我送的是 JSON 格式、長度 30 位元組」。 空行之後就是實際的 Payload（載荷）。

常見 HTTP 方法

方法語義有 BodyESP32 應用範例
GET讀取資源✕取得感測器數值、系統狀態
POST建立 / 執行動作✓發送馬達指令、更新設定
PUT完整更新資源✓覆寫整組設定檔
DELETE刪除資源✕清除日誌、重置設定

HTTP 回應（Response）結構

ESP32 收到請求後，必須回傳一個結構正確的 HTTP 回應。回應同樣包含狀態行、 標頭與主體三部分：

HTTP/1.1 200 OK
Content-Type: application/json
Connection: keep-alive

{ "status": "ok", "speed": 200 }

狀態碼（Status Code）是回應中最關鍵的資訊。它用三位數字告訴客戶端「請求的處理結果」：

狀態碼含義ESP32 使用時機
200OK — 成功指令已執行、數據已回傳
400Bad Request — 請求格式錯誤缺少必要參數或 JSON 解析失敗
404Not Found — 路徑不存在存取了未定義的路由
500Internal Error — 伺服器內部錯誤硬體操作異常或記憶體不足
備註 在 ESP32 上實作 API 時，務必為所有回應加上正確的 Content-Type 標頭。 若回傳 JSON 卻沒有宣告 application/json，瀏覽器可能會將回應當作純文字處理， 導致前端的 fetch().json() 解析失敗。

請求生命週期

當你在瀏覽器輸入 ESP32 的 IP 位址按下 Enter 後，看似瞬間完成的「載入網頁」， 背後其實經歷了多個精密的網路階段。理解這條完整的請求生命週期（Request Lifecycle）， 你才能在控制延遲時精準定位「瓶頸到底出在哪一層」。

四大階段詳解

階段一：DNS 解析（Domain → IP）

如果你使用的是網域名稱（如 mycar.local），瀏覽器必須先透過 DNS 查詢將名稱轉為 IP 位址。 在 ESP32 的 AP 模式下，由於客戶端直接使用 192.168.4.1 連線，這個階段通常耗時 0ms。 但如果啟用了 mDNS（如 MDNS.begin("mycar")），則首次解析可能需要 20–100ms。

階段二：TCP 三向交握（Three-Way Handshake）

HTTP 基於 TCP 協定，因此在傳送任何資料前，客戶端與 ESP32 必須先建立可靠的連線通道。 這個過程需要三個封包來回：SYN → SYN-ACK → ACK。 在區域網路中，這個步驟通常耗時 1–5ms；但如果 Wi-Fi 訊號不佳（RSSI < -75dBm）， 封包重傳可能將耗時拉高到 50ms 以上。

階段三：HTTP 請求與回應

TCP 連線建立後，瀏覽器發送 HTTP 請求，ESP32 的 Web Server 接收、處理、並回傳回應。 從「請求送出」到「收到回應第一個位元組」的時間稱為 TTFB（Time To First Byte）。 TTFB 直接反映了 ESP32 的處理效能——如果你的 handler 中有 delay() 或耗時的感測器讀取， TTFB 就會飆升。

階段四：內容下載與渲染

瀏覽器收到回應後，開始下載 HTML/CSS/JS 並渲染畫面。 對 ESP32 而言，關鍵是回應的Content-Length—— 如果 HTML 檔案過大（超過 10KB），在低頻寬的 Soft-AP 環境下可能需要數百毫秒才能傳輸完畢。

階段典型耗時（區網）瓶頸來源優化方向

提示 打開 Chrome 或 Edge 的 F12 開發者工具 → Network 面板， 點擊任一請求即可看到 Timing（時序）標籤，裡面會將每個階段的耗時以瀑布圖（Waterfall）呈現。 這是診斷 ESP32 通訊延遲最有力的工具。

用 F12 擷取時序資訊

在瀏覽器開發者工具的 Network 面板中，每筆請求的 Timing 標籤都會顯示以下關鍵指標。 學會閱讀這些數字，你就能從「覺得慢」進化到「知道哪裡慢」：

  • Stalled / Queueing：瀏覽器排隊等待的時間（通常因為連線數已滿）
  • DNS Lookup：名稱解析耗時
  • Initial Connection：TCP 交握 + TLS 協商（HTTP 無 TLS 則只有 TCP）
  • Waiting (TTFB)：等待伺服器回應第一個位元組
  • Content Download：下載回應本體的時間
備註 若你在 AP 模式下使用 IP 位址連線，DNS Lookup 應該始終為 0ms。 如果看到非零值，表示瀏覽器可能在嘗試反向 DNS 查詢——這可以透過直接輸入 http://192.168.4.1（加上 http:// 前綴）來避免。

Keep-Alive 的力量

在預設情況下，每次 HTTP 請求完成後，TCP 連線就會被關閉（Connection: close）。 下一次請求必須重新執行完整的三向交握。當你的應用需要高頻率通訊時—— 例如每 50ms 發送一次馬達控制指令——這些重複的握手開銷將嚴重拖慢反應速度。

Keep-Alive（長連線）的核心思想很簡單：一次握手，多次通訊。 TCP 連線在第一次請求完成後不關閉，後續請求直接複用同一條連線通道。 這意味著從第二次請求開始，TCP 交握的時間成本降為零。

一發一斷 vs. 連線複用

以下對比清楚展示了兩種模式在高頻通訊場景下的效能差異。 假設 TCP 交握需要 3ms，每次 HTTP 往返需要 2ms：

指標Connection: closeConnection: keep-alive
50 次請求的 TCP 交握次數50 次1 次
TCP 交握總耗時150ms (50 × 3ms)3ms
HTTP 往返總耗時100ms (50 × 2ms)100ms (50 × 2ms)
50 次請求總耗時250ms103ms
節省時間—約 59%
提示 在 ESPAsyncWebServer 中，Keep-Alive 是預設啟用的，你不需要額外設定。 但如果你使用的是同步版的 WebServer.h，需要手動在回應標頭中加入 Connection: keep-alive。

在 ESP32 上啟用 Keep-Alive

以下範例展示如何在同步式 Web Server 中手動設定 Keep-Alive 標頭。 注意 Content-Length 在長連線中特別重要——它告訴客戶端「這筆回應到哪裡結束」， 否則客戶端無法判斷何時開始發送下一個請求。

server.on("/api/status", HTTP_GET,
  [](AsyncWebServerRequest *req) {
    String json = "{\"heap\":" + String(ESP.getFreeHeap()) + "}";

    AsyncWebServerResponse *res =
      req->beginResponse(200, "application/json", json);
    res->addHeader("Connection", "keep-alive");
    req->send(res);
});

TTFB：診斷 ESP32 效能的黃金指標

TTFB（Time To First Byte，首位元組時間）衡量的是從「請求送出」到「收到回應第一個位元組」的延遲。 在 ESP32 上，TTFB 直接反映了你的 handler 函式的執行效率：

  • TTFB < 5ms：優秀。handler 中沒有阻塞操作。
  • TTFB 5–50ms：可接受。可能有感測器讀取或少量計算。
  • TTFB > 100ms：需要優化。檢查是否有 delay()、同步 I2C 讀取或大量字串拼接。
注意 ESP32 的 TCP 連線數量有限（預設為 最多 4–5 條同時連線）。 如果多個客戶端同時使用 Keep-Alive 佔滿連線池，新的請求將被拒絕。 在多使用者場景中，應設定合理的 Keep-Alive: timeout 值（如 5–10 秒）， 讓閒置連線能被自動回收。

掌握了 Keep-Alive 與 TTFB 的概念後，你已經具備了對物聯網通訊進行「量化診斷」的能力。 差 200ms 的延遲在普通網頁上或許感覺不出來，但對一台高速移動的遙控車而言， 那就是「順利過彎」和「直接撞牆」的區別。

2. 作業任務部署細節

任務 1：「封包捕手」F12 請求時序解析 (Network Diagnostic Lab)

  • 目標：透過瀏覽器開發者工具將抽象的網路階層具象化，識別出「連線階段」與「資料傳輸階段」的性能界線。
  • Classroom 部署建議：
    • 時序解剖：存取 ESP32 網頁並擷取時序圖。
    • 瓶頸診斷：核心關鍵點。學員必須識別出 TTFB 代表的是「ESP32 的處理時間」，而 TCP Connect 代表的是「連線握手開銷」。
    • 驗證要點：README 中是否準確標註了各階段的毫秒數並提出了延遲的主因假設？

任務 2：「自建路由」RESTful API 實作 (API Architect Lab)

  • 目標：建立符合現代 Web 開發規範的通訊接口，掌握 JSON 標頭配置與狀態碼語義。
  • Classroom 部署建議：
    • 規範實施：實作 /api/status 回傳 JSON（如 { "free_heap": 12345 }）。
    • 命令接收：實作 POST /api/command 解析 Body 指令。
    • 標準化回應：成功時回傳 200 OK，若指令格式不對則回傳 400 Bad Request。
    • 驗證要點：藉由 Postman 或網頁測試，確認標頭中 Content-Type 是否正確宣告為 application/json。

任務 3：「連線優化師」Keep-Alive 效能對比 (Performance Lab)

  • 目標：親手驗證 TCP 三向握手的成本，學習在「高頻通訊」場景下如何透過保持連線來節省超過 50% 的延遲。
  • Classroom 部署建議：
    • 壓力測試：執行 50 次高頻控制封包。
    • 數據量化：對比「一發一斷」與「長連線」的總時間差。
    • 原理深挖：核心思想考題。請學員在報告中說明：為什麼長連線在第二次以後的請求中，不再出現 TCP Connect 的耗時？
    • 最終 Commit：封裝具備效能優化意識的通訊代碼並 Push。

3. 通訊診斷與效能優化點評標準 (Diagnostics & Latency Standards)

此單元的價值在於 「對通訊細節的洞察力與對網路協定成本的量化控制能力」：

  • [ ] 時序解析正確性：是否能正確區分連線建立時間 (TCP) 與伺服器反應時間 (TTFB)？
  • [ ] RESTful 專業度：路由設計是否簡潔？狀態碼的選用是否符合開發慣例？
  • [ ] 優化結論驅動：是否能從數據中得出「Keep-Alive 對即時控制（如遙控車）至關重要」的工程結論？
