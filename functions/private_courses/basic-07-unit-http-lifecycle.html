數據的一趟旅程：
HTTP 封包結構與生命週期

拆解 HTTP 通訊細節，跟隨數據包在無線森林中的跳轉，掌握網路效能的診斷藝術。

📦 1. HTTP 報文結構：封包裡裝了什麼？

無論是網頁瀏覽還是無人車控制，底層都是透過一封封「HTTP 報文 (Message)」在傳遞訊息。

REQ

Request (請求)

包含 Method (GET/POST)、Headers (自定義標頭) 與 Body (控制指令 JSON)。

RES

Response (回應)

關鍵在於 Status Code (200 OK, 404, 500) 與回應正文 (狀態數據)。

Protocol Anatomy

POST /api/move HTTP/1.1

Host: 192.168.4.1

Content-Type: application/json

{ "dir": "F", "speed": 100 }

// 這就是控制馬達的封包 Payload

🚀 2. 請求生命週期：一趟數據的旅程

1

DNS / IP

將名稱轉為 IP，找到正確的門牌號碼。

2

TCP Connect

建立可靠的三向握手，確保連線穩定通訊。

3

Send Payload

正式傳輸控制指令 Body，並等待伺服器解析。

4

Render Result

接收回應狀態碼，前端頁面更新 UI 狀態。

⚡ 3. 效能優化：Keep-Alive 的力量

每次發送指令都要握手 (Handshake) 嗎？頻繁控制時，傳統的「一發一斷」會浪費大量時間在建立連線上。

Optimized Connectivity

  • Connection: Keep-Alive：讓 TCP 連線保持暢通，後續 50 次指令只需握手 1 次。
  • TTFB (Time To First Byte)：觀察伺服器開始回應前的第一時間，這是診斷 ESP32 負載的黃金標準。
  • 批次通訊：在高頻控制(20Hz+)下，降低 JSON 轉義開銷是效能起飛的關鍵。

Latency Inspector

任務 2：「自建路由」：RESTful API 實作 (API Architect)

角色扮演：嵌入式 API 開發者 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 路由封裝實作：在 WebServer 中定義 /api/status (GET) 路徑，回傳結構化的系統運行 JSON 數據。
  2. 命令模式開發：實作 /api/command (POST) 路徑，接收並解析從網頁傳入的 JSON 控制指令。
  3. 協定標準化：所有回應必須正確配置 Content-Type: application/json 標頭，並按結果發送 200 或 400 狀態碼。
  4. 代碼提交：Push 具備完整路由處裡邏輯的代碼，並附帶 API 定義規格書。

任務 3：「連線優化師」：Keep-Alive 效能對比 (Performance Lab)

角色扮演：系統效能優化專家 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 實驗架構搭建：在代碼中切換處理機制，對比 Connection: Close 與 Connection: Keep-Alive 的通訊表現。
  2. 壓力數據擷取：模擬 10 秒內連續連發 50 次控制包，量化總傳輸耗時的差異。
  3. 優化結論筆記：在 README 中彙整數據對比表，並根據 TCP 握手次數的減少，解釋 Keep-Alive 的具體加速原理。
  4. 正式提交：上傳實驗數據與結論，並標註該優化在物聯網賽車即時控制中的關鍵性。
