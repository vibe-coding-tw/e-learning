課程單元 基礎 6 - 網路安全基礎

CORS 與安全性基礎 (CORS & Web Security)

本程式庫是關於如何在 ESP32 上正確處理跨來源資源共用 (CORS) 與網路安全標頭的練習。你將學習如何診斷瀏覽器的跨域封鎖，並實作伺服器端的手段來建立安全且標準的通訊。

任務目標

任務 1：「邊界診斷員」CORS 錯誤追蹤 (CORS Trace Lab)

  • 目標：學會使用瀏覽器開發者工具 (DevTools) 定位跨域阻斷原因。
  • 要求：
    • 開啟 web/index.html，觀察並截圖 Console 顯示的 CORS 報錯資訊。
    • 在說明文件中詳細說明「同源政策 (SOP)」的三個判定要素。

任務 2：「開發期繞過方案」分析 (Legacy Bypass Lab)

  • 目標：了解臨時的修正方案及其風險。
  • 要求：
    • 測試使用瀏覽器 CORS 插件繞過限制。
    • 在 docs/BYPASS_RISK.md 中撰寫風險分析，說明為何這在生產環境中是危險的。

任務 3：「硬體安全官」ESP32 標頭注入 (Hardware Sentry Lab)

  • 目標：從伺服器端解決跨域需求。
  • 要求：
    • 在 src/main.cpp 中實作 Access-Control-Allow-Origin 標頭。
    • 實作 OPTIONS 預檢請求 的處理邏輯。
    • 關閉插件後，確保 web/index.html 依然能成功控制硬體。

提交要求

  1. 程式碼：完成 src/main.cpp 中的標頭注入與 OPTIONS 處理。
  2. 安全性計畫：完成 assets/security_plan.md 中的設計思考。
  3. 風險評估：完成 docs/BYPASS_RISK.md 中的心得紀錄。
  4. 更新程式庫：完成後請 commit 並 push 你的更改。

[!TIP] 記得在 platformio.ini 中確認已加入 ESPAsyncWebServer 與 AsyncTCP 的依賴。

針對 basic-06-unit-cors-security (CORS 與安全性基礎) 單元，這是學員從「區域網路開發」跨向「雲端網頁整合」最常遇到的 第一座技術大山（跨網域讀取封鎖）。它要求學員具備 通訊安全與合規 (Compliance) 的思維，學習為何瀏覽器會保護使用者，並掌握如何在硬體端正確處理「預檢請求 (Preflight)」以達成安全的跨平台互動。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template 程式庫) 配置建議

CORS 作業的核心在於「診斷與配置」，範本應結合前端網頁與後端韌體，建議包含：

  • 📂 src/main.cpp：ESP32 標頭注入區。學員需在此實作 Access-Control-Allow-Origin 標頭的配置，並處理 HTTP OPTIONS 方法的嚮應，確保「敲門機制」運作正常。
  • 📂 web/index.html：故障診斷網頁。提供一個發送 fetch() 請求到 ESP32 IP 地址的簡單網頁，初期這會因為跨域限制而報錯。
  • 📂 README.md：安全診斷報表。要求學員截圖瀏覽器 Console 顯示的紅字報錯資訊，並詳細說明「同源政策 (SOP)」的三個判定要素（協議、網域、埠）。
  • 📂 docs/BYPASS_RISK.md：開發風險評估。學員需記錄使用「CORS 繞過插件」的心得，並解釋為何這在生產環境 (Production) 中是極大的安全漏洞。

2. 作業任務部署細節

任務 1：「邊界診斷員」CORS 錯誤追蹤 (CORS Trace Lab)

  • 目標：學會使用瀏覽器開發者工具 (DevTools) 定位跨域阻斷原因，理解瀏覽器的防禦機制。
  • Classroom 部署建議：
    • 報錯模擬：開啟 index.html，點擊控制鈕。此時瀏覽器會因 ESP32 未回傳 CORS 標頭而阻斷連接。
    • 證據採樣：學員需擷取 Console 與 Network Tab 的詳細資訊。
    • 驗證要點：README 中需正確解釋：為什麼 http://localhost 存取 http://192.168.x.x 會被判定為跨域？

任務 2：「開發期繞過方案」代理與插件測試 (Legacy Bypass Lab)

  • 目標：了解臨時的修正方案，並建立對開發工具與真實環境差異的認知。
  • Classroom 部署建議：
    • 插件繞過：安裝瀏覽器 CORS 插件並開啟，確認網頁能成功控制硬體。
    • 安全評析：核心思想考題。撰寫報告說明：為什麼如果我們要求所有使用者都安裝這款插件是不合理且危險的？
    • 驗證要點：提交一份風險分析，展現對網路安全邊界的理解。

任務 3：「硬體安全官」ESP32 標頭注入 (Hardware Sentry Lab)

  • 目標：實作底層標頭注入技術，徹底從伺服器端解決跨域需求，這是開發物聯網控制台的工業標準。
  • Classroom 部署建議：
    • 標頭實作：使用 DefaultHeaders::Instance().addHeader("Access-Control-Allow-Origin", "*");（或其他特定 Origin）。
    • OPTIONS 預檢處理：關鍵技術點。實作一個 Handler 專門回傳成功的 OPTIONS 請求（HTTP 200/204），否則複雜請求（如帶 Header 的 POST）依然會失敗。
    • 閉環驗證：關閉瀏覽器插件，確認 index.html 依然能流暢控制硬體。
    • 最終 Commit：Push 具備完整標頭配置的韌體專業版本。

3. 網路安全與標頭合規點評標準 (Security Headers & Preflight Standards)

此單元的價值在於 「對 Web 規格的尊重與對伺服器端配置的精確掌控」：

  • [ ] 標頭配置精確度：Access-Control-Allow-Origin 是否拼寫正確且放置在正確的 Handler 中？
  • [ ] 預檢請求處理：是否理解並實作了 OPTIONS 方法的嚮應？（這是跨域封鎖最常見的死穴）。
  • [ ] 防禦性意識：是否理解並非所有 CORS 標頭都應設為 *，並具備針對特定域名開放的維護意識？
