基礎 6 - 網路安全

瀏覽器的安全邊界：
CORS 與硬體防禦策略

掌握跨來源資源共用 (CORS) 的運作機制，為你的物聯網專案建立正確的安全屏障。

🛑 1. 什麼是 CORS？瀏覽器的邊界感

CORS (Cross-Origin Resource Sharing) 跨來源資源共用，是一套讓瀏覽器決定是否允許網頁讀取另一個網域資料的防禦機制。

Same-Origin Policy

為了防止惡意網站偷取隱私（如銀行 Cookie），瀏覽器預設禁止「網域 A」的腳本去讀取「網域 B」的回傳內容。

判定標準：協議 (http)、網域 (domain)、連接埠 (port) 只要有一個不同，即為「跨來源」。

Origin Matching Quiz

http://vibe.car/a.js
http://vibe.car/b.js Same
https://vibe.car
http://vibe.car Cross
http://192.168.1.1:80
http://192.168.1.1:8080 Cross

🙋‍♂️ 2. 預檢請求：敲門機制 (Preflight Request)

當發送「複雜」請求（如 POST 或自定義 Header）時，瀏覽器會像個禮貌的訪客，先發送一個 OPTIONS 請求去「敲門」。

The Knock

OPTIONS 方法

瀏覽器自動發起，詢問：「我可以進來嗎？我打算使用這類標頭」。

The Ticket

CORS Header

伺服器需回傳 Allow-Origin。若無，瀏覽器直接阻斷通訊。

ESP32 CONFIG 
// 解決 ESP32 的 CORS 封鎖：
DefaultHeaders::Instance().addHeader(
  "Access-Control-Allow-Origin", "*"
);
DefaultHeaders::Instance().addHeader(
  "Access-Control-Allow-Methods", "GET, POST, OPTIONS"
);

🔐 3. 硬體安全防禦思維

能通訊只是第一步，能「安全通訊」才是專業。在開放的網路環境下，防禦性編程是物聯網的核心。

Defense Strategy

  • 權限最小化：生產環境中，不要使用 * (星號)，只開放必要的 Trusted Origins。
  • 指令簽章：利用雜湊效驗 (Hash Key) 確保指令在傳輸過程中未被中間人竄改。
  • 敏感資料遮蔽：不在前端網址傳送 API Key，所有關鍵密碼應封裝在 POST Body 或更安全的 Cookie 中。

Warning Zone

"CORS 不是真正的防火牆，它是瀏覽器保護「使用者」的一種手段，後端仍需實作驗證。"

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。

  4. 判定分析：在 README 中清晰解釋「協議、網域、連接埠」三者如何決定該次通訊被歸類為「跨來源」。

  1. 重點 1：瀏覽器不是你的對手 - 引導學員理解 CORS 的存在是為了保護使用者，而非阻礙開發。
