🏛️ 1. Web Speech API 的兩大支柱 (30 min)

現代瀏覽器內建了處理人類自然語言的能力。我們可以透過 SpeechRecognition 讓網頁跟機器人「聽」指令，並利用 SpeechSynthesis 讓它開口說話。

System Architecture

  • SpeechRecognition：STT (音轉文)。通常需要雲端引擎支援，具備語義語法分析。
  • SpeechSynthesis：TTS (文轉音)。利用端點裝置內建語音包，反應速度極快。

🔒 3. 安全與隱私限制 (20 min)

因為麥克風涉及使用者隱私，Web Speech API 受到瀏覽器的嚴格限制。它必須在 HTTPS 環境下運作，且每一次的「持續監聽」都需要明確的使用者授權。

Security Constraints

  • HTTPS Mandatory：非安全來源禁止訪問麥克風設備。
  • 自動結束機制：靜默過久會自動 `onend`。如何透過手動重啟維持機器人的「長聽」狀態？
  • 權限持久化：理解不同作業系統下，對於網頁權限彈窗的記憶策略。

🔄 45 min

「長效監聽整合」

將辨識器導入 FSM 生命週期。實作 onend 自動重啟與 continuous: true 以防中斷。說出「進入極速模式」後，同步更新 UI 全域狀態。

任務 1：模糊意圖解析 (Intent Mapping Lab)

角色扮演：VUI 體驗設計師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 詞典建立：在 intent_parser.js 定義語音與指令的映射表（包含多種同義詞）。
  3. 中間結果測試：開啟 interimResults，測量從開始說話到「初步辨識出指令」的反應毫秒數。
  4. README 紀錄：列出所有可接受的口語範例，並說明如何利用字串模糊比對提高系統容錯。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 語義覆蓋率：指令是否具備基本的同義詞容錯？（如：停、停止、不要動）。
  • [ ] 指令純淨度：是否能從複雜的句子中精準分離出關鍵動作，而不發生誤觸？

任務 2：語音對話回饋 (Two-way Feedback Lab)

角色扮演：人機互動工程師 | 難度：⭐️⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 狀態同步器：辨識成功後，利用 window.speechSynthesis 讓機器人「復誦」確認（如："收到指令，即將右轉"）。
  2. 置信度過濾：只在 Confidence > 0.85 時執行馬達寫入。
  3. HTTPS 診斷：在本地利用 mkcert 或隧道工具建立 HTTPS 測試環境，解決各瀏覽器的麥克風權限封鎖問題。
  4. Push Final：提交具備「聽/說」雙向互動能力的無人車控制面板。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 互動流暢性：語音回饋是否有與辨識衝突？（需考量回饋時是否應暫停辨識以防自說自聽）。
  • [ ] 環境適應力：在 README 錄製噪音環境下的辨識成功率測試結果。

任務 3：全天候自動辨識核心 (Continuous VUI Engine)

角色扮演：系統可靠度專家 | 難度：⭐️⭐️⭐️ | 估計時長：45 min

  1. 自動重啟路徑：在 onend 事件中，監測全域控制變數，實作無縫重啟邏輯。
  2. FSM 狀態橋接：將「語音控制中」作為 FSM 的一個子狀態。當偵測到關鍵字後，發送 VOICE_EVENT 事件。
  3. Push Final：提交具備工業級「自我修復」能力的連續語音監聽代碼。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 穩定性分析：針對辨識器偶發的 network 錯誤，是否有實作指數退避式的重連策略？

GitHub Classroom 詳細操作指南

本單元教學重點：讓學生理解網頁端語音識別的「雲端依賴性」。理解為什麼它適合做高層指令，但不適合做低延遲急停。

  1. 重點 1：事件與非同步 - 語音辨識是長時間運行的非同步流程。必須導正學員「一次性調用」的錯誤思維，建立「長駐狀態機」的概念。
  2. 重點 2：隱私與 Sandbox - 全面說明 HTTPS 的重要性。解釋為何瀏覽器會自動關閉不活躍的麥克風連結。
  3. 重點 3：意圖識別 (Intent Parsing) - 辨識結果通常是字串。如何從冗長的字串中準確撈出關鍵動作（Action）是軟體工程的核心。
