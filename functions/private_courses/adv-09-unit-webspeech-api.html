Web Speech API 的兩大支柱

Web Speech API 是 W3C 標準化的瀏覽器原生介面，讓前端開發者無需安裝額外套件即可在網頁中實現語音互動。它由兩個獨立的子系統組成：語音辨識（Speech Recognition）將人聲轉為文字，語音合成（Speech Synthesis）則將文字轉為人聲。

SpeechRecognition：語音轉文字 (STT)

SpeechRecognition（在 Chrome 中需使用 webkitSpeechRecognition）啟動後會開啟麥克風，將收音的音訊串流送至雲端語音辨識引擎（Google Speech Services），返回辨識出的文字結果。

// 基礎語音辨識設定
const recognition = new webkitSpeechRecognition();
recognition.lang = 'zh-TW'; // 繁體中文
recognition.continuous = true; // 持續監聽（不自動停止）
recognition.interimResults = true; // 取得中間結果

recognition.onresult = (event) => {
  const result = event.results[event.resultIndex];
  const transcript = result[0].transcript;
  const confidence = result[0].confidence;
  const isFinal = result.isFinal;
  console.log(`[${isFinal ? '最終' : '中間'}] ${transcript} (${confidence})`);
};

recognition.start();

SpeechSynthesis：文字轉語音 (TTS)

與 STT 不同，TTS 完全在本地端執行——利用作業系統內建的語音引擎與語音包。這意味著它不需要網路連線，延遲極低（通常 < 50ms），非常適合用於即時回饋。

// 語音合成：讓瀏覽器開口說話
function speak(text) {
  const utterance = new SpeechSynthesisUtterance(text);
  utterance.lang = 'zh-TW';
  utterance.rate = 1.2; // 語速（0.1~10）
  utterance.pitch = 1.0; // 音調（0~2）
  speechSynthesis.speak(utterance);
}

speak('收到指令，即將前進');

兩者的關鍵差異

面向 SpeechRecognition (STT) SpeechSynthesis (TTS)
處理位置 雲端（音訊上傳至 Google 伺服器） 本地端（使用系統語音引擎）
網路需求 必須有網路連線 離線可用
延遲 200~800ms（含網路往返） < 50ms
瀏覽器支援 僅 Chromium 系列完整支援 主流瀏覽器皆支援
隱私風險 音訊會傳至第三方伺服器 資料不離開裝置
備註 在 IoT 控制場景中，你可以建立一個「聽說閉迴路」：STT 接收使用者語音指令 → 解析意圖 → 透過 BLE 發送控制碼 → TTS 回報執行狀態。這種雙向語音互動大幅提升了操控的安全性與使用者體驗。

事件驅動與非同步生命週期

SpeechRecognition 是一個典型的事件驅動非同步 API——你無法同步地「呼叫一個函式取得辨識結果」。取而代之的是，你註冊一系列事件處理器，在辨識過程的不同階段接收通知。深入理解這些事件的觸發時機與順序，是建構穩定語音控制系統的核心基礎。

生命週期事件一覽

事件 觸發時機 常見用途
onstart 辨識引擎成功啟動，麥克風開始收音 更新 UI 狀態指示燈為「聆聽中」
onaudiostart 偵測到音訊輸入（使用者開始說話） 顯示音波動畫
onresult 收到辨識結果（中間或最終） 解析語意、觸發控制指令
onspeechend 偵測到使用者停止說話 準備處理最終結果
onerror 發生錯誤（網路斷線、權限拒絕等） 錯誤處理與重試邏輯
onend 辨識服務斷開連線 自動重啟以維持長效監聽

中間結果 vs. 最終結果

當 interimResults = true 時，辨識引擎會在使用者還在說話的過程中不斷回傳「中間猜測」。這些中間結果可能隨時改變（例如先猜「前」再修正為「前進」），但對於即時性要求極高的控制場景非常有價值——你可以在使用者話還沒說完時就預先準備動作。

// 區分中間結果與最終結果
recognition.onresult = (event) => {
  const result = event.results[event.resultIndex];

  if (result.isFinal) {
    // 最終確認結果 → 執行指令
    const intent = parseIntent(result[0].transcript);
    if (intent && result[0].confidence > 0.8) {
      executeCommand(intent);
    }
  } else {
    // 中間結果 → 即時顯示在 UI
    updateLiveTranscript(result[0].transcript);
  }
};

自動重啟與指數退避

瀏覽器會在多種情況下自動觸發 onend：長時間靜默、網路中斷、系統資源不足等。對於需要「永遠在聽」的語音控制系統，你必須在 onend 中實作自動重啟。但若重啟過於頻繁（如網路持續斷線），會觸發瀏覽器的節流機制。建議使用指數退避（Exponential Backoff）策略：

let retryDelay = 100; // 初始重試延遲 100ms

recognition.onend = () => {
  setTimeout(() => {
    try {
      recognition.start();
      retryDelay = 100; // 成功後重置
    } catch (e) {
      retryDelay = Math.min(retryDelay * 2, 5000);
    }
  }, retryDelay);
};
提示 在 UI 上顯示辨識引擎的即時狀態（IDLE / LISTENING / PROCESSING / ERROR）對於除錯與使用者體驗都非常重要。建議維護一個簡單的狀態機，在每個事件觸發時更新狀態並反映到介面的指示燈或文字上。

安全與隱私限制

麥克風是高度敏感的硬體資源——它能錄下使用者的私密對話、環境聲音與個人資訊。因此，瀏覽器對 Web Speech API 施加了嚴格的安全限制。理解這些限制不僅能幫助你避免開發期間的詭異 Bug，更能讓你設計出尊重使用者隱私的負責任應用。

核心安全限制

限制項目 行為說明 開發者因應方式
HTTPS 強制 非安全來源（HTTP）下 SpeechRecognition 完全無法啟動 使用 mkcert 建立本地開發憑證，或使用 localhost（例外允許）
使用者授權 首次呼叫 .start() 會彈出麥克風權限提示 在明確的使用者互動（按鈕點擊）後才啟動辨識
靜默自動停止 Chrome 約在靜默 5~7 秒後自動觸發 onend 實作 onend 自動重啟邏輯
雲端傳輸 所有音訊片段都會上傳至 Google 伺服器處理 在隱私政策中明確告知使用者
瀏覽器相容 Firefox 與 Safari 不支援 STT 部分 進行 Feature Detection，提供降級方案

瀏覽器相容性與降級策略

Web Speech API 的語音辨識功能目前僅在 Chromium 系列瀏覽器（Chrome、Edge、Opera）中完整可用。在不支援的瀏覽器中，你的應用應優雅降級而非直接崩潰：

// Feature Detection：檢查 API 可用性
const SpeechRecognition = window.SpeechRecognition
  || window.webkitSpeechRecognition;

if (!SpeechRecognition) {
  // 降級方案：顯示文字輸入框替代語音
  showTextInputFallback();
  console.warn('此瀏覽器不支援語音辨識，已切換為文字輸入模式');
} else {
  const recognition = new SpeechRecognition();
  // 正常初始化...
}

自激迴路問題（Echo Loop）

當你的應用同時使用 STT（聽）和 TTS（說）時，會遇到一個經典問題：機器人說出的話被自己的麥克風接收，觸發新的辨識結果，導致無限迴圈。解決方案是在 TTS 播放期間暫停辨識：

function speakAndPause(text) {
  recognition.stop(); // 暫停聆聽
  const utterance = new SpeechSynthesisUtterance(text);
  utterance.onend = () => {
    setTimeout(() => recognition.start(), 300); // 延遲恢復
  };
  speechSynthesis.speak(utterance);
}
注意 Chrome 在使用者未與頁面互動前，會阻止自動播放音訊（包括 TTS）。這是 Autoplay Policy 的一部分。確保你的 TTS 呼叫是在使用者按鈕點擊事件鏈中觸發的，否則 speechSynthesis.speak() 會靜默失敗。
備註 對於需要離線語音辨識的場景（如嵌入式設備、隱私敏感環境），Web Speech API 並非合適選擇。替代方案包括：本地端的 Vosk（開源離線 STT）、Picovoice Porcupine（輕量喚醒詞引擎）、或直接在 ESP32 上執行的 TFLite Micro KWS 模型。這些方案犧牲了語言理解深度，但換來了完全離線與零隱私風險。

adv-09-unit-webspeech-api (Web Speech API)

這是一個讓學員體驗 「語音用戶介面 (VUI)」 的核心單元。它強調了雲端識別的非同步特性，以及如何透過「意圖解析 (Intent Parsing)」與「語音合成 (TTS)」建立一個會聽也會說的人機互動閉迴路。

作業目標

掌握 Web Speech API 的整合後，你將擁有了開發「智慧語音助理」或「輔助科技 (Assistive Technology)」的核心工程技能，這能顯著提升機器人系統的易用性與人性化程度。

任務細節

任務 1：模糊意圖解析 (Intent Mapping Lab)

  • 目標：將「不精確的口語」轉化為「精確的機器碼」。
  • 實作要求：
    • 解析邏輯：實作一個解析器，能從 SpeechRecognitionEvent 的長字串中抓取關鍵動詞。
    • 同義詞覆蓋：請在此 README 中列出你支援的指令集（應包含至少 3 組以上的同義詞表）。例如：「停」、「不要跑」、「煞車」都應對應到 STOP 指令。

任務 2：語音對話回饋 (Two-way Feedback Lab)

  • 目標：透過 TTS (Text-to-Speech) 建立「指令確認」機制，提升控制安全性。
  • 實作要求：
    • 語音合成實作：加入雙向確認回饋機制。例如，當解析到前進指令且信心度夠高時，讓系統回饋語音「收到指令，即將前進」後再執行。
    • 自激迴避設計：請在此 README 中說明，你如何防止「機器人說話的聲音被自己的麥克風辨識」所造成的死循環（提示：在 speak() 時可以暫停辨識）。
    • 驗證影片：上傳一小段影片，展示錄音確認到連動的過程。

任務 3：長效監聽核心 (Continuous VUI Engine)

  • 目標：克服瀏覽器自動結束錄音的限制，建立工業級監聽服務。
  • 實作要求：
    • 生命週期管理：實作 onend 自動重啟邏輯，並加入「指數退避 (Exponential Backoff)」重連策略，以應對網路斷線。
    • 狀態機整合：將語音指令作為 FSM 的一個全局事件源。無論是在什麼狀態，語音 STOP 必須擁有最高切換權限。
    • 進階挑戰：嘗試運用 interimResults，在使用者還在說話時（中間結果）就預先載入或處理指令。

    • 解析邏輯檢核：學員需實作一個解析器，能從 SpeechRecognitionEvent 的長字串中抓取關鍵動詞。
    • 同義詞覆蓋：要求在 README 中列出其支援的指令集（應包含至少 3 組以上的同義詞表）。
    • Autograding：餵入字串如 "哈囉，請幫我往後退"，檢核輸出是否為 BACKWARD。

任務 2：語音對話回饋 (Two-way Feedback Lab)

  • 目標：透過 TTS 建立「指令確認」機制，提升控制安全性。
  • Classroom 部署建議：
    • 語音合成實作： 
      // 學生需實作：雙向確認回饋
if (intent === 'GO' && confidence > 0.85) {
    speak("收到指令，即將前進");
    sendBLECommand('forward');
}
    • 自激迴避設計：學員需在 README 中說明，他們如何防止「機器人說話的聲音被自己的麥克風辨識」所造成的死循環（提示：在 speak() 時暫停辨識）。
    • 驗證影片：要求上傳一小段影片，展示錄音確認到車體連動的過程。

任務 3：長效監聽核心 (Continuous VUI Engine)

  • 目標：克服瀏覽器自動結束錄音的限制，建立工業級監聽服務。
  • Classroom 部署建議：
    • 生命週期管理：學員需實作 onend 自動重啟邏輯，並加入「指數退避 (Exponential Backoff)」重連策略，以應對網路斷線。
    • 狀態機整合：將語音指令作為 FSM 的一個全局事件源。無論車體是在什麼狀態，語音 STOP 必須擁有最高切換權限。
    • PR 深度核核：點評其對 interimResults 的運用。是否能在使用者還在說話時（中間結果）就預先載入指令？

3. VUI 系統工程導師點評標準 (VUI System Benchmarks)

此單元的價值在於 「對自然語言不確定性的處理能力」：

  • [ ] 語義匹配率：意圖詞典設計是否周全？是否能正確過濾贅詞與噪音字詞？
  • [ ] 反饋實效性：TTS 的語音回饋是否能提供明確的狀態指引？
  • [ ] 韌性架構：自動重啟邏輯是否穩健？是否能處理 low-confidence 的辨識失敗？
