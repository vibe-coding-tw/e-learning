簡介

馬達是無人車的動力核心，但「如何啟動」決定了車輛的穩定性。 直接將 PWM 從 0 拉到最大值，就像把油門從零踩到底——結果不是加速，而是失控。 本模組將教你用 Ramping 演算法讓加速曲線平滑可控，同時透過 BLE 二進位協議即時調校參數。

情境
小明用遙控器操控他組裝好的 ESP32 無人車。他興奮地推下油門，車子猛地衝出去，打了一個轉就撞上桌腳。 問題不在馬達太強，而在 PWM 從 0 瞬間跳到 255——沒有任何緩衝。 這就是本模組要解決的問題：用 Ramping 演算法讓加速曲線變得平滑可控。
本模組重點 你將學會三件事：(1) PWM 控制馬達速度的原理；(2) BLE 二進位通訊的資料結構映射；(3) Ramping 加速曲線的調校與安全機制。

裝置連線與設定

在操控無人車之前，你需要先理解三個核心概念：PWM 如何控制馬達轉速、BLE 二進位通訊為何比文字協議更適合嵌入式裝置， 以及如何用 JavaScript 的 DataView 將參數精確映射到 C 語言的結構體記憶體佈局。

PWM（脈衝寬度調變）與馬達速度

PWM 不是直接控制電壓大小，而是透過快速開關訊號的「高電位時間比例」（Duty Cycle）來調節平均功率。 當 Duty Cycle 為 0% 時馬達不轉，100% 時全速運轉。ESP32 的 LEDC 模組可輸出 0–255 共 256 級的 PWM 值， 對應 0%–100% 的轉速比例。

Note PWM 值 127 不代表「半速」——馬達的轉速與 PWM 之間是非線性關係，受到靜摩擦力、負載與電池電壓影響。這也是為什麼需要 pwmStartKick 參數來克服啟動死區。

BLE 二進位通訊 vs. 文字協議

傳統做法是用 JSON 字串傳送參數（如 {"pwmLimit":200}），但這對嵌入式裝置有兩個問題： 一是字串解析需要消耗 CPU 與記憶體（ESP32 僅 520KB SRAM），二是 BLE 單次傳輸最大 512 Bytes， JSON 的冗餘字元浪費了寶貴的頻寬。二進位協議將 7 個參數壓縮成固定的 28 Bytes，嵌入式端只需一次 memcpy 即可完成解析。

比較項目JSON 文字協議Binary 二進位協議
封包大小（7 個 int32）~120–150 Bytes固定 28 Bytes
嵌入式端解析方式需要 JSON Parser 函式庫memcpy(&config, buf, 28)
CPU 開銷高（逐字元解析）趨近零
型別安全性低（字串轉數字可能出錯）高（固定偏移量 + 型別）

C Struct 與 DataView 的對應關係

ESP32 韌體定義了一個 MotorConfig_t 結構體，共 28 Bytes。網頁端使用 JavaScript 的 ArrayBuffer（原始記憶體）配合 DataView（帶偏移量的讀寫介面）來組裝完全相同的位元組排列。 關鍵在於兩端必須使用相同的位元組序（Little-Endian）與型別寬度。

1

建立 ArrayBuffer

const buffer = new ArrayBuffer(28); — 分配 28 Bytes 的連續記憶體空間。

2

包裝為 DataView

const view = new DataView(buffer); — DataView 提供 setInt32()、getUint32() 等方法，可指定偏移量和位元組序。

3

逐欄寫入（Little-Endian）

view.setInt32(offset, value, true); — 第三個參數 true 表示 Little-Endian，與 ESP32 的記憶體排列一致。

Tip 忘記設定 Little-Endian（省略第三個參數）是最常見的錯誤——DataView 預設使用 Big-Endian，與 ESP32 相反，導致讀出的數值完全錯誤。

Ramping 原理示意

Ramping 的核心概念是：不要讓 PWM 輸出瞬間到達目標值，而是每隔固定時間（例如 10ms）遞增或遞減一個步進量， 讓實際輸出如同「爬坡」般逐漸逼近目標。這個步進量就是 rampAccelStep 參數。

為什麼需要 Ramping？

當你在操控無人車時，若直接將 PWM 從 0 瞬間拉到 255，會觸發三個連鎖問題： 首先是電流突波——馬達啟動瞬間的堵轉電流可達穩態的 5–8 倍，可能導致 ESP32 的 3.3V 電源軌壓降， 造成 Brown-out Reset（低電壓重啟）。其次是機械應力——齒輪箱的塑膠齒輪承受瞬間衝擊力， 長期使用容易斷裂。最後是打滑失控——輪胎瞬間突破靜摩擦力極限，失去抓地力後車輛打轉。

Warning 電流突波不只影響馬達——同一條電源線上的所有元件（ESP32、感測器、LED）都會受到瞬間壓降影響。嚴重時可能造成 Flash 寫入中斷、導致韌體毀損。

rampAccelStep：虛擬彈簧的剛性

你可以把 rampAccelStep 想像成一條虛擬彈簧，連接「目標 PWM」與「實際 PWM」。 數值越小，彈簧越軟，加速越平滑但反應越慢；數值越大，彈簧越硬，反應越靈敏但衝擊越大。 選擇合適的數值需要在「操控手感」與「硬體安全」之間取得平衡。

Note Ramping 的週期固定為 10ms（100Hz 更新率）。這意味著 rampAccelStep = 5 時，從 0 加速到 255 需要 255 ÷ 5 × 10ms = 510ms（約半秒）。

pwmStartKick：克服靜摩擦力的起步推力

馬達在低 PWM 值時可能無法轉動——靜摩擦力 + 齒輪箱阻力需要最低啟動電流才能克服。 pwmStartKick 定義了「當目標 PWM > 0 且實際 PWM 為 0 時，直接跳到的最低值」。 例如設為 40，代表起步時 PWM 會直接從 0 跳到 40，跳過馬達無法動彈的死區。

Tip 每台車的 pwmStartKick 最佳值不同，取決於馬達品質、齒輪比與輪胎摩擦力。建議從 30 開始測試，逐步增加直到車輛能穩定起步。

技術開發細節

本單元深入探討二進位通訊協議的實作細節。你需要理解 ESP32 韌體端的 C Struct 如何定義記憶體佈局， 以及 JavaScript 端如何精確對齊每個欄位的偏移量與型別。

C++ Struct 定義

ESP32 韌體中定義了以下結構體，共 7 個欄位、28 Bytes。所有整數型別為 32-bit（4 Bytes）， 因此每個欄位的偏移量恰好是 4 的倍數，不存在 Padding 問題。 

struct MotorConfig_t {
  uint32_t controlTimeoutMs;   // Offset 0  (4 bytes)
  int32_t  pwmEffectiveLimitT; // Offset 4  (4 bytes)
  int32_t  rampAccelStepT;     // Offset 8  (4 bytes)
  int32_t  pwmStartKickT;      // Offset 12 (4 bytes)
  int32_t  pwmEffectiveLimitS; // Offset 16 (4 bytes)
  int32_t  rampAccelStepS;     // Offset 20 (4 bytes)
  int32_t  pwmStartKickS;      // Offset 24 (4 bytes)
};

Little-Endian 編碼說明

ESP32 使用 Xtensa LX6/LX7 處理器，採用 Little-Endian 位元組序。這代表一個 32-bit 整數的「最低有效位元組」 存放在最低地址。例如數值 500（十六進位 0x000001F4）在記憶體中的排列是 F4 01 00 00，而非 Big-Endian 的 00 00 01 F4。

JavaScript DataView 的第三個參數 littleEndian 設為 true 時， 會自動以 Little-Endian 格式讀寫。若省略此參數（預設 false），則使用 Big-Endian， 韌體端將解析出完全錯誤的數值。

Important Big-Endian / Little-Endian 錯誤不會產生任何 JavaScript 例外——程式照常執行，但韌體收到的數值是錯的。例如你傳送 200，韌體可能解讀為 3355443200。這是最難除錯的 Bug 之一。

常見錯誤排查

無法掃描到裝置 請重新插拔 ESP32 電源，並確認瀏覽器已授予藍牙權限。部分瀏覽器需要在 HTTPS 環境下才能使用 Web Bluetooth API。
數據長度錯誤 讀取時若出現長度不符的錯誤（如收到 24 Bytes 而非 28 Bytes），代表網頁端定義的結構與韌體端不一致。請確認韌體版本是否已更新至最新的 7 欄位版本。
寫入後裝置沒有反應 確認輸入值是否為整數（不含小數點）。DataView.setInt32() 接收到 NaN 時不會報錯，但寫入的位元組全為 0，導致參數被清空。建議在寫入前使用 parseInt() 強制轉型。

  • 驗證方式：學生需撰寫一個 verify-bytes.js，將填充好的 ArrayBuffer 轉換為十六進位字串（Hex String）並與導師提供的預期結果比對。

任務 3：斷電與超時保護驗證 (Fail-Safe Check)

  • 目標：驗證韌體的「安全停車」機制。
  • Classroom 部署建議：
  • 操作證明：由於這涉及物理行為，要求學生提交一段 GIF 錄影 或 序列埠日誌截圖。
  • 任務內容：設定 controlTimeoutMs 為 500ms，啟動馬達後關閉網頁。截圖需證明馬達在半秒內自動停止（終端機出現 [SAFETY] Timeout - Motor Stop）。

3. 翻轉教室與導師評估指標

  • 教學重難點：
  • Endians (位元組序)：很多學生會卡在為什麼數據讀出來是亂碼（因為沒設 true 開啟 Little-Endian）。
  • Struct Alignment (記憶體對齊)：引導學生思考為什麼 uint32 佔 4 個位元組。
  • 評語建議 (Tutor Benchmarks)：
  • [ ] 封裝正確性：DataView 寫入時是否使用了正確的整數型別 (Int32 vs Uint32)？
  • [ ] 容錯邏輯：是否處理了「空值」或「非數字」輸入的例外狀況？
  • [ ] 協議文件化：是否在 README 中清楚標註了 28 Bytes 分別代表哪些資料項？

這套部署方案能將原本抽象的「二進位協議」轉化為「可測試、可觀察」的具體任務，讓學生在學習 Web 控制硬體時，建立起極致效能與物理安全性併重的觀念。
