簡介

ESPAsyncWebServer 讓 ESP32 在處理網頁請求的同時不會阻塞主迴圈， 這對即時控制類應用至關重要——你的馬達 PWM 更新與感測器讀取不該因為一個 HTTP 請求而卡住。 本單元將帶你認識非同步伺服器的核心架構、路由設定與回應方式。

情境
假設你的 ESP32 同時要做三件事：每 20ms 更新馬達速度、每 100ms 讀取感測器、還要隨時回應手機送來的控制指令。 如果用同步式 Web Server，一個請求就可能卡住整條 loop()。 非同步架構把「等待網路 I/O」的時間還給主迴圈，讓三件事並行不悖。

非同步：網頁守門員

在 ESP32 上建立 Web Server 有兩條路線：同步（Synchronous）與非同步（Asynchronous）。 兩者的根本差異在於「處理網路請求時，是否會佔用 loop() 函式的執行時間」。 對於需要同時控制馬達、讀取感測器、又要回應網頁請求的即時系統而言，這個差異攸關生死。

同步 vs. 非同步：核心差異

同步式 Web Server（如 Arduino 內建的 WebServer.h） 需要你在 loop() 中呼叫 server.handleClient()。 每次呼叫時，如果有請求進來，它會阻塞在那裡直到處理完畢。 這意味著在處理一個 HTTP 請求的期間，你的馬達 PWM 不會更新、感測器不會讀取、 LED 不會閃——整個系統「凍結」了。

非同步式 Web Server（如 ESPAsyncWebServer）則完全不同。 它在底層使用事件驅動機制——當有 HTTP 請求到達時，系統會自動觸發你預先註冊的回呼函式（Callback）， 處理完畢後立即釋放控制權。整個過程不佔用 loop()， 你的主迴圈可以專心做它該做的事。

特性同步式 WebServer.h非同步式 ESPAsyncWebServer
loop() 佔用需呼叫 handleClient()完全不佔用
並行處理一次只能處理一個請求可同時處理多個請求
即時性影響處理請求時，控制邏輯暫停控制邏輯不受影響
程式碼風格輪詢式（Polling）回呼式（Callback / Lambda）
適用場景簡單展示頁面即時控制系統、多客戶端

安裝 ESPAsyncWebServer

在 PlatformIO 中，你需要在 platformio.ini 加入兩個依賴套件：

; platformio.ini
lib_deps =
  ESP Async WebServer
  AsyncTCP

然後在程式碼中引入標頭檔，建立伺服器實體並啟動：

#include <ESPAsyncWebServer.h>

AsyncWebServer server(80);  // 監聽 Port 80

void setup() {
  // ... WiFi 連線設定 ...

  server.on("/", HTTP_GET, [](AsyncWebServerRequest *req) {
    req->send(200, "text/plain", "ESP32 is alive!");
  });

  server.begin();
}

void loop() {
  // 完全空的！Web Server 在背景自動運作
  // 這裡可以專心做馬達控制、感測器讀取
}
注意 雖然非同步伺服器不阻塞 loop()，但回呼函式本身仍然是在主執行緒中執行的。 因此在回呼函式中絕不可以使用 delay()，也不應該執行耗時超過數毫秒的操作。 如需讀取慢速感測器，應提前將數值快取到全域變數中，在回呼中直接回傳快取值。

路由與 Lambda 魔法

路由（Route）是 Web Server 的靈魂——它定義了「什麼 URL 路徑 + 什麼 HTTP 方法」 會觸發什麼處理邏輯。在 ESPAsyncWebServer 中，路由的定義使用了 C++11 的 Lambda 表達式（匿名函式），讓你可以用極簡的語法將路徑與處理邏輯綁定在一起。

server.on() 基本語法

server.on() 接受三個參數：URL 路徑、HTTP 方法、以及一個 Lambda 處理函式。 當有請求匹配路徑與方法時，對應的 Lambda 會被自動呼叫。

// 根路徑 GET — 回傳首頁
server.on("/", HTTP_GET,
  [](AsyncWebServerRequest *request) {
    request->send(200, "text/html", htmlPage);
});

// 控制 LED — 帶 URL 參數
server.on("/set_led", HTTP_GET,
  [](AsyncWebServerRequest *request) {
    if (request->hasParam("value")) {
      int val = request->getParam("value")->value().toInt();
      analogWrite(LED_PIN, constrain(val, 0, 255));
      request->send(200, "text/plain", "LED set to " + String(val));
    } else {
      request->send(400, "text/plain", "Missing 'value' param");
    }
});

URL 參數解析

當使用者存取 /set_led?value=128 時，? 之後的部分就是查詢參數（Query Parameter）。 ESPAsyncWebServer 提供了便利的 API 來提取這些參數：

API功能回傳值
request->hasParam("key")檢查參數是否存在bool
request->getParam("key")取得參數物件AsyncWebParameter*
param->value()取得參數值（字串）String
request->params()取得參數總數int
重要 永遠先檢查參數是否存在，再嘗試取值。如果直接呼叫 request->getParam("value")->value() 而參數不存在， 將會造成空指標存取（Null Pointer Dereference）， 導致 ESP32 立即崩潰重啟。使用 hasParam() 是防禦性程式設計的基本功。

處理 404 — 未匹配的路由

當使用者存取一個你沒有定義的路徑時，ESPAsyncWebServer 允許你設定一個全域的 「未匹配」處理器，回傳友善的 404 訊息，而不是讓客戶端看到連線逾時：

server.onNotFound(
  [](AsyncWebServerRequest *request) {
    request->send(404, "text/plain",
      "Not Found: " + request->url());
});
提示 Lambda 中的 [] 是捕獲列表（Capture List）。 若你需要在 Lambda 中存取外部變數，可以使用 [&]（按參考捕獲所有外部變數） 或 [=]（按值捕獲）。但在嵌入式環境中建議謹慎使用， 因為捕獲過多變數可能增加堆疊使用量。

範本引擎：內容與數據分離

隨著你的 ESP32 Web 介面越來越複雜，你會發現一個痛點：HTML 字串直接寫在 C++ 程式碼中， 既難閱讀又難維護。更糟的是，如果你需要在 HTML 中顯示動態數據（如感測器即時數值）， 就得用大量的 String 拼接——又慢又耗記憶體。

ESPAsyncWebServer 的 Processor 佔位符機制優雅地解決了這個問題。 你可以在 HTML 字串中放入 %PLACEHOLDER% 標記， 伺服器在回傳網頁時會自動呼叫你定義的 Processor 函式，將佔位符替換為真實數據。 這就是嵌入式版本的伺服器端渲染（Server-Side Rendering, SSR）。

R"rawliteral" — 保持 HTML 原始格式

C++ 的 R"rawliteral(...)rawliteral" 語法允許你在程式碼中直接嵌入多行字串， 不需要跳脫引號或換行符號。這讓你的 HTML 模板能保持完整的排版格式， 大幅提升可讀性：

const char htmlPage[] PROGMEM = R"rawliteral(
<!DOCTYPE html>
<html>
<body>
  <h1>ESP32 Dashboard</h1>
  <p>Uptime: <strong>%UPTIME%</strong> seconds</p>
  <p>Free Heap: <strong>%HEAP%</strong> bytes</p>
  <p>WiFi RSSI: <strong>%RSSI%</strong> dBm</p>
</body>
</html>
)rawliteral";

Processor 回呼函式

接下來，定義一個 processor 函式。每當伺服器在 HTML 中遇到 %...% 標記時， 就會呼叫這個函式，傳入佔位符名稱，並以回傳值替換之：

String processor(const String &var) {
  if (var == "UPTIME")
    return String(millis() / 1000);
  if (var == "HEAP")
    return String(ESP.getFreeHeap());
  if (var == "RSSI")
    return String(WiFi.RSSI());
  return String();  // 未知佔位符回傳空字串
}

最後，在路由中將 HTML 字串與 Processor 函式綁定：

server.on("/", HTTP_GET,
  [](AsyncWebServerRequest *request) {
    request->send_P(200, "text/html", htmlPage, processor);
});

常用佔位符設計參考

佔位符Processor 中的回傳值用途
%UPTIME%millis() / 1000顯示系統運行秒數
%HEAP%ESP.getFreeHeap()監測剩餘記憶體
%RSSI%WiFi.RSSI()Wi-Fi 訊號強度
%TEMP%感測器讀值變數溫度即時顯示
%LED_STATE%"ON" 或 "OFF"LED 當前狀態
備註 使用 send_P()（注意結尾的 _P）可以直接從 PROGMEM（Flash 記憶體）讀取 HTML 字串， 避免將整個網頁載入 RAM。當你的 HTML 超過 4KB 時，這個技巧能顯著節省寶貴的堆疊空間。
提示 佔位符名稱中不可以包含 % 字元本身。 如果你的 HTML 中需要顯示百分比符號（如 CSS 的 width: 50%）， Processor 可能會誤判。解決方法是使用 %%（兩個百分號）來跳脫， 或將 CSS 移到 <style> 標籤外的獨立檔案中。

ESP32 非同步網頁伺服器實作 (ESP32 Async Web Server)

本程式庫是關於如何在 ESP32 上建立非同步網頁伺服器的練習。你將學習如何處理網頁請求而不阻塞控制邏輯，並實作動態內容顯示與參數控制。

任務目標

任務 1：建立非同步首頁 (Async Landing Page)

  • 目標：建立基礎的 HTTP 伺服器，並在根路徑 / 顯示首頁。
  • 要求：確保網頁能正確載入，且硬體端的序列埠輸出（Serial Monitor）在連線時依然保持流暢。

任務 2：互動式參數控制 (Interactive Control)

  • 目標：透過 URL 參數控制 ESP32 的硬體狀態（例如 LED 亮度）。
  • 路由：/set_led?value=X (X 為 0-255 的整數)。
  • 要求：
    • 實作 防禦性檢查：必須使用 request->hasParam("value") 檢查參數。
    • 若參數缺失或數值超出範圍，必須回傳 400 Bad Request。

任務 3：動態範本引擎 (Template Engine)

  • 目標：將 HTML 結構與動態數據分離。
  • 要求：
    • 在 include/web_pages.h 中使用佔位符（如 %UPTIME%）。
    • 實作 processor 函式，在回傳網頁時將佔位符替換為真實的系統運行時間。

提交要求

  1. 程式碼：完成 src/main.cpp 中的路由與邏輯實作。
  2. 測試報告：在 docs/API_Test.md 中記錄你的測試結果，並附上截圖。
  3. 更新程式庫：完成後請 commit 並 push 你的更改。

3. 指令架構與安全合規點評標準 (API Architecture & Safety Standards)

此單元的價值在於 「對硬體狀態的 Web 化抽象能力與對通訊流程的非阻塞掌控」：
