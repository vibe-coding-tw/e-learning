數據流與取樣率

理解機器人學的隱藏法則：如何在高頻更新與通訊瓶頸之間找到完美的「平衡點」。

📡 1. 什麼是取樣率 (Sampling Rate)？

取樣率是指每秒鐘讀取感測器或發送控制指令的次數，單位是 Hz (赫茲)。它是衡量控制精準度的核心指標。

1 Hz

"機器人動作極其頓挫，無法穩定直線。"

20 Hz

"基本遙控感，適合低速巡航。"

50+ Hz

"專業級響應，操控感覺如絲般順滑。"

Discrete Control Theory

"控制頻率越高，系統對誤差的修正就越及時。"

⚠️ 2. 數據流瓶頸：頻寬與延遲

取樣率不能無限提高。硬體瓶頸（如藍牙 MTU 限制、解析 JSON 的時間）限制了我們的傳輸速度。

Golden Rule

\[Hz \times PacketSize < Bandwidth\]

🌀 3. 奈奎斯特定理與滑動濾波

若要精確捕捉動作，取樣率必須大於信號頻率的兩倍。同時，高頻取樣會帶來更多噪聲，我們需要「滑動平均」來平滑數據。

Signal Processing

  • 混疊 (Aliasing)：取樣太慢導致方向轉向誤判（就像電風扇倒著轉）。
  • 滑動平均 (SMA)：利用 FIFO Buffer 取均值，消除感測器的毛刺噪聲。
  • 節流 (Throttling)：確保發送頻率封頂，防止藍牙緩衝區堆棧性當機。

Bandwidth Optimization

"優雅的數據流在於：發送適量的資訊，並賦予其最大的執行價值。"

數據流與信號處理實驗室

任務 1：抖動測試：取樣頻率與手感 (Jitter & Control Feel)

角色扮演：操縱感調校工程師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並拉取基礎 App 框架。
  2. 取樣率控制器實作：在前端 App 介面加入滑桿，使指令發送週期能在 10ms (100Hz) 到 1000ms (1Hz) 間動態切換。
  3. 實機手感對比：分別在 5Hz 與 50Hz 設定下挑戰狹隘的 S 彎道。記錄不同取樣頻率對操控「精準度」與「頓挫感」的具體影響。
  4. 擁塞邊界探究：不斷提高頻率，尋找並記錄該藍牙環境下開始發生封包丟失 (Packet Loss) 或明顯卡頓的臨界頻寬值。

任務 2：感測器數據降噪：滑動平均濾波 (Moving Average Filter)

角色扮演：數據科學家 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 噪雜採樣觀察：以 100Hz 高頻連續輸出原始超音波測距日誌。量化靜止狀態下數據的「隨機抖動」範圍。
  2. FIFO 緩衝區實作：建立一個長度為 8-16 的循環 Buffer 陣列，儲存最新的歷史採樣數據。
  3. 均值平滑演算：計算 Buffer 內的數據平均值。對比處理前後的感測器數據流平滑程度。
  4. 延遲補償分析：觀察加大 Buffer 長度後，「噪聲消除效果」與「物理反應遲滯」之間的權衡結果。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 濾波品質優異性：處理後的感測器數據流是否成功消除了 80% 以上的隨機噪聲，且無明顯的數值突跳？
  • [ ] 計算邏輯穩健度：數據緩衝區的索引環繞 (Wraparound) 邏輯是否正確，各項邊界條件處理是否妥當？

任務 3：通訊節流與協定優化 (Throttling & Protocol Optimization)

角色扮演：通訊協定優化專家 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 通訊壓力測試：以超高頻率（亂動搖桿）產發封包，觀察未處理狀態下藍牙底層 Buffer 是否會因寫入過快而當機。
  2. 節流閥機制 (Throttling)：在發送指令前實施 millis() 判定，強迫發送間隔絕不低於 20ms，平滑化發送負載。
  3. 協定壓縮優化：重寫 JSON 數據交換協定。壓縮 Key 名稱（如用 "v" 代替 "voltage"）並量化單個封包節省的位元組數。
  4. 最終提交：將具備節流保護與高効率壓縮協定邏輯的代碼 Push 至 GitHub PR。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 資源効能指標：README 中是否清晰展示了「協定瘦身」對整體吞吐量與實時響應性的貢獻？
