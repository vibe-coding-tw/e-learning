📡 1. 什麼是取樣率 (Sampling Rate)？

每秒讀取次數 (Hz)。衡量控制精準度的核心。

1 Hz

頓挫嚴重。無法穩定。

20 Hz

基本遙控感。低速巡航。

50+ Hz

專業響應。絲般順滑。

🌀 3. 滑動濾波與節流

取樣頻率需與物理動作匹配。使用「滑動平均」平滑噪聲。

Signal Processing

  • 混疊 (Aliasing)：取樣太慢導致方向轉向誤判。
  • 滑動平均 (SMA)：利用 Buffer 取均值，消除毛刺噪聲。
  • 節流 (Throttling)：確保發送頻率封頂，防止緩衝區溢位。

Bandwidth Optimization

作業單元 09：數據流與取樣率 (Sampling Rate & DSP)

歡迎來到「數位訊號處理 (DSP)」的進階基礎課程！在這個單元中，你將學習如何掌握 取樣定理 (Nyquist-Shannon)，並透過頻率管理來平衡「操控流暢度」與「硬體頻寬限制」。

🎯 學習目標

  1. 取樣頻率與感官體驗：量化數位頻率對操控手感的影響。
  2. 數位濾波實作：學習滑動平均濾波 (SMA) 的原理與 FIFO Buffer 實作。
  3. 通訊優化：實作 Throttling (節流) 機制與 JSON 協定縮身。

📂 程式庫結構

  • src/main.cpp: ESP32 硬體濾波核心與 FIFO Buffer 實作。
  • src/DataHandler.js: 控制端通訊節流與協定優化實作。
  • docs/Quality_Report.md: 訊號品質實驗日誌與數據分析。

🚀 任務說明

任務 1：取樣頻率實驗

  • 動態調整發送頻率 (5Hz ~ 100Hz)。
  • 記錄不同頻率下的操控手感，找出系統的「黃金頻率」。

任務 2：滑動平均濾波 (Moving Average Filter)

  • 實作長度為 12 的循環緩衝區 (FIFO Buffer)。
  • 觀察 Buffer 長度對「訊號平滑度」與「反應延遲」的影響。

任務 3：通訊節流與協定優化

  • 設定發送頻率上限 (Throttling)，確保通訊穩定。
  • 優化 JSON 封包格式，縮減數據負擔。

📝 繳交要求

  1. 完成 src/ 中的程式碼實作。
  2. 在 docs/Quality_Report.md 中記錄實驗數據與分析結果。
  3. 將更新後的程式碼 push 到 GitHub 程式庫。

針對 basic-09-unit-sampling-rate (數據流與取樣率 Sampling Rate) 單元，這是學員從「數位代碼」跨入「數位訊號處理 (DSP)」的進階基礎課程。它要求學員掌握 取樣定理 (Nyquist-Shannon) 的實務應用，學習如何透過頻率管理平衡「操控流暢度」與「硬體頻寬限制」，並利用「滑動平均濾波 (SMA)」在噪聲與延遲之間尋找工程最優解。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本倉庫 (範本程式庫) 配置建議

取樣率與訊號處理作業的核心在於「數據流控」與「平滑演算」，範本應包含通訊引擎與濾波框架，建議包含：

  • 📂 src/DataHandler.js：通訊節流實作區。學員在此實作控制端的 Throttling 機制，並練習「協定縮身 (Protocol Minification)」以優化頻寬。
  • 📂 src/main.cpp：硬體濾波核心。學員在 ESP32 端實作 FIFO Buffer 與滑動平均演算法，處理來自感測器的原始抖動數據。
  • 📂 README.md：訊號品質實驗日誌。提供表格讓學員記錄：不同取樣頻率下的「響應延遲感」評分，以及不同濾波長度對「穩定度 vs. 遲滯」的量化對比。
  • 📂 docs/BANDWIDTH_ANALYSIS.md：頻寬負載分析。要求學員根據 JSON 封包大小與發送頻率，計算理論上的每秒數據負擔 (bps)，並解釋優化協定後的效益。

2. 作業任務部署細節

任務 1：抖動測試：取樣頻率與手感 (Jitter & Control Feel Lab)

  • 目標：透過主觀感官量化數位頻率的影響，找到控制系統的「黃金頻率」。
  • Classroom 部署建議：
    • 頻率滑動介面：在控制 App 中實作滑桿，可動態調整發送頻率 (例如 5Hz ~ 100Hz)。
    • 極效對比：記錄在 5Hz（嚴重頓挫、方向混疊）與 50Hz（絲般順滑）下的操控表現。
    • 驗證要點：README 中是否能量化描述低頻下的「控制斷裂感」？是否找到了能兼顧流暢度且不造成網路擁塞的頻率平衡點？

任務 2：感測器數據降噪：滑動平均濾波 (Moving Average Filter Lab)

  • 目標：掌握數位濾波的基本建構，理解 Buffer 長度對訊號相位（延遲）的影響。
  • Classroom 部署建議：
    • FIFO Buffer 建立：實作長度為 12 的循環緩衝區。技術要領：每次讀取新值時需覆蓋最舊的值，而非重新分配記憶體。
    • 均值演算：對比「原始數據（多毛刺）」與「濾波後數據（平穩）」的實時差異。
    • 延遲與過濾平衡：核心挑戰。要求學員觀察：當 Buffer 長度增加到 50 時，雖然數據極度平穩，但「反應遲滯」是否變得無法接受？
    • 驗證要點：FIFO 指針處理是否妥當？濾波邏輯是否高效？能在 README 中正確描述延遲與降噪的拉鋸關係？

任務 3：通訊節流與協定優化 (Throttling & Protocol Lab)

  • 目標：培養「網路資源管理」意識。透過發送頻率封頂與報文瘦身，確保系統在高頻通訊下具備極高的魯棒性 (Robustness)。
  • Classroom 部署建議：
    • Throttling 節流閥：不論 loop 多快，發送間隔嚴禁低於 20ms（封頂 50Hz）。
    • 協定瘦身計畫：將原本的 {"linear_velocity": 1.0, "angular_velocity": 0.5} 優化為 {"v": 1.0, "w": 0.5}。
    • 穩定度驗證：連續執行 5 分鐘高頻操控，驗證 ESP32 是否不再因為 Serial/BT 緩衝區溢位而當機。
    • 驗證要點：Throttling 是否有效防止了通訊擁塞？JSON 協定是否達成顯著縮身？

3. 訊號處理與通訊效率點評標準 (DSP & Protocol Standards)

此單元的價值在於 「對原始數據的數學轉化能力與對通訊頻寬的工程管理能力」：

  • [ ] 濾波演算法嚴謹度：循環 Buffer 是否正確管理索引？是否有效消除了隨機噪聲？
  • [ ] 頻寬優化見解：是否能量化證明縮減 JSON Key 名稱對系統即時性的實質貢獻？
  • [ ] 人機工程權衡 (Trade-off)：是否理解了「沒有最完美的濾波，只有最適合場景的參數選擇」？
