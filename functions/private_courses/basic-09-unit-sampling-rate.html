什麼是取樣率 (Sampling Rate)？

取樣率（Sampling Rate）是指系統每秒讀取感測器數據的次數，單位為赫茲（Hz）。例如 50 Hz 代表每秒讀取 50 次，每次間隔 20 毫秒。取樣率直接決定了你的控制系統「看世界」的解析度——取樣越密集，對現實變化的追蹤就越精確。

在數位控制系統中，物理世界是連續的，但處理器只能在離散的時間點讀取數據。兩次取樣之間發生的變化，系統完全無從得知。這就是為什麼取樣率對控制品質如此重要。

Nyquist 取樣定理

奈奎斯特-夏農取樣定理（Nyquist-Shannon Sampling Theorem）是數位訊號處理的基石：

fsample ≥ 2 × fmax

要忠實還原一個訊號，取樣頻率必須至少是訊號最高頻率成分的兩倍。若違反此條件，就會產生混疊（Aliasing）——高頻訊號被誤判為低頻，導致控制系統做出錯誤的修正。

實務經驗法則 理論上 2 倍就足夠，但實務中建議取樣頻率為目標訊號頻率的 5~10 倍。例如遙控車的方向盤操作頻率約 2~5 Hz，取樣率設為 20~50 Hz 就能獲得流暢的操控體驗。

取樣率對操控體驗的影響

取樣率 間隔 (ms) 操控手感 適用場景
1 Hz 1000 嚴重頓挫，幾乎無法控制 環境溫度監測
5 Hz 200 明顯延遲感，方向操控困難 低速巡邏（不推薦用於操控）
20 Hz 50 基本可操控，低速行駛尚可 簡單遙控車、慢速機器人
50 Hz 20 流暢自然，專業操控感 競速車、無人機、工業機器人
100+ Hz ≤10 極致響應（但頻寬壓力大） 高速 PID、自平衡系統

ESP32 上的取樣率實作

使用 millis() 或硬體定時器控制取樣間隔：

const unsigned long SAMPLE_INTERVAL = 20; // 50 Hz = 每 20ms 取樣一次 unsigned long lastSample = 0; void loop() { unsigned long now = millis(); if (now - lastSample >= SAMPLE_INTERVAL) { lastSample = now; int sensorValue = analogRead(SENSOR_PIN); processSample(sensorValue); } // 不取樣時可處理通訊等其他任務 handleCommunication(); }
取樣率不是越高越好 過高的取樣率會帶來三大問題：(1) CPU 被大量 ADC 讀取佔用，無暇處理其他任務；(2) 通訊頻寬被數據灌滿導致擁塞；(3) 電池壽命因高頻運算而縮短。選擇取樣率時，必須在「解析度」和「資源成本」之間取得平衡。

瓶頸：頻寬與延遲

即使你的感測器能以 1000 Hz 產生數據，通訊鏈路也不一定吞得下。在物聯網系統中，有效取樣率受限於整條數據管線中最慢的一環——這就是瓶頸（Bottleneck）。理解瓶頸在哪裡，才能設計出高效且穩定的數據流架構。

數據管線的瓶頸分析

從感測器讀取到控制執行，數據經過以下階段：

1

感測器讀取（ADC 轉換）

ESP32 SAR ADC 每次轉換約 10~20 μs，理論上可達 100 kHz 取樣率——通常不是瓶頸。

2

數據封裝（JSON 序列化）

將數據打包為 JSON 字串需要字串操作時間，一個包含 5 個欄位的 JSON 約花 50~200 μs。

3

通訊傳輸（BLE / WiFi）

BLE 4.2 的有效吞吐量約 2~6 KB/s（受 MTU、連線間隔影響）。這通常是最大的瓶頸所在。

4

接收端解析與執行

手機 App 或接收端需要解析 JSON 並更新 UI，高頻數據可能導致 UI 卡頓。

頻寬計算公式

判斷通訊是否會擁塞的核心公式：

發送頻率 (Hz) × 封包大小 (Bytes) ≤ 可用頻寬 (Bytes/s)

通訊方式 典型有效頻寬 50Hz 時可容許封包大小
BLE 4.2 (預設 MTU) ~2,000 Bytes/s 40 Bytes
BLE 5.0 (大 MTU) ~6,000 Bytes/s 120 Bytes
WiFi UDP ~50,000 Bytes/s 1,000 Bytes
Serial (115200 baud) ~11,520 Bytes/s 230 Bytes
超出頻寬的後果 當數據產生速度超過傳輸速度時，緩衝區會逐漸填滿。最終結果是：(1) 封包被丟棄導致數據遺失、(2) 延遲持續累積導致控制滯後、(3) 緩衝區溢位導致 ESP32 當機重啟。

封包瘦身策略

在頻寬有限的情況下，縮小每個封包的大小可以有效提高可用頻率。以下是常見的優化手段：

// ❌ 冗長的 JSON（67 Bytes） {"linear_velocity":1.23,"angular_velocity":0.45,"battery":78} // ✅ 壓縮 Key 名稱（25 Bytes，節省 63%） {"v":1.23,"w":0.45,"b":78} // ✅✅ 二進位協定（12 Bytes，節省 82%） // [0xAA][float v][float w][uint16 bat][checksum]
延遲 vs 頻寬的權衡 降低發送頻率可以減少頻寬壓力，但會增加控制延遲。最佳策略是同時做到：(1) 壓縮封包大小、(2) 只在數值有顯著變化時才發送（差值觸發）、(3) 設定合理的頻率上限（Throttling）。

滑動濾波與節流

真實世界的感測器數據往往帶有雜訊（Noise）——ADC 的量化誤差、電磁干擾、機械振動都會在讀數中產生毛刺。直接使用原始數據進行控制，會導致馬達輸出抖動、轉向不穩。數位濾波是解決這個問題的關鍵技術。

同時，即使感測器能高頻產生數據，通訊鏈路的頻寬是有限的。節流（Throttling）機制確保發送頻率不超過通訊能力的上限，防止緩衝區溢位和系統崩潰。

滑動平均濾波 (Simple Moving Average, SMA)

滑動平均是最基本也最實用的數位濾波器。它維護一個固定長度的循環緩衝區（FIFO Buffer），每次讀取新值時覆蓋最舊的值，然後計算整個緩衝區的平均值作為輸出。

const int BUFFER_SIZE = 12; int buffer[BUFFER_SIZE]; int bufIndex = 0; long bufSum = 0; bool bufFull = false; int addSample(int newValue) { bufSum -= buffer[bufIndex]; // 減去即將被覆蓋的舊值 buffer[bufIndex] = newValue; // 寫入新值 bufSum += newValue; // 加上新值 bufIndex = (bufIndex + 1) % BUFFER_SIZE; if (bufIndex == 0) bufFull = true; int count = bufFull ? BUFFER_SIZE : bufIndex; return bufSum / count; // 回傳平均值 }
O(1) 效率 注意上面的實作使用「累加和」技巧：每次只加減一個值，不需要遍歷整個陣列重新求和。無論 Buffer 多長，每次更新都是常數時間 O(1)——這對即時系統非常重要。

Buffer 長度的取捨

Buffer 長度是「平滑度」與「反應延遲」之間的拉鋸：

Buffer 長度 降噪效果 反應延遲 適用場景
4 輕微平滑 極低（4 個取樣週期） 需要快速反應的控制迴圈
12 有效消除毛刺 中等 一般遙控系統（推薦起始值）
32 非常平穩 明顯遲滯 儀表板顯示、非即時監測
64+ 極度平穩 嚴重遲滯 長期趨勢分析（不適合控制）
過度濾波的危險 在即時控制系統中，Buffer 過長會導致系統「反應遲鈍」。例如自平衡車使用 64 點的滑動平均，當它開始傾斜時，濾波後的數值要等到大量新讀數進入才會反映真實角度——此時修正已經太遲，車子可能已經倒了。

通訊節流 (Throttling)

節流的目的是：無論感測器取樣多快，對外發送的頻率不超過通訊能力上限。感測器可以高頻取樣並進行濾波，但只在固定間隔才將結果發送出去。

const unsigned long MIN_SEND_INTERVAL = 20; // 最高 50Hz 發送 unsigned long lastSendTime = 0; void loop() { // 高頻取樣 + 濾波（每次 loop 都執行） int raw = analogRead(SENSOR_PIN); int filtered = addSample(raw); // 節流發送：不超過頻率上限 unsigned long now = millis(); if (now - lastSendTime >= MIN_SEND_INTERVAL) { lastSendTime = now; sendData(filtered); // 只發送濾波後的值 } }

整體架構：取樣 → 濾波 → 節流

一個設計良好的數據流系統，會將三個層次解耦：

  1. 取樣層：以適當頻率讀取感測器（如 100 Hz），由定時器或 millis 控制。
  2. 濾波層：每筆取樣都送入滑動平均 Buffer，即時產生平滑的輸出值。
  3. 發送層：以通訊頻寬允許的最高頻率（如 50 Hz）將濾波結果發送出去。
進階優化：差值觸發 (Dead-band) 除了固定頻率發送外，還可以加入「差值觸發」機制：只有當新值與上次發送的值差距超過閾值時才發送。靜止狀態下幾乎不佔頻寬，動態變化時才密集傳輸——這對電池供電的物聯網裝置尤其有價值。

    • 極效對比：記錄在 5Hz（嚴重頓挫、方向混疊）與 50Hz（絲般順滑）下的操控表現。
    • 驗證要點：README 中是否能量化描述低頻下的「控制斷裂感」？是否找到了能兼顧流暢度且不造成網路擁塞的頻率平衡點？

任務 2：感測器數據降噪：滑動平均濾波 (Moving Average Filter Lab)

  • 目標：掌握數位濾波的基本建構，理解 Buffer 長度對訊號相位（延遲）的影響。
  • Classroom 部署建議：
    • FIFO Buffer 建立：實作長度為 12 的循環緩衝區。技術要領：每次讀取新值時需覆蓋最舊的值，而非重新分配記憶體。
    • 均值演算：對比「原始數據（多毛刺）」與「濾波後數據（平穩）」的實時差異。
    • 延遲與過濾平衡：核心挑戰。要求學員觀察：當 Buffer 長度增加到 50 時，雖然數據極度平穩，但「反應遲滯」是否變得無法接受？
    • 驗證要點：FIFO 指針處理是否妥當？濾波邏輯是否高效？能在 README 中正確描述延遲與降噪的拉鋸關係？

任務 3：通訊節流與協定優化 (Throttling & Protocol Lab)

  • 目標：培養「網路資源管理」意識。透過發送頻率封頂與報文瘦身，確保系統在高頻通訊下具備極高的魯棒性 (Robustness)。
  • Classroom 部署建議：
    • Throttling 節流閥：不論 loop 多快，發送間隔嚴禁低於 20ms（封頂 50Hz）。
    • 協定瘦身計畫：將原本的 {"linear_velocity": 1.0, "angular_velocity": 0.5} 優化為 {"v": 1.0, "w": 0.5}。
    • 穩定度驗證：連續執行 5 分鐘高頻操控，驗證 ESP32 是否不再因為 Serial/BT 緩衝區溢位而當機。
    • 驗證要點：Throttling 是否有效防止了通訊擁塞？JSON 協定是否達成顯著縮身？
