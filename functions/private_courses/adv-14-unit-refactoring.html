從「義大利麵代碼」進化到優雅的軟體架構。學習償還技術債，設計高內聚、低耦合的精品系統。

👃 1. 識別「代碼壞味道」 (Code Smells)

重構不是增加新功能，而是優化內在結構。好的代碼應該像優美的散文，讓人讀起來賞心悅目。學會聞出代碼中的「技術債」。

Common Code Symptoms

  • 冗長函式 (Long Method)：一個 loop() 超過 50 行，且混雜了多種職責。
  • 魔術數字 (Magic Numbers)：代碼中充滿寫死的比例與閾值，缺乏具名常數。
  • 義大利麵依賴：改動 A 卻導致完全不相關的 B 壞掉。

Refactoring Transformation

spaghetti_logic() { ... 500 lines ... }

🧱 2. 關注點分離：三層神經架構

將系統拆解為不互相依賴的獨立模組。感知 (Perception)、決策 (Logic) 與執行 (Action) 三權分立，實現高內聚低耦合。

🧩 3. 插件化與管道設計模式

追求硬核的擴展性。設計「處理鏈」模式，在不修改核心引擎的前提下，只需幾行代碼即可動態載入新的濾鏡或控制策略。

Open-Closed Design

  • 對擴展開放 (Open)：新增濾鏡只需開發獨立的 Plugin，不影響主迴圈。
  • 對修改封閉 (Closed)：核心處理邏輯一旦穩定，不再因為細微需求變動而改寫。
  • Pipeline 模式：像生產線一樣，各模組按序處理數據流。

重構實驗室：代碼病態分析與精品架構設計

任務 1：「代碼手術台」：解構大雜燴 (Spaghetti Code Surgery)

角色扮演：代碼外科醫生 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 壞味道偵測：審視專案中的 spaghetti-logic.js。標記出：哪裡出現了超過三層的巢狀迴圈？哪裡全域變數正在被濫用？
  3. 單一職責拆解：利用 Extract Function 手法，將原始龐大的迴圈拆解為「感知」、「處理」、「輸出」三個標準函數。
  4. 具名常數化：將代碼中散落的魔術數字（如：0.35, 255）統一抽取至外部的 Config 設定物件中。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 命名精確度：拆解後的函式名稱是否具備高可讀性，無需註解即可看懂用途？
  • [ ] 技術債削減度：全域變數是否被最小化？代碼的循環複雜度 (Cyclomatic Complexity) 是否有顯著降低？

任務 2：打造「通訊模組」Class (Encapsulating BLE Module)

角色扮演：軟體架構師 | 難度：⭐️⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. OOP 封裝：定義 CarComms 類別，將 Web Bluetooth 的事件監聽與數據序列化封裝在內部成員中。
  2. 介面合約制定：確保主程式與通訊模組僅透過 send() 與 onReceive() 介面互動，不得存取內部底層 Buffer。
  3. 異常靜默處理：在類別內部實作「短暫斷線自動重連」邏輯，對外部僅回報目前連線健康度。
  4. 類別測試：嘗試在 PR 範例中同時建立兩個測試實例，驗證模組是否達成了真正的隔離（無互斥干擾）。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 封裝層次感：主邏輯代碼是否已完全從底層 BLE 的繁雜 API 中解放？
  • [ ] 系統穩定性：類別是否能正確處理「非同步初始化」與「突發性中斷」的邊界狀態？

任務 3：「插件化」影像處理鏈 (Pipeline Pattern Plugin)

角色扮演：系統擴充工程師 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 處理管線設計：設計一個「影像處理管線 (ImagePipeline)」。實作 addPlugin(plugin) 與 process(frame) 方法。
  2. 插件規範：定義一個簡單的插件物件結構（Interface），要求每個插件必須具備 apply 函式。
  3. 動態配置實踐：實作一個範例：透過 pipeline.add(gray).add(binarize) 的鏈式調用完成視覺優化。
  4. Push Final：提交具備「高度彈性、易於擴展」特質的最終影像處理引擎。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 擴展開放性檢核：當我想要新增一個新的顏色過濾器時，是否完全「不需要」改動 Pipeline 核心類別的代碼？
  • [ ] 架構優雅度：數據在各插件間的流動是否清晰且具備一致性（Input Frame -> Output Frame）？

GitHub Classroom 詳細操作指南

本單元教學重點：引導學生從「程式員 (Coder)」晉升為「工程師 (Engineer)」。學習不僅讓程式動起來，更要讓它具備生命力（易維護）。

  1. 重點 1：對抗複雜度 - 所有的重構手段都是為了「降低認知負荷」。如果拆解後反而更難讀，那就不是好的重構。
  2. 重點 2：技術債的感知力 - 讓學員「聞到」味道。培養對 Long Method 與 Global Variables 的不耐受感。
  3. 重點 3：開閉原則 (OCP) 的實戰體悟 - 透過插件化任務，讓學員感受到「穩定核心 + 二方套件」的現代軟體開發模式之美。
