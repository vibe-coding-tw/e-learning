👃 1. 識別「代碼壞味道」 (Code Smells)

重構（Refactoring）不是增加新功能，而是在不改變外部行為的前提下，優化代碼的內部結構。Martin Fowler 在《重構：改善既有程式的設計》中提出了「代碼壞味道」（Code Smells）的概念——這些是代碼中暗示深層問題的表面徵兆，就像食物發出的異味預示著它已經開始腐敗。

學會識別這些壞味道，是重構的第一步。以下是在物聯網控制系統中最常見的五種技術債徵兆：

五大代碼壞味道

壞味道 症狀描述 危害 重構手法
冗長函式
(Long Method)		 主迴圈 loop() 超過 50 行，混雜感測讀取、PID 計算與馬達輸出 認知負荷過高，改一處壞全局 函式提取（Extract Method）
魔法數字
(Magic Numbers)		 代碼中充滿 255、0.35、1024 等裸數值 含義不明，修改時容易遺漏 具名常數（Named Constants）
緊密耦合
(Tight Coupling)		 模組間共享全域變數，改動通訊層導致 PID 計算損毀 牽一髮而動全身 介面隔離（Interface Segregation）
重複代碼
(Duplicated Code)		 相似的邊界檢查邏輯在 3 個以上的地方重複出現 修改時必須同步更新所有副本 提取為共用函式
深度巢狀
(Deep Nesting)		 3 層以上的 if-else 巢狀結構 邏輯分支難以追蹤，易出錯 衛語句（Guard Clauses）提早返回

壞味道辨識實例

以下是一段典型的「義大利麵代碼」，請嘗試辨識其中包含了哪些壞味道：

// 壞味道示範：一鍋義大利麵
void loop() {
  int val = analogRead(A0); // 魔法數字 A0
  if (val > 512) { // 512 代表什麼？
    if (connected) { // 全域變數
      int spd = val * 0.35; // 魔法數字 0.35
      analogWrite(5, spd); // 魔法腳位 5
      Serial.println(spd); // 除錯殘留
    }
  }
}
備註 上述短短 10 行代碼就包含了至少 4 種壞味道：魔法數字（512、0.35、5）、全域變數依賴（connected）、職責混合（感測 + 計算 + 輸出都擠在同一個 loop()）以及除錯殘留（Serial.println）。在真實的 500 行專案中，這些問題會以指數級膨脹。

技術債的複利效應

技術債就像金融債務——如果不償還，它會產生複利。今天你多花 5 分鐘寫出乾淨的代碼，可以省下未來數小時的除錯時間。反之，今天偷懶寫出的「能動就好」的代碼，會在每一次後續修改時收取利息——更長的理解時間、更高的出錯概率、更難以預測的副作用。

警告 重構的黃金法則：永遠不要在新增功能的同時進行重構。這兩件事必須分開做——先重構使代碼結構足夠清晰，確認測試通過；然後再新增功能。混在一起做，你將無法分辨 Bug 是來自重構還是新功能。

🧱 2. 關注點分離：模組化神經架構

關注點分離（Separation of Concerns, SoC）是軟體工程中最重要的設計原則之一。其核心思想是：系統中的每個模組應該只負責一件事，且模組之間的依賴關係應該降到最低。就像人體的神經系統——感覺神經負責接收、中樞神經負責處理、運動神經負責執行——三者各司其職，透過標準化的神經訊號溝通。

在物聯網控制系統中，最自然的分層方式是將邏輯拆分為三個獨立的子系統：感知層（Perception）、決策層（Control Logic）與執行層（Actuation）。

三層架構：感知 → 決策 → 執行

架構層 職責 輸入 輸出
感知層
(Perception)		 讀取感測器數據、接收通訊封包、數據預處理 原始硬體訊號 標準化的數據結構
決策層
(Control Logic)		 PID 運算、路徑規劃、狀態機管理 標準化的數據結構 控制指令（純數值）
執行層
(Actuation)		 馬達驅動、LED 控制、通訊回傳 控制指令 硬體操作

重構後的主迴圈應該像一份目錄一樣簡潔易讀，每一行都是對子模組的一次呼叫：

// 重構後的主迴圈：清晰的三層呼叫結構
void loop() {
  SensorData data = perception.update();
  ControlCommand cmd = controller.compute(data);
  actuator.execute(cmd);
}
提示 注意上述三行代碼的精妙之處：controller.compute() 完全不知道數據是來自紅外線感測器還是超音波——它只接收標準化的數據結構。這代表你可以在不修改控制邏輯的情況下更換感測器，這就是關注點分離帶來的擴展彈性。

OOP 封裝：隱藏複雜性

物件導向程式設計（OOP）的核心價值在於封裝——將複雜的實作細節隱藏在類別內部，僅對外暴露簡潔的公開介面。以通訊層為例，底層的 Web Bluetooth API 涉及 UUID 配對、特徵值讀寫、異步連接等繁瑣細節。透過 OOP 封裝，主程式只需要呼叫 comm.send(data)，無需了解任何底層協議。

// OOP 封裝範例：通訊類別
class VibeBotComm {
  #device; // 私有屬性：外部不可見
  #characteristic;

  async connect() { /* 封裝 BLE 配對流程 */ }
  async send(payload) { /* 序列化 + 寫入 */ }
  onReceive(callback) { /* 事件監聽 */ }
}

介面合約化：模組間的溝通協議

模組之間不應該共享全域變數。正確的做法是透過函式參數或事件機制來傳遞數據。這就像模組之間簽訂了一份「合約」——只要輸入格式符合合約規定，模組內部可以任意重構而不影響其他模組。

重要 判斷封裝是否成功的終極測試：如果你要將 BLE 通訊切換為 WebSocket，主程式邏輯是否一字不改？如果答案是「否」，代表你的介面隔離做得不夠徹底。這正是作業 14-2 要你證明的核心能力。

🧩 3. 插件化與管道設計模式

當你的架構已經完成了關注點分離，下一步是追求更高層次的設計目標：可擴展性（Extensibility）。理想的系統架構應該像一條工業生產線——當你需要增加新的加工步驟時，只需要在產線上「插入」一個新的工作站，而不需要拆掉整條產線重新設計。這就是插件化管道設計模式（Plugin Pipeline Pattern）的精髓。

開閉原則（Open-Closed Principle）

SOLID 設計原則中的「O」代表開閉原則：對擴展開放，對修改封閉。這意味著當你需要新增功能（例如加入卡爾曼濾波器）時，你應該能夠透過撰寫一個獨立的插件類別來實現，而完全不需要修改已經驗證穩定的核心引擎代碼。

備註 開閉原則的實踐價值在於：核心引擎一旦通過嚴格測試，就可以「鎖定」不再修改。所有新功能都透過外部插件注入。這大幅降低了因為新功能而引入回歸 Bug（Regression Bug）的風險。

管道框架的實作

管道（Pipeline）框架的核心是一個有序的處理鏈。每個插件都實作相同的介面——接收數據、處理數據、傳遞給下一站。框架本身只負責按順序呼叫各插件，不關心每個插件的具體邏輯。

// 管道框架核心
class DataPipeline {
  #plugins = [];

  addPlugin(plugin) {
    this.#plugins.push(plugin);
    return this; // 支援鏈式呼叫
  }

  process(data) {
    return this.#plugins.reduce(
      (result, plugin) => plugin.execute(result), data
    );
  }
}

撰寫獨立插件

每個插件只需要實作一個 execute(data) 方法。以下是兩個典型插件的範例：

// 插件 1：數值安全鉗位（Clamping）
class ClampPlugin {
  execute(data) {
    data.speed = Math.min(255, Math.max(0, data.speed));
    return data;
  }
}

// 插件 2：移動平均平滑濾波
class SmoothPlugin {
  #history = [];
  execute(data) {
    this.#history.push(data.speed);
    if (this.#history.length > 5) this.#history.shift();
    data.speed = this.#history.reduce((a,b) => a+b) / this.#history.length;
    return data;
  }
}

動態組裝：即插即用

管道框架的威力在於動態組裝。你可以根據不同的運行模式，在啟動時選擇性地載入不同的插件組合：

// 動態組裝管道——根據運行模式載入不同插件
const pipeline = new DataPipeline()
  .addPlugin(new ClampPlugin())
  .addPlugin(new SmoothPlugin());

// 使用管道處理數據——核心引擎代碼零修改
const result = pipeline.process({ speed: 280, direction: 45 });
// ClampPlugin 先將 280 鉗位到 255
// SmoothPlugin 再進行移動平均平滑
提示 要驗證你的管道框架是否真正符合開閉原則，試著回答這個問題：新增一個卡爾曼濾波插件需要修改幾個檔案？如果答案是「只需要建立一個新的 Plugin 檔案，然後在啟動配置中加一行 .addPlugin(new KalmanPlugin())」，那麼你的架構就已達到工業級的擴展彈性。

設計模式的價值總結

插件化管道模式帶來的三大價值：

  • 安全擴展——新功能透過獨立插件加入，既有穩定代碼零風險。
  • 靈活配置——不同的任務場景可以組裝不同的插件組合，一套框架適應多種需求。
  • 獨立測試——每個插件都可以被單獨測試（Unit Test），而不需要啟動整個系統。
警告 管道中的插件執行順序至關重要。例如，鉗位插件（Clamp）必須放在平滑插件之前——如果先平滑再鉗位，溢位值可能會汙染平均計算。在設計管道時，務必仔細考慮每個插件的前後依賴關係。

軟體架構實踐：從 Spaghetti 到 Clean Code

歡迎來到 adv-14-unit-refactoring 程式庫。本單元是從「程式員 (Coder)」蛻變為「軟體架構師 (Architect)」的核心訓練。你將學習如何處理混亂的單體代碼，並將其重構為具備可維護性、可擴展性與美感的專業架構。

🎯 學習目標

  1. 關注點分離 (SoC)：將混亂的單體檔案拆解為具備單一職責的模組。
  2. OOP 封裝：透過類別 (Class) 隱藏底層複雜性，實現乾淨的介面隔離。
  3. 開閉原則 (OCP)：打造具備擴展能力的數據管道 (Pipeline)，實踐「對擴展開放，對修改封閉」。

🚀 任務說明

任務 1：義大利麵代碼大手術 (Legacy Decoupling)

  • 對象：src/legacy_monolith.js
  • 要求：
    • 識別代碼中的「壞味道」(Code Smells)，如全域變數、深度巢狀、過長的函式。
    • 將邏輯拆解並搬移至 src/modules/ 目錄下（例如：Perception.js, Actuation.js）。
    • 確保主程式邏輯變得像目錄一樣易讀。
  • 記錄：請在 docs/Migration_Diary.md 中記錄你的重構日誌。

任務 2：通訊層 OOP 封裝

  • 要求：
    • 定義 VibeBotComm 類別，封裝底層通訊細節（如 Web Bluetooth API）。
    • 隱藏內部狀態（如 UUID、連線狀態），僅對外暴露必要的 API。
    • 證明其靈活性：如果未來要切換通訊協議，主程式邏輯應能保持不變。

任務 3：插件化處理管線 (Pipeline Framework)

  • 要求：
    • 設計一個數據處理管道 (Pipeline)，支持按序執行插件 (Middleware)。
    • 實作至少兩個插件（例如：平滑濾波、數值安全限制）。
    • 展示如何在不修改主框架的情況下，動態增加新功能。

📁 程式庫結構

  • src/legacy_monolith.js: 待重構的原始混亂代碼。
  • src/modules/: 存放重構後的各類功能模組。
  • src/plugins/: 存放符合 Pipeline 規範的插件。
  • docs/Migration_Diary.md: 你的重構過程記錄。
  • tests/: 自動化測試腳本，用於驗證重構後的架構品質。

📝 提交要求

  1. 完成所有重構任務，確保程式碼能通過 tests/ 中的架構檢核。
  2. 更新 docs/Migration_Diary.md，詳細描述你解決了哪些壞味道，以及使用了哪些重構手法。
  3. 確保 README.md（本檔案）最後包含你的結案感言。

「任何傻瓜都能寫出電腦懂的代碼。好的工程師能寫出人類懂的代碼。」 — Martin Fowler

針對 adv-14-unit-refactoring (代碼重構) 單元，這是從「程式員 (Coder)」蛻變為「軟體架構師 (Architect)」的核心訓練。它不再關注功能實現，而是關注代碼的 可維護性、可擴展性與閱讀美感。在 Vibe Coding 這種跨平台系統中，優雅的架構是防止系統隨功能增加而崩潰的唯一防線。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template Repo) 配置建議

重構教學需要一個「待手術」的正反面教材，建議範本包含：

  • 📂 src/legacy_monolith.js：重構手術對象。一個滿是「壞味道」的 500 行單體大檔案，裡面充斥著全域變數、深度巢狀的 if-else 以及職責不分的混亂 loop。
  • 📂 src/modules/：目標組件目錄。學員需將拆解後的邏輯（如 Perception.js, Actuation.js）放置於此。
  • 📂 docs/REFACTOR_LOG.md：重構手術日誌。記錄：原始行號 ➡ 提取後的函式/類別 ➡ 解決的壞味道（如：消除全域依賴）。
  • 📂 tests/architecture.test.js：架構完整性測試。這不是測功能，而是檢核：1. 主程式是否還有超過 50 行的函式？ 2. 通訊層是否實現了真正的 OOP 封裝？

2. 作業任務部署細節

任務 1：義大利麵代碼之大手術 (Legacy Decoupling Surgery Lab)

  • 目標：練習「關注點分離 (SoC)」手法，解決單體代碼的認知負荷問題。
  • Classroom 部署建議：
    • 職責解構：學員需將 legacy_monolith.js 拆解為具備單一職責的純函數。
    • 介面合約化：確保子模組間透過引數傳遞而非共享全域變數。
    • README 日誌：說明你用了哪些 Clean Code 技巧（如：函式提取、衛句取代巢狀結構）。
    • 驗證要點：主 loop 代碼是否變得像目錄一樣易讀？

任務 2：通訊層 OOP 封裝 (Communication Layer OOP Lab)

  • 目標：透過類別封裝，隱藏底層複雜性，實現「介面隔離」。
  • Classroom 部署建議：
    • Class 封裝實作：定義 VibeBotComm 類別。封裝 Web Bluetooth API 的異步連接與特徵值讀寫。
    • 私有域保護：確保 UUID 與連線狀態對外部不可見。
    • 測試切換能力：要求學員證明：如果未來要切換到 WebSocket 傳輸，主程式邏輯是否「一字不改」？
    • 提交結果：提交一個乾淨的、可重用的通訊組件 JS 檔。

任務 3：插件化處理管線架構 (Dynamic Pipeline Framework Lab)

  • 目標：實踐「開閉原則 (OCP)」，打造具備擴展能力的數據管道。
  • Classroom 部署建議：
    • 處理管道實作：設計一個載體，能按序執行 Middleware。
    • 插件注入演示：學員撰寫兩個對應插件（如：平滑濾波插件、數值安全鉗位插件）。
    • 動態擴展展示：在 PR 中展示如何只撰寫 10 行代碼新增一個功能，且不需要修改既有的 Pipeline 主框架。
    • Final Push：提交具備「工業級彈性」的高階控制引擎架構。

3. 軟體架構組件導師點評標準 (Clean Code & Architecture Standards)

此單元的價值在於 「對對抗代碼熵增的防禦深度」：

  • [ ] 封裝完備度：模組間是否做到了「知道的越少越好 (Least Knowledge)」？
  • [ ] API 語意化：設計的類別介面是否自然流暢，能讓其他開發者快速上手而無需閱讀內部實作？
  • [ ] 架構彈性：系統是否支持「不修改原始碼即能增加功能」？（展現對 OCP 的理解）。
