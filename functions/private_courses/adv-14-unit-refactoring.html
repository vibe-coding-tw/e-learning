從「義大利麵代碼」進化到優雅的軟體架構。學習償還技術債，設計高內聚、低耦合的控制系統。

👃 1. 識別「代碼壞味道」 (Code Smells)

重構不是增加新功能，而是優化內部結構。好的代碼應該像優美的散文，讓人讀起來賞心悅目。學會「聞出」隱藏在開發初期的技術債。

Common Technical Debt Symptoms

  • 冗長函式 (Long Method)：主迴圈超過 50 行，且混雜了感測、計算與輸出職責。
  • 魔法數字 (Magic Numbers)：代碼中充滿寫死的數值標籤，缺乏具名常數定義。
  • 緊密耦合 (Spaghetti)：模組間相互依賴，改動通訊層卻導致 PID 計算損毀。

🧱 2. 關注點分離：模組化神經架構

將系統拆解為不互相依賴的獨立模組。感知 (Perceive)、決策 (Decide) 與執行 (Act) 三權分立，實現極高內聚且低耦合的軟體結構。

🧩 3. 插件化與管道設計模式

追求硬核的系統擴展性。設計「處理鏈」模式，在不修改核心引擎的前提下，僅需注入獨立組件即可動態載入新的控制策略或遙測濾波器。

SOLID Design Principles

  • 對擴展開放 (Open)：新增功能只需撰寫獨立 Plugin Class，無需改動既有穩定代碼。
  • 對修改封閉 (Closed)：核心驅動邏輯經過驗證後，不再因為業務變動而被頻繁改寫。
  • 數據管道模式 (Pipeline)：像工業生產線一樣，數據流按序流經各個異步處理站。

核心任務：打造具備生命力的精品代碼

償還技術債，設計高內聚、低耦合的系統架構。當你能寫出讓後繼者願意維護的代碼，你就是資深工程師。透過 Git Workflow 實踐大規模代碼解構與封裝任務。

✂️ 60 min

【作業 14-1】

「代碼外科手術」

挑戰拆解一段 500 行的單體控制代碼。利用「函式提取」手法將職責混亂的 loop 邏輯抽像為功能單一的模組（如：遙測層、控制層），落實 Clean Code 核心精神。

📦 60 min

【作業 14-2】

「通訊類別封裝」

使用 JavaScript Class 封裝 BLE 連線邏輯。建立具備事件觸發能力的通訊組件，確保主程式與底層 API 相隔離。證明當切換傳輸介面時，你的核心邏輯完全無需修改。

🧩 45 min

【作業 14-3】

「數據插件管線」

設計一個處理管線框架。實現「即插即用 (PnP)」架構，撰寫兩個獨立插件（如：平滑濾波、閾值攔截），並證明只需透過 addPlugin 即可動態改變系統行為。

單元課程 進階 14 - 重構藝術

重構實驗室：架構病態分析與精品組件設計

任務 1：義大利麵代碼之大手術 (Legacy Decoupling Surgery Lab)

角色扮演：代碼重構專家 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 嗅覺掃描 (Smell Detection)：找出專案中「神之類別」或「魔法常數」，並將其標記為重構目標。
  3. 關注點分離 (SoC)：將原始單體 loop 拆解為 Perception 與 Action 模組，確保邏輯流清晰且具備命名意義。
  4. Push Final：提交具備「高度可讀性、職責單一」的重構後代碼與思路說明。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 命名精準度：變數與函式命名是否能「自我解釋 (Self-documenting)」，減少註解依賴？
  • [ ] 解耦成效：全域變數是否被有效限制？代碼的巢狀深度是否有顯著改善？

任務 2：軟體通訊層 OOP 封裝 (Communication Layer OOP Lab)

角色扮演：軟體架構組件開發者 | 難度：⭐️⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 介面定義：定義通訊模組 Class。封裝 Web Bluetooth API，對外僅暴露 write(data) 與 onMessage()。
  2. 私有狀態保護：確保底層的特徵值 UUID 與連結 Buffer 隱藏在類別私有域中，防止外部非法篡改。
  3. 連線生命週期管理：在 Class 內部處理「重連指數退避 (Exponential Backoff)」邏輯，優化使用者控制體驗。
  4. Push Final：推送具備「高穩定性、可重用性」的獨立通訊模組組件。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 封裝完整度：主邏輯檔案中是否完全不再直接呼叫瀏覽器原生 BLE API？
  • [ ] API 簡潔性：設計的 Class 介面是否直觀易用，能否讓同學在 5 分鐘內學會調用？

任務 3：動態處理管線架構 (Dynamic Pipeline Framework Lab)

角色扮演：資深框架開發者 | 難度：⭐️⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 框架架構設計：實作一個 Pipeline 容器類別，具備線性執行各級插件 (Middleware) 的能力。
  2. 組件對接實施：撰寫「低通濾波插件」與「數值鉗位插件」，並證明在不重啟系統下可動態抽換順序。
  3. 擴展性驗證：撰寫一份 10 行以內的「新插件範本」，證明系統滿足「開閉原則 (OCP)」。
  4. Push Final：提交具備「現代化軟體設計美感」的高階數據處理管線。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 架構優雅度：插件間的數據傳遞是否具備標準格式（Pipeline Contract）？
  • [ ] OCP 達成率：當新增處理邏輯時，是否真正做到了「無須修改 Pipeline Host 代碼」？

GitHub Classroom 詳細操作指南

本單元教學重點：引導學生從「程式員 (Coder)」轉型為「架構師 (Architect)」。將關注點從功能實現轉向維護性。

  1. 重點 1：對抗認知負荷 - 強調重構不是為了炫技，而是為了讓其他人能用更短的時間看懂代碼。
  2. 重點 2：技術債的感知教育 - 引導學員識別「代碼壞味道」。培養對魔術數字與重複邏輯的高敏感度。
  3. 重點 3：開閉原則 (OCP) 的力量 - 讓學員理解為什麼現代工業級系統必須透過「插件化」來應對頻繁變動的需求。
