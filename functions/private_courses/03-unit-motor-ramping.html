Ramping 原理示意

為什麼需要 Ramping？

當您在操控無人車時，若直接將 PWM 從 0 瞬間拉到 255，會發生以下問題：

  • 電流突波：瞬間大電流可能導致 ESP32 重啟。
  • 機械應力：齒輪箱承受瞬間撞擊，容易損壞。
  • 打滑失控：輪胎失去抓地力，導致車輛打轉。
rampAccelStep 就像是「虛擬彈簧」，數值越小，彈簧越軟（加速越平滑）；數值越大，反應越直接。

技術開發細節

🛠️ 技術開發細節 (For Developer)

C++ Struct Definition Total: 28 Bytes 
struct MotorConfig_t {
  uint32_t controlTimeoutMs;   // Offset 0  (4 bytes)
  int32_t  pwmEffectiveLimitT; // Offset 4  (4 bytes)
  int32_t  rampAccelStepT;     // Offset 8  (4 bytes)
  int32_t  pwmStartKickT;      // Offset 12 (4 bytes)
  int32_t  pwmEffectiveLimitS; // Offset 16 (4 bytes)
  int32_t  rampAccelStepS;     // Offset 20 (4 bytes)
  int32_t  pwmStartKickS;      // Offset 24 (4 bytes)
};

為什麼使用二進位傳輸？

1. 極致效能：28 Bytes vs JSON 字串 (~150 Bytes)。

2. 零解析開銷：嵌入式端只需 memcpy，無需 JSON Parser。

錯誤排查

❌ 無法掃描到裝置

請重新插拔 ESP32 電源，並確認藍牙權限已開啟。

⚠️ 數據長度錯誤

若讀取時發生 Error，代表網頁端定義的 28 Bytes 與韌體端不一致，請檢查 Struct 對齊。

❌ 寫入後沒反應

請確認數值是否為整數 (Integer)。DataView 對型別非常敏感。

馬達 Ramping 組態設定 (Motor Ramping Configuration)

歡迎來到本單元！在這個練習中，你將學習如何透過網頁介面與硬體進行「二進位通訊協議 (Binary Protocol)」的對接，並實作關鍵的物理安全控制邏輯。

🎯 學習目標

  1. 前端安全防護：學習如何在 UI 層級進行數值驗證，防止非法參數導致硬體損毀。
  2. 二進位資料處理：熟練使用 JavaScript 的 DataView 與 ArrayBuffer 來封裝二進位封包。
  3. 通訊協議對齊：確保網頁端封包與硬體端 (ESP32) 的記憶體結構完全吻合（28 Bytes, Little-Endian）。
  4. 物理安全驗證：理解並驗證硬體端的超時保護 (Fail-Safe) 機制。

🛠️ 任務說明

任務 1：軟體保險絲 - 前端輸入驗證

請在 ramping-app.html 中實作輸入驗證邏輯：

  • 油門限制 (PWM Limit)：數值必須在合法範圍內。
  • 驗證機制：若輸入非法數值，應禁用「傳送」按鈕或顯示警告紅框。
  • 重要性：這是防止馬達因非法輸入（如設為 999）而燒毀的第一道防線。

任務 2：二進位橋樑 - 資料結構映射

你需要實作二進位封包的填充邏輯：

  • 目標檔案：ramping-app.html 中的 // TODO 區塊。
  • 規格：
    • 總長度：28 Bytes。
    • 編碼格式：Little-Endian。
    • 關鍵偏移量：例如 pwmStartKickT 應位於偏移量 12 的位置。
  • 驗證方式：執行 verify-bytes.js（如有提供）或查看序列埠輸出的十六進位字串。

任務 3：物理安全保護驗證

驗證硬體的「安全停車」機制：

  1. 將 controlTimeoutMs 設定為 500ms。
  2. 啟動馬達後，立即關閉控制網頁。
  3. 截圖證明：從序列埠日誌中截圖，證明馬達在 0.5 秒內自動停止（應出現 [SAFETY] Timeout - Motor Stop 訊息）。

📂 程式庫結構

  • ramping-app.html：主要的網頁控制程式（需完成 UI 驗證與 DataView 邏輯）。
  • test-protocol.js：用於本地測試二進位解析的腳本。
  • firmware/：對應的 ESP32 韌體源碼（供參考與實機測試）。

📝 繳交規範

  1. 確保 ramping-app.html 已通過自動評分測試。
  2. 提交包含所有程式碼修改的 Commit。
  3. （選繳）上傳任務 3 的安全停車截圖。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫配置建議

由於此單元涉及精確的位元組對齊（28 Bytes），建議範本中包含：

  • 📂 ramping-app.html：提供 UI 骨架，但將 DataView 的讀寫邏輯改為 // TODO。
  • 📂 test-protocol.js：預置一個單元測試，模擬一組 28 位元組的二進位數據，讓學生練習是否能正確解析回數值。
  • 📂 firmware/main.cpp：提供對應的 ESP32 韌體源碼，讓有硬體的學生能進行真實測試。

2. 作業任務部署細節

任務 1：軟體保險絲 - 前端輸入驗證 (UI Validation)

  • 目標：防止非法數值（如 PWM 限制設為 999）導致馬達損毀。
  • Classroom 部署建議：
  • 自動評分 (Autograding)：使用無頭瀏覽器測試（如 Playwright）。
  • 測試案例：嘗試在油門限制欄位輸入 999。如果「傳送」按鈕沒有被禁用 (Disabled) 或沒有出現紅框警告，則自動評分失敗。這能訓練學生重視「前端第一線安全控管」。

任務 2：二進位橋樑 - 資料結構映射 (DataView Lab)

  • 目標：精確操作位元組偏移量 (Offset) 與位元編碼格式 (Little-Endian)。
  • Classroom 部署建議：
  • 核心代碼檢核： 
    // 學生需實作：偏移量 12 應對應到 pwmStartKickT
view.setInt32(12, parseInt(inputs.pwmStartKickT.value), true);
  • 驗證方式：學生需撰寫一個 verify-bytes.js，將填充好的 ArrayBuffer 轉換為十六進位字串（Hex String）並與導師提供的預期結果比對。

任務 3：斷電與超時保護驗證 (Fail-Safe Check)

  • 目標：驗證韌體的「安全停車」機制。
  • Classroom 部署建議：
  • 操作證明：由於這涉及物理行為，要求學生提交一段 GIF 錄影 或 序列埠日誌截圖。
  • 任務內容：設定 controlTimeoutMs 為 500ms，啟動馬達後關閉網頁。截圖需證明馬達在半秒內自動停止（終端機出現 [SAFETY] Timeout - Motor Stop）。

3. 翻轉教室與導師評估指標

  • 教學重難點：
  • Endians (位元組序)：很多學生會卡在為什麼數據讀出來是亂碼（因為沒設 true 開啟 Little-Endian）。
  • Struct Alignment (記憶體對齊)：引導學生思考為什麼 uint32 佔 4 個位元組。
  • 評語建議 (Tutor Benchmarks)：
  • [ ] 封裝正確性：DataView 寫入時是否使用了正確的整數型別 (Int32 vs Uint32)？
  • [ ] 容錯邏輯：是否處理了「空值」或「非數字」輸入的例外狀況？
  • [ ] 協議文件化：是否在 README 中清楚標註了 28 Bytes 分別代表哪些資料項？

這套部署方案能將原本抽象的「二進位協議」轉化為「可測試、可觀察」的具體任務，讓學生在學習 Web 控制硬體時，建立起極致效能與物理安全性併重的觀念。
