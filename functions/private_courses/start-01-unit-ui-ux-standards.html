Module 1: Thumb Zone

拇指操作區與人體工學

在設計手機遙控介面時，**Thumb Zone（拇指熱區）** 是最關鍵的概念。了解雙手持握時，拇指能輕鬆觸及的範圍，決定了核心按鈕的佈局。

色彩編碼

顏色傳遞訊息，不需文字。紅色與黃色能瞬間抓住注意力，綠色則給人安全感。

📏

大小差異

最重要的功能應該最大。根據 Fitts's Law，目標越大、距離越近，操作所需時間越短。

🧠

資訊降噪

FPV 競速時，駕駛者沒有時間閱讀。只顯示當下必要的數據，隱藏次要資訊，降低認知負荷。

🎯 學習目標

預習如何減少操作者的認知負擔，讓使用者在高速飛行時不需要「思考」按鈕在哪裡，而是憑「直覺」操作。

1. 「盲操挑戰」紀錄表 (Blind Operation Challenge)

實驗說明：請閉上眼睛（或不看螢幕），嘗試點擊畫面上的「STOP」按鈕與「方向鍵」。請他人協助紀錄點擊成功與否。

盲操成功率：90 % (目標：> 80%)

2. 設計決策問答 (Design Decisions)

  • Q1: 為什麼將「電量顯示」與「訊號強度」放在螢幕頂端邊緣？ 回答： 這些屬於持續監看但不需要頻繁點擊的狀態資訊，放在頂端邊緣可以保留中央視野，並避免干擾主要操控區。
  • Q2: 為什麼核心控制按鈕（STOP, FWD）集中在底部 1/3 區域？這符合哪種人體工學理論？ 回答： 底部 1/3 是拇指自然最容易到達的區域，能降低誤觸與操作時間，這符合 Thumb Zone 與行動裝置人體工學的操作原則。
  • Q3: 當系統進入 .state-warning 時，你設計了哪些視覺反饋？為什麼「單純改變顏色」是不夠的？ 回答： 我加入了紅色脈衝動畫、警示文案與高對比色塊，因為只有顏色變化可能在緊張情境或低注意力狀態下不夠明顯，動畫與文字能提高即時辨識率。

3. 視野安全區證明 (Safety Zone Verification)

請在 assets/ 資料夾中上傳一張截圖 test_screenshot.png，證明你的 HUD 數據沒有遮擋中央 40% 的核心視野區。

針對 start-01-unit-ui-ux-standards (UI/UX 設計規範：從視覺到體驗) 單元

這是學員從「排版」跨入「產品設計」的重要門檻。它要求學員具備 人機工程 (Ergonomics) 的意識，學習如何根據「拇指熱區 (Thumb Zone)」分配功能權限，並實作 FPV 抬頭顯示器 (HUD) 與異常狀態的視覺反饋，確保控制介面在高速操作下依然直覺且安全。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 配置建議

UI/UX 規範作業的核心在於「層級設計」與「用戶體驗測試」，範本應包含介面原型與測試框架，建議包含：

  • 📂 index.html：HUD 疊加架構。預設包含一個 video-container 與空的 hud-overlay 容器，讓學員專注於資訊層級的堆疊。
  • 📂 style.css：視覺規範定義區。學員在此實作半透明背景 (rgba)、高對比顏色警示以及異常狀態下的 @keyframes 閃爍動畫。
  • 📂 README.md：UX 研究報告樣板。提供數據紀錄表，讓學員紀錄「盲操挑戰」的誤觸率，並要求寫下：為什麼電量顯示要放在特定位置？
  • 📂 assets/：提供一套標準的 UI 圖標（電池、信號、警告），確保學員能快速建立視覺一致性。

2. 作業任務部署細節

任務 1：「盲操」佈局挑戰 (Blind Operation Layout Lab)

  • 目標：實踐拇指熱區理論。設計一個讓使用者在不看螢幕的情況下，也能憑觸覺位置完成核心控制的佈局。
  • Classroom 部署建議：
    • 熱區佈局規劃：將核心按鈕（停止、前進、轉向）集中於底部 1/3 的「Natural Zone」。
    • 測試紀錄提交：學員必須在 README 中紀錄至少 10 次盲操的成功次數。
    • 驗證要點：人體工學審核。檢查佈局是否避免了將關鍵按鈕放在螢幕頂端角落？按鈕尺寸是否確實大於觸控建議值？

任務 2：FPV HUD 抬頭顯示器 (FPV HUD Design Lab)

  • 目標：掌握資訊降噪與視覺 Z 軸。在動態影像上疊加重要遙測數據，且不干擾駕駛視線。
  • Classroom 部署建議：
    • 資訊層次化：要求電量、連線延遲 (Ping) 必須具備半透明背景，確保在不同背景畫面下都可讀。
    • 視野安全區實作：核心視野（畫面中央 50%）嚴禁放置非必要的文字資訊。
    • 驗證要點：HUD 元件是否具備足夠的對比度？文字是否太細小以至於在移動中難以辨識？

任務 3：異常狀態 UX 處理 (Abnormal States UX Lab)

  • 目標：設計「生存等級」的反饋機制。當系統出錯（斷線、低電量）時，介面必須主動且強烈地提醒操作者。
  • Classroom 部署建議：
    • 狀態驅動設計：定義一組 CSS 類別（如 .state-warning），當加載於容器時，邊框會產生紅色脈衝閃爍。
    • 多維警示實作：要求低電量提示必須包含「文案 + 顏色 + 動畫」三要素。
    • 驗證要點：警告動畫的節奏是否合適（既能引起注意又不過度干擾操作）？顏色編碼是否符合通用的安全規範？

3. 可用性與視覺一致點評標準 (Usability & Visual Standards)

此單元的價值在於 「對使用者操作負荷的同理心與對界面信息優先級的決斷力」：

  • [ ] 操作直覺性：在盲操測試中，核心功能的佈局是否符合人類生理習慣？
  • [ ] 視覺降噪純度：HUD 是否達成了「只顯示當下必要數據」的工程目標？
  • [ ] 異常反應速度：當狀態改變時，介面是否能量化地表現出「緊迫感」？
