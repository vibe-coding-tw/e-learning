簡介

HTTP 是網頁、伺服器與物聯網設備互相「對話」的標準語言——每一次交互都由一組請求（Request）與回應（Response）構成。 本單元將帶你拆解 HTTP 的三大核心元素：方法（GET / POST）、標頭（Headers）與狀態碼（200 / 404 / 500）， 讓你的 ESP32 能像專業的 API 客戶端一樣，精確地與雲端服務進行數據交換。

情境
你的 ESP32 嘗試將感測器數據上傳到雲端，卻收到了一個神秘的「500 Internal Server Error」回覆。 你不確定問題出在 JSON 格式、缺少 Content-Type 標頭，還是伺服器端本身的邏輯錯誤——這時你才發現， 理解 HTTP 的「對話規則」是排除一切網路故障的起點。

理解 HTTP 對話

HTTP（HyperText Transfer Protocol，超文本傳輸協定）是全球資訊網（WWW）的基礎通訊協議。 無論你在瀏覽器中載入一個網頁、透過手機 App 查看天氣，還是讓 ESP32 將感測器數據上傳到雲端，底層都是透過 HTTP 在傳遞訊息。 理解 HTTP 的運作機制，是成為全端開發者與物聯網工程師的第一步。

餐廳點餐：HTTP 的直覺類比

最直觀的理解方式就是把 HTTP 想像成「餐廳點餐」的流程。在這個類比中，客戶端（Client）是顧客， 伺服器（Server）是廚房，而 HTTP 就是服務員在兩者之間傳遞的那張「點餐單」和「出餐單」。

每一次完整的 HTTP 互動都由兩個部分組成：

  • 請求（Request）：客戶端向伺服器發出的「點餐單」——告訴伺服器你要什麼資源、用什麼方法取得、附帶哪些額外說明。
  • 回應（Response）：伺服器回覆的「出餐單」——告訴客戶端請求是否成功、附帶狀態碼與實際的資料內容。
注意 HTTP 是一種無狀態（Stateless）協議。這意味著伺服器不會記住任何一次請求的前後文——每一次的 Request/Response 都是獨立的。 就像餐廳服務員不會自動記得你上次點了什麼菜，每次你都需要重新告訴他。如果需要「記憶」，則需要額外機制如 Cookie 或 Session。

HTTP 對話的完整結構

一次完整的 HTTP 對話可以拆解為以下流程。無論是瀏覽器存取網頁，還是 ESP32 上傳感測器數據，都遵循相同的結構：

階段 描述 餐廳類比
建立連線 客戶端透過 TCP 與伺服器建立連線（三次握手） 走進餐廳、坐下、服務員走過來
發送 Request 客戶端發出請求，包含方法、URL、標頭與正文 把寫好的點餐單交給服務員
處理請求 伺服器解析請求內容，執行對應的邏輯 廚房收到菜單，開始備料烹調
回傳 Response 伺服器回覆狀態碼、標頭與回應正文 菜做好了，服務員把餐點端上桌
關閉連線 傳輸完成後，連線結束（HTTP/1.1 可重用） 吃完結帳離開（或繼續加點）

ESP32 與 HTTP：物聯網的對話場景

在物聯網（IoT）場景中，ESP32 通常扮演 HTTP 客戶端的角色——它主動向雲端伺服器發送請求， 上傳感測器數據或取得指令。但在某些情境下（例如建立本地控制台網頁），ESP32 也可以同時扮演 HTTP 伺服器的角色， 接收來自手機或電腦瀏覽器的請求。

// ESP32 作為 HTTP Client 發送請求的基本流程
HTTPClient http;
http.begin("http://api.example.com/data");

int httpCode = http.GET(); // 發送 GET 請求
String payload = http.getString(); // 取得回應正文

http.end(); // 關閉連線
提示 在學習 HTTP 時，建議使用 Postman 或瀏覽器的 DevTools（開發者工具）觀察真實的 Request 與 Response。 你可以清楚看到每次請求的方法、標頭、狀態碼與回應內容，這比閱讀文件更加直覺。

方法、標頭、正文

每一個 HTTP 請求都像一封格式嚴謹的信件，由三個核心部分組成：方法（Method）告訴伺服器你想做什麼動作、 標頭（Headers）提供這次請求的附加資訊、正文（Body）承載要傳送的數據內容。 理解這三個要素，就能讓你的 ESP32 與任何伺服器進行精確的對話。

HTTP 方法：GET 與 POST

HTTP 定義了多種請求方法（也稱為「動詞」），其中最常用的兩個是 GET 和 POST。 在物聯網開發中，你幾乎每天都會使用它們：

比較項目 GET POST
用途 讀取資源（查詢、獲取資料） 提交資料（新增、修改）
參數位置 附在 URL 的 Query String 中 封裝在請求的 Body 中
安全性 參數暴露在網址列，不適合傳密碼 參數隱藏在 Body 中，較安全
資料長度 受 URL 長度限制（約 2048 字元） 理論上無限制
IoT 場景 讀取感測器數值、查詢設備狀態 上傳溫濕度數據、發送控制指令
警告 切勿使用 GET 方法傳輸敏感資料！GET 的參數會出現在瀏覽器歷史紀錄、伺服器日誌，甚至會被搜尋引擎索引。 例如 http://api.com/login?password=1234 會讓密碼完全暴露。

URL 結構解析

在 ESP32 開發中，你經常需要手動組裝請求的目標 URL。一個完整的 URL 由以下部分組成：

// URL 結構拆解
http://192.168.4.1:80/move?dir=forward&speed=200

// 拆解：
// 協議 (Protocol) : http://
// 主機 (Host) : 192.168.4.1
// 埠號 (Port) : 80 (預設可省略)
// 路徑 (Path) : /move
// 查詢 (Query) : dir=forward&speed=200

HTTP 標頭（Headers）

標頭是 HTTP 訊息中的「附加說明」，以 Key: Value 的形式傳送。它們不承載實際的數據內容， 但提供了伺服器理解與處理請求所需的關鍵後設資料（metadata）。以下是物聯網開發中最常見的標頭：

標頭名稱 說明 範例值
Content-Type 告訴伺服器 Body 中的資料格式 application/json
Authorization 攜帶身分驗證憑證（Token） Bearer eyJhbG...
User-Agent 識別客戶端的類型與版本 ESP32HTTPClient
Accept 告訴伺服器客戶端能接受的回應格式 application/json
重要 當你使用 POST 傳送 JSON 數據時，必須手動設定 Content-Type: application/json 標頭。 如果遺漏了這個標頭，伺服器將無法正確解析你傳送的 JSON 資料，通常會回傳 400 Bad Request 錯誤。

ESP32 POST 請求實作範例

以下是一個完整的 ESP32 POST 請求範例，展示如何設定標頭並將感測器數據以 JSON 格式傳送到伺服器：

HTTPClient http;
http.begin("http://api.example.com/sensor");

// 設定標頭——告訴伺服器我們傳的是 JSON
http.addHeader("Content-Type", "application/json");

// 組裝 JSON 正文
String body = "{\"temperature\":25.3,\"humidity\":68}";

// 發送 POST 並取得狀態碼
int code = http.POST(body);
Serial.printf("HTTP Code: %d\n", code);
提示 手動拼接 JSON 字串容易出錯（漏掉逗號、引號跳脫錯誤等）。建議使用 ArduinoJson 函式庫來序列化數據， 它會自動處理格式問題，產出合法的 JSON 字串。

狀態碼

每次 HTTP 回應的第一行都會包含一個三位數的狀態碼（Status Code），它就像伺服器給你的「成績單」—— 一看數字，就能立即判斷這次請求是成功、失敗，還是需要進一步處理。 對 ESP32 開發者來說，正確解讀狀態碼是實現自動化錯誤處理與重試機制的基礎。

狀態碼的五大家族

HTTP 狀態碼的第一位數字代表「回應類別」，共分為五大家族。在物聯網開發中，你最常遇到的是 2xx、4xx、5xx 三個類別：

範圍 類別 說明 IoT 場景範例
1xx 資訊性 請求已收到，繼續處理中 較少遇到，通常由底層處理
2xx 成功 請求已被成功接收、理解並處理 數據成功上傳到雲端
3xx 重新導向 需要進一步操作才能完成請求 API 網址已搬遷
4xx 客戶端錯誤 請求包含錯誤，伺服器無法處理 ESP32 發送的 URL 拼錯
5xx 伺服器錯誤 伺服器無法完成合法的請求 雲端伺服器當機或程式崩潰

ESP32 開發者必知的關鍵狀態碼

以下是你在物聯網開發中最常遇到的狀態碼，以及對應的處理策略：

狀態碼 意義 ESP32 應對策略
200 OK 請求成功，回應包含數據 正常解析回應正文，繼續執行
201 Created 資源已成功建立 POST 上傳數據成功，可記錄日誌
400 Bad Request 請求格式錯誤 檢查 JSON 格式或缺少的 Header
404 Not Found 請求的資源不存在 檢查 URL 是否正確，不重試
500 Internal Error 伺服器內部錯誤 等待後重試（伺服器端問題）
警告 ESP32 的 HTTPClient 函式庫在網路層級錯誤（如 DNS 解析失敗、連線逾時）時， 回傳的不是標準 HTTP 狀態碼，而是負數的錯誤碼（例如 -1 表示連線失敗）。 在撰寫判斷邏輯時，必須先排除負數的情況。

實作狀態碼分流處理

以下範例展示如何根據不同的狀態碼執行不同的處理邏輯，並實作基於 millis() 的非阻塞重試機制：

int code = http.GET();

if (code == 200) {
  String data = http.getString();
  Serial.println("成功取得: " + data);
} else if (code == 404) {
  Serial.println("資源不存在，請檢查 URL");
} else if (code == 500) {
  Serial.println("伺服器錯誤，稍後重試...");
  retryAt = millis() + 5000; // 5 秒後重試
} else if (code < 0) {
  Serial.println("網路連線失敗!");
}
提示 在重試邏輯中，嚴禁使用 delay()！它會阻塞整個 loop() 循環， 導致感測器採樣、LED 閃爍等其他任務全部停擺。請使用基於 millis() 的「非阻塞狀態機」來實現定時重試。

作業指南：HTTP 協議 - 請求與回應 (basic-06-unit-http-request)

歡迎來到物聯網通訊的核心單元！在這個實驗中，你將學習如何讓你的 ESP32 與網路世界進行「對話」。你將不再只是發送盲目的數據，而是學會使用標準的 HTTP 協議與伺服器溝通。

🎯 學習目標

  1. 通訊解析：理解 HTTP Headers 的作用，並學會區分「請求 (Request)」與「回應 (Response)」。
  2. 數據封裝：掌握 POST 方法，並學習如何將感測器數據包裝成標準的 JSON 格式。
  3. 系統韌性：實作非阻塞式的錯誤處理與重試機制，確保設備在網路不穩定時仍能正常運行。

🚀 任務說明

任務 1：「數位點餐員」- Postman GET 測試

  • 使用 Postman 發送一個 GET 請求到測試端點。
  • 分析並記錄回應中的 Content-Type、Status Code 以及 Date 標頭。
  • 在 docs/ 目錄下建立一個紀錄，說明為什麼 GET 請求不適合傳輸機密數據。

任務 2：「數據整合師」- 實作 POST 發送 JSON

讓你的 ESP32 將感測器數據上傳到雲端伺服器。

  • 在 src/main.cpp 中實作 POST 請求。
  • 關鍵要求：必須手動添加 Content-Type: application/json 的標頭。
  • 使用 ArduinoJson 函式庫將數據（如溫度、濕度）序列化為 JSON 格式。

任務 3：「通訊守門員」- 狀態碼攔截與防錯

建立一個強大的通訊邏輯。

  • 根據伺服器回傳的 Status Code（如 200, 404, 500）執行不同的處理邏輯。
  • 核心工程挑戰：實作基於 millis() 的 非阻塞式重試機制。
  • ⚠️ 嚴禁 在通訊重試循環中使用 delay()，這會卡死設備的其他任務。

📁 程式庫結構

  • src/main.cpp: 核心邏輯實作區。
  • include/api_config.h: 設定你的 WiFi 資訊與 API 端點。
  • assets/retry_logic.md: 關於非阻塞重試策略的參考說明。

📝 繳交要求

  1. 確保代碼可以正確通過 GitHub Actions 的自動化測試。
  2. 程式碼中必須包含正確的 JSON 標頭設定。
  3. loop() 函數中不得包含阻塞式的 delay()。

2. 作業任務部署細節

任務 1：「數位點餐員」Postman GET 測試 (Digital Waiter Lab)

  • 目標：學會使用專業工具解構 HTTP 封包，理解 Headers 在通訊中的「語境提供」作用。
  • Classroom 部署建議：
    • 端點勘查：發送 GET 到一個 JSON API。
    • 屬性拆解：找出 Response 中的 Content-Type 與 Date。
    • 安全性反思：撰寫報告說明：為什麼 GET 參數放在網址後面（Query String）對機密資料（如密碼）是不安全的？
    • 驗證要點：README 中的 JSON 欄位解釋是否精確對齊 API 規範。

任務 2：「數據整合師」實作 POST 發送 JSON (Data Integrator Lab)

  • 目標：掌握 POST 方法與 JSON 序列化，建立「韌體推播數據」的標準流程。
  • Classroom 部署建議：
    • 封裝實作：在 ESP32 中將感測器數據包裝成 C++ 字串或使用 ArduinoJson 函式庫。
    • 標頭配置：核心關鍵點。必須手動添加 http.addHeader("Content-Type", "application/json");。
    • 數據提交：執行 POST 請求並驗證伺服器端的成功回傳。
    • 驗證要點：檢核 POST Body 是否為合法的 JSON 格式。

任務 3：「通訊守門員」狀態碼攔截與防錯 (Communication Sentry Lab)

  • 目標：建立強韌的異常處理模型，學會根據狀態碼分流邏輯，並確保通訊重試「不阻塞」設備運行。
  • Classroom 部署建議：
    • 碼分流診斷：模擬 404（網址錯）與 500（伺服器崩潰）情境，並在 Serial Monitor 輸出對應的提示。
    • 非阻塞重試：若失敗，實作基於 millis() 的非阻塞重試機制。
    • 使用者反饋：透過螢幕或 LED 閃爍模式，告知目前通訊狀態（成功/失敗/重試中）。
    • 驗證要點：關鍵工程審核。學員是否在通訊重試循環中使用了 delay()？若是，則扣分（這會卡死感測器採樣），應使用「非阻塞狀態機」。

3. 通訊協議與格式化點評標準 (Protocol Compliance & Resilience Standards)

此單元的價值在於 「對通訊規範的嚴格遵循與對異常狀態的分類處理」：

  • [ ] 標頭賦能意識：是否理解 Content-Type 在伺服器解析 JSON 時的決定性作用？
  • [ ] 封裝正確性：JSON 字串拼接是否考慮了跳脫字元 (Escaping) 與逗號位置？
  • [ ] 韌性表現：請求失敗時，設備是否能繼續執行主線程任務，而非卡死在網路等待中？
