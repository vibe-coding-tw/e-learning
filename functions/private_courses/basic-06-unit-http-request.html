基礎 6 - 網路通訊基礎

數位世界的溝通橋樑：
HTTP 協議與 Fetch API

從餐廳點餐模型出發，掌握讓網頁、伺服器與物聯網設備互相「對話」的核心技術。

🍽️ 1. 餐廳模型：理解 HTTP 對話

HTTP (HyperText Transfer Protocol) 是網路上交換數據的標準格式。最直觀的理解方式就是「餐廳點餐」流程：

🙋‍♂️ 請求 (Request)

你向服務生點餐。包含你想看的頁面網址、你想做的動作（讀取或儲存）以及你提供的參數。

🍛 回應 (Response)

服務生送來餐點。包含狀態碼（成功或賣完）以及具體的內文（網頁內容、圖片或 JSON 數據）。

🧠 無狀態 (Stateless)

伺服器很健忘！每次連線都是獨立的，它不會主動記得你是誰，除非透過 Cookie 或 Token。

💡 Why Stateless?

無狀態性讓伺服器能同時服務成千上萬的請求，而不需要為每個連線保留龐大的記憶體背景，這正是網際網路能大規模擴展的關鍵。

✉️ 2. 請求組成：方法、標頭與正文

一封標準的 HTTP 請求信件包含三個核心部分：

GET
讀取菜單

向伺服器請求數據。參數通常直接掛載在網址後面 (Query Strings)。

POST
送交訂單

提交新數據給伺服器，資料封裝在 Body 中，更安全且不限長度。

Headers
信封標註

備註資訊。例如：告訴對方「我傳的是 JSON 格式」或「我的 Token 是 XXX」。

URL ANATOMY 
http://192.168.4.1/move?dir=forward&speed=200


// GET 請求參數 (Query Strings) 解析：
Host: 192.168.4.1 (ESP32)
Path: /move
Params: { dir: "forward", speed: 200 }

🏷️ 3. 狀態碼：伺服器的表情

200 OK

請求成功！資料已經準時送達你的手機或設備。

404 Not Found

找不到東西。可能是 API 網址打錯，或資源已被移除。

500 Server Error

伺服器崩潰。通常代碼邏輯寫爛了，导致處理請求時拋出異常。

實踐挑戰

「在瀏覽器網址列直接貼上指令網址，觀察序列監控視窗接收到的數據。這就是最原始的 Remote Control。」

HTTP 協議與數據交換實驗室

任務 1：「數位點餐員」：Postman GET 測試 (Digital Waiter: GET Lab)

角色扮演：API 測試工程師 | 難度：⭐️ | 估計時長：30 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 端點請求實作：使用 Postman 或瀏覽器發送 GET 請求。目標：解析 JSON 客戶端回應。
  3. 協議解構記錄：在 README 中彙整 Headers 中的 Content-Type、Status Code 與 Response Body 截圖。
  4. 理論分析：撰寫簡短分析，說明為什麼 GET 參數掛載在網址上具有安全隱憂。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 協議解構正確性：是否能清楚識別 HTTP 標頭與正文在該次通訊中的各別功能？
  • [ ] 數據擷取精確度：所擷取的 JSON 欄位是否與 API 原始定義完全吻合？

任務 2：「數據整合師」：實作 POST 發送 JSON (Data Integrator: POST/JSON)

角色扮演：後端數據整合工程師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 封裝 JSON 實作：在 ESP32 代碼中利用 ArduinoJson 或字串拼接，封裝模擬傳感器數據 Payload。
  2. 同步通訊實作：配置 HTTPClient 標頭 Content-Type: application/json 並執行 POST 動作。
  3. 端點接收驗證：在 Webhook.site 或測試 Server 端截圖，證明 JSON 被正確解析為結構化物件。
  4. 最終 Commit：提交具備數據封裝上報能力的穩定代碼。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] Payload 規格性：發送的 JSON 字串是否完全符合 RFC 規範，無格式溢位或非法字元？
  • [ ] Headers 宣告：是否有正確配置 Content-Type 標頭，否則伺服器端將無法解析正文。

任務 3：「通訊守門員」：狀態碼攔截與防錯 (Communication Sentry: Error Handling)

角色扮演：韌體穩定性工程師 | 難度：⭐️⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 異步狀態捕獲：實作 if (httpCode > 0) 的狀態碼分流邏輯。
  2. 錯誤攔截實驗：故意訪問錯誤路徑，並使設備能攔截 404 狀態碼並在 Serial 噴出對應報警。
  3. 自動自我修復：針對 500 以上或逾時 (Timeout) 的請求，實作「限制次數的自動重新嘗試 (Retry)」邏輯。
  4. 穩定性報告：在 README 中說明此邏輯如何避免設備因短暫網路不穩而永久佔用 CPU 資源。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 異常分流度：是否涵蓋了 4xx、5xx 與 Timeout 三大類核心網路異常處理？
  • [ ] 非阻塞性：在異常重試期間，代碼是否能優雅應對，不導致設備主程序完全卡死？
