🧠 什麼是 Vibe Coding？

Vibe Coding 描述了一種高層次的程式開發模式：開發者不再逐行編寫邏輯，而是透過自然語言與 AI 進行對話，定義應用的 「氛圍」(Vibe) —— 即系統的核心功能、使用者體驗與整體邏輯 —— 並由 AI 代理完成低層次的程式碼實現。

版本控制作為學習足跡

每一筆 Git Commit 都是你的思考軌跡。這能幫助導師區分哪些是「AI 生成」，哪些是你基於理解所做的「人類修正」。

Autograding 即時回饋

在快速迭代中，自動化評分系統提供了客觀的指標（測試案例），確保你的 Vibe 沒有偏離正確的邏輯軌道。

Vibe 時代的提交規範

在 AI 協助開發的時代，一次性提交（複製貼上）會導致學習過程不對稱。我們建議採用 「微提交 (Micro-commits)」 策略：

範例提交訊息：
• git commit -m "[AI-Generated] 實作 BLE 連線基礎邏輯"
• git commit -m "[Human-Modified] 修正 MTU 限制下的分段傳輸 Bug"
• git commit -m "[Review-Only] 完成單元測試驗證"

💡 每一筆 Commit 都是你的開發思考痕跡。

自動評分 (Autograding) 邏輯

系統會模擬真實的產品驗收，對你的程式碼進行功能測試。

✅

❌
測試失敗
點擊紅叉查看報錯，修正後再次 Push。

Prompt 審查作業

除了提交代碼，你也需要記錄與 AI 的對話過程。我們關注的是你如何精確描述需求，以及如何糾正 AI 的「幻覺」。這才是 Vibe Coding 真正的基本功。

規格說明書 (Spec) 的定義權

訓練撰寫高品質的 Markdown 文檔，定義介面、資料結構與流程。在 2026 年，定義問題的能力遠比寫出語法更重要。

GitHub Classroom 與 Vibe Coding 實務

【任務 1-1】定義你的「Vibe」 (Defining the Vibe)

角色扮演：產品經理 (PM) | 難度：⭐️⭐️ | 估計時長：30 min

🛠️ 執行技術流程 (Git Workflow)

  1. 接受挑戰：點擊 GitHub Classroom 連結，獲取本單元的實作倉庫。
  2. 撰寫 Vibe Prompt：在 VIBE.md 中，針對一個「智慧家庭控制面板」撰寫高質量的描述語：
    • 包含視覺風格 (e.g. Glassmorphism, Dark Mode)
    • 定義核心功能與佈局 (Joystick for lights, status bar)
  3. 提交設計：執行 git add 並 push，讓 AI 或導師審視你的「定義力」。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 描述精確度：是否包含具體的色彩計畫 (Color Palette) 與組件佈局？
  • [ ] 層次化邏輯：Prompt 是否按照「背景 -> 功能 -> 細節」的層級撰寫？

【任務 1-2】糾正 AI 的「幻覺」 (Hallucination Audit)

角色扮演：系統架構師 | 難度：⭐️⭐️⭐️ | 估計時長：40 min

🛠️ 執行技術流程 (Git Workflow)

  1. 檢視代碼：閱讀 buggy_code.js，這是一份由 AI 生成但包含明顯邏輯錯誤的程式碼。
  2. 修復與紀錄：不要只手動修改，請撰寫一段「修正 Prompt」來引導 AI 修復它。
  3. 提交紀錄：在 Commit Message 中註明：[Human-Corrected] Fix logical error in array processing。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 問題識別：是否正確找出 AI 生成代碼中的邊界案例 (Edge Case) 錯誤？
  • [ ] 引導效率：修正 Prompt 是否簡潔且能讓 AI 產出正確結果？

【任務 1-3】建立你的第一個「Vibe 願景」

角色扮演：全端開發者 | 難度：⭐️⭐️ | 估計時長：20 min

🛠️ 執行技術流程 (Git Workflow)

  1. 生成原型：使用 AI 快速生成一個包含「毛玻璃效果 (Glassmorphism)」的時鐘組件。
  2. 微粒子提交：將生成的 HTML/CSS 拆分為 3 個微小的 Commit 同步。
  3. 成果展示：確保 index.html 可以在 GitHub Pages 正常預覽。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 視覺達成度：是否展現出 Vibe 中的質感細節？
  • [ ] Git 軌跡：是否實踐了「微提交」策略而非單一巨型提交？

【任務 2-1】接受邀請與首次推送 (The Lifecycle Loop)

角色扮演：實習開發者 | 難度：⭐️ | 估計時長：15 min

🛠️ 執行技術流程 (Git Workflow)

  1. 接受挑戰：點擊老師提供的 GitHub Classroom 邀請連結。
  2. Clone 倉庫：將建立好的倉庫 git clone 到你的開發環境。
  3. 驗證推送：修改 README.md，增加一行「我已完成工作流設定」，並執行 git push。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 連線成功：GitHub Actions 是否成功觸發？
  • [ ] 權限驗證：是否能正確在 Classroom Dashboard 看到學員的名單？

【任務 2-2】微提交實踐 (Micro-Commit Mastery)

角色扮演：資深工程師 (Senior) | 難度：⭐️⭐️ | 估計時長：30 min

🛠️ 執行技術流程 (Git Workflow)

  1. 邏輯實作：在一個小型專案中，分別完成「骨架定義」與「內容填充」。
  2. 提交規範：為這兩個動作分別建立 Commit：
    • feat: Define structural components
    • docs: Add descriptive content
  3. 同步提交：以此展現你的開發並非大量複製貼上，而是有層次的構建過程。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] Commit 質量：提交訊息是否清晰描述了變動內容？
  • [ ] 細粒度控管：是否成功將工作拆分為獨立的邏輯單元？

【任務 2-3】紅燈轉綠燈 (Autograding Fix)

角色扮演：測試工程師 (QA) | 難度：⭐️⭐️⭐️ | 估計時長：40 min

🛠️ 執行技術流程 (Git Workflow)

  1. 故意失敗：推送一段不符合 assignment_test.js 規範的代碼（故意讓測試失敗）。
  2. 查閱日誌：在 GitHub 介面的 "Actions" 分頁中找出報錯原因。
  3. 修正與通過：根據錯誤訊息修正代碼再次推送，直到看到 ✅ 指標。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 修復紀錄：是否有從失敗到成功的 Git Commit 紀錄？
  • [ ] 測試覆蓋：最終是否通過了所有的自動評分測試案例？

【任務 3-1】規格說明書撰寫實踐 (Spec-First)

角色扮演：技術負責人 (Tech Lead) | 難度：⭐️⭐️ | 估計時長：30 min

🛠️ 執行技術流程 (Git Workflow)

  1. 定義需求：針對一個「物聯網氣象站」，撰寫 SPEC.md。
  2. 結構化描述：包含：
    • API 接口協議 (JSON 格式定義)
    • 硬體腳位配置 (Pinout Mapping)
    • 驗收標準 (Acceptance Criteria)
  3. 提交規格：讓 AI 根據這份 Spec 產生第一版代碼，並 commit 你的原始規格。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 嚴謹性：Spec 是否足以讓另一個開發者（或 AI）在不詢問的情況下完成開發？
  • [ ] 維護性：是否正確使用了 Markdown 進行結構化排版？

【任務 3-2】安全性與漏洞審計 (Security Audit)

角色扮演：安全專家 (SecOps) | 難度：⭐️⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 掃描弱點：檢視 server_logic.js，找出 3 個潛在的安全風險（例如：Hardcoded Key, No Authentication, SQL Injection）。
  2. 修復建議：在 AUDIT.md 中記錄這些風險，並提交修復後的代碼。
  3. 同步分析：推送你的審計報告與修補程式。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 風險識別：是否能找出非顯而易見的邏輯漏洞？
  • [ ] 修補正確性：修補後的程式碼是否在修正漏洞的同時，保持原有功能正常？

【任務 3-3】自動化工作流提案 (CI/CD Concept)

角色扮演：運維工程師 (DevOps) | 難度：⭐️⭐️ | 估計時長：20 min

🛠️ 執行技術流程 (Git Workflow)

  1. 流程設計：構思一個自動化腳本，當學生提交作業時，自動檢查 `README.md` 是否包含必要的手機截圖。
  2. 撰寫提案：在 WORKFLOW.md 中用文字或偽代碼描述這個 GitHub Actions 的邏輯。
  3. 提交提案：將你的設計思維同步到 GitHub。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 可行性：提案的邏輯是否在 GitHub Actions 環境下可被實踐？
  • [ ] 效率提升：該工作流是否能顯著減少導師的人工檢查負擔？

GitHub Classroom 教學指引

本單元是 GitHub Classroom、Vibe Coding Prompt 與教師評核觀點的整合入口。授課時請先確認學生已完成邀請接受、倉庫 clone 與首次 push，再把重點放在 prompt 品質、提交節奏與 autograding 除錯，而不是只檢查最後結果。

建議教師把九個任務拆成三段節奏：先練習「定義與修正 AI」，再驗證「Classroom 工作流」，最後提升到「規格、安全與自動化提案」。如此才能讓學生理解 GitHub Classroom 不是單純交作業平台，而是支撐 AI 時代工程素養的學習基礎設施。
