簡介

在 AI 工具尚未普及的年代，評斷一位開發者的標準是：「你會寫幾種語言？」 而到了 2026 年，這個問題已經失去意義——幾乎所有人都能透過 AI 在數秒內生成可運行的程式碼。

情境
小明和小華同時拿到了同一份 GitHub Classroom 作業。兩人都用了 AI 生成大部分的程式碼， Autograding 也都通過了。但導師翻開 commit 紀錄，發現小明只有一筆「上傳全部」， 小華則有 8 筆，每一筆都記錄了「為什麼這樣修」。 你覺得，哪一位學員真正學到了東西？

GitHub Classroom 在這個脈絡下，不只是繳交作業的管道，而是你留下開發思考軌跡的空間。 每一筆 commit，都是你與 AI 協作的過程紀錄；每一次 Autograding，都是系統對你思考的即時驗證。

GitHub Classroom：AI 時代的工程化學習空間

GitHub Classroom 是 GitHub 為教育場景打造的作業管理工具。 老師透過它建立作業範本（Template Repository），每位學生接受邀請後，系統會自動複製出一個屬於自己的私人 Repository， 並在每次 Push 時觸發自動評分（Autograding）。

為什麼是「工程化學習空間」？

版本控制即學習足跡

每一筆 Git Commit 都記錄了你的思考過程。導師可以區分「AI 生成」與「人類理解後修正」的部分，讓學習真正被看見。

自動評分即即時回饋

Autograding 提供客觀的測試指標，在快速迭代中確保你的程式碼沒有偏離正確的邏輯軌道，不需要等待人工批改。

Classroom 的角色分工

角色在 Classroom 中的工作使用的功能
老師 / 導師 建立作業、設定 Autograding、監控進度、Code Review GitHub Classroom Dashboard、Actions YAML
學員 接受邀請、Clone Repository、實作、Push、查看測試結果 GitHub Repository、Git CLI 或 VS Code
系統（GitHub Actions） 每次 Push 後自動執行測試腳本，回傳 ✅ 或 ❌ CI/CD Pipeline、Autograding Workflow
2026 年的定位 GitHub Classroom 已從單純的「作業繳交系統」演進為「工程化學習空間」—— 它連接了教育者與業界標準工具（Git、CI/CD、Code Review）， 讓你在學習過程中就累積真實的工程素養。

學員操作：5 步驟完成作業

從收到作業連結到看見 ✅，整個流程可以拆解成 5 個清晰的步驟。 熟悉這個循環，你就能將注意力從「怎麼操作」移到「怎麼思考」。

1
接受邀請
點擊作業連結，授權 GitHub Classroom
2
Clone Repository
將作業下載到本機或 Codespaces
3
開始實作
透過 Vibe 引導 AI，微提交記錄過程
4
Push 同步
上傳程式碼，觸發自動化測試
5
確認結果
查看 ✅ / ❌，修正後再 Push

步驟詳解

1

接受邀請

導師會透過 LINE 群組或平台傳送一個專屬連結。點擊後登入 GitHub，授權 Classroom 存取你的帳號。系統會自動複製一個範本，建立屬於你的私人作業 Repository（格式：作業名稱-你的GitHub帳號）。

2

Clone Repository

在 Repository 頁面點擊 Code ▾，複製 HTTPS 網址，在終端機執行 git clone 網址；或直接用 . 在 github.dev 開啟，或啟動 Codespaces。

3

開始實作

依照作業說明（README）開始撰寫程式碼。使用 AI 協助時，請記得採用微提交策略（下一個單元詳述），留下你的思考脈絡。

4

Push 同步

執行 git push 將程式碼上傳。Push 成功後，GitHub Actions 會自動觸發 Autograding 工作流，執行測試腳本。

5

確認結果

前往 Repository 頁面，點擊 Actions 分頁，查看測試結果。✅ 代表通過；❌ 代表失敗——點擊紅色的 ❌ 查看詳細錯誤報告，修正後再次 Push。

截止日期 GitHub Classroom 支援設定作業截止日期。截止後仍可 Push，但導師可以看到是否在期限內完成。 養成提前 Push 的習慣，確保最後一刻的網路問題不會影響成績。

Vibe 時代的提交規範：微提交策略

當 AI 能在幾秒內生成數百行程式碼，「一次上傳全部」成為最危險的壞習慣。 它不只讓導師看不出你的學習過程，也讓你自己失去了理解與除錯的機會。 微提交（Micro-commits）是解方。

什麼是微提交？

微提交指的是：每完成一個有意義的小步驟，就立刻 commit 一次。 不等到整個功能完成，而是把開發過程切割成一系列可追溯的快照。

AI 時代的黃金法則 在 Vibe Coding 課程中，我們要求每一筆 commit 都標示它的「來源」： 是 AI 生成的？是你理解後修正的？還是你驗證完成的？ 這是建立可信開發足跡的核心紀律。

Commit 訊息格式

我們採用以下標籤規範，幫助導師（和未來的你）快速理解每一筆 commit 的性質：

commit message 範例
# AI 生成的程式碼（直接使用）
[AI-Generated] 實作 BLE 連線基礎邏輯
# 你理解後手動修正的部分
[Human-Modified] 修正 MTU 限制下的分段傳輸 Bug
# 通過測試驗證後的里程碑
[Review-Only] 完成單元測試驗證，所有測試通過
# 修正 AI 的錯誤（最有價值的 commit！）
[AI-Fixed] 糾正 AI 生成的邊界條件錯誤：陣列超界問題

為什麼這樣做？

做法導師看到的你得到的
一次上傳全部 無法判斷過程，難以評估理解程度 代碼可能跑通，但不知道為什麼
微提交 + 標籤 清楚看見你的思考脈絡與 AI 協作比例 強迫自己理解每個步驟，真正學會
頻率建議 每 10–20 分鐘，或每完成一個可獨立運行的功能單元，就 commit 一次。 寧可 commit 太多次，不要太少。Git 是可以回滾的，但遺失的思考紀錄無法重建。

自動評分（Autograding）：從紅燈到綠燈

每次你 Push 程式碼，GitHub Classroom 就會在背景自動執行一組測試腳本。 這個機制叫做 Autograding，背後由 GitHub Actions 驅動。 它模擬的是真實的產品驗收流程，讓你在學習階段就習慣「以測試為準」的工程思維。

Autograding 的運作方式

1

你執行 git push

程式碼上傳至 GitHub Repository。

2

GitHub Actions 自動觸發

系統依照導師設定的 YAML 腳本，啟動一個乾淨的 Linux 環境，安裝相依套件並執行測試。

3

測試結果回傳

Repository 首頁的 commit 旁邊會出現 ✅（通過）或 ❌（失敗）。點擊圖示可進入 Actions 查看詳細日誌。

解讀測試結果

✅
測試通過

你的程式碼符合作業規格定義的所有條件，邏輯正確。可以繼續下一個功能，或加入額外優化。

❌
測試失敗

至少有一個測試案例未通過。點擊 ❌ 進入 Actions → 展開失敗的步驟 → 閱讀錯誤訊息 → 修正 → 再次 Push。

如何閱讀錯誤日誌？

GitHub Actions 錯誤日誌範例
Run autograding tests...
✗ test_wifi_connect: Expected True, got False
  AssertionError: WiFi 連線函式應回傳布林值 True
  at line 42 in wifi_module.py
Tests run: 5 | Passed: 4 | Failed: 1

從這個日誌可以讀出：

  • 失敗的測試名稱：test_wifi_connect
  • 預期值 vs 實際值：預期 True，實際得到 False
  • 問題位置：wifi_module.py 第 42 行
測試失敗不是失敗 在工程實務中，測試失敗是開發流程的一部分，不是終點。 重要的是你能讀懂錯誤訊息，找到問題，並修正它。 這個「紅燈 → 修正 → 綠燈」的循環，正是 TDD（測試驅動開發）的核心精神。

核心策略：掌握定義問題的能力

在 AI 工具普及的今天，「寫程式」的門檻已大幅降低， 但「定義清楚問題」的能力卻越來越稀缺且值錢。 本單元介紹兩個核心策略：Prompt 審查與規格說明書（Spec）撰寫。

策略一：Prompt 審查與紀錄

在 Vibe Coding 課程中，我們評量的不只是你的程式碼，也包含你如何與 AI 對話的過程。 精確描述需求、識別並糾正 AI 的「幻覺（Hallucination）」，才是真正的核心基本功。

❌ 低品質 Prompt
「幫我寫一個 WiFi 連線的程式」

AI 會自行假設許多細節，結果往往需要大量修改。

✅ 高品質 Prompt
「使用 ESP32 Arduino 框架，
實作 WiFi Station 模式的連線函式。
輸入：SSID 字串、密碼字串。
輸出：布林值（連線成功/失敗）。
需包含 10 秒超時機制。」

明確的規格讓 AI 第一次就能生成接近正確的程式碼。

作業要求 部分作業要求你在 commit 中附上關鍵的 Prompt 紀錄，或在 README 中說明你如何糾正 AI 的錯誤。 這些記錄是評估你「Prompt 工程能力」的重要依據。

策略二：規格說明書（Spec）的定義權

在 2026 年，定義問題的能力遠比寫語法更重要。 撰寫一份清晰的規格說明書，能讓 AI、隊友，甚至三個月後的自己，都能快速理解你的設計意圖。

規格說明書範例（Markdown 格式）
## WiFi 連線模組規格
### 功能描述
負責管理 ESP32 的 WiFi Station 模式連線生命週期。
### 介面定義
```cpp
bool connectWiFi(const char* ssid, const char* password, int timeout_ms);
```
### 驗收標準
- [ ] 連線成功時回傳 true
- [ ] 超過 timeout_ms 未連線時回傳 false
- [ ] 連線期間 Serial 輸出連線進度（每秒一個 `.`）
Spec 的三個核心要素 一份好的規格說明書包含：
① 功能描述：這個模組做什麼？
② 介面定義：輸入什麼、輸出什麼？
③ 驗收標準：怎樣算做好了？

03-unit-github-classroom：GitHub Classroom 實戰工作流

歡迎來到 GitHub Classroom 實戰單元！本作業旨在讓你親自動手體驗 GitHub Classroom 的核心機制，從啟動雲端 IDE、觸發自動評分，到在 Feedback Pull Request 中進行 Code Review，建立真實的 Vibe Coding 工作流直覺。

🚀 快速開始

  1. 接受作業邀請：點擊老師提供的 GitHub Classroom 邀請連結，系統會自動為你建立專屬 Repository。
  2. 啟動 Codespaces：進入你的 Repo 頁面，點擊 Code → Codespaces → Create codespace on main，在瀏覽器中開啟雲端 IDE。
  3. 確認環境：在 Codespaces Terminal 執行以下指令，確認 Node.js 環境正常： 
    node --version
npm install
npm test

    此時測試預期是「失敗」的（紅燈），這是正常現象。

📋 任務說明

本作業共三個任務，請依序完成：

☁️ 任務 3-1：啟動 Codespaces 雲端環境（20 分鐘）

目標：體驗「一鍵啟動雲端開發環境」的 Vibe Coding 基礎設施。

步驟：

  1. 點擊老師提供的 GitHub Classroom 邀請連結，建立你的專屬 Repository。
  2. 進入 Repo，點擊 Code → Codespaces → Create codespace on main。
  3. 等待 Codespace 初始化完成（約 1-2 分鐘）。
  4. 在 Codespaces 的 Terminal 中執行 node --version 與 npm install，確認環境正常。

提交內容：

  • 你的 Codespace 畫面截圖（顯示 Repo 名稱與 Terminal 畫面）。
  • 你的 Repo 網址。

🚦 任務 3-2：原子化提交與自動評分（30 分鐘）

目標：體驗「Push → 自動評分 → 紅燈 → 修正 → 綠燈」的完整 Autograding 循環。

步驟：

  1. 在 Codespaces 中查看 src/buggy_code.js，找出並修正程式中的 Bug。
  2. 使用原子化提交 (Atomic Commits)：每修正一個邏輯點，就執行一次獨立 commit： 
    git add src/buggy_code.js
git commit -m "fix: correct array sorting logic"
git push
  3. 前往你的 Repo → Actions 分頁，觀察 Autograding 工作流的執行結果。
  4. 重複「修正 → Push → 查看 Actions」流程，直到 Autograding 出現 ✅ 綠燈。

提交內容：

  • GitHub Actions 頁面的截圖，顯示 Autograding 測試由 ❌ 轉為 ✅ 的過程。
  • 至少包含 2 個以上獨立 commit 的 Git 歷史截圖（git log --oneline）。

💬 任務 3-3：Feedback PR 與 Code Review（20 分鐘）

目標：練習使用 GitHub 的 Pull Request 功能進行程式碼審查（Code Review）。

背景：GitHub Classroom 會在你的 Repo 中自動建立一個名為 Feedback 的 Pull Request，這是老師與你的程式碼對話的主要管道。

步驟：

  1. 進入你的 Repo → Pull requests 分頁，找到 Feedback PR。
  2. 切換到 Files changed 頁籤，瀏覽程式碼差異。
  3. 對程式碼的至少一行進行 Inline Comment（點擊行號旁的 + 按鈕新增評論），說明：
    • 這段程式碼的作用是什麼？
    • 你認為有沒有更好的寫法？
  4. 提交你的 Review（選擇 Comment 即可，不需要 Approve 或 Request changes）。

提交內容：

  • 你在 Feedback PR 中留下的 Inline Comment 連結（複製 Comment 的永久連結）。

📤 提交方式

完成以上三個任務後，在 Vibe Coding 平台的本作業頁面點擊「提交作業」，並填寫：

  1. 你的 GitHub Repository 網址。
  2. 三個任務的截圖或連結（依各任務要求）。

導師指南 (Tutor Guide) - 03-unit-github-classroom

單元簡介

本單元 03-unit-github-classroom（GitHub Classroom 實戰工作流） 是課程中第一個讓學生「真正動手操作 GitHub Classroom 完整流程」的單元。透過三個由淺至深的任務，學生將建立「雲端 IDE → Commit → 自動評分 → Code Review」的核心工作習慣，為後續所有實作課程打下基礎。

作業總覽：三個核心任務

任務 重點技能 預計時間 自動評分
3-1 啟動 Codespaces 雲端環境、Classroom 邀請流程 20 分鐘 無（人工審核截圖）
3-2 原子化提交與自動評分 Atomic Commits、GitHub Actions、Autograding 30 分鐘 有（tests/lab.test.js）
3-3 Feedback PR 與 Code Review Pull Request、Inline Comment、程式審查 20 分鐘 無（人工審核 PR 連結）

導師評核重點與技巧

☁️ 任務 3-1：啟動 Codespaces 雲端環境

常見卡點：

  • 學生點擊邀請連結後找不到「Create codespace」按鈕 → 確認學生已登入 GitHub 且接受組織邀請（前往 GitHub Settings / Organizations 接受）。
  • Codespace 初始化失敗 → 建議直接重新建立（Delete → New），通常一次就成功。

評核標準：

  • 截圖需清楚顯示：① Repo 名稱（含學生 GitHub ID）② Codespaces Terminal 已成功載入（能看到提示符）。
  • 若截圖只有 Repo 首頁而無 Codespaces 畫面，請退回要求補充。

導師提示（L1-L3）：

  • L1：「你在 Repo 頁面看到 Code 按鈕了嗎？點擊後有 Codespaces 的 Tab 嗎？」
  • L2：「若找不到 Codespaces Tab，請確認你是否已接受組織邀請。可以到 GitHub 右上角的鈴鐺確認是否有待處理通知。」
  • L3：「請前往 https://github.com/settings/organizations，確認 vibe-coding-classroom 組織邀請狀態。」

🚦 任務 3-2：原子化提交與自動評分

部署說明：

  • 範本程式庫中的 src/buggy_code.js 包含 刻意設計的 Bug（例如：陣列排序邏輯錯誤、邊界條件錯誤）。
  • tests/lab.test.js 預設會因 Bug 而失敗（紅燈），學生必須修正後才能過關。

評核重點：

  1. Autograding 過程：Actions 歷史中必須有至少一次失敗記錄，才能證明學生真正理解了「紅燈→修正→綠燈」的循環。若第一次就成功，請詢問是否直接查看了答案。
  2. Atomic Commits：查看 Commit 歷史，確認有 2 個以上獨立 Commit，且每個 Commit 訊息有意義（如 fix: correct boundary check）。單一大型 commit 視為未達標。
  3. 評分同步：Autograding 分數會自動回寫到 Vibe Coding Dashboard。

常見錯誤：

  • 學生直接 git commit -m "fix all" 把所有修改放在一個 Commit → 提示：「請把每個修正分成獨立的 Commit，想像每個 Commit 是一個思考步驟。」
  • 學生看不懂 Actions 的錯誤訊息 → 指引他在 Actions 頁面點開 Autograding Job，展開失敗的 Step，閱讀具體的測試錯誤。

導師提示（L1-L3）：

  • L1：「你有看到 Actions 裡 Autograding 的結果嗎？點進去展開看看是哪個測試失敗了。」
  • L2：「測試錯誤訊息顯示的是 Expected: X, Received: Y，表示你的函數回傳值不符合預期。請對照 tests/lab.test.js 的測試案例，逆推應該修改什麼。」
  • L3：「在 src/buggy_code.js 第 12 行，sort() 預設會按字串排序而非數字。你需要傳入比較函數：arr.sort((a, b) => a - b)。」

💬 任務 3-3：Feedback PR 與 Code Review

背景：

  • GitHub Classroom 會在學生接受作業後自動建立 Feedback Pull Request（從 feedback 分支合入 main）。這個 PR 是導師與學生進行程式碼對話的主要管道。

評核重點：

  1. Comment 是否有實質內容（不是只寫「我看了」）：理想的評論應說明「這段程式碼做什麼」或「可以怎麼改進」。
  2. 若學生沒有看到 Feedback PR → 引導他前往 Repo → Pull requests → 確認是否有 Feedback 標題的 PR。若無，可能是 Classroom 設定問題，請聯繫管理員。

導師後續動作：

  • 收到學生提交後，在 Feedback PR 中針對學生的最重要 Commit留下導師自己的 Review Comment，肯定做得好的地方，並對可以改進的地方提供具體建議（而非直接給答案）。
  • 對於 Commit 訊息品質差的學生（如 fix bug、update），在 PR Comment 中鼓勵他們參考 Conventional Commits 規範。

導師提示（L1-L3）：

  • L1：「你有找到 Repo 裡的 Pull requests 分頁嗎？裡面應該有一個叫 Feedback 的 PR。」
  • L2：「進入 Feedback PR 後，切到 Files changed 頁籤，把滑鼠移到任何一行程式碼上，會出現 + 按鈕，點它就能新增 Inline Comment。」
  • L3：「你可以在 Comment 中寫：『這段 sort((a, b) => a - b) 的作用是強制讓陣列按數值大小排序，因為 JS 預設的 sort 是字串排序，對數字會產生錯誤結果。』這樣的說明就很有價值！」

整體評分建議

  • 3-1（30分）：截圖清楚 → 30分；截圖不夠清楚或缺少必要資訊 → 15-20分。
  • 3-2（40分）：Autograding ✅ 且有 2+ atomic commits → 40分；✅ 但只有 1 個 commit → 25分；❌ 仍未通過 → 引導修正後補交。
  • 3-3（30分）：有實質性 Inline Comment → 30分；Comment 過於敷衍（少於 10 字）→ 15分；無 Comment → 0分。

重要提醒

本單元的核心不在於「程式寫得多難」，而在於讓學生建立 Git Workflow 的肌肉記憶。導師在評核時，應特別關注學生的 Commit 節奏與訊息品質，這是區分「有在認真實踐 Vibe Coding」與「只是交差了事」的關鍵指標。
