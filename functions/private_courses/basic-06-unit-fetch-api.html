從等待到非同步：
Fetch API 與流暢體驗

學習如何發送一個「背景請求」，讓網頁在不重新整理的情況下，優雅地與遠端伺服器交換數據。

⏳ 1. 從等待到非同步：Fetch 的進化

在過去，處理 AJAX 需要冗長的語句。Fetch API 的出現改寫了這一切，它基於 Promise，讓網頁通訊變得直覺且簡潔。

Why Fetch?

  • Promise 驅動：告別回呼地獄，邏輯一條線。
  • 語義化介面：函式名稱即是動作（fetch = 去拿）。
  • 原生支援：無需引入外部套件即可處理數據流。

Promise Lifecycle

Pending

請求發送中（等待回應）。

Fulfilled

成功獲取數據（走 .then / await）。

Rejected

網路斷線或 DNS 失敗（走 .catch）。

✨ 2. Async / Await：寫出優雅的同步感

雖然 .then() 已經很好，但當邏輯複雜時，async/await 能讓非同步代碼看起來像同步一樣順暢。

Step 01

等待 Header

伺服器回傳狀態碼 (如 200 OK)。此時 Body 尚未載入。

Step 02

解析 Body

轉換二進位數據流為 JSON 對象。這也是非同步過程。

CLEAN CODE 
async function fetchData() {
  // 1. 送出請求並等待回應標頭
  const res = await fetch(url);
  
  // 2. 等待數據解析完畢
  const data = await res.json();
  console.log(data);
}

🛡️ 3. 強韌通訊：攔截與逾時

現實中的網路是脆弱的。優秀的工程師不會只考慮「成功」，而會優先處理「異常」。

容錯三箭

  • res.ok 檢查：攔截 404 或 500 等邏輯錯誤。fetch 本身不會捕捉這些錯誤。
  • try/catch 捕捉：處理物理斷網、DNS 解析失敗。
  • AbortController：設定逾時保護（例如 5s），避免請求無限期待命。

Network Sentry

GitHub Classroom 詳細操作指南

本單元教學重點：掌握非同步編程的核心心法與異步 UI 的處理技巧。

  1. 重點 1：兩次 await 的本質 - 解釋為什麼 fetch 返回 Response 後還需要第二次 await 解析 Body 位元流。
  2. 重點 2：不阻塞主執行緒 - 引導學員觀察在 fetch 期間頁面按鈕是否仍可點擊（非同步的魅力）。
  3. 重點 3：Fetch 的「過於強壯」特性 - 強調 fetch 不會因為 404/500 而進 catch 的陷阱，必須手動檢查 res.ok。
