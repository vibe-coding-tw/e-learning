簡介

Fetch API 是現代 JavaScript 發送網路請求的標準工具，基於 Promise 的非同步設計讓程式碼告別回呼地獄。 本單元將帶你掌握 async/await 語法、JSON 解析流程，以及如何用 AbortController 實作逾時保護， 讓你的控制面板在網路不穩定時也能優雅降級，而非靜默卡死。

Fetch 的進化

在早期的 Web 開發中，要從伺服器取得資料，開發者必須使用 XMLHttpRequest（XHR）—— 一套笨重、語法繁瑣且容易掉入「回呼地獄（Callback Hell）」的 API。 Fetch API 是現代瀏覽器提供的全新網路請求介面，基於 Promise 設計， 讓非同步通訊的程式碼變得直覺、簡潔且可讀性極高。

從 XMLHttpRequest 到 Fetch

以下比較展示了同一個功能（從 API 取得 JSON 資料）在兩種方式下的寫法差異：

比較項目 XMLHttpRequest（舊） Fetch API（新）
語法風格 事件監聽 + 回呼函式，巢狀層層包裹 Promise 鏈 / async-await，線性易讀
錯誤處理 需手動檢查 readyState 與 status 統一用 .catch() 或 try/catch
串流支援 有限，需自行處理 原生支援 ReadableStream
取消請求 xhr.abort() AbortController 搭配 signal
瀏覽器支援 所有瀏覽器（含 IE） 所有現代瀏覽器（IE 除外）

Promise 的三種狀態

Fetch API 回傳的是一個 Promise 物件。Promise 代表一個「尚未完成但最終會有結果」的非同步操作， 它在整個生命週期中會處於以下三種狀態之一：

  • Pending（等待中）：請求已發出，但尚未收到回應。此時 UI 應顯示 Loading 狀態。
  • Fulfilled（已實現）：請求成功完成，回傳了 Response 物件。可以繼續解析 JSON 或文字內容。
  • Rejected（已拒絕）：請求因網路層級錯誤（如斷網、DNS 解析失敗）而失敗。進入 .catch() 處理。
重要 fetch() 的 Promise 只有在網路層級錯誤時才會被 reject（進入 catch）。 即使伺服器回傳 404 或 500 錯誤，Promise 仍然會 resolve（進入 then）—— 你必須自己檢查 response.ok 或 response.status 來判斷請求是否真的成功。

最基本的 Fetch 範例

以下是使用 fetch() 發送 GET 請求的最簡寫法：

// 最基本的 fetch 用法
fetch("http://192.168.4.1/sensor")
  .then(res => res.json())   // 解析 JSON
  .then(data => {
    console.log(data.temperature); // 25.3
  })
  .catch(err => {
    console.error("網路錯誤：", err);
  });
提示 雖然 .then().catch() 寫法可以運作，但在現代開發中，更推薦使用 async/await 語法—— 它讓非同步程式碼讀起來像同步程式碼一樣直覺。下一節將詳細介紹這種更優雅的寫法。

Async / Await：優雅的非同步語法

async/await 是 ES2017 引入的語法糖，建立在 Promise 之上，讓非同步程式碼讀起來就像同步程式碼一樣流暢。 在物聯網 Web 控制台的開發中，你會大量使用它來處理與 ESP32 的通訊——從發送請求、等待回應、到解析數據， 每一步都清晰明瞭，不再有層層巢狀的回呼函式。

Fetch 的兩階段等待

使用 fetch() 取得資料其實需要經過兩次 await， 這是初學者最容易忽略的重點：

階段 程式碼 說明
第一次 await await fetch(url) 等待伺服器回傳回應標頭（Headers）。此時你可以檢查 res.ok、res.status，但 Body 還沒有完全下載。
第二次 await await res.json() 等待回應正文（Body）完整下載並解析為 JavaScript 物件。這是一個耗時操作，因此也是非同步的。
注意 為什麼需要兩次 await？因為 HTTP 回應是串流（Stream）傳輸的——伺服器先傳送狀態碼與標頭， 然後才逐步傳送 Body 內容。第一次 await 在收到標頭時就完成了， 但 Body 可能還在傳輸中，所以需要第二次 await 來等待完整的資料。

完整的 async/await 範例

以下是使用 async/await 搭配 fetch() 的標準寫法：

async function loadSensorData() {
  const res = await fetch("http://192.168.4.1/sensor");

  if (!res.ok) {
    throw new Error(`HTTP 錯誤：${res.status}`);
  }

  const data = await res.json();
  document.getElementById("temp")
    .textContent = data.temperature + " °C";
}

POST 請求的 async/await 寫法

當你需要向 ESP32 發送控制指令（如前進、後退）時，需要使用 POST 請求並傳送 JSON Body：

async function sendCommand(direction, speed) {
  const res = await fetch("http://192.168.4.1/move", {
    method: "POST",
    headers: {
      "Content-Type": "application/json"
    },
    body: JSON.stringify({ direction, speed })
  });

  const result = await res.json();
  return result;
}
警告 使用 async/await 時，一定要用 try/catch 包裹你的 fetch 呼叫。 沒有 try/catch 的 await 在遇到網路錯誤時會直接拋出未捕獲的例外， 可能導致整個應用程式的後續邏輯被中斷。
提示 async 函式永遠回傳一個 Promise。這意味著你可以在外部使用 .then() 接續處理， 也可以在另一個 async 函式中用 await 等待它完成。兩種風格可以自由混用。

攔截與逾時保護

在物聯網場景中，網路環境遠比一般 Web 開發更加不穩定——ESP32 可能斷電、Wi-Fi 訊號弱、或者伺服器暫時過載。 一個專業的 Web 控制台必須能優雅地處理所有異常情況，而不是讓使用者面對一個永遠轉圈的 Loading 圖示。 這一節將教你建立一套完整的容錯機制。

三層防禦策略

完善的 Fetch 錯誤處理應包含以下三層：

防禦層 處理的問題 實作方式
res.ok 檢查 伺服器回傳了錯誤狀態碼（404、500 等），但 fetch 仍然 resolve if (!res.ok) throw new Error()
try/catch 捕獲 網路斷線、DNS 失敗等導致 fetch 被 reject try { ... } catch (err) { ... }
AbortController ESP32 無回應，請求無限等待（無 timeout） 搭配 setTimeout 主動取消
重要 fetch() 本身沒有內建的逾時（timeout）機制。如果 ESP32 斷電了， fetch() 會一直等待直到瀏覽器自行放棄（通常需要數分鐘），期間使用者看到的就是一個「凍結」的介面。 你必須使用 AbortController 自行實作逾時保護。

AbortController 逾時保護實作

AbortController 是瀏覽器原生的請求取消機制。搭配 setTimeout， 你可以在指定時間內自動取消尚未完成的 fetch 請求：

async function robustFetch(url, timeout = 5000) {
  const controller = new AbortController();
  const timer = setTimeout(() =>
    controller.abort(), timeout
  );

  try {
    const res = await fetch(url, {
      signal: controller.signal
    });

    if (!res.ok) throw new Error(`${res.status}`);
    return await res.json();

  } catch (err) {
    if (err.name === "AbortError") {
      throw new Error("請求逾時，ESP32 無回應");
    }
    throw new Error("網路連線失敗");
  } finally {
    clearTimeout(timer);
  }
}

在 UI 中整合錯誤提示

好的錯誤處理不只是在 Console 印出訊息，而是要讓使用者在畫面上看到明確的回饋。 以下是將 robustFetch 整合到 UI 的完整範例：

async function refreshDashboard() {
  statusEl.textContent = "載入中...";

  try {
    const data = await robustFetch("/sensor");
    renderData(data);
    statusEl.textContent = "更新完成";
  } catch (err) {
    statusEl.textContent = err.message;
    statusEl.style.color = "red";
  }
}
提示 建議將 robustFetch 封裝為專案中的通用工具函式，讓所有 API 呼叫都自動具備逾時保護與錯誤分類功能。 這種「一次封裝、處處受益」的模式，是專業前端工程的最佳實踐。
注意 finally 區塊非常適合用來關閉 Loading 狀態。無論請求成功或失敗， finally 都一定會執行，確保 Loading 轉圈不會永遠卡在畫面上。

3. 非同步通訊與體驗優化點評標準 (Asynchronous UX & Robustness Standards)

此單元的價值在於 「對通訊生命週期的全面掌控與對異常狀態的優雅處理」：

  • [ ] Body 解析深度：是否理解 await res.json() 的必要性（解析 Body 是耗時操作）？
  • [ ] HTTP 狀態檢核：是否具備 res.ok 的檢查意識？
  • [ ] 系統穩健度：在模擬斷網或伺服器宕機情況下，網頁是否能正確顯示錯誤訊息而非「靜默卡死」？
