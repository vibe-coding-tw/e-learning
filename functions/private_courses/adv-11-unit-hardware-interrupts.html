🧐 1. 什麼是中斷？主動 vs. 被動

輪詢 (Polling) 就像是每隔一秒看一次手機有沒有簡訊；中斷 (Interrupt) 則是等簡訊來時讓鈴聲主動通知你。

System Efficiency

  • 搶佔執行流：中斷能立即暫停 Main Loop，確保緊急事件（如限位開關、感測脈衝）不漏失。
  • 釋放資源：CPU 不再需要浪費時間在無效的判斷上，可以全速處理主要的演算法。

📜 2. 中斷服務函式 (ISR) 的嚴苛規則

ISR 程式碼必須「極速執行」。在裡面做太多事會導致主程式崩潰，甚至造成硬體重啟。

No Delay

絕對不准調用延遲函式。

No Serial

Serial 輸出太慢，請避免使用。

Volatile

共享變數必須宣告為 volatile。

// 專業 ISR 範例
volatile int count = 0;

void IRAM_ATTR handleEvent() {
  count++; // 極簡邏輯，僅遞增
}

// setup() 內綁定
attachInterrupt(PIN, handleEvent, RISING);

⚡ 3. 中斷觸發類型與防抖

數位訊號的物理邊緣 (Edge) 是觸發核心。但機械開關的「彈跳雜訊」常導致錯誤的多重觸發。

Trigger Strategy

  • RISING / FALLING：偵測電位從 0 變 1 或 1 變 0 的瞬間。
  • Software Debounce：在 ISR 內利用時間檢查，剔除短於 50ms 的重複跳火。
  • IRAM_ATTR：將處理程式存放在內存中，加速中斷反應。

任務 1：光速反應測試：中斷 vs. 輪詢 (Reaction Speed Challenge)

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 模擬 CPU 忙碌：在 loop() 內加入一組 100ms 的 Busy Wait 計算任務。
  3. 輪詢法故障再現：用 digitalRead() 判斷按鍵，記錄在忙碌狀態下連續嘗試觸發按鈕的成功率（漏失紀錄）。
  4. 中斷法重構：引入 attachInterrupt()，將按鍵事件掛載到 ISR 中，驗證即使 CPU 在做繁重計算，按鈕是否能「瞬間奪回」控制權。

任務 2：「彈跳惡夢」：軟體去彈跳實作 (Software Debouncing)

角色扮演：訊號濾波工程師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 機械抖動監測：觀察並輸出按一下按鈕後中斷觸發的次數。紀錄物理抖動造成的 False Positive 次數。
  2. 微秒計時過濾：在 ISR 中實作 mills() 判斷邏輯，過濾掉任何在 50ms 以內的二連跳訊號。
  3. 變數同步一致性：正確使用 volatile 關鍵字宣告全域變數，防止編譯器優化導致數據讀取過期。
  4. 正確性驗證報告：確保連續快速手點 10 次後，變數累計值精確等於 10。

任務 3：馬達編碼器實戰：高頻壓力計數 (High-Frequency Stress Test)

角色扮演：系統可靠性工程師 | 難度：⭐️⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 極速脈衝輸入：將光電編碼器訊號接入中斷引腳，模擬馬達全速轉動時的高頻脈衝流（>1000 次/秒）。
  2. 內存響應優化：為 ISR 函式添加 IRAM_ATTR，確保處理程式直接在靜態記憶體中極速運行。
  3. 亂序與丟包分析：在 README 中對比在高頻下，使用 IRAM_ATTR 與否在數據穩定度上的差異。
  4. 代碼 Push：將具備高效能中斷計數框架的代碼提交至 GitHub 倉庫。

GitHub Classroom 詳細操作指南

本單元教學重點：將處理器從「被動等待」轉為「事件驅動」的高效架構。

  1. 第一步：建立搶佔意識 - 透過「忙碌計算 vs. 按鈕漏失」的實驗，讓學員親手證實輪詢法的不可信。
  2. 第二步：對抗物理雜訊 - 強調整個嵌入式世界皆有「彈跳」，去彈跳是從實驗室進入工業產品的第一道工序。
  3. 第三步：極致效能優化 - 解釋 `IRAM_ATTR` 的內在機制，讓學員體悟即使是微秒級的優化，在高頻感知中也有巨大差異。
