🧐 1. 什麼是中斷？主動 vs. 被動

輪詢 (Polling) 就像是每隔一秒看一次手機有沒有簡訊；中斷 (Interrupt) 則是等簡訊來時讓鈴聲主動通知你。這就是事件驅動架構的精髓。

System Efficiency

  • 搶佔執行流：中斷能立即暫停 Main Loop，確保緊急事件不漏失。
  • 釋放資源：CPU 可以全速處理主要演算法，而非無謂等待。

📜 2. 中斷服務函式 (ISR) 的嚴苛規則

ISR 程式碼必須「極速執行」。在裡面做太多事會導致主程式崩潰，甚至造成系統看門狗（Watchdog）重啟。

No Delay

不准調用延遲函式。

No Serial

Serial 輸出太慢，請避免。

Volatile

共享變數需加 volatile。

// ISR Safe Pattern
volatile int count = 0;

void IRAM_ATTR handleISR() {
  count++; // Mini Logic
}

// attach at setup()

⚡ 3. 中斷觸發類型與防抖

數位訊號的物理邊緣 (Edge) 是觸發核心。但機械開關的「彈跳雜訊」常導致錯誤的多重觸發，需透過硬體或軟體層級的濾波來解決。

Trigger Strategy

  • RISING / FALLING：偵測電位從 0 變 1 或 1 變 0 的瞬間。
  • Software Debounce：利用時間檢查，剔除極短時間內的重複跳火。
  • IRAM_ATTR：將處理程式存放在 IRAM，加速中斷反應。

單元課程 進階 11 - 中斷機制

硬體中斷實驗 (Hardware Interrupts Lab)

專案簡介

歡迎來到 adv-11-unit-hardware-interrupts 實驗！在這個單元中，我們將學習如何讓微處理器具備「本能反應」。硬體中斷是嵌入式系統處理緊急或高頻事件的關鍵技術。

本程式庫提供了一個基礎框架，讓你體驗「輪詢 (Polling)」與「中斷 (Interrupt)」在效能上的巨大落差。

學習目標

  1. 理解硬體中斷的觸發機制 (Rising, Falling, Change)。
  2. 掌握 volatile 關鍵字在多執行緒環境（ISR 與 Loop）中的重要性。
  3. 實作 ISR (Interrupt Service Routine) 並進行微秒級的去彈跳 (Debouncing) 處理。
  4. 體驗 IRAM_ATTR 如何優化中斷響應速度。

實驗任務

任務 1：實作去彈跳邏輯

在 src/main.ino 中，你需要完善 handleButtonInterrupt 函式。目前它僅有基本的骨架，你需要：

  • 使用 millis() 或 micros() 判斷兩次中斷之間的時間差。
  • 只有當時間差大於 DEBOUNCE_DELAY (20ms) 時，才增加 interruptCount。

任務 2：響應對比觀察

  1. 觀察序列號監視器 (Serial Monitor) 的輸出。
  2. 注意 loop() 函式中有一個 delay(100) 的忙碌迴圈。
  3. 嘗試在忙碌期間點擊按鈕，觀察中斷是否依然能精準捕捉到每一次點擊（相較於傳統的 digitalRead 輪詢）。

任務 3：效能分析報告

完成實驗後，請將你的觀察結果記錄在 docs/latency_report.md 中。

開發指南

核心檔案結構

  • src/main.ino: 主要邏輯實作位置。
  • include/interrupt_cfg.h: 腳位配置與共享變數宣告。
  • docs/latency_report.md: 實驗結果記錄表。

重要提示

  • 不要在 ISR 中使用 Serial.print() 或 delay()：這會導致系統不穩定甚至崩潰。
  • 共享變數必須宣告為 volatile：確保編譯器不會過度優化這些在主迴圈外被修改的變數。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template 程式庫) 配置建議

中斷程式碼對執行時間與記憶體配置極其敏感，建議範本包含：

  • 📂 include/interrupt_handlers.h：中斷配置檔。定義 volatile 全域變數與 ISR 函式原型，留下 IRAM_ATTR 實作與邏輯 TODO。
  • 📂 src/latency_test.ino：效能對比實驗。預置一個 100ms 的「忙碌迴圈 (Busy Loop)」，讓學員在同步執行的 loop 中體驗輪詢與中斷的反應落差。
  • 📂 tests/isr_sanity.test.js：自動化檢查腳本。使用正則表達式檢核 ISR 內部是否「誤用」了 delay() 或 Serial.print() 等阻塞式函式（這會導致系統崩潰）。
  • 📂 ANALYSIS.md：效能分析報告模板。提供對比表格，讓學員記錄在不同 CPU 負載下，中斷法的「事件不漏失率」。

2. 作業任務部署細節

任務 1：響應對比實驗 (Reaction Battle Lab)

  • 目標：證明中斷能「搶佔」CPU 執行權，解決低頻輪詢的感知盲區。
  • Classroom 部署建議：
    • 忙碌模擬檢核：學員必須在 loop 中加入長延遲，觀測輪詢法完全漏失快速點選的情況。
    • 事件劫持實作：使用 attachInterrupt(pin, ISR, RISING)，證明即使 CPU 在忙碌，計數器依然能精準跳動。
    • Autograding：透過程式模擬高負載環境，核測計數器是否能在 loop 阻塞時依然維持正常。

任務 2：中斷去彈跳演算法 (Interrupt Debouncing Lab)

  • 目標：解決機械彈跳產生的「幽靈觸發」，建立純淨的數位訊號。
  • Classroom 部署建議：
    • 計時濾波實作：學員需在 ISR 中加入基於 millis() 的時間窗口判斷。 
      // 學生需實作：微秒級防抖
void IRAM_ATTR handleButton() {
    if (millis() - lastTime > 20) {
        count++;
        lastTime = millis();
    }
}
    • 原子性與同步：要求學員在 README.md 中說明為什麼共享變數必須宣告為 volatile。
    • 感知一致性驗證：手動點擊 20 次，檢核計數器是否確實回報 20（不多不少）。

任務 3：高頻壓力測試 (High-Speed Stress Lab)

  • 目標：因應馬達全速運作下的編碼器計數，驗證系統的極限響應能力。
  • Classroom 部署建議：
    • IRAM 優化要求：學員必須為 ISR 添加 IRAM_ATTR，將代碼強制存放於高速內存。
    • 計數完整性報告：全速轉動馬達 1 分鐘，記錄計數器的數值並與公式推導的轉速進行對比。
    • PR 深度核核：點評其對「中斷嵌套限制」的理解。中斷程式碼是否足夠簡潔？是否在 10 微秒內執行完畢？

3. 即時系統工程導師點評標準 (Real-Time Engineering Benchmarks)

此單元的價值在於 「對 CPU 執行流的深度控制力」：

  • [ ] ISR 潔淨度：是否完全排除了 slow I/O（如 Serial）？執行路徑是否最短化？
  • [ ] 資料競爭防護：變數宣告是否考慮到了多執行緒環境（ISR vs. Loop）？
  • [ ] 系統穩定性：在高頻脈衝下，ESP32 是否發生過 Watchdog Timer (WDT) 重啟？
