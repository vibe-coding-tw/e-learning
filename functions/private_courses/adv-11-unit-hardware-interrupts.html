🧐 1. 什麼是中斷？主動 vs. 被動

在微控制器的世界中，CPU 如何感知外部事件？最直覺的方式是輪詢 (Polling)——像不斷翻看手機是否有新訊息。 但這種方式讓 CPU 忙於檢查，無法專注做其他事。中斷 (Interrupt) 是更優雅的解方：讓硬體在事件發生時主動「拍 CPU 的肩膀」， CPU 才需要暫停手邊工作、處理事件、再返回。這就是事件驅動架構 (Event-Driven Architecture) 的精髓。

注意：中斷是一種硬體機制。當外部腳位偵測到電位變化，CPU 會立即搶佔 (Preempt) 正在執行的程式碼， 跳轉到預先註冊的中斷服務函式 (ISR, Interrupt Service Routine)。ISR 執行完畢後，CPU 自動返回原本的程式流。

想像你正在寫報告（Main Loop），門鈴響了（中斷觸發）。你放下筆、去開門處理事情（ISR），然後回到座位繼續寫。 重要的是：開門這件事必須快速完成——如果你在門口聊了一小時，報告進度就會嚴重落後。 這正是 ISR 必須「極速、極簡」的原因。

輪詢 vs. 中斷 比較表
特性 輪詢 (Polling) 中斷 (Interrupt)
CPU 使用率 高（持續檢查） 低（僅事件時啟動）
反應速度 取決於檢查頻率 微秒級即時響應
是否漏失事件 可能（若檢查間隔太長） 不會（硬體保證觸發）
程式複雜度 簡單直觀 需處理共享變數與競態
適用場景 低頻率、非即時性 高頻脈衝、精確計時
C++ (Arduino) — 中斷基礎範例
volatile int pulseCount = 0;   // 共享變數必須加 volatile

void IRAM_ATTR handlePulse() {
    pulseCount++;              // ISR 內只做最小操作
}

void setup() {
    Serial.begin(115200);
    pinMode(SENSOR_PIN, INPUT_PULLUP);
    // 在 SENSOR_PIN 偵測到下降緣時，呼叫 handlePulse
    attachInterrupt(digitalPinToInterrupt(SENSOR_PIN),
                    handlePulse, FALLING);
}

void loop() {
    // 主程式可安心做其他事，中斷會自動累加 pulseCount
    Serial.println(pulseCount);
    delay(500);
}
提示：ESP32 的每個 GPIO 都支援中斷，但 Arduino UNO 只有 pin 2 與 pin 3 支援外部中斷。 開發前請先查閱你的開發板的中斷腳位對照表。

📜 2. 中斷服務函式 (ISR) 的嚴苛規則

ISR 是在中斷發生時由硬體自動呼叫的函式。由於它會搶佔主程式的執行流，因此有極其嚴格的編寫規範。 違反這些規則輕則導致資料損壞、序列埠亂碼，重則造成系統當機或看門狗 (Watchdog) 重啟。

警告：在 ISR 內呼叫 delay()、Serial.print()、millis()（在某些平台） 等依賴中斷的函式，會導致系統死鎖。因為這些函式本身也需要中斷才能運作，但在 ISR 執行期間，中斷是被禁用的。
ISR 黃金守則一覽
守則 說明 違反後果
禁止 delay() delay() 依靠計時器中斷，在 ISR 中無法運作 系統凍結
禁止 Serial.print() 序列通訊耗時且使用緩衝區中斷 資料損毀、輸出亂碼
共享變數加 volatile 告訴編譯器不要最佳化此變數，每次都從記憶體讀取 主程式讀到過時的快取值
加上 IRAM_ATTR 將 ISR 放入 IRAM（內部 RAM），避免從 Flash 讀取的延遲 ESP32 可能觸發 LoadProhibited 例外
程式碼極短 只做旗標設定或計數器累加，複雜邏輯留給 loop() 阻塞其他中斷、影響系統即時性

正確的模式是：ISR 內只設旗標或累加計數器，所有「慢操作」（列印、計算、通訊）都在 loop() 中根據旗標來執行。 這種「旗標驅動」的設計模式，是嵌入式系統中最常見且最安全的中斷處理架構。

C++ (Arduino) — 安全 ISR 模式
volatile bool dataReady = false;    // 旗標：ISR 設定，loop() 檢查
volatile unsigned long lastTime = 0;

void IRAM_ATTR sensorISR() {
    lastTime = micros();   // micros() 在 ESP32 ISR 中可安全使用
    dataReady = true;      // 只設旗標，不做任何慢操作
}

void loop() {
    if (dataReady) {
        dataReady = false;
        noInterrupts();                // 暫時禁用中斷
        unsigned long t = lastTime;    // 安全複製 volatile 變數
        interrupts();                  // 重新啟用中斷

        Serial.print("Event at: ");    // 慢操作在 loop() 中執行
        Serial.println(t);
    }
    // ... 其他主程式邏輯 ...
}
重要：當主程式需要讀取 ISR 修改的變數時，應使用 noInterrupts() / interrupts() 包裹讀取操作，防止在讀取多位元組變數（如 unsigned long）時，ISR 介入導致讀到「半新半舊」的值。 這稱為原子操作 (Atomic Operation) 保護。

⚡ 3. 中斷觸發類型與防抖

中斷觸發的核心在於偵測數位訊號的邊緣 (Edge)——電位從低到高或從高到低的瞬間。 ESP32 與 Arduino 提供多種觸發模式，開發者需根據感測器的輸出特性選擇正確的模式。 但在實際應用中，機械開關與訊號線上的雜訊會產生彈跳 (Bounce)，導致一次按壓觸發數十次中斷。

中斷觸發模式
模式 觸發條件 適用場景
RISING 電位從 LOW → HIGH（上升緣） 光電遮斷器離開遮蔽區
FALLING 電位從 HIGH → LOW（下降緣） 按鈕按下（INPUT_PULLUP 配置）
CHANGE 任何電位變化 需要雙倍解析度的編碼器
LOW 電位維持在 LOW 時持續觸發 少用；適合喚醒深度睡眠

在測速應用中，光電遮斷器的輸出通常非常乾淨（因為使用光學感測，不存在機械接觸）， 所以防抖的需求較低。但如果你使用的是機械式按鈕或簧片開關，軟體防抖就至關重要。

C++ (Arduino) — 軟體防抖 ISR
#define DEBOUNCE_US  200   // 200 微秒內的重複觸發視為彈跳

volatile unsigned long lastTrigger = 0;
volatile int cleanCount = 0;

void IRAM_ATTR debouncedISR() {
    unsigned long now = micros();
    if (now - lastTrigger > DEBOUNCE_US) {
        cleanCount++;
        lastTrigger = now;
    }
}

void setup() {
    Serial.begin(115200);
    pinMode(SENSOR_PIN, INPUT_PULLUP);
    attachInterrupt(digitalPinToInterrupt(SENSOR_PIN),
                    debouncedISR, FALLING);
}

void loop() {
    noInterrupts();
    int c = cleanCount;
    interrupts();
    Serial.printf("Clean pulses: %d\n", c);
    delay(500);
}
提示：防抖時間的選擇取決於硬體特性。光電遮斷器通常只需 50–200μs； 機械按鈕可能需要 5–20ms。建議先用示波器或 micros() 實測彈跳持續時間，再設定合適的門檻值。
觀念釐清：對於光電編碼盤測速，通常不需要軟體防抖，因為光學訊號的邊緣非常銳利。 過度防抖反而會「吞掉」合法的高頻脈衝，導致高速時計數不足。 只有在低速手動旋轉時，才可能因為格柵邊緣停留在感測器縫隙上而產生誤觸發。

硬體中斷實驗 (Hardware Interrupts Lab)

專案簡介

歡迎來到 adv-11-unit-hardware-interrupts 實驗！在這個單元中，我們將學習如何讓微處理器具備「本能反應」。硬體中斷是嵌入式系統處理緊急或高頻事件的關鍵技術。

本程式庫提供了一個基礎框架，讓你體驗「輪詢 (Polling)」與「中斷 (Interrupt)」在效能上的巨大落差。

學習目標

  1. 理解硬體中斷的觸發機制 (Rising, Falling, Change)。
  2. 掌握 volatile 關鍵字在多執行緒環境（ISR 與 Loop）中的重要性。
  3. 實作 ISR (Interrupt Service Routine) 並進行微秒級的去彈跳 (Debouncing) 處理。
  4. 體驗 IRAM_ATTR 如何優化中斷響應速度。

實驗任務

任務 1：實作去彈跳邏輯

在 src/main.ino 中，你需要完善 handleButtonInterrupt 函式。目前它僅有基本的骨架，你需要：

  • 使用 millis() 或 micros() 判斷兩次中斷之間的時間差。
  • 只有當時間差大於 DEBOUNCE_DELAY (20ms) 時，才增加 interruptCount。

任務 2：響應對比觀察

  1. 觀察序列號監視器 (Serial Monitor) 的輸出。
  2. 注意 loop() 函式中有一個 delay(100) 的忙碌迴圈。
  3. 嘗試在忙碌期間點擊按鈕，觀察中斷是否依然能精準捕捉到每一次點擊（相較於傳統的 digitalRead 輪詢）。

任務 3：效能分析報告

完成實驗後，請將你的觀察結果記錄在 docs/latency_report.md 中。

開發指南

核心檔案結構

  • src/main.ino: 主要邏輯實作位置。
  • include/interrupt_cfg.h: 腳位配置與共享變數宣告。
  • docs/latency_report.md: 實驗結果記錄表。

重要提示

  • 不要在 ISR 中使用 Serial.print() 或 delay()：這會導致系統不穩定甚至崩潰。
  • 共享變數必須宣告為 volatile：確保編譯器不會過度優化這些在主迴圈外被修改的變數。

void IRAM_ATTR handleButton() {

  • Classroom 部署建議：
    • IRAM 優化要求：學員必須為 ISR 添加 IRAM_ATTR，將代碼強制存放於高速內存。
    • 計數完整性報告：全速轉動馬達 1 分鐘，記錄計數器的數值並與公式推導的轉速進行對比。
    • PR 深度核核：點評其對「中斷嵌套限制」的理解。中斷程式碼是否足夠簡潔？是否在 10 微秒內執行完畢？

3. 即時系統工程導師點評標準 (Real-Time Engineering Benchmarks)

此單元的價值在於 「對 CPU 執行流的深度控制力」：

  • [ ] ISR 潔淨度：是否完全排除了 slow I/O（如 Serial）？執行路徑是否最短化？
  • [ ] 資料競爭防護：變數宣告是否考慮到了多執行緒環境（ISR vs. Loop）？
  • [ ] 系統穩定性：在高頻脈衝下，ESP32 是否發生過 Watchdog Timer (WDT) 重啟？
