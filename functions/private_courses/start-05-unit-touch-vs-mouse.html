Module 1: Intuitive Lab

滑鼠 vs. 手指：速度對決

為什麼手機上 click 比較慢？

在觸控螢幕剛誕生時，為了偵測「雙擊縮放 (Double Tap to Zoom)」，瀏覽器在使用者點擊後會等待約 300ms，確認沒有第二次點擊才會觸發 `click` 事件。

這對於一般網頁瀏覽沒問題，但對於即時遙控車來說，300ms 的延遲就像是一世紀那麼長！

⚠️ 實驗觀察重點

  • 使用滑鼠點擊方塊，觀察 log。
  • 使用手機模式 (DevTools Cmd+Shift+M) 輕觸方塊。
  • 注意 touchstart 與 click 的時間差。

Module 2: Core Principles

事件結構與監聽陷阱

特性 滑鼠事件 (Mouse) 觸控事件 (Touch)
關鍵事件 mousedown, mousemove, mouseup touchstart, touchmove, touchend
座標取得 e.clientX, e.clientY e.touches[0].clientX (多層結構)
多點支援 無 (單游標) 有 (TouchList 陣列)
預設行為 選取文字、拖曳圖片 捲動頁面、雙擊縮放、長按選單
JavaScript - Universal Coordinate Helper 
function getPosition(e) {
  let x, y;
  
  if (e.type.includes('mouse')) {
    // 滑鼠直接取座標
    x = e.clientX;
    y = e.clientY;
  } else {
    // 觸控要判斷事件類型
    // touchmove/start 看 touches
    // touchend 看 changedTouches
    let touch = e.touches[0] || e.changedTouches[0];
    x = touch.clientX;
    y = touch.clientY;
  }
  
  return { x, y };
}

! 被動監聽器 (Passive Listener)

為了提升捲動效能，現代瀏覽器預設將 `touchstart` 和 `touchmove` 視為 { passive: true }。 這意味著你無法在這些事件中呼叫 preventDefault() 來阻止網頁捲動。

若要製作搖桿或拖曳功能，必須明確設定 { passive: false }：

element.addEventListener('touchmove', handler, { passive: false });

觸控 vs. 滑鼠：打造全平台通用的操控體驗

在本單元中，你將學習如何處理跨裝置的輸入差異，打造一個在桌面端（滑鼠）與行動端（觸控）都能流暢運行的 Web 介面。

🎯 學習目標

  1. 掌握 事件序列 (Event Sequencing)：理解 touchstart -> touchend -> mousedown -> click 的觸發順序。
  2. 攔截 幽靈點擊 (Ghost Clicks)：消除行動端 300ms 的點擊延遲。
  3. 實作 輸入正規化 (Input Normalization)：撰寫通用的座標解析函數。
  4. 優化 UX 體感：封鎖不必要的瀏覽器預設行為（如捲動鎖定、右鍵選單）。

🚀 任務說明

任務 1：打造全平台萬用按鈕 (Omni-Button)

  • 目標：同時監聽 mousedown 與 touchstart，但確保在行動端點擊時僅觸發一次邏輯，且無延遲感。
  • 關鍵點：在 touchstart 事件中正確使用 e.preventDefault() 攔截後續的模擬事件。

任務 2：防抖滑動控制 (Interaction Locking)

  • 目標：實作一個滑動條，當使用者在滑動時，防止行動端瀏覽器發生意外的頁面捲動。
  • 關鍵點：實作通用的 getPosition(e) 輔助函數，並對觸控事件設定 { passive: false } 以便進行攔截。

任務 3：原生 App 體感優化 (Native Feel UX)

  • 目標：封鎖長按時的系統右鍵選單，並清除行動端預設的點擊高亮色，提升 Web App 質感。
  • 關鍵點：攔截 contextmenu 事件，並使用 CSS 調整觸控回饋視覺。

2. 作業任務部署細節

任務 1：打造全平台萬用按鈕 (Omni-Button Implementation Lab)

  • 目標：掌握「事件競態條件 (Race Condition)」。同時監聽 mousedown 與 touchstart，但確保在行動端僅觸發一次執行邏輯，徹底消除 300ms 延遲。
  • Classroom 部署建議：
    • 幽靈點擊攔截實作：在 touchstart 事件中使用 e.preventDefault() 斬斷後續模擬的 mousedown 鍊條。
    • 異步處理挑戰：確保按鈕在快速連續點擊下，依然維持高度的穩定性。
    • 驗證要點：唯一性與速度感審核。在手機上點擊時，後台日誌是否僅出現一筆 Action？延遲是否感覺「歸零」？當切換回電腦滑鼠時，功能是否依然正常？

任務 2：防抖滑動控制 (Interaction Locking Lab)

  • 目標：掌握「被動監聽器 (Passive Listener) 攔截術」。製作一個橫向滑動條，確保使用者在拖曳時，行動端網頁不會發生「意外的上下捲動」。
  • Classroom 部署建議：
    • 視窗滾動鎖定實作：對滑桿容器設定 { passive: false }，並在 touchmove 中精確執行 preventDefault()。
    • 統一座標輔助函數：實作一個 getPosition(e) 函數，能自動適應滑鼠物件或觸控物件的座標提取路徑。
    • 驗證要點：操作流暢度審核。拖曳滑桿時，背景頁面是否完全靜止？手指快速滑動是否有位移或跳動感？

任務 3：原生 App 體感優化 (Native Feel UX Lab)

  • 目標：掌握「瀏覽器行為防禦」。封鎖長按時的系統右鍵選單 (Context Menu) 與預設的點擊高亮色，讓 Web App 質感升級。
  • Classroom 部署建議：
    • 系統選單屏蔽實作：監聽 contextmenu 事件並回傳 false (或執行 preventDefault)，消除所有阻礙操控的對話框。
    • 視覺質感清潔實作：使用 CSS 清除行動端預設的藍色 Tap 框，並防止在激烈操作中誤選取文字。
    • 驗證要點：APP 質感模擬審核。長按按鈕是否不再跳出「複製/儲存圖片」？按鈕按下時是否有自定義的物理回饋（如震動感或 CSS 陰影變化）？
