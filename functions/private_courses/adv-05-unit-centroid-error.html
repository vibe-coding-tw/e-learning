質心的數學本質

質心 (Centroid) 是一組數據點的「加權平均位置」。在影像處理中， 想像所有被偵測到的目標像素（例如：黑色循跡線上的白色像素）都是等重的小球， 質心就是這群小球的平衡點——如果你把一片標記了所有目標像素的薄板放在指尖上， 能讓它保持水平的那個支撐點，就是質心。

對循跡車而言，質心的 X 座標直接反映了「目標線在畫面中的水平位置」。 這個單一數值取代了成千上萬個像素，成為控制系統唯一需要的導航輸入。 從數萬個像素到一個座標——這就是質心計算的核心價值。

質心計算公式

給定二值化影像中所有目標像素的座標 (x₁, y₁), (x₂, y₂), ..., (xₙ, yₙ)，質心座標的計算公式為：

// 質心公式（加權平均）
centroid_x = (x₁ + x₂ + ... + xₙ) / N
centroid_y = (y₁ + y₂ + ... + yₙ) / N

// 其中 N = 目標像素的總點數

這個公式在數學上等價於求所有目標像素在 X 軸與 Y 軸上的算術平均。 當目標物體偏向畫面右側時，大多數像素的 X 座標較大，質心自然偏右；反之亦然。

程式碼實作

以下是在 JavaScript 中遍歷二值化影像矩陣、計算質心的標準實作模式：

function calculateCentroid(binaryData, width, height) {
  let sumX = 0, sumY = 0, count = 0;

  for (let y = 0; y < height; y++) {
    for (let x = 0; x < width; x++) {
      if (binaryData[y * width + x] > 128) {
        sumX += x;
        sumY += y;
        count++;
      }
    }
  }

  // 安全防護：避免除以零
  if (count === 0)
    return { x: width / 2, y: height / 2 };

  return { x: sumX / count, y: sumY / count };
}
警告 當畫面中完全沒有目標像素（count === 0）時，若直接執行除法將產生 NaN， 可能導致下游控制邏輯崩潰甚至讓馬達失控。務必在計算前加入零點數防護，回傳影像中心點作為安全預設值。

手算練習：L 型物體的質心

假設一個 4×4 的方格中有一個 L 型物體，目標像素座標分別為： (0,0)、(0,1)、(0,2)、(1,2)，共 4 個像素。

座標軸 總和 平均（質心）
X 軸 0 + 0 + 0 + 1 = 1 1 / 4 = 0.25
Y 軸 0 + 1 + 2 + 2 = 5 5 / 4 = 1.25
提示 質心座標 (0.25, 1.25) 並不落在任何一個像素上——這是正常的。質心是一個數學上的連續值， 它代表的是物體的「重心平衡點」，而非某個特定像素的位置。在後續的誤差計算中， 我們只需要質心的 X 座標來判斷偏移方向。

視覺誤差 (Visual Error) 的定義

有了質心座標，下一步是回答一個關鍵問題：目標偏離了畫面中心多遠、往哪個方向？ 這就是「視覺誤差 (Visual Error)」的定義——它是質心位置與參考基準點（通常是影像中心）之間的有方向距離。

視覺誤差是連接「感知」與「控制」的數學橋樑。它將攝影機看到的空間資訊， 轉譯為一個帶有正負號的數值——正值代表目標偏右，負值代表偏左，零代表完美對齊。 馬達控制系統只需要這一個數字，就能決定「該往哪轉、轉多少」。

誤差計算公式

以水平方向為例，假設影像寬度為 W 像素，影像中心點的 X 座標為 W / 2：

// 視覺誤差計算
let imageCenter = width / 2; // 例如：320 / 2 = 160
let error = centroidX - imageCenter;

// error > 0 → 目標偏右 → 車子需右轉修正
// error < 0 → 目標偏左 → 車子需左轉修正
// error = 0 → 完美對齊 → 直行

誤差的正負號與物理意義

誤差的正負號決定了修正方向，絕對值決定了修正力道。 以下表格列出不同場景下的誤差值與對應行為：

質心位置 (centroidX) 影像中心 誤差 (Error) 物理意義
240 160 +80 目標大幅偏右，需要強力右轉修正
170 160 +10 目標微偏右，輕微修正即可
160 160 0 完美對齊，保持直行
80 160 −80 目標大幅偏左，需要強力左轉修正
備註 上述誤差值的單位是「像素」。在 320 像素寬的影像中，最大誤差為 ±160 像素。 但不同解析度的攝影機會產生不同範圍的數值，這會導致同一套控制參數在不同裝置上表現不一致。 解決方案就是下一單元要學習的正規化 (Normalization)。

邊界情況的防護

在實際運作中，有幾種特殊情況需要特別處理：

  • 目標消失 (N = 0)：畫面中完全沒有目標像素。若直接計算質心會產生除以零錯誤。 建議回傳誤差 0（保持直行）或使用上一幀的誤差值。
  • 多目標干擾：背景雜訊產生了非目標的像素群，拉偏了質心。 可透過「最小像素門檻」過濾：當 N < 10 時視為雜訊，不更新質心。
  • 目標跨越邊界：目標物體部分超出畫面。此時質心會被拉向可見部分， 誤差可能比實際偏移量小。
警告 當 N = 0 時，切勿讓車輛保持上一次的轉向指令——這會導致車輛「盲衝」出賽道。 安全的做法是將誤差歸零並降低車速，或觸發 Failsafe 停車機制。 這個觀念將在閉迴路控制單元中深入探討。

數據正規化 (Normalization)

上一單元計算出的視覺誤差是以「像素」為單位的原始值。 問題在於：同一個物體在 320 像素寬的影像中最大誤差為 ±160，但在 640 像素寬的影像中最大誤差為 ±320。 如果控制參數（Kp）是針對特定解析度調校的，更換攝影機或調整解析度後就會完全失效。

正規化 (Normalization) 解決了這個問題：將誤差縮放到 [-1.0, 1.0] 的標準區間。 正規化後，-1.0 代表目標在畫面最左邊，+1.0 代表最右邊，0 代表正中央—— 無論攝影機解析度是 160 還是 1280，這套控制邏輯都能通用。這就是演算法的抽象化。

正規化公式

// 正規化：將像素誤差縮放到 [-1.0, 1.0]
let halfWidth = width / 2;
let rawError = centroidX - halfWidth;
let normalizedError = rawError / halfWidth;

// 範例（320px 寬影像）：
// centroidX = 240 → rawError = 80 → normalized = 0.5
// centroidX = 80 → rawError = -80 → normalized = -0.5
// centroidX = 160 → rawError = 0 → normalized = 0.0

正規化的數學本質是線性映射：將原始值域 [0, W] 映射到 [-1.0, 1.0]。 這保證了誤差與偏移量之間的線性關係——目標偏離一倍，正規化誤差也精確翻倍。

死區 (Deadzone) 機制

在現實環境中，即使車輛已經對齊賽道中心，質心仍會因為攝影機抖動、光影變化而產生微小的來回跳動。 如果控制器對這些微小波動都做出反應，馬達將不停地左右微調，造成不必要的能耗與機械磨損——甚至使車身輕微蛇行。

死區 (Deadzone) 是解決這個問題的工程策略： 當正規化誤差的絕對值小於一個設定的門檻值時，直接將誤差歸零，不觸發修正。

function normalizeError(centroidX, width, deadzone) {
  let halfWidth = width / 2;
  let rawError = centroidX - halfWidth;

  // 死區判定：微小偏移不觸發修正
  if (Math.abs(rawError) < deadzone)
    return 0;

  return rawError / halfWidth;
}

// 使用範例：deadzone = 5 像素
let error = normalizeError(163, 320, 5);
// rawError = 3, |3| < 5 → 回傳 0（不修正）
注意 死區大小需要仔細權衡：設太大會讓車輛在偏離中心較多時仍不修正（反應遲鈍）； 設太小則無法有效抑制抖動。典型的起始值是 3–8 像素， 實際值需根據攝影機的雜訊程度與賽道寬度進行調校。

ROI（感興趣區域）進階優化

正規化解決了解析度依賴問題，但還有一個進階技巧可以大幅提升循跡穩定性： ROI (Region of Interest)。不掃描整張影像，而是只計算畫面特定區域的質心。

  • 下半部 ROI：只掃描影像下半部（靠近車輛的區域），減少遠處雜訊干擾，質心更穩定。
  • 多區域 ROI：將影像分為上、中、下三段分別計算質心，上段可預測前方彎道走向。
  • 動態 ROI：根據上一幀的質心位置，動態調整下一幀的掃描區域，進一步減少運算量。
提示 在作業中，嘗試將 160×120 的影像分為上下兩半，分別計算質心。觀察上半部質心是否能提前預測即將到來的彎道方向。 這個技巧在進階賽道中可以顯著提升車輛的過彎流暢度。

質心計算與視覺誤差 (Centroid Calculation & Vision Error)

本專案是一個關於自動駕駛視覺系統核心算法的練習。你將學習如何從原始像素數據中提取「質心」(Centroid)，並將其轉換為導航系統可以使用的「誤差」(Error) 訊號。

學習目標

  1. 實作空間加權平均算法來計算物體質心。
  2. 實作誤差正規化 (Normalization) 與死區 (Deadzone) 邏輯。
  3. 處理空視野與雜訊等例外狀況，提高系統魯棒性。

任務說明

任務 1：質心提取器 (Centroid Extractor)

在 src/vision_math.js 中實作 calculateCentroid 函式：

  • 遍歷影像像素。
  • 計算所有「開啟」像素 (像素值 > 128) 的 X 與 Y 座標總和。
  • 回傳質心的平均座標。
  • 重要：必須處理 totalPixels === 0 的情況（例如回傳畫面中心點），以避免除以零導致的系統崩潰。

任務 2：誤差量化與正規化 (Error Normalization & Deadzone)

在 src/vision_math.js 中實作 normalizeError 函式：

  • 將質心的 X 座標轉換為相對於中心點的偏移量。
  • 將偏移量縮放至 [-1.0, 1.0] 的範圍（左側為負，右側為正）。
  • 實作「死區」邏輯：當偏移量絕對值小於門檻值（預設為 5）時，回傳 0，以避免在中心附近的微小抖動。

任務 3：環境干擾下的魯棒性

確保你的算法能處理 data/ 中的各種模擬情況，包括空視野與帶有雜訊的畫面。

  1. 複製此程式庫到你的本地環境。
  2. 使用瀏覽器開啟 ui/dashboard.html 以查看視覺化儀表板。
  3. 在 src/vision_math.js 中撰寫你的實作邏輯。
  4. 執行測試以驗證你的程式碼。

if (Math.abs(centroidX - imageMidPoint) < 5) error = 0;
    • 驗證方式：在儀表板上模擬物體在中心微小擺動，檢核「死區」是否能有效抑制指針抖動。這展現了軟體對硬體抖動的控制策略。

任務 3：環境干擾下的抗性測試 (Noise Robustness Challenge)

  • 目標：設計過濾機制，確保質心不會被背景雜訊「拉走」。
  • Classroom 部署建議：
    • 魯棒性實作：要求學員實作一個「區域權重」或「最小像素門檻」。例如：若偵測到的點數 $N < 10$，則視為雜訊並維持上一幀的質心。
    • 實驗報告：學員需在 README 中記錄：當背景出現「散點雜訊」時，其質心計算值偏離主體的最大百分比。
    • Tutor View (PR 審核)：導師点評學員是否考慮到了 ROI（感興趣區域）的概念，例如只計算畫面下半部的質心以提高循跡穩定性。

3. 量化感知導師點評標準 (Vision Quantization Benchmarks)

此單元的價值在於 「對底層數據處理的嚴謹度」：

  • [ ] 例外防護意識：代碼中是否包含了對「全黑視野（$N=0$）」的顯式判斷，以防止 NaN 導致系統連鎖崩潰？
  • [ ] 數學轉換能力：輸出的 Error 是否具備線性特徵？（即：位移一倍，Error 是否也隨之翻倍）。
  • [ ] 控制靈敏度權衡：設定的「死區大小」是否兼顧了「抑制抖動」與「及時回正」？
