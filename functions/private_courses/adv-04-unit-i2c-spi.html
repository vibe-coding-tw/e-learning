📢 I2C：省線的「廣播系統」

  • ● 僅需兩根線：SDA（數據）與 SCL（時鐘）。
  • ● 定址機制：主機透過廣播「地址」來決定與哪個設備通訊。
  • ● Open-Drain 架構：理解為什麼 I2C 總線需要上拉電阻。

⚡ SPI：高速的「專線系統」

  • ● 通常四根線：SCK、MOSI、MISO、CS（片選）。
  • ● 全雙工傳輸：數據可同時雙向移動，速度極快 (MHz)。
  • ● 點對點機制：主機利用專屬的 CS 線啟動特定設備。

📈 時序圖判讀與訊號擷取

開始位元 (Start Condition)

I2C 的獨家通訊協議：當 SCL 為高電平時，SDA 從高變低，標誌著對話的開始。

時鐘極性與相位 (CPOL/CPHA)

SPI 的四種模式。決定數據是在時鐘的「上升沿」還是「下降沿」被固定並讀取。

💡 Datasheet 尋寶：嘗試找出 MPU6050 的預設 I2C 地址，以及它在通訊時的最高時鐘頻率限制。

adv-04-unit-i2c-spi: 通訊協議與硬體實戰

歡迎來到「通訊協議與硬體實戰」單元！在這個單元中，你將深入學習 I2C 與 SPI 這兩種最常見的硬體通訊協議。你將不再只是單純地呼叫 Library，而是直接與晶片對話，理解物理訊號的邊界。

🎯 學習目標

  1. 掌握 I2C 協議：理解地址 (Address) 與應答 (ACK) 的物理過程。
  2. 理解 SPI 性能：探索數據通訊的物理極限與訊號完整性。
  3. 協議解析：學習將低階二進位數據 (Raw Data) 解析為具備語義的物理量 (JSON)。

🚀 任務說明

任務 1：I2C 設備大掃描 (I2C Scanner Lab)

  • 內容：實作一個 I2C 掃描工具，偵測連接到 ESP32 的所有設備。
  • 要求：
    • 實作掃描迴圈與 ACK 狀態判斷。
    • 在 README.md (或指定報告中) 列出偵測到的設備地址（例如 OLED 的 0x3C）。
    • 能夠區分「無設備」與「總線異常」。

任務 2：SPI 刷新率挑戰 (SPI Speed Benchmark)

  • 內容：測試不同 SPI 時脈頻率下的螢幕刷新表現。
  • 要求：
    • 分別測試 1MHz, 10MHz, 40MHz 的刷新 FPS。
    • 觀察並紀錄高頻下的訊號品質（是否有雪花或錯位）。
    • 確保 SPI_MODE (CPOL/CPHA) 配置正確。

任務 3：協議翻譯官：Raw to JSON

  • 內容：讀取感測器（如 MPU6050）的原始資料並轉換。
  • 要求：
    • 將 High/Low Bytes 正確拼接為 16-bit 整數。
    • 根據量程換算為物理單位（如 $m/s^2$）。
    • 輸出格式化後的 JSON 字串。

📂 程式庫結構

  • src/main.cpp: 整合專案核心邏輯。
  • src/bus_scanner.cpp: I2C 掃描實作。
  • include/ic_driver.h: SPI 驅動參數配置。
  • include/sensor_map.h: 暫存器映射與比例換算。
  • docs/: 實驗報告與參考文件。

📝 提交要求

  1. 完成所有 TODO 標記的程式碼。
  2. 確保程式碼符合位元運算的正確性（注意 Big-endian/Little-endian）。
  3. 更新 docs/scope_results.md 紀錄實驗結果。
  4. 將程式碼 push 到你的程式庫。

針對 adv-04-unit-i2c-spi (通訊協議與硬體實戰) 單元，這是一個深入探討「機器之間的對話規範」與「物理訊號限界」的課程，是所有嵌入式系統開發的核心基本功。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template 程式庫) 配置建議

硬體通訊單元的範本應強調「暫存器級別 (Register-level)」的理解，建議包含：

  • 📂 src/i2c_prober.cpp：提供基礎的 Wire 庫框架，留下掃描迴圈與 ACK 狀態判斷的 TODO。
  • 📂 src/spi_driver.h：預置圖形顯示初始化邏輯，留下可調整的 SPI_CLOCK_SPEED 宏定義位元。
  • 📂 include/registers.h：提供常見感測器（如 MPU6050）的暫存器地址表，讓學員練習定義「位元掩碼 (Bitmask)」。
  • 📂 docs/logic_analyze_guide.md：一份圖文並茂的指南，教導學員如何截取串列訊號並進行解碼分析。

2. 作業任務部署細節

任務 1：I2C 設備大掃描 (I2C Scanner Lab)

  • 目標：理解地址 (Address) 與應答 (ACK) 的物理過程。
  • Classroom 部署建議：
    • 驗證方式：學員需在 README 中提交其 ESP32 掃描到的設備地址列表（通常包含 OLED 的 0x3C 與 IMU 的 0x68）。
    • Autograding：檢查掃描代碼是否具備基本的錯誤處理邏輯（例如：能區分「沒設備」與「總線被拉低」的錯誤狀態）。

任務 2：SPI 刷新率挑戰 (SPI Speed Benchmark)

  • 目標：探索數據通訊的物理極限（訊號完整性）。
  • Classroom 部署建議：
    • 極限測試：要求學員紀錄在 1MHz、10MHz、40MHz 下的螢幕刷新 FPS 與畫面品質。
    • 截圖提交：上傳一張一張在高頻通訊下產生「雪花」或「圖像錯位」的照片，並在報告中分析原因（如：導線過長產生的寄生電容影響）。
    • 性能核核：導師點評其對 SPI_MODE (CPOL/CPHA) 配置的理解是否正確。

任務 3：協議翻譯官：Raw to JSON (Protocol Converter)

  • 目標：將「低階二進位流」解讀為「高階語義化數據」。
  • Classroom 部署建議：
    • 核心代碼檢核： 
      // 學生需實作：將 High/Low Bytes 拼接為 16-bit 整數
int16_t rawAccX = (buf[0] << 8) | buf[1];
float realAccX = rawAccX / 16384.0; // 依據敏感度量程換算
    • Autograding：透過串列埠測試框架，餵入一組 Mock 的 Hex 數據，檢查學員產出的 JSON 字串是否精確對應物理量（例如：加速度是否正確換算為 $m/s^2$）。

3. 硬體通訊導師點評標準 (Hardware Protocol Benchmarks)

此單元的價值在於 「對硬體通訊細節的掌控力」：

  • [ ] 位元運算正確性：是否理解大端 (Big-endian) 與小端 (Little-endian) 的順序？
  • [ ] 異常排查能力：README 中是否紀錄了當 I2C 鎖死 (Bus Hang) 時的「手動重放時鐘 (Clock Replay)」解決方案？
  • [ ] 數據解耦意識：輸出的 JSON 是否簡潔？是否將「硬體換算係數」與「應用層邏輯」進行了有效分離？
