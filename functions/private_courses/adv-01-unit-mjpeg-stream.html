MJPEG：每一幀都是主角

當你在瀏覽器上看到 ESP32-CAM 的即時影像時，你可能會以為它在播放「影片」——但事實上，你看到的是一連串獨立的 JPEG 圖片被快速輪播。這就是 MJPEG（Motion JPEG）的核心概念：它不是影片壓縮格式，而是一種「不斷替換靜態圖片」的傳輸策略。

MJPEG 與現代影片編碼的根本差異

現代影片編碼格式（如 H.264 / H.265）使用了極為複雜的幀間壓縮 (Inter-frame Compression) 技術。它們會分析相鄰幀之間的差異，只傳送「變化的部分」，從而達到極高的壓縮率。然而，這種演算法需要大量的運算資源——在 ESP32-S3 這種等級的 MCU 上，即使有硬體加速，即時編碼 H.264 仍然幾乎不可能。

MJPEG 則完全不同：每一幀都是獨立壓縮的 JPEG 圖片，幀與幀之間沒有任何依賴關係。這意味著編碼器不需要記住「上一幀長什麼樣子」，只需要專注於壓縮當前這一幀。對於運算資源有限的嵌入式裝置來說，這是一個非常務實的選擇。

特性 MJPEG H.264
壓縮方式 幀內壓縮 (Intra-frame) 幀間壓縮 (Inter-frame)
編碼運算量 低（適合 MCU） 極高（需要 DSP/GPU）
壓縮率（同等畫質） 較低（檔案大） 極高（檔案小 5～10 倍）
延遲 極低（無需等待參考幀） 較高（GOP 結構造成延遲）
隨機存取 每幀皆可獨立解碼 需從最近的 I-frame 開始解碼
音訊支援 通常無 完整支援
提示 MJPEG 的「低延遲」特性對即時控制類應用（如遙控車、無人機 FPV）非常重要。H.264 的 GOP（Group of Pictures）結構可能引入數百毫秒的額外延遲，而 MJPEG 的延遲理論上只有「一幀的擷取時間 + 壓縮時間 + 傳輸時間」。

JPEG 檔案的二進位結構

每一幀 MJPEG 影像本質上就是一個標準的 JPEG 檔案。JPEG 規範定義了明確的標記 (Marker) 來界定檔案的開始與結束，這是接收端辨識影像邊界的基礎：

  • SOI (Start of Image)：0xFF 0xD8 — 每個 JPEG 檔案的開頭兩個位元組，就像一本書的封面。
  • EOI (End of Image)：0xFF 0xD9 — 每個 JPEG 檔案的結尾兩個位元組，標記這張圖片的資料到此為止。

在 SOI 與 EOI 之間，還包含了多個功能區段：DQT（量化表）、DHT（霍夫曼表）、SOF（幀起始標記，含解析度資訊）以及壓縮後的掃描資料 (SOS)。理解這個結構，對於排查「影像損毀」或「綠屏」問題至關重要。

// JPEG 二進位結構示意（以 hex 表示）
FF D8 ← SOI: 影像開始
FF E0 xx xx ← APP0: JFIF 標頭 (含版本資訊)
FF DB xx xx ← DQT: 量化表 (控制壓縮品質)
FF C0 xx xx ← SOF0: 幀起始 (含寬度、高度)
FF C4 xx xx ← DHT: 霍夫曼編碼表
FF DA xx xx ← SOS: 掃描開始 (壓縮影像資料)
... 壓縮資料 ...
FF D9 ← EOI: 影像結束
警告 在壓縮資料區段中，0xFF 本身是合法的像素值。JPEG 規範使用位元組填充 (Byte Stuffing) 來避免混淆：當壓縮資料中出現 0xFF 時，會緊跟一個 0x00。因此，你不能單純地搜尋 0xFF 0xD9 來尋找 EOI——必須排除 0xFF 0x00 的填充序列。

為什麼 ESP32 選擇 MJPEG？

ESP32-S3 的硬體 JPEG 編碼器可以在不佔用 CPU 的情況下將攝影機擷取的原始 YUV/RGB 像素壓縮為 JPEG。這個硬體編碼器的效能足以在 VGA 解析度下達到 15～30 FPS 的即時壓縮速率。結合 MJPEG 協定簡單、不需要複雜的前後幀分析的特性，使它成為 ESP32 平台上最實用、最可靠的即時影像串流方案。

HTTP Boundary 的「隔板」原理

上一單元我們知道了 MJPEG 是「一連串 JPEG 圖片的快速輪播」。但問題來了：TCP 是一個連續的位元組流 (Byte Stream)，它不會告訴接收端「第一張圖從哪裡結束、第二張圖從哪裡開始」。HTTP Boundary 就是為了解決這個「封包邊界辨識」問題而設計的機制——它就像壽司輸送帶上的隔板，把每一盤壽司（每一幀影像）清楚地分開。

multipart/x-mixed-replace 協議

MJPEG over HTTP 的核心在於一個特殊的 Content-Type 標頭：

Content-Type: multipart/x-mixed-replace; boundary=frame_boundary

這個標頭告訴瀏覽器三件事：

  1. multipart：回應內容包含多個部分（多張圖片）。
  2. x-mixed-replace：每收到一個新的部分，就用它「替換 (replace)」前一個部分的顯示內容。
  3. boundary=frame_boundary：使用字串 frame_boundary 作為每個部分之間的分隔符。

瀏覽器收到這個標頭後，會保持 HTTP 連線不關閉，持續監聽後續傳入的資料。每當它偵測到 boundary 字串，就知道前一張圖已經傳完，會立刻渲染新收到的圖片。

注意 x-mixed-replace 是一個非標準但被廣泛支援的 MIME 類型，最早由 Netscape 在 1990 年代提出。現代主流瀏覽器（Chrome、Firefox、Edge）都原生支援它。Safari 在某些版本上有相容性問題，可能只顯示第一幀後就停止更新。

完整的幀封裝結構

每一幀 JPEG 影像在傳送前，都需要被「包裝」在嚴格的 HTTP 多部份格式中。以下是伺服器端每次推送一幀影像時，必須依序發送的完整位元組序列：

// 每一幀的完整 HTTP 封裝格式
--frame_boundary\r\n ← 分隔符 (注意前面的 --)
Content-Type: image/jpeg\r\n ← 這個部分是 JPEG 圖片
Content-Length: 28456\r\n ← 圖片大小 (位元組)
\r\n ← 空行：標頭與資料的分隔
[28456 bytes of JPEG data] ← FF D8 ... 壓縮資料 ... FF D9
\r\n ← 資料後的換行

這裡有幾個極易出錯的細節，值得特別標注：

常見錯誤 症狀 原因
Boundary 前少了 -- 瀏覽器只顯示第一幀後卡死 RFC 規定分隔符前必須加兩個連字號
使用 \n 而非 \r\n Chrome 正常但 Safari 綠屏 HTTP 規範強制使用 CRLF 換行
缺少空行（標頭與資料間） 影像完全無法解析 HTTP 用空行區分標頭與正文
Content-Length 數值錯誤 畫面破碎或後續幀錯位 長度必須精確匹配 JPEG 位元組數
警告 \r\n（CRLF，即 Carriage Return + Line Feed）與 \n（LF，僅 Line Feed）是完全不同的位元組序列。HTTP/1.1 規範明確要求使用 CRLF 作為行終止符。在 C/C++ 程式碼中，務必寫成 "\r\n" 而非僅 "\n"。這個「差一個位元組」的錯誤是 MJPEG 串流開發中最隱蔽、最難除錯的問題之一。

ESP32 端的實作要點

在 ESP32 的 HTTP 伺服器中實作 MJPEG 串流時，核心邏輯是在一個迴圈中不斷擷取新幀並推送給客戶端。以下程式碼片段展示了手動拼接 HTTP 多部份封包的關鍵步驟：

const char* BOUNDARY = "esp32_frame";

// 步驟 1：發送 HTTP 回應標頭 (僅在連線建立時發送一次)
client.print("HTTP/1.1 200 OK\r\n");
client.printf("Content-Type: multipart/x-mixed-replace; boundary=%s\r\n", BOUNDARY);
client.print("\r\n");

// 步驟 2：迴圈推送每一幀
while (client.connected()) {
camera_fb_t* fb = esp_camera_fb_get();
if (!fb) continue;

client.printf("--%s\r\n", BOUNDARY);
client.print("Content-Type: image/jpeg\r\n");
client.printf("Content-Length: %u\r\n\r\n", fb->len);
client.write(fb->buf, fb->len);
client.print("\r\n");

esp_camera_fb_return(fb);
}
提示 Boundary 字串的命名沒有強制規範，但建議使用不可能出現在 JPEG 二進位資料中的 ASCII 字串。像 "123frame" 這種簡短字串理論上沒問題，但業界常見做法是使用較長且含隨機字元的字串（如 "----esp32cam_frame_boundary"）以避免極端情況下的碰撞。

串流的生命週期

一個穩健的 MJPEG 串流服務，不僅要能「送」影像，還要能優雅地「停」，並且要正確處理「多個觀眾同時觀看」的情境。理解串流的完整生命週期——從連線建立到資源回收——是打造生產級影像服務的必備知識。

串流的三個階段

一次完整的 MJPEG 串流會經歷以下三個階段：

階段一：連線建立 (Handshake)

當使用者在瀏覽器輸入 http://192.168.4.1/stream 時，瀏覽器會對 ESP32 的 HTTP 伺服器發起 TCP 連線，並發送一個標準的 HTTP GET 請求。ESP32 端的路由處理器匹配到 /stream 端點後，回應一個特殊的 HTTP 標頭（即 multipart/x-mixed-replace），告訴瀏覽器：「準備好接收一個不會結束的影像串流」。

階段二：持續推送 (Streaming Loop)

連線建立後，ESP32 進入一個無窮迴圈：擷取攝影機幀 → JPEG 壓縮 → 封裝 HTTP 多部份格式 → 透過 TCP 傳送 → 釋放幀緩衝區 → 重複。這個迴圈會持續運作，直到客戶端斷線或發生錯誤。

階段三：連線終止 (Teardown)

當使用者關閉瀏覽器分頁或網路斷線時，TCP 連線會被終止。ESP32 必須偵測到這個事件並正確清理資源。

注意 ESP32 端偵測客戶端斷線的方式有兩種：(1) 呼叫 client.connected() 返回 false；(2) 呼叫 client.write() 時返回錯誤（寫入失敗）。在串流迴圈中，每次發送前都應檢查連線狀態，否則 ESP32 會持續向一個已死的 socket 發送資料，浪費 CPU 與記憶體資源。

連線終止的偵測機制

在 TCP 層面，連線終止有兩種方式，ESP32 的處理策略各有不同：

終止方式 觸發情境 ESP32 端的表現
TCP FIN（正常關閉） 使用者關閉分頁、點擊停止 client.connected() 返回 false
TCP RST（異常重置） 網路斷線、瀏覽器崩潰 client.write() 返回 -1 或觸發錯誤
// 健壯的串流迴圈：正確處理連線終止
void handleStream(WiFiClient& client) {
sendStreamHeader(client);

while (client.connected()) {
camera_fb_t* fb = esp_camera_fb_get();
if (!fb) {
ESP_LOGE(TAG, "攝影機擷取失敗");
break;
}

int sent = sendFrame(client, fb);
esp_camera_fb_return(fb);

if (sent < 0) {
ESP_LOGW(TAG, "客戶端已斷線，結束串流");
break;
}
}

client.stop();
ESP_LOGI(TAG, "串流結束，資源已回收");
}

多客戶端與資源競爭

在實際部署中，可能有多個使用者同時觀看串流。ESP32 的 HTTP 伺服器需要為每個客戶端維護一個獨立的 TCP 連線，每個連線都會佔用一個 FreeRTOS 任務堆疊（通常 4～8 KB）。考慮到 ESP32-S3 有限的記憶體，建議限制同時連線數為 2～4 個。

另一個常見的陷阱是幀緩衝區的競爭。ESP-IDF 的攝影機驅動通常只配置 2～3 個幀緩衝區（frame buffer）。如果第一個客戶端正在讀取某個緩衝區時，第二個客戶端也嘗試讀取同一個緩衝區，就會產生資料競爭 (Race Condition)。解決方案是使用 FreeRTOS 信號量 (Semaphore) 或雙緩衝 (Double Buffering) 策略。

警告 如果串流迴圈中忘記呼叫 esp_camera_fb_return(fb) 歸還幀緩衝區，緩衝區很快就會被耗盡，導致 esp_camera_fb_get() 永遠返回 NULL。這是一種典型的記憶體洩漏 (Memory Leak)，而且症狀（攝影機「凍結」）很容易被誤診為硬體故障。養成「取得後必歸還」的習慣至關重要。
提示 要觀察 MJPEG 串流的原始封包結構，最簡單的方法是用瀏覽器的 F12 開發者工具開啟 Network 面板，找到串流請求後查看 Response 的 Raw Data。你會清楚看到 boundary 字串、Content-Type 標頭與每一幀 JPEG 的二進位資料交錯排列。

MJPEG 編碼與 HTTP Boundary 原理 (adv-01-unit-mjpeg-stream)

課程目標

本單元的目標是讓學員從「函式庫使用者」進化為「協議理解者」。你將學習如何不依賴高階影像串流庫，手動透過 TCP Socket 實作符合 HTTP 標準的 MJPEG 影像串流。

任務說明

任務 1：封包解剖師 (Packet Analysis Lab)

  • 目標：透過觀察真實流量，理解 MJPEG 在 HTTP 層級的運作機制。
  • 要求：
    1. 使用瀏覽器開發者工具 (F12) 或 Wireshark 擷取 HTTP 串流。
    2. 找出 Content-Type: multipart/x-mixed-replace; boundary=... 這一行。
    3. 在 docs/protocol_report.md 中解釋 boundary 的具體作用。
    4. 找出單張 JPEG 的 SOI (0xFFD8) 與 EOI (0xFFD9) 在二進位流中的位置並紀錄。

任務 2：手動拼接 MJPEG 數據流 (Manual Streamer Implementation)
