MJPEG 編碼與 HTTP Boundary 原理

🎞️ 1. MJPEG：每一幀都是主角 (20 min)

編碼邏輯：MJPEG (Motion JPEG) 並非真正的影片壓縮格式（如 H.264/H.265），它其實只是將一連串的 JPEG 圖片快速播放。

✅ 優勢

對 ESP32-S3 運算壓力極小。它不需要計算「前後幀的差異 (Inter-frame compression)」，只需專心把每一幀靜態圖片壓縮好。

❌ 缺點

檔案大、佔頻寬，且通常不包含聲音軌道。

🔍 學習任務

在 hex 編輯器中觀察一個 JPEG 檔案，尋找它的「身份證明」：

  • 開頭標誌 (SOI)：0xFFD8
  • 結尾標誌 (EOI)：0xFFD9

🚧 2. HTTP Boundary 的「隔板」原理 (25 min)

Content-Type: multipart/x-mixed-replace; boundary=12345

原理

瀏覽器收到這個特殊的 Header 後，會知道這不是一個普通的網頁，而是一個「永遠不會結束」的傳輸。它會保持連線開啟，等待後續的數據。

Boundary 標籤

伺服器在每張圖片之間插入一個自定義的字串（如 --12345）。瀏覽器一看到這個標籤，就知道上一張圖傳完了，會立刻「清空」畫面並準備「換上」新收到的圖片。

🧠 思考任務

如果沒有 Boundary，瀏覽器如何區分第一張圖的結尾與第二張圖的開始？

提示：TCP 串流本身是連續的位元組流 (Byte Stream)，沒有天生的邊界。

🔄 3. 串流的生命週期 (15 min)

一個健康的串流服務，不僅要能「送」，還要在適當的時候「停」。

🛑 停止傳送的時機

當你關閉網頁視窗時，ESP32 端的伺服器是如何得知「停止傳送影像」的？

  • TCP Fin/Rst：瀏覽器主動斷開 TCP 連線。
  • 伺服器偵測：在發送下一幀數據時，檢查 client.connected() 或寫入錯誤，發現客戶端已離線。

🌐 線上任務

在電腦上觀察 MJPEG 串流的原始碼 (View Source)，你能看到每張圖片之間的「邊界符號 (Boundary)」嗎？

提示：通常瀏覽器會隱藏這些細節，你需要用 F12 Network 工具的 Response 分頁來查看 Raw Data。

進階課程 1 - 單元 2

MJPEG 編碼與 HTTP Boundary 原理 (adv-01-unit-mjpeg-stream)

課程目標

本單元的目標是讓學員從「函式庫使用者」進化為「協議理解者」。你將學習如何不依賴高階影像串流庫，手動透過 TCP Socket 實作符合 HTTP 標準的 MJPEG 影像串流。

任務說明

任務 1：封包解剖師 (Packet Analysis Lab)

  • 目標：透過觀察真實流量，理解 MJPEG 在 HTTP 層級的運作機制。
  • 要求：
    1. 使用瀏覽器開發者工具 (F12) 或 Wireshark 擷取 HTTP 串流。
    2. 找出 Content-Type: multipart/x-mixed-replace; boundary=... 這一行。
    3. 在 docs/protocol_report.md 中解釋 boundary 的具體作用。
    4. 找出單張 JPEG 的 SOI (0xFFD8) 與 EOI (0xFFD9) 在二進位流中的位置並紀錄。

任務 2：手動拼接 MJPEG 數據流 (Manual Streamer Implementation)

  • 目標：不依賴第三方庫，手動透過 WiFiClient 物件推送影像流。
  • 要求：
    1. 在 src/mjpeg_driver.cpp (或指定檔案) 中實作 HTTP 多部份 (Multipart) 數據包的發送邏輯。
    2. 確保正確使用 CRLF (\r\n) 換行符。
    3. 正確封裝每張影像的標頭 (Boundary, Content-Type, Content-Length)。

任務 3：串流穩定性與 MTU 優化 (Stream Stability Lab)

  • 目標：解決高解析度下的影像撕裂 (Tearing) 問題。
  • 要求：
    1. 在 VGA 解析度下測試傳輸穩定性。
    2. 紀錄調整傳輸參數前後的流暢度差異。
    3. 將測試數據紀錄於 test/stability_test.csv 中。

提交要求

  1. 完成 src/ 中的程式碼實作。
  2. 完成 docs/protocol_report.md 的分析報告。
  3. 確保你的程式庫程式碼乾淨且包含必要的註解。
  4. 將所有更改提交並推送到 GitHub 程式庫。

評分標準

  • 協議嚴謹度：HTTP 標頭與邊界封裝是否完全符合規範？
  • 故障排除能力：是否能正確診斷串流失敗的原因？
  • 穩定性：影像串流是否平滑且無撕裂現象？

任務 1：封包解剖師 (Packet Analysis Lab)

任務 2：手動拼接 MJPEG 數據流 (Manual Streamer Implementation)

  • 目標：不依賴第三方庫，手動透過 TCP Socket 推送影像流。
  • Classroom 部署建議：
    • 核心代碼檢核： 
      // 學生需實作：手動傳送邊界與長度標頭

    • 成果展示：上傳一段錄影或截圖，對比「優化前（影像腰斬/撕裂）」與「優化後（完整幀平滑切換）」的視覺效果。
