MJPEG 編碼與 HTTP Boundary 原理

🎞️ 1. MJPEG：每一幀都是主角 (20 min)

編碼邏輯：MJPEG (Motion JPEG) 並非真正的影片壓縮格式（如 H.264/H.265），它其實只是將一連串的 JPEG 圖片快速播放。

✅ 優勢

對 ESP32-S3 運算壓力極小。它不需要計算「前後幀的差異 (Inter-frame compression)」，只需專心把每一幀靜態圖片壓縮好。

❌ 缺點

檔案大、佔頻寬，且通常不包含聲音軌道。

🔍 學習任務

在 hex 編輯器中觀察一個 JPEG 檔案，尋找它的「身份證明」：

  • 開頭標誌 (SOI)：0xFFD8
  • 結尾標誌 (EOI)：0xFFD9

🚧 2. HTTP Boundary 的「隔板」原理 (25 min)

Content-Type: multipart/x-mixed-replace; boundary=12345

原理

瀏覽器收到這個特殊的 Header 後，會知道這不是一個普通的網頁，而是一個「永遠不會結束」的傳輸。它會保持連線開啟，等待後續的數據。

Boundary 標籤

伺服器在每張圖片之間插入一個自定義的字串（如 --12345）。瀏覽器一看到這個標籤，就知道上一張圖傳完了，會立刻「清空」畫面並準備「換上」新收到的圖片。

🧠 思考任務

如果沒有 Boundary，瀏覽器如何區分第一張圖的結尾與第二張圖的開始？

提示：TCP 串流本身是連續的位元組流 (Byte Stream)，沒有天生的邊界。

🔄 3. 串流的生命週期 (15 min)

一個健康的串流服務，不僅要能「送」，還要在適當的時候「停」。

🛑 停止傳送的時機

當你關閉網頁視窗時，ESP32 端的伺服器是如何得知「停止傳送影像」的？

  • TCP Fin/Rst：瀏覽器主動斷開 TCP 連線。
  • 伺服器偵測：在發送下一幀數據時，檢查 client.connected() 或寫入錯誤，發現客戶端已離線。

🌐 線上任務

在電腦上觀察 MJPEG 串流的原始碼 (View Source)，你能看到每張圖片之間的「邊界符號 (Boundary)」嗎？

提示：通常瀏覽器會隱藏這些細節，你需要用 F12 Network 工具的 Response 分頁來查看 Raw Data。

GitHub Classroom 詳細操作指南

本單元教學重點：讓學生從「調庫俠」進化為「協議理解者」。掌握 HTTP 1.1 的持久連線 (Keep-Alive) 脈絡。

  1. 重點 1：無聲的對話 - 透過 2-1 任務，讓學生看見 Header 是如何指揮瀏覽器渲染行為的。
  2. 重點 2：位元級精確度 - MJPEG 拼接中，哪怕少一個 \n 都會導致串流崩潰。鍛鍊學生的嚴謹度。
  3. 重點 3：底層性能調校 - 影像撕裂不只是 FPS 問題，更是網路層的隊列問題。引導學生思考 TCP 窗口與緩衝。
