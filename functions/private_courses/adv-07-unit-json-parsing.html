💎 1. 為什麼選擇 JSON？ (20 min)

JSON 是跨平台通用的純文字格式。無論是 ESP32 (C++) 還是網頁 (JS)，都能透過結構化標籤精準交換數據，免除逗號分隔字串的易錯性。

Protocol Benefits

  • 易讀性：Key-Value 結構一目了然。
  • 擴展性：增加感測器欄位不影響舊有解析代碼。

Data Schema Payload

"{
\"v_bat\": 12.6,
\"sensors\": [ 102, 98 ]
}"
Byte Stream String

🎨 2. 物件解構 (Destructuring) 的藝術 (25 min)

利用 ES6 解構語法，只需一行代碼即可從資料包中提取數值。配合「預設值 (Defaults)」與「巢狀提取」，打造強健且簡潔的數據處理鏈。

Modern Syntax Pattern

const { rpm = 0, bat } = data;

Nested Object Mapping

const data = { motor: { left: 100 } };
const { motor: { left: speed } } = data;
console.log(speed); // 100

Deep Property Extraction

🔄 3. 序列化與反序列化 (15 min)

傳輸媒介只能傳送「位元組流」。stringify 將物件拆解為木屑 (字串) 塞入水管，而 parse 則在出口將其重組回椅子 (物件)。

Data Integrity Lifecycle

  • JSON.stringify()：將物件轉化為標準 JSON 字串準備發送至 BLE。
  • JSON.parse()：將接收到的原始封包字串還原為 JS 可操作的物件。
  • Exception Safety：使用 try-catch 攔截格式錯誤，防止解析失敗導致網頁崩潰。

單元課程 進階 7 - JSON 解析

ADV-07：JSON 解析與物件解構

課程單元：adv-07-unit-json-parsing 難度等級：進階 ⚡

📖 作業說明

在自駕車系統中，Web 介面與 ESP32 控制器之間透過 JSON 交換數據。本次作業的目標是讓你學會如何安全解析 JSON 封包、優雅處理格式錯誤的資料，以及用 ES6 進階語法高效提取巢狀物件中的數值。

🎯 學習目標

  • 使用 try-catch 實作防禦性解析，避免程式因壞封包而崩潰
  • 掌握 ES6 解構賦值 (Destructuring) 與預設值 (Default Values) 語法
  • 理解 JSON Payload 大小對藍牙 MTU 頻寬的影響，並進行精簡優化

✅ 作業任務

Task 1：封包解碼器 (js/decoder_app.js)

開啟 js/decoder_app.js，找到 TODO 標記，完成以下實作：

  1. 使用 JSON.parse() 解析傳入的原始字串 rawString
  2. 用 try-catch 包覆解析邏輯，捕捉格式錯誤的封包
  3. 解構提取 v_bat 與 rpm 欄位並更新 UI
  4. 若解析失敗，保留上一次成功的數值（不得顯示 undefined）

驗收標準：系統在收到「半截字串」或「缺少欄位」的封包時，看板仍能穩定顯示最後一次正確值。

Task 2：配置管理器 (js/config_manager.js)

開啟 js/config_manager.js，找到 TODO 標記，完成以下實作：

  1. 使用巢狀解構語法提取 PID 參數（具備雙重預設值保障）： 
    const { pid: { p = 1.0, i = 0, d = 0 } = {} } = config;
  2. 在 UI 調整參數後，將物件序列化為符合協議的 JSON 字串
  3. 確保任何欄位缺失時，系統套用預設值而非回傳 undefined

驗收標準：刪除 JSON 中任一非必填欄位後，系統仍能正確顯示預設值。

Task 3：協議精簡文件 (docs/protocol_spec.md)

在 docs/protocol_spec.md 中完成以下內容：

  1. 設計一套縮寫版 JSON Key 對照表（例如：battery_voltage → bv）
  2. 記錄優化前後的 Blob.size 數據（可使用 js/size_calculator.js 測量）
  3. 說明你在「可讀性 vs 效能」之間的取捨理由

驗收標準：對照表清晰完整，並附上量化的 Byte 節省數據。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template Repo) 配置建議

理解數據交換需要「破壞性測試」與「語法規範」，建議範本包含：

  • 📂 js/packet_decoder.js：封包解碼器。提供一個模擬數據流產生器，故意夾雜「格式錯誤」或「欄位缺失」的 JSON 字串，留下 decode() 與 try-catch 的 TODO。
  • 📂 js/config_handler.js：配置管理器。預置多層次物件架構，留下 ES6 解構語法 (Destructuring) 與「預設值 (Safe Fallbacks)」的實作空間。
  • 📂 docs/protocol_spec.md：協議規範文件。要求學員在此定義新設計的「精簡版 JSON 協議」映射表。
  • 📂 tests/json_robustness.test.js：自動化評分脚本。專門測試邊界情況（如：空字串、巢狀深度過大），檢核學員的代碼是否具備足夠的健壯性 (Robustness)。

2. 作業任務部署細節

任務 1：「封包拆解員」：動態資訊看板 (JSON Packet Decoder Lab)

  • 目標：學會防禦性編程 (Defensive Programming)，確保通訊異常時系統不崩潰。
  • Classroom 部署建議：
    • 健壯解析檢核： 
      // 學生需實作：try-catch 與解構提取
try {
    const { v_bat, rpm } = JSON.parse(rawString);
    updateUI(v_bat, rpm);
} catch (e) {
    console.error("Packet Corrupted"); // 防止網頁報錯停止運行
}
    • Autograding：餵入一組包含「半截字串」的樣本，檢核學員的程式是否能跳過錯誤封包並維持 UI 顯示最後一次的正確值。
    • 視覺驗證：要求提交看板成功顯示 v_bat 與 rpm 的即時刷新截圖。

任務 2：多層次配置管理器 (Nested Config Manager)

  • 目標：掌握 ES6 進階語法，寫出具備「專業感」且強健的數據提取代碼。
  • Classroom 部署建議：
    • 巢狀提取檢核：要求學員實作 const { pid: { p = 1.0, i = 0 } = {} } = config; 這種具備「雙重保險」的解構寫法。
    • 序列化測試：學員需在 UI 調整參數後，生成一個符合協議的 JSON 字串。
    • Autograding：模擬刪除 JSON 中某個非必填欄位，檢核系統是否能正確套用「預設值」而非顯示 undefined。

任務 3：數據壓縮挑戰：精簡 JSON Payload (Payload Slimming Challenge)

  • 目標：在高頻電感通訊中，透過縮減字節提升系統反應速度。
  • Classroom 部署建議：
    • 優化指標：要求學員將 Key 名稱縮寫（如 battery_voltage → bv），並將浮點數四捨五入。
    • 量化分析：要求在 README 中列出優化前後的 Blob.size 數據。
    • PR 深度核核：點評其對「可讀性 vs 效能」的權衡思維。是否在 docs/protocol_spec.md 中留下了清晰的縮寫對應表？

3. 通訊協議導師點評標準 (Data Protocol Benchmarks)

此單元的價值在於 「數據契約 (Data Contract) 的嚴謹度」：

  • [ ] 容錯能力：代碼是否能優雅地處理「髒數據」而不拋出未捕獲的異常？
  • [ ] 語法現代化程度：是否拋棄了舊式的 obj.a.b.c 寫法，改用更簡潔、具備預設值保障的解構語法？
  • [ ] 頻寬意識：是否理解 JSON Key 的長度會直接佔用藍牙 MTU 空間？（這是判斷是否具備嵌入式開發思維的關鍵）。
