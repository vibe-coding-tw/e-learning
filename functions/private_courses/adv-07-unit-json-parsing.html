💎 1. 為什麼選擇 JSON？ (20 min)

JSON 是跨平台通用的純文字格式。無論是 ESP32 (C++) 還是網頁 (JS)，都能透過結構化標籤精準交換數據，免除逗號分隔字串的易錯性。

Protocol Benefits

  • 易讀性：Key-Value 結構一目了然。
  • 擴展性：增加感測器欄位不影響舊有解析代碼。

Data Schema Payload

"{
\"v_bat\": 12.6,
\"sensors\": [ 102, 98 ]
}"
Byte Stream String

🎨 2. 物件解構 (Destructuring) 的藝術 (25 min)

利用 ES6 解構語法，只需一行代碼即可從資料包中提取數值。配合「預設值 (Defaults)」與「巢狀提取」，打造強健且簡潔的數據處理鏈。

Modern Syntax Pattern

const { rpm = 0, bat } = data;

Nested Object Mapping

const data = { motor: { left: 100 } };
const { motor: { left: speed } } = data;
console.log(speed); // 100

Deep Property Extraction

🔄 3. 序列化與反序列化 (15 min)

傳輸媒介只能傳送「位元組流」。stringify 將物件拆解為木屑 (字串) 塞入水管，而 parse 則在出口將其重組回椅子 (物件)。

Data Integrity Lifecycle

  • JSON.stringify()：將物件轉化為標準 JSON 字串準備發送至 BLE。
  • JSON.parse()：將接收到的原始封包字串還原為 JS 可操作的物件。
  • Exception Safety：使用 try-catch 攔截格式錯誤，防止解析失敗導致網頁崩潰。

數據實驗室：結構化協議與容錯性處理

任務 1：封包拆解員 (JSON Packet Decoder Lab)

角色扮演：數據通訊工程師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 數據源連接：訂閱來自傳感器的模擬數據流（字串格式）。
  3. 健壯解析實作：撰寫 decodePayload(raw) 函式。使用 try-catch 攔截 JSON 語法錯誤。
  4. 看板渲染：利用 ES6 解構提取 { speed, temp }，並動態更新至看板 HTML。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 崩潰測試：當手動傳入破壞性的惡意字串（如 "abc"）時，網頁是否能正常報錯而非卡死？
  • [ ] 解構語法成熟度：代碼中是否正確運用了 ES6 解構而非傳統的 data.x.y.z 鏈式寫法？

任務 2：配置管理器 (Nested Config Manager)

角色扮演：系統架構師 | 難度：⭐️⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 深層協議設計：建立一個巢狀配置物件 { pid: { p: 1.2, i: 0 }, ... }。
  2. 安全提取邏輯：在解構時賦予 Default Values，如 const { kp = 1.0 } = data.pid;。
  3. 即時序列化：當使用者調整滑桿後，自動調用 JSON.stringify 並顯示輸出的字串。
  4. Push Final：提交具備「配置一致性檢查」能力的管理器。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 預設值健壯性：當刪除 JSON 中某個非必填欄位時，系統顯示的預設數據是否正確？
  • [ ] 協議擴展力：若要增加一個「馬達電平」欄位，學員是否能在不破壞舊有解析架構下完成？

任務 3：數據壓縮挑戰 (Payload Slimming Challenge)

角色扮演：帶寬優化專家 | 難度：⭐️⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 指標建立：計算初始 JSON 封包的 Byte 長度。
  2. 重構 Key 命名：將冗長的 battery_voltage 替換為 bv。
  3. 數據截斷：將 3.1415926 處理為 3.14。
  4. 對比數據提交：在 README 中列出優化前後的位元組減少比例。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 壓縮效率：封包大小是否至少減少了 30% 以上？
  • [ ] 可讀性權衡：縮寫後的 Key 是否依然具備足夠的語意辨識度（或有完整的 Documentation 對應表）？

GitHub Classroom 詳細操作指南

本單元教學重點：讓學生掌握「數據契約 (Data Contract)」的概念。這是專業通訊與前後端分離的基礎。

  1. 重點 1：安全性優先 - 學生往往會直接 JSON.parse 數據。必須強迫他們使用 try-catch，因為通訊過程中的亂碼是常態。
  2. 重點 2：寫得少做得多 - 重點展示解構語法 (Destructuring) 的美學。讓程式碼更具備「聲明式 (Declarative)」的質感。
  3. 重點 3：頻寬意識 - 尤其在藍牙 BLE 這種窄頻通訊中，Key 名稱的長度直接影響發送頻率。
