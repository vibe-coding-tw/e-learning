簡介

本單元聚焦於 JSON 封包的解析技巧與 JavaScript 物件解構語法。物聯網裝置傳回的資料通常以 JSON 格式封裝多層巢狀結構，你需要快速且安全地取出目標欄位。掌握解構賦值、可選鏈與預設值等技巧，能讓你的程式碼更簡潔、更具防禦性，同時降低因欄位缺失而導致執行錯誤的風險。

情境
你的氣象站裝置每 5 秒傳回一筆 JSON 封包，內含溫度、濕度、氣壓、風速等感測值，且部分欄位可能因硬體異常而缺失。前端需要在毫秒級時間內安全地提取這些數據並更新 UI。如果直接存取未定義的巢狀屬性，頁面就會因 TypeError 而崩潰。如何優雅地處理這些不確定結構？本單元將教你實用的解構策略。

為什麼選擇 JSON？

在物聯網 (IoT) 開發中，ESP32 控制器與網頁前端之間需要一個共通的資料交換格式。常見選項包括：純文字逗號分隔 (CSV)、自訂二進位封包 (Custom Binary)、以及 JSON (JavaScript Object Notation)。三者各有優劣，但 JSON 已成為 Web 領域的事實標準，其原因值得深入了解。

資料格式比較

當 ESP32 透過藍牙 (BLE) 傳送感測器數值給網頁時，最直覺的做法是用逗號分隔："12.6,1024,98"。這看起來簡潔，但一旦欄位順序改變或新增感測器，接收端就必須同步修改解析邏輯。自訂二進位協議雖然位元組效率最高，但除錯困難且兩端都需要手動位元操作。JSON 則以 Key-Value 結構在可讀性、擴展性與跨平台相容性之間取得了最佳平衡。

特性 JSON CSV 自訂二進位
可讀性 Key-Value 一目了然 僅靠欄位順序辨識 需對照規格書
擴展性 新增欄位不破壞舊解析 新增欄位須同步修改索引 需重新定義位元偏移
跨平台支援 C++/JS/Python 原生支援 需自行 split 解析 需自行實作編解碼器
位元組效率 中等（Key 佔用空間） 較佳（無 Key 開銷） 最佳
除錯友善度 瀏覽器 Console 直接檢視 需手動比對欄位 需 HEX 查看器

ESP32 端的 JSON 輸出範例

以下是 ESP32 透過 BLE 傳送給網頁的典型 JSON 封包。每個欄位都有明確的 Key 名稱，接收端可以依照 Key 提取數值，而非依賴欄位在字串中的位置。

// ESP32 (C++) — 使用 ArduinoJson 產生封包
StaticJsonDocument<200> doc;
doc["v_bat"] = 12.6;
doc["rpm"] = 1024;
doc["sensors"][0] = 102;
doc["sensors"][1] = 98;

String output;
serializeJson(doc, output);
// output → {"v_bat":12.6,"rpm":1024,"sensors":[102,98]}

同樣的資料用 CSV 傳送

如果改用 CSV 格式，相同的資料會被壓縮為一行字串。看起來更短，但問題在於：接收端必須記住每個欄位的順序。一旦 ESP32 端新增一個 temp 欄位插入中間，所有索引都會錯位，導致解析邏輯全面崩潰。

// CSV 格式 — 簡潔但脆弱
"12.6,1024,102,98"

// 接收端解析：依賴欄位索引
const parts = raw.split(",");
const v_bat = parseFloat(parts[0]); // 萬一順序變了呢？
const rpm = parseInt(parts[1]); // 新增欄位就全部錯位
備註 JSON 是純文字格式，你可以直接在瀏覽器的 Console 或 BLE 除錯工具中閱讀封包內容。這在開發階段大幅縮短了除錯時間——你不需要對照偏移量表來理解每個位元組的含義。
提示 在 ESP32 (C/C++) 端，推薦使用 ArduinoJson 函式庫來組裝與解析 JSON。它提供了 StaticJsonDocument（堆疊配置，適合小封包）與 DynamicJsonDocument（堆積配置，適合大封包）兩種模式，並能自動處理記憶體管理與字串逸出 (Escaping)。

物件解構 (Destructuring) 的藝術

ES6 引入的解構賦值 (Destructuring Assignment) 是處理 JSON 資料的利器。傳統寫法需要逐行用 obj.key 提取數值，而解構語法讓你在一行之內就能同時提取多個欄位、重新命名變數、並設定預設值。對於頻繁處理 IoT 封包的前端程式來說，這不只是語法糖——它是寫出簡潔且防禦性代碼的關鍵技巧。

基本解構：從封包提取欄位

假設從 ESP32 接收到一筆 JSON 封包，已經透過 JSON.parse() 還原為 JavaScript 物件。最基本的解構寫法如下：

// 傳統寫法
const v_bat = data.v_bat;
const rpm = data.rpm;

// ES6 解構寫法 — 一行搞定
const { v_bat, rpm } = data;

兩種寫法的結果完全相同，但解構語法在欄位數量增加時優勢更加明顯——你不需要為每個欄位寫一行 const x = obj.x。

預設值：防禦欄位缺失

IoT 裝置傳回的封包不一定每次都包含所有欄位。硬體異常、韌體版本差異、或傳輸中斷都可能導致某些 Key 消失。解構語法允許你為每個欄位指定預設值，當該欄位為 undefined 時自動套用：

const { v_bat = 0, rpm = 0, temp = 25.0 } = data;

// 若 data = { v_bat: 12.6 }（缺少 rpm 和 temp）
// 結果：v_bat = 12.6, rpm = 0, temp = 25.0

巢狀解構：深入多層物件

真實的 IoT 封包經常包含巢狀結構。例如馬達控制資料可能是 { motor: { left: 100, right: 80 } }。巢狀解構讓你一步到位地提取深層屬性，甚至可以在提取的同時重新命名變數：

const packet = {
  motor: { left: 100, right: 80 },
  pid: { p: 1.2, i: 0.01, d: 0.5 }
};

// 巢狀解構 + 變數重新命名
const {
  motor: { left: leftSpeed, right: rightSpeed },
  pid: { p: kp, i: ki, d: kd }
} = packet;

console.log(leftSpeed); // 100
console.log(kp); // 1.2
警告 當巢狀的父層物件不存在時，解構會拋出 TypeError: Cannot destructure property 'left' of undefined。例如，若封包缺少整個 motor 欄位，直接寫 const { motor: { left } } = packet 會導致程式崩潰。你必須為父層提供預設值或使用可選鏈來防禦。

可選鏈 (Optional Chaining) 作為替代防線

ES2020 引入的可選鏈運算子 ?. 是另一種防禦策略。當你不確定物件的某一層是否存在時，可選鏈會在遇到 null 或 undefined 時短路回傳 undefined，而不會拋出錯誤：

// 可選鏈 — 安全存取深層屬性
const leftSpeed = packet?.motor?.left ?? 0;
const kp = packet?.pid?.p ?? 1.0;

// 即使 packet.motor 是 undefined，也不會拋出 TypeError
// ?? 是「空值合併運算子」，僅在左側為 null/undefined 時套用右側預設值

雙重保險模式：解構 + 預設物件

在實務開發中，最穩健的做法是結合巢狀解構與父層預設值，形成所謂的「雙重保險」模式。這確保即使整個子物件都不存在，解構仍然可以順利完成並套用預設值：

// 雙重保險：父層預設 {} + 子層預設值
const {
  pid: { p = 1.0, i = 0, d = 0 } = {}
} = config;

// 情境 A：config = { pid: { p: 2.5 } }
// → p = 2.5, i = 0, d = 0 (i, d 套用子層預設值)

// 情境 B：config = {} (完全沒有 pid 欄位)
// → p = 1.0, i = 0, d = 0 (父層 = {} 防止 TypeError，再套用子層預設值)

// 情境 C：config = { pid: { p: 0.8, i: 0.02, d: 0.3 } }
// → p = 0.8, i = 0.02, d = 0.3 (全部使用封包實際值)
提示 = {} 是關鍵所在。它告訴 JavaScript：「如果 config.pid 是 undefined，就把它當成空物件 {}，然後繼續對空物件做解構並套用預設值。」沒有這個 = {}，當 pid 不存在時就會直接拋出 TypeError。建議在所有處理 IoT 封包的解構語句中都採用此模式。

序列化與反序列化

BLE（藍牙低功耗）等傳輸媒介只能傳送位元組流（byte stream）——它不認得 JavaScript 物件，也不認得陣列或數字。因此，在發送資料前你必須將物件「打平」成一串純文字字串，這個過程稱為序列化 (Serialization)。而接收端收到字串後，需要將其「還原」為可操作的 JavaScript 物件，這就是反序列化 (Deserialization)。

在 JavaScript 中，這兩個操作分別由 JSON.stringify() 和 JSON.parse() 完成。它們是 JSON 資料交換的核心雙子星，理解其用法與陷阱是穩健開發的基礎。

JSON.stringify()：物件 → 字串

JSON.stringify() 接受一個 JavaScript 物件，回傳對應的 JSON 字串。它還提供了兩個可選參數：replacer（過濾函式或陣列）和 space（縮排格式化），方便你在除錯時美化輸出。

const packet = {
  v_bat: 12.6,
  rpm: 1024,
  sensors: [102, 98]
};

// 基本序列化 — 產生最精簡的字串
const raw = JSON.stringify(packet);
// → '{"v_bat":12.6,"rpm":1024,"sensors":[102,98]}'

// 美化輸出 — 第三個參數指定縮排空格數
console.log(JSON.stringify(packet, null, 2));
// {
// "v_bat": 12.6,
// "rpm": 1024,
// "sensors": [102, 98]
// }

// 使用 replacer 過濾欄位 — 只保留指定的 Key
const slim = JSON.stringify(packet, ["v_bat", "rpm"]);
// → '{"v_bat":12.6,"rpm":1024}'

JSON.parse()：字串 → 物件

接收端從 BLE 拿到的是一串原始字串。JSON.parse() 負責將其還原為 JavaScript 物件，讓你可以用 . 或解構語法存取欄位。但這裡有一個致命陷阱：如果字串不是合法的 JSON 格式，JSON.parse() 會直接拋出 SyntaxError，導致後續程式碼完全停止執行。

// 防禦性解析 — 永遠用 try-catch 包覆 JSON.parse()
function safeParsePacket(rawString) {
  try {
    const data = JSON.parse(rawString);
    const { v_bat = 0, rpm = 0 } = data;
    return { v_bat, rpm, ok: true };
  } catch (err) {
    console.error("封包解析失敗:", err.message);
    return { v_bat: null, rpm: null, ok: false };
  }
}

// 使用範例
const result = safeParsePacket('{"v_bat":12.6,"rpm":1024}');
if (result.ok) updateUI(result.v_bat, result.rpm);
重要 在 BLE 通訊中，try-catch 不是可選的——它是必要的。藍牙封包可能因訊號干擾而被截斷（例如只收到 {"v_bat":12），也可能因 MTU 限制而被拆分成多段。任何一次未被捕捉的 SyntaxError 都會中斷整個事件處理迴圈，導致網頁介面完全凍結。

常見 JSON.parse() 錯誤對照表

錯誤輸入範例 錯誤訊息 原因說明
{"v_bat":12.6, Unexpected end of JSON input BLE 傳輸中斷，封包被截斷
{v_bat: 12.6} Expected property name or '}' Key 未加雙引號（非合法 JSON）
{'v_bat': 12.6} Expected property name or '}' 使用單引號（JSON 僅允許雙引號）
""（空字串） Unexpected end of JSON input BLE 回傳空資料
undefined Unexpected token u in JSON 變數未初始化即傳入 parse

測量 Payload 大小：BLE MTU 考量

BLE 4.2 的預設 MTU（最大傳輸單元）僅有 23 bytes（扣除 ATT 標頭後實際可用約 20 bytes）。雖然 BLE 5.0 支援協商至 251 bytes，但在設計 JSON 封包時仍應盡量精簡——更短的封包意味著更快的傳輸與更低的功耗。

// 使用 Blob 測量 JSON 字串的實際位元組大小
const payload = JSON.stringify({ v_bat: 12.6, rpm: 1024 });
const byteSize = new Blob([payload]).size;

console.log(payload); // '{"v_bat":12.6,"rpm":1024}'
console.log(byteSize); // 26 (bytes)

// 精簡版：縮短 Key 名稱
const slim = JSON.stringify({ vb: 12.6, r: 1024 });
console.log(new Blob([slim]).size); // 18 (bytes) — 省下 31%
提示 new Blob([str]).size 回傳的是 UTF-8 編碼的實際位元組數（非字元數）。這是衡量 BLE 封包是否超出 MTU 限制最準確的方式。中文字元佔 3 bytes，因此 JSON Key 應盡量使用短英文縮寫以節省空間。在作業 3-3 中，你將實際運用此技巧來優化封包大小。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template Repo) 配置建議

理解數據交換需要「破壞性測試」與「語法規範」，建議範本包含：

  • 📂 js/packet_decoder.js：封包解碼器。提供一個模擬數據流產生器，故意夾雜「格式錯誤」或「欄位缺失」的 JSON 字串，留下 decode() 與 try-catch 的 TODO。
  • 📂 js/config_handler.js：配置管理器。預置多層次物件架構，留下 ES6 解構語法 (Destructuring) 與「預設值 (Safe Fallbacks)」的實作空間。
  • 📂 docs/protocol_spec.md：協議規範文件。要求學員在此定義新設計的「精簡版 JSON 協議」映射表。
  • 📂 tests/json_robustness.test.js：自動化評分脚本。專門測試邊界情況（如：空字串、巢狀深度過大），檢核學員的代碼是否具備足夠的健壯性 (Robustness)。

2. 作業任務部署細節

任務 1：「封包拆解員」：動態資訊看板 (JSON Packet Decoder Lab)

  • 目標：學會防禦性編程 (Defensive Programming)，確保通訊異常時系統不崩潰。
  • Classroom 部署建議：
    • 健壯解析檢核： 
      // 學生需實作：try-catch 與解構提取
try {
    const { v_bat, rpm } = JSON.parse(rawString);
    updateUI(v_bat, rpm);
} catch (e) {
    console.error("Packet Corrupted"); // 防止網頁報錯停止運行
}
    • Autograding：餵入一組包含「半截字串」的樣本，檢核學員的程式是否能跳過錯誤封包並維持 UI 顯示最後一次的正確值。
    • 視覺驗證：要求提交看板成功顯示 v_bat 與 rpm 的即時刷新截圖。

任務 2：多層次配置管理器 (Nested Config Manager)

  • 目標：掌握 ES6 進階語法，寫出具備「專業感」且強健的數據提取代碼。
  • Classroom 部署建議：
    • 巢狀提取檢核：要求學員實作 const { pid: { p = 1.0, i = 0 } = {} } = config; 這種具備「雙重保險」的解構寫法。
    • 序列化測試：學員需在 UI 調整參數後，生成一個符合協議的 JSON 字串。
    • Autograding：模擬刪除 JSON 中某個非必填欄位，檢核系統是否能正確套用「預設值」而非顯示 undefined。

任務 3：數據壓縮挑戰：精簡 JSON Payload (Payload Slimming Challenge)

  • 目標：在高頻電感通訊中，透過縮減字節提升系統反應速度。
  • Classroom 部署建議：
    • 優化指標：要求學員將 Key 名稱縮寫（如 battery_voltage → bv），並將浮點數四捨五入。
    • 量化分析：要求在 README 中列出優化前後的 Blob.size 數據。
    • PR 深度核核：點評其對「可讀性 vs 效能」的權衡思維。是否在 docs/protocol_spec.md 中留下了清晰的縮寫對應表？

3. 通訊協議導師點評標準 (Data Protocol Benchmarks)

此單元的價值在於 「數據契約 (Data Contract) 的嚴謹度」：

  • [ ] 容錯能力：代碼是否能優雅地處理「髒數據」而不拋出未捕獲的異常？
  • [ ] 語法現代化程度：是否拋棄了舊式的 obj.a.b.c 寫法，改用更簡潔、具備預設值保障的解構語法？
  • [ ] 頻寬意識：是否理解 JSON Key 的長度會直接佔用藍牙 MTU 空間？（這是判斷是否具備嵌入式開發思維的關鍵）。
