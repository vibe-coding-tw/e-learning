作業指引 (Assignment Guide)

作業 3-1：Web Serial 硬體讀取測試

  • 角色扮演： 遠端除錯員。
  • 執行方式： 在 vscode.dev (按下 .) 中，使用序列埠監控器讀取 ESP32 訊息。
  • 評分重點： 確認學生理解瀏覽器權限請求，並成功連結硬體。

作業 3-2：Codespaces 雲端編譯

  • 角色扮演： 雲端架構師。
  • 執行方式： 在 GitHub 開啟 Codespaces，並執行一個終端機指令（如 npm --version 或 pio --version）。
  • 評分重點： 區分「純網頁介面」與「具備運算力容器」的差異。

作業 3-3：跨裝置同步除錯

  • 角色扮演： 敏捷開發者。
  • 執行方式： 用手機或平板在 GitHub 修改 README 並 Commit。
  • 評分重點： 觀察 Git 的即時同步能力與 Commit Message 的品質。

GitHub Classroom 詳細操作指南

第一步：準備範本倉庫 (Template Repo)

  1. 包含 .devcontainer 設定，確保學生開啟 Codespace 時工具已就位。
  2. 勾選 "Template repository"。

第二步：建立 Classroom 作業

  1. 在 Classroom 設定中連結此位範本倉庫。
  2. 重點設定：勾選 "Enable feedback pull requests" 以便進行代碼審查。

第三步：學生端操作 (Flipped Workflow)

  1. 學生按下 "." 進入 Web VS Code 或啟動 Codespaces。
  2. 修改程式碼後 Push 代碼。

第四步：教師回饋 (Review)

  1. 檢查學生 PR 中的 Commit 時序，確認其理解跨裝置同步的邏輯。

在 GitHub Classroom 的實戰部署建議

此單元建議搭配 "Cloud Development Challenge" 作業，建議範本倉庫設置如下：

A. 範本倉庫 (Template Repo) 結構

.
├── .devcontainer/   # (核心) 讓學生一進入 Codespace 就有自動配置好的環境
├── src/             # 存放學生的練習程式碼
└── cloud-test.js    # 一個簡單的小程式供學生運行

B. 參考範例代碼 (Reference Example Code)

.devcontainer/devcontainer.json (自動環境配置)

{
    "name": "Node.js & PlatformIO",
    "image": "mcr.microsoft.com/devcontainers/javascript-node:20",
    "features": {
        "ghcr.io/devcontainers/features/common-utils:1": {}
    },
    "postCreateCommand": "npm install -g platformio"
}

cloud-test.js (雲端運行測試)

C. 自動評分 (Autograding) 的設計

  • Codespace 啟動檢查：： 雖然難以自動偵測是否「在雲端跑」，但可點評 cloud-test.js 的執行結果截圖。
  • Commit 紀錄檢查： 透過 GitHub API 或 Dashboard 檢查學生是否有「.」變身後的 Commit 紀錄。

翻轉教室的實施策略

階段 學生行動 (Git Workflow) 教師/平台角色
課前 (預習) 使用互動模擬器體驗按下 . 的變身效果，理解雲端與本機差異。 Dashboard：檢查學生預習紀錄。
課中 (實作) 開啟自己的 Repo 並建立 Codespace，測試 3-1 與 3-2。 助教角色：引導學生區分 Web 版 VS Code 的限制（無法上傳程式碼）。
課後 (精進) 回家用不同裝置（如平板）進行 3-3 跨端 Commit 挑戰。 Feedback PR：檢視 Commit 紀錄，確認其跨裝置操作連貫性。
