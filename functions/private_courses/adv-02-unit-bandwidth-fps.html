網路頻寬與幀率 (FPS) 的數學關係

🔢 1. 影像容量的數學拆解 (20 min)

原始數據量 (Raw Data)：一張未壓縮影像的大小取決於解析度與每個像素佔用的位元組。

核心公式

Width × Height × Bytes_per_pixel

💡 範例

$160 \times 120$ 的 8-bit 灰階影像 $= 19,200$ Bytes ($\approx 19$ KB)。
FPS 的乘法效應：當你每秒要傳送 20 張圖時，數據量會瞬間放大 20 倍！

🔍 學習任務

計算解析度從 $160 \times 120$ 提升到 $640 \times 480$ 時，所需的頻寬會增加幾倍？
（提示：長寬各加 4 倍，總面積增加多少倍？）

🗜️ 2. 壓縮率 (Compression Ratio) 的變數 (20 min)

JPEG 壓縮的影響：壓縮不是固定的。複雜的畫面（如森林）比簡單的畫面（如白牆）產生的 JPEG 檔案更大。

位元率 (Bitrate) 公式

Bitrate (bps) = 平均幀大小 (Bytes) × 8 × FPS

📊 學習任務

理解為什麼在相同的 Wi-Fi 頻寬下，降低解析度可以換取更高的 FPS（流暢度）。

⚖️ 3. 延遲與頻寬的權衡 (20 min)

網路瓶頸：如果頻寬只有 2Mbps，但你的影像數據流是 5Mbps，會發生什麼事？（提示：封包堆積、延遲增加、連線斷開）。

🚀 預習任務：思考與設定

思考對於「自動導航」的無人車，是「高解析度但每秒 2 幀」比較好，還是「低解析度但每秒 30 幀」比較好？

線上任務：如果要達成 30 FPS 且不卡頓，在 2.4GHz Wi-Fi 環境下，影像品質應該如何設定？

關鍵概念

解析度、品質 (Quality) 與數據量的平衡。

計算公式

FPS × 單張圖片大小 = 所需頻寬

進階課程 2 - 單元 3

針對 adv-02-unit-bandwidth-fps (網路頻寬與幀率的數學關係) 單元，這是一個深入探討「資源取捨 (Trade-off)」的進階工程課程，核心在於解析度、品質與流暢度的平衡。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template Repo) 配置建議

進階效能單元的範本應強調「量化測量」工具，建議包含：

  • 📂 src/main.cpp：內建精確的 $FPS$ 與 $Bytes/sec$ 監測與輸出邏輯，減少學員在底層測量上的重複勞動。
  • 📂 src/abr_controller.cpp：預置一個「自動適應頻寬 (ABR)」的框架類別，將核心的降級邏輯 (Downscaling Logic) 留給學員實作。
  • 📂 analysis/bandwidth_profile.csv：一個預先格式化的 CSV 檔案，讓學員填入在不同解析度下的頻寬消耗數據。
  • 📂 .github/workflows/bench_checker.yml：自動化腳本，驗證 bandwidth_profile.csv 是否填寫完整且數值邏輯正確。

2. 作業任務部署細節

任務 1：頻寬消耗標定法 (Bandwidth Profiling)

  • 目標：透過實驗數據，畫出解析度、品質與頻寬的三維關係圖。
  • Classroom 部署建議：
    • 數據提交：學員需在 analysis/bandwidth_profile.csv 中紀錄 QQVGA、QVGA、VGA 解析度在 Quality=20 下的真實頻寬佔用。
    • Autograding：檢查數據是否符合「面積加倍 -> 流量近乎加倍」的線性趨勢，並核對學員是否發現了 Wi-Fi 傳輸的物理瓶頸（例如：當流量超過 2MB/s 時，FPS 是否開始異常跳動）。

任務 2：動態擁塞控制實驗 (Congestion Control Lab)

  • 目標：實作簡單的 ABR (Adaptive Bitrate) 算法，確保系統在網路惡化時不會斷連。
  • Classroom 部署建議：
    • 核心邏輯檢核： 
      // 學生需實作：當 FPS 連續 3 秒低於 5 幀時，自動切換至更低解析度
if (consecutive_low_fps_seconds >= 3) {
    current_resolution = get_lower_res(current_resolution);
    update_camera_config();
}
    • 挑戰項 (Extra Credit)：學員是否實作了「遲滯 (Hysteresis)」邏輯？避免解析度在臨界點來回跳動 (Resolution Flickering)。

任務 3：硬體加速與 FPS 極限挑戰 (XCLK Optimization)

  • 目標：調教感測器時鐘 (XCLK)，找出硬體同步與功耗的極限。
  • Classroom 部署建議：
    • 現象紀錄：要求學員紀錄將 XCLK 從 20MHz 提升後，影像是否出現「畫面撕裂 (Sync Lost)」或顏色失真。
    • 驗證方式：在 README 中說明為什麼盲目提升時鐘頻率不一定能提升有效 FPS（提示：後端的 DMA 搬運速度或 JPEG 編碼能力可能已經封頂）。

3. 進階效能單元導師點評標準 (Advanced Benchmarks)

此單元的價值在於 「數據驅動的工程決策」：

  • [ ] 量化嚴謹度：提交的數據表是否完整，且能清晰標出系統的「雪崩點 (Performance Drop)」？
  • [ ] 算力感應：是否能區分出 $FPS$ 下降是因為「封包塞在大氣中」還是「CPU 來不及編碼」？
  • [ ] 自癒能力：實作的 ABR 邏輯是否能在模擬干擾撤除後，平滑地恢復至高品質影像？
