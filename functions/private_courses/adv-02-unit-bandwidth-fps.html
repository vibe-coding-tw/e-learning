網路頻寬與幀率 (FPS) 的數學關係

🔢 1. 影像容量的數學拆解 (20 min)

原始數據量 (Raw Data)：一張未壓縮影像的大小取決於解析度與每個像素佔用的位元組。

核心公式

Width × Height × Bytes_per_pixel

💡 範例

$160 \times 120$ 的 8-bit 灰階影像 $= 19,200$ Bytes ($\approx 19$ KB)。
FPS 的乘法效應：當你每秒要傳送 20 張圖時，數據量會瞬間放大 20 倍！

🔍 學習任務

計算解析度從 $160 \times 120$ 提升到 $640 \times 480$ 時，所需的頻寬會增加幾倍？
（提示：長寬各加 4 倍，總面積增加多少倍？）

🗜️ 2. 壓縮率 (Compression Ratio) 的變數 (20 min)

JPEG 壓縮的影響：壓縮不是固定的。複雜的畫面（如森林）比簡單的畫面（如白牆）產生的 JPEG 檔案更大。

位元率 (Bitrate) 公式

Bitrate (bps) = 平均幀大小 (Bytes) × 8 × FPS

📊 學習任務

理解為什麼在相同的 Wi-Fi 頻寬下，降低解析度可以換取更高的 FPS（流暢度）。

⚖️ 3. 延遲與頻寬的權衡 (20 min)

網路瓶頸：如果頻寬只有 2Mbps，但你的影像數據流是 5Mbps，會發生什麼事？（提示：封包堆積、延遲增加、連線斷開）。

🚀 預習任務：思考與設定

思考對於「自動導航」的無人車，是「高解析度但每秒 2 幀」比較好，還是「低解析度但每秒 30 幀」比較好？

線上任務：如果要達成 30 FPS 且不卡頓，在 2.4GHz Wi-Fi 環境下，影像品質應該如何設定？

關鍵概念

解析度、品質 (Quality) 與數據量的平衡。

計算公式

FPS × 單張圖片大小 = 所需頻寬

頻寬與 FPS 優化實戰：實作任務

透過實測與除錯，掌握數據傳輸的工程取捨 (Trade-off)。

進階課程 2 - 單元 3

影像實驗室：頻寬、解析度與 FPS 的三角博弈

任務 1：頻寬消耗標定法 (Bandwidth Profiling)

角色扮演：網路效能分析師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 解析度掃描：分別設置 QQVGA、QVGA、VGA 解析度。
  3. 流量測算：計算每秒傳送的數據量 (Bytes/sec)，並標記當 FPS 跌破 15 時的臨界頻寬。
  4. Push Final：提交包含「解析度-頻寬-FPS 對照表」的數據報告。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 關係識別：能否準確描述解析度加倍時，頻寬與 FPS 的反比變動規律？
  • [ ] 瓶頸分析：報告中是否能指出系統瓶頸是發生在「編碼速度」還是「網路傳輸」？

任務 2：動態擁塞控制實驗 (Congestion Control Lab)

角色扮演：系統開發工程師 | 難度：⭐️⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 干擾導入：在串流過程中開啟大型檔案下載，模擬網路擁塞。
  2. 自動降級：開發一段邏輯，當偵測到 FPS 低於 5 秒持續 3 秒時，自動將解析度調低。
  3. 恢復測試：當擁塞消除後，系統是否能自動恢復至高品質串流？
  4. Push Final：提交具備「網路彈性 (Resilience)」的動態調整模組。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 狀態轉換：解析度切換瞬間，系統是否發生溢位 (Buffer Overflow) 或重啟？
  • [ ] 邏輯嚴謹度：README 是否解釋了「遲滯 (Hysteresis)」設計以避免解析度頻繁跳動？

任務 3：硬體加速與 FPS 極限挑戰 (Clock Speed Optimization)

角色扮演：底層韌體工程師 | 難度：⭐️⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 時鐘調整：嘗試調整 XCLK (感測器主時鐘) 頻率 (10MHz vs 20MHz)。
  2. 效能對比：觀察 CPU 溫度與影像幀率的變動關係。
  3. 穩定性評估：紀錄在「高幀率模式」下，影像是否出現同步錯誤 (Sync Lost)。
  4. Push Final：提交經過「極限調教」後的韌體配置參數。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 硬體深度：能否解釋 XCLK 與感測器取樣週期及 I2C 通訊的同步關係？
  • [ ] 安全邊際：在追求極限 FPS 的同時，是否能保留合理的硬體安全邊際？

GitHub Classroom 詳細操作指南

本單元教學重點：掌握影像傳輸中的「平衡藝術」。理解解析度、壓縮率與網路頻寬之間的數學權衡。

  1. 重點 1：數據秤重能力 - 帶領學生從「感性觀察」轉向「理性測量」。通過具體 Bytes/sec 數據理解頻寬瓶頸。
  2. 重點 2：動態適應思維 - 介紹 ABR (Adaptive Bitrate) 核心概念，理解如何在惡劣路徑下保持系統可用性。
  3. 重點 3：底層硬體限界 - 探討 XCLK 與感測器取樣的物理限制，識別「軟體優化」與「硬體限界」的邊界。
