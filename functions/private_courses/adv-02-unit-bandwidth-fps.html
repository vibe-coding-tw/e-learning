簡介

「為什麼我的串流只有 3 FPS？」——這個問題的答案通常不在程式碼裡，而在一條簡單的數學公式中。 頻寬、解析度、壓縮率與幀率之間存在嚴格的物理約束關係，掌握這些數學工具能讓你在動手寫程式之前就準確預測系統的效能上限。 本模組將教你用位元率公式計算最大可達 FPS，並學會在解析度與流暢度之間做出量化決策。

情境
你的 ESP32-CAM 專案在 QVGA（320×240）解析度下跑得很順暢，達到 25 FPS。但客戶要求升級到 VGA（640×480）以看清車牌號碼。你改了一行設定，FPS 立刻從 25 暴跌到 3，畫面卡成投影片。「解析度只放大了一級，怎麼差這麼多？」你需要用數學來解釋這個現象，並算出在現有頻寬下，能達到的最佳解析度與 FPS 組合。

影像容量的數學拆解

在開始優化影像串流之前，你必須先理解一個基本問題：一張影像到底佔了多少資料量？ 這個問題的答案取決於三個變數：解析度（寬 × 高）、每像素的色彩深度（Bytes per Pixel），以及壓縮方式。掌握這些數學關係，你就能在動手寫程式之前精準預測系統的效能上限。

原始影像大小公式

一張未壓縮的影像，其資料量可以用一條簡單的公式計算：

// 原始影像大小公式 Raw_Size = Width × Height × Bytes_per_Pixel // 範例：QVGA RGB 影像 320 × 240 × 3 = 230,400 Bytes ≈ 225 KB // 範例：VGA RGB 影像 640 × 480 × 3 = 921,600 Bytes ≈ 900 KB

注意：RGB 全彩影像每像素佔 3 Bytes（R、G、B 各 1 Byte），灰階影像每像素僅佔 1 Byte。ESP32-CAM 常用的 YUV422 格式則為每像素 2 Bytes。選擇不同的色彩空間，資料量會有顯著差異。

常見解析度的資料量對比

解析度名稱 尺寸 (px) 總像素數 RGB 原始大小 灰階原始大小
QQVGA 160 × 120 19,200 56 KB 19 KB
QVGA 320 × 240 76,800 225 KB 75 KB
VGA 640 × 480 307,200 900 KB 300 KB
SVGA 800 × 600 480,000 1.37 MB 469 KB
HD (720p) 1280 × 720 921,600 2.64 MB 900 KB
重要 從 QVGA 升級到 VGA，解析度的長寬各放大 2 倍，但總像素數放大了 4 倍（2 × 2 = 4）。這就是為什麼「只是把解析度調高一級」就讓 FPS 暴跌的根本原因——資料量是以面積（二次方）成長的。

FPS 的乘法效應

影像串流不是傳送一張圖，而是每秒傳送數十張。因此，串流所需的原始資料頻寬為：

// 原始串流頻寬公式 Raw_Bandwidth = Width × Height × Bytes_per_Pixel × FPS // 範例：VGA RGB @ 15 FPS (未壓縮) 640 × 480 × 3 × 15 = 13,824,000 Bytes/s ≈ 105 Mbps
警告 VGA 解析度在未壓縮的情況下，15 FPS 就需要 105 Mbps 的頻寬！ESP32 的 Wi-Fi 實際吞吐量約為 2–5 Mbps，這意味著壓縮是絕對必要的——沒有壓縮，串流根本不可能運作。
提示 想驗證這些計算？在 ESP32 的串流程式中，你可以用 fb->len 取得每幀壓縮後的實際位元組數。將它除以原始大小，就能得到「真實壓縮率」——這個數字會隨畫面內容而大幅變動。

壓縮率 (Compression Ratio) 的變數

上一節計算的是未壓縮的原始資料量，但實際串流中，每一幀都經過了 JPEG 壓縮。壓縮率並非一個固定數字——它會隨著畫面內容的複雜度和Quality 參數而劇烈變動。理解這些變數，是精準預測串流頻寬的關鍵。

JPEG 壓縮的物理本質

JPEG 壓縮的核心是離散餘弦變換（DCT），它將 8×8 像素的區塊從「空間域」轉換到「頻率域」，然後丟棄人眼不敏感的高頻細節。Quality 參數控制的就是「丟掉多少高頻成分」——Quality 越低，丟得越多，檔案越小，但畫質也越差。

備註 JPEG 的壓縮是有損的（Lossy）——被丟掉的高頻資訊無法復原。這就是為什麼反覆儲存 JPEG 會讓畫質越來越差。但在即時串流中，每一幀都是從原始感測器資料重新壓縮的，不存在「反覆壓縮」的問題。

影響壓縮率的兩大變數

變數一：Quality 參數

ESP32-CAM 的 JPEG Quality 參數範圍通常是 10（最高壓縮）到 63（最低壓縮）。以下是在 VGA 解析度下的實測數據參考：

Quality 平均幀大小 壓縮率（vs 原始） 畫質感受
10 ~8 KB ~112:1 嚴重馬賽克，僅能辨識大輪廓
20 ~15 KB ~60:1 細節模糊但可用於偵測
40 ~35 KB ~26:1 良好平衡，適合多數應用
63 ~80 KB ~11:1 高畫質，但頻寬消耗大幅增加

變數二：畫面內容複雜度

即使 Quality 參數相同，不同的畫面內容也會產生截然不同的檔案大小。高頻細節越多（如樹葉、草地、碎石），壓縮後的檔案就越大；反之，大面積的單色區域（如白牆、天空）壓縮後檔案很小。

警告 這意味著你不能假設「每幀都是 X KB」來計算頻寬！在實際部署時，你必須用最壞情況（最複雜的場景）來評估頻寬需求，否則一旦畫面變複雜，FPS 就會突然暴跌。

位元率 (Bitrate) 公式

有了平均幀大小，就能計算串流的位元率（Bitrate）——這是衡量串流佔用多少網路頻寬的標準指標：

// 位元率公式 Bitrate (bps) = 平均幀大小 (Bytes) × 8 × FPS // 範例：VGA, Quality=20, 15 FPS 15,000 Bytes × 8 × 15 = 1,800,000 bps = 1.8 Mbps // 反向推導：已知頻寬求最大 FPS Max FPS = 可用頻寬 (bps) ÷ (平均幀大小 × 8) // 範例：2 Mbps 頻寬下的最大 FPS 2,000,000 ÷ (15,000 × 8) ≈ 16.7 FPS
提示 在相同的 Wi-Fi 頻寬下，降低解析度可以換取更高的 FPS。例如從 VGA 降到 QVGA，幀大小減少約 4 倍，FPS 就能提升近 4 倍。這就是「解析度 vs 流暢度」的核心取捨——你無法同時擁有兩者，必須根據應用場景做出選擇。

延遲與頻寬的權衡

當你的影像串流產生的資料量超過了網路的可用頻寬，會發生什麼事？答案不是「畫面變模糊」那麼簡單——而是引發一連串的連鎖崩潰：封包在傳送佇列中堆積、延遲急劇上升、TCP 重傳機制介入導致更多壅塞，最終可能導致連線完全斷開。理解這個物理極限，是設計穩定串流系統的前提。

頻寬瓶頸的物理現象

網路頻寬就像水管的直徑——你灌入的水量（資料量）不能超過水管的容量。當超量時，多餘的水會在入口堆積，形成「壅塞佇列」。對應到串流系統，現象如下：

1

封包堆積

編碼器產出的資料速率超過網路傳輸速率，封包開始在傳送緩衝區中排隊等待。佇列越長，每個封包的等待時間（延遲）就越久。

2

延遲飆升

當佇列深度超過 200–500 ms 時，使用者感受到「畫面嚴重落後」。對遙控應用而言，操控變得完全無法預測。

3

連線崩潰

TCP 偵測到丟包後觸發重傳，而重傳的封包又加劇了壅塞。在最壞的情況下，系統進入「壅塞崩潰（Congestion Collapse）」，有效吞吐量趨近於零。

重要 ESP32 的 2.4 GHz Wi-Fi 在理想條件下的理論傳輸速率為 72 Mbps，但實際可用吞吐量通常只有 2–5 Mbps。這是因為 Wi-Fi 協定的開銷（標頭、ACK、退避機制）、環境干擾、以及多裝置競爭等因素。你的頻寬計算必須以「實測值」而非「理論值」為依據。

解析度 vs 流暢度：不可能三角

在固定的頻寬下，你面對的是一個不可能三角——解析度、畫質（Quality）、流暢度（FPS）三者最多只能同時滿足兩個。提升任何一項，必然要犧牲另一項：

策略 解析度 Quality FPS 適用場景
高解析度 + 高品質 VGA 40 2–5 靜態監控截圖（不需要動態）
高解析度 + 高流暢度 VGA 10 10–15 動態監控（接受畫質損失）
低解析度 + 高流暢度 QVGA 20 25–30 遙控車、FPV 飛行（流暢優先）
極低解析度 + 極高流暢度 QQVGA 20 30+ AI 視覺避障（資料量最小）

ABR 動態擁塞控制概念

Netflix 和 YouTube 使用的ABR（Adaptive Bitrate）技術，其核心思想很簡單：偵測到 FPS 下降時，自動降低解析度或品質，讓串流保持流暢。反之，當網路恢復時，再逐步提升品質。這種「自我調節」的機制可以避免壅塞崩潰。

// ABR 擁塞控制的虛擬碼 if (連續 3 秒 FPS < 5) { current_resolution = 降一級(current_resolution); 更新攝影機設定(); log("⚠ 偵測到壅塞，降級至", current_resolution); } if (連續 10 秒 FPS > 20 && current_resolution < 最高級) { current_resolution = 升一級(current_resolution); 更新攝影機設定(); log("✓ 網路恢復，升級至", current_resolution); }
提示 注意升級條件比降級條件更嚴格（10 秒 vs 3 秒）——這種「不對稱閾值」設計稱為遲滯（Hysteresis）。它可以避免解析度在臨界點來回跳動（Resolution Flickering），讓使用者看到穩定的畫面品質。
備註 在實戰挑戰中，你將親手實作這套 ABR 邏輯，並在模擬弱網環境下測試它的自癒能力。思考：對於自動導航的無人車，究竟是「高解析度但每秒 2 幀」好，還是「低解析度但每秒 30 幀」好？答案取決於 AI 模型需要多少細節——而你的 ABR 邏輯必須做出即時判斷。

}
    • 挑戰項 (Extra Credit)：學員是否實作了「遲滯 (Hysteresis)」邏輯？避免解析度在臨界點來回跳動 (Resolution Flickering)。

任務 3：硬體加速與 FPS 極限挑戰 (XCLK Optimization)

  • 目標：調教感測器時鐘 (XCLK)，找出硬體同步與功耗的極限。
  • Classroom 部署建議：
    • 現象紀錄：要求學員紀錄將 XCLK 從 20MHz 提升後，影像是否出現「畫面撕裂 (Sync Lost)」或顏色失真。
    • 驗證方式：在 README 中說明為什麼盲目提升時鐘頻率不一定能提升有效 FPS（提示：後端的 DMA 搬運速度或 JPEG 編碼能力可能已經封頂）。

3. 進階效能單元導師點評標準 (Advanced Benchmarks)

此單元的價值在於 「數據驅動的工程決策」：

  • [ ] 量化嚴謹度：提交的數據表是否完整，且能清晰標出系統的「雪崩點 (Performance Drop)」？
  • [ ] 算力感應：是否能區分出 $FPS$ 下降是因為「封包塞在大氣中」還是「CPU 來不及編碼」？
  • [ ] 自癒能力：實作的 ABR 邏輯是否能在模擬干擾撤除後，平滑地恢復至高品質影像？

4. 自我檢測與自動評分 (Autograding)

本專案內建自動化評分腳本，您可以隨時在本地端執行以驗證您的進度：

  1. 安裝依賴： 
    pip install pandas
  2. 執行評分： 
    python tools/grader.py

評分腳本會檢查：

  • analysis/throughput.csv 是否已填寫完整數據。
  • src/abr_logic.cpp 中的 TODO 是否已處理（即已實作 ABR 邏輯）。
  • docs/bottleneck_report.md 是否已完成分析且勾選了對應項。

透過這種部署方式，學生能體會到 「數據是不會說謊的」。在面對網路多變的真實世界時，學會如何透過數學公式與動態邏輯，在受限的頻寬中榨取最高的視覺品質。
