📦 1. 為什麼要封裝 API？ (15 min)

在上一個單元中，我們推導了 ICC 的數學公式，能夠從目標半徑 \(R\) 和速度 \(V\) 計算出左右輪速。 但如果每次需要轉彎時，都得在應用層直接操作三角函數、處理 PWM 映射和邊界條件， 不僅容易出錯，更會讓程式碼變得極難維護。這正是 API 封裝（API Encapsulation）的價值所在。

資訊隱藏（Information Hiding）的力量

軟體工程大師 David Parnas 在 1972 年提出的「資訊隱藏」原則指出：模組應該隱藏其內部的設計決策， 只透過穩定的介面與外部溝通。對差速驅動車而言，這意味著上層程式碼不需要知道 \(R = \frac{L}{2} \cdot \frac{V_L + V_R}{V_L - V_R}\) 這個公式怎麼推導的——它只需要呼叫 driveArc(radius, speed) 就能精確控制車體軌跡。

備註 好的 API 就像方向盤：駕駛者不需要理解齒輪齒條機構或動力輔助泵浦的液壓原理，只需轉動方向盤就能控制車輛方向。 這就是抽象層的精髓——降低使用者的認知負擔。

封裝帶來的三大工程優勢

優勢 未封裝（直接操作 PWM） 封裝後（透過 API）
可讀性 analogWrite(pinL, 200); analogWrite(pinR, 133); driveArc(0.5, 100);
可維護性 更換馬達驅動板需修改每一處 PWM 呼叫 只需修改 API 內部的硬體抽象層
安全性 忘記邊界檢查可能導致馬達過載 API 內部自動攔截非法參數

一個具體的對比範例

假設你的隊友需要讓機器人沿著半徑 50 cm 的圓弧行駛。以下是「未封裝」和「封裝後」的程式碼對比：

// ❌ 未封裝：隊友必須理解完整的運動學公式
float R = 0.50; // 目標半徑 (m)
float V = 0.20; // 目標線速度 (m/s)
float omega = V / R;
float V_L = omega * (R + L/2);
float V_R = omega * (R - L/2);
int pwmL = map(V_L, 0, maxV, 0, 255);
int pwmR = map(V_R, 0, maxV, 0, 255);
// 還需自行處理飽和、死區、負值...

// ✅ 封裝後：一行搞定
drive.arc(0.50, 0.20); // 半徑 50cm，速度 0.2 m/s
警告 封裝不只是「讓程式碼變短」。API 內部應包含安全攔截邏輯： 當輸入的半徑 \(R\) 小於物理可達到的最小值，或速度超過 PWM 上限時， API 應自動進行 Clamp（箝位）或 Scaling（縮放），而不是讓硬體進入未定義行為。
提示 設計 API 時，問自己一個問題：「如果我的隊友完全不懂 ICC 公式，他能不能只看函式名稱和參數就知道怎麼用？」 如果答案是「可以」，那你的 API 就達到了自述性（Self-documenting）的標準。

🎛️ 2. 設計良好的控制接口 (20 min)

API 設計不只是「寫個函式」這麼簡單。一個設計良好的控制接口，應該讓使用者在不閱讀原始碼的情況下， 僅憑函式名稱和參數就能正確使用。這種品質稱為自述性（Self-documenting）—— 它是區分業餘程式碼與專業程式庫的關鍵分水嶺。

參數層級：幾何 vs. 時間

在設計差速驅動 API 時，首先要決定的是參數的語意層級。 我們可以用不同抽象度的參數來描述同一個動作：

層級 參數形式 函式範例 使用者需要理解的知識
底層（PWM） 左右 PWM 值 setMotors(200, 100) PWM 與馬達的映射關係
幾何層 半徑 \(R\)、線速度 \(V\) driveArc(0.5, 0.2) 基本幾何概念（半徑、速度）
任務層 角度、距離 turnAngle(90, 30) 幾乎不需要額外知識
備註 良好的 API 應該提供多個層級的接口。進階使用者可以直接操作幾何層以獲得最大彈性， 而初學者可以使用任務層快速完成目標。這就是所謂的漸進式揭露（Progressive Disclosure）設計原則。

命名規範與一致性

函式命名是 API 設計中最容易被忽略、卻影響最深遠的決策。以下是差速驅動控制 API 的命名建議：

// ✅ 好的命名：動詞 + 名詞，語意清晰
void driveArc(float radius, float speed); // 沿圓弧行駛
void driveStraight(float speed); // 直線行駛
void spinInPlace(float angularVelocity); // 原地旋轉
void stopGradually(unsigned long ms); // 緩減速停車
void emergencyStop(); // 緊急煞車

// ❌ 壞的命名：模糊、不一致
void go(int a, int b); // a 和 b 是什麼？
void doThing(float x); // 做什麼事？
void motor_set_L_R(...); // 暴露了硬體細節

邊界保護與防禦性設計

良好的 API 不會讓使用者輕易犯錯。每個公開函式都應包含參數驗證邏輯， 在輸入值超出合理範圍時，自動進行箝位（Clamp）或拋出清楚的錯誤訊息。

void driveArc(float radius, float speed) {
// 邊界保護：半徑過小 → 自動切換為原地旋轉
if (abs(radius) < 0.001) {
spinInPlace(speed > 0 ? 1.0 : -1.0);
return;
}
// 邊界保護：半徑過大 → 視為直線行駛
if (abs(radius) > 1000) {
driveStraight(speed);
return;
}
// 正常 ICC 運算...
}
提示 設計 API 時，用這個測試檢驗你的命名品質：把函式簽名（不含實作）列印出來， 拿給一位不懂運動學的同學看。如果他能在 10 秒內猜出每個函式的用途，你的 API 就通過了自述性測試。

🏗️ 3. 數據結構與返回類型

一個成熟的控制 API 不應只是「發出指令然後結束」——它還需要負責資料的打包與狀態回報。 透過精心設計的自定義結構體（Struct），我們可以讓 API 同步回傳運動學計算結果、 馬達的實際物理速率、以及執行狀態碼，為上層的 PID 閉環控制或遙測系統提供可靠的數據介面。

設計返回結構體

與其讓函式回傳單一的 void 或 bool， 不如設計一個包含完整運算結果的結構體。以下是差速驅動控制 API 推薦的數據結構設計：

// 運動學計算結果結構體
struct KinematicsResult {
float left_speed_ms; // 左輪目標線速度 (m/s)
float right_speed_ms; // 右輪目標線速度 (m/s)
int left_pwm; // 左輪 PWM 輸出值 (0-255)
int right_pwm; // 右輪 PWM 輸出值 (0-255)
bool was_clamped; // 是否觸發了飽和縮放
float actual_radius; // 經箝位後的實際半徑
};

// 控制狀態枚舉
enum DriveState {
IDLE, // 停止中
DRIVING_ARC, // 沿圓弧行駛中
DRIVING_LINE, // 直線行駛中
SPINNING, // 原地旋轉中
TURNING // 角度轉向中（非阻塞）
};

API 方法總覽

下表列出了完整的 DriveSystem 類別應該提供的公開方法及其簽名：

方法簽名 返回類型 說明
driveArc(float R, float V) KinematicsResult 沿指定半徑的圓弧行駛
driveStraight(float V) KinematicsResult 直線行駛
spinInPlace(float omega) KinematicsResult 原地旋轉
turnAngle(float deg, float omega) void 非阻塞式角度轉向
getState() DriveState 查詢當前運動狀態
emergencyStop() void 立即停止所有馬達
備註 注意 driveArc() 回傳的 KinematicsResult 包含一個 was_clamped 欄位。 這讓上層程式碼可以得知「系統是否因為 PWM 飽和而進行了速度縮放」， 從而決定是否需要降低整體速度或調整軌跡規劃。

封裝的存取控制

良好的 C++ 類別設計應嚴格區分 public 和 private 成員。 物理參數（輪距 \(L\)、輪徑、死區值等）應設為 private， 只能透過建構函式初始化，防止外部程式碼意外修改。

class DriveSystem {
public:
DriveSystem(float wheelBase, float wheelDiam,
int pinL, int pinR);
KinematicsResult driveArc(float R, float V);
void emergencyStop();

private:
float _wheelBase; // 外部無法直接修改
float _wheelDiam; // 只能透過建構函式設定
int _deadZoneL, _deadZoneR;
void _applyPWM(int left, int right);
};
提示 在你的 DriveSystem.h 標頭檔中，只放置公開介面的宣告。 所有實作細節都放在 DriveSystem.cpp 中。 這就是 C++ 的介面與實作分離原則——讓使用者只看到「能做什麼」， 而不需要關心「怎麼做」。

🤖 進階 10：API 封裝設計 (API Design & Abstraction)

歡迎來到 adv-10-unit-api-design 專案！本單元是讓你從「硬體愛好者」蛻變為「軟體架構師」的關鍵。我們將學習 「資訊隱藏 (Information Hiding)」 與 「抽象層設計 (Abstraction Layer)」。在這裡，你不再直接對馬達下達 PWM 指令，而是要設計一套具備物理語義的控制 API。

🎯 作業目標與任務

本次作業分為三個核心任務，請依照順序完成：

任務 1：Kinematics 類別設計 (Abstraction Lab)

  • 目標：實作「運動大腦」類別，建立從幾何指令到馬達轉速的強健映射。
  • 實作檔案：src/DriveSystem.cpp
  • 要求：
    • 實作物理參數的封裝，確保輪距 $L$、輪徑等只能透過 constructor 初始化。
    • 處理數學邊界情況：
      • 半徑過小 (abs(radius) < 0.001)：切換為「原地自轉」模式。
      • 半徑過大 (radius > 1000)：切換為「直線行駛」模式。

任務 2：API 調用挑戰：8 字形軌跡 (API Sequence Challenge)

  • 目標：驗證 API 接口的簡潔性與行為一致性。
  • 實作檔案：examples/Figure8.ino
  • 要求：
    • 使用你封裝好的 API 完成「8 字形」繞行任務。
    • 嚴禁在應用層 (Sketch) 中出現任何直接控制 PWM 的代碼。
    • 請觀察多次運行後的累積誤差，並在下方的【觀察與心得】區塊記錄並分析為什麼「開環控制」下的 8 字形會隨時間漂移。

任務 3：目標角度轉向接口 (Angle Command Engine)

  • 目標：將「時間」因素導入 API，建立具備回饋意識的高階控制。
  • 實作檔案：src/DriveSystem.cpp / include/DriveSystem.h
  • 要求：
    • 實作轉向精準度換算：void turnAngle(float degree, float angular_velocity)。
    • 非阻塞設計：讓車子在旋轉的同時，程式仍能繼續執行（例如監聽感測器），不可使用 delay()。
    • 在下方的【觀察與心得】說明你的非阻塞設計思路。

🚀 評分與繳交標準 (Software Arch Benchmarks)

你的程式碼將依據以下標準進行評估：

  1. 封裝完備性：成員存取控制 (Private/Public) 是否合理？是否成功隱藏了硬體細節？
  2. 接口易用性：API 調用是否直覺？是否具備「自述性 (Self-documenting)」？
  3. 數學邊界處理：針對極端參數是否有做平滑降級或保護？
  4. 自動化測試：透過 tests/api_logic.test.cpp 檢核運動學轉換精準度。

  • 📂 src/DriveSystem.cpp：實作檔。提供基礎的 C++ 類別骨架，引導學員處理 $R=\infty$ (直線) 等數學邊界情況。
  • 📂 examples/Figure8.ino：應用範例。這是一個空的 Sketch，學員需在此調用自己寫好的 API 來完成「8 字形」繞行任務。
  • 📂 tests/api_logic.test.cpp：單元測試腳本。專門測試「運動學映射」——輸入特定的半徑 $R$ 與速度 $V$，檢核輸出的左右輪速是否符合幾何理論值。
