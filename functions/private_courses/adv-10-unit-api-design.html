從幾何到代碼：
API 封裝設計

釋放運動控制的潛能。將複雜的 ICC 數學封裝為易用的 API，打造直覺且健壯的控制介面。

📦 1. 為什麼要封裝 API？ (15 min)

好的 API 就像駕駛座的方向盤，駕駛者不需要知道引擎內部的閥門如何開合，只需要轉動方向盤即可。這就是資訊隱藏的力量。

Abstraction Layer

  • 解譯複雜度：呼叫 move(radius, speed) 而非處理三角函數。
  • 解耦硬體：更換馬達只需修改 API 內部，上層邏輯無需變動。

Safety Intercept

if (radius < MIN_LIMIT) {
radius = MIN_LIMIT; // Auto-Clamp
}

Shielding Hardware Limits

🎛️ 2. 設計良好的控制接口 (20 min)

🏗️ 3. 數據結構與返回類型

API 不應只是執行動作，還應負責資料的打包與狀態回報。透過自定義結構體，我們可以同步回傳左右馬達的實際物理速率，供上層進行 PID 閉環。

【作業 2-1】

「運動大腦」

將 ICC 公式封裝進 C++ 類別。實作 calculateIK(radius, speed)，並處理 $R=\infty$ (直線) 與 $R \to 0$ (自轉) 的邊界數學情境。

♾️ 60 min

【作業 2-2】

「8 字型執法官」

嚴禁操作 PWM！僅限調用自研 API 組合指令。由左轉半圓與右轉半圓交替，挑戰回到初始座標的精準度，檢驗 API 的穩定度。

📐 45 min

【作業 2-3】

「角度精確接口」

結合時間與角速度。實作 turnAngle(degree, speed)。讓車子能精準轉向 90 度後主動停止，嘗試導入「非阻塞式」計時設計。
