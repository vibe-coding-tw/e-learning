📦 1. 為什麼要封裝 API？ (15 min)

好的 API 就像駕駛座的方向盤，駕駛者不需要知道引擎內部的閥門如何開合，只需要轉動方向盤即可。這就是資訊隱藏的力量。

Abstraction Layer

  • 解譯複雜度：呼叫 move(radius, speed) 而非處理三角函數。
  • 解耦硬體：更換馬達只需修改 API 內部，上層邏輯無需變動。

Safety Intercept

if (radius < MIN_LIMIT) {
radius = MIN_LIMIT; // Auto-Clamp
}

🤖 進階 10：API 封裝設計 (API Design & Abstraction)

歡迎來到 adv-10-unit-api-design 專案！本單元是讓你從「硬體愛好者」蛻變為「軟體架構師」的關鍵。我們將學習 「資訊隱藏 (Information Hiding)」 與 「抽象層設計 (Abstraction Layer)」。在這裡，你不再直接對馬達下達 PWM 指令，而是要設計一套具備物理語義的控制 API。

🎯 作業目標與任務

本次作業分為三個核心任務，請依照順序完成：

任務 1：Kinematics 類別設計 (Abstraction Lab)

  • 目標：實作「運動大腦」類別，建立從幾何指令到馬達轉速的強健映射。
  • 實作檔案：src/DriveSystem.cpp
  • 要求：
    • 實作物理參數的封裝，確保輪距 $L$、輪徑等只能透過 constructor 初始化。
    • 處理數學邊界情況：
      • 半徑過小 (abs(radius) < 0.001)：切換為「原地自轉」模式。
      • 半徑過大 (radius > 1000)：切換為「直線行駛」模式。

任務 2：API 調用挑戰：8 字形軌跡 (API Sequence Challenge)

  • 目標：驗證 API 接口的簡潔性與行為一致性。
  • 實作檔案：examples/Figure8.ino
  • 要求：
    • 使用你封裝好的 API 完成「8 字形」繞行任務。
    • 嚴禁在應用層 (Sketch) 中出現任何直接控制 PWM 的代碼。
    • 請觀察多次運行後的累積誤差，並在下方的【觀察與心得】區塊記錄並分析為什麼「開環控制」下的 8 字形會隨時間漂移。

任務 3：目標角度轉向接口 (Angle Command Engine)

  • 目標：將「時間」因素導入 API，建立具備回饋意識的高階控制。
  • 實作檔案：src/DriveSystem.cpp / include/DriveSystem.h
  • 要求：
    • 實作轉向精準度換算：void turnAngle(float degree, float angular_velocity)。
    • 非阻塞設計：讓車子在旋轉的同時，程式仍能繼續執行（例如監聽感測器），不可使用 delay()。
    • 在下方的【觀察與心得】說明你的非阻塞設計思路。

🚀 評分與繳交標準 (Software Arch Benchmarks)

你的程式碼將依據以下標準進行評估：

  1. 封裝完備性：成員存取控制 (Private/Public) 是否合理？是否成功隱藏了硬體細節？
  2. 接口易用性：API 調用是否直覺？是否具備「自述性 (Self-documenting)」？
  3. 數學邊界處理：針對極端參數是否有做平滑降級或保護？
  4. 自動化測試：透過 tests/api_logic.test.cpp 檢核運動學轉換精準度。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template 程式庫) 配置建議

API 封裝需要嚴謹的類別結構與介面定義，建議範本包含：

  • 📂 include/DriveSystem.h：介面定義檔。定義 KinematicsResult 結構體與 DriveSystem 類別的公有/私有成員，留下運動學計算的 TODO。
  • 📂 src/DriveSystem.cpp：實作檔。提供基礎的 C++ 類別骨架，引導學員處理 $R=\infty$ (直線) 等數學邊界情況。
  • 📂 examples/Figure8.ino：應用範例。這是一個空的 Sketch，學員需在此調用自己寫好的 API 來完成「8 字形」繞行任務。
  • 📂 tests/api_logic.test.cpp：單元測試腳本。專門測試「運動學映射」——輸入特定的半徑 $R$ 與速度 $V$，檢核輸出的左右輪速是否符合幾何理論值。

2. 作業任務部署細節

任務 1：Kinematics 類別設計 (Abstraction Lab)

  • 目標：實作「運動大腦」類別，建立從幾何指令到馬達轉速的強健映射。
  • Classroom 部署建議：
    • 封裝規範檢核：學員必須將輪距 $L$、輪徑等物理參數設定為 private，只能透過 constructor 初始化。
    • 異常攔截邏輯： 
      // 學生需實作：邊界處理
if (abs(radius) < 0.001) {
    // 自動切換為「原地自轉」模式
} else if (radius > 1000) {
    // 自動切換為「直線行駛」模式
}
    • Autograding：透過單元測試檢核 calculateIK() 的數學轉換精準度。

任務 2：API 調用挑戰：8 字形軌跡 (API Sequence Challenge)

  • 目標：驗證 API 接口的簡潔性與行為一致性。
  • Classroom 部署建議：
    • 指令純淨度要求：學員在完成 8 字形繞行時，嚴禁出現任何 PWM 設定代碼。
    • 累積誤差錄試：要求學員在 README.md 中紀錄多次運行後，圓心偏移的數據，並分析為什麼「開環控制」下的 8 字形會隨時間漂移。
    • PR 深度核核：點評其 API 命名是否具備「自述性 (Self-documenting)」。

任務 3：目標角度轉向接口 (Angle Command Engine)

  • 目標：將「時間」因素導入 API，建立具備回饋意識的高階控制。
  • Classroom 部署建議：
    • 轉向精準度實作： 
      // 學生需實作：角度換算時間公式
void turnAngle(float degree, float angular_velocity) {
    unsigned long duration = (abs(degree) / angular_velocity) * 1000;
    // 非阻塞式執行或狀態標記
}
    • 非阻塞設計檢核：要求學員在 README.md 說明，他們如何讓車子在旋轉 90 度的同時，依然能監聽感測器數據（即不使用 delay()）。
    • Push Final：提交具備「狀態碼（任務中/已完成）」回傳機制的最終版 DriveSystem 程式庫。
