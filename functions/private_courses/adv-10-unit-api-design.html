API 封裝設計

釋放運動控制的潛能。將複雜的 ICC 數學封裝為易用的 API，打造直覺且健壯的控制介面。

📦 1. 為什麼要封裝 API？ (15 min)

好的 API 就像駕駛座的方向盤，駕駛者不需要知道引擎內部的閥門如何開合，只需要轉動方向盤即可。這就是資訊隱藏的力量。

Abstraction Layer

  • 解譯複雜度：呼叫 move(radius, speed) 而非處理三角函數。
  • 解耦硬體：更換馬達只需修改 API 內部，上層邏輯無需變動。

Safety Intercept

if (radius < MIN_LIMIT) {
radius = MIN_LIMIT; // Auto-Clamp
}

Shielding Hardware Limits

API 實驗室：從幾何公式到產業級封裝

任務 1：Kinematics 類別設計 (Abstraction Lab)

角色扮演：軟體架構師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 成員保護：將輪距 $L$、輪徑等設定為 `private`，透過 `constructor` 初始化。
  3. 異常處理：在 API 內部偵測除以零（$R=0$）的非法運算，並自動映射到自轉模式。
  4. README 紀錄：說明為何使用 `float` 而非 `int` 來建立運動學結構體。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 結構對稱性：API 是否正確區分了 $V_{left}$ 與 $V_{right}$ 的計算分支？
  • [ ] 封裝完整度：主程式 (`main.ino`) 是否完全沒有出現顯式的 PWM 操作？

任務 2：軌跡調用挑戰 (Traj Mapping Lab)

角色扮演：測試工程師 | 難度：⭐️⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 指令鏈路設計：利用 driveArc(R, V) 指令，將一個圓弧軌跡拆解為兩個半圓組合。
  2. 累積誤差分析：在 README 錄製運行影片，觀察物理上的 8 字圓心是否逐漸偏移，並探討原因。

任務 3：目標轉向接口 (Angle Command Engine)

角色扮演：韌體工程師 | 難度：⭐️⭐️⭐️ | 估計時長：45 min

GitHub Classroom 詳細操作指南

本單元教學重點：讓學生掌握「資訊隱藏 (Information Hiding)」的軟體工程核心。從單純寫 Code 轉向「設計介面」。

  1. 重點 1：封裝的心理邊界 - 要求學生思考：如果我的馬達換了，API 使用者需要被通知嗎？答案應是否定的。
  2. 重點 2：防呆的必要性 - 在 API 層級攔截除以零或溢位，比在全域攔截更容易除錯。
  3. 重點 3：介面的自癒性 - 講解為什麼 `driveArc(30, 100)` 比 `setMotor(128, 64)` 更具「自述性 (Self-documenting)」。
