為何電腦看不懂紅色：
HSV 的故事

深入探討 HSV 相比於 RGB 在變動光影環境下的穩定性，學習如何定義精準的顏色區間。

💔 1. RGB 的致命傷：光影耦合 (20 min)

在 RGB 空間中，如果你調暗燈光，$R, G, B$ 三個數值會同時下降。這導致針對「紅色」的門檻值會隨著環境光變動而頻繁失效。

Coupling Problem

  • 數值漂移：同一物體在陽光與陰影下的特徵不一致。
  • 非互斥性：R 分量包含亮度與顏色訊息。

RGB Space Observation

Light: (255, 60, 60) Bright Red
Dark: (110, 20, 20) Deep Red

📐 2. HSV：幾何上的「分而治之」 (30 min)

將亮度 (Value) 與顏色 (Hue) 徹底分離。在 HSV 中，只要顏色不變，即使燈光強弱改變，Hue (色相) 依然保持高度穩定。

⭕ 3. 色環的連續性與斷點

紅色在色環的「斷點」處 ($0^{\circ}$ 與 $360^{\circ}$)。學習如何處理數值上的跨區過濾，讓追蹤不因為微小的角度跳變而失效。

Binary Mask Strategy

  • 雙重門檻 (Dual Threshold)：同時檢查 $H > 350$ 或 $H < 10$。
  • 飽和過濾 (S-Filter)：排除灰度背景雜訊。
  • 多區間合併 (Union)：將多段色值結果整合成最終 Mask。
360/0 Target

單元課程 進階 6 - HSV 數學

adv-06-unit-hsv-math (HSV 顏色空間的數學優勢)

歡迎來到 HSV 顏色空間的數學優勢單元！在這個單元中，我們將學習如何利用 HSV 色彩模型來處理電腦視覺中的顏色識別問題。

專案目標

本單元的核心在於理解 HSV 如何將「亮度」與「顏色」解耦，並學會處理色環（Hue）的循環邊界問題。你將完成以下任務：

  1. 色彩空間壓力測試 (Color Stress Test)：量化證明 HSV 在變動環境下的穩定性。
  2. 動態 HSV 遮罩與「紅色斷點」處理：處理色相在 0/360 度處的非線性邊界問題。
  3. 紅綠燈行為優先權判定：將色彩感知轉化為具備安全性優先權的系統決策。

任務說明

任務 1：色彩空間壓力測試

請在 docs/stress_test_report.md 中記錄不同光照條件下的數據，並分析為什麼 HSV 模型比 RGB 更適合用於視覺追蹤。

任務 2：動態 HSV 遮罩實作

在 js/mask_logic.js 中實作遮罩邏輯。特別注意「紅色」在色環上的斷點處理。

任務 3：行為決策邏輯

在 js/decision_maker.js 中實作決策引擎。當系統同時偵測到多種顏色時，應遵循「安全優先」的原則進行決策（例如：紅燈優先於綠燈）。

程式庫結構

  • js/hsv_math.js: 核心數學運算，包含 RGB 到 HSV 的轉換。
  • js/mask_logic.js: 顏色遮罩判定邏輯。
  • js/decision_maker.js: 行為決策邏輯。
  • docs/stress_test_report.md: 實驗紀錄與分析報告。
  • tests/: 自動化評分脚本，用於驗證你的程式碼正確性。

1. 範本程式庫 (Template 程式庫) 配置建議

色彩空間的轉換與二值化（Masking）對穩定性要求極高，建議範本包含：

  • 📂 js/hsv_core.js：核心數學檔案，提供 ImageData 操作框架，留下 rgbToHsv() 公式實作與 inRange() 判定邏輯的 TODO。
  • 📂 js/decision_engine.js：決策引擎模組。用於 Task 3，讓學員實作基於像素數量的行為觸發（如紅燈停、綠燈行）。
  • 📂 STRESS_TEST.md：實驗紀錄表。預置表格，要求學員填入同一目標在不同光照（陽光、陰影、強光）下的 RGB 與 HSV 對比數據。
  • 📂 tests/hsv_conversion.test.js：自動化評分脚本。利用 Jest 驗證 rgbToHsv() 轉換函式的數學精確度（比對標準色碼轉換結果）。

2. 作業任務部署細節

任務 1：色彩空間壓力測試 (Color Stress Test)

  • 目標：透過實驗數據量化證明 HSV 在變動環境下的穩定性（抗噪性）。
  • Classroom 部署建議：
    • 數據報告要求：學員需在 STRESS_TEST.md 中計算變異係數 ($\sigma / \mu$)。分析當環境變暗時，雖然 RGB 數值暴跌，但 HSV 的 $H$ 值是否依然保持穩定。
    • PR 深度核核：導師點評其對「數值耦合 (Coupling)」概念的理解。是否能解釋為什麼 RGB 模型不適合用於戶外追蹤？

任務 2：動態 HSV 遮罩與「紅色斷點」處理 (Red Gap Handling)

  • 目標：處理色環在 $0/360$ 度處的非線性邊界問題，這是開發者常犯錯的「天坑」。
  • Classroom 部署建議：
    • 邊界邏輯檢核： 
      // 學生需實作：應對紅色斷點的邏輯 OR
function isRed(h) {
    return (h > 350 || h < 10); // 而不是簡單的 h > h_min && h < h_max
}
    • Autograding：餵入一個 H 值為 $358$ 的像素，檢核學員的遮罩函式是否能正確判定其為「紅色」。如果學員只用簡單區域判斷，此處會失效。
    • 互動 UI 實作：要求在網頁上建立 HSV 滑桿，並即時渲染出二值化（黑白）遮罩圖形，驗證提取品質。

任務 3：紅綠燈行為優先權判定 (Behavior Decision Logic)

  • 目標：將「感知」轉化為「語義化的系統決策」，並考慮安全性優先權。
  • Classroom 部署建議：
    • 決策邏輯實作： 
      // 學生需實作：安全優先的決策
if (redPixels > threshold) return "STOP"; 
else if (greenPixels > threshold) return "FOLLOW";
    • 驗證方式：在畫面積前同時晃動紅板與綠板（模擬衝突情境）。系統必須能根據「安全優先」原則穩定輸出 STOP。
    • Tutor View (PR 審核)：檢查學員是否加入了「飽和度 (S)」與「亮度 (V)」的過濾，以防止淺灰色背景或反光引起的「虛假觸發 (False Positive)」。

3. 色彩感知導師點評標準 (HSV Perception Benchmarks)

此單元的價值在於 「對數位訊號物理特性的深刻理解」：

  • [ ] 邊界處理能力：代碼中是否正確處理了 Hue 的 $360 \to 0$ 循環特性？
  • [ ] 抗噪優化意識：是否利用 S (飽和度) 成功過濾掉了黑白灰等「無色」背景噪音？
  • [ ] 行為決策一致性：當數據發生閃動時，是否有實作濾波或計數確認，以防止控制指令頻繁切換產生的「抖動」？
