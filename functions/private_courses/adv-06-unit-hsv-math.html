RGB 的致命傷：光影耦合

RGB 是我們最熟悉的色彩模型——每個像素由紅（R）、綠（G）、藍（B）三個 $[0, 255]$ 的數值組合而成。這套模型在螢幕顯示上運作完美，但用於電腦視覺的顏色辨識時，卻有一個致命的結構性缺陷：亮度資訊與顏色資訊彼此耦合。

耦合問題的具體表現

想像你用攝影機對準一顆紅色球。在陽光直射下，感測器回報的數值可能是 (255, 60, 60)；但當一片雲遮住太陽時，同一顆球的讀值可能驟降為 (110, 20, 20)。兩組數值在 RGB 空間中距離甚遠，但它們代表的是同一顆紅球。

這就是「耦合」的問題：R 通道的數值同時包含了「這是什麼顏色」和「環境有多亮」兩種資訊，而且無法透過簡單的加減法將它們分開。

光照條件RGB是否為紅色？門檻判定
強光直射2556060是 通過 R > 200
一般室內光1804035是 通過
陰影區域1102020是 未通過 R > 200
極暗環境5088是 未通過
注意 上表清楚顯示：使用固定的 RGB 門檻值（如 R > 200）來判定「紅色」，在光線變暗時必然失效。你不得不頻繁調整門檻值，這在自動駕駛的動態場景中是不可接受的。

為什麼不能靠「放寬門檻」解決？

直覺上，你可能會想：「那就把門檻降低到 R > 50 不就好了？」但問題在於，當 R 門檻放寬後，許多非紅色的物體（如棕色桌面、膚色皮膚）也會被誤判為紅色。放寬門檻同時提高了召回率（Recall）卻犧牲了精確度（Precision）——這個取捨在 RGB 空間中是無解的。

警告 RGB 模型的根本問題在於：它是為螢幕顯示（加色混色）設計的，而非為顏色辨識設計。在任何需要於變動光線下穩定追蹤特定顏色的應用中，RGB 都不是正確的選擇。下一節介紹的 HSV 色彩空間才是解答。
提示 你可以用手電筒照射一張紅色色紙，在 Canvas 上即時觀察 RGB 三個通道的數值變化。你會發現：光線越強，$R, G, B$ 三者同步上升——這就是耦合的具體表現。

HSV：幾何上的「分而治之」

HSV 色彩模型的全名是 Hue（色相）、Saturation（飽和度）、Value（亮度）。它的核心設計理念是將顏色的「種類」與「強度」徹底解耦——色相決定「這是什麼顏色」，飽和度決定「顏色有多純」，亮度決定「有多明亮」。這三個維度彼此獨立，使得 HSV 在變動光照環境下的辨色穩定性遠超 RGB。

三大分量的意義

分量範圍物理意義對光照變化的敏感度
H（色相）$0° \sim 360°$顏色在色環上的角度位置。紅 ≈ 0°、綠 ≈ 120°、藍 ≈ 240°極低（幾乎不受光照影響）
S（飽和度）$0 \sim 1$（或 $0 \sim 255$）顏色的純度。$S=1$ 為鮮豔純色，$S=0$ 為灰色中等（極暗時飽和度會下降）
V（亮度）$0 \sim 1$（或 $0 \sim 255$）明暗程度。$V=0$ 為全黑極高（直接反映光照強度）

RGB 到 HSV 的轉換公式

以下是標準的轉換演算法。首先將 $R, G, B$ 正規化到 $[0, 1]$，然後依序計算 $V$、$S$、$H$：

function rgbToHsv(r, g, b) {
  r /= 255; g /= 255; b /= 255;
  const max = Math.max(r, g, b);
  const min = Math.min(r, g, b);
  const d = max - min;

  let h = 0;
  if (d !== 0) {
    if (max === r) h = 60 * (((g - b) / d) % 6);
    else if (max === g) h = 60 * ((b - r) / d + 2);
    else h = 60 * ((r - g) / d + 4);
  }
  if (h < 0) h += 360;

  const s = max === 0 ? 0 : d / max;
  const v = max;
  return { h, s, v }; // H:0-360, S:0-1, V:0-1
}

注意 觀察上表的 H 欄位：無論光照如何劇烈變化，色相角始終穩定在 $0° \sim 2°$ 之間。這意味著我們只要設定 H ∈ [350°, 10°]，就能在所有光照條件下穩定辨識紅色——這正是 HSV 的威力所在。
提示 在 HSV 遮罩中，除了限制 H（色相）外，還應同時設定 S（飽和度）的下限（如 S > 0.3）。這能有效過濾掉灰色與白色背景，避免誤判。

色環的連續性與斷點

HSV 的色相（Hue）被定義在一個 $0°$ 到 $360°$ 的圓環上。大多數顏色的辨識只需設定一個簡單的區間即可（如綠色 $H \in [90°, 150°]$）。然而，紅色恰好坐落在色環的起點/終點交界處——它的色相值橫跨 $350° \sim 360°$ 與 $0° \sim 10°$ 兩段。這個「斷點」是開發者最容易踩進的陷阱。

斷點問題的本質

色環在數學上是連續的（$360°$ 等於 $0°$），但在程式實作中，角度值被儲存為一個普通的數字。當你寫下 h > h_min && h < h_max 來判斷紅色時，如果 h_min = 350、h_max = 10，由於 $350 > 10$，這個條件永遠不會成立。

// 錯誤寫法：紅色永遠無法被偵測到
if (h > 350 && h < 10) { // 350 < h < 10 → 永遠為 false
  mask[i] = 255;
}

解決方案：雙區間 OR 判定

正確的做法是將紅色的色相判定拆成兩個獨立區間，用邏輯 OR 運算合併：

// 正確寫法：用 OR 處理色環斷點
function isRed(h, s, v) {
  const hueMatch = (h > 350 || h < 10); // 兩段區間
  const satMatch = (s > 0.3); // 排除灰色
  const valMatch = (v > 0.15); // 排除極暗
  return hueMatch && satMatch && valMatch;
}

警告 自動測試會餵入一個 $H = 358°$ 的紅色像素。如果你的遮罩函式只使用簡單的 h > min && h < max，這個測試會直接失敗。務必使用 OR 邏輯處理紅色的跨區判定。

多色追蹤與決策優先權

在「紅綠燈」場景中，系統需要同時運行紅色與綠色兩組 HSV 遮罩。當兩者的目標像素都超過面積門檻時，必須依照安全優先原則做出決策：紅色（停車）永遠優先於綠色（前進）。

// 安全優先的行為決策邏輯
if (redPixels > MIN_AREA) return "STOP";
else if (greenPixels > MIN_AREA) return "FOLLOW";
else return "IDLE";
提示 在實作 HSV 遮罩 UI 時，建議為每個 HSV 分量都提供獨立的滑桿控制（H_min / H_max / S_min / V_min），並在 Canvas 上即時渲染二值化遮罩。這能讓你快速找到最佳的過濾參數。

任務 3：紅綠燈行為優先權判定 (Behavior Decision Logic)

  • 目標：將「感知」轉化為「語義化的系統決策」，並考慮安全性優先權。
  • Classroom 部署建議：
    • 決策邏輯實作： 
      // 學生需實作：安全優先的決策
if (redPixels > threshold) return "STOP"; 
else if (greenPixels > threshold) return "FOLLOW";
    • 驗證方式：在畫面積前同時晃動紅板與綠板（模擬衝突情境）。系統必須能根據「安全優先」原則穩定輸出 STOP。
    • Tutor View (PR 審核)：檢查學員是否加入了「飽和度 (S)」與「亮度 (V)」的過濾，以防止淺灰色背景或反光引起的「虛假觸發 (False Positive)」。

3. 色彩感知導師點評標準 (HSV Perception Benchmarks)

此單元的價值在於 「對數位訊號物理特性的深刻理解」：

  • [ ] 邊界處理能力：代碼中是否正確處理了 Hue 的 $360 \to 0$ 循環特性？
  • [ ] 抗噪優化意識：是否利用 S (飽和度) 成功過濾掉了黑白灰等「無色」背景噪音？
  • [ ] 行為決策一致性：當數據發生閃動時，是否有實作濾波或計數確認，以防止控制指令頻繁切換產生的「抖動」？
