JSON 序列化與反序列化

📦 1. 為什麼要序列化？ (20 min)

物理限制：理解通訊媒介（藍牙、Wi-Fi）只能傳送一連串的「位元組（Bytes）」，無法直接傳送一個「JavaScript 物件」。

📤 序列化 (Serialization)

將記憶體中的物件狀態「壓平」成文字格式（如 JSON 字串）的過程。

📥 反序列化 (Deserialization)

將接收到的文字格式「重新組裝」成可操作物件的過程。

🪑 學習任務：家具郵寄比喻

想像你要郵寄一張椅子給朋友。你必須先拆解它並裝箱（序列化），朋友收到後再按照說明書組裝（反序列化）。

📜 2. JSON 格式規範與類型 (25 min)

合法格式：理解 JSON 支援的數據類型（字串、數字、布林值、陣列、物件）。

⚠️ 常見錯誤：雙引號

✔ {"speed": 50} // 正確

✘ {speed: 50} // 錯誤：Key 必須用雙引號

🔍 預習任務

觀察一段複雜的無人車配置 JSON，嘗試識別出哪些是「數值（Number）」，哪些是「嵌套物件（Nested Object）」。

🛠️ 3. 語法工具箱 (15 min)

🌐 JavaScript 端

  • JSON.stringify(obj)
  • JSON.parse(string)

🦾 C++ (ESP32) 端

  • serializeJson(doc, output)
  • deserializeJson(doc, input)

💡 節省頻寬的技巧

如何設計精簡的 Key 值？
例如使用 {"d": 10} 代替 {"distance": 10}。

學習目標：在線上編譯器中寫出一個能產生目標 JSON 格式的代碼片段。

進階課程 3 - 單元 2 JSON 序列化與反序列化實作

數據通訊實驗室：JSON 序列化與防禦性解析

任務 1：「封包工廠」：狀態物件序列化 (Packet Factory Lab)

角色扮演：前端資料處理員 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 物件封裝：在 JavaScript 中定義一個包含 battery, speed, status 的感測器物件。
  3. 序列化轉換：調用 JSON.stringify() 將物件轉換為字串，並在 Console 輸出其 Byte 長度。
  4. Push Final：提交具備「符合 RFC 規範」且結構完整的序列化範例。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 格式嚴謹度：產出的 JSON 字串是否符合雙引號規範（特別是 Key 部分）？
  • [ ] 資料完整性：物件中的所有屬性是否都正確轉化為對應的 JSON 數據類型？

任務 2：「解析挑戰」：防禦型 JSON 除錯 (Parsing Challenge)

角色扮演：軟體除錯工程師 | 難度：⭐️⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 異常注入：嘗試解析一組故意損毀（缺少括號或引號）的非法 JSON 字串。
  2. 捕獲實作：使用 try...catch 語法包裹 JSON.parse() 邏輯。
  3. 安全降級：當發生解析錯誤時，確保程式能自動返回一組「安全預設值」而非報錯中止。
  4. Push Final：提交具備「高度魯棒性」且能應對通訊雜訊的解析引擎。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 防崩潰機制：當面對損毀數據時，網頁 UI 是否能維持運作而不發生白屏或斷線？
  • [ ] 錯誤處理邏輯：Catch 區塊內是否提供了明確的錯誤提示或日誌記錄？

任務 3：效能優化：數據精簡賽 (Payload Optimization Plan)

角色扮演：系統架構師 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 瘦身計畫：將冗長的 Key（如 temperature）優化為縮寫（如 t）。
  2. 精度控制：實作對浮點數的截斷（例如保留一位小數），測量封包減輕的具體 Bytes。
  3. 效率紀錄：在 README 中紀錄優化前後的頻寬節省百分比與無線傳輸時延變化。
  4. Push Final：提交經過「極致優化」且兼顧可讀性的傳輸 Payload 方案。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 優化成效：最終封包體積是否達成 30% 以上的精簡目標？
  • [ ] 結構合理性：精簡後的 Key 是否仍具備足夠的唯一性與開發可讀性？

GitHub Classroom 詳細操作指南

本單元教學重點：數據封裝的嚴絲合縫。理解序列化在跨平台通訊中的角色，並培養防禦性編程習慣。

  1. 重點 1：序列化與通訊模型 - 強調「物件（Object）」是記憶體中的私有格式，「字串（String）」才是網路中的通用語。
  2. 重點 2：防護牆思維 - 在 2-2 任務中，引導學生思考：當通訊環境不安穩時，如何用程式邏輯建置最後一道防線。
  3. 重點 3：頻寬與記憶體的權衡 - 透過精簡任務，讓學生看見在嵌入式設備中，每一個 Byte 的節省對於時延與頻寬的巨大補償。
