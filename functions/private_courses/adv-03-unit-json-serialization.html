簡介

網頁與 ESP32 之間的溝通，本質上就是在傳送一串位元組。JSON 序列化是將你的控制指令與感測器數據轉換為雙方都能理解的文字格式的橋樑。 一個格式錯誤、一個多餘的逗號，都可能讓 JSON.parse() 直接拋出例外，導致整個通訊鏈路斷裂。 本模組將教你精確掌握 JSON 的格式規範、在 MCU 端高效序列化，以及在前端安全地反序列化與容錯處理。

情境
你讓 ESP32 透過 BLE 回傳感測器數據，格式是 {speed:50, angle:30}。網頁端呼叫 JSON.parse() 卻直接報錯：Unexpected token s in JSON。明明看起來很像 JSON 啊？原來問題出在 Key 沒有加雙引號——這在 JavaScript 物件語法中合法，但在 JSON 規範中是非法的。這類「差一點就對了」的陷阱，正是本模組要幫你系統性排除的。

為什麼要序列化？

通訊媒介（無論是藍牙、Wi-Fi 還是序列埠）只能傳送一連串的位元組（Bytes）——它們無法直接理解什麼是「JavaScript 物件」或「C++ struct」。序列化（Serialization）就是將記憶體中的結構化數據「壓平」成一段文字字串，使其能夠在物理通道上安全傳輸的過程。

序列化與反序列化的定義

序列化（Serialization）：將程式語言中的物件、陣列、結構等複雜數據型態，轉換為一段有規範的純文字字串（例如 JSON 格式）。這個過程就像把一張立體的椅子拆解、壓平後裝進紙箱——讓它能通過狹窄的郵遞通道。

反序列化（Deserialization）：在接收端將收到的文字字串還原回原始的數據結構，使程式能直接存取其中的欄位。就像朋友收到紙箱後，按照說明書把椅子重新組裝起來。

// 序列化：物件 → JSON 字串 → BLE 位元組
const sensorData = { speed: 80, angle: 45 };
const jsonStr = JSON.stringify(sensorData);
// → '{"speed":80,"angle":45}'

// 反序列化：BLE 位元組 → JSON 字串 → 物件
const received = JSON.parse('{"speed":80,"angle":45}');
console.log(received.speed); // → 80

為什麼 BLE 通訊需要 JSON？

在 ESP32 與網頁之間的 BLE 通訊中，雙方使用的程式語言完全不同（C++ vs JavaScript）。JSON 作為一種跨語言、跨平台的文字格式，恰好扮演了「通用翻譯官」的角色。以下表格比較了常見的序列化格式：

格式 可讀性 體積 跨語言支援 BLE 適用性
JSON 高（純文字） 中等 極廣泛 最佳選擇
Protocol Buffers 低（二進位） 極小 廣泛 需要額外函式庫
MessagePack 低（二進位） 小 中等 MCU 端支援有限
CSV / 自訂分隔符 中等 小 手動解析 不適合巢狀結構
備註 JSON 雖然不是體積最小的格式，但它是唯一在 JavaScript（JSON.stringify / JSON.parse）和 C++（ArduinoJson 函式庫）兩端都具備原生級支援的文字格式。在 BLE 頻寬受限的場景下，我們可以透過「縮短 Key 名稱」等技巧來進一步壓縮體積。
警告 一個常見的初學者錯誤是直接透過 BLE 發送 JavaScript 物件的 toString() 結果（例如 [object Object]），而非正確呼叫 JSON.stringify()。這會導致接收端拿到完全無法解析的亂碼。

JSON 格式規範與類型

JSON（JavaScript Object Notation）雖然源自 JavaScript，但它是一個獨立的數據交換標準（RFC 8259）。許多在 JavaScript 中合法的語法，在 JSON 規範中是非法的。精確掌握 JSON 的格式規則，是避免 JSON.parse() 拋出 SyntaxError 的關鍵。

JSON 支援的六種數據類型

類型 JSON 範例 說明
字串 (String) "hello" 必須使用雙引號包裹
數字 (Number) 42, 3.14, -7 整數或浮點數，不支援 NaN / Infinity
布林 (Boolean) true, false 全小寫，不可使用大寫或數字代替
空值 (Null) null 全小寫，代表「無值」
物件 (Object) {"key": "value"} Key 必須是雙引號字串，支援巢狀
陣列 (Array) [1, 2, 3] 有序值列表，元素可混合類型
重要 JSON 不支援以下 JavaScript 中常見的值：undefined、NaN、Infinity、-Infinity、函式（Function）、日期物件（Date）。如果嘗試對包含這些值的物件執行 JSON.stringify()，它們會被自動省略或轉換為 null。

常見格式錯誤

以下是初學者最容易犯的 JSON 格式錯誤，每一個都會導致 JSON.parse() 失敗：

// ✅ 正確的 JSON
{"speed": 50, "active": true, "mode": "auto"}

// ❌ Key 沒有雙引號（JavaScript 物件合法，JSON 非法）
{speed: 50}

// ❌ 使用單引號（JSON 只接受雙引號）
{'speed': 50}

// ❌ 結尾多了逗號（Trailing Comma）
{"speed": 50, "angle": 30,}

// ❌ 使用註解（JSON 不支援任何形式的註解）
{"speed": 50 /* 單位: cm/s */}
警告 在 ESP32 的 C++ 韌體中手動拼接 JSON 字串時（例如 String json = "{speed:" + String(val) + "}"），極容易遺漏 Key 的雙引號。強烈建議使用 ArduinoJson 函式庫，而非手動拼接，以確保產出的字串符合嚴格的 JSON 規範。

巢狀結構：無人車配置範例

JSON 支援物件與陣列的任意巢狀，讓你能以結構化方式描述複雜的設備狀態：

{
  "id": "car-01",
  "pos": { "x": 102, "y": -45 },   ← 巢狀物件
  "sensors": [30, 45, 60, 90], ← 陣列
  "active": true,              ← 布林值
  "error": null                 ← 空值
}
提示 開發過程中可使用線上 JSON 驗證工具（如 jsonlint.com）即時檢查你手動拼接的 JSON 是否合法。許多難以發現的解析錯誤，都能透過驗證工具在幾秒內定位。

語法工具箱

在 BLE 通訊的兩端——網頁（JavaScript）與 ESP32（C++）——各有一套 JSON 處理工具。掌握這些 API 的正確用法與常見陷阱，是實現穩定跨平台數據交換的基礎。

JavaScript 端：原生 JSON API

JavaScript 內建了兩個核心函式，無需安裝任何套件即可使用：

// 序列化：物件 → JSON 字串
const sensor = { speed: 80, angle: 45, active: true };
const json = JSON.stringify(sensor);
// → '{"speed":80,"angle":45,"active":true}'

// 反序列化：JSON 字串 → 物件（必須 try-catch 防護）
try {
  const data = JSON.parse(receivedString);
  console.log(data.speed); // → 80
} catch (e) {
  console.warn('JSON 解析失敗，使用安全預設值');
  const data = { speed: 0, angle: 0, active: false };
}
重要 JSON.parse() 在接收到殘缺或格式錯誤的字串時會直接拋出 SyntaxError。在 BLE 通訊中，封包遺失與截斷是常態，因此任何呼叫 JSON.parse() 的地方都必須包裹在 try-catch 中。

C++ (ESP32) 端：ArduinoJson 函式庫

ESP32 端推薦使用 ArduinoJson 函式庫來處理 JSON，它提供了安全且高效的序列化 / 反序列化 API：

// ESP32 C++：序列化
#include <ArduinoJson.h>

StaticJsonDocument<128> doc;
doc["speed"] = 80;
doc["angle"] = 45;
doc["active"] = true;

String output;
serializeJson(doc, output);
// output → '{"speed":80,"angle":45,"active":true}'

// ESP32 C++：反序列化
StaticJsonDocument<128> incoming;
DeserializationError err = deserializeJson(incoming, bleData);
if (err) {
  Serial.println("JSON 解析錯誤");
  return;
}
int spd = incoming["speed"]; // → 80
備註 StaticJsonDocument<N> 中的 N 是分配給 JSON 文件的記憶體大小（Bytes）。可使用 ArduinoJson 官方提供的 Assistant 工具（arduinojson.org/v6/assistant/）計算精確的所需容量。分配過小會導致數據截斷，過大則浪費 ESP32 寶貴的 RAM。

頻寬優化：精簡 Key 名稱

在 BLE 20 Bytes 的嚴格限制下，每一個字元都很珍貴。透過縮短 JSON 的 Key 名稱，可以顯著減少封包體積，甚至讓原本需要分段傳輸的數據縮減到單一封包內：

語義化 Key 精簡 Key 節省空間
"speed" (7 Bytes) "s" (3 Bytes) 4 Bytes
"angle" (7 Bytes) "a" (3 Bytes) 4 Bytes
"distance" (10 Bytes) "d" (3 Bytes) 7 Bytes
"battery" (9 Bytes) "b" (3 Bytes) 6 Bytes
// 優化前：47 Bytes（需要分 3 片）
{"speed":80,"angle":45,"distance":120,"battery":85}

// 優化後：25 Bytes（只需 2 片，減少 47%）
{"s":80,"a":45,"d":120,"b":85}
提示 精簡 Key 名稱時，務必在專案中維護一份對照表文件（例如 docs/key_mapping.md），記錄每個縮寫對應的完整語義。這樣在除錯時不會因為看到 "b":85 而搞不清楚它代表 battery 還是 bearing。

單元作業：JSON 序列化與反序列化 (JSON Serialization & Deserialization)

🎯 作業目標

在本單元中，你將學習如何將複雜的 JavaScript 物件轉換為標準的 JSON 字串（序列化），以及如何安全地將接收到的字串還原為物件（反序列化）。此外，我們還將探索如何在頻寬受限的環境下進行數據精簡。

🚀 任務說明

任務 1：封包工廠 - 狀態物件序列化

請實作 src/codec.js 中的 serialize() 函式。

  • 將傳入的感測器狀態物件轉換為 JSON 字串。
  • 確保包含巢狀物件（如 location: { x, y }）的資料能正確轉換。
  • 驗證輸出的字串符合標準 JSON 規範（雙引號、正確的逗號分隔）。

任務 2：解析挑戰 - 防禦型解析

實作 src/codec.js 中的 safeParse() 函式。

  • 接收可能損毀或格式錯誤的 JSON 字串。
  • 使用 try-catch 機制防止程式崩潰。
  • 當解析失敗時，應返回預設的安全狀態物件並記錄警告資訊。

任務 3：效能優化 - 數據精簡賽

在滿足通訊需求的前提下，盡可能縮減 JSON 封包的大小。

  • 縮短鍵名：例如將 temperature 改為 t。
  • 數值優化：控制浮點數精度，移除不必要的空格。

    • 防禦性邏輯檢核： 
      // 學生需實作：Try-Catch 保護
try {
    return JSON.parse(inputString);
} catch (e) {
    console.warn("解析失敗，採用安全預設值");
    return { speed: 0, status: "error" }; // 返回安全狀態

    • 驗證方式：透過自動化腳本持續餵入損毀數據，檢核網頁 UI 是否能顯示「錯誤狀態碼」而不是直接崩潰白屏。

任務 3：效能優化：數據精簡賽 (Payload Optimization Plan)

  • 目標：在極度受限的頻寬下，練習數據精簡術.
  • Classroom 部署建議：
    • 精簡策略檢核：要求學員將 Key 名稱縮寫（如 temperature -> t），並控制浮點數精度（如 long: 121.123456 -> l: 121.1）。
    • 效能評比：學員需在 README 中紀錄優化程度。導師在 PR (Pull Request) 中審核：雖然 Key 變短了，但其語義是否仍具備基本的可讀性與唯一性？這展現了在「機器效率」與「人類維護性」之間的權衡判斷。

3. 資料交換導師點評標準 (Data Exchange Benchmarks)

此單元的價值在於 「對通訊魯棒性的深度執念」：

  • [ ] 異常捕獲完整性：是否考慮到了 null 數據或空字串的特殊情況？
