JSON 序列化與反序列化

📦 1. 為什麼要序列化？ (20 min)

物理限制：理解通訊媒介（藍牙、Wi-Fi）只能傳送一連串的「位元組（Bytes）」，無法直接傳送一個「JavaScript 物件」。

📤 序列化 (Serialization)

將記憶體中的物件狀態「壓平」成文字格式（如 JSON 字串）的過程。

📥 反序列化 (Deserialization)

將接收到的文字格式「重新組裝」成可操作物件的過程。

🪑 學習任務：家具郵寄比喻

想像你要郵寄一張椅子給朋友。你必須先拆解它並裝箱（序列化），朋友收到後再按照說明書組裝（反序列化）。

📜 2. JSON 格式規範與類型 (25 min)

合法格式：理解 JSON 支援的數據類型（字串、數字、布林值、陣列、物件）。

⚠️ 常見錯誤：雙引號

✔ {"speed": 50} // 正確

✘ {speed: 50} // 錯誤：Key 必須用雙引號

🔍 預習任務

觀察一段複雜的無人車配置 JSON，嘗試識別出哪些是「數值（Number）」，哪些是「嵌套物件（Nested Object）」。

🛠️ 3. 語法工具箱 (15 min)

🌐 JavaScript 端

  • JSON.stringify(obj)
  • JSON.parse(string)

🦾 C++ (ESP32) 端

  • serializeJson(doc, output)
  • deserializeJson(doc, input)

例如使用 {"d": 10} 代替 {"distance": 10}。

進階課程 3 - 單元 2 JSON 序列化與反序列化實作

單元作業：JSON 序列化與反序列化 (JSON Serialization & Deserialization)

🎯 作業目標

在本單元中，你將學習如何將複雜的 JavaScript 物件轉換為標準的 JSON 字串（序列化），以及如何安全地將接收到的字串還原為物件（反序列化）。此外，我們還將探索如何在頻寬受限的環境下進行數據精簡。

🚀 任務說明

任務 1：封包工廠 - 狀態物件序列化

請實作 src/codec.js 中的 serialize() 函式。

  • 將傳入的感測器狀態物件轉換為 JSON 字串。
  • 確保包含巢狀物件（如 location: { x, y }）的資料能正確轉換。
  • 驗證輸出的字串符合標準 JSON 規範（雙引號、正確的逗號分隔）。

任務 2：解析挑戰 - 防禦型解析

實作 src/codec.js 中的 safeParse() 函式。

  • 接收可能損毀或格式錯誤的 JSON 字串。
  • 使用 try-catch 機制防止程式崩潰。
  • 當解析失敗時，應返回預設的安全狀態物件並記錄警告資訊。

任務 3：效能優化 - 數據精簡賽

在滿足通訊需求的前提下，盡可能縮減 JSON 封包的大小。

  • 縮短鍵名：例如將 temperature 改為 t。
  • 數值優化：控制浮點數精度，移除不必要的空格。
  • 在下方「效能對比表」中記錄你的優化結果。

2. 作業任務部署細節

任務 1：「封包工廠」：狀態物件序列化 (Packet Factory Lab)

  • 目標：掌握將記憶體資料固化為標準傳輸格式的能力。
  • Classroom 部署建議：
    • 代碼檢核：要求學員實作將包含 nested object（如 pos: {x: 1, y: 2}）的複雜物件序列化。
    • Autograding：使用 JSON.parse() 重新解析學員產出的字串，驗證數據類型（如數字是否被誤轉為字串）是否保持原樣。

任務 2：「解析挑戰」：防禦型 JSON 除錯 (Parsing Challenge)

  • 目標：建立高度魯棒的接收端邏輯，防止系統被髒數據崩潰。
  • Classroom 部署建議：
    • 防禦性邏輯檢核： 
      // 學生需實作：Try-Catch 保護
try {
    return JSON.parse(inputString);
} catch (e) {
    console.warn("解析失敗，採用安全預設值");
    return { speed: 0, status: "error" }; // 返回安全狀態
}
    • 驗證方式：透過自動化腳本持續餵入損毀數據，檢核網頁 UI 是否能顯示「錯誤狀態碼」而不是直接崩潰白屏。

任務 3：效能優化：數據精簡賽 (Payload Optimization Plan)

  • 目標：在極度受限的頻寬下，練習數據精簡術.
  • Classroom 部署建議：
    • 精簡策略檢核：要求學員將 Key 名稱縮寫（如 temperature -> t），並控制浮點數精度（如 long: 121.123456 -> l: 121.1）。
    • 效能評比：學員需在 README 中紀錄優化程度。導師在 PR (Pull Request) 中審核：雖然 Key 變短了，但其語義是否仍具備基本的可讀性與唯一性？這展現了在「機器效率」與「人類維護性」之間的權衡判斷。

3. 資料交換導師點評標準 (Data Exchange Benchmarks)

此單元的價值在於 「對通訊魯棒性的深度執念」：

  • [ ] 異常捕獲完整性：是否考慮到了 null 數據或空字串的特殊情況？
  • [ ] 數據類型嚴謹度：在 C++ (ESP32) 與 JS 之間傳輸時，是否發生了浮點數精度丟失（Precision Loss）？
  • [ ] 通訊開銷意識：是否能解釋為什麼在 115200 波特率下，節省 50 Bytes 的 JSON 長度對系統響應速度有巨大的貢獻？
