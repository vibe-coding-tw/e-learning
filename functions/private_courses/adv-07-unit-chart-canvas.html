🖌️ 1. 渲染原理：畫布 vs. 物件 (25 min)

Canvas 是一塊即時畫布，你必須手動下指令「抹除再重繪」。而 Chart.js 則是基於 Canvas 的高度封裝，讓你專注於數據本身而非座標計算。

Comparison Architecture

  • 低階 Canvas：極致性能，適合處理數萬個頻繁跳動的像素點。
  • 高階 Chart.js：高度語意化，自動處理座標軸、縮放與適應性佈局。

⚡ 3. 更新頻率與視覺感知

人眼難以感知超過 60Hz 的數值跳變。學習如何使用 requestAnimationFrame 配合螢幕刷新率，並關閉不必要的圖形動畫以減少 CPU 負載。

Efficiency Checklist

  • 停用動畫 (Animation: false)：在即時圖表中，過度動畫會造成渲染卡頓與數據延遲。
  • 分組更新 (Batching)：如果數據爆量，則每 5-10 筆數據才刷新一次 UI。
  • 滾動軸優化：固定 X 軸的顯示跨度，維持視覺上的連續性。

✅ 任務一：「滑動視窗」動態圖表

目標：在 js/chart_logic.js 中實作滑動視窗（Sliding Window）緩衝邏輯，讓圖表始終只顯示最近 50 筆數據，並且不能隨著時間拉長而佔用更多記憶體。

需求規格

  • 呼叫 updateBuffer(newValue) 時，將新值加入 dataBuffer 陣列尾端。
  • 若陣列長度超過 50，移除最舊的一筆（使用 shift()）。
  • 呼叫 chart.update('none') 以關閉動畫，維持高更新頻率。

提交驗證

  • [ ] 執行 npm test，Sliding Window 相關測試全部通過（陣列長度恆 ≤ 50）。
  • [ ] 附上波形截圖：X 軸隨時間向右滾動，畫面維持流暢。

✅ 任務二：多軸同步監控儀

目標：在 js/chart_logic.js 中擴充圖表，同時顯示「目標值 (Setpoint)」與「回授值 (Feedback)」兩條曲線，讓 PID 追隨過程一目了然。

需求規格

  • Y 軸範圍固定為 0 ~ 255，禁止使用 Chart.js 的自動縮放。
  • 兩條線的 label 須使用工程語意命名（如 Setpoint、Feedback），不得使用 line1、data2 等無意義名稱。
  • 啟用 Tooltip，讓滑鼠懸停時能同時顯示兩條線的當前值。

提交驗證

  • [ ] 附上截圖或影片：手動模擬一個階躍輸入，展示目標線（方波）與回授線（響應曲線）在同一比例尺下的收斂過程。

✅ 任務三：原生 Canvas 高頻示波器

目標：在 js/raw_canvas.js 中，不依賴任何圖表庫，直接以原生 Canvas API 繪製波形，並挑戰 100Hz 的刷新率。

需求規格

  • 使用 requestAnimationFrame 驅動渲染循環，禁止使用 setInterval。
  • 每幀呼叫 clearRect 清除舊畫面，再用 lineTo 重繪所有數據點。
  • 注意 Canvas 座標系：Y 軸向下為正，需進行翻轉換算： 
    let y_pixel = canvasHeight - (value / 255 * canvasHeight);
ctx.lineTo(x_pixel, y_pixel);

提交驗證

  • [ ] 在 docs/tuning_view.md 中記錄：以 100Hz 刷新時，瀏覽器 FPS 表現為何？是否出現卡頓？
  • [ ] 執行 npm test，requestAnimationFrame 相關檢核通過。
