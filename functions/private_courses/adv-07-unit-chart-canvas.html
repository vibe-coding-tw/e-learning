渲染原理：畫布 vs. 物件

在即時數據視覺化的世界中，有兩種截然不同的繪圖模型：即時模式 (Immediate Mode) 與保留模式 (Retained Mode)。理解這兩者的差異，是開發高效能圖表的基礎。

即時模式：HTML5 Canvas API

HTML5 的 <canvas> 元素是一塊「像素畫布」，採用即時模式渲染。這意味著每一幀的繪圖都是一次性的：你必須手動清除畫布 (clearRect)，然後重新繪製所有內容。Canvas 本身不會記住你畫了什麼——一旦像素被描繪到畫布上，它就變成了靜態的點陣圖。

這種模式的優勢在於極致的控制力與性能。由於沒有物件管理的額外負擔，Canvas 可以輕鬆處理數萬個數據點的高頻渲染，非常適合示波器、頻譜分析器等需要像素級控制的應用場景。

Canvas 繪圖基本流程

// 取得 Canvas 繪圖上下文
const canvas = document.getElementById('myCanvas');
const ctx = canvas.getContext('2d');

function draw(data) {
  // 步驟 1：清除整個畫布
  ctx.clearRect(0, 0, canvas.width, canvas.height);

  // 步驟 2：開始新路徑並設定樣式
  ctx.beginPath();
  ctx.strokeStyle = '#0078d4';
  ctx.lineWidth = 2;

  // 步驟 3：逐點繪製折線（注意 Y 軸翻轉）
  for (let i = 0; i < data.length; i++) {
    const x = (i / data.length) * canvas.width;
    const y = canvas.height - (data[i] / 255 * canvas.height);
    i === 0 ? ctx.moveTo(x, y) : ctx.lineTo(x, y);
  }

  // 步驟 4：將路徑描繪到畫布
  ctx.stroke();
}

保留模式：Chart.js 抽象層

Chart.js 是一個建構在 Canvas 之上的圖表函式庫，採用保留模式的設計理念。你只需要宣告「資料是什麼」和「圖表長什麼樣」，Chart.js 會自動處理座標軸的計算、刻度的排列、圖例的繪製，以及響應式的尺寸調整。

當資料發生變化時，你只需更新資料陣列並呼叫 chart.update()，Chart.js 會自動計算差異並重新渲染——開發者完全不需要操心像素座標的換算。

Chart.js 初始化範例

const chart = new Chart(ctx, {
  type: 'line',
  data: {
    labels: timeLabels,
    datasets: [{
      label: '溫度 (°C)',
      data: sensorData,
      borderColor: '#0078d4',
      tension: 0.3
    }]
  },
  options: {
    animation: false,
    scales: {
      y: { min: 0, max: 100 }
    }
  }
});

兩種模式比較

比較項目 Canvas API（即時模式） Chart.js（保留模式）
抽象層級 低階——直接操作像素與路徑 高階——宣告資料即可繪圖
座標計算 開發者手動換算 自動處理座標軸與刻度
響應式佈局 需自行監聽 resize 事件 內建自動縮放
渲染性能 極高，適合 >10,000 點 中高，適合 ≤2,000 點
互動功能 需自行實作 hit-testing 內建 Tooltip、Legend、Zoom
學習曲線 較陡——需理解圖形學基礎 較平緩——查閱 API 文件即可
適用場景 示波器、遊戲、粒子系統 儀表板、報表、監控面板
備註 在物聯網即時監控的場景中，大多數開發者會優先選擇 Chart.js，因為它能以最少的程式碼實現專業級的圖表。只有當數據點數量極大（例如每秒 100+ 筆且需同時顯示上千點）或需要自訂繪圖效果時，才需要回歸原生 Canvas API。
效能提示 使用 Chart.js 時，務必將 animation 設為 false，或在呼叫 chart.update('none') 時傳入 'none' 參數。在即時數據流的場景下，每次資料更新的過場動畫不僅沒有意義，還會嚴重拖慢渲染速度，導致數據顯示延遲。

滑動窗口 (Sliding Window) 數據緩衝

當感測器持續傳送數據時，資料量會隨時間線性增長。如果將所有歷史數據都保留在陣列中並餵給圖表，瀏覽器的記憶體用量與渲染時間都會不斷攀升，最終導致頁面凍結甚至崩潰。

滑動窗口 (Sliding Window) 是一種經典的緩衝策略：維持一個固定長度的陣列，每當新數據抵達時，將它推入陣列尾端 (push)，同時移除最舊的數據 (shift)。這樣一來，陣列長度始終恆定，記憶體用量穩定可控，圖表也能呈現出自然的「向右滾動」效果。

核心實作模式

滑動窗口的實作只需三行核心程式碼。以下範例展示如何將此模式整合到 Chart.js 的即時更新流程中：

const BUFFER_SIZE = 50;
const dataBuffer = [];
const labelBuffer = [];

function updateBuffer(newValue) {
  // 1. 將新數據推入尾端
  dataBuffer.push(newValue);
  labelBuffer.push(new Date().toLocaleTimeString());

  // 2. 若超過上限，移除最舊的一筆
  if (dataBuffer.length > BUFFER_SIZE) {
    dataBuffer.shift();
    labelBuffer.shift();
  }

  // 3. 通知 Chart.js 重繪（關閉動畫）
  chart.update('none');
}
記憶體洩漏警告 如果忘記呼叫 shift()，陣列會無限增長。在長時間運行的物聯網儀表板中（例如 24 小時不間斷監控），每秒 10 筆數據累計一天就是 864,000 筆。這不僅會耗盡瀏覽器的可用記憶體，Chart.js 在處理如此龐大的資料集時，每次 update() 的渲染時間也會從毫秒級退化到秒級。

緩衝區大小與效能關係

選擇合適的緩衝區大小 (BUFFER_SIZE) 是一項需要權衡的設計決策。太小會損失歷史上下文，太大則增加渲染負擔。下表列出常見的緩衝區大小及其對應的性能特徵：

緩衝區大小 可見時間跨度 (10Hz) 記憶體用量 Chart.js 渲染延遲 適用場景
20 2 秒 極低 < 1ms 嵌入式小元件、即時數值監控
50 5 秒 低 1–2ms 馬達 PWM 波形、PID 調校
200 20 秒 中 3–8ms 環境感測趨勢、溫濕度走勢
500 50 秒 中高 8–15ms 長期趨勢分析（需注意 CPU）
1000+ >100 秒 高 >20ms 不建議——考慮降採樣或分頁

完整整合範例

以下展示一個完整的 WebSocket 即時數據接收與滑動窗口更新的流程：

const ws = new WebSocket('ws://esp32.local/sensor');

ws.onmessage = (event) => {
  const sensorValue = JSON.parse(event.data).temperature;
  updateBuffer(sensorValue);
};
設計提示 緩衝區大小的選擇取決於兩個因素：數據頻率與使用者需要觀察的時間範圍。公式為：BUFFER_SIZE = 數據頻率 (Hz) × 期望顯示秒數。例如，若感測器每秒送 10 筆數據，而使用者需要看到最近 5 秒的趨勢，則 BUFFER_SIZE = 10 × 5 = 50。

更新頻率與視覺感知

人類視覺系統對畫面更新的感知存在生理上限。研究顯示，大多數人能流暢感知的畫面更新頻率約為 60 幀/秒 (60fps)，對應約 16.67 毫秒的幀間隔。超過這個頻率的數值跳動，人眼幾乎無法分辨。因此，在即時圖表的設計中，盲目追求超高更新率不僅沒有視覺上的改善，反而會白白消耗 CPU 資源。

在瀏覽器環境中，控制畫面更新頻率主要有兩種方式：setInterval 計時器與 requestAnimationFrame API。兩者的設計理念與效能表現有著本質差異。

setInterval vs requestAnimationFrame

比較項目 setInterval requestAnimationFrame
觸發機制 固定時間間隔觸發回呼 由瀏覽器在下一次重繪前觸發
與螢幕刷新同步 否——可能在兩幀之間觸發，導致畫面撕裂 是——與顯示器垂直同步 (vsync)
背景分頁行為 持續執行，浪費 CPU 自動暫停，節省資源
精確度 受事件迴圈延遲影響，最小間隔 4ms 由瀏覽器優化調度，精確對齊 vsync
效能影響 容易造成任務堆積與卡頓 瀏覽器自動節流，確保流暢
適用情境 非視覺定時任務（輪詢、心跳） 所有涉及畫面更新的渲染迴圈

使用 requestAnimationFrame 的渲染迴圈

以下範例展示如何使用 requestAnimationFrame 建立一個與螢幕刷新率同步的渲染迴圈，用於驅動原生 Canvas 的即時波形繪製：

let animationId;

function renderLoop() {
  // 清除畫布
  ctx.clearRect(0, 0, canvas.width, canvas.height);

  // 繪製波形
  ctx.beginPath();
  for (let i = 0; i < dataBuffer.length; i++) {
    const x = (i / dataBuffer.length) * canvas.width;
    const y = canvas.height - (dataBuffer[i] / 255 * canvas.height);
    i === 0 ? ctx.moveTo(x, y) : ctx.lineTo(x, y);
  }
  ctx.strokeStyle = '#0078d4';
  ctx.stroke();

  // 排程下一幀——瀏覽器會自動對齊螢幕刷新率
  animationId = requestAnimationFrame(renderLoop);
}

// 啟動渲染迴圈
renderLoop();

// 需要停止時：cancelAnimationFrame(animationId);

Chart.js 關閉動畫

Chart.js 預設會為每次 update() 加上平滑的過場動畫——在靜態報表中這能提升視覺體驗，但在即時數據流的場景下卻是災難性的。當數據以 10Hz 或更高頻率湧入時，上一次的動畫還沒播完，下一筆數據就已經到達，導致動畫佇列堆積、畫面嚴重延遲。

// 方法一：在建立圖表時全域關閉動畫
const chart = new Chart(ctx, {
  type: 'line',
  data: { /* ... */ },
  options: {
    animation: false,
    responsive: true
  }
});

// 方法二：在每次更新時指定不使用動畫
chart.update('none');
重要 在即時圖表中，animation: false 不是「優化建議」，而是必要設定。如果你的圖表每秒更新超過 2 次，卻沒有關閉動畫，使用者看到的圖表將永遠落後於實際數據，失去「即時」的意義。這是從「做網頁」跨入「做工程系統」的關鍵認知轉變。

分組更新 (Batching) 策略

在極高頻的數據流中（例如每秒 100 筆以上），即使關閉了動畫，逐筆呼叫 chart.update() 仍然可能造成 CPU 瓶頸。更有效的做法是將多筆數據蒐集到批次中，再一次性更新圖表：

let pendingData = [];
let rafScheduled = false;

function onSensorData(value) {
  pendingData.push(value);

  if (!rafScheduled) {
    rafScheduled = true;
    requestAnimationFrame(() => {
      pendingData.forEach(v => updateBuffer(v));
      pendingData = [];
      chart.update('none');
      rafScheduled = false;
    });
  }
}
效能提示 分組更新結合 requestAnimationFrame 是最佳實踐：無論感測器數據以多快的速度湧入，畫面最多只在每次螢幕重繪時更新一次（通常為 60fps）。這種模式下，100Hz 的數據流與 1000Hz 的數據流對 CPU 的渲染負擔幾乎相同，因為所有數據都被合併到每 16.67ms 的單次重繪中。
