⏰ 1. 什麼是硬體定時器？

處理器內部「小鬧鐘」。時間一到，「強行暫停」主程式 (Loop) 執行緊急任務。

Active vs Passive

millis() - 被動

loop 中手動檢查。易受阻塞失效。

Timer - 主動

強迫中斷 loop。精度極高。

ESP32 Hardware Resource

\[4 \times Hardware Timers\]

Resolution: Microseconds (\(\mu s\))

🚫 2. ISR 三大禁忌

中斷代碼必須「快進快出」。執行太久會導致系統 Watchdog 重啟甚至崩潰。

NO delay() & Serial.print()

NO Complex Calculation

IRAM ATTRIBUTE 
// ISR 存放在 RAM 獲取極速響應
void IRAM_ATTR onTimer() {
  digitalWrite(LED_P, !digitalRead(LED_P));
}

// Fast in, Fast out.

🛡️ 3. volatile 變數共享

當 Loop 與中斷共用變數時，告訴編譯器：不准優化暫存。

Memory Visibility

  • volatile 修飾字：每次都從內存讀取，不可讀取暫存器副本。
  • 數據完整性：防止主程式讀到「舊時空」過期數據。
  • 臨界區保護：多位元數值需原子操作。

Data Integrity

精通硬體定時器與 ISR 中斷服務 (basic-09-unit-hardware-timer)

本單元將帶領你建立「實時系統 (Real-time System)」的核心思維。你將學習如何利用硬體定時器超越 millis() 的限制，並掌握 ISR (中斷服務程序) 的三大禁忌與 volatile 記憶體可見性防護。

🎯 學習目標

  1. 理解硬體定時器的「優先權搶佔 (Priority Preemption)」機制。
  2. 掌握 ISR 的「快進快出 (Fast-In, Fast-Out)」原則。
  3. 正確使用 volatile 關鍵字與臨界區 (Critical Section) 保護共享變數。
  4. 實作硬體級的 Watchdog 安全機制。

🚀 作業任務

任務 1：精確頻率：硬體定時器與閃燈

  • 目標：配置 1Hz 的硬體定時器控制 LED。
  • 實驗：在 loop() 中加入 delay(5000)，觀察 LED 是否依然精準閃爍。
  • 驗證：確保使用了 IRAM_ATTR 且不受主程式阻塞影響。

任務 2：轉速計實作：馬達 Encoder 採樣

  • 目標：配置 100ms 定時器作為採樣點，計算 RPM。
  • 關鍵：所有共享變數必須宣告為 volatile。
  • 保護：使用 portENTER_CRITICAL() 確保讀取數據的原子性。

任務 3：安全保護：硬體級 Watchdog

  • 目標：實作失效安全 (Fail-safe) 機制。
  • 邏輯：定時器每 500ms 檢查計數，主程式需及時「餵狗」。
  • 驗證：若主程式死鎖，ISR 必須能主動切斷動力。

⚠️ ISR 代碼規範 (Grading Standards)

  • [ ] 禁用法令：ISR 內絕不可出現 Serial.print() 或 delay()。
  • [ ] 記憶體防護：共享變數是否正確使用 volatile 與臨界區保護？
  • [ ] 快進快出：ISR 邏輯是否保持簡潔？

📁 程式庫結構

  • src/timer_isr.cpp：ISR 實作與變數維護。
  • include/types.h：共享數據結構定義。
  • docs/Stress_Test.md：記錄你的實驗觀察報告。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template Library) 配置建議

硬體定時器作業的核心在於「頻率穩定性」與「共享記憶體安全」，範本應包含定時器框架與壓力測試工具，建議包含：

  • 📂 src/timer_tasks.cpp：定時器邏輯實作區。學員在此撰寫 timerBegin() 配置，並實作帶有 IRAM_ATTR 屬性的中斷回呼函式。
  • 📂 include/shared_memory.h：變數共享規範。引導學員正確宣告 volatile 變數與臨界區 (Critical Section) 保護結構。
  • 📂 README.md：壓力實驗日誌。提供表格讓學員記錄：當 loop() 函數被一個無限循環阻塞時，硬體定時器任務是否依然能維持精確執行。
  • 📂 docs/ISR_FLOW.md：中斷流程圖。要求學員繪製中斷「搶佔」主程式的過程，並標註為何其 ISR 代碼符合「快進快出 (Fast-In, Fast-Out)」原則。

2. 作業任務部署細節

任務 1：精確頻率：硬體定時器與閃燈 (Precision Timer Blink Lab)

  • 目標：理解「硬體級搶佔」的本質，體會定時器如何無視軟體阻塞，強行執行任務。
  • Classroom 部署建議：
    • 頻率固定化：配置 1Hz 的硬體定時器控制 LED。
    • 阻塞實驗：技術審核重點。在 loop() 中加入 delay(5000) 或複雜迴圈。要求學員觀察 LED 是否依然以 1 秒為週期精準閃爍？
    • 驗證要點：是否正確使用了 IRAM_ATTR？閃爍頻率是否真的完全不受 loop() 程式碼長短的影響？

任務 2：轉速計實作：馬達 Encoder 採樣 (Tachometer & RPM Lab)

  • 目標：掌握「等間隔採樣 (Sampling Window)」技術，並正確處理中斷與主程式間的數據競爭 (Race Condition)。
  • Classroom 部署建議：
    • 採樣窗口設計：配置 100ms 的硬體定時器作為採樣點。
    • 變數共享防護：核心關鍵點。所有在 ISR 中增加、在主程式讀取的計數變數，必須添加 volatile 修飾字。
    • 原子操作 (Atomic)：實作 portENTER_CRITICAL() 讀取數據，確保在讀取 16-bit 以上變數時不會被中斷截斷。
    • 驗證要點：RPM 數值是否跳動過大？代碼中是否包含了正確的臨界區保護邏輯？

任務 3：安全保護：硬體級 Watchdog (Watchdog Guard Lab)

  • 目標：打造工業級的「失效安全 (Fail-safe)」機制。當主程式崩潰或網路失聯時，硬體定時器應主動奪權並切斷動力。
  • Classroom 部署建議：
    • 心跳監控邏輯：定時器每隔 500ms 檢查一次計數。主程式必須在期限內「餵狗」重置計數。
    • 強制煞車中斷：若計數歸零，ISR 直接調用馬達暫停函式（不經過 loop 的判斷）。
    • 崩潰模擬：提交一個會在 10 秒後陷入死鎖的代碼，驗證硬體是否能自動執行緊急制動。
    • 驗證要點：制動動作是否在 ISR 中完成？系統是否能在軟體徹底掛掉的情況下依然保護物理安全？

3. 多工穩定度與 ISR 代碼規範點評標準 (Multitasking & ISR Standards)

此單元的價值在於 「對系統執行層級的洞察力與對數據一致性的嚴謹守護」：

  • [ ] 中斷規範遵從度：ISR 內是否有違禁的 Serial.print() 或 delay()？
  • [ ] 數值存取原子性：是否深刻理解了 volatile 與臨界區在保護「跨時空數據」中的角色？
  • [ ] 失效安全性 (Failsafe)：系統設計是否預見了軟體可能故障，並留下了硬體的「最終執行權」？
