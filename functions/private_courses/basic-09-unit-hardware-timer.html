簡介

當你需要「精確到微秒」的定時觸發——例如固定頻率的 PID 控制迴圈或感測器取樣——軟體 millis() 就不夠用了。 ESP32 內建硬體定時器，能透過中斷服務常式（ISR）在指定週期自動觸發，完全不受主迴圈負載影響。 本單元教你設定硬體 Timer、撰寫安全的 ISR，並理解 ISR 中的限制與注意事項。

情境
你的循線車需要每 10ms 精確讀取一次光感測器陣列，再即時計算 PID 修正量。 如果把讀取邏輯放在 loop() 裡用 millis() 計時，任何一段阻塞程式碼都會導致取樣抖動。 掛載硬體定時器後，無論主程式在忙什麼，每 10ms ISR 都會準時觸發——穩定性天差地別。

什麼是硬體定時器？

硬體定時器（Hardware Timer）是微控制器內部的一個獨立計數電路。它由處理器的時鐘訊號驅動，以固定頻率遞增或遞減一個暫存器。當計數器達到預設值時，它會自動產生一個中斷訊號，強制 CPU 暫停當前工作、跳轉到你指定的中斷服務常式（ISR）執行。

與 millis() 相比，硬體定時器的核心差異在於「主動」vs「被動」：

  • millis() 是被動的——你必須在 loop() 中主動檢查時間差。如果 loop 被某段長程式碼阻塞，檢查就會延遲。
  • 硬體定時器是主動的——無論 CPU 正在執行什麼程式碼，時間一到就「搶佔」CPU 的控制權，跳轉執行 ISR。

ESP32 的定時器資源

ESP32 內建 4 組通用硬體定時器（Timer 0~3），分為 2 組（Group 0 和 Group 1），每組各 2 個。這些定時器的特性如下：

特性 規格
計數器位元數 64-bit（可計數極長時間）
基準時鐘 APB Clock = 80 MHz
預分頻器 (Prescaler) 16-bit（1 ~ 65,536）
最小解析度 12.5 ns（80MHz 下）
實用精度 1 μs（prescaler = 80 時）
中斷觸發 警報值匹配時自動產生 IRQ
Prescaler 的作用 預分頻器將 80 MHz 的時鐘除頻後送入計數器。設定 prescaler = 80 時，計數器每 1 微秒（μs）加 1，非常方便換算。設定 prescaler = 8000 時，每 100μs 加 1，適合較長週期的定時。

配置硬體定時器（Arduino-ESP32）

以下範例配置 Timer 0 以 1 秒為週期觸發中斷：

hw_timer_t *timer = NULL; void setup() { // 第 1 步：初始化 Timer 0，prescaler = 80 → 每 1μs 計數一次 timer = timerBegin(0, 80, true); // timer_id, prescaler, count-up // 第 2 步：綁定 ISR 函式 timerAttachInterrupt(timer, &onTimer, true); // 第 3 步：設定警報值 = 1,000,000 μs = 1 秒 timerAlarmWrite(timer, 1000000, true); // 值, 自動重載 // 第 4 步：啟動定時器 timerAlarmEnable(timer); }

millis() vs 硬體定時器比較

面向 millis() 軟體計時 Hardware Timer
觸發方式 被動：loop 中手動檢查 主動：硬體中斷搶佔
受阻塞影響 是 否
精度 毫秒級（受 loop 速度影響） 微秒級（硬體保證）
適用場景 LED 閃爍、定期回報 PID 控制、精確取樣、安全監控

何時該用硬體定時器？ 當你的任務對時間精度有嚴格要求（如固定頻率 PID、精確的 PWM 產生、或安全相關的超時監控），且不容許被主程式中的任何延遲干擾時，就該使用硬體定時器。一般的 LED 閃爍或狀態回報，用 millis() 就足夠了。

ISR 三大禁忌

中斷服務常式（ISR）是在中斷觸發時由硬體「強行」呼叫的函式。在 ISR 執行期間，正常的程式流程被暫停，其他中斷也可能被遮罩。因此 ISR 必須遵守「快進快出」原則——只做最必要的操作，然後立即返回。

如果 ISR 執行時間過長，會導致：系統看門狗 (WDT) 觸發重啟、其他中斷被延遲、主程式的執行時間被壓縮。以下是 ESP32 ISR 中的三條鐵律。

禁忌一：禁止使用 delay() 或任何阻塞函式

delay() 依賴系統 tick 中斷來計時，但在 ISR 內部，中斷已被禁用——呼叫 delay() 會導致程式永久卡死（因為計時中斷永遠不會觸發）。同理，任何依賴中斷的阻塞操作（如 pulseIn()、Wire.requestFrom()）都不可使用。

禁忌二：禁止使用 Serial.print() 或任何 I/O 函式

Serial.print() 需要操作 UART 緩衝區，涉及中斷驅動的佇列管理。在 ISR 內呼叫會造成遞迴中斷衝突或資料損毀。同理，WiFi、BLE、SPI 等通訊函式也絕對禁止在 ISR 中使用。

禁忌三：禁止複雜運算或長時間邏輯

浮點數運算、迴圈遍歷、動態記憶體分配（malloc/new）都不應出現在 ISR 中。ESP32 的 Xtensa 核心在 ISR 內的浮點運算會觸發額外的上下文保存開銷，嚴重影響效能。

違反後果 違反以上規則的常見症狀包括：系統無預警重啟（WDT reset）、Serial Monitor 出現亂碼「Guru Meditation Error」、程式行為不可預測。這些問題非常難以除錯，因為症狀可能與觸發原因在時間上分離。

ISR 的正確寫法

ISR 的標準模式是：在中斷內只設定旗標 (flag)，在 loop() 中檢查旗標並執行實際工作。

volatile bool timerFired = false; // ✅ 正確的 ISR：只做最少的事 void IRAM_ATTR onTimer() { timerFired = true; // 設定旗標，立即返回 } void loop() { if (timerFired) { timerFired = false; // 在這裡安全地執行複雜操作 Serial.println("Timer triggered!"); readSensorData(); } }

IRAM_ATTR 屬性的必要性

ESP32 的程式碼預設儲存在外部 Flash 記憶體中，透過 Cache 載入 CPU 執行。但中斷發生時，若 ISR 的程式碼恰好不在 Cache 中，就需要從 Flash 載入——這會產生不可預測的延遲。

加上 IRAM_ATTR 屬性，強制將 ISR 函式存放在內部 RAM（IRAM），確保中斷觸發時能零延遲執行：

// IRAM_ATTR 確保此函式永遠在 RAM 中 void IRAM_ATTR onTimer() { digitalWrite(LED_PIN, !digitalRead(LED_PIN)); }
忘記 IRAM_ATTR 的後果 若省略此屬性，在 Flash Cache Miss 時呼叫 ISR 會觸發 LoadProhibited 例外，導致系統崩潰。這是 ESP32 開發中非常常見的初學者錯誤。

volatile 變數共享

當 ISR 與 loop() 需要共享資料時（例如 ISR 遞增計數器，loop 讀取計數器計算 RPM），就會面臨一個隱藏的陷阱：編譯器最佳化可能讓 loop() 永遠讀不到 ISR 更新的值。

為什麼需要 volatile？

C/C++ 編譯器在最佳化時，會假設變數只能被「可見的程式流程」修改。如果它發現 loop() 中沒有任何程式碼修改某個變數，就可能把這個變數快取到 CPU 暫存器中，不再每次從記憶體讀取。

但 ISR 是在你看不到的時間點被硬體呼叫的——編譯器無法預見這件事。結果就是 loop() 可能永遠讀到舊值，即使 ISR 早已更新了記憶體中的資料。

// ❌ 錯誤：缺少 volatile，loop 可能永遠看不到 ISR 的更新 int counter = 0; void IRAM_ATTR onTimer() { counter++; } void loop() { // 編譯器可能把 counter 暫存在暫存器，永不重新讀取 Serial.println(counter); // 可能永遠印出 0 }
// ✅ 正確：volatile 強制每次從記憶體讀取 volatile int counter = 0; void IRAM_ATTR onTimer() { counter++; } void loop() { Serial.println(counter); // 每次都讀到最新值 }
volatile 的真正含義 volatile 告訴編譯器：「這個變數可能在任何時候被外部因素改變（如中斷、DMA、硬體暫存器），你不可以對它做任何快取或優化假設。每次存取都必須直接讀寫記憶體。」

臨界區保護（Critical Section）

volatile 只解決了「可見性」問題，但對於多位元組的資料（如 unsigned long、float、結構體），還存在「原子性」問題。

ESP32 是 32-bit 處理器，讀寫一個 32-bit 整數是原子操作（一條指令完成）。但 64-bit 的 unsigned long long 需要兩條指令——如果讀取到一半時 ISR 觸發並修改了值，你就會讀到「一半舊、一半新」的錯亂資料。

解決方案是使用臨界區 (Critical Section)：暫時禁用中斷，確保讀取操作不被打斷。

volatile unsigned long pulseCount = 0; portMUX_TYPE timerMux = portMUX_INITIALIZER_UNLOCKED; void IRAM_ATTR onEncoder() { portENTER_CRITICAL_ISR(&timerMux); pulseCount++; portEXIT_CRITICAL_ISR(&timerMux); } void loop() { unsigned long count; // 安全讀取：禁用中斷期間複製變數 portENTER_CRITICAL(&timerMux); count = pulseCount; pulseCount = 0; // 讀取後歸零 portEXIT_CRITICAL(&timerMux); // 安全地使用 count 進行計算 float rpm = count * 60.0 / PULSES_PER_REV; Serial.println(rpm); }
臨界區時間必須極短 portENTER_CRITICAL() 會禁用中斷。如果在臨界區內執行太長時間，其他中斷會被延遲，可能導致資料遺失或系統行為異常。只在臨界區內做「複製」操作，複雜計算放在臨界區之外。

共享變數防護清單

資料型別 需要 volatile？ 需要臨界區？ 說明
bool / uint8_t 是 通常不需要 8-bit 讀寫是原子操作
int / uint32_t 是 通常不需要 32-bit 在 ESP32 上是原子的
uint64_t / double 是 需要 64-bit 需要兩次讀寫
結構體 / 陣列 是 需要 多欄位必須整體保護
最佳實踐模式 養成習慣：所有在 ISR 中寫入、在 loop() 中讀取的變數，一律加上 volatile。遇到多位元組或需要同時讀取多個相關變數時，一律使用臨界區。寧可多保護也不要少保護——Data Race 的 Bug 極難復現和除錯。

2. 作業任務部署細節

任務 1：精確頻率：硬體定時器與閃燈 (Precision Timer Blink Lab)

  • 目標：理解「硬體級搶佔」的本質，體會定時器如何無視軟體阻塞，強行執行任務。
  • Classroom 部署建議：
    • 頻率固定化：配置 1Hz 的硬體定時器控制 LED。
    • 阻塞實驗：技術審核重點。在 loop() 中加入 delay(5000) 或複雜迴圈。要求學員觀察 LED 是否依然以 1 秒為週期精準閃爍？
    • 驗證要點：是否正確使用了 IRAM_ATTR？閃爍頻率是否真的完全不受 loop() 程式碼長短的影響？

任務 2：轉速計實作：馬達 Encoder 採樣 (Tachometer & RPM Lab)

  • 目標：掌握「等間隔採樣 (Sampling Window)」技術，並正確處理中斷與主程式間的數據競爭 (Race Condition)。
  • Classroom 部署建議：
    • 採樣窗口設計：配置 100ms 的硬體定時器作為採樣點。
    • 變數共享防護：核心關鍵點。所有在 ISR 中增加、在主程式讀取的計數變數，必須添加 volatile 修飾字。
    • 原子操作 (Atomic)：實作 portENTER_CRITICAL() 讀取數據，確保在讀取 16-bit 以上變數時不會被中斷截斷。
    • 驗證要點：RPM 數值是否跳動過大？代碼中是否包含了正確的臨界區保護邏輯？

任務 3：安全保護：硬體級 Watchdog (Watchdog Guard Lab)

  • 目標：打造工業級的「失效安全 (Fail-safe)」機制。當主程式崩潰或網路失聯時，硬體定時器應主動奪權並切斷動力。
  • Classroom 部署建議：
    • 心跳監控邏輯：定時器每隔 500ms 檢查一次計數。主程式必須在期限內「餵狗」重置計數。
    • 強制煞車中斷：若計數歸零，ISR 直接調用馬達暫停函式（不經過 loop 的判斷）。
    • 崩潰模擬：提交一個會在 10 秒後陷入死鎖的代碼，驗證硬體是否能自動執行緊急制動。
    • 驗證要點：制動動作是否在 ISR 中完成？系統是否能在軟體徹底掛掉的情況下依然保護物理安全？

3. 多工穩定度與 ISR 代碼規範點評標準 (Multitasking & ISR Standards)

此單元的價值在於 「對系統執行層級的洞察力與對數據一致性的嚴謹守護」：

  • [ ] 中斷規範遵從度：ISR 內是否有違禁的 Serial.print() 或 delay()？
  • [ ] 數值存取原子性：是否深刻理解了 volatile 與臨界區在保護「跨時空數據」中的角色？
  • [ ] 失效安全性 (Failsafe)：系統設計是否預見了軟體可能故障，並留下了硬體的「最終執行權」？
