為什麼數據會抖動？

在物聯網與嵌入式系統開發中，感測器回傳的原始數據幾乎不可能是完美的平滑線條。無論是超音波測距、紅外線循線、還是 IMU 慣性量測，每一筆讀數都夾帶著程度不等的「雜訊 (Noise)」。理解雜訊的來源，是設計有效濾波策略的第一步。

雜訊的三大來源

感測器數據中的抖動主要來自三個層面：環境干擾、電子元件內部雜訊，以及數位轉換過程中的量化誤差。每一種雜訊的特性不同，應對策略也不同。

雜訊類型 來源 特徵 典型情境
電磁干擾 (EMI) 馬達、電源供應器、WiFi 模組 高頻脈衝式雜訊，與馬達轉速正相關 無人車馬達驅動時，ADC 讀數劇烈跳動
熱雜訊 (Thermal) 電阻、半導體元件內部電子熱運動 隨機高斯分布，無法完全消除 溫濕度感測器在靜態環境下仍有 ±0.1°C 波動
量化誤差 (Quantization) ADC 解析度不足（例如 10-bit = 1024 階） 階梯狀跳動，振幅固定為 1 LSB ESP32 的 12-bit ADC 在 3.3V 下解析度約 0.8mV
備註 在實際系統中，這三種雜訊通常會同時存在並疊加。一個良好的濾波策略需要綜合考量所有雜訊源的特性，而非僅針對單一類型。

雜訊對控制系統的實際影響

假設你的無人車使用攝影機進行循線，質心座標的原始讀數在每一幀之間跳動 ±15 像素。如果直接將此數值餵入 PID 控制器，馬達的 PWM 輸出也會以相同的頻率劇烈振盪，導致以下問題：

  • 機械磨損加速：馬達齒輪箱承受高頻正反轉衝擊，壽命大幅縮短。
  • 功耗激增：頻繁加減速導致電池消耗速度加倍，續航時間減半。
  • 控制失穩：過度修正引發「震盪 (Oscillation)」，車輛左右擺頭無法直行。
警告 在高速運動場景（如機器人競速）中，未經濾波的感測器數據可能導致控制迴路發散，造成設備碰撞或損壞。濾波不是「錦上添花」，而是系統穩定運作的基礎要求。

如何觀察數據抖動？

要設計有效的濾波器，你必須先「看見」雜訊的樣貌。以下是在 ESP32 上快速擷取原始感測器數據的範例程式碼：

void loop() { int rawValue = analogRead(36); Serial.print("RAW:"); Serial.println(rawValue); delay(10); // 100 Hz 取樣率 }
提示 將上述程式碼的輸出貼入 Arduino IDE 的「序列繪圖器 (Serial Plotter)」，你會立即看到鋸齒狀的波形——這就是需要被「磨平」的原始數據。觀察波形的振幅與頻率，將幫助你在後續單元中選擇正確的濾波參數。

均值濾波器 (Moving Average)

均值濾波器（又稱移動平均濾波器）是數位信號處理中最直覺的演算法之一。它的核心概念極為簡單：取最近 N 筆數據的算術平均值作為輸出，藉此「抹平」短期波動，保留長期趨勢。它在時域濾波中扮演著「入門教科書」的角色，也是理解更複雜濾波器的基石。

滑動窗口原理

均值濾波器使用一個固定大小為 N 的滑動窗口 (Sliding Window)。每當新的數據點進入時，最舊的數據點被移除（先進先出 FIFO），然後計算窗口內所有數據的平均值。這個過程就像一個移動的「觀察窗」，持續追蹤最近一段時間的平均表現。

1

初始化一個長度為 N 的陣列

這個陣列就是你的「窗口」，用來暫存最近的 N 筆原始數據。

2

新數據進入，最舊數據移除

每當感測器回傳一筆新讀數，將它推入陣列尾端，同時移除陣列首端的最舊數據（FIFO 機制）。

3

計算窗口內的算術平均

將窗口中所有數值加總後除以 N，得到的結果就是濾波後的輸出值。

C++ 實作範例（Arduino / ESP32）

const int WINDOW_SIZE = 5; int buf[WINDOW_SIZE]; int idx = 0, cnt = 0, total = 0; float movingAverage(int newValue) { if (cnt == WINDOW_SIZE) { total -= buf[idx]; // 移除最舊的值 } else { cnt++; } buf[idx] = newValue; total += newValue; idx = (idx + 1) % WINDOW_SIZE; return (float)total / cnt; }
備註 上述實作使用環形緩衝區 (Circular Buffer) 搭配累計加總，每次更新只需 O(1) 運算量，而非每次都重新遍歷整個窗口計算 O(N) 加總。這是嵌入式系統上的標準最佳化技巧。

窗口大小 N 的權衡

選擇窗口大小 N 是均值濾波器調校的核心問題。N 越大，輸出越平滑；但同時，系統對真實訊號變化的反應也越遲鈍。這就是「延遲代價」(Lag Penalty)。

窗口大小 N 平滑效果 反應延遲 適用場景
N = 3 輕微平滑 約 1–2 個取樣週期 高速避障、即時追蹤
N = 5~10 中度平滑 約 3–5 個取樣週期 循線控制、一般巡航
N = 20+ 極致平滑 約 10+ 個取樣週期 環境監測、靜態測量
警告 在高速運動場景中，過大的窗口可能導致災難性後果。例如：無人車以 1m/s 前進、取樣率 50Hz 時，N=20 的濾波器意味著輸出延遲了約 0.4 秒——車子已經向前移動了 40 公分！如果前方出現障礙物，這 40 公分的「盲區」可能導致碰撞。

手算練習

數據流：[10, 12, 11, 25, 12]，窗口大小 N = 3。當最後一筆數據（12）進入後，窗口內容為 [11, 25, 12]，均值 = (11 + 25 + 12) / 3 = 16.0。請注意，真實值應接近 12，但離群值 25 將平均值拉高到了 16——這就是均值濾波器面對尖峰雜訊的弱點，也是下一個作業「中值濾波器」要解決的問題。

一階低通濾波 (Low-pass Filter)

均值濾波器需要一個陣列來存放歷史數據，這在記憶體受限的嵌入式系統上可能是奢侈的。一階低通濾波器 (First-order Low-pass Filter) 提供了一個更優雅的替代方案：它只需要保存上一次的輸出值和一個係數 α，就能達到與均值濾波器相當甚至更好的平滑效果。

核心公式：指數加權移動平均 (EMA)

一階低通濾波的數學表達式如下：

output = α × newValue + (1 − α) × lastOutput

其中 α（alpha）是一個介於 0 到 1 之間的平滑係數。這個公式本質上是對「新數據」和「歷史趨勢」做加權混合：α 越大，新數據的權重越高；α 越小，歷史趨勢的影響力越大。

α 值 新數據權重 歷史權重 行為特性 適用場景
α = 0.1 10% 90% 極度穩定、反應遲鈍 環境溫度監測、靜態基準校正
α = 0.3 30% 70% 適中平衡 循線巡航、穩定追蹤
α = 0.7 70% 30% 靈敏、輕微平滑 緊急避障、快速轉向
α = 1.0 100% 0% 無濾波（等同原始值） 僅供測試比對

C++ 實作範例（Arduino / ESP32）

float alpha = 0.3; float filteredValue = 0; bool initialized = false; float lowPassFilter(float newValue) { if (!initialized) { filteredValue = newValue; initialized = true; return filteredValue; } filteredValue = alpha * newValue + (1.0 - alpha) * filteredValue; return filteredValue; }
備註 與均值濾波器相比，低通濾波器的最大優勢在於記憶體效率：不需要陣列，只需 2 個浮點數變數（filteredValue 與 alpha）。這使它特別適合資源受限的微控制器，或是需要同時對多個通道進行濾波的場景。

均值濾波 vs 低通濾波：如何選擇？

比較項目 均值濾波 (Moving Average) 低通濾波 (EMA)
記憶體需求 O(N) — 需要長度 N 的陣列 O(1) — 僅需 2 個變數
對離群值的抵抗力 中等（會被拉偏） 中等（α 小時較佳）
參數調校 調整窗口大小 N 調整係數 α
歷史衰減方式 等權重（窗口內每筆數據影響力相同） 指數衰減（越舊的數據影響力越小）
提示 在作業 2-3「動態 α 濾波器」中，你將挑戰更進階的概念：讓 α 不再是固定值，而是根據「當前誤差的大小」自動調整。當偵測到真實位移時提高 α 以快速追隨，當數據平穩時降低 α 以極致穩定。這就是自適應濾波 (Adaptive Filtering) 的核心思想。

adv-04-unit-filter-algorithms (數據抖動與數位濾波演算法)

歡迎來到數據濾波演算法實作單元！在這個練習中，你將學習如何處理來自感測器的原始「髒數據」，並透過不同的數學演算法將其提純為平滑且準確的物理訊號。這對於打造穩定的自動駕駛系統、遙控機器人或任何感測器應用都至關重要。

任務目標

你需要實作並優化以下三種核心濾波器：

  1. 線性平均濾波 (Moving Average)：掌握基礎的時域濾波，理解視窗大小對平滑度與延遲的影響。
  2. 中值濾波器 (Median Filter)：學習如何剔除極端的離群值（Spikes），保護系統不受突發噪音干擾。
  3. 自適應濾波 (Adaptive Alpha Filter)：實作智能係數調整邏輯，在「靈敏反應」與「極致平穩」之間找到完美的動態平衡。

專案結構

  • src/：核心演算法存放處。
    • MovingAverage.js：線性平均邏輯。
    • MedianFilter.js：排序取中值邏輯。
    • AdaptiveFilter.js：自適應係數邏輯。
  • simulator/：視覺化工具。你可以直接開啟 index.html 觀看你的演算法在真實感測器數據上的表現。
  • data/：包含一組包含高頻雜訊與突發尖峰的模擬感測器數據 sensor_noise.json。
  • test/：自動化評分與性能測試腳本。

如何開始

  1. 安裝依賴（如果你需要執行本地測試）：

    npm install

  2. 實作演算法： 依序打開 src/ 下的檔案，根據註解完成實作。

  3. 開啟模擬器： 使用瀏覽器打開 simulator/index.html。

    • 你可以切換不同的演算法。
    • 調整參數（如 Window Size 或 Alpha 值）。
    • 即時觀察「原始數據」與「濾波後數據」的波形對比。

  4. 執行測試： 執行以下命令來檢查你的演算法是否符合品質標準：

    npm test

    此測試會評估你的濾波器在消除噪音方面的有效性以及造成的相位延遲（Phase Lag）。

提交準則

  • 確保你的濾波器在面對突發尖峰（Spikes）時具有足夠的魯棒性。
  • 試著找到最優的參數配置，使得信號既平滑又不會產生過大的反應延遲。
  • 提交前請確保所有測試皆能通過。

this.currentValue = this.alpha * newValue + (1 - this.alpha) * this.lastValue;
    • Autograding：給予一組階躍輸入，測量輸出的「上升時間 (Rise Time)」。如果學員的 $\alpha$ 設得太小，卻沒發現數據反應變得過於遲鈍，則需提供回饋提示。

任務 2：中值濾波抗干擾挑戰 (Median Filter Spike Protection)

  • 目標：處理線性濾波器無法解決的「離群值 (Outliers)」。
  • Classroom 部署建議：
    • 異常測試：在測試數據中插入一個極大的錯誤值（如：正常值 10，突然變成 1000）。
    • 驗證方式：檢核輸出曲線。均值濾波會被 1000 拉起一個大斜坡，而正確的中值濾波應該能「完全無視」這個尖峰，維持在 10 附近。這能帶給學員極大的成功感。

任務 3：動態 α 濾波器 (Adaptive Alpha Lab)

  • 目標：設計具備「智能」的濾波強度，解決靈敏度與平穩度的零和博弈。
  • Classroom 部署建議：
    • 權衡邏輯設計：要求學員實作一個根據「誤差值 (Error)」自動調整 $\alpha$ 的函式。
      • 當 $Error$ 很大時（代表發生真位移），調大 $\alpha$ 以求快速追隨。
      • 當 $Error$ 很小時（代表只是雜訊），調小 $\alpha$ 以求極致平穩。
    • Tutor View (PR 審核)：導師在 PR 中點評學員的公式曲線。是否有設置「死區 (Deadband)」以避免在平衡點產生微小震盪？

3. 演算法導師點評標準 (Algorithm Benchmarks)

此單元的價值在於 「對數據背後物理意義的理解」：

  • [ ] 延遲感知力：學員是否能解釋為什麼視窗 $N$ 不能無限加大？（提示：無人車會因為濾波延遲而撞牆）。
  • [ ] 排序效率考量：在實作中值濾波時，是否因頻繁的 Array.sort() 導致瀏覽器運算負載過重？
  • [ ] 場景自適應性：README 是否說明了：在「靜態測速」與「高速避障」時，濾波器參數應如何動態切換？
