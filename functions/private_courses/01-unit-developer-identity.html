認識雲端身分：OAuth 是什麼？

在建立帳號之前，我們先來了解「用 Google 帳號登入 GitHub」這件事背後的原理。 這套機制叫做 OAuth 2.0，是目前全球最廣泛使用的身分授權標準。

為什麼需要 OAuth？

過去，如果你想用 A 服務的帳號登入 B 服務，你必須把帳號密碼直接告訴 B。這非常危險—— 萬一 B 的資料庫被入侵，你的密碼就外洩了。

OAuth 解決了這個問題。它讓你只需要向身分提供者（Identity Provider）（如 Google） 登入一次，再由身分提供者發一張限時的「通行證（Token）」給第三方服務（如 GitHub）。 第三方服務永遠拿不到你的原始密碼。

重點 在整個 OAuth 流程中，GitHub 從未拿到你的 Google 密碼。 它只收到一張由 Google 簽發的臨時通行證（Token），有效期限結束後就失效。 即使 GitHub 資料庫被入侵，駭客也無法用這張 Token 取得你的 Google 帳號。

三個角色

角色 範例 職責
使用者（User） 你 發起授權請求、同意授權範圍
應用服務（Client） GitHub 請求 Token、用 Token 存取用戶資料
身分提供者（Identity Provider） Google 驗證使用者身分、簽發 Token
思考一下 你還記得哪些網站提供「用 Google / Apple / Facebook 登入」的選項嗎？ 現在你知道了，那些都是 OAuth 在運作。

建立你的 Google 帳號

Google 帳號是你踏入雲端世界的第一把鑰匙。它不只是一個信箱，更是你未來使用 GitHub、Google Drive、Firebase 等開發工具的統一身分入口。

注意 如果你已有 Google 帳號，可以直接跳到 GitHub 帳號設定。 建議使用一個你長期會使用的 Google 帳號作為開發者主帳號。

申請步驟

1

前往 Google 帳號建立頁面

開啟瀏覽器，前往 accounts.google.com/signup，或開啟 Gmail 首頁點選「建立帳戶」。

2

選擇「個人用途」

在用途選擇頁面，選擇「個人用途」繼續。

3

填寫基本資料

輸入你的姓名、生日與性別，以及你想要的 Gmail 使用者名稱（即 xxxxx@gmail.com 中的 xxxxx 部分）。

4

建立強密碼

密碼需包含至少 8 個字元，建議混合大小寫字母、數字與符號（例如 Abc@2024!）。避免使用生日、姓名等易猜測的資訊。

5

設定帳號救援方式（建議）

填寫備援信箱或手機號碼。當你忘記密碼或帳號被鎖定時，這是找回帳號的重要管道。

建議 為你的開發活動建立一個專屬 Google 帳號，與個人生活帳號分開， 有助於日後管理 Firebase、Google Cloud 等開發資源，也更符合專業習慣。

完成後你擁有的服務

Google 服務開發用途
Gmail接收 GitHub 通知、驗證碼
Google Drive存放課程講義、備份資料
Google Docs / Slides協作文件、簡報
Firebase（需額外連結）雲端資料庫、部署網站
Google Cloud（需額外連結）伺服器、AI 服務

建立你的 GitHub 帳號

GitHub 是全球最大的程式碼協作平台，超過一億名開發者在這裡存放、分享、協作程式碼。 你的 GitHub 帳號就是你的開發者履歷——未來面試官、合作夥伴， 甚至世界各地的開發者都可能透過它認識你。

情境
想像你是一位剛畢業的工程師，面試官要求提供 GitHub 連結。 一個有完整 Profile、清楚 README 和穩定提交紀錄的帳號， 往往比一份制式履歷更能展現你的實力。

Step 1：註冊帳號

1

前往 GitHub 官網

開啟 github.com，點擊右上角的 Sign up。

2

填寫帳號資訊

Email：填入你的 Google 信箱。
密碼：使用強密碼（與 Google 密碼不同）。
Username：這是你的公開開發者識別碼，格式為 github.com/你的名字。

Username 選擇建議 Username 一旦設定後雖可修改，但更動會影響所有連結。建議：
  • 使用真實姓名的英文拼音（如 mingchen、rover-chen）
  • 避免使用隨機數字、綽號或不雅詞彙
  • 字元只能包含字母、數字和連字號（-）

Step 2：帳號驗證

GitHub 會進行兩道驗證，確保你是真人：

  • 圖形驗證：完成旋轉圖片或拼圖等視覺驗證。
  • Email 驗證：前往你的 Gmail，輸入 GitHub 寄來的 8 位數驗證碼。

Step 3：回答問卷

GitHub 會問幾個關於你用途的問題（學生、個人開發者等）。 這些答案用於個人化推薦，可以快速勾選，不影響帳號功能。

學生方案 如果你是在學學生，可以申請 GitHub Student Developer Pack， 免費獲得許多付費開發工具的使用權。

強化帳號安全：兩步驟驗證（2FA）

帳號建立完成後，最重要的事不是立刻開始寫程式，而是開啟兩步驟驗證（Two-Factor Authentication，2FA）。 你的 GitHub 帳號存放著你所有的程式碼，一旦被盜，損失可能難以估計。

為什麼只有密碼不夠？

驗證手段 本質 主要風險
密碼（Password） 你知道的東西 可被猜測、釣魚信件竊取、資料庫外洩
2FA 驗證碼（OTP） 你擁有的東西（手機） 駭客需同時取得你的手機才能登入

啟用 2FA 後，即使駭客拿到你的密碼，沒有你的手機就無法登入。 這就是雙重保護的核心概念。

重要 GitHub 官方建議所有使用者啟用 2FA。 部分組織 Repository 甚至強制要求成員必須開啟 2FA 才能貢獻程式碼。

設定步驟

1

安裝驗證器 App

在手機上安裝 GitHub Mobile 或 Google Authenticator（iOS / Android 皆有）。

2

前往 GitHub 安全設定

登入 GitHub → 右上角頭像 → Settings → Password and authentication。

3

啟用 2FA

點擊 Enable two-factor authentication，選擇「Authenticator App」方式。

4

掃描 QR Code

用手機 App 掃描畫面上的 QR Code，App 會開始每 30 秒產生一組新的 6 位數驗證碼。

5

儲存備用碼

GitHub 會提供一組備用恢復碼（Recovery Codes），請務必截圖或列印存放在安全的地方。手機遺失時，你可以用這組碼登入。

備用碼保管 備用碼只會顯示一次，且每個碼只能使用一次。 請務必將它們存放在 GitHub 以外的安全位置（如加密的記事本、紙本保險箱）。

GitHub 隱藏彩蛋：Special Repository

建立好 GitHub 帳號後，這裡要向你介紹一個非常有趣的 GitHub 隱藏彩蛋功能——Special Repository（特殊倉庫）。 它可以讓你在 GitHub 個人檔案首頁，放上一篇漂亮的自我介紹，打造專屬的個人名片看板！

GitHub 隱藏彩蛋（Easter Egg）功能：Special Repository

只要你建立一個與你的 GitHub 帳號名稱完全相同的專案倉庫（Repository），你就可以在你的 GitHub 個人檔案首頁，放上一篇漂亮的自我介紹（README.md）。

舉例說明
如果你的 GitHub 帳號名稱（Username）叫做 chen-yu-kuan：
  1. 當你新增一個 Repository，並且把它的名字也輸入為 chen-yu-kuan。
  2. GitHub 系統就會跳出煙火特效，告訴你發現了彩蛋，這是一個 Special Repository（特殊倉庫）。
  3. 當你在這個倉庫裡寫入 README.md 檔案時，這份檔案的內容就會直接變成你的 GitHub 個人主頁大看板！

這個「Special Repository」有什麼特別的地方？

平常別人點進你的 GitHub 主頁（例如：https://github.com/你的帳號），預設只能看到你最近參與的專案與綠色的貢獻格子（Contribution Graph）。

但如果你啟用了這個功能，你就可以把它當作你的數位名片、個人履歷或技術網誌首頁。你可以用 Markdown 語法自由地在上面放置：

  • 👋 自我介紹與研究領域：例如你目前專注的 AI 代理人、Vibe Coding、ESP32 韌體開發等。
  • 🛠️ 擅長的技術棧：放置如 Python, C++, Node.js 等生動的技術圖標。
  • 📊 動態統計圖：自動動態更新的 GitHub 數據統計圖（例如你的程式碼行數、最常使用的語言）。
  • 🔗 個人連結：你的個人網站、社群媒體或聯絡方式。

如何建立它？（簡單三步驟）

1
新建倉庫：登入 GitHub，點擊右上角的 「+」 → New repository。
2
精準命名（關鍵）：在 Repository name 格子裡，輸入跟你個人帳號完全一樣的英文字母（大小寫需一致）。
此時下方會跳出綠色框提示：「You found a secret! ...」
3
設定公開並初始化： 務必將可見度設為 Public（公開），並勾選 Add a README file（非常重要，因為自我介紹將寫在此處），最後點擊 Create repository。
開始填寫名片： 建立完成後，點擊該 README.md 檔案右上角的「編輯（鉛筆圖示）」，寫下你想讓大家看到的自我介紹並存檔。回到你的 GitHub 個人首頁，就能看到精美的個人大看板了！

開發者身分與數位通行證 (Developer Identity)

歡迎來到 Vibe Coding 課程的第一步！在這個單元中，你將建立屬於自己的開發者身分，並在 GitHub 上發射你的第一個訊號。這不只是關於技術操作，更是關於如何在這個開源世界中展示你自己。

🎯 學習目標

  • 掌握 Markdown 語法，撰寫專業的技術文檔。
  • 理解 GitHub 個人主頁與程式庫 (Repository) 的運作機制。
  • 強化帳戶安全性（啟用 2FA），保護你的數位資產。

📝 作業任務說明

本單元包含兩個主要任務，請依照以下步驟完成：

任務 1：打造數位身分通行證 (Digital Profile)

你的 my_profile.md 是你在開發者社群的「名片」。

  1. 編輯檔案：打開 my_profile.md 檔案。
  2. 更新內容：
    • 將標題改為你的名字。
    • 在 ## 🎯 About Me 寫下簡短的個人介紹。
    • 列出你的技術技能與感興趣的專案（必須包含 Markdown 清單語法）。
    • 放入你的 GitHub 或 LinkedIn 連結（必須包含 Markdown 超連結語法）。
  3. 上傳圖片：將你的頭像或代表圖片放入 /assets/ 資料夾中（此資料夾不可為空），並在 Markdown 中正確引用路徑。
  4. 安全切結：在檔案末尾確認你已完成 2FA（必須包含「2FA」或「兩步驟驗證」關鍵字）。

任務 2：發射第一個訊號 (Hello World)

練習建立並發布你的第一個公開程式庫。

  1. 建立程式庫：前往你的 GitHub 個人首頁，建立一個名為 hello-world 的 Public 程式庫。
  2. 初始化：在建立時勾選 "Add a README file"。
  3. 提交連結：將該程式庫的連結貼回本程式庫的 submission.md 檔案中。
    • 格式必須為：https://github.com/[你的帳號]/hello-world
    • 重要：請確保該程式庫設為 Public，否則自動評分系統將無法存取（404 錯誤）。

🛡️ 安全性提醒 (Security First)

在開發者世界，帳戶安全至關重要。請務必：

  • 進入 GitHub 設定，啟用 Two-Factor Authentication (2FA)。
  • 妥善保存你的 Recovery Codes。
  • 在 my_profile.md 的底部勾選確認你已完成此安全性設置。

🎓 導師指南：單元 01 開發者身分 (Tutor Guide)

針對 01-unit-developer-identity (開發者身分) 的作業部署與管理，本指南旨在協助導師有效利用 GitHub Classroom 建立學習流，並結合自動化檢測提升教學效率。

📌 單元教學核心

本單元不只是 Markdown 語法練習，更核心的目標是讓學生理解：

  • 數位身分 (Digital Identity)：如何在技術社群呈現專業形象。
  • 開源流程 (OSS Workflow)：建立、公開並提交一個真實的程式庫。
  • 安全意識 (Security)：實踐開發者必備的帳戶防護 (2FA)。

🛠️ 作業設計與部署建議

任務 1：數位身分通行證 (Digital Profile)

  • 部署方式：使用 Classroom Template 提供 my_profile.md 與 /assets/ 資料夾。
  • 教學邏輯：讓學生在受控環境（Classroom）練習，完成後引導其將內容同步至個人的 username/username 特殊程式庫，達成「學以致用」。
  • 評分重點：
    • [ ] Markdown 語法是否標籤正確（標題、清單、連結）。
    • [ ] 圖片資源路徑引用是否正確。
    • [ ] 個人介紹內容的完整度。

任務 2：發射第一個訊號 (Hello World)

  • 部署技巧：要求學生「手動建立」一個獨立於 Classroom 之外的公共程式庫，並將 URL 貼回 submission.md。
  • 教學目標：跳脫範本框架，讓學生練習真實的 GitHub 建立流程與權限設定（Public vs Private）。

🤖 自動化檢測機制 (Autograding)

本單元已整合 .github/scripts/check_assignment.sh 進行自動化評分，主要檢核指標如下：

檢查項目 檢測方式 失敗原因舉例
Markdown 語法 grep 檢查 my_profile.md 漏寫連結、未使用清單符號
資源完整性 檢查 assets/ 資料夾是否非空 忘記上傳圖片檔案
安全性切結 檢查關鍵字「2FA」或「兩步驟驗證」 未在檔案底部進行確認
連結有效性 curl -I 檢查 submission.md 內的 URL 連結錯誤或程式庫設為 Private (404)

💡 教學與反饋心法 (Tutor Tips)

  1. 回饋機制 (Feedback Pull Request)： 強烈建議開啟 GitHub Classroom 的 "Feedback Pull Request" 功能。導師可直接在學生的 Markdown 代碼處留言，點評其選圖或排版細節，建立「代碼審查 (Code Review)」的初步體驗。
  2. 關於 2FA 的驗證： 由於 Action 無法直接存取學生帳戶設定，因此採用「口頭切結」方式。建議導師在課堂上抽查學生的 GitHub 設定頁面，確保其不僅是填寫關鍵字，而是真正啟用了保護。
