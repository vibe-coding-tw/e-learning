Module 1: Intuitive Lab

同步 vs 非同步：為什麼畫面卡住了？

SYNC

🚫 同步 (Synchronous)

程式碼一行一行執行。當遇到耗時操作（如連線）時，整個網頁的主執行緒會被「阻塞 (Block)」，導致按鈕無法點擊、動畫停止，使用者體驗極差。

ASYNC

⚡ 非同步 (Asynchronous)

使用 Promise 把耗時任務丟給瀏覽器背景處理。主執行緒繼續回應使用者操作。當任務完成時，再透過回呼或 await 繼續執行後續動作。

1 告別 Callback Hell，擁抱語法糖

The Old Way (Callback Hell) Painful 😫 
navigator.bluetooth.requestDevice({...})
  .then(device => {
    return device.gatt.connect()
      .then(server => {
        return server.getPrimaryService(...)
          .then(service => {
             // ... 波動拳 👊
             return service.getCharacteristic(...)
          })
      })
  })
  .catch(error => { ... });
Async / Await Elegant ✨ 
async function connectBLE() {
  try {
    const device = await navigator.bluetooth.requestDevice(...);
    const server = await device.gatt.connect();
    const service = await server.getPrimaryService(...);
    const char = await service.getCharacteristic(...);
    
    console.log('Ready!');
  } catch (error) {
    console.error('Oops', error);
  }
}

2 Web BLE 連線鏈

Web BLE 的連線過程是嚴格的「階層式」操作。每一層都必須等待上一層 Resolved 才能繼續。

錯誤捕獲：Try...Catch

藍牙連線變數極多（距離遠、被拒絕、斷線）。使用 try...catch 包裹整個 async 函式，是保護 App 不會崩潰的唯一防線。

Step 1

requestDevice()

彈出視窗，等待使用者選擇設備。

Step 2

gatt.connect()

建立低功耗藍牙連線。這是最容易失敗的一步（Timeout, Error 133 等）。

Step 3

getPrimaryService()

尋找該設備提供的特定服務（Service UUID）。

Target

getCharacteristic()

取得最終的讀寫控制點 (Characteristic)。

JavaScript 非同步編程：操作 BLE 的靈魂 (start-02-unit-ble-async)

歡迎來到 Start 02 單元！在這個練習中，你將學習如何使用 JavaScript 的非同步特性（async/await、Promise）來控制藍牙低功耗 (BLE) 裝置。

在無線通訊的環境中，連線往往是不穩定的，且各項操作需要時間等待。掌握非同步流程控制，是開發穩定硬體互動介面的關鍵。

🎯 學習目標

  1. 非同步流程控制：熟練運用 async/await 處理 BLE 連線鏈。
  2. 狀態機 UI 更新：在非同步操作的各個階段即時更新 UI 狀態。
  3. 競爭條件 (Race Condition) 防範：利用旗標 (Flag) 防止重複觸發導致的 GATT 錯誤。
  4. 超時保護 (Timeout)：使用 Promise.race 實作連線超時處理，提升使用者體驗。

🚀 任務說明

任務 1：「逐步點亮」狀態機實作

在 ble-handler.js 中實作連線邏輯，並在每個關鍵步驟（搜尋、連線、掃描服務）之間更新 index.html 中的進度條或狀態文字。

  • 關鍵點：確保在連線過程中將「連線」按鈕設為 disabled。

任務 2：處理並發與競爭條件

當使用者快速連續點擊按鈕時，可能會發生 GATT operation already in progress 錯誤。請在 ble-handler.js 中使用 isBusy 旗標或 try...finally 結構來保護你的連線流程。

  • 關鍵點：無論連線成功或失敗，最後都必須釋放鎖定。

任務 3：非同步超時保護

實作一個超時機制。如果在 10 秒內無法完成連線，應主動取消等待並告知使用者「連線逾時」，而不是讓 UI 永久處於等待狀態。

  • 提示：可以使用 Promise.race 來達成此功能。

📂 程式庫結構

  • index.html：包含進度條與狀態提示的控制面板。
  • ble-handler.js：你主要實作非同步邏輯的地方。
  • style.css：介面樣式。
  • assets/：參考圖檔（如連線時序圖）。

    • 驗證要點：超時邏輯是否確實使用了非同步競爭模式？處理流程是否能將 UI 完整推回初始重試狀態？
