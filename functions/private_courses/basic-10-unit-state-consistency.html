簡介

競爭條件 (Race Condition)

在物聯網系統中，網頁前端與硬體之間透過無線通訊交換資料。然而，無線環境本質上是不可靠且無序的——封包可能延遲、丟失、甚至以相反的順序抵達。當多個操作同時影響同一個狀態變數時，最終結果取決於「誰先到達」，而不是「誰先發出」。這就是競爭條件（Race Condition）。

競爭條件在單機嵌入式系統中也會發生：當 ISR（中斷服務程序）與主迴圈同時讀寫同一個狀態變數時，若缺乏保護機制，主迴圈可能讀到「半更新」的資料。在分散式系統（如 BLE 遙控車）中，這個問題更加嚴重，因為網路延遲讓時序完全不可預測。

典型失敗情境

以下是一個常見的競爭條件場景：

1

使用者點擊「進入自動模式」

網頁發出指令封包 {cmd: "SET_MODE", mode: "AUTO", seq: 42}。

2

硬體因電力不足自動退回待機

在收到指令之前，ESP32 偵測到電壓過低，已自行切回 IDLE 並發出狀態通知 {state: "IDLE", seq: 43}。

3

封包亂序抵達

網頁先收到舊的 ACK（確認已切換到 AUTO），後收到新的狀態通知（已退回 IDLE）。若無序號過濾，UI 最終顯示「AUTO」——但車子其實在待機！

重要 UI 與硬體狀態不同步不只是「顯示錯誤」——它可能導致使用者基於錯誤資訊做出危險操作。例如，使用者以為車子在自動模式（不需介入），實際上車子正待機等待指令而朝障礙物前進。

版本序號過濾器

解決亂序問題最直接的方法是版本序號（Sequence Number）。每個封包都附帶一個遞增的整數序號，接收端只接受序號大於目前已知序號的封包，丟棄所有「遲到的舊資料」。

// 前端 JavaScript：序號過濾邏輯
let lastSeq = 0;

function onMessageReceived(packet) {
    if (packet.seq > lastSeq) {
        lastSeq = packet.seq;
        updateUI(packet.state); // 更新顯示
    } else {
        console.warn(`丟棄過期封包: seq=${packet.seq}, current=${lastSeq}`);
    }
}

封包過濾策略比較

策略 原理 適用場景 缺點
序號遞增 丟棄 seq ≤ lastSeq 的封包 狀態同步、感測器數據 序號溢位需處理環繞
時間戳比對 丟棄時間戳早於已知的封包 時鐘同步良好的系統 需要精確的時鐘同步
向量時鐘 多節點因果關係追蹤 多裝置協同控制 實作複雜度高
提示 對於 ESP32 BLE 通訊的場景，使用 uint16_t 序號即可（0–65535）。以每秒 10 個封包的頻率，需要將近兩小時才會溢位。當偵測到序號從極大值跳到極小值時，可以判定為合法的環繞而非亂序。

樂觀 UI 與真值來源 (SSoT)

在 Web 應用開發中，「樂觀更新（Optimistic UI）」是一種常見的設計模式：當使用者執行操作時，UI 立即顯示預期結果，不等待伺服器確認。這在普通網頁（如按讚）中很好用，但在控制實體硬體的場景下卻可能致命。

想像你點擊「切換到自動模式」，UI 立刻將按鈕變為綠色——但此時 BLE 封包可能根本還沒送達硬體。如果指令最終失敗（丟包、硬體拒絕、電力不足），UI 就會「說謊」：螢幕顯示自動模式，車子實際在待機。這種「認知不對稱」在遙控機器人場景中極其危險。

樂觀 UI vs. SSoT 架構

特性 樂觀更新 (Optimistic) 真值來源 (SSoT)
UI 更新時機 使用者點擊後立刻更新 收到硬體確認後才更新
操作感受 極快（0ms 延遲感） 略有延遲（50–200ms）
失敗處理 需要「回滾」UI，產生閃爍 不需回滾，狀態始終正確
一致性保證 可能短暫不一致 100% 一致
適用場景 低風險操作（按讚、收藏） 高風險操作（馬達控制、模式切換）
警告 在控制物理裝置的介面中，絕對不要使用樂觀更新。一個「螢幕說自動、車子其實待機」的狀態不一致，可能導致使用者放手不管——而車子正朝牆壁衝去。

SSoT 架構實作

SSoT（Single Source of Truth，真值來源）的核心原則很簡單：UI 狀態完全由硬體回傳的 NOTIFY 封包驅動。前端不自行猜測、不樂觀假設——只呈現硬體告訴它的真實狀態。

// SSoT 架構：UI 狀態完全由硬體驅動
class StateManager {
    constructor() {
        this.currentMode = 'UNKNOWN'; // 開機時不假設
        this.pendingCmd = null;
    }

    // 只有硬體通知才能改變 UI 狀態
    onHardwareNotify(packet) {
        this.currentMode = packet.state;
        this.pendingCmd = null;
        renderPanel(this.currentMode);
    }

    // 發送指令時只顯示「處理中」
    sendCommand(cmd) {
        this.pendingCmd = cmd;
        showPendingIndicator(); // 旋轉圖示，不改變狀態
        bleTransmit(cmd);
    }
}

實作要點

  • 開機初始化：UI 啟動時狀態設為 UNKNOWN，顯示「等待連線」，直到收到第一個硬體心跳。
  • 指令發送：點擊按鈕後不改變 UI 狀態，僅顯示「處理中」動畫（如 spinner）。
  • 狀態接收：只有收到硬體的 NOTIFY 封包時才更新 UI。
  • 超時處理：若超過閾值（如 2 秒）未收到回應，顯示「通訊異常」警告。
提示 SSoT 架構的「延遲感」可以透過 UI 設計來緩解：在按下按鈕後立即顯示一個優雅的過渡動畫（如脈衝光環），讓使用者知道指令已送出、正在等待確認。這比假裝已完成要誠實得多。

ACK 與等級機制

序號過濾器解決了「亂序」問題，但還有另一個致命挑戰：封包丟失（Packet Loss）。當你發出「切換到自動模式」的指令，如何確認硬體真的收到了？如果封包在空中消失了呢？這就需要 ACK（Acknowledgement，應答確認）機制。

ACK 機制的核心思想借鑒自 TCP 協議：每個關鍵指令都帶有唯一 ID，接收端執行完畢後必須回傳帶有相同 ID 的確認封包。若發送端在超時時間內沒收到 ACK，就自動重傳——但重傳不能太頻繁，否則會造成網路壅塞，因此需要指數退避策略。

ACK 通訊協議流程

1

封裝指令

前端將指令封裝為帶有唯一 cmdId 的 JSON 物件，並記錄發送時間。

2

顯示 Pending 狀態

UI 進入「等待確認」模式：按鈕顯示旋轉動畫，禁止重複點擊。

3

硬體執行並回覆 ACK

ESP32 收到指令 → 執行動作 → 回傳 {ack: true, cmdId: 101}。

4

前端匹配確認

收到 ACK 後，匹配 cmdId，清除 Pending 狀態，由 NOTIFY 驅動 UI 更新。

指數退避重傳 (Exponential Backoff)

若超時未收到 ACK，不能立即瘋狂重傳——這會在網路已經壅塞時雪上加霜。正確的做法是指數退避：每次重傳的等待時間加倍，直到達到上限或重傳次數耗盡。

async function sendWithRetry(cmd, maxRetries = 3) {
    let delay = 500; // 初始等待 500ms

    for (let attempt = 0; attempt < maxRetries; attempt++) {
        bleTransmit(cmd);

        const ack = await waitForAck(cmd.cmdId, delay);
        if (ack) return { success: true, attempts: attempt + 1 };

        delay *= 2; // 500 → 1000 → 2000ms
        console.warn(`重傳 #${attempt + 1}，下次等待 ${delay}ms`);
    }

    // 全部重傳失敗
    showConnectionError();
    return { success: false, attempts: maxRetries };
}

指令等級分類

並非所有指令都需要 ACK。持續發送的搖桿座標（每秒 20 次）若都要求 ACK，會造成巨大的通訊負擔。正確做法是按指令的重要程度分級：

等級 類型 需要 ACK？ 範例
Critical 模式切換 / E-Stop 必須，含重傳 SET_MODE、EMERGENCY_STOP
Normal 參數設定 需要 ACK，不重傳 SET_SPEED_LIMIT、CALIBRATE
Streaming 即時控制數據 不需要 搖桿座標、感測器輪詢
備註 ESP32 端需要實作「指令冪等性（Idempotency）」：即使收到重複的指令（同一個 cmdId），也只執行一次。收到重複指令時直接回傳 ACK 而不重複執行動作，避免因重傳導致馬達被啟動兩次。

基礎 10：非同步下的狀態一致性 (Solving Communication Lies)

歡迎來到本單元！在開發物聯網或機器人系統時，網路延遲、封包丟失（Packet Loss）與亂序抵達是常態。本作業的目標是讓你學習如何建立「工業級強壯通訊」，確保你的控制面板與硬體狀態達成 100% 同步，徹底解決「通訊謊言」問題。

🎯 學習目標

  1. 版本序號同步法 (Sequence Number)：防止舊數據覆蓋新狀態。
  2. 應答確認機制 (ACK/Retry)：確保關鍵指令確實落地執行。
  3. 真值來源 (SSoT)：建立「眼見即真實」的訂閱驅動架構。

🛠 任務說明

任務 1：實作版本序號過濾器

在分散式系統中，封包抵達順序不一定等於發送順序。

  • 硬體端：在 src/main.cpp 中，為每個回傳的狀態封包加上遞增的 seq。
  • 前端：在 src/UIController.js (或對應邏輯檔) 中實作過濾邏輯：只有當新序號大於目前序號時，才更新 UI。

任務 2：建立 ACK 與自動重傳機制

當你點擊按鈕時，如何確定機器人真的收到了指令？

  • 指令封裝：將發送指令封裝為異步操作，在等待回應時顯示「處理中」狀態。
  • ACK 實作：硬體收到指令後，回傳帶有該指令 ID 的確認封包。
  • 重傳邏輯：若 500ms 內未收到 ACK，實作自動重傳（建議加入指數退避機制）。

任務 3：重構為 SSoT 架構

徹底消除「樂觀更新」。

  • 變更前：按鈕點擊後立即改變顏色（假裝成功）。
  • 變更後：UI 狀態完全由硬體回傳的封包驅動。只有收到硬體回傳「狀態已變更」的通知後，面板才更新顯示。

📁 程式庫結構

  • src/main.cpp：硬體通訊協議實作。
  • src/UIController.js：前端通訊與過濾邏輯。
  • docs/CONSISTENCY_PROTOCOL.md：定義你的通訊握手流程。
  • assets/retry_log.txt：記錄你測試重傳機制的日誌。

📝 提交要求

  1. 確保在網路不穩定的模擬環境下，面板不會出現「跳動」或「閃爍」。
  2. 當斷線時，UI 應能正確顯示「通訊異常」而非誤導使用者。
  3. 將你的測試結果與一致性協議定義填寫在 docs/CONSISTENCY_PROTOCOL.md 中。

  • 📂 docs/CONSISTENCY_PROTOCOL.md：一致性協議規範。要求學員定義其自訂的「握手 (Handshake)」流程，描述指令發出到收到確認的完整生命週期。

2. 作業任務部署細節

任務 1：版本序號同步法 (Sequence Number Logic Lab)

  • 目標：修復亂序跳動 Bug。體會「時間順序」在分散式系統中的重要性。
  • Classroom 部署建議：
    • 序號注入實作：在 ESP32 回傳的 JSON 中加入 seq: n++ 字段。
    • 過濾器代碼：前端 JavaScript 實作：if (newSeq > lastSeq) updateUI(); else discard();。
    • 亂序實驗：手動發送一個帶有較小 seq 的偽造封包，驗證面板是否能冷靜地將其忽視（不產生跳動）。
    • 驗證要點：過濾邏輯是否嚴密？是否成功根處理因為無線網絡延遲導致的「數值閃爍」現象？

任務 2：應答確認機制 (ACK/NACK Framework Lab)

  • 目標：從「盲發通知」進化到「雙向握手」，確保關鍵指令（如：切換自動模式）一定有執行落地。
  • Classroom 部署建議：
    • 異步指令封裝：將發送代碼封裝為 Promise，並在發出時顯示「Pending」動畫。
    • ACK 回軌邏輯：硬體執行完畢後必須回傳帶有該指令 ID 的 ACK 封包。
    • 重傳機制 (Retry)：實作超時偵測。若 500ms 沒收到 ACK，自動觸發「指數退避 (Exponential Backoff)」重傳。
    • 驗證要點：重傳邏輯是否運作？當連續斷線時，UI 是否能正確顯示「通訊異常」而非假裝成功？

任務 3：真值來源 (SSoT) 重構 (Single Source of Truth Lab)

  • 目標：徹底放棄「樂觀更新 (Optimistic UI)」，實踐「眼見即真實」的權威架構。
  • Classroom 部署建議：
    • 主權移交設計：刪除按鈕下方的「按了就變色」代碼。強制 UI 只能在收到硬體正式回傳的 NOTIFY 封包後才更新狀態。
    • 狀態追蹤分析：在 README 中分析 SSoT 如何解決「螢幕顯示自動、車子其實待機」的致命衝突。
    • 驗證要點：UI 狀態是否達成 100% 訂閱驅動？在延遲環境下，面板是否能量化地展示出「等待硬體響應」的真實過程？
