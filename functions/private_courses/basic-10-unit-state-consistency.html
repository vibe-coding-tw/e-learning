🔄 3. ACK 與等級機制

「應答 (ACK)」是最高效方案：硬體執行後主動通知。

Protocol Strategy

  • 版本序號 (Sequence)：封包加上遞增編號。
  • 雙向握手 (ACK)：包含指令 ID。
  • 指數退避 (Backoff)：失敗時重試。
// 发送具備 ID 指令
sendCmd({ id: 101, act: "START" });

// ACK 回传
{ "ack": true, "id": 101 }

任務 1：版本序號同步法 (Sequence Number Logic)

角色扮演：系統工程師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫。
  2. 注入封裝：封包新增遞增 seq 字段。
  3. 過濾機制：網頁端僅接受 seq 大於 local 值的數據。
  4. 驗證：觀察顯示是否穩定不跳動。

任務 2：應答確認機制 (ACK/NACK Framework)

角色扮演：協定設計師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 非同步化：發送函式改寫為 Promise。
  2. 確認回軌：硬體完成後回傳 ACK。
  3. 重傳邏輯：實作超時偵測與指數退避重傳。
  4. 路徑驗證：失敗後正確顯示警告。

🎯 教師點評標準 (Tutor Benchmarks)

任務 3：真值來源 (SSoT) 重構 (Single Source of Truth)

角色扮演：系統架構師 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 主權剝離：徹底移除樂觀更新。
  2. 全訂閱制：面板變更由 NOTIFY 觸發。
  3. 差異分析：分析 SSoT 實質優勢。
  4. 交付物：推送重構後版本。
