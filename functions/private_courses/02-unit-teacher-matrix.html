進入入門課程

2026 導師能力矩陣

任務 1：規格說明書撰寫實踐 (Spec-First)

角色扮演：技術負責人 (Tech Lead) | 難度：⭐️⭐️ | 估計時長：30 min

🛠️ 執行技術流程 (Git Workflow)

  1. 定義需求：針對一個「物聯網氣象站」，撰寫 SPEC.md。
  2. 結構化描述：包含：
    • API 接口協議 (JSON 格式定義)
    • 硬體腳位配置 (Pinout Mapping)
    • 驗收標準 (Acceptance Criteria)
  3. 提交規格：讓 AI 根據這份 Spec 產生第一版代碼，並 commit 你的原始規格。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 嚴謹性：Spec 是否足以讓另一個開發者（或 AI）在不詢問的情況下完成開發？
  • [ ] 維護性：是否正確使用了 Markdown 進行結構化排版？

任務 2：安全性與漏洞審計 (Security Audit)

角色扮演：安全專家 (SecOps) | 難度：⭐️⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 掃描弱點：檢視 server_logic.js，找出 3 個潛在的安全風險（例如：Hardcoded Key, No Authentication, SQL Injection）。
  2. 修復建議：在 AUDIT.md 中記錄這些風險，並提交修復後的代碼。
  3. 同步分析：推送你的審計報告與修補程式。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 風險識別：是否能找出非顯而易見的邏輯漏洞？
  • [ ] 修補正確性：修補後的程式碼是否在修正漏洞的同時，保持原有功能正常？

任務 3：自動化工作流提案 (CI/CD Concept)

角色扮演：運維工程師 (DevOps) | 難度：⭐️⭐️ | 估計時長：20 min

🛠️ 執行技術流程 (Git Workflow)

  1. 流程設計：構思一個自動化腳本，當學生提交作業時，自動檢查 `README.md` 是否包含必要的手機截圖。
  2. 撰寫提案：在 WORKFLOW.md 中用文字或偽代碼描述這個 GitHub Actions 的邏輯。
  3. 提交提案：將你的設計思維同步到 GitHub。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 可行性：提案的邏輯是否在 GitHub Actions 環境下可被實踐？
  • [ ] 效率提升：該工作流是否能顯著減少導師的人工檢查負擔？

導師能力矩陣操作指南

本單元是「領導力」的展現。教師應鼓勵學生從更高維度的開發者視角（架構、安全、維運）來思考問題，而非僅僅是單純的實作功能。
