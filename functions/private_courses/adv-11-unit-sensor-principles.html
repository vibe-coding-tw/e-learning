👁️ 1. 光電遮斷器：數位化的眼睛

透過紅外線對管與編碼盤（Encoder Disk）的組合，將旋轉的機械運動轉化為 0 與 1 的數位訊號。

Working Principle

  • 遮斷 (Blocked)：輸出高電位 (3.3V)。
  • 通過 (Clear)：輸出低電位 (0V)。
  • 解析度 (PPR)：一圈產生的脈衝數計為格數。

📏 3. 解析度與物理量映射

理解「一個脈衝代表前進多少公分」是自動導航的基石。這涉及到輪胎直徑與格數的幾何關係。

Kinematic Calibration

  • 解析度 (Ticks)：格數越多，解析度越高，能感知的最小移動量越小。
  • 遲滯 (Hysteresis)：防止在臨界位置時訊號因微小晃動而瘋狂跳動。
  • 映射邏輯：建立從脈衝到 RPM，再從 RPM 到 cm/s 的穩定轉換管線。

光電感測器與物理標定實驗室

任務 1：訊號觀測站：方波與波形分析 (Signal Observation & Waveform)

角色扮演：訊號分析師 | 難度：⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 建立倉庫：從 GitHub Classroom 連結建立個人作業倉庫並 Clone 到本地。
  2. 波形觀測：手動轉動馬達，利用 Serial Plotter 或示波器觀察光電感測器的輸出波形。
  3. 邊緣定義：識別並記錄數位訊號的上升沿 (Rising Edge) 與下降沿 (Falling Edge)。
  4. 電平驗證：測量並確認遮斷時的高電位 (High) 與通過時的低電位 (Low) 是否符合 ESP32 規範。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 訊號解讀力：能否在 README 中正確標註 0V 與 3.3V 代表的物理意義（光線遮斷 vs. 通過）？
  • [ ] 銳利度分析：觀察在高低電位切換時是否具備足夠的切換速度（Edge Sharpness），以支持後續的高速中斷計數？

任務 2：外部中斷計數器：精確脈衝統計 (Interrupt Counter & PPR)

角色扮演：數據採集工程師 | 難度：⭐️⭐️⭐️ | 估計時長：60 min

🛠️ 執行技術流程 (Git Workflow)

  1. 中斷配置：在 ESP32 上實作 attachInterrupt()，設定為 RISING 或 FALLING 觸發。
  2. 物理校準：手動旋轉輪胎剛好一圈，記錄產生的總脈衝數 (Ticks)。
  3. 解析度定義 (PPR)：根據實驗結果定義出該編碼盤的 PPR (Pulses Per Revolution) 參數。
  4. 穩定性測試：快速旋轉馬達，驗證在高速下計數值是否依然穩定，不發生漏噴或異常跳動。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 計數精準度：實測得到的一圈脈衝數是否與編碼器實體格數準確吻合？
  • [ ] 代碼規範：中斷服務常式 (ISR) 是否保持最精簡，無違禁的 Serial.print 或 delay 呼叫？

任務 3：即時轉速計：物理單位轉換 (Tachometer & Unit Scaling)

角色扮演：儀表板開發者 | 難度：⭐️⭐️ | 估計時長：45 min

🛠️ 執行技術流程 (Git Workflow)

  1. 公式推導：推導從脈衝數 (Ticks) 轉換為物理距離 (cm) 與每分鐘轉速 (RPM) 的數學比例尺。
  2. 即時演算實作：在主程式中定時計算過去一段時間內的脈衝增量，並換算為 cm/s 單位。
  3. 網頁同步顯示：將換算後的物理數據透過 BLE 發送，並在網頁 UI 儀表板上即時更新數值。
  4. 成果交付：將具備物理單位換算邏輯的代碼提交至 GitHub。

🎯 教師點評標準 (Tutor Benchmarks)

  • [ ] 轉換正確性：輸出的轉速與距離數據是否符合物理邏輯，且具備正確的量綱 (Dimension)？
  • [ ] UI 即時性：網頁端數據顯示是否能流暢反映馬達轉速的動態變化，延遲感低於 200ms？

GitHub Classroom 詳細操作指南

本單元教學重點：將「光與電」的連續物理現象轉化為「0 與 1」的高精度數位模型。

  1. 第一步：理解波形的意義 - 引導學員觀察真實波形中的雜訊與邊緣，建立對數位感知的硬體直覺。
  2. 第二步：標定物理比例尺 - 強調 PPR 參數是銜接虛擬代碼與真實世界的關鍵，必須透過實驗親手測得。
  3. 第三步：數據到價值的轉換 - 指導學員將抽象的 Ticks 轉換為有意義的 RPM，這是開發導航系統的最底層基礎。
