🎼 1. 硬體 PWM 三部曲

在 ESP32 中，PWM 是具備層級架構的精密模組。

⏱️ 定時器 (Timer)

決定頻率。共用定時器的頻道，頻率必須一致。

🔢 頻道 (Channel)

16 個獨立頻道 (0-15)，獨立控制強度。

🧩 解析度 (Resolution)

控制精細度。8-bit 對應 256 階；13-bit 對應 8192 階。

🔗 腳位綁定 (Attach)

頻道可映射到幾乎任何物理 GPIO 腳位。

⚙️ 2. 關鍵語法與核心 API

ledcSetup(ch, freq, res);

初始化：指定頻道、頻率、解析度。

ledcAttachPin(gpio, ch);

映射：腳位導通至邏輯頻道。

ledcWrite(ch, val);

輸出：更新占空比。上限取決於 Resolution。

思考題

播放「音階」時，改變 Freq 還是 Duty？為什麼？

C++ FRAMEWORK 
// 步驟：設定 -> 綁定 -> 驅動
void setup() {
  ledcSetup(0, 5000, 8);  // 5kHz, 8bit
  ledcAttachPin(14, 0);   // GPIO14 -> CH0
}

void loop() {
  ledcWrite(0, 128);      // 50% Power
  delay(1000);
}

⚡ 3. 為什麼不直接用 analogWrite？

LEDC 提供「硬體層級」精確資源控制。零延遲修改頻率（變頻）或更改映射，是運動控制必備能力。

動態變頻

優化馬達在不同轉速下的發熱與噪音。

位元對比

12-bit 下 4096 階控制，讓起步更穩健不暴衝。

實驗四：ESP32 LEDC PWM 控制與 API 封裝 (Student Assignment)

任務目標

在本單元中，你將學習如何利用 ESP32 的 LEDC (LED Control) 周邊來控制馬達速度與 LED 亮度。這項任務將幫助你從單純的「功能實作」進階到「韌體架構設計」，學習如何精確分配硬體資源。

任務清單

1. 硬體資源規劃 (Resource Management)

  • 目標：建立對硬體定時器 (Timer) 共享機制的認知。
  • 任務說明：在 include/ledc_init.h 中定義引腳與頻道。
  • 要求：馬達 (20kHz, 10-bit) 與燈光 (5kHz, 8-bit) 必須使用不同的定時器以避免衝突。

2. 速度控制 API 封裝 (API Builder)

  • 目標：學會隱藏底層位元 (Bit) 細節，提供友善的速度控制接口。
  • 任務說明：在 src/main.cpp 中實作 setMotorSpeed(int channel, int percent)。
  • 安全性設計：實作數值限制 (Clamping)，確保輸入超出 0-100% 範圍時能安全處理，防止資料溢位。

3. 諧振頻率掃描 (Frequency Sweep - 進階)

  • 目標：體會頻率對硬體效率與噪訊的影響。
  • 任務說明：測試不同 PWM 頻率（例如從 100Hz 到 40kHz），觀察馬達的表現與噪音。
  • 紀錄：將你的觀察與工程決策紀錄在 assets/sweep_log.csv 中。

如何開始

  1. 複製此 程式庫 (Repository) 到你的本地端。
  2. 使用 PlatformIO 打開專案。
  3. 按照 docs/API_Guide.md 的說明進行開發。

提交方式

  1. 完成程式碼開發並通過本地測試。
  2. 將更改 commit 並 push 到你的 程式庫。
  3. GitHub Actions 將自動運行評分腳本，請查看 Action 標籤以確認結果。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template Repo) 配置建議

LEDC 作業的核心在於「資源映射」與「API 抽象」，範本應提供清晰的資源邊界，建議包含：

  • 📂 src/main.cpp：核心實作區。學員需在此實作一個多頻道的初始化邏輯，並撰寫一個自定義的 setMotorSpeed(int percent) 函式，處理百分比與位元深度 (Resolution) 之間的數學轉換。
  • 📂 include/ledc_config.h：引腳與頻道定義中心。預先定義多個邏輯頻道（如 CH_L_MOTOR, CH_R_MOTOR, CH_HEADLIGHT），引導學員建立抽象層。
  • 📂 README.md：資源規劃報表。提供表格讓學員記錄：「Timer 索引」 vs 「頻道頻率」 vs 「解析度」。並在「諧振掃描實驗」紀錄馬達在不同頻率下的表現。
  • 📂 docs/API_DOCUMENTATION.md：API 設計說明。要求學員說明其 setMotorSpeed 的數學公式與邊界處理邏輯（如處理負值或超過 100% 的狀況）。

2. 作業任務部署細節

任務 1：「頻道接力賽」多路 PWM 資源管理 (Resource Audit Lab)

  • 目標：建立對硬體定時器 (Timer) 共享機制的認知，這是未來進行複雜機器人開發的基礎。
  • Classroom 部署建議：
    • 衝突測試：要求學員將兩個不同頻率的需求分配給同一個 Timer，觀察硬體報錯或頻率連動的現象。
    • 映射表撰寫：在 README 中彙整一套不產生衝突的資源清單：馬達 (20kHz, 10-bit), LED (5kHz, 8-bit)。
    • 驗證要點：確保程式碼中正確使用了 ledcSetup 指向不同的定時器以實現不同頻率的共存。

任務 2：「百分比適配器」自定義動力 API (API Builder Lab)

  • 目標：學會隱藏底層位元 (Bit) 細節，為其他開發者或導航算法提供友善的接口。
  • Classroom 部署建議：
    • 數學映射挑戰：撰寫 setMotorSpeed(int percent)。函式需能根據當前的解析度參數（如 10-bit 對應 1023），自動計算對應的 ledcWrite 值。
    • 保護邏輯：實作範圍限制 (Clamping)，確保輸入 150% 時僅輸出 100%，防止資料溢位導致馬達暴衝。
    • 提交結果：展示透過一個簡單的 Loop 呼叫該 API，馬達能平緩、線性地從 0% 加速到 100%。

任務 3：「諧振頻率掃描」尋找馬達甜蜜點 (Resonance Seek Lab)

  • 目標：體會頻率對應硬體效率與噪訊的影嚮，培養工程直覺。
  • Classroom 部署建議：
    • 自動掃描算法：撰寫一段程式碼，每隔 5 秒改變一次 PWM 頻率（從 100Hz 漸進到 40kHz）。
    • 感官實測紀錄：學員需在 README 中紀錄在哪些頻段會有明顯的高頻噪音（尖叫聲），以及在哪些頻段馬達轉動最為平穩。
    • 工程決策：解釋為何最終選擇了該特定的工作頻率（例如：選用 20kHz 是為了避開人耳聽覺範圍）。

3. LEDC 專業與資源管理點評標準 (Peripheral Design & Optimization Standards)

此單元的價值在於 「對硬體資源的精細規劃與軟體抽象的深度」：

  • [ ] 資源利用率：Timer 與 Channel 的分配是否考慮了未來擴充性？
  • [ ] API 魯棒性：setMotorSpeed() 是否處理了各種極限輸入？數學轉換公式是否簡潔高效？
  • [ ] 調優意識：是否理解解析度 (Resolution) 越高、可控頻率越低的硬體權衡 (Trade-off)？
