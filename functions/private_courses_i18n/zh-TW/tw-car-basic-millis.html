簡介

delay() 會讓整個程式「睡著」，在等待期間無法回應任何事件——這對需要同時處理多項任務的物聯網裝置是致命的。 millis() 提供了一種非阻塞的計時方式：不斷檢查「時間到了沒」，沒到就繼續做其他事。 這是從「新手寫法」邁向「專業架構」最重要的第一步。

情境

為什麼 delay() 是邪惡的？

在學習嵌入式開發的初期，delay() 是最直覺的計時方法——呼叫 delay(1000) 就能讓程式暫停一秒鐘。但這個看似方便的函式，實際上是一個「忙等待 (Busy-Wait)」迴圈：它讓 CPU 原地空轉，完全不做任何有意義的事情，直到倒數結束為止。

對於 ESP32 這類單執行緒的微控制器而言（Arduino 框架下的 loop() 只在一個核心上跑一條主迴圈），delay() 的影響是災難性的：在等待期間，所有輸入偵測、通訊處理、安全監控全部停擺。

delay() 的內部機制

當你呼叫 delay(ms) 時，Arduino 核心實際執行的是：

void delay(unsigned long ms) { unsigned long start = millis(); while (millis() - start < ms) { // CPU 在這裡空轉... 什麼都不做 yield(); // ESP32 會餵看門狗，但你的程式碼無法執行 } }

這意味著在 delay(1000) 的整整一秒鐘內，你的 loop() 函式被完全凍結——無法讀取按鈕、無法接收藍牙封包、無法檢查安全逾時。

安全警告 在控制馬達的系統中使用 delay()，意味著在等待期間機器人無法接收緊急停止指令。若發生碰撞或故障，系統將無法及時反應——這是真實世界中的安全隱患。

具體影響分析

影響面向 使用 delay() 嚴重程度
使用者輸入 按鈕、遙桿指令在等待期間全部被忽略 高
通訊可靠性 BLE/WiFi 封包堆積導致緩衝區溢位或連線逾時 高
感測器精度 取樣間隔不穩定，PID 控制出現抖動 高
安全機制 Watchdog 逾時、碰撞偵測延遲 致命
多工能力 無法同時處理閃燈 + 通訊 + 感測 高

實際案例：LED 閃爍阻斷遙控

考慮以下常見的新手寫法——讓 LED 每 500ms 閃爍一次，同時監聽藍牙指令：

void loop() { digitalWrite(LED_PIN, HIGH); delay(500); // 🚨 此處凍結 500ms digitalWrite(LED_PIN, LOW); delay(500); // 🚨 再凍結 500ms // 🔥 這裡的藍牙指令最多延遲 500ms 才被處理！ if (BLE.available()) { handleCommand(); } }

在最壞情況下，使用者送出「緊急停止」指令後，系統需要等待最多 500ms 才能收到——對時速 1 公尺/秒的小車而言，這等於白白多衝出 50 公分。

關鍵認知 在專業的嵌入式開發中，loop() 應該盡可能快速循環（理想情況下每次迭代 < 1ms）。任何「等待」都應該用非阻塞的時間檢查來實現，而非讓 CPU 停擺。下一節我們將學習如何用 millis() 達成這個目標。

millis() 碼錶邏輯

millis() 是 Arduino 框架中的系統時鐘函式，它回傳自開機以來經過的毫秒數。與 delay() 不同的是，呼叫 millis() 只是「看一眼時鐘」——瞬間完成，不會阻塞任何程式碼。

非阻塞計時的核心思想是：不要叫 CPU 等待，而是讓它不斷檢查「時間到了沒」。如果還沒到，就繼續做其他事情；如果到了，就執行對應的任務並更新時間戳。這就像你看手錶決定是否該行動，而不是閉眼數秒。

基礎公式

非阻塞計時的判斷條件為：

currentMillis - previousMillis >= interval

其中：

  • currentMillis：當前的系統時間（呼叫 millis() 取得）
  • previousMillis：上一次執行任務時的時間戳記
  • interval：期望的時間間隔（毫秒）

完整範例：非阻塞 LED 閃爍

const int LED_PIN = 2; const unsigned long BLINK_INTERVAL = 500; // 500ms 閃爍週期 unsigned long previousMillis = 0; bool ledState = false; void setup() { pinMode(LED_PIN, OUTPUT); } void loop() { unsigned long currentMillis = millis(); if (currentMillis - previousMillis >= BLINK_INTERVAL) { previousMillis = currentMillis; // 更新時間戳 ledState = !ledState; digitalWrite(LED_PIN, ledState); } // ✅ 程式不會停在這裡——可以繼續處理其他任務 handleBluetooth(); checkSensors(); }
注意 務必使用 unsigned long 型別儲存時間戳。millis() 的回傳值會在約 49.7 天後溢位歸零，但只要使用減法判定（now - last >= interval），即使發生溢位，邏輯依然正確——這是無號整數減法的數學特性。

為什麼減法判定能防溢位？

假設 unsigned long 為 32 位元，最大值為 4,294,967,295。當 millis() 溢位：

變數 值 說明
previousMillis 4,294,967,290 接近溢位邊界
currentMillis 5（溢位後） 已經繞回起點
current - previous 11 無號減法自動得到正確差值
常見錯誤 切勿使用 if (millis() > previousMillis + interval) 這種加法寫法——當 previousMillis + interval 溢位時，條件判斷會失效。永遠使用減法形式。

millis() 模式的運作流程

每次 loop() 迭代的執行流程如下：

1

取得當前時間

呼叫 millis() 記錄「現在幾點」。

2

比較時間差

計算 now - last，判斷是否已達到設定的間隔。

3

條件執行

若時間到了，執行任務並更新 previousMillis = currentMillis。

4

繼續循環

無論時間到或沒到，loop() 都會立即繼續下一輪——不會停頓。

合作式多工：任務並行

掌握了 millis() 的非阻塞計時後，下一步就是將多個獨立任務整合到同一個 loop() 中。這種架構稱為合作式多工 (Cooperative Multitasking)——每個任務各自管理自己的時間戳，輪流佔用極短的 CPU 時間，整體看起來就像「同時運行」。

與作業系統的搶佔式多工不同，合作式多工要求每個任務主動讓出 CPU（即快速完成並返回），不能獨佔處理器。只要遵守這個規則，你就能在單一核心上同時處理感測器讀取、LED 指示、通訊回應、安全監控等多項任務。

多任務調度架構

以一輛遙控車為例，同時需要以下任務：

任務 頻率 間隔 (ms) 說明
感測器讀取 100 Hz 10 超音波測距或循線陣列
PID 修正 50 Hz 20 馬達速度控制運算
LED 狀態指示 2 Hz 500 系統心跳閃爍
電量回報 1 Hz 1000 ADC 讀取電池電壓
安全看門狗 0.5 Hz 2000 逾時強迫停機

完整實作範例

// ─── 各任務獨立的時間戳 ─── unsigned long lastSensor = 0; unsigned long lastBlink = 0; unsigned long lastBattery = 0; unsigned long lastCmdTime = 0; const unsigned long SENSOR_INTERVAL = 10; const unsigned long BLINK_INTERVAL = 500; const unsigned long BATTERY_INTERVAL = 1000; const unsigned long WATCHDOG_TIMEOUT = 2000; void loop() { unsigned long now = millis(); // 任務 1：高頻感測器讀取 (100Hz) if (now - lastSensor >= SENSOR_INTERVAL) { lastSensor = now; readSensors(); } // 任務 2：LED 心跳閃爍 (2Hz) if (now - lastBlink >= BLINK_INTERVAL) { lastBlink = now; toggleHeartbeat(); } // 任務 3：電量回報 (1Hz) if (now - lastBattery >= BATTERY_INTERVAL) { lastBattery = now; reportBatteryLevel(); } // 安全任務：Watchdog 逾時停機 if (now - lastCmdTime > WATCHDOG_TIMEOUT) { emergencyStop(); } // 持續處理通訊（非定時，每次 loop 都執行） handleBluetooth(); }
設計原則 每個任務函式（如 readSensors()）的執行時間應控制在微秒等級。如果某個任務需要較長時間（例如等待感測器回應），應該將它拆分成「啟動」和「讀取」兩個階段，用狀態機模式實作。

合作式多工的黃金守則

  1. 絕不阻塞：loop() 中禁止任何 delay()、while(等待) 或長時間計算。
  2. 各自管理時間：每個任務擁有獨立的 previousMillis 變數，互不干擾。
  3. 快進快出：每個任務函式執行後立即返回，讓出 CPU 給下一個任務。
  4. 優先處理安全：Watchdog 檢查可以放在每次 loop 都執行（不需要計時器），確保最快反應。
Jitter（抖動）問題 若某個任務的執行時間超出預期（例如 I2C 通訊卡住），會導致其他任務的觸發時間被延後——這就是「抖動」。在作業中你將學習如何量測 Jitter，並透過程式架構優化將其最小化。

  • 目標：透過量化實驗證明非阻塞邏輯在安全性上的巨大優勢，並實作超時強迫停機邏輯。
  • Classroom 部署建議：
    • 量化對照實驗：在 README 中記錄：當 loop 中有 delay(1000) 時，按下剎車後的反應時間；與改用 millis() 後的反應時間。
    • 安全守護 (Watchdog)：實作一個獨立的計時器。若超過 1.5 秒未接收到遙控指令，系統必須立即強迫停機，且此邏輯不能被閃燈任務干擾。
    • 驗證要點：實驗日誌是否展示了非阻塞邏輯對剎車距離的顯著改善？失效停機邏輯是否能在閃燈運行的同時可靠發揮作用？

3. 多工邏輯與系統流暢度點評標準 (Concurrency & Fluidity Standards)

此單元的價值在於 「對 CPU 時間片價值的極致克扣與對異步任務編排的架構能力」：

  • [ ] 代碼潔淨度 (Delay-Free)：是否徹底將阻塞代碼從主循環中驅逐出境？
  • [ ] 時間戳管理嚴謹度：是否使用了 unsigned long 並正確處理了溢位風險？
  • [ ] 安全韌實力：系統是否展現出「眼觀四路、耳聽八方」的靈敏反應，尤其是在緊急制動場景下？
