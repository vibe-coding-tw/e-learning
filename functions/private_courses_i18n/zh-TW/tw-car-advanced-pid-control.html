🙈 1. 開迴路控制：盲目的信任

在控制工程中，開迴路控制 (Open-Loop Control) 是最原始也最簡單的控制方式。系統僅根據事先設定好的指令執行動作，完全不觀察輸出結果是否符合預期。就像你蒙著眼睛走路——你知道自己要走十步，但完全不知道走到了哪裡。

想像你用微波爐加熱牛奶：你設定了 30 秒，按下開始。微波爐不會量測牛奶的溫度，它只管轉 30 秒就停。如果今天放的是冷凍牛奶，30 秒可能還是冰的；如果是常溫牛奶，30 秒可能已經燙到溢出。這就是開迴路控制的本質——輸出完全不影響輸入決策。

開迴路系統的方塊圖

開迴路系統的信號流非常單純：由控制器產生一個固定指令，經由致動器 (Actuator) 作用於受控體 (Plant)，產生輸出。但輸出結果並沒有任何路徑回傳給控制器做比較。

// 開迴路系統方塊圖示意
┌─────────┐ ┌──────────┐ ┌────────┐
│ 固定指令 │───▶│ 馬達驅動 │───▶│ 車輪轉動│───▶ 輸出 (位置)
│ (PWM=200)│ │(Actuator) │ │ (Plant) │
└─────────┘ └──────────┘ └────────┘
// ↑ 無反饋路徑！

程式碼範例：固定 PWM 驅動

以下是一段典型的開迴路 Arduino 程式碼。馬達以固定 PWM 值運轉 2 秒後停止，完全不管車子跑到了哪裡：

void loop() {
  // 開迴路：給定固定出力，持續固定時間
  analogWrite(MOTOR_PIN, 200);  // 固定 PWM = 200
  delay(2000);                 // 跑 2 秒
  analogWrite(MOTOR_PIN, 0);    // 停止
  while(true);                // 永久停機
}
注意 這段程式在磁磚地板上可能停在 1.5 公尺處，但換到地毯上可能只走了 0.8 公尺。同樣的程式、同樣的指令，結果卻天差地遠——這就是開迴路控制在真實物理世界中的致命弱點。

開迴路控制的特性總結

特性 說明
無反饋 輸出不影響輸入，系統僅憑事先設定的時序或數值運行
抗干擾力極弱 電壓波動、路面摩擦改變、負載變化都會直接導致偏差
結構簡單 不需要感測器，硬體成本低，程式碼短
適用場景有限 僅適合環境極度穩定且不需要精準度的場合（如定時烤箱）
學習重點 開迴路控制本身並不是「壞的」設計——在環境完全可預測的情境下，它反而是最經濟高效的方案。但在真實物理世界中，擾動 (Disturbance) 無處不在，因此我們需要引入反饋機制來應對不確定性。下一單元將揭示閉迴路控制如何解決這個問題。

👁️ 2. 閉迴路控制：觀察與修正的智慧

閉迴路控制 (Closed-Loop Control) 的核心思想只有一句話：觀察結果、計算誤差、修正行為。系統具備感測器，能將目前的實際輸出（如位置、速度、溫度）即時回傳，與目標值進行比較，計算出「誤差 (Error)」，然後根據誤差大小調整出力。

回到淋浴調水溫的比喻：你用手感受水溫（感測器），發現太冷（誤差為正），於是轉大熱水（增加出力）；感覺差不多了（誤差趨近零），就停止調整。這個「感知→比較→修正」的持續循環，就是閉迴路控制的本質。

負反饋的核心公式

閉迴路系統最關鍵的運算只需一行：

// 負反饋核心公式
error = target - measured;

// 其中：
// target = 你希望到達的目標值（如 150 cm）
// measured = 感測器回報的當前值（如 92 cm）
// error = 150 - 92 = 58 cm（還差 58 公分）

當 error 為正，代表尚未到達目標，系統應持續出力；當 error 為負，代表超過目標，系統需要反向修正；當 error 等於零，系統完美地停在目標位置。

閉迴路系統方塊圖

// 閉迴路系統 — 反饋路徑是關鍵差異
              ┌─────────┐   ┌──────────┐   ┌────────┐
Target ──(+)──▶│ 控制器   │──▶│ 馬達驅動  │──▶│ 車輪轉動│──▶ Output
          ▲   └─────────┘   └──────────┘   └────────┘    │
          │                                            │
          │           ┌──────────┐                   │
          └────(-)─────│ 感測器   │◀────────────────────┘
                     └──────────┘
//       ↑ 這條回傳路徑就是「反饋 (Feedback)」
觀念 「負反饋」中的「負」不代表壞事——它代表系統會朝減少誤差的方向修正。如果沒有這個「負號」，系統會越跑越偏，形成災難性的正反饋（如麥克風靠近喇叭產生尖嘯）。

開迴路 vs. 閉迴路比較

比較項目 開迴路 (Open-Loop) 閉迴路 (Closed-Loop)
反饋機制 無 有感測器回傳
抗干擾能力 極弱，環境一變就失準 強，能即時補償擾動
精準度 取決於校正品質，難以保證 可持續逼近目標值
硬體複雜度 低（無需感測器） 較高（需要感測器與運算）
穩定性風險 不會震盪，但可能永遠偏離 若控制器設計不當，可能產生震盪

最簡單的閉迴路控制實作

以下程式碼展示了最基本的閉迴路架構：讀取感測器 → 計算誤差 → 決定出力方向：

void loop() {
  float currentPos = readEncoder();  // 讀取當前位置
  float error = targetPos - currentPos; // 計算誤差

  if (error > 0) {
    motorForward();               // 尚未到達，前進
  } else {
    motorStop();                  // 到達或超過，停止
  }
}
思考題 上面的程式已經是閉迴路了，但它只做了「到達就停」的簡單判斷。如果目標還很遠，馬達全速衝；快到的時候呢？它依然全速衝——直到衝過頭才急停。這就引出了下一個問題：我們需要更聰明的控制策略，讓出力與誤差「成比例」。這正是 PID 登場的前奏。

🎛️ 3. 進化：為什麼需要 PID？

你可能會想：閉迴路控制已經有了反饋機制，為什麼還需要 PID？答案很簡單——單純的「有誤差就出力」太粗暴了。比例控制 (P) 雖然讓系統學會了「看路」，但它有三個致命缺陷：靜態誤差、震盪、慣性過衝。PID 正是為了解決這三個問題而誕生。

純比例控制的三大困境

情境
你的機器人使用 P 控制，目標距離 150 cm。一開始跑得很好，但到了 145 cm 時，剩餘誤差只有 5 cm，P 控制器的出力太小，不足以克服靜摩擦力——機器人卡住了，永遠差那最後 5 公分。
困境 現象描述 PID 解法
靜態誤差
(Steady-State Error)		 誤差很小時，P 輸出低於馬達啟動電壓，系統卡在終點前 I (積分)：隨時間累積誤差，持續增加推力直到克服阻力
震盪
(Oscillation)		 Kp 太大時，系統在目標附近來回擺動，無法收斂 D (微分)：偵測誤差變化速率，在衝過頭之前提前踩煞車
慣性過衝
(Overshoot)		 高速接近目標時，機械慣性使系統衝過目標點 D (微分)：預判趨勢，提前減速讓系統平滑收斂

PID 三兄弟的直覺

PID 控制器由三個獨立的項組合而成，各自處理不同的時間維度：

  • P (Proportional) — 現在：「離目標還有多遠？」——誤差越大，出力越猛。提供即時反應力。
  • I (Integral) — 過去：「已經偏離目標多久了？」——累積歷史誤差，剷除頑固的殘留偏差。
  • D (Derivative) — 未來：「正在以多快的速度接近目標？」——預判趨勢，防止過衝。
// PID 完整公式（連續時間）
Output = Kp × Error
       + Ki × ∫ Error dt
       + Kd × d(Error)/dt

// 離散化實作（嵌入式系統）
output = Kp * error
       + Ki * (integral += error * dt)
       + Kd * (error - lastError) / dt;
關鍵觀念 PID 的威力不在於三個公式有多複雜，而在於它們協同作戰：P 提供反應速度，I 提供精度保證，D 提供穩定性。三者缺一不可——只有 P 會卡住，加了 I 會衝過頭，再加 D 才能優雅收斂。

從學術到實戰

PID 控制器發明於 1920 年代，至今仍是工業界使用最廣泛的控制演算法。據統計，全世界超過 90% 的工業控制迴路使用某種形式的 PID。從你家的冷氣溫控、汽車的定速巡航、到無人機的姿態穩定——底層都是同一套 PID 邏輯。

歡迎來到 PID 控制原理 單元！在這個作業中，你將從「盲目駕駛」進化到「有感反饋」，親身體會如何透過比例控制 (P Control) 讓機器人更精準地到達目標位置。

任務 3：靜態誤差與阻力分析 (Steady-State Error Analysis)

  • 說明：在機器人上增加負荷（例如放一瓶水），觀察 P 控制為何會「差一點點」停在目標點前。

1. 範本程式庫 (Template Repo) 配置建議

控制理論需要嚴謹的實驗對照與數據記錄，建議範本包含：

  • 📂 src/open_loop_drive.ino：開迴路測試腳本。僅憑時序控制，讓學員在 README 記錄不同摩擦力地表的慘重誤差。
  • 📂 src/p_regulator.ino：P 控制器實作。提供 target 與 current 讀取介面，留下 Output = Kp * (Target - Current) 的 TODO。
  • 📂 PERFORMANCE_LOG.md：效能對比日誌。提供標準化表格，讓學員紀錄「地表類型」、「停車誤差公分數」以及「震盪頻率」。
  • 📂 tests/feedback_integrity.test.js：自動化邏輯檢核。驗證當誤差減小時，輸出量是否也隨之線性下降（確保負回饋方向正確）。

2. 作業任務部署細節

任務 1：開迴路盲測實驗 (Blind Open-Loop Task)

  • 目標：打破「程式寫好就一定會跑準」的幻想，理解環境擾動的威力。
  • Classroom 部署建議：
    • 故障模擬檢測：學員需在磁磚與地毯上執行同一個 delay(2000) 指令。
    • 誤差量化紀錄：在 README 中上傳照片，展示在兩種環境下停車點的巨大落差（通常會有 10-30 公分不等的偏差）。
    • 原因分析報告：說明為什麼純時序控制無法應對物理世界的變動。

任務 2：P 比例控制器實作 (Proportional Feedback Design)

  • 目標：賦予機器人「視覺（反饋）」，實現自動導向目標。
  • Classroom 部署建議：
    • 反饋邏輯實作： 
      // 學生需實作：負回饋計算
float error = targetPos - currentPos;
float output = Kp * error;
motorDrive(output);
    • 穩定性調校：挑戰尋找能讓機器人「最快到達、且不發生劇烈震盪」的黃金 $K_p$ 值。
    • 震盪觀測驗證：要求學員錄影展示：當 $K_p$ 太大時，機器人是如何像沒對準的舵機一樣瘋狂抖動。

任務 3：靜態誤差與阻力分析 (Steady-State Error Analysis)

  • 目標：發現 P 控制的阿基里斯之踵——無法克服靜摩擦力的「最後一哩路」。
  • Classroom 部署建議：
    • 負重挑戰：在機器人上放一瓶水，觀察 P 控制為何會「差一點點」停在目標點前，且馬達只發出嗡鳴聲而不動。
    • 物理死區分析：要求學員解釋「為什麼當誤差極小時，比例出力會低於馬達啟動電壓？」。
    • PR 深度核核：點評其對解決方案的預見。是否能正確推導出「我們需要一個能隨著時間不斷累積力量的項（積分 I）」來解決這個問題？
    • 數據趨勢圖：在 Serial Plotter 中截圖展示 Error 曲線最終停在一條水平線上（非零）的現象。

3. 控制工程導師點評標準 (Control Engineering Benchmarks)

此單元的價值在於 「對系統行為的量化診斷能力」：

  • [ ] 誤差敏感度：代碼是否能隨時根據位置變化調整驅動力？
  • [ ] 穩定性邊界意識：學員是否理解了增益過大會導致系統發散（震盪）？
  • [ ] 物理侷限洞察：是否能清楚描述「靜態誤差 (Steady-state error)」的成因？
