簡介

BLE 預設的 MTU 只有區區 20 Bytes 可用載荷——這意味著一段稍微複雜的 JSON 指令就會被無聲地截斷，導致接收端解析失敗。 理解 MTU 的物理限制與分段傳輸策略，是讓你的藍牙通訊從「偶爾能動」變成「穩定可靠」的關鍵一步。 本模組將教你如何交涉更大的 MTU，以及當交涉失敗時如何在軟體層實作可靠的分段與重組機制。

情境
你透過 BLE 從網頁發送一段 JSON 控制指令 {"cmd":"move","speed":80,"angle":45,"mode":"auto"} 給 ESP32，但 ESP32 那端只收到 {"cmd":"move","speed 就斷了，後面全部消失。你反覆檢查程式碼，傳送端明明送出了完整字串，為什麼接收端永遠只拿到前 20 個字元？這個「幽靈截斷」的元凶，正是 BLE 協議的 MTU 限制。

什麼是 MTU (Maximum Transmission Unit)？

MTU（Maximum Transmission Unit，最大傳輸單元）是指在通訊鏈路中，單一封包（Packet）所能承載的最大位元組數。在 BLE（Bluetooth Low Energy）協議中，MTU 直接決定了每一次無線傳輸可以攜帶多少有效數據——超過此限制的部分將被無聲地截斷，而不會產生任何錯誤提示。

BLE 封包結構解析

要理解 MTU 的限制，首先必須了解 BLE 封包的分層結構。一個 BLE 數據封包由多層協議標頭（Header）層層包裹，每一層都會「吃掉」一部分空間，最終留給應用層的有效載荷（Payload）其實非常有限。

層級 協議 佔用空間 說明
鏈路層 BLE Link Layer 依實作而定 無線電封包框架與位址資訊
邏輯鏈路層 L2CAP 4 Bytes 封包長度 + 通道 ID (CID)
屬性協議層 ATT（Attribute Protocol） 3 Bytes 操作碼 (Opcode) + 屬性控制碼 (Handle)
應用層 你的 JSON 數據 20 Bytes（預設） 實際可用的有效載荷空間
備註 BLE 4.0/4.1 的預設 ATT MTU 為 23 Bytes。扣除 3 Bytes 的 ATT 標頭後，留給應用數據的空間恰好是 23 - 3 = 20 Bytes。這就是為什麼你的 JSON 字串一旦超過 20 個字元就會被截斷。

生活比喻：電梯限重

MTU 就像一台電梯的載重限制。假設你需要將一台鋼琴（一段大型 JSON 數據）送到 10 樓，但電梯最多只能承載 20 公斤。你有兩個選擇：向管委會申請換一台更大的貨梯（MTU 交涉），或者先把鋼琴拆成零件分批運送（軟體分段）。兩種策略各有適用場景，而最穩健的做法是——兩者同時實作。

實際案例：被截斷的 JSON

以下範例展示了一段常見的無人車控制指令如何被 MTU 限制無聲截斷：

// 原始 JSON 指令（49 Bytes）
const cmd = '{"cmd":"move","speed":80,"angle":45,"mode":"auto"}';

// ESP32 在預設 MTU (20 Bytes) 下實際收到的內容
// → '{"cmd":"move","spee' （只有前 20 個字元）

// JSON.parse() 嘗試解析 → 直接拋出 SyntaxError！
警告 BLE 的截斷行為是靜默的（Silent Truncation）——發送端不會收到任何錯誤回報，接收端也不會知道數據不完整。這使得問題極難排查，因為程式碼看起來完全正確，但接收端總是只拿到殘缺的片段。

不同情境下的 MTU 範圍

情境 ATT MTU 有效載荷 適用場景
BLE 4.0 預設 23 Bytes 20 Bytes 短指令、單一數值傳送
交涉後（iOS 常見） 185 Bytes 182 Bytes 中型 JSON 封包傳輸
交涉後（Android / BLE 5.0+） 517 Bytes 514 Bytes 大型封包、高效率傳輸
提示 即使你的裝置支援 MTU 交涉，也應該同時實作軟體分段機制作為最終防禦策略。因為交涉結果取決於雙方硬體的共同上限，你無法保證每台手機都能協商到理想的 MTU 值。

分段傳輸與 MTU 交涉

當你的數據長度超過 MTU 限制時，有兩個主要策略可以確保完整傳輸：MTU 交涉（Negotiation）向硬體請求更大的封包空間，以及軟體分段（Software Segmentation）在應用層將數據手動切片。實務上，最穩健的架構是兩者並行——交涉提升效率，分段保證容錯。

策略一：MTU 交涉（Negotiation）

在 BLE 連線建立後，客戶端可以向伺服器端發送 Exchange MTU Request，請求提升雙方的 MTU 上限。最終的 MTU 值由雙方各自支援的最大值中取較小的那個決定。例如，若手機支援 512 Bytes、ESP32 支援 247 Bytes，則最終協商結果為 247 Bytes。

備註 Web Bluetooth API 目前不提供直接呼叫 requestMTU() 的方法。MTU 交涉通常由瀏覽器在連線時自動完成。在 Android 上，BLE 堆疊會嘗試交涉至最大 517 Bytes；在 iOS 上，通常協商到 185 Bytes 左右。

各平台交涉結果差異

平台 典型協商結果 有效載荷 是否可手動控制
Android（Chrome） 517 Bytes ~514 Bytes 自動（系統控制）
iOS（Safari / Bluefy） 185 Bytes ~182 Bytes 自動（系統控制）
Windows（Chrome） 變動 變動 依藍牙驅動而定
ESP32（Arduino BLE） 最高 517 Bytes ~514 Bytes 可透過 pServer->setMTU() 設定

策略二：軟體分段（Software Segmentation）

軟體分段是最可靠的策略，完全不依賴硬體交涉結果。核心思想是：在發送端將長字串依照 MTU 大小切割為多個小片段（Chunks），逐一發送；在接收端將片段累積到緩衝區，收齊後再重組還原。

分段演算法實作

function fragmentData(text, mtu) {
  const chunks = [];
  for (let i = 0; i < text.length; i += mtu) {
    chunks.push(text.substring(i, i + mtu));
  }
  return chunks;
}

// 範例：將 JSON 切成 20 Bytes 的片段
const json = '{"cmd":"move","speed":80,"angle":45}';
const parts = fragmentData(json, 20);
// → ['{"cmd":"move","spee', 'd":80,"angle":45}']

逐片發送流程

1

切割數據

使用 fragmentData() 將完整 JSON 字串依照 MTU 大小切割為字串陣列。

2

逐片寫入 Characteristic

使用 for...of 迴圈逐一呼叫 characteristic.writeValue()，每次送出一個片段。兩次寫入之間建議加入短暫延遲（如 20ms）以避免 BLE 堆疊溢位。

3

追加終止符

在最後一個片段的末尾加上約定的終止符號（如 \n），讓接收端知道一筆完整的數據已傳送完畢。

重要 即使 MTU 交涉成功，仍強烈建議同時實作軟體分段邏輯。不同品牌的手機、不同版本的作業系統、甚至不同的藍牙晶片，都可能導致協商結果不一致。軟體分段是你的「最終防禦線」。

計算練習

假設無人車需要傳送以下狀態 JSON：{"x":102,"y":-45,"v":120,"b":85}（長度為 35 Bytes）。在預設 MTU = 20 Bytes 的限制下，需要切成 ⌈35 ÷ 20⌉ = 2 個片段才能完成傳輸。如果再加上終止符 \n（+1 Byte = 36 Bytes），仍然是 2 個片段（第一片 20 Bytes，第二片 16 Bytes）。

提示 片段數量的計算公式為 Math.ceil(data.length / mtu)。在規劃 BLE 通訊協議時，片段越多意味著傳輸延遲越高（每個片段需要一次完整的 BLE 寫入交易），因此縮短 JSON 長度同樣是提升效率的關鍵。

數據完整性與終止符

當數據被切成多個片段分批發送後，接收端面臨一個關鍵問題：如何判斷所有片段已經全部到齊？接收端既不知道原始數據有多長，也不知道總共會收到幾個片段。此時，終止符（Terminator）就是解決這個問題的核心機制——它就像一封信的最後一頁蓋上「全文完」的印章。

終止符的運作原理

終止符是發送端與接收端事先約定的特殊字元，附加在完整數據的末尾。接收端持續將收到的片段累加到緩衝區（Buffer），直到偵測到終止符，才認定數據已完整並觸發解析動作。未見到終止符之前，所有片段都只是暫存，不會嘗試解析。

常見的終止符選擇

終止符 字元表示 優點 注意事項
換行符 \n 通用性高，Serial Monitor 友善 JSON 內部不應包含裸換行符
空字元 \0 C 語言字串的天然邊界 部分 JS 函式庫可能忽略此字元
井號 # 可見字元，方便人工除錯 需確保 JSON 數據本體不含此字元
自訂封包頭 LEN:DATA 支援長度預告，可提前分配記憶體 實作較複雜，需額外解析邏輯
提示 對於 JSON 數據傳輸，推薦使用 \n（換行符）作為終止符。因為合法的 JSON 字串內部不會出現裸換行（換行必須以 \\n 跳脫序列表示），所以不會造成誤判。

緩衝區重組實作

接收端的重組器需要維護一個字串緩衝區，持續累積收到的片段，偵測到終止符後才觸發解析：

let buffer = '';

function onDataReceived(chunk) {
  buffer += chunk;

  if (buffer.includes('\n')) {
    const complete = buffer.split('\n')[0];
    try {
      const data = JSON.parse(complete);
      console.log('解析成功:', data);
    } catch (e) {
      console.warn('JSON 解析失敗:', e.message);
    }
    buffer = ''; // 無論成敗，都重置緩衝區
  }
}
警告 JSON.parse() 必須包裹在 try-catch 中。在分段傳輸的場景下，封包遺失是常態——如果某個片段因無線干擾而丟棄，拼湊出的字串將是殘缺的 JSON，直接呼叫 JSON.parse() 會拋出 SyntaxError，可能導致整個應用程式崩潰。

生產環境的安全機制

在正式部署的系統中，重組器還需要額外的防護措施：

  • 最大長度限制：設定 Buffer 的最大容量（例如 4096 Bytes），超過時自動清空並記錄警告，防止惡意或錯誤數據導致記憶體溢出（OOM）。
  • 超時機制：如果一段時間內（例如 5 秒）都沒有收到終止符，自動丟棄 Buffer 中的殘餘數據，避免過期片段干擾後續傳輸。
  • 每次解析後歸零：無論解析成功或失敗，Buffer 都必須在處理後清空，確保下一筆數據從乾淨的狀態開始累積。
備註 在 ESP32 韌體端，Arduino 的 String 類型會自動處理字串拼接，但要注意 ESP32 的可用 RAM 僅約 320 KB。如果 Buffer 無限成長而未設上限，很容易導致堆記憶體耗盡、觸發看門狗（Watchdog），造成裝置自動重啟。

adv-03-unit-ble-mtu: MTU 與分段傳輸

本程式庫是關於 BLE (Bluetooth Low Energy) 通訊中 MTU (Maximum Transmission Unit) 與 數據分段 (Fragmentation) 的進階實作練習。你將學習如何突破 BLE 的物理傳輸限制，穩定地傳輸大型 JSON 數據。

🎯 學習目標

  1. 理解 BLE MTU 的概念及其對傳輸效能的影響。
  2. 實作數據分段演算法，將大型字串切分為符合 MTU 限制的片段。
  3. 實作接收端緩衝區重組器，利用終止符 (Terminator) 完整還原數據。
  4. 分析 MTU 協商 (Negotiation) 前後的效能差異。

🛠 任務說明

任務 1：「電梯拆解工」：手動分段發送器

在 src/ble_fragmenter.js 中實作 fragmentData(text, mtu) 函數。

  • 目標：依據指定的 MTU 大小，將長字串正確切分為字串陣列。
  • 關鍵點：確保合併後的片段與原文字串完全一致。

任務 2：「拼圖高手」：緩衝區重組器

在 src/ble_reassembler.js 中完善重組邏輯。

  • 目標：建立一個累計緩衝區，只有在偵測到終止符（如 \n）時才嘗試解析 JSON。
  • 關鍵點：必須包含 try-catch 錯誤處理，防止因為數據殘缺導致程式當機。

任務 3：效能分析與實驗紀錄

使用 ESP32 開發板配合 Web Bluetooth API 進行測試。

  • 目標：比較預設 MTU (23 Bytes) 與請求增大 MTU (如 247 Bytes) 後的傳輸時間。
  • 產出：將觀察結果紀錄於 docs/mtu_benchmarks.md。

  • 📂 ANALYSIS.md：預置效能對比表，讓學員紀錄提升 MTU 前後的傳輸成功率與耗時延遲。

2. 作業任務部署細節

任務 1：「電梯拆解工」：手動分段發送器 (MTU Fragmenter Lab)

  • 目標：掌握字串切割演算法，突破 BLE 物理載荷限制。
  • Classroom 部署建議：
    • 核心邏輯檢核： 
      // 學生需實作：依據 MTU 大小將字串切分為 Array
function fragmentData(text, mtu) {
    let chunks = [];
    for (let i = 0; i < text.length; i += mtu) {
        chunks.push(text.substring(i, i + mtu));
    }
    return chunks;
}
    • Autograding：測試一段 100 字元的字串，設定 MTU=20，檢查是否回傳恰好 5 個片段，且合併後與原文字串完全一致（包含空格與符號）。

任務 2：「拼圖高手」：緩衝區重組器 (Puzzle Reassembler)

  • 目標：實作狀態機緩衝區，利用「終止符 (Terminator)」觸發原子解析。
  • Classroom 部署建議：
    • 驗證方式：學員需在接收端實作累積 Buffer。只有當偵測到 \n 或 \0 時，才執行 JSON.parse()。
    • 核核重點：導師應在 PR 中檢查學員是否對 JSON.parse 進行了 Try-Catch 封裝。在分段傳輸中，數據丟包是常態，若解析到殘缺 JSON 導致前端 JS 當機，即視為失敗。

任務 3：MTU 請求與效能分析 (MTU Request & Analysis)

  • 目標：理解硬體協商 (Negotiation) 的價值與頻寬開銷 (Overhead) 的關係。
  • Classroom 部署建議：
    • 實驗紀錄：學員需嘗試調用 requestMTU(247)，並對比在傳輸 1KB 數據時，「多個 20B 小包」與「少數 247B 大包」的總傳輸時間差。
    • 理論分析：在 ANALYSIS.md 中解釋為什麼大包傳輸更快（提示：減少了每個封包重複的 L2CAP/ATT 標頭開銷）。

3. 進階通訊導師點評標準 (Communication Benchmarks)

此單元的價值在於 「對通訊邊界案例 (Edge Cases) 的處理」：

  • [ ] 邊界嚴謹度：當數據長度恰好是 MTU 的倍數時，分段邏輯是否會產生空包或遺失末尾？
  • [ ] 內存安全性：緩衝區 (Buffer) 是否在每次成功解析後歸零？是否有設置最大容量上限（防止惡意超大包導致 OOM）？
  • [ ] 異質平台理解：學員是否能指出不同手機系統對 MTU 協商結果的差異，並提出軟體分段作為「最終防禦線」的必要性？
