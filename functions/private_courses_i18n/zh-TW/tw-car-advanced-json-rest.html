數據打包

在物聯網系統中，感測器採集到的原始數據需要被「打包」成結構化的格式，才能在設備與伺服器之間有效傳遞。JSON (JavaScript Object Notation) 是目前最廣泛使用的輕量級數據交換格式，它具備人類可讀、機器可解析的雙重優勢，已成為 RESTful API 的事實標準。

為什麼不能只傳原始數字？

許多初學者傾向使用逗號分隔的純文字格式（如 "25.5,60,1,0.85"）來傳送感測器數據。雖然這種方式簡潔省頻寬，但在實際開發中會快速陷入維護地獄：

  • 缺乏語義：接收端無法知道 25.5 是溫度、距離還是電壓，必須仰賴雙方約定的「欄位順序」。
  • 無法擴充：當你需要新增「電池電壓」欄位時，所有解析程式碼都必須同步修改。
  • 錯誤難排：少了一個逗號或多了一個空格，整串數據就會錯位解析。
重要 JSON 格式的自我描述性（每個值都有對應的鍵名）是它最大的優勢。單看一段 JSON，就能立即理解每個數值的物理意義——這在多人協作與長期維護中至關重要。

JSON 基礎語法

JSON 的核心結構是鍵值對 (Key-Value Pair)，支援字串、數字、布林值、陣列和巢狀物件等資料型別。以下是一個典型的感測器數據封包：

{ "device_id": "ESP32-CAR-001", "timestamp": 1716600000, "sensors": { "temperature": 25.5, "humidity": 60, "distance_cm": 34.2 }, "battery": { "voltage": 3.85, "percent": 80 }, "status": "running" }

在 ESP32 上使用 ArduinoJson

在嵌入式環境中，我們使用 ArduinoJson 程式庫來處理 JSON 的序列化（物件 → 字串）與反序列化（字串 → 物件）。以下範例示範如何將感測器讀數封裝為 JSON 字串：

#include <ArduinoJson.h> String buildTelemetryPayload(float temp, int humidity) { JsonDocument doc; doc["device_id"] = "ESP32-CAR-001"; doc["timestamp"] = millis(); JsonObject sensors = doc["sensors"].to<JsonObject>(); sensors["temperature"] = temp; sensors["humidity"] = humidity; String output; serializeJson(doc, output); return output; }
提示 ArduinoJson v7 使用 JsonDocument 取代了舊版的 StaticJsonDocument，能自動管理記憶體配置。在記憶體受限的 ESP32 上，建議使用 serializeJson() 直接輸出至 WiFiClient，避免建立中間字串佔用額外 RAM。

格式比較：CSV vs JSON

特性 CSV（逗號分隔） JSON（鍵值對）
可讀性 低（需對照欄位順序表） 高（鍵名即說明文件）
擴充性 低（新增欄位需修改所有解析器） 高（新增鍵值不影響既有解析）
封包大小 小（約 20–30 bytes） 較大（約 100–200 bytes）
巢狀結構 不支援 原生支援
前端整合 需自行解析 JSON.parse() 一行搞定

RESTful 動詞與語意化操作

REST (Representational State Transfer) 是一種軟體架構風格，它將系統中的每個實體視為「資源 (Resource)」，並使用標準化的 HTTP 方法（動詞）來對這些資源執行操作。在物聯網場景中，你的 ESP32 就是一個提供多種資源（感測器數據、控制指令、設備狀態）的微型伺服器。

HTTP 方法與 IoT 操作對應

RESTful API 的核心在於語義化：看到 URL 和 HTTP 方法就能理解這個操作的意圖，無需額外文件說明。

HTTP 方法 語義 IoT 範例 是否有副作用
GET 讀取資源 GET /api/sensors — 取得所有感測器數據 否（安全、冪等）
POST 建立資源 / 執行動作 POST /api/control — 下達移動指令 是
PUT 完整更新資源 PUT /api/config — 更新 PID 參數組 是（冪等）
DELETE 刪除資源 DELETE /api/tasks/3 — 取消排程任務 是（冪等）
備註 冪等 (Idempotent) 是指重複執行同一操作，結果不會改變。GET 和 PUT 是冪等的（讀 10 次、更新 10 次，結果與 1 次相同），而 POST 通常不是（每次 POST 可能產生新資源）。理解冪等性有助於設計可靠的重試機制。

路徑命名規範

良好的 REST API 路徑應遵循「名詞代表資源，動詞由 HTTP 方法承擔」的原則：

設計 路徑範例 評價
正確 GET /api/car/status 資源名詞 + HTTP 動詞，語義清晰
正確 POST /api/car/control 動作由 POST 承擔，路徑為資源名
錯誤 GET /api/getCarStatusNow 動詞放入路徑，違反 REST 原則
錯誤 POST /api/doMoveForward 動詞洩漏，應改用 POST Body 攜帶指令

ESP32 WebServer 實作範例

以下範例示範如何在 ESP32 上使用 WebServer 程式庫建立 RESTful 端點：

#include <WebServer.h> WebServer server(80); void handleGetStatus() { String json = buildTelemetryPayload(); server.send(200, "application/json", json); } void handlePostControl() { String body = server.arg("plain"); JsonDocument doc; if (deserializeJson(doc, body)) { server.send(400, "application/json", "{\"error\":\"Invalid JSON\"}"); return; } server.send(200, "application/json", "{\"status\":\"ok\"}"); } void setup() { server.on("/api/car/status", HTTP_GET, handleGetStatus); server.on("/api/car/control", HTTP_POST, handlePostControl); server.begin(); }
提示 注意每個回應都明確設定了 Content-Type: application/json。這不只是「好習慣」，而是 HTTP 標準要求——它告訴客戶端如何正確解析回應 Body。遺漏此 Header 可能導致瀏覽器的 fetch() 或 XMLHttpRequest 解析失敗。

狀態碼與回饋機制

HTTP 狀態碼是伺服器對客戶端請求的「回覆簡訊」：三位數字就能精確傳達請求的處理結果。在物聯網場景中，正確使用狀態碼不僅能讓前端準確判斷設備狀態，更能在異常發生時提供清晰的診斷線索。一個好的 API 不會在出錯時「沉默不語」。

IoT 常用 HTTP 狀態碼

狀態碼 名稱 語義 ESP32 應用場景
200 OK 請求成功，回傳結果 成功讀取感測器數據並回傳 JSON
201 Created 資源建立成功 成功註冊新的排程任務
400 Bad Request 客戶端送出無效請求 JSON 格式錯誤、缺少必要欄位、參數超出範圍
404 Not Found 資源不存在 請求了不存在的 API 路徑
500 Internal Server Error 伺服器內部異常 感測器讀取失敗、記憶體不足、I2C 匯流排鎖死
警告 最常見的反模式是「萬用 200」——無論成功或失敗都回傳 200 OK，把錯誤訊息塞在 Body 裡。這會讓前端的錯誤處理邏輯變得混亂。請務必使用正確的狀態碼，讓 fetch() 的 .catch() 或 response.ok 判斷能正常運作。

錯誤回應格式標準化

除了正確的狀態碼，回應 Body 中也應包含結構化的錯誤資訊，幫助開發者快速定位問題。以下是建議的標準錯誤回應格式：

// 400 Bad Request 回應範例 { "error": "INVALID_PAYLOAD", "message": "Missing required field: cmd", "timestamp": 1716600123 } // 500 Internal Server Error 回應範例 { "error": "SENSOR_FAILURE", "message": "I2C bus timeout on MPU6050 (0x68)", "timestamp": 1716600456 }

ESP32 完整錯誤處理實作

以下範例示範如何在 ESP32 WebServer 中實作完整的請求驗證與錯誤回饋機制：

void handlePostControl() { String body = server.arg("plain"); JsonDocument doc; // 階段 1：JSON 格式驗證 if (deserializeJson(doc, body)) { server.send(400, "application/json", "{\"error\":\"PARSE_ERROR\"}"); return; } // 階段 2：必要欄位檢查 if (!doc["cmd"].is<const char*>()) { server.send(400, "application/json", "{\"error\":\"MISSING_FIELD\"}"); return; } // 階段 3：執行控制邏輯 bool ok = executeCommand(doc["cmd"]); if (!ok) { server.send(500, "application/json", "{\"error\":\"EXEC_FAILED\"}"); return; } server.send(200, "application/json", "{\"status\":\"ok\"}"); }
提示 在生產環境中，建議將錯誤碼設計為可被程式判讀的常數字串（如 PARSE_ERROR），而 message 欄位則提供人類可讀的描述。這樣前端可以根據 error code 自動處理已知錯誤，同時也方便開發者在 console 中快速定位問題。

前端 JavaScript 搭配範例

以下展示前端如何配合 ESP32 的 RESTful API 進行數據請求與錯誤處理：

async function sendCommand(cmd) { try { const res = await fetch("/api/car/control", { method: "POST", headers: { "Content-Type": "application/json" }, body: JSON.stringify({ cmd }) }); const data = await res.json(); if (!res.ok) throw new Error(data.error); console.log("Success:", data); } catch (err) { console.error("API Error:", err.message); } }

adv-04-unit-json-rest (JSON 數據打包與 RESTful 回傳)

本單元將帶領你將底層硬體（ESP32）接入現代網際網路生態。你將學習如何使用標準化的 JSON 格式封裝數據，並透過 RESTful API 與 Web 客端進行通訊。

學習目標

  1. 掌握 JSON 序列化與反序列化 (ArduinoJson)。
  2. 實作符合 RESTful 規範的 HTTP 端點 (Endpoints)。
  3. 理解 HTTP 狀態碼 (200, 400, 500) 的語義。
  4. 評估 API 效能與 Payload 大小對傳輸效率的影響。

任務說明

任務 1：標準化封包產生器 (GET /api/telemetry)

實作一個 GET 請求處理器，回傳設備的即時數據。

  • URL: /api/telemetry
  • 回傳格式: JSON
  • 必要欄位:
    • device_id: 設備唯一識別碼。
    • sensors: 包含各項感測器數值的物件。
    • timestamp: 數據採集的時間戳。
  • 評分標準: Header 必須包含 Content-Type: application/json。

任務 2：指令接收器 (POST Command Receiver)

實作一個 POST 請求處理器，接收外部指令並控制硬體。

  • URL: /api/control
  • 處理邏輯:
    • 解析傳入的 JSON Payload。
    • 驗證是否包含 cmd 欄位。
    • 如果格式錯誤，應回傳 400 Bad Request 並附帶錯誤訊息。
    • 如果成功執行，回傳 200 OK。

任務 3：API 效能與壓力測試

分析通訊協議的開銷。

  • 量化測試: 比較「詳細版 Key」與「精簡版 Key」的數據包大小。
  • 報告撰寫: 在 docs/api_performance_report.md 中記錄不同負載下的延遲數據 (RTT)。
  • 反思: 在什麼情況下應該優先考慮開發者可讀性？什麼時候必須追求極致的傳輸效率？

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template Repository) 配置建議

RESTful API 的開發涉及「請求 (Request)」與「回應 (Response)」的精準對接，建議範本包含：

  • 📂 src/rest_server.ino：提供 ESP32 Web Server 的基礎框架，預置 WiFi 連線代碼，留下 on("/api/telemetry", ...) 與 on("/api/control", ...) 的路由 TODO。
  • 📂 web_client/api_test.js：一個基於 fetch 的 JS 測試工具。學員只需在本地瀏覽器打開，即可向其 ESP32 發送各種 REST 指令，觀察狀態碼與結果。
  • 📂 docs/status_map.md：一份 HTTP 狀態碼對照表。要求學員定義在其系統中，什麼情況下回傳 200, 400 或 500。
  • 📂 tests/api_validator.test.js：自動化評分腳本。模擬不同的 JSON Payload，檢核 API 是否能正確解析並回傳符合格式的 Response。

2. 作業任務部署細節

任務 1：標準化封包產生器 (Standardized Payload Generator)

  • 目標：掌握 REST 的「資源導向 (Resource-Oriented)」建模與 JSON 序列化。
  • Classroom 部署建議：
    • 數據結構檢核：要求學員實作一個包含 device_id (串列號), sensors (物件), timestamp (時間戳) 的層次化 JSON。
    • Autograding：透過腳本請求 GET /api/telemetry，驗證 Header 是否包含 Content-Type: application/json，且 Body 的 JSON 欄位是否齊全。

任務 2：指令接收器 (POST Command Receiver)

  • 目標：處理具有「副作用 (Side Effects)」的 API 指令，並強化異常報錯機制。
  • Classroom 部署建議：
    • 邏輯魯棒性檢核： 
      // 學生需實作：POST 請求解析與錯誤回饋
if (!doc.containsKey("cmd")) {
    server.send(400, "application/json", "{\"error\":\"Missing cmd field\"}");
    return;
}
    • 除錯任務：導師提供幾組錯誤的指令格式（如：參數超限、格式損毀），學員需實作對應的處理邏輯，確保硬體不會因為髒數據而重啟。

任務 3：API 效能與壓力測試 (Payload Benchmark)

  • 目標：直觀感受「序列化開銷」與「通訊延遲」的關聯。
  • Classroom 部署建議：
    • 量化對比：學員需在 BENCHMARK.md 中提交數據。
      • 測試「冗長版 Key」(如 power_consumption_mw) vs 「精簡版 Key」(如 p_mw) 在每秒 10 次連續請求下的平均 RTT (Round Trip Time)。
    • 架構思考度：要求學員在 PR (Pull Request) 中回答：在什麼情境下應該優先考慮開發者可讀性（冗長 Key），什麼情境下必須考慮傳輸效率（精簡 Key）？

3. REST API 導師點評標準 (RESTful Benchmarks)

此單元的價值在於 「對 Web 標準通訊禮儀的內化」：

  • [ ] 語義化路徑設計：路由命名是否使用了「名詞代表資源」？（例如 /api/car/status 而非 /api/getCarStatusNow）。
  • [ ] 狀態碼準確性：當參數無效時，是否正確使用了 400 而不是隱瞞錯誤的 200？或者是內部崩潰時的 500？
  • [ ] 封包原子性：返回的 JSON 是否具有「自我描述性」？（即：單看這一串文字，就知道它是什麼設備、在什麼狀態）。
