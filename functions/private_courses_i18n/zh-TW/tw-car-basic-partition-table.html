簡介

ESP32 的 Flash 記憶體容量有限（通常 4MB），而你的韌體、OTA 備份區、檔案系統與設定資料全都要擠在裡面。 分區表就是這塊 Flash 的「藍圖」，決定了每個區域的起始位址與大小。配置不當，你的程式會因為空間不足而無法燒錄，或 OTA 更新因為沒有備份分區而失敗。 掌握分區表的規劃邏輯，才能在有限資源中做出最佳分配。

為什麼需要分區表？

ESP32 的 Flash 記憶體就像一塊有限的「數位土地」——通常只有 4MB 到 16MB。你的應用程式、OTA 備份、Wi-Fi 設定、檔案系統資源全都要存放在這塊土地上。分區表（Partition Table）就是這塊土地的「地籍圖」，它精確定義了每個分區的起始位址（Offset）、大小（Size）與用途（Type/SubType）。

如果沒有分區表，ESP32 不知道程式碼從哪裡開始、設定資料存在哪裡、哪個區域可以安全地被覆寫。分區表是 Bootloader 啟動時讀取的第一份「指令文件」，它決定了整個系統的記憶體佈局。

Flash 記憶體的基本結構

ESP32 的 Flash 從位址 0x0000 開始，前 0x8000（32KB）被保留給 Bootloader 使用。分區表本身佔據 0x8000 到 0x9000 的 4KB 空間。從 0x9000 開始，才是你可以自由規劃的區域。

分區類型 SubType 用途說明 典型大小
app factory / ota_0 / ota_1 存放可執行的韌體二進位檔 1～2 MB
data nvs Non-Volatile Storage，鍵值對存儲 16～24 KB
data ota OTA 引導資訊（otadata） 8 KB
data spiffs / littlefs 檔案系統，存放網頁、圖片等靜態資源 512 KB～2 MB
data coredump 程式崩潰時的除錯記憶體傾印 64 KB

4KB 對齊規則

ESP32 的 Flash 硬體以 4KB（0x1000）為最小擦除單位。這意味著所有分區的起始位址和大小都必須是 4KB 的整數倍。若你在 partitions.csv 中設定了非對齊的數值，編譯時會產生錯誤。

注意 常見的對齊換算：16KB = 0x4000、64KB = 0x10000、1MB = 0x100000、2MB = 0x200000。在規劃分區表時，建議使用十六進位表示法以避免計算錯誤。

沒有分區表會發生什麼事？

嚴格來說，ESP32 無法在沒有分區表的情況下運作——Bootloader 會因為找不到有效的分區資訊而拒絕啟動。但更常見的問題是使用了不適當的預設分區表：

  • 韌體超過預設 App 分區大小 → 編譯錯誤 Region 'iram0_0_seg' overflowed
  • 沒有 OTA 分區 → 無法進行無線更新，每次都要接線燒錄
  • 檔案系統空間不足 → 網頁資源或資料檔案無法存入裝置
  • 分區地址重疊 → 資料互相覆蓋，導致不可預期的行為
提示 當你遇到「sketch too big」或「region overflowed」錯誤時，第一步就是檢查分區表。通常只需要將 App 分區從 1.2MB 擴大到 2MB 就能解決問題。

預設 vs. 自定義分區

ESP32 開發框架（Arduino Core 或 ESP-IDF）提供了數種預設分區表供你選用。但隨著專案功能增長——加入 BLE、Wi-Fi Web Server、OTA、TLS 等大型函式庫——預設的 1.2MB App 空間很快就會不夠用，導致令人頭痛的 Region overflowed 編譯錯誤。

這時，你需要學會建立自定義的 partitions.csv 檔案，依照專案需求重新分配 Flash 空間。這就像城市規劃師重新劃分土地用途——你可以縮減不需要的區域，將空間讓給需要擴張的功能。

預設分區表的限制

Arduino ESP32 最常使用的預設分區表為 default.csv，其配置如下：

# Name,   Type, SubType, Offset,   Size,    Flags
nvs,      data, nvs,     0x9000,   0x5000,
otadata,  data, ota,     0xe000,   0x2000,
app0,     app,  ota_0,   0x10000,  0x140000,
app1,     app,  ota_1,   0x150000, 0x140000,
spiffs,   data, spiffs,  0x290000, 0x170000,

在這個配置中，每個 App 分區只有 0x140000（1.25MB）。一旦你的韌體編譯輸出超過 1.25MB，就會立即觸發溢位錯誤。常見的觸發場景包括：

  • 同時啟用 Wi-Fi + BLE（各自佔約 300～500KB）
  • 引入 TLS/SSL 加密（增加約 200KB）
  • 使用 ArduinoJSON、AsyncWebServer 等大型函式庫
  • 啟用 OTA 除錯資訊或 Core Dump 功能

自定義 partitions.csv

建立自定義分區表的步驟很簡單：在專案根目錄建立一個 partitions.csv 文件，並在 platformio.ini 中指向它。CSV 格式的每一行代表一個分區：

# Name,   Type, SubType, Offset,   Size,     Flags
nvs,      data, nvs,     0x9000,   0x4000,
otadata,  data, ota,     0xd000,   0x2000,
app0,     app,  ota_0,   0x10000,  0x200000,
app1,     app,  ota_1,   0x210000, 0x200000,
storage,  data, spiffs,  0x410000, 0x1F0000,

上方的配置將每個 App 分區擴大至 2MB（0x200000），適用於 4MB Flash 的開發板。注意每個分區的 Offset 等於前一個分區的 Offset + Size，確保不會重疊。

警告 所有分區的總大小不得超過 Flash 晶片的實際容量。若你的開發板使用 4MB Flash（0x400000），所有分區加上 Bootloader 區域的總和必須 ≤ 4MB。超出範圍會導致燒錄失敗。

在 PlatformIO 中啟用自定義分區表

1

建立分區表檔案

在專案根目錄建立 partitions.csv，依照上方格式填入各分區定義。

2

修改 platformio.ini

加入 board_build.partitions = partitions.csv 設定項，讓 PlatformIO 使用你的自定義分區表而非預設值。

3

重新編譯並燒錄

執行 pio run --target upload。PlatformIO 會自動將新的分區表連同韌體一起寫入 Flash。首次變更分區表時建議先執行 pio run --target erase 清除整個 Flash。

提示 若你不需要 OTA 功能，可以將雙 App 分區改為單一 factory 分區，最大可達 3.5MB 以上。但這表示未來每次更新都需要接線燒錄。在專案初期就規劃好 OTA 支援，能為日後的維護省下大量時間。

持久化存儲的決擇

ESP32 提供了兩種截然不同的持久化存儲方案：NVS（Non-Volatile Storage）與檔案系統（LittleFS / SPIFFS）。它們各有專長的使用場景——選錯了方案不會導致程式崩潰，但會讓你的程式碼效率低落、維護困難。理解兩者的差異，是設計穩健 IoT 系統的基礎能力。

NVS：鍵值對存儲

NVS 採用 Key-Value（鍵值對）結構，專為小型、結構化的設定資料而設計。每筆資料以一個字串鍵（最長 15 字元）對應一個值（支援整數、字串、Blob 等型別）。NVS 的底層使用磨損平衡（Wear Leveling）演算法，能有效延長 Flash 的壽命。

在 Arduino 框架中，你可以透過 Preferences 函式庫輕鬆操作 NVS：

#include <Preferences.h>
Preferences prefs;

void setup() {
  Serial.begin(115200);
  prefs.begin("my-app", false); // namespace, readOnly

  // 讀取開機次數（若不存在則預設為 0）
  unsigned int bootCount = prefs.getUInt("boot_count", 0);
  bootCount++;
  prefs.putUInt("boot_count", bootCount);

  Serial.printf("第 %u 次開機\n", bootCount);
}

NVS 最適合的使用場景包括：

  • Wi-Fi SSID 與密碼（裝置重啟後不需重新設定）
  • 裝置 ID、校正係數、PID 控制參數
  • 開機次數、最後運行狀態、錯誤計數器
  • 使用者偏好設定（亮度、音量、模式）

LittleFS / SPIFFS：檔案系統

當你需要存放大型、結構化的檔案時（如 HTML 頁面、CSS 樣式表、JavaScript 腳本、JSON 設定檔、圖片等），應使用檔案系統。LittleFS 是 SPIFFS 的現代替代方案，支援目錄結構、具備更好的斷電保護，且在大量小檔案的場景下效能更優。

#include <LittleFS.h>

void setup() {
  if (!LittleFS.begin(true)) { // true = 若掛載失敗則格式化
    Serial.println("LittleFS 掛載失敗");
    return;
  }

  // 讀取 Web 頁面並回傳給客戶端
  File file = LittleFS.open("/index.html", "r");
  if (file) {
    String content = file.readString();
    file.close();
  }
}

NVS vs. LittleFS 完整比較

特性 NVS (Preferences) LittleFS
資料結構 Key-Value 鍵值對 檔案與目錄樹
適合資料類型 小型參數（整數、短字串） 大型檔案（HTML、圖片、Log）
單筆資料大小 最大約 4KB 僅受分區大小限制
磨損平衡 內建 內建
目錄支援 否（使用 namespace 分類） 是
上傳工具 程式碼中直接寫入 PlatformIO Upload Filesystem Image
典型分區大小 16～24 KB 512 KB～2 MB
注意 SPIFFS 已被官方標記為「不建議用於新專案」。LittleFS 在所有方面（效能、可靠性、功能）都優於 SPIFFS。若你正在開始新專案，請直接使用 LittleFS。在 PlatformIO 中，只需在 platformio.ini 加入 board_build.filesystem = littlefs 即可切換。

如何選擇？

簡單的判斷原則：如果資料可以用一個變數表示，用 NVS；如果資料是一個完整的檔案，用 LittleFS。 在實務中，大多數 IoT 專案會同時使用兩者——NVS 存放運行參數與狀態，LittleFS 存放網頁資源與大型設定檔。

提示 在 PlatformIO 中上傳檔案至 LittleFS 非常簡單：將檔案放入專案根目錄下的 data/ 資料夾，然後在終端機執行 pio run --target uploadfs。所有 data/ 中的檔案都會被打包並寫入 Flash 的 LittleFS 分區。

基礎 02：ESP32 儲存架構與分區規劃 (Partition Table & Persistence)

歡迎來到本單元！在本實驗中，你將學習如何從「黑盒開發」轉向「系統架構規劃」，學會精細管理 ESP32 的 Flash 資源。你將挑戰重新規劃分區表、實作 NVS 數據持久化，以及部署 LittleFS 檔案系統。

🎯 學習目標

  1. 分區規劃：學會使用 partitions.csv 重新分配 Flash 空間。
  2. 數據持久化：使用 NVS (Non-volatile Storage) 儲存裝置狀態。
  3. 檔案系統：實作 LittleFS 掛載並從 Flash 讀取網頁資源。

🚀 作業任務

任務 1：「擴容手術」自定義分區規劃

你的任務是修改 partitions.csv，打破預設限制，為 App 與 檔案系統騰出空間：

  • 目標 1：將 factory 分區（App 空間）擴大至 2MB。
  • 目標 2：將 storage 分區（LittleFS）規劃為 1.5MB。
  • 注意：必須確保所有分區地址符合 4KB (0x1000) 對齊，且各分區不可重疊。
  • 配置：在 platformio.ini 中手動添加 board_build.partitions = partitions.csv。

任務 2：「數據黑盒子」NVS 參數持久化

實作一個能「記住開機次數」的功能：

  • 使用 Preferences 函式庫。
  • 每次重新開機時，讀取並增加計數值，然後存回 NVS。
  • 驗證：拔掉電源 10 秒後重新通電，確認計數值是否能正確累加。

任務 3：「資源檔案庫」LittleFS 檔案部署

將 data/ 目錄下的 index.html 部署到 ESP32：

  1. 執行 PlatformIO 的 Upload Filesystem Image 任務。
  2. 在程式碼中掛載 LittleFS。
  3. 實作 Web Server 路由，當訪問 / 時，讀取並回傳 index.html 的內容。

📝 提交要求

  1. 程式碼：完成 src/main.cpp 中的邏輯。
  2. 分區表：提交修正後的 partitions.csv。
  3. 報告：在 docs/MemoryMap.md 中填寫你最終規劃的各分區 Offset 與 Size。
  4. 驗證：在 README.md (或提交訊息) 中說明你如何驗證 NVS 的持久化功能。

  • 📂 src/main.cpp：存儲實作區。預設空的 NVS 與 LittleFS 掛載框架，學員需寫出能「記住開機次數」與「從 Flash 讀取網頁」的代碼。
  • 📂 docs/MEMORY_MAP.md：存儲地圖。學員需在此填寫最終規劃的各分區 Offset (起始地址) 與 Size，建立對 4KB 對齊的物理直覺。

2. 作業任務部署細節

任務 1：「擴容手術」自定義分區規劃 (Flash Expansion Surgery Lab)

  • 目標：打破 1MB 的預設限制，為未來複雜的 AI 或 Web 專案騰出空間。
  • Classroom 部署建議：
    • 規劃挑戰：學員需在 CSV 中將 factory 分區擴大至 2MB。
    • 地址計算：正確計算下一個分區（如 nvs 或 storage）的 Offset，防止空間重疊 (Overlap)盤。
    • 驗證要點：提交燒錄時的終端機截圖。證明 Firmware 已成功依照自定義分區表進行部署。

任務 2：「數據黑盒子」NVS 參數持久化 (NVS Persistence Lab)

  • 目標：掌握 Key-Value 存儲，讓裝置在斷電後仍能像擁有「長期記憶」般記住設定。
  • Classroom 部署建議：
    • 持久化實作：使用 Preferences 函式庫，實作一個能自動記錄「馬達最後位置」或「開機累計次數」的功能。
    • 邏輯強健性：學員需處理「初次使用」時 NVS 為空的預設值返回邏輯。
    • 驗證要點：在 README 中說明：拔掉電源 10 秒後重新通電，數據是否如預期被正確還原？

任務 3：「資源檔案庫」LittleFS 檔案部署 (LittleFS Asset Deployment Lab)

  • 目標：實作「動靜分離」。將 UI 與程式碼隔離，建立專業的嵌入式 Web 開發結構。
  • Classroom 部署建議：
    • 檔案上傳實戰：學員需執行 PlatformIO 的 Upload Filesystem Image 任務，將 data/ 目錄寫入 Flash。
    • 路徑映射：在代碼中掛載 LittleFS，並寫出一個簡單的 Web 路由，讓瀏覽器請求 /index 時，能直接從 Flash 返回檔案內容。
    • 提交結果：展示瀏覽器成功渲染出存儲於 LittleFS 中 HTML 頁面的畫面。
