🏗️ 1. 三層神經架構：從螢幕到馬達

理解系統全貌是架構師的核心競爭力。當你在螢幕上滑動搖桿，這個看似簡單的動作背後，數據需要穿越三個截然不同的技術領域：瀏覽器中的 JavaScript 運算環境、藍牙射頻的無線傳輸通道、以及 ESP32 微控制器上的即時韌體。每一層都有各自的資料格式、時間尺度和失敗模式。將系統拆解為清晰的層級架構，是定位瓶頸與設計容錯機制的基礎。

感知控制層 (Dashboard Layer)

感知控制層運行在使用者的瀏覽器中，負責三項核心任務：UI 事件捕捉（搖桿位移、按鈕觸發）、遙測資料視覺化（即時速度、電池電壓、感測器數值的圖形呈現）、以及控制指令序列化（將浮點數座標轉換為 BLE 可傳輸的封包格式）。

此層的關鍵挑戰在於「人類感知的即時性要求」。研究顯示，操控延遲超過 100ms 時使用者會明顯感受到「黏滯感」，超過 200ms 則幾乎無法精確操控。因此感知層的處理時間預算極為有限，通常必須在 5-10ms 內完成座標運算與封包組裝。

備註 搖桿事件的觸發頻率取決於瀏覽器的 pointermove 取樣率，通常在 60-120Hz 之間。但 BLE 的傳輸頻寬有限，因此必須在此層實施節流 (Throttle)，將發送頻率限制在 20-50Hz。

通訊傳輸層 (Transport Layer)

通訊傳輸層由 Web Bluetooth GATT 協議承擔，是數據從軟體世界跨越到硬體世界的「空中橋樑」。BLE 4.2 的單次封包最大承載為 20 bytes（BLE 5.0 可擴展至 244 bytes），這意味著每一個 byte 都極為珍貴。此層的設計目標是在有限頻寬內實現最低延遲與最高可靠性。

傳輸參數典型值影響
Connection Interval7.5ms - 4s決定封包往返的最小時間單位
MTU (最大傳輸單元)23 bytes (預設)單次可攜帶的最大數據量
Slave Latency0 (即時模式)周邊裝置可跳過的連線事件數
Supervision Timeout4000ms超過此時間無回應視為斷線

硬體執行層 (Firmware Layer)

硬體執行層運行在 ESP32 的 FreeRTOS 即時作業系統上，負責將接收到的封包解析為具體的物理動作。此層的工作包括：JSON/二進制協議解析、差分驅動運動學計算（將線速度 v 與角速度 w 轉換為左右輪 PWM）、以及安全限幅（確保 PWM 值不超出硬體安全範圍）。

// ESP32 韌體端：接收 BLE 封包並轉換為馬達 PWM
void onBleReceive(const uint8_t* data, size_t len) {
  int8_t v = (int8_t)data[0];  // 線速度 [-128, 127]
  int8_t w = (int8_t)data[1];  // 角速度 [-128, 127]
  int pwmL = constrain(v + w, -255, 255);
  int pwmR = constrain(v - w, -255, 255);
  setMotor(MOTOR_L, pwmL);
  setMotor(MOTOR_R, pwmR);
}
警告 韌體層的解析迴圈必須在 1ms 以內完成。如果使用 ArduinoJson 解析複雜 JSON 結構，在 ESP32 240MHz 主頻下可能需要 200-500μs。當封包頻率超過 50Hz 時，累積的解析延遲可能導致 Serial 緩衝區溢位。考慮使用二進制協議取代 JSON 以提升解析效率。

🔄 2. 數據變身術：序列化與動態映射

同一個「向前走」的控制意圖，在系統的不同層級有著截然不同的數位表達方式：在網頁端是 JavaScript 的浮點數 0.78，經過 BLE 傳輸時變成 JSON 字串 {"v":12}，到了 ESP32 韌體端則轉化為 uint8_t 類型的 PWM 值 200。這個「數據變身」的過程稱為序列化 (Serialization) 與動態映射 (Dynamic Mapping)——設計不當會導致精度損失、溢位錯誤，甚至讓馬達做出完全相反的動作。

從浮點到位元組：類型轉換管道

搖桿控制器輸出的座標通常是 [-1.0, 1.0] 範圍的正規化浮點數，但 BLE 封包的每一個 byte 都極為珍貴。傳輸前必須將浮點數映射為整數以壓縮體積。常見的映射策略如下：

映射方案網頁端輸入封包內容韌體端還原精度損失
JSON 字串 0.78 {"v":0.78} (10 bytes) parseFloat() 零損失，但體積大
Int8 映射 0.78 0x63 (1 byte) val / 127.0 ~0.8%（可接受）
UInt8 映射 0.78 0xC7 (1 byte) val / 255.0 ~0.4%

JSON vs. 二進制協議的取捨

JSON 格式具有極佳的可讀性與除錯便利性——你可以直接在 Serial Monitor 中看到 {"v":12,"w":-4}，一目了然。但 JSON 的代價是體積膨脹：一個簡單的二維座標用 JSON 表示需要 15-20 bytes，而用二進制緊湊封包只需 2 bytes。在 BLE 4.2 的 20-byte MTU 限制下，這個差距意味著 JSON 協議每次只能傳送一筆數據，而二進制協議可以打包多達 10 筆。

// 網頁端：將正規化座標打包為二進制封包
function packCommand(x, y) {
  const buf = new ArrayBuffer(3);
  const view = new DataView(buf);
  view.setUint8(0, 0xA5);         // 同步標記 (Sync Byte)
  view.setInt8(1, Math.round(x * 127));// X: [-127, 127]
  view.setInt8(2, Math.round(y * 127));// Y: [-127, 127]
  return buf;
}
備註 同步標記 (Sync Byte) 是封包起始的魔術數字（如 0xA5），用於在數據流中定位有效封包的開頭。當通訊因干擾產生位元錯位時，解析器可搜尋下一個同步標記重新對齊。這是串流協議中不可或缺的容錯機制。

韌體端的安全解析

韌體端接收到原始位元組後，必須執行嚴格的驗證：檢查同步標記是否正確、封包長度是否符合預期、數值是否在合法範圍內。任何一項驗證失敗都應丟棄該封包，而非使用可能損毀的數據驅動馬達。

// ESP32 韌體端：安全解析二進制封包
void parsePacket(const uint8_t* data, size_t len) {
  if (len < 3 || data[0] != 0xA5) return;
  int8_t x = (int8_t)data[1];
  int8_t y = (int8_t)data[2];
  float normX = x / 127.0f; // 還原至 [-1.0, 1.0]
  float normY = y / 127.0f;
  applyMotorControl(normX, normY);
}
警告 JavaScript 與 C/C++ 的位元組順序 (Endianness) 可能不同。JavaScript DataView 預設使用大端序 (Big-Endian)，而 ESP32（Xtensa 架構）使用小端序 (Little-Endian)。對於多位元組數值（如 int16_t），務必在 DataView 中明確指定 littleEndian: true，否則韌體端解析出的數值將完全錯誤。

📈 3. 性能指標：端到端延遲測量

「感覺有點卡」是工程師最危險的自我欺騙。端到端延遲 (End-to-End Latency) 必須用數據而非直覺來量化。從搖桿位移的瞬間到馬達實際轉動的瞬間，中間經歷了至少六個處理階段，每個階段都可能是瓶頸。學會在關鍵路徑上埋入時間戳，是從「碰運氣」進化到「精準除錯」的分水嶺。

全鏈路延遲拆解

一次完整的控制指令從觸發到執行，需要穿越以下六個階段：

階段位置典型耗時測量方法
1. UI 事件處理 瀏覽器 JS 1-3 ms performance.now()
2. 封包序列化 瀏覽器 JS < 1 ms performance.now() 差值
3. BLE 傳輸 空中介面 7.5-30 ms Connection Interval 決定
4. 封包解析 ESP32 韌體 0.05-0.5 ms micros() 差值
5. PID 運算 ESP32 韌體 0.01-0.1 ms micros() 差值
6. PWM 輸出 硬體外設 < 0.01 ms 示波器
備註 BLE 傳輸延遲（階段 3）通常佔據總延遲的 70-90%，是最大的瓶頸。此延遲主要取決於 Connection Interval 設定，最小值為 7.5ms。要降低此延遲，可在連線後請求更短的 Connection Interval——但代價是更高的功耗。

RTT 測量實作

最直接的延遲測量方式是往返時間 (Round-Trip Time, RTT)：網頁端在發送指令時記錄時間戳，ESP32 收到後立即回傳一個 Echo 封包，網頁端收到 Echo 後計算時間差。RTT 除以 2 即為單程延遲的近似值。

// 網頁端 RTT 測量
async function measureRTT() {
  const t0 = performance.now();
  const pingPacket = new Uint8Array([0xEE]); // Ping 標記
  await txChar.writeValueWithResponse(pingPacket);

  return new Promise(resolve => {
    rxChar.addEventListener('characteristicvaluechanged',
      function onEcho(e) {
        if (e.target.value.getUint8(0) === 0xEE) {
          rxChar.removeEventListener('characteristicvaluechanged', onEcho);
          resolve(performance.now() - t0);
        }
      });
  });
}

壓力測試與崩潰邊界

壓力測試的目的是找出系統的「崩潰邊界」——從哪個發送頻率開始，封包遺失率急劇上升？測試方法是逐步提高發送頻率（從 10Hz 到 200Hz），同時在 ESP32 端統計實際收到的封包數量。當丟包率超過 5% 時，即為該系統配置下的安全上限。

提示 在壓力測試期間，同時監測 ESP32 的可用堆積記憶體 (ESP.getFreeHeap())。如果記憶體持續下降，可能表示封包處理速度跟不上接收速度，未處理的封包正在佇列中堆積。這是記憶體洩漏的前兆，最終會導致系統崩潰。
警告 performance.now() 在某些瀏覽器中受到「時間精度降級 (Reduced Time Precision)」的安全限制，精度可能被截斷至 100μs 甚至 1ms。在 Chrome 中啟用 Cross-Origin-Opener-Policy 和 Cross-Origin-Embedder-Policy 標頭可恢復高精度時間戳。

數據流分析 (Data Flow Analysis)

「好的架構師，是能像觀測心電圖一樣，看穿數據流中每一毫秒跳動背後的工程邏輯。」

歡迎來到進階單元 15。在本單元中，你將不再只是開發功能，而是要像「系統診斷醫師」一樣，深入觀察數據在感知層、傳輸層到解析層之間的每一分傳遞、變形與損耗。你的目標是定義出系統的生命脈動，並在高壓與干擾環境下確保系統的絕對穩定。

🎯 學習目標

  1. 全鏈路觀測 (Full-Stack Observability)：實作高性能埋點工具，量化端到端的延遲 (RTT)。
  2. 性能極限定義：透過壓力測試找出系統的崩潰臨界點，並定義 SLA 規格。
  3. 系統韌性設計 (Resilience Design)：重構固件解析器，使其具備在髒數據環境下自動同步與復位的能力。

  • 目標：透過科學實驗定義系統的「官方規格」。

    • 逐步提升發送頻率（從 10Hz 至 100Hz），記錄丟包率與 Serial 緩衝區狀態。
    • 產出：在 docs/Performance_Map.md 中繪製性能曲線圖，並在 README.md 的結案報告中給出具體的硬體配置建議。

任務 3：髒數據注入與自動復位解析器

  • 目標：在電磁干擾或亂碼注入下，確保系統「永不當機」。
  • 要求：

    • 使用混沌測試工具，驗證系統在封包損毀後能否在 50ms 內恢復控制。

1. 範本程式庫 (Template Repo) 配置建議

系統分析需要高精度的埋點工具與回報框架，建議範本包含：

  • 📂 src/latency_tracer.js：發送端高性能埋點工具。預置時間戳計算法，自動追蹤每一筆指令發出與回傳的精確耗時。
  • 📂 src/naive_parser.cpp：重構手術對象。一個「天真」的固件解析器（遇到亂碼會卡住或延遲），學員需將其升級為具備「自癒能力」的高級解析器。
  • 📂 docs/LINK_AUDIT.md：全鏈路審計報告。要求學員記錄：感知層耗時 | 傳輸層耗時 | 解析層耗時 | 最終執行延遲。
  • 📂 tests/chaos_injections.js：混沌工程注入器。這是一個自動化腳本，會隨機發送「損毀的 JSON」或「超長無效字串」，用來檢測系統的崩潰臨界點。

2. 作業任務部署細節

任務 1：端到端數據流追蹤分析 (E2E Data Flow Trace Lab)

  • 目標：看穿數據在跨平台傳遞時隱形的耗時分景。
  • Classroom 部署建議：
    • 數據路徑量測：學員需在網頁點擊到馬達轉動的完整路徑上設定 3 個以上的觀測點，量化總延遲 (RTT)。
    • 映射精確度驗證：比對網頁端 Normalized 資料 (01.0) 轉換為 C 語言 Uint8 (0255) 過程中的數據損耗。
    • 驗證要點：分析報告是否能清楚定位「哪一層」是目前的性能瓶頸？

任務 2：通訊頻寬與延遲極限壓測 (Throughput & Latency Stress Lab)

  • 目標：透過實驗找出系統的「崩潰臨界點」，並定義其 SLA 規格。
  • Classroom 部署建議：
    • 頻率/丟包曲線：逐步將發送頻率從 10Hz 拉升至 100Hz，紀錄系統開始出現丟包或 Serial 緩衝區溢位的臨界點。
    • 官方規格定義：學員需在 README 中定義：「在此硬體配置下，系統的最佳控制頻率為 XX Hz，延遲保證在 XX ms 內」。
    • PR 提交內容：提交具備「工業背書強度」的性能曲線圖與規格建議書。

任務 3：髒數據注入與自動復位解析器 (Dirty Data & Robust Parser Lab)

  • 目標：在現實世界的電磁干擾環境中，確保系統「永不當機」。
  • Classroom 部署建議：
    • 韌性重構：優化固件端解析器，使其具備「同步標記搜索 (Sync Marker Search)」能力。即使前一個封包損毀，也能在下一毫秒立即找回有效的數據開頭。
    • 復位速度挑戰：利用混沌工具注入亂碼，量測系統「恢復控制」所需的平均毫秒數。目標應小於 50ms。
    • Push Final：提交具備「高度通訊韌性、不因亂碼崩潰」的最終加固版解析代碼與測試 log。

3. 全鏈路系統工程師導師點評標準 (Full-Stack Observability Standards)

此單元的價值在於 「對全系統脈動的科學觀察力與抗壓設計」：

  • [ ] 診斷嚴謹度：報告是否揭示了延遲產生的物理原因（如：BLE 跳頻、Serial 波特率限制）？
  • [ ] 解析器防護力：面對連續的非法字串注入，ESP32 是否依然能穩定運行且不丟失後續的有效控制？
  • [ ] 規格承諾一致性：測量出的數據是否能在 PR 審查現場被導師複現？
