什麼是歸一化 (Normalization)？

在遙控系統中，搖桿產生的原始數據通常是像素偏移量——例如使用者在手機螢幕上拖動搖桿控件時， 瀏覽器回報的是 (deltaX, deltaY)，單位是像素 (px)。問題在於：不同裝置的螢幕尺寸與解析度各不相同， 同樣「推到底」的動作，在 iPhone SE 上可能只有 60px，在 iPad Pro 上卻可能超過 150px。 如果直接把像素值送給馬達，同一套程式碼在不同裝置上的行為就會完全不一致。

歸一化 (Normalization) 就是解決此問題的數學手段：將原始物理量除以其最大可能值， 使輸出落在一個與裝置無關的標準區間內。以搖桿為例，我們將偏移量除以搖桿最大半徑， 得到一個 -1.0 到 +1.0 的無因次量 (dimensionless value)。 這個標準化比例才是後續所有運算——映射、死區、混控——的起點。

歸一化公式

對搖桿的每一個軸 (X 或 Y)，歸一化計算如下：

\[V_{norm} = \frac{\text{Offset}}{\text{MaxRadius}}\]

其中 Offset 是搖桿控件在該軸上的位移量（帶正負號），MaxRadius 是搖桿可拖動的最大半徑。 結果 \(V_{norm}\) 的範圍為 \([-1.0,\;1.0]\)：

  • +1.0：搖桿推到該軸的正向極限（例如 Y 軸全力前進）
  • 0.0：搖桿位於中心（靜止）
  • -1.0：搖桿推到該軸的負向極限（例如 Y 軸全力後退）
備註 歸一化後的數值不帶任何物理單位——它只是一個「比例」。無論搖桿 UI 的設計半徑是 50px 還是 200px， 歸一化後的 0.5 都代表「推到一半」，系統行為完全一致。這就是「與硬體解耦」的核心意義。

程式碼實作

以下是 JavaScript 中的典型歸一化函式，結合了 clamp 保護以防止溢出：

// 將搖桿原始偏移歸一化至 [-1.0, 1.0]
function normalize(offset, maxRadius) {
  const raw = offset / maxRadius;
  return Math.max(-1.0, Math.min(1.0, raw));
}

// 範例用法
const maxR = 120;  // 搖桿 UI 半徑 120px
const normX = normalize(78,  maxR); // → 0.65
const normY = normalize(-54, maxR); // → -0.45
提示 使用 Math.max 與 Math.min 的 clamp 技巧可以防止搖桿在邊界被拖出圓形區域時， 數值超過 ±1.0 的安全上限。這種「防禦性計算」在控制系統中至關重要。

跨裝置行為比較

下表說明歸一化如何消除裝置差異，讓同一套控制邏輯適用於所有螢幕：

裝置 搖桿半徑 (px) 推到底偏移 (px) 歸一化結果
iPhone SE 60 60 1.0
iPhone 15 Pro 90 90 1.0
iPad Pro 150 150 1.0
iPad Pro (半推) 150 75 0.5

無論螢幕大小如何變化，歸一化後的 1.0 永遠代表「全力」，0.5 永遠代表「半力」。 這就是後續所有映射計算的基石——沒有歸一化，就沒有跨裝置的一致性保證。

數值映射 (Mapping)

歸一化給了我們一個統一的 [-1.0, 1.0] 區間，但這只是中間產物。真正驅動硬體的是具有物理意義的數值： 馬達 PWM 需要 0–255 的整數、舵機需要 0–180 的角度、速度百分比需要 -100 到 +100。 映射 (Mapping) 就是將歸一化後的標準比例，線性或非線性地轉換到目標硬體所需的數值區間。

線性映射公式

最基本的映射是線性插值 (Linear Interpolation, lerp)。給定歸一化輸入 \(x \in [a_{in},\;b_{in}]\)， 要映射到目標區間 \([a_{out},\;b_{out}]\)，公式為：

\[\text{output} = a_{out} + \frac{(x - a_{in})}{(b_{in} - a_{in})} \times (b_{out} - a_{out})\]

在搖桿場景下，由於歸一化後輸入已經是 [0, 1] 或 [-1, 1]，公式可以簡化。 例如 0 ~ 1.0 映射到 0 ~ 255 只需 output = x × 255。

常見映射目標

控制類型 歸一化輸入 目標範圍 映射公式
馬達 PWM (單向) 0.0 ~ 1.0 0 ~ 255 output = x × 255
馬達 PWM (雙向) -1.0 ~ 1.0 -255 ~ 255 output = x × 255
舵機角度 -1.0 ~ 1.0 0° ~ 180° output = (x + 1) × 90
速度百分比 -1.0 ~ 1.0 -100% ~ 100% output = x × 100

程式碼實作

以下範例展示了通用的線性映射函式，以及在搖桿場景中的典型用法：

// 通用線性映射
function mapRange(x, inMin, inMax, outMin, outMax) {
  return outMin + (x - inMin) / (inMax - inMin) * (outMax - outMin);
}

// 搖桿 → 馬達 PWM
const pwm = mapRange(0.72, 0, 1, 0, 255);   // → 183.6

// 搖桿 → 舵機角度
const angle = mapRange(-0.5, -1, 1, 0, 180); // → 45°
備註 線性映射雖然簡單直觀，但它假設「搖桿推多少，出力就多少」。在實際操控中， 這種等比例關係往往讓低速微調變得困難（稍微推一下就衝太快）。 下一個單元將介紹「非線性映射」來解決這個問題。

非線性映射預覽

線性映射之外，我們可以在歸一化值上套用冪函數來改變響應曲線。 例如 output = pow(x, 2.0) 會讓低速區域的斜率變平緩（更容易微調）， 而高速區域的斜率變陡峭（動力爆發更集中）。這就是「指數映射」的核心概念，也是打造「職人級」操控手感的數學秘密。

提示 使用非線性映射時，務必確保端點閉合：當輸入為 1.0 時，輸出也必須精確到達 1.0。 否則全力推搖桿時，馬達永遠無法達到 100% 出力。

死區 (Deadzone)

理想中，搖桿放在中央時輸出應該是精確的 0.0。但現實並非如此——觸控螢幕的感測精度、 使用者手指微微的顫抖、甚至搖桿 UI 控件的浮點運算誤差，都可能讓靜止狀態下的歸一化值出現 0.02、-0.03 這類微小雜訊。如果這些雜訊直接被映射到馬達， 車輛在「靜止」時會不斷發出微弱的嗡嗡聲甚至緩慢漂移，這在專業系統中是不可接受的。

死區 (Deadzone) 就是為此設計的過濾機制： 在搖桿中心的一小段區域內（通常為 5%–15%），強制將輸出設為零。 只有當搖桿偏移超過這個閾值時，才開始輸出有效數值。

警告 單純地「把小於閾值的值設為 0」會造成一個跳躍：當搖桿離開死區邊界的瞬間， 輸出值會從 0.0 突然跳到 0.10。這個不連續的「斷層」 會讓馬達產生明顯的頓挫感。正確的做法是同時進行縮放補償。

死區實作的四個步驟

  1. 取得歸一化值：從搖桿獲取 [-1.0, 1.0] 區間的原始歸一化座標
  2. 閾值判定：計算絕對值，檢查是否小於設定的死區門檻（例如 10%，即 0.10）
  3. 歸零處理：若在死區內，強制輸出為 0.0
  4. 縮放補償：若超出死區，將有效範圍重新映射至 [0.0, 1.0]，確保起步平滑、端點閉合

程式碼實作

以下函式實作了含縮放補償的死區過濾：

// 帶縮放補償的死區過濾
function applyDeadzone(value, threshold) {
  const abs = Math.abs(value);
  if (abs < threshold) return 0.0;

  // 重新映射：將 [threshold, 1.0] 壓縮到 [0.0, 1.0]
  const scaled = (abs - threshold) / (1.0 - threshold);
  return Math.sign(value) * scaled;
}

// 使用範例
applyDeadzone(0.05, 0.10);  // → 0.0 (在死區內)
applyDeadzone(0.55, 0.10);  // → 0.5 (超出死區)
applyDeadzone(1.00, 0.10);  // → 1.0 (端點閉合)

有無死區的行為比較

歸一化輸入 無死區輸出 有死區輸出 (10%) 馬達行為
0.03 0.03 0.0 無死區：微弱抖動；有死區：完全靜止
0.10 0.10 0.0 死區邊界，剛好被過濾
0.55 0.55 0.50 縮放補償後平滑過渡
1.00 1.00 1.00 端點閉合，全力輸出不受影響
提示 死區閾值的大小需要根據實際硬體來調校。觸控搖桿通常 5%–10% 就足夠； 物理搖桿（如 RC 遙控器）因為機械結構老化，可能需要更大的死區。 驗證方式很簡單：放開搖桿後，觀察 Serial Monitor 輸出是否穩定停在 0.00。

任務 3：數據發送節流 (Throttling)

  • 目標：防止數據過度發送導致網路擁塞。
  • 實作重點：
    • 限制發送頻率（建議每 100ms 傳送一次）。
    • 限制數值精度至小數點後兩位。
    • 驗證：在瀏覽器控制台觀察，快速滑動搖桿時，封包發送頻率是否穩定？

    • 死區防護：核心關鍵點。實作 10% 閥值的死區。當搖桿在中心附近微動時，輸出必須被強制拉回「絕對 0.0」。
    • 驗證要點：放開搖桿後，介面顯示的數據是否會因為雜訊而出現 0.02 等細微抖動？（若會，則死區實作失敗）。

任務 2：非線性映射：指數型油門 (Exponential Mapping Lab)

  • 目標：學習如何利用數學函式模擬「高級感」，提升低速精準操控的能力，避免起步過於突兀。
  • Classroom 部署建議：
    • 曲線實作：實作公式 output = sgn(input) * pow(abs(input), exponent)。
    • 正負號保留：技術審核重點。學員必須正確處理負數區間，確保油門推後與推前都能運作。
    • 感官對比：在 README 中描述在「倒車入庫」等精細動作中，指數映射如何協助使用者比線性映射更容易成功。
    • 驗證要點：推到底時是否能精準達到 1.0 或 -1.0？低速區間的數據增長是否變得更加平緩？

任務 3：數據打包與發送節流 (Throttling & Packaging Lab)

  • 目標：建立網路效率與負擔的權衡意識，防止數據「濫發」造成網路擁塞 (Network Congestion)。
  • Classroom 部署建議：
    • 負擔輕量化：封裝成 JSON 物件，並將座標限制在小數點後兩位（如 0.85 而非 0.8499999）。
    • 節流機制 (Throttling)：實作時間計數器。限制發送頻率（如：每 100ms 傳送一次），即使使用者瘋狂轉動搖桿。
    • 靜默抑制：當數值與上一次相比「變化微小」時，停止發送以節省流量。
    • 驗證要點：在瀏覽器控制台觀察 Network Tab。快速滑動搖桿時，封包發送頻率是否維持穩定而非爆炸式增長？

3. 控制品質與操控演算法點評標準 (Control Quality & UX Standards)

此單元的價值在於 「對數值的極限掌控力與對使用者感覺的數學量化能力」：

  • [ ] 中心點絕對靜默：死區是否能讓馬達在靜止時遠離震顫？
  • [ ] 對應精準力：最大推力與最小推力是否能完美對準物理邊界？
  • [ ] 通訊專業度：是否具備節流意識，將網路資源花在「變動」的瞬間而非重複發送？
