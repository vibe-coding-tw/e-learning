📖 1. 英雄之旅：技術敘事的骨架

你花了數週打造的精密控制系統，實作了 PID 演算法、二進位通訊協議和斷線自癒機制—— 但如果你只用一句「我做了一台遙控車」來介紹，所有的技術深度都將被埋沒。 在科展、面試或開源社群中，技術的價值不在於它有多複雜，而在於它能被多少人理解。 這就是技術敘事 (Technical Narrative) 的力量。

好萊塢編劇大師 Joseph Campbell 提出的「英雄之旅 (Hero's Journey)」是人類最古老且最有效的敘事框架。 它的結構——平凡世界 → 遭遇挑戰 → 歷經掙扎 → 獲得突破 → 帶著成果回歸—— 同樣適用於技術專案的展示。你的專案就是一場英雄之旅，而你就是那位克服技術挑戰的英雄。

技術敘事的三幕結構

1

挑戰 (The Problem)

清楚定義你要解決的問題與痛點。不是「我想做一台遙控車」，而是「如何在 BLE 頻寬僅 20 bytes/packet 的限制下，實現低於 40ms 延遲的即時控制？」越具體、越有技術張力的問題描述，越能吸引專業聽眾的注意。

2

掙扎 (The Struggle)

誠實展示開發中遇到的障礙與失敗。例如：「初版使用 JSON 協議，導致控制延遲高達 200ms，車子轉彎時像在水中移動。」「第一次斷線測試時，車子直接撞牆。」這些掙扎讓你的故事有了人性與張力。

3

突破 (The Solution)

展示你的解決方案與成果數據。「透過自訂二進位協議，將封包從 42 bytes 壓縮到 2 bytes，延遲降低到 35ms，更新率提升 5 倍。」用具體的數字證明你的方案有效——這才是技術人的說服力。

弱敘事 vs. 強敘事的差異

面向弱敘事強敘事
專案簡介「我做了一台藍牙遙控車」「我打造了一套延遲 <40ms 的即時控制系統」
技術描述「我用了 PID」「PID 讓循跡誤差從 ±15cm 收斂到 ±2cm」
挑戰描述「遇到一些困難」「JSON 封包佔滿 BLE 頻寬，更新率僅 12Hz」
成果展示「車子可以動了」「斷線 1.2s 內自動煞停，0 次失控紀錄」
30 秒法則 無論是科展評審、面試官還是 GitHub 訪客，他們通常只會給你 30 秒的注意力。 你的 README 第一段、影片前 10 秒、口頭簡報的開場白，都必須在這段時間內傳達出最核心的技術亮點與成果。 如果 30 秒內無法吸引他們，後面的內容再精彩也無濟於事。
適應受眾 面對不同受眾時，敘事的重點應有所調整：對技術面試官著重演算法與架構決策； 對科展評審強調問題解決過程與實驗數據； 對非技術受眾則用類比和視覺化來取代專業術語。 同一個專案，可以有三種不同的敘事版本。

🎨 2. 邏輯視覺化：讓代碼「被看見」

程式碼是抽象的——它住在文字編輯器裡，看不見、摸不著。 但你用程式碼建構的邏輯、架構與數據流，卻可以透過視覺化手段被「看見」。 一張精心繪製的架構圖，往往比一千行程式碼更能讓人理解你的系統設計。 這就是為什麼頂尖的技術文件和開源專案的 README，幾乎都會包含架構圖、流程圖和性能圖表。

視覺化不僅是展示工具，更是思考工具。 當你試圖用圖表描繪系統架構時，你會被迫釐清每個模組之間的關係、數據如何流動、 哪些是關鍵路徑。這個過程本身就是一次深度的架構審視。

三類核心視覺資產

1. 系統架構圖 (Architecture Diagram)

展示系統中所有主要模組及其連接方式。對於遙控車專案，應涵蓋三個層次： 物理層（ESP32、馬達、感測器）、 通訊層（BLE/WebSocket 協議）、以及 介面層（Web Dashboard）。 讓觀看者能在 10 秒內理解數據是如何從搖桿流向馬達的。

2. 通訊協議規格圖 (Protocol Specification)

用圖表展示封包格式、欄位定義與編解碼流程。 這不僅幫助他人理解你的設計，也是團隊協作時不可或缺的文件。

3. 性能實測波形 (Performance Waveform)

用截圖或 GIF 展示 Dashboard 上的即時波形——PID 修正過程、延遲分佈、更新率曲線。 這些是你技術實力的視覺證據，比任何文字描述都更有說服力。

使用 Mermaid 繪製架構圖

Mermaid 是一個基於文字描述的圖表語法， 可以直接嵌入 Markdown 文件中，GitHub 會自動渲染為圖表。 這意味著你的架構圖可以和程式碼一起被版本控制，無需額外的設計工具。

```mermaid
graph LR
  A[📱 Web Dashboard] -->|WebSocket| B[🌐 ESP32 Server]
  B -->|PWM Signal| C[⚙️ Motor Driver]
  C -->|Encoder Feedback| B
  B -->|Telemetry JSON| A
```
視覺工具比較 Mermaid 適合快速繪製流程圖與架構圖，且可嵌入 Markdown； Excalidraw 適合手繪風格的概念圖，視覺親和力高； Figma / draw.io 適合需要精確排版的專業級視覺資產。 建議 README 使用 Mermaid，作品集使用 Figma 導出的高解析度圖片。

視覺資產檢查清單

  • 系統架構圖——標註物理層、通訊層與介面層
  • 通訊封包格式圖——展示 JSON vs 精簡協議的結構
  • PID 修正波形截圖——優化前 vs 優化後的對比
  • 性能對比表格——延遲、更新率、穩定性的量化數據
  • Dashboard 操控 GIF——展示即時控制的流暢度
10 秒測試 完成架構圖後，找一個不了解你專案的同學看 10 秒鐘，然後問他：「數據是怎麼從手機到馬達的？」 如果他能大致說出來，你的圖就成功了。如果不能，代表圖表還需要簡化或加上更清楚的標註。

🎬 3. 實機演示與電梯簡報

在科技業有句話：「Pics or it didn't happen」（沒有影片就等於沒發生）。 無論你的系統有多精密，如果沒有一段展示影片，觀看者就無法感受到那種「指尖滑動搖桿，車子即時回應」的魔力。 一段精心錄製的 Demo 影片，是你技術作品集中最具感染力的資產。

專業的 Demo 影片不是隨手拿手機拍攝就好。它需要經過腳本設計、畫面構圖 和敘事節奏的安排，讓觀看者在 60 秒內就能理解你做了什麼、為什麼厲害、以及你克服了哪些挑戰。 其中，子母畫面 (Picture-in-Picture, PiP) 是展示軟硬體整合專案的黃金格式—— 同一個畫面中同時呈現 Dashboard 的數據動態和車體的實際運動，證明兩者的同步性。

60 秒電梯簡報結構

1

0–10 秒：炸裂開場 (The Hook)

直接展示最令人驚豔的功能。例如：車子在全速行駛時突然斷線，然後在 1 秒內自動煞停。這個畫面比任何介紹語都有衝擊力。

2

10–30 秒：技術亮點 (The Substance)

用 PiP 畫面展示 2–3 個核心技術特點。例如：Dashboard 上的延遲數字穩定在 35ms；PID 波形顯示精確的誤差修正曲線；協議壓縮讓更新率從 12Hz 飆升到 60Hz。

3

30–50 秒：挑戰故事 (The Story)

用旁白簡述開發過程中最大的挑戰與突破。「初版使用 JSON 協議，控制延遲高達 200ms。透過自訂二進位協議，延遲降低了 83%。」

4

50–60 秒：收尾呼籲 (The Close)

總結專案的技術規格並引導觀看者到 GitHub 查看完整程式碼與文件。「完整原始碼與架構文件請見 GitHub，歡迎 Star 與 Fork。」

影片製作注意事項

項目建議做法避免做法
片長60–90 秒，精準不拖沓超過 3 分鐘，觀眾會跳離
開場前 5 秒展示最驚豔的功能用 30 秒介紹自己和背景
畫面PiP 同時展示軟體 + 硬體只拍車子或只錄螢幕
音訊清晰的技術旁白 + 字幕背景噪音或無聲影片
數據畫面上標註關鍵數值只有定性描述（「很快」「很穩」）
常見失誤 最常見的 Demo 影片錯誤是錄太長和沒有重點。 在錄製前先寫好 60 秒的逐句腳本，精確到每 10 秒要展示什麼內容。 用手機的慢動作模式拍攝精確停靠或高速過彎的瞬間，能大幅提升影片的專業感。
PiP 錄製工具 macOS 可使用 OBS Studio（免費）建立子母畫面佈局： 將螢幕擷取（Dashboard）設為主畫面，手機相機透過 Camo 或 EpocCam 串流作為子畫面。錄完後用 iMovie 或 DaVinci Resolve 進行簡單剪輯與字幕添加。

🛠️ 三大核心任務

任務 1：專業級系統架構圖 (Professional Architecture Lab)

  • 要求：在 README.md 中嵌入一張系統架構圖（建議使用 Mermaid 語法）。

任務 2：動態性能證據集 (Evidence-based Portfolio)

  • 要求：
    1. 建立一個 性能對比表，對比優化前（Week 1）與優化後（Final）的延遲、更新率等數據。
    2. 採集 Dashboard 的 即時波形對比圖 (GIF)，展示系統穩定性的提升。
  • 重點：用數據說話，而不僅僅是口頭宣稱「速度很快」。

任務 3：一分鐘技術簡報影片 (Professional Pitch Video)

  • 要求：錄製一段 60 秒的影片，採用 PiP (子母畫面) 格式。
  • 內容：展示 Web Dashboard 的數據動態與機器人實體運動的同步性。
  • 敘事：遵循「挑戰 ➡ 掙扎 ➡ 突破」的敘事框架。

祝你成功封裝出一份令人驚艷的技術作品集！

針對 adv-13-unit-technical-narrative (技術敘事) 單元，這是整門進階課程的 總結與作品集封裝。它要求學員從「代碼撰寫者」晉升為「技術佈道者」，學習如何透過架構圖、性能數據證據以及專業的 Demo 影片，將過去十二週的艱澀研發成果轉化為一眼就能吸引評審或面試官的高質量資產。

以下是在 GitHub Classroom 部署其作業 (Assignments) 的具體建議：

1. 範本程式庫 (Template Repo) 配置建議

作為作品集單元，範本應著重於「結構化展示」，建議範本包含：

  • 📂 assets/images/：圖片目錄。引導學員存放系統架構圖、流程圖以及 PID 修正波形圖。
  • 📂 assets/analysis/：性能白皮書與數據。存放優化前後的 Latency 與數據精簡率對比表。
  • 📂 README.md (Premium Template)：核心作品集模板。預置「英雄之旅」敘事架構、技術亮點標籤、PiP 影片嵌入區位。這將是 GitHub Classroom 的主要提交文件。
  • 📂 MERMAID_GUIDE.md：繪圖指南。提供 Mermaid 語法範例，協助學員快速繪製專業的系統與通訊藍圖。

2. 作業任務部署細節

任務 1：專業級系統架構圖設計 (Professional Architecture Lab)

  • 目標：將抽象的代碼與硬體連線轉化為清晰、具備工業美感的視覺藍圖。
  • Classroom 部署建議：
    • 分層建模要求：學員需在 README 中嵌入一張架構圖。圖中必須標註出「物理層 (PWM/Encoder)」、「自定義通訊層 (Binary/JSON)」與「VUI 反饋層」。
    • 工具應用：推薦使用 Mermaid 直接在 Markdown 中繪製，或導出 Figma/Excalidraw 圖片。
    • 驗證要點：架構圖是否能讓「非開發者」在 10 秒內看懂數據是如何從馬達流向網頁儀表板的？

任務 2：動態性能證據集 (Evidence-based Performance Portfolio)

  • 目標：用「數據與視覺證據」取代口頭宣稱，展現系統的真實實力。
  • Classroom 部署建議：
    • 波形 GIF 採集：要求學員採集 Dashboard 上的即時波形。必須包含一個「優化前（亂跳/緩慢）」與「優化後（平滑/即時）」的對比 GIF。
    • 性能對比表：在 README 中實作一個 Markdown 表格：方案類型 | 延遲 (ms) | 更新率 (Hz) | 抖動率。
    • PR 深度核核：點評其對數據的「註解能力」。是否能指出波形中的某個峰值代表了「積分飽和的抑制」？

任務 3：一分鐘技術簡報影片 (Professional Pitch Video)

  • 目標：結合實機演示與技術口述，打造具備感染力的 Demo。
  • Classroom 部署建議：
    • PiP (子母畫面) 錄製：要求學員錄影。影片左側為 Web Dashboard 的邏輯動態，右側為車體實體運動，展示兩者完美的同步性。
    • 敘事模型應用：腳本需遵循「遇到挑戰（如連線卡頓）➡ 掙扎過程（如協議重構）➡ 最終突破（如極速控制）」的敘事張力。
    • 作品集發布：將影片連結（或 GIF 精華）放在 README 最上方。
    • Push Final：提交最終版作品集 程式庫，這將是其結業的最重要資產。

3. 高級技術文案導師點評標準 (Engineering Narrative Benchmarks)

此單元的價值在於 「對技術實力的視覺轉譯與說服力」：

  • [ ] 視覺透明度：架構圖與波形圖是否能準確反映代碼的素質與複雜度？
  • [ ] 敘事張力： README 是否能讓觀看者快速入戲，理解該專案解決了多少硬核問題？
  • [ ] 專業細節：所有的命名、表格與圖表標註是否具備工業標準的精確感？
