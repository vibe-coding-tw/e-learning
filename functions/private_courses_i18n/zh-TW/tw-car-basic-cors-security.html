簡介

當網頁嘗試存取不同網域的資源時，瀏覽器會啟動 CORS（跨來源資源共用） 安全機制進行攔截。 本單元將帶你理解同源政策（Same-Origin Policy）的判定邏輯、預檢請求（OPTIONS Preflight）的敲門機制， 以及如何在 ESP32 伺服器端正確注入安全標頭，從根本上解決跨域封鎖問題。

情境
你開發了一個漂亮的遙控介面網頁，在瀏覽器中打開後點擊「連線」按鈕，Console 卻跳出一整片紅字： 「Access to fetch at 'http://192.168.4.1' has been blocked by CORS policy」。 ESP32 端明明收到了請求，但瀏覽器卻拒絕讀取回應——你需要從伺服器端配置正確的安全標頭來打通這道隱形牆。

什麼是 CORS？

CORS（Cross-Origin Resource Sharing，跨來源資源共用）是一套由瀏覽器執行的安全機制， 用來控制「網頁是否被允許讀取來自不同來源（Origin）的資源」。當你在 localhost 的網頁中 嘗試用 fetch() 連接 ESP32 的 IP 位址時，瀏覽器會先檢查 CORS 規則——如果伺服器沒有回傳正確的授權標頭， 瀏覽器就會直接封鎖這次請求，即使 ESP32 端已經正確處理並回傳了數據。

同源政策（Same-Origin Policy）

要理解 CORS，首先要認識它的基礎——同源政策（SOP）。這是瀏覽器最重要的安全防線之一， 規定網頁腳本只能讀取同源的資源。所謂「來源（Origin）」由三個要素組成：

  • 協議（Protocol）：http 或 https
  • 網域（Host）：example.com 或 192.168.4.1
  • 埠號（Port）：80、443、3000 等

只要三個要素中任何一個不同，就會被判定為「跨來源（Cross-Origin）」，觸發 CORS 機制。

來源比對實例

以下表格列出常見的來源比對情境，幫助你快速判斷哪些情況會觸發 CORS：

來源 A（網頁所在） 來源 B（請求目標） 結果 原因
http://vibe.car/a.js http://vibe.car/b.js 同源 協議、網域、埠號皆相同
https://vibe.car http://vibe.car 跨域 協議不同（https vs http）
http://localhost:3000 http://192.168.4.1 跨域 網域不同（localhost vs IP）
http://vibe.car:80 http://vibe.car:8080 跨域 埠號不同（80 vs 8080）
注意 CORS 限制是瀏覽器端的行為。像 Postman、curl 或 ESP32 自身發出的請求不會受 CORS 影響。 這就是為什麼「同樣的 API，Postman 能用但瀏覽器不行」——問題不在伺服器，而在瀏覽器的安全策略。

為什麼需要同源政策？

同源政策的存在是為了保護使用者。想像以下場景：你登入了銀行網站 bank.com， 然後不小心打開了一個惡意網站 evil.com。如果沒有同源政策，evil.com 的 JavaScript 就能直接讀取你在 bank.com 上的登入狀態、帳戶餘額，甚至發起轉帳請求。 同源政策正是阻止這種「跨站資料竊取」的第一道防線。

警告 在開發 ESP32 控制面板時，你的網頁（通常在 localhost 或從檔案系統開啟）與 ESP32 伺服器（http://192.168.x.x） 天生就是不同來源。如果不在 ESP32 端配置 CORS 標頭，瀏覽器將會封鎖所有的 API 請求。

預檢請求：敲門機制

當瀏覽器偵測到一個跨域請求屬於「複雜請求」時，不會直接發送，而是會先自動發送一個 OPTIONS 預檢請求（Preflight Request） 來「敲門」——詢問伺服器：「我可以用這個方法、帶這些標頭來存取你的資源嗎？」只有伺服器回應「准許」之後， 瀏覽器才會發送真正的請求。這就是 CORS 的核心安全機制。

簡單請求 vs 複雜請求

瀏覽器會根據請求的內容自動判斷它是「簡單請求」還是「複雜請求」。只有複雜請求才會觸發預檢：

類型 條件 是否預檢
簡單請求 方法為 GET/HEAD/POST，且只使用 Accept、Content-Language、Content-Type（僅限 text/plain、multipart/form-data、application/x-www-form-urlencoded）等安全標頭 否，直接發送
複雜請求 使用 PUT/DELETE 方法，或帶有自訂標頭，或 Content-Type 為 application/json 是，先預檢
重要 在物聯網開發中，當你的網頁透過 fetch() 發送帶有 Content-Type: application/json 標頭的 POST 請求到 ESP32 時， 這會被視為複雜請求，瀏覽器會自動先發送 OPTIONS 預檢。如果 ESP32 沒有正確處理 OPTIONS 請求， 你真正的 POST 請求永遠不會送出。

預檢流程詳解

完整的預檢流程分為以下步驟：

1

瀏覽器自動發送 OPTIONS 請求

瀏覽器攔截你的 fetch() 呼叫，先發送一個 OPTIONS 方法的請求到目標伺服器，附帶 Access-Control-Request-Method 和 Access-Control-Request-Headers 標頭。

2

伺服器回傳授權標頭

伺服器收到 OPTIONS 後，回傳 Access-Control-Allow-Origin、Access-Control-Allow-Methods、Access-Control-Allow-Headers 等標頭，告知瀏覽器哪些操作被允許。

3

瀏覽器驗證並發送實際請求

瀏覽器比對伺服器的回應，若符合規範，才會發送你原本的 POST/PUT 請求。若不符合，瀏覽器會直接封鎖並在 Console 顯示紅字錯誤。

在 ESP32 上配置 CORS 標頭

使用 ESPAsyncWebServer 函式庫時，你可以透過 DefaultHeaders 全域注入 CORS 標頭， 讓所有回應都自動攜帶授權資訊：

// 在 setup() 中，於 server.begin() 之前配置
DefaultHeaders::Instance().addHeader(
  "Access-Control-Allow-Origin", "*"
);
DefaultHeaders::Instance().addHeader(
  "Access-Control-Allow-Methods", "GET,POST,OPTIONS"
);
DefaultHeaders::Instance().addHeader(
  "Access-Control-Allow-Headers", "Content-Type"
);

// 處理 OPTIONS 預檢請求
server.on("/*", HTTP_OPTIONS,
  [](AsyncWebServerRequest *req) {
    req->send(204); // 204 No Content
});
提示 你可以在瀏覽器 DevTools 的 Network 面板中觀察預檢請求。成功的預檢會顯示為一個獨立的 OPTIONS 請求， 狀態碼為 204 或 200，緊接著才是你真正的 POST 或 GET 請求。

硬體安全防禦

正確配置 CORS 標頭只是第一步。在真實的物聯網部署環境中，你的 ESP32 可能直接暴露在區域網路甚至公共網路中， 任何知道 IP 位址的人都可以嘗試發送請求。因此，CORS 只是前端瀏覽器的防線，後端（ESP32）自身也需要建立額外的安全機制。

CORS 的安全邊界

一個常見的誤解是：「只要設好 CORS，我的 ESP32 就安全了。」事實上，CORS 只保護瀏覽器中的使用者， 防止惡意網站透過瀏覽器發送未授權的跨域請求。但以下情況不受 CORS 保護：

  • 使用 Postman、curl 等工具直接發送的請求（繞過瀏覽器）
  • 其他 ESP32 或物聯網設備發送的請求
  • 伺服器端的程式碼（Node.js、Python 後端）發送的請求
警告 CORS 不是伺服器的安全機制，而是瀏覽器的安全機制。千萬不要依賴 CORS 來保護你的 API。 即使設置了嚴格的 Access-Control-Allow-Origin，攻擊者仍然可以用非瀏覽器工具直接存取你的 ESP32 端點。

防禦性設計原則

在部署物聯網設備時，你應該遵循以下安全設計原則，在 CORS 之外建立多層防禦：

原則 說明 ESP32 實作方式
權限最小化 不使用萬用符號 *，只開放必要的來源與方法 指定具體的 Origin，如 http://192.168.4.2
輸入驗證 驗證所有傳入的參數，防止注入攻擊 檢查 JSON 欄位的型別與範圍
Token 認證 要求請求攜帶密鑰或令牌，驗證發送者身分 在 Header 中加入自訂的 API Key
資料遮蔽 敏感資訊封裝在 Body 中，不放在 URL 使用 POST + JSON Body 傳送密碼與設定
速率限制 限制單位時間內的請求次數，防止暴力攻擊 用 millis() 追蹤請求頻率

安全配置範例：限定來源

在開發階段，使用 * 萬用符號可以讓任何來源的網頁存取你的 ESP32，方便快速測試。 但在正式部署時，你應該將 Origin 限定為特定的控制台網頁位址：

// 開發環境：允許所有來源（方便測試）
DefaultHeaders::Instance().addHeader(
  "Access-Control-Allow-Origin", "*"
);

// 正式環境：只允許特定來源
DefaultHeaders::Instance().addHeader(
  "Access-Control-Allow-Origin",
  "http://192.168.4.2" // 只有控制台 IP 可存取
);
注意 當你使用特定 Origin 取代 * 時，如果需要支援多個來源，你需要在程式碼中動態判斷請求的 Origin 標頭， 並根據白名單回傳對應的值。因為 Access-Control-Allow-Origin 標頭只能設定一個值或 *。

CORS 安全檢核清單

在部署你的 ESP32 物聯網專案前，請確認以下安全項目：

  • Access-Control-Allow-Origin 已從 * 改為特定 Origin
  • Access-Control-Allow-Methods 只開放實際需要的 HTTP 方法
  • ESP32 端有處理 OPTIONS 預檢請求並回傳 204
  • 敏感操作（如韌體更新、重啟）已加入 Token 驗證
  • 所有傳入的參數都有進行型別與範圍驗證
提示 在開發期間，你可以使用瀏覽器的 CORS 繞過插件（如 Allow CORS）來暫時跳過限制，加速測試。 但請務必記住，這只是開發時的權宜之計——正式上線前，一定要從 ESP32 端正確配置 CORS 標頭，而非依賴使用者安裝插件。

    • 驗證要點：提交一份風險分析，展現對網路安全邊界的理解。

任務 3：「硬體安全官」ESP32 標頭注入 (Hardware Sentry Lab)

  • 目標：實作底層標頭注入技術，徹底從伺服器端解決跨域需求，這是開發物聯網控制台的工業標準。
  • Classroom 部署建議：
    • 標頭實作：使用 DefaultHeaders::Instance().addHeader("Access-Control-Allow-Origin", "*");（或其他特定 Origin）。
    • OPTIONS 預檢處理：關鍵技術點。實作一個 Handler 專門回傳成功的 OPTIONS 請求（HTTP 200/204），否則複雜請求（如帶 Header 的 POST）依然會失敗。
    • 閉環驗證：關閉瀏覽器插件，確認 index.html 依然能流暢控制硬體。
    • 最終 Commit：Push 具備完整標頭配置的韌體專業版本。

3. 網路安全與標頭合規點評標準 (Security Headers & Preflight Standards)

此單元的價值在於 「對 Web 規格的尊重與對伺服器端配置的精確掌控」：

  • [ ] 標頭配置精確度：Access-Control-Allow-Origin 是否拼寫正確且放置在正確的 Handler 中？
  • [ ] 預檢請求處理：是否理解並實作了 OPTIONS 方法的嚮應？（這是跨域封鎖最常見的死穴）。
  • [ ] 防禦性意識：是否理解並非所有 CORS 標頭都應設為 *，並具備針對特定域名開放的維護意識？
